🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — now 6 months old (continuing from Project 1)
Situation:
→ 180 days of CSV files sitting in S3 (from Project 1 upload pipeline)
→ s3://quickcart-raw-data-prod/daily_exports/orders/orders_YYYYMMDD.csv
→ s3://quickcart-raw-data-prod/daily_exports/customers/customers_YYYYMMDD.csv
→ s3://quickcart-raw-data-prod/daily_exports/products/products_YYYYMMDD.csv
→ ~180 files per entity → ~540 total files → ~11 GB total

Marketing Manager asks:
"How many orders had status='completed' last month?
 What's the total revenue by product category?"

Current options the team is considering:
→ Option A: Download all CSVs, open in Excel → crashes at 1M+ rows
→ Option B: Load into MariaDB on EC2 → takes hours, disk almost full (Project 8-9)
→ Option C: Spin up Redshift cluster → $0.25/hr minimum, overkill for ad-hoc queries
→ Option D: ??? Something that queries S3 directly with SQL

CTO: "I don't want to maintain another database just for ad-hoc questions.
      Can we just SQL these files WHERE THEY SIT?"

YOU: "Yes. That's exactly what Amazon Athena does."

🟡 HOW TO APPROACH IT

Step 1: Understand what Athena IS
→ Serverless SQL query engine
→ Based on Presto (open-source distributed SQL engine by Facebook)
→ Points at S3 → reads files → returns SQL results
→ No infrastructure to manage — no servers, no clusters, no loading

Step 2: Understand what Athena NEEDS to work
→ It needs to know the SCHEMA of the files (columns, types)
→ It needs to know the FORMAT of the files (CSV, JSON, Parquet, ORC)
→ It needs to know the LOCATION in S3 (which prefix)
→ All of this is stored in the AWS Glue Data Catalog (Project 15)

Step 3: Understand the cost model
→ $5.00 per TB of data SCANNED
→ Not per query — per bytes read from S3
→ 11 GB scanned = $0.055 (5.5 cents)
→ Cancelled queries still charged for data already scanned
→ Failed queries with zero scan = $0.00

Step 4: Understand the flow
→ User writes SQL in Athena console
→ Athena reads table definition from Glue Data Catalog
→ Athena scans files in S3 matching the table's location
→ Athena processes, aggregates, filters
→ Results stored in a separate S3 output location
→ Results displayed in console or returned via API

Step 5: Plan our approach
→ Create a Glue Catalog Database (logical namespace)
→ Create Glue Catalog Tables (schema definition pointing to S3)
→ Configure Athena output location
→ Run SQL queries
→ No data movement — files stay in S3 untouched

🔵 WHAT COULD BE DONE

Option A: Create tables manually via Athena DDL (CREATE TABLE)

→ Full control over schema definition
→ Best when you KNOW the exact schema
→ ✅ WE USE THIS — teaches fundamentals

Option B: Use Glue Crawler to auto-discover schema (Project 23)
→ Crawler reads sample files, infers schema automatically
→ Great for unknown/changing schemas
→ Sometimes infers wrong types (string vs int)
→ Covered in Project 23

Option C: Use Athena CTAS to create optimized copies (Project 36)
→ CREATE TABLE AS SELECT — transforms data during creation
→ Converts CSV to Parquet, adds partitions
→ Covered in Project 36

Option D: Use Athena Federated Query to query databases directly (Project 46)
→ Query MariaDB, Redshift, DynamoDB via Lambda connectors
→ Covered in Project 46

DECISION: Option A — manual DDL
→ Teaches exactly what Athena needs and why
→ No magic — every column, type, and setting is explicit

✅ WHAT SHOULD DO

1. Create a SEPARATE S3 bucket (or prefix) for Athena query results
   → Athena writes result CSVs to S3 after every query
   → Never mix results with source data
   → Set lifecycle policy: delete results after 7 days (they pile up fast)

2. Use a Workgroup to control costs
   → Set per-query data scan limit (e.g., 1 GB)
   → Prevents accidental full-table scan on 10 TB dataset
   → Tracks query history per team/user

3. Define column types precisely
   → "quantity" should be INT, not STRING
   → Wrong types = wrong aggregations (SUM of strings = error)
   → Athena reads files lazily — bad types cause runtime failures, not DDL errors

4. Use skip.header.line.count for CSVs with headers
   → Without this: header row "order_id" becomes a data row
   → Silent data corruption — Athena won't error, just wrong results

5. Use backticks for reserved words as column names
   → `status`, `date`, `order` are SQL reserved words
   → CREATE TABLE with bare "status" → syntax error

6. Match SerDe to file format exactly
   → CSV → OpenCSVSerDe or LazySimpleSerDe
   → JSON → org.openx.data.jsonserde.JsonSerDe
   → Parquet → org.apache.hadoop.hive.ql.io.parquet (auto-detected)
   → Wrong SerDe = garbled data or query failure

7. Point LOCATION to the PREFIX, not individual files
   → LOCATION 's3://bucket/daily_exports/orders/'
   → Athena scans ALL files under that prefix
   → Adding new files = instant availability (no re-registration)

❌ WHAT SHOULD NOT DO

1. DO NOT point Athena output to the same prefix as source data
   → Athena writes .csv and .csv.metadata files
   → Next query scans its own results as if they were source data
   → Creates infinite feedback loop of garbage results

2. DO NOT forget skip.header.line.count for CSV
   → Header row becomes data row silently
   → "order_id" appears as a customer name
   → Aggregations like SUM("unit_price") fail on string "unit_price"

3. DO NOT use SELECT * on large datasets without LIMIT
   → Scans ENTIRE dataset every time
   → 1 TB scan = $5.00 per query
   → 10 analysts running SELECT * = $50/day

4. DO NOT assume Athena validates data on table creation
   → CREATE TABLE succeeds even if S3 location is empty
   → CREATE TABLE succeeds even if schema doesn't match files
   → Errors appear only at QUERY TIME

5. DO NOT ignore Athena query result accumulation
   → Every query generates result files in output bucket
   → 100 queries/day × 30 days = 3,000 result files
   → Set S3 lifecycle to auto-delete after 7 days

6. DO NOT use Athena for low-latency (<1 second) queries
   → Minimum query time: ~2-3 seconds (engine startup overhead)
   → Not designed for application-facing queries
   → Use Redshift, DynamoDB, or ElastiCache for low latency

7. DO NOT create tables in the "default" database
   → Messy — everything from every project piled together
   → Always create a dedicated database per domain/project

🚫 WHEN NOT TO USE ATHENA

1. Repeated queries on same data (>20 queries/day on same tables)
   → Every query re-scans S3 from scratch
   → No persistent indexes, no caching of data (only results cached 
     if enabled)
   → Cost: $5/TB × 20 queries × 100 GB = $10/day = $300/month
   → Cheaper: Load into Redshift, query unlimited times

2. Sub-second query latency requirements
   → Athena cold start: 2-5 seconds minimum
   → Complex queries on large data: 30-300 seconds
   → Use: Redshift, DynamoDB, ElastiCache

3. UPDATE/DELETE operations
   → Athena is read-only on source data (with Iceberg exception)
   → Cannot: UPDATE orders SET status='shipped' WHERE id=123
   → Use: MariaDB (Project 9), Redshift (Project 10)

4. Streaming / real-time data
   → Athena queries are batch — submit, wait, get results
   → No continuous query or streaming window
   → Use: Kinesis Analytics, Flink (Project 72)

5. Data < 1 MB
   → Minimum charge: 10 MB per query regardless of actual scan
   → Querying a 1 KB file costs same as 10 MB file
   → At tiny scale: just download and use pandas

6. Highly concurrent workload (>20 simultaneous queries)
   → Default: 20-30 concurrent DML queries per account per region
   → Can increase via service quota request
   → But Athena is not designed for 1000-user dashboard backend
   → Use: Redshift with concurrency scaling

┌─────────────────────────────────────────────────────────────────────────────┐
│                     ATHENA INTERNAL ARCHITECTURE                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  LAYER 1: QUERY SUBMISSION                                                  │
│  ─────────────────────────                                                  │
│  User submits SQL → Athena API (StartQueryExecution)                        │
│  → Query gets a unique QueryExecutionId                                     │
│  → Query enters queue in the Athena engine                                  │
│                                                                             │
│  LAYER 2: QUERY PLANNING                                                    │
│  ─────────────────────────                                                  │
│  Athena reads table metadata from Glue Data Catalog:                        │
│  → Column names and types                                                   │
│  → SerDe (serializer/deserializer) — how to parse the file format           │
│  → S3 LOCATION — where to find the data files                               │
│  → Partition info (if any) — which S3 prefixes to scan                      │
│                                                                             │
│  Query planner creates execution plan:                                      │
│  → Which files to scan                                                      │
│  → Which columns to read (column projection)                                │
│  → Which filters to push down (predicate pushdown — Parquet/ORC only)       │
│  → How to distribute work across worker nodes                               │
│                                                                             │
│  LAYER 3: EXECUTION ENGINE (Trino/Presto)                                   │
│  ─────────────────────────────────────────                                   │
│  Athena v2: Presto 0.217                                                    │
│  Athena v3: Trino (Presto fork, current)                                    │
│                                                                             │
│  Execution flow:                                                            │
│  ┌─────────────┐                                                            │
│  │ Coordinator │ ← Parses SQL, builds plan, assigns splits                  │
│  └──────┬──────┘                                                            │
│         │ distributes work                                                  │
│    ┌────┴─────┬──────────┬──────────┐                                       │
│    ▼          ▼          ▼          ▼                                        │
│  Worker 1   Worker 2   Worker 3   Worker N                                  │
│  (reads     (reads     (reads     (reads                                    │
│   files     files      files      files                                     │
│   1-10)     11-20)     21-30)     31-N)                                     │
│    │          │          │          │                                        │
│    └────┬─────┴──────────┴──────────┘                                       │
│         ▼                                                                   │
│  Coordinator merges partial results                                         │
│  → Final aggregation                                                        │
│  → Sorting (if ORDER BY)                                                    │
│  → Writes result to S3 output location                                      │
│                                                                             │
│  LAYER 4: SerDe — THE KEY TO FORMAT UNDERSTANDING                           │
│  ─────────────────────────────────────────────────                           │
│                                                                             │
│  SerDe = Serializer / Deserializer                                          │
│  → Tells Athena HOW to read each file format                                │
│                                                                             │
│  ┌─────────────────┬───────────────────────────────────────────┐            │
│  │ File Format     │ SerDe Used                                │            │
│  ├─────────────────┼───────────────────────────────────────────┤            │
│  │ CSV (simple)    │ LazySimpleSerDe                           │            │
│  │ CSV (quoted)    │ OpenCSVSerDe (handles quotes, escapes)    │            │
│  │ JSON            │ org.openx.data.jsonserde.JsonSerDe        │            │
│  │ Parquet         │ org.apache.hadoop.hive.ql.io.parquet      │            │
│  │ ORC             │ org.apache.hadoop.hive.ql.io.orc          │            │
│  │ TSV             │ LazySimpleSerDe with '\t' delimiter       │            │
│  │ CloudTrail logs │ com.amazon.emr.hive.serde.CloudTrailSerde│            │
│  └─────────────────┴───────────────────────────────────────────┘            │
│                                                                             │
│  OpenCSVSerDe specifics (used in our project):                              │
│  → ALL columns treated as STRING internally                                 │
│  → Athena casts to declared types at query time                             │
│  → Handles: quoted fields, escaped commas, newlines in fields               │
│  → Does NOT support: arrays, maps, structs (use LazySimpleSerDe)            │
│                                                                             │
│  LAYER 5: COLUMN PROJECTION & PREDICATE PUSHDOWN                           │
│  ─────────────────────────────────────────────────                           │
│                                                                             │
│  CSV files:                                                                 │
│  → Athena must read ENTIRE row even for SELECT single_column                │
│  → CSV is row-oriented — cannot skip columns physically                     │
│  → Projection happens AFTER reading — saves nothing on scan cost            │
│                                                                             │
│  Parquet/ORC files (Project 25):                                            │
│  → Columnar format — each column stored separately                          │
│  → SELECT single_column reads ONLY that column's bytes                      │
│  → 100 columns but need 2 → reads 2% of data → 98% cost savings            │
│  → Predicate pushdown: WHERE filters applied at storage level               │
│  → Parquet footer contains min/max stats per column per row group           │
│  → Athena skips row groups where filter cannot match                        │
│                                                                             │
│  THIS IS WHY PARQUET > CSV FOR ATHENA (Project 25)                          │
│                                                                             │
│  LAYER 6: QUERY RESULT OUTPUT                                               │
│  ────────────────────────────                                                │
│  → Results written as CSV to: s3://output-bucket/query-id.csv               │
│  → Metadata file: s3://output-bucket/query-id.csv.metadata                  │
│  → Max result size: 2 GB (per query)                                        │
│  → If result > 2 GB → query fails → use CTAS instead (Project 36)          │
│                                                                             │
│  LAYER 7: PRICING MECHANICS                                                 │
│  ─────────────────────────                                                  │
│  → Charged per BYTES SCANNED from S3                                        │
│  → $5.00 per TB scanned                                                     │
│  → Minimum charge: 10 MB per query                                          │
│  → DDL (CREATE TABLE, SHOW TABLES): FREE                                    │
│  → Failed queries: charged for data scanned before failure                  │
│  → Cancelled queries: charged for data scanned before cancel                │
│  → Compressed files: charged on COMPRESSED size (not decompressed)          │
│  → Columnar (Parquet/ORC): charged ONLY for columns read                   │
│                                                                             │
│  COST FORMULA:                                                              │
│  cost = (bytes_scanned / 1_099_511_627_776) × $5.00                        │
│  cost = max(cost, $0.00000476)  ← 10 MB minimum                            │
│                                                                             │
│  EXAMPLE:                                                                   │
│  → 11 GB CSV, SELECT * → scans 11 GB → $0.055                              │
│  → 11 GB CSV, SELECT single_col → STILL scans 11 GB → $0.055 (CSV!)        │
│  → 11 GB Parquet, SELECT single_col (of 10 cols) → scans ~1.1 GB → $0.0055│
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "All my numeric columns show NULL"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: OpenCSVSerDe treats everything as STRING internally
→ If you declare INT but CSV has "12.99" → cast fails → NULL
→ Fix: declare DOUBLE for decimals, or use STRING and CAST in queries
→ Deeper fix: use LazySimpleSerDe which handles type casting better

PITFALL 2: "First row of results is the CSV header"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: forgot 'skip.header.line.count' = '1'
→ Fix: add to TBLPROPERTIES in CREATE TABLE
→ Impact: header "order_id" becomes a row → COUNT is off by 1 per file
→ 180 files = count inflated by 180

PITFALL 3: "LOCATION points to wrong prefix — query returns 0 rows"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: typo in S3 path or missing trailing slash
→ LOCATION must end with "/" → 's3://bucket/prefix/'
→ Without trailing slash: Athena may not find files
→ No error at CREATE TABLE time — only empty results at query time

PITFALL 4: "Query scans way more data than expected"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: CSV format → must read entire row regardless of columns selected
→ Also: no partitioning → scans ALL files even if you need only 1 day
→ Fix: convert to Parquet (Project 25) + add partitions (Project 26)

PITFALL 5: "Athena result bucket fills up with thousands of files"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Every query = 2 files (result.csv + result.csv.metadata)
→ 50 queries/day × 30 days = 3,000 files
→ Fix: S3 lifecycle policy → delete objects after 7 days (Project 4 concept)

PITFALL 6: "Column order mismatch — wrong data in wrong columns"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ CSV is POSITIONAL — column 1 in DDL maps to column 1 in file
→ If CREATE TABLE has (order_id, customer_id, product_id)
→ But CSV has (customer_id, order_id, product_id)
→ Data silently maps wrong → order_id contains customer_id values
→ No error, just wrong answers
→ Fix: always verify with SELECT * LIMIT 5 and visually check

PITFALL 7: "HIVE_CURSOR_ERROR or HIVE_BAD_DATA"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: data doesn't match declared schema
→ Example: column declared INT but file contains "N/A" in that column
→ Fix: declare as STRING, clean in query with TRY_CAST()
→ Or: clean data upstream before landing in S3

PITFALL 8: "Query works on one file but fails on 180 files"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: one file has different column count or encoding
→ File 87 has 7 columns, rest have 8 → breaks parsing
→ Fix: validate file structure before upload (Project 1 concept)
→ Debug: use $path pseudo-column to find which file has the issue
   SELECT "$path", COUNT(*) FROM orders GROUP BY 1

📈 SCALABILITY CONSIDERATIONS

DATA SIZE SCALING:
━━━━━━━━━━━━━━━━━
→ 10 GB: Athena handles in 3-10 seconds (CSV)
→ 100 GB: 15-60 seconds (CSV), 3-10 seconds (Parquet)
→ 1 TB: 1-5 minutes (CSV), 15-60 seconds (Parquet partitioned)
→ 10 TB: Athena still works but COST matters → $50 per full scan
→ 100 TB: Must have partitioning + Parquet + column pruning

COST SCALING:
━━━━━━━━━━━━━
→ 1 analyst, 5 queries/day on 11 GB CSV:
  11 GB × 5 × $5/TB = $0.275/day = $8.25/month ✅ Cheap

→ 10 analysts, 20 queries/day on 500 GB CSV:
  500 GB × 20 × $5/TB = $50/day = $1,500/month ⚠️ Expensive

→ Same 500 GB converted to Parquet (5:1 compression):
  100 GB × 20 × $5/TB = $10/day = $300/month ✅ 80% savings

→ Same + partitioning (only scan 1 day of 365):
  (100 GB / 365) × 20 × $5/TB = $0.027/day = $0.82/month ✅ 99.9% savings

CONCURRENCY SCALING:
━━━━━━━━━━━━━━━━━━━━
→ Default: 20-30 concurrent DML queries per account per region
→ DDL queries (CREATE, DROP): separate limit, ~20 concurrent
→ Can request increase to 100+ via AWS support

   → Every Athena API call logged

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ Athena = serverless SQL engine that queries S3 files in-place
→ No servers, no loading, no infrastructure — just SQL
→ Built on Trino (Presto fork) — distributed query engine
→ Schema stored in Glue Data Catalog — Athena has NO metastore of its own
→ Cost = $5 per TB scanned — pay only for what you read
→ EXTERNAL TABLE = metadata only — DROP TABLE never deletes S3 data
→ CSV is row-oriented → Athena scans ENTIRE file regardless of columns selected
→ OpenCSVSerDe reads everything as STRING internally → casts at query time
→ skip.header.line.count is CRITICAL for CSVs with headers
→ Workgroups control costs with per-query scan limits
→ $path pseudo-column reveals which S3 file each row came from
→ DDL operations (CREATE/DROP) are FREE — only DML (SELECT) costs money
