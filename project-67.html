🟡 HOW TO APPROACH IT

Step 1: Understand Redshift Serverless vs Provisioned

  PROVISIONED (current):
  → Fixed cluster: N nodes × node_type
  → Pay per HOUR regardless of usage
  → You manage: node type, count, WLM, VACUUM, resize
  → Scaling: manual resize or elastic resize (minutes)
  → Cost predictable but wasteful for variable workloads

  SERVERLESS:
  → No cluster to manage — AWS handles infrastructure
  → Pay per RPU-HOUR (Redshift Processing Unit)
  → Auto-scales: 8 RPU base → up to 512 RPU under load
  → Scales DOWN to near-zero when idle
  → Storage: separate billing ($0.024/GB/month — same as RA3)
  → No node types, no WLM configuration, no resize operations

  RPU EXPLAINED:
  → RPU = Redshift Processing Unit (abstract compute unit)
  → 1 RPU ≈ roughly equivalent to 1 slice of a dc2.large node
  → Base capacity: 8 RPU minimum (configurable 8-512)
  → Max capacity: 512 RPU (configurable)
  → Scales automatically based on query complexity and concurrency
  → Billing: per second, minimum 60 seconds per query
  → Idle: scales to base RPU, charges only base RPU-seconds

   → "We'll save money by deleting immediately"
   → If Serverless has compatibility issues: no fallback
   → Always: parallel run for 30 days → then decommission

2. DO NOT set base RPU too high
   → Base RPU = minimum you pay even when idle
   → Base 128 RPU: $48/hr idle = $1,152/day = WORSE than provisioned
   → Start with base 8 RPU → auto-scales up when needed
   → Increase base only if cold-start latency is unacceptable

3. DO NOT skip usage limits
   → Runaway query (SELECT * FROM 500M rows, no WHERE): consumes 200+ RPU
   → 200 RPU × 1 hour = $75 for ONE query
   → Usage limits: cap at budget → prevent cost explosions
   → Set both per-query and per-day limits

4. DO NOT assume all Redshift features work on Serverless
   → Serverless supports MOST features but check:
     - Stored procedures: ✅ supported
     - Materialized views: ✅ supported
     - Spectrum: ✅ supported
     - Data sharing: ✅ supported
     - Concurrency scaling: ❌ not applicable (auto-scales natively)
     - WLM queues: ❌ not applicable (auto-manages)
     - AQUA: ❌ not applicable
     - Elastic resize: ❌ not applicable
     - Snapshot scheduling: ✅ via recovery points

5. DO NOT use same JDBC connection string
   → Provisioned endpoint ≠ Serverless endpoint
   → If you just change the endpoint: port is still 5439 ✅
   → But: hostname format is completely different
   → Must update: Glue connections, Superset config, Lambda env vars

6. DO NOT forget to recreate Redshift users and groups
   → Snapshot preserves users if restoring from same account
   → But: password policies may differ
   → IAM-based auth: must update trust policies for Serverless
   → Verify: all service accounts can authenticate after migration

7. DO NOT run VACUUM on Serverless manually
   → Serverless handles VACUUM automatically
   → Manual VACUUM wastes RPU-hours (= money)
   → Same for ANALYZE — automatic on Serverless

8. DO NOT panic about cold-start latency
   → First query after idle period: may take 10-30 seconds (cold start)
   → Serverless spins up compute → then runs query
   → Subsequent queries: normal speed
   → Mitigation: schedule a lightweight query at 7:55 AM to "warm up"

🚫 WHEN NOT TO USE SERVERLESS

1. When workload is consistently heavy (> 8 hrs/day active queries)
   → Provisioned with RI is cheaper at sustained usage
   → Break-even: ~6-8 hrs/day depending on RPU consumption
   → Above break-even: Serverless costs MORE

2. When cold-start latency is unacceptable
   → Real-time dashboard requiring < 1 second response
   → Cold start: 10-30 seconds for first query after idle
   → Provisioned: always warm, consistent latency
   → Mitigation: warm-up query or higher base RPU (adds cost)

3. When you need WLM queue control
   → Provisioned WLM: dedicate queues to different user groups
   → Serverless: no WLM — auto-manages concurrency
   → If strict resource isolation required: provisioned + WLM

4. When budget requires ABSOLUTE predictability
   → Provisioned: exact monthly cost (node × hours)
   → Serverless: variable based on usage (even with limits)
   → CFO who demands exact budget: provisioned + RI

5. When using Redshift-specific features not in Serverless
   → Check latest AWS documentation — feature gap closing rapidly
   → As of early 2025: most features supported
   → Edge cases: some older cluster parameters, custom parameter groups

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Paused cluster won't resume in time for ETL"
→ Cause: Resume takes 30-120 seconds, but ETL starts at exactly 2:00 AM
→ EventBridge schedules resume at 1:55 AM → ETL at 2:00 AM → 5 min buffer
→ But: occasionally resume takes 3 minutes → ETL starts before cluster ready
→ Fix: resume at 1:30 AM (30 min buffer) OR Step Function waits for cluster status

PITFALL 2: "Serverless cold start: dashboard takes 30 seconds"
→ Cause: no queries for 2 hours → Serverless scaled down
→ First query: allocate compute → 10-30 second delay
→ Fix: "warm-up" scheduled query at 7:55 AM (SELECT 1)
→ Or: increase base RPU (8→32) — costs more but always warm

PITFALL 3: "Serverless bill is $2,000 — expected $200"
→ Cause: no usage limits set → runaway query consumed 500 RPU for 2 hours
→ Or: analyst ran SELECT * FROM 500M row table with no WHERE
→ Fix: ALWAYS set usage limits before any user connects
→ Per-query timeout: 300 seconds max
→ Daily RPU-hour limit: 100 RPU-hours
→ Monthly RPU-hour limit: 2000 RPU-hours

PITFALL 4: "Glue connection fails after Serverless migration"
→ Cause: Glue JDBC connection still points to old provisioned endpoint
→ Provisioned: cluster-name.xxxx.us-east-2.redshift.amazonaws.com
→ Serverless: workgroup-name.account-id.us-east-2.redshift-serverless.amazonaws.com
→ Fix: update Glue connection JDBC URL
→ Also: update Lambda env vars, dashboard configs, any hardcoded endpoints

PITFALL 5: "Stored procedure works on provisioned, fails on Serverless"
→ Cause: some PL/pgSQL features have subtle differences
→ Most procedures work identically
→ Edge cases: RAISE with specific error codes, cursor behaviors
→ Fix: test all stored procedures on Serverless before migration

PITFALL 6: "Materialized Views missing after snapshot restore"
→ Cause: some MV types may not transfer cleanly via snapshot
→ Fix: keep MV creation SQL (Project 63) → run after restore
→ Verify: SELECT * FROM STV_MV_INFO after restore

PITFALL 7: "Resume at 1:30 AM on Monday but no ETL needed until 2 AM"
→ Cause: paying for 30 minutes of idle compute unnecessarily
→ $0.50/hr × 0.5 hr × 22 weekdays = $5.50/month — negligible
→ Trade-off: $5.50/month insurance vs risk of ETL failure
→ Keep the buffer

PITFALL 8: "Serverless can't be paused like provisioned"
→ CORRECT — Serverless doesn't have pause/resume
→ It auto-scales including down to near-zero
→ But "near-zero" still means base RPU charges when active
→ To truly stop billing: delete the workgroup (data in namespace persists)
→ For dev/test: delete workgroup Friday → recreate Monday

PITFALL 9: "Cross-account Glue connection fails on Serverless"
→ Cause: Serverless uses different IAM actions
→ Provisioned: redshift:GetClusterCredentials
→ Serverless: redshift-serverless:GetCredentials
→ IAM policies must be updated for Serverless-specific actions

PITFALL 10: "Data API calls fail after migration"
→ Cause: Data API parameter changes
→ Provisioned: ClusterIdentifier="cluster-name"
→ Serverless: WorkgroupName="workgroup-name"
→ All Lambda functions using Data API must update parameter names

→ Snapshot → Serverless restore: transfers tables, procedures, users
→ MVs may need recreation after restore
→ ALWAYS run parallel (provisioned + Serverless) for 30 days before cutting over
