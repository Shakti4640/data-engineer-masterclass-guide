🔵 WHAT COULD BE DONE

OPTION A: Kinesis Data Streams → Firehose → S3 → Glue → Redshift ✅
→ Firehose handles buffering, format conversion, partitioning
→ Glue handles dedup, quality, compaction
→ Step Functions orchestrates micro-batch
→ 5-15 min latency achievable
→ SELECTED: best balance of cost, reliability, and mesh integration

OPTION B: Kinesis Data Streams → Redshift Streaming Ingestion (direct)
→ Redshift can natively consume from Kinesis
→ Sub-minute latency possible
→ Problem: data doesn't land in S3 (no Athena, no data lake)
→ Problem: tightly couples Kinesis to Redshift
→ PARTIAL USE: could add later for real-time dashboard layer

OPTION C: Kinesis → Lambda → S3 + Redshift
→ Lambda processes each batch from Kinesis
→ Problem: Lambda 15-min timeout, 10 GB /tmp limit
→ Problem: at 5K events/sec, Lambda costs explode
→ REJECTED: not cost-effective at this scale

OPTION D: Kinesis → Firehose → S3 (only) + Redshift Spectrum
→ Skip Glue ETL entirely
→ Spectrum queries Bronze Parquet directly
→ Problem: no dedup, no quality check, no schema enforcement
→ Problem: small files from Firehose kill Spectrum performance (Project 62)
→ REJECTED: violates Data Mesh quality principle

DECISION: Option A + Gold-layer Spectrum as bonus query path

✅ WHAT SHOULD DO

1. USE KINESIS DATA STREAMS as the front door
   → Decouples producers (website) from consumers (Firehose, Lambda, etc.)
   → Multiple consumers can read the same stream independently
   → 24-hour retention = replay capability if something fails

2. USE FIREHOSE for S3 delivery
   → Handles buffering (60s or 5MB — whichever first)
   → Converts JSON → Parquet automatically (saves Athena cost)
   → Partitions by date/hour automatically
   → Handles backpressure and retries

3. BUFFER BEFORE PROCESSING — DON'T TRIGGER ON EVERY FILE
   → Firehose writes a file every 60 seconds
   → That's 60 files/hour, 1440 files/day
   → Glue ETL per file = too many jobs
   → Step Functions waits 60s to accumulate files, then ONE Glue job

4. COMPACT IN GOLD LAYER
   → Bronze: many small Parquet files from Firehose
   → Gold: compacted into fewer, larger Parquet files
   → Glue ETL coalesces (Project 62 pattern)
   → Critical for Athena and Spectrum performance

5. DEDUPLICATE IN SILVER LAYER
   → At-least-once delivery means duplicates possible
   → Silver layer: GROUP BY event_id, keep latest
   → Gold layer: clean, deduplicated, compacted

6. PARTITION GOLD BY HOUR (not minute)
   → Hourly partitions = 24/day = manageable
   → Minute partitions = 1440/day = too many small files
   → Athena partition pruning works well with hour granularity

7. USE ENHANCED FAN-OUT for Kinesis consumers if needed
   → Standard: 2 MB/s shared across all consumers per shard
   → Enhanced: 2 MB/s DEDICATED per consumer per shard
   → Use when: multiple consumers (Firehose + Lambda + analytics)

8. PUBLISH TO DATA MESH after each micro-batch
   → SNS event: "clickstream_events micro-batch published"
   → Follows Project 71 publish/subscribe pattern
   → Analytics domain reacts immediately

❌ WHAT SHOULD NOT DO

1. DO NOT send events directly to S3
   → No buffering = one file per event = millions of tiny files
   → S3 LIST becomes unusable, Athena scans explode
   → Always buffer through Firehose

2. DO NOT use Firehose JSON delivery for analytical queries
   → JSON is 5-10x larger than Parquet
   → Athena scans 10x more data = 10x more cost
   → Always convert to Parquet in Firehose

3. DO NOT trigger Glue job on EVERY S3 file from Firehose
   → 60 files/hour × $0.44/DPU-hour = $26.40/hour = $19,000/month
   → Batch multiple files into one Glue run
   → Use Step Functions with Wait state to accumulate

4. DO NOT set Firehose buffer to 900 seconds to reduce files
   → Kills your 15-minute latency budget
   → 900s buffer + 300s Glue = 20 minutes (SLA breach)
   → Use 60-120s buffer, handle small files in Glue compaction

5. DO NOT skip the deduplication step
   → Kinesis guarantees at-least-once delivery
   → Network retries, Firehose retries = duplicate events
   → Without dedup: inflated metrics, wrong revenue numbers

6. DO NOT put Kinesis Data Stream and Firehose in different regions
   → Cross-region Kinesis reading not supported
   → Both must be in same region as S3 bucket

7. DO NOT use on-demand Kinesis capacity for predictable workloads
   → On-demand costs 50% more than provisioned
   → Clickstream traffic is predictable (peak during business hours)
   → Use provisioned with auto-scaling

8. DO NOT skip CloudWatch alarms on Kinesis iterator age
   → Iterator age = how far behind the consumer is
   → If iterator age > 5 minutes → consumer is falling behind
   → Could cause data loss when records expire from stream

🚫 WHEN NOT TO USE THIS PATTERN

1. LATENCY REQUIREMENT < 1 SECOND
   → This micro-batch pattern = 5-15 minutes
   → For sub-second: Kinesis → Lambda → DynamoDB or Kinesis Analytics
   → Or Redshift Streaming Ingestion (Materialized View from stream)

2. EVENT VOLUME < 100 EVENTS/SECOND
   → Kinesis minimum cost: 1 shard = $0.015/hour = $11/month
   → Firehose minimum cost: per-record pricing
   → At low volume: cheaper to use SQS → Lambda → S3 batch

3. NO NEED FOR REPLAY OR ORDERING
   → If events are independent and order doesn't matter
   → SQS is simpler and cheaper
   → Kinesis value = ordered replay within shard

4. EVENTS NEED COMPLEX ROUTING TO 10+ DESTINATIONS
   → Kinesis fan-out limited to 5 consumers per stream (standard)
   → Enhanced fan-out: up to 20 consumers
   → For 10+: use MSK (Kafka) with consumer groups

5. BUDGET < $100/MONTH FOR STREAMING
   → Kinesis + Firehose + Glue micro-batch = $300-1000+/month
   → Alternative: increase cron frequency to every 15 minutes
   → Use S3 → EventBridge → Glue pattern (Project 66) with faster schedule

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  Firehose reads from Kinesis Data Stream continuously               │
│                                                                     │
│  BUFFERING LOGIC:                                                   │
│  ┌─────────────────────────────────────────────────────┐           │
│  │  Buffer accumulates records until:                   │           │
│  │                                                       │           │
│  │  CONDITION 1: Buffer SIZE ≥ 5 MB                     │           │
│  │       OR                                              │           │
│  │  CONDITION 2: Buffer TIME ≥ 60 seconds               │           │
│  │                                                       │           │
│  │  WHICHEVER COMES FIRST → flush to S3                 │           │
│  │                                                       │           │
│  │  At 800 events/sec × 1 KB each:                      │           │
│  │  → 800 KB/sec → reaches 5 MB in ~6.25 seconds        │           │
│  │  → SIZE condition triggers before TIME condition      │           │
│  │  → Result: ~1 file every 6 seconds at average load   │           │
│  │                                                       │           │
│  │  At 100 events/sec × 1 KB each:                      │           │
│  │  → 100 KB/sec → reaches 5 MB in ~51 seconds          │           │
│  │  → SIZE condition triggers before TIME (60s)          │           │
│  │                                                       │           │
│  │  At 10 events/sec × 1 KB each:                       │           │
│  │  → 10 KB/sec → reaches 5 MB in ~512 seconds          │           │
│  │  → TIME condition (60s) triggers first                │           │
│  │  → File will be ~600 KB (small file — need compaction)│           │
│  └─────────────────────────────────────────────────────┘           │
│                                                                     │
│  FORMAT CONVERSION (JSON → Parquet):                                │
│  1. Firehose accumulates JSON records in buffer                    │
│  2. References Glue Catalog table for target schema                │
│  3. Converts JSON → Apache Parquet using schema                    │
│  4. Applies Snappy compression                                      │
│  5. Writes Parquet file to S3 with dynamic partitioning            │
│                                                                     │
│  DYNAMIC PARTITIONING:                                              │
│  → Firehose extracts partition keys from each record               │
│  → Routes records to S3 prefix:                                     │
│    year=!{timestamp:yyyy}/month=!{timestamp:MM}/                    │
│    day=!{timestamp:dd}/hour=!{timestamp:HH}/                        │
│  → Each partition gets its own buffer                               │
│  → Enables Athena partition pruning from Day 1                     │
│                                                                     │
│  RETRY & ERROR HANDLING:                                            │
│  → S3 write fails → Firehose retries for up to 24 hours           │
│  → Records that fail conversion → sent to error prefix              │
│    s3://bucket/errors/processing-failed/                            │
│  → NO DATA LOSS — records buffered in Firehose until delivered     │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "ProvisionedThroughputExceededException" — shard capacity exceeded
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Each shard: 1,000 records/sec OR 1 MB/sec write
→ Black Friday: 25,000 events/sec needs 25 shards minimum
→ Fix: enable auto-scaling or pre-provision shards before peak
→ Fix: use PutRecords (batch) instead of PutRecord (single)

PITFALL 2: Firehose writes thousands of tiny files
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Dynamic partitioning + short buffer = many small files per partition
→ 24 hour partitions × many flushes = 100+ files per hour per partition
→ Athena scans slow down (Project 62 problem)
→ Fix: Glue ETL compaction in Gold layer — merge into fewer large files

PITFALL 3: Glue ETL takes 10+ minutes for micro-batch
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Glue cold start: 1-3 minutes
→ Processing: 2-5 minutes for typical micro-batch
→ If Glue takes too long → SLA breached
→ Fix: use Glue warm pools (pre-initialized workers)
→ Fix: reduce DPU count for small batches (2 DPUs enough)

PITFALL 4: Duplicate events in Gold layer
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Kinesis: at-least-once delivery
→ Firehose: may write same record twice on retry
→ Producer retry: same event sent twice to Kinesis
→ Fix: deduplicate on event_id in Silver → Gold ETL
→ Use: window function ROW_NUMBER() PARTITION BY event_id ORDER BY timestamp DESC

PITFALL 5: Step Functions triggered too frequently
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Firehose writes every 6 seconds at peak → 10 files/minute
→ Each file triggers EventBridge → Step Functions
→ 10 concurrent state machines → 10 concurrent Glue jobs → expensive
→ Fix: use deduplication in Step Functions (only start if not already running)
→ Fix: EventBridge → SQS → single poller → trigger once per batch window

PITFALL 6: Kinesis iterator age growing — consumer falling behind
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ GetRecords.IteratorAgeMilliseconds increasing over time
→ Means Firehose can't keep up with stream volume
→ Risk: records expire from stream before being read
→ Fix: add more shards, or enable enhanced fan-out
→ Alarm: if iterator age > 300,000 ms (5 min) → alert immediately

PITFALL 7: Parquet schema mismatch between Firehose and Glue
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Firehose references Glue Catalog table for Parquet conversion
→ If event JSON has new field not in schema → field DROPPED silently
→ If event JSON has wrong type → conversion ERROR
→ Fix: update Glue Catalog table before deploying new event schema
→ Fix: use schema evolution strategy (Project 45)

📈 SCALABILITY CONSIDERATIONS

SCALING DIMENSION 1: Event Volume
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ 800 events/sec (normal): 2 shards sufficient
→ 5,000 events/sec (peak): 5-6 shards needed
→ 25,000 events/sec (Black Friday): 25-30 shards
→ 100,000 events/sec (viral event): 100+ shards
→ Kinesis supports up to 500 shards per stream (soft limit)
→ Auto-scaling: UpdateShardCount API or Application Auto Scaling

SCALING DIMENSION 2: Firehose Throughput
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Firehose auto-scales — no shard concept
→ Soft limit: 1,000 records/sec or 1 MB/sec per delivery stream
→ Can request increase to 100,000 records/sec
→ For higher: create multiple delivery streams from same Kinesis stream

SCALING DIMENSION 3: Glue ETL Micro-Batch
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Small batch (1 minute): 2 DPUs, 2-3 minutes runtime
→ Medium batch (5 minutes): 4 DPUs, 3-5 minutes runtime
→ Large batch (15 minutes): 10 DPUs, 5-8 minutes runtime
→ Use Glue auto-scaling (available since 2023)
→ Warm pools reduce cold start from 3 min to 30 seconds

SCALING DIMENSION 4: Redshift COPY Frequency
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Every 5 minutes: 288 COPYs/day — manageable
→ Every 1 minute: 1,440 COPYs/day — Redshift handles it
→ Consider: Redshift Streaming Ingestion for sub-minute
→ Or: batch COPY every 15 minutes, Spectrum for interim queries

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY THE STREAMING PIPELINE IS HEALTHY:

1. KINESIS STREAM HEALTH:
   → CloudWatch: IncomingRecords (count/sec)
   → CloudWatch: IncomingBytes (bytes/sec)
   → CloudWatch: WriteProvisionedThroughputExceeded (should be 0)
   → CloudWatch: GetRecords.IteratorAgeMilliseconds (< 60000)

2. FIREHOSE DELIVERY HEALTH:
   → CloudWatch: DeliveryToS3.Records (count per period)
   → CloudWatch: DeliveryToS3.Success (should be 100%)
   → CloudWatch: DataFreshness (seconds behind — should be < buffer time)

3. GLUE MICRO-BATCH HEALTH:
   → CloudWatch: glue.driver.aggregate.numCompletedTasks
   → Glue job run history: duration, DPU-hours, records processed
   → Custom metric: records_processed_per_batch

4. END-TO-END LATENCY:
   → Custom CloudWatch metric: event_timestamp vs queryable_timestamp
   → P99 latency should be < 900 seconds (15 min SLA)
   → Alert if P99 > 600 seconds (10 min — early warning)

5. DATA QUALITY:
   → Dedup effectiveness: count(event_id) vs count(distinct event_id)
   → Schema validation failures: Firehose error records count
   → Null rate per column: Glue DQ rules (Project 68)

6. COST TRACKING:
   → Tag all resources: pipeline=clickstream-realtime
   → Cost Explorer: filter by tag
   → Alert if daily cost > $10 (unexpected spike)
