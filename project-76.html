🔴 HYPOTHETICAL PROBLEM

Company: QuickCart — Full data lake platform (Project 73)
         DR across regions (Project 75)
         50+ Glue jobs, 200+ catalog tables, Redshift DDL, Step Functions

THE INCIDENT:
→ Friday 4:30 PM: Junior engineer updates Glue ETL job for daily_orders
→ Changes column mapping: "order_total" → "total_amount" (rename)
→ Uploads new script to S3 manually via Console
→ Saturday 2:00 AM: Nightly pipeline runs with new script
→ Glue job SUCCEEDS but writes wrong column name to Gold Parquet
→ Saturday 8:00 AM: BI dashboards show $0 revenue (column mismatch)
→ Saturday 10:00 AM: On-call engineer discovers the issue
→ Saturday 12:00 PM: Engineer can't find the OLD version of the script
→ Saturday 2:00 PM: Engineer rewrites from memory, deploys fix
→ Saturday 3:00 PM: Re-runs nightly pipeline, dashboards recover

DAMAGE:
→ 13 hours of broken dashboards
→ Weekend executive report showed $0 revenue (sent to board)
→ No version history of the change
→ No code review before deployment
→ No automated test caught the column mismatch
→ No rollback mechanism

✅ WHAT SHOULD DO

1. MONOREPO for all data platform code
   → Single repository: quickcart-data-platform
   → Folder structure mirrors AWS services
   → One PR can change Glue script + CloudFormation + Redshift DDL together
   → Atomic deployments: all related changes deploy together

2. CLOUDFORMATION for ALL infrastructure
   → Every AWS resource defined in YAML templates
   → No Console clicks, no ad-hoc boto3 scripts
   → Reproducible: same template deploys to dev, staging, prod
   → Drift detection: know when someone changes resources manually

3. SEPARATE TESTS by speed and scope
   → Unit tests: < 30 seconds, no AWS calls, run on every push
   → Integration tests: 2-5 minutes, uses dev AWS account, run on PR
   → Smoke tests: 1-2 minutes, verifies deployed resources, run post-deploy

4. PARAMETERIZE everything by environment
   → Bucket names: {project}-data-{env} (prod, staging, dev)
   → Glue database names: {domain}_{layer}_{env}
   → Redshift workgroup: {project}-{env}
   → Same CloudFormation template, different parameter files

5. MIGRATION PATTERN for Redshift DDL
   → Numbered SQL migration files: 001_create_tables.sql, 002_add_column.sql
   → Applied in order, tracked in migrations table
   → Never ALTER existing migrations — only add new ones
   → Rollback = new migration that reverses the change

6. GLUE SCRIPT VERSIONING via S3 + Git
   → Scripts stored in S3: s3://bucket/scripts/{version}/script.py
   → Glue job updated to point to new S3 path
   → Rollback: update Glue job to previous S3 path
   → Git tag marks each production deployment

❌ WHAT SHOULD NOT DO

1. DO NOT upload Glue scripts to S3 via Console
   → This is how the incident happened
   → All deployments through CI/CD pipeline only
   → Console write access removed from developer IAM roles

2. DO NOT run CloudFormation directly from laptop
   → "It works on my machine" → different IAM permissions
   → All deploys through CodePipeline (consistent IAM role)
   → Exception: emergency hotfix with mandatory post-deploy PR

3. DO NOT skip the approval gate for production
   → Every production deploy requires human approval
   → Automated tests necessary but NOT sufficient
   → Team lead must review CloudFormation changeset

4. DO NOT test against production data
   → Integration tests use synthetic test datasets
   → Test data stored in dev S3 bucket
   → Never read from or write to production S3 in tests

5. DO NOT version Glue scripts with timestamps
   → s3://bucket/scripts/transform_20250115.py → unmaintainable
   → Use Git commit SHA: s3://bucket/scripts/{commit_sha}/transform.py
   → Deterministic, traceable, reproducible

6. DO NOT mix infrastructure and application deploys
   → Infrastructure (CloudFormation): separate pipeline stage
   → Application (Glue scripts): separate pipeline stage
   → Infrastructure deploys first, then application
   → Prevents: new script references resource that doesn't exist yet

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: CloudFormation stack update fails mid-way
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ CFN creates new Glue job but fails on Step Functions update
→ Stack in UPDATE_ROLLBACK_IN_PROGRESS state
→ Rollback may also fail → UPDATE_ROLLBACK_FAILED (stuck)
→ Fix: use CFN change sets with careful dependency ordering
→ Fix: avoid replacing resources (update in-place when possible)

PITFALL 2: Glue script deployed but references non-existent catalog table
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ New script references analytics_derived.new_table
→ CloudFormation creates table AFTER script deploys
→ Nightly run fails: "Table not found"
→ Fix: deploy infrastructure BEFORE application code
→ Fix: integration test catches this before production

PITFALL 3: Redshift migration runs in wrong order
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Developer creates 003_add_column.sql
→ Another developer creates 003_create_view.sql (same number!)
→ Which runs first depends on filesystem sort order
→ Fix: use timestamp-based naming: 20250115_143000_add_column.sql
→ Fix: migration runner checks for gaps and duplicates

PITFALL 4: Tests pass in dev but fail in staging
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Dev has different IAM permissions than staging
→ Dev S3 bucket has different data than staging
→ Fix: environments must be identical except data volume
→ Fix: use same CloudFormation template with different parameters

PITFALL 5: Pipeline takes 45 minutes — developers skip it
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ If pipeline is too slow, people find workarounds
→ "I'll just upload to S3 directly this one time"
→ Fix: unit tests < 2 min, integration tests < 10 min
→ Fix: cache dependencies in CodeBuild
→ Fix: parallel test execution

PITFALL 6: Rollback leaves orphaned resources
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ CloudFormation rolls back Glue job definition
→ But S3 data written by the failed job remains
→ Stale/corrupt data in Bronze/Silver layer
→ Fix: smoke test verifies data BEFORE marking deploy as success
→ Fix: write to timestamped partition — old data unaffected
