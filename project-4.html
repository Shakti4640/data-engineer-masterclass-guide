→ Each optimized for different access patterns

→ HOT (0-7 days):      S3 Standard         — $0.023/GB, instant access

→ ARCHIVE (365+ days):  S3 Glacier Deep     — $0.00099/GB, 12 hour retrieval

→ With transitions: ~$0.65/month (76% savings)
→ At 10 TB: $230/month → $55/month (same percentage)

→ Project 2 handled NON-CURRENT versions (old versions after overwrite)

🔵 WHAT COULD BE DONE

Option A: Manual Transitions via API
→ Change storage class by copying object with new StorageClass parameter
→ Works but requires scripting, scheduling, tracking
→ Does not scale — must track every file individually

Option B: Lifecycle Rules (RECOMMENDED)
→ S3 evaluates rules daily — automatic, zero maintenance
→ Handles transitions AND expirations
→ Can target by prefix, tag, or entire bucket
→ Set it once, works forever

Option C: S3 Intelligent Tiering
→ S3 automatically moves objects between tiers based on access patterns
→ No lifecycle rules needed — fully automatic
→ Small monthly monitoring fee: $0.0025 per 1000 objects
→ Best for: unpredictable access patterns
→ Limitation: doesn't cover Glacier Flexible or Deep Archive

Option D: Combination — Lifecycle + Intelligent Tiering
→ Use Intelligent Tiering for 0-90 day window (unpredictable access)
→ Use lifecycle to transition to Glacier Flexible after 90 days (predictable)
→ Use lifecycle to transition to Deep Archive after 365 days (predictable)
→ Most cost-effective for mixed patterns

DECISION: Option B for QuickCart (predictable patterns)
→ We KNOW files are hot for 7 days, warm for 30, cold after 90
→ Lifecycle rules are cheaper than IT monitoring fee for known patterns
→ We'll also demonstrate Option C for interview knowledge

✅ WHAT SHOULD DO

1. Analyze ACTUAL access patterns before choosing tiers
   → S3 Storage Lens: free dashboard showing access frequency
   → S3 Analytics — Storage Class Analysis: recommends transitions
   → Don't guess — data-driven decisions

2. Understand MINIMUM storage duration charges
   → S3-IA: minimum 30 days (delete at day 15 = still charged for 30)
   → Glacier IR: minimum 90 days
   → Glacier Flexible: minimum 90 days
   → Glacier Deep Archive: minimum 180 days
   → Transition ONLY when object will stay long enough to justify

3. Understand MINIMUM object size charges
   → S3-IA: objects < 128KB are charged as 128KB
   → Glacier IR: objects < 128KB are charged as 128KB
   → Small files (< 128KB) may COST MORE in IA/Glacier than Standard
   → QuickCart's files are 2-50MB — no issue here

4. Account for RETRIEVAL costs
   → S3 Standard: $0.00 per GB retrieved
   → S3-IA: $0.01 per GB retrieved
   → Glacier IR: $0.03 per GB retrieved
   → Glacier Flexible (standard): $0.01 per GB + 3-5 hour wait
   → Glacier Deep Archive: $0.02 per GB + 12 hour wait
   → If files are retrieved frequently, savings evaporate

5. Set up S3 Storage Lens BEFORE implementing transitions
   → Baseline current access patterns
   → Monitor after transitions to verify cost savings
   → Required evidence for CFO's cost report

6. Use TAGS in lifecycle rules for different data categories
   → Tag: retention=7years → transition to Deep Archive after 365 days
   → Tag: retention=1year → delete after 365 days
   → More flexible than prefix-based rules alone

7. Test retrieval from each tier BEFORE a real emergency
   → Glacier Flexible restore takes 3-5 hours — know this BEFORE you need data
   → Deep Archive restore takes 12 hours — plan ahead
   → Same lesson as Project 2: test recovery before disaster

❌ WHAT SHOULD NOT DO

1. DO NOT transition tiny files to S3-IA or Glacier
   → Files < 128KB are charged as 128KB in IA/Glacier
   → A 1KB file in S3-IA costs MORE than in Standard
   → Rule of thumb: only transition files > 128KB
   → Use lifecycle rule filter with ObjectSizeGreaterThan

2. DO NOT transition files that are still being accessed
   → S3-IA retrieval: $0.01/GB
   → File retrieved 10 times/month × 50MB = $0.50/month retrieval
   → Standard storage for same file: $0.00115/month
   → Retrieval cost >> storage savings = net LOSS

3. DO NOT skip minimum duration calculation
   → Transition to Glacier IR (90-day minimum) on day 31
   → Delete the file on day 45
   → Charged for FULL 90 days of Glacier IR storage
   → Plus: early deletion fee in some tiers

4. DO NOT use Glacier for data that needs instant access
   → Glacier Flexible: 3-5 HOURS to retrieve (expedited: 1-5 minutes at $$$)
   → Deep Archive: 12 HOURS minimum
   → If dashboard query needs this data → use Glacier IR (millisecond access)

5. DO NOT create conflicting lifecycle rules
   → Rule A: transition to S3-IA after 30 days
   → Rule B: transition to Standard-IA after 60 days
   → These conflict — S3 uses the EARLIEST matching rule
   → Keep transitions in one rule with clear progression

6. DO NOT forget the "waterfall" restriction
   → Transitions must go "downward" in the storage class hierarchy
   → Standard → S3-IA → Glacier IR → Glacier Flexible → Deep Archive
   → CANNOT go backward: Glacier → Standard via lifecycle
   → To go backward: must RESTORE then COPY (manual process)

7. DO NOT apply Intelligent Tiering to files with known access patterns
   → IT charges $0.0025 per 1000 objects/month for monitoring
   → If you KNOW the pattern → lifecycle rules are cheaper
   → IT is for UNPREDICTABLE patterns only

🚫 WHEN NOT TO USE

1. DO NOT use S3-IA for objects accessed more than once per month
   → Breakeven: storage savings must exceed retrieval cost
   → 50MB file: Standard = $0.00115/mo, S3-IA = $0.000625/mo + $0.0005/retrieval
   → 1 retrieval/month: S3-IA wins by $0.0000250
   → 3 retrievals/month: S3-IA LOSES by $0.000875
   → Rule: < 1 access/month → S3-IA; otherwise → Standard

2. DO NOT use Glacier Flexible for data needed within minutes
   → Standard retrieval: 3-5 hours
   → Expedited retrieval: 1-5 minutes BUT costs $10 per 1000 requests + $0.03/GB
   → If you need sub-minute access → use Glacier IR or S3-IA

3. DO NOT use Glacier Deep Archive for data needed within hours
   → Standard retrieval: 12 hours
   → Bulk retrieval: 48 hours
   → No expedited option
   → Only for true archive: compliance, legal hold, historical backup

4. DO NOT use S3 One Zone-IA for critical/irreplaceable data
   → Stored in SINGLE Availability Zone (not 3+ like other classes)
   → If that AZ has hardware failure → data LOST
   → Use for: derived data, reproducible data, cached results
   → NOT for: source data, compliance data, anything irreplaceable

5. DO NOT use Intelligent Tiering for short-lived objects
   → Objects deleted within 30 days never benefit from auto-tiering
   → Monitoring fee paid regardless
   → Use Standard + lifecycle expiration for transient data

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Transitioned to S3-IA but bill INCREASED"
→ Cause: files are accessed frequently after transition
→ S3-IA retrieval: $0.01/GB per access
→ 50MB file × 20 accesses/month = $0.01 × 0.05 × 20 = $0.01/month retrieval
→ vs Standard storage: $0.023 × 0.05 = $0.00115/month
→ Fix: only transition files with < 1 access/month
→ Use S3 Analytics to validate access patterns BEFORE transitioning

PITFALL 2: "Transitioned 1 million tiny files to Glacier — bill went UP"
→ Cause: 128KB minimum charge per object
→ 1 million × 1KB files at Standard: 1 GB × $0.023 = $0.023/month
→ 1 million × 1KB files at Glacier (charged as 128KB each):
   128 GB × $0.0036 = $0.46/month
→ 20x MORE expensive in Glacier!
→ Fix: filter by ObjectSizeGreaterThan: 131072 (128KB) in lifecycle rule

PITFALL 3: "Deleted file from Glacier after 10 days — charged for 90 days"
→ Cause: Glacier Flexible has 90-day minimum storage duration
→ Pro-rated early deletion fee applies
→ Same for: S3-IA (30-day min), Glacier IR (90-day), Deep Archive (180-day)
→ Fix: don't delete objects early from cold tiers — let lifecycle handle it

PITFALL 4: "GetObject on Glacier Flexible returns 403 InvalidObjectState"
→ Cause: Glacier Flexible objects MUST be restored before reading
→ Cannot GET directly — must call RestoreObject first, wait hours
→ This is the #1 surprise for developers new to Glacier
→ Glacier IR does NOT have this problem — instant access

PITFALL 5: "Lifecycle rule set but objects not transitioning"
→ Cause 1: rule evaluates on creation date, not upload date
→ Cause 2: lifecycle runs once per day — may take 24-48 hours
→ Cause 3: rule has wrong prefix filter — doesn't match objects
→ Cause 4: object already in target storage class
→ Debug: check S3 Inventory report for StorageClass per object

PITFALL 6: "Intelligent Tiering monitoring fee eating our savings"
→ 10 million objects × $0.0025/1000 = $25/month just for monitoring
→ If objects are small (few KB each) and storage cost is $5/month
→ Monitoring fee > storage savings = net loss
→ Fix: IT is cost-effective for objects > 128KB with unpredictable access

PITFALL 7: "Restored from Glacier but restore expired before I downloaded"
→ Cause: RestoreObject creates temporary copy for N days (you specify)
→ If you specify Days=1 and don't download within 24 hours → copy gone
→ Must restore again (another 3-5 hour wait + retrieval cost)
→ Fix: set restore duration generously (Days=7 or more)
