Current S3 Structure (from Projects 1 & 2):
s3://quickcart-raw-data-prod/
├── daily_exports/
│   ├── orders/          ← Finance needs this
│   ├── customers/       ← Marketing needs this

Current Problem:
→ Everyone uses the same IAM user with AdministratorAccess
→ Marketing analyst can see finance_reports/ (salary data, revenue)

→ CTO demands: "Lock it down — each team sees only their data"
→ Compliance officer: "We may face SOC 2 audit next quarter"

🟡 HOW TO APPROACH IT

Step 1: Understand the TWO places access is controlled in AWS
→ IDENTITY-BASED POLICIES (IAM): attached to users/groups/roles
   "What can THIS PERSON do?"
→ RESOURCE-BASED POLICIES (Bucket Policy): attached to the S3 bucket
   "Who can touch THIS BUCKET?"
→ BOTH are evaluated together — access granted only if BOTH allow

Step 2: Map out the access matrix
┌──────────────────┬──────────────────┬────────────────┬──────────────┐
│ Prefix           │ Data Engineering │ Finance        │ Marketing    │
├──────────────────┼──────────────────┼────────────────┼──────────────┤
│ daily_exports/   │ Read + Write     │ Read Only      │ Read Only    │
│ finance_reports/ │ Read + Write     │ Read Only      │ NO ACCESS    │
│ marketing_data/  │ Read + Write     │ NO ACCESS      │ Read Only    │
├──────────────────┼──────────────────┼────────────────┼──────────────┤
│ Interns          │                  │ Read Only on   │ Read Only on │
│                  │                  │ their team's   │ their team's │
│                  │                  │ prefixes       │ prefixes     │
└──────────────────┴──────────────────┴────────────────┴──────────────┘

Step 3: Choose the right mechanism
→ IAM Groups: group users by team, attach team-specific policies
→ IAM Policies: define WHAT each group can do on WHICH prefixes
→ Bucket Policy: add DENY rules as a safety net (defense in depth)
→ S3 Block Public Access: already done in Project 1 — verify

Step 4: Implement with Principle of Least Privilege
→ Start with ZERO access
→ Grant ONLY what's needed
→ DENY explicitly where critical
→ Test every permission path

SOLUTION ARCHITECTURE:

AWS ACCOUNT
│

│   ├── DataEngineering-Group
│   │   └── Policy: full S3 access to all prefixes in this bucket
│   ├── Finance-Group
│   │   └── Policy: read-only to daily_exports/ + finance_reports/
│   ├── Marketing-Group
│   │   └── Policy: read-only to daily_exports/ + marketing_data/

│   ├── alice → DataEngineering-Group
│   ├── bob → Finance-Group
│   ├── carol → Marketing-Group
│   └── intern_dave → Interns-Group
│

    └── Bucket Policy (resource-based control):
        ├── EXPLICIT DENY: delete on finance_reports/ unless DataEngineering
        └── EXPLICIT DENY: any access to finance_reports/ from Marketing

EVALUATION FLOW:
User makes request → IAM Policy checked → Bucket Policy checked
→ If EITHER has explicit DENY → DENIED (always wins)
→ If BOTH have ALLOW → ALLOWED
→ If one ALLOWS and other is silent → ALLOWED
→ If neither ALLOWS → DENIED (implicit deny)

🔵 WHAT COULD BE DONE

Option A: IAM Policies Only (no bucket policy)
→ Works for simple setups
→ All control lives in IAM
→ Risk: someone creates a new IAM user and forgets to restrict
→ No safety net on the resource side

Option B: Bucket Policy Only (no IAM restrictions)
→ Works but harder to manage per-user
→ Bucket policies reference principals (users/roles/accounts)
→ Gets messy with many users
→ Better for cross-account access (Project 56)

Option C: IAM Groups + Bucket Policy DENY (RECOMMENDED)
→ IAM Groups: ALLOW what each team needs (positive permissions)
→ Bucket Policy: DENY what must never happen (safety net)
→ Defense in depth: even if IAM policy is misconfigured,
   bucket policy DENY still blocks unauthorized access
→ DENY always wins over ALLOW in AWS policy evaluation

Option D: S3 Access Points (enterprise scale)
→ Named endpoints with dedicated policies per application/team
→ Overkill for 4 teams, designed for hundreds of consumers
→ Covered conceptually but not implemented here

DECISION: Option C — IAM Groups + Bucket Policy DENY
→ Reason: clean separation, defense in depth, interview gold

✅ WHAT SHOULD DO

1. Create IAM GROUPS — never attach policies to individual users
   → Users come and go; groups persist
   → New hire joins Finance → add to Finance-Group → done
   → Employee leaves → remove from group → access revoked

2. Use PREFIX-BASED resource ARNs in policies
   → arn:aws:s3:::quickcart-raw-data-prod/finance_reports/*
   → This restricts to SPECIFIC "folders" within the bucket
   → Two ARNs needed: bucket-level (for ListBucket) + object-level (for GetObject)

3. Always include BOTH bucket-level AND object-level permissions
   → s3:ListBucket on arn:aws:s3:::bucket-name (with Condition for prefix)
   → s3:GetObject on arn:aws:s3:::bucket-name/prefix/*
   → Missing either = confusing "Access Denied" errors

4. Use EXPLICIT DENY in bucket policy for critical data
   → Deny ALWAYS overrides Allow in AWS
   → Even if someone accidentally grants admin access,
     bucket policy DENY still blocks the action

5. Use Conditions in policies for fine-grained control
   → s3:prefix condition limits ListBucket to specific "folders"
   → aws:PrincipalTag condition for tag-based access (advanced)
   → s3:x-amz-server-side-encryption for encryption enforcement

6. Test with AWS Policy Simulator BEFORE applying
   → IAM → Policy Simulator → select user → select action → simulate
   → Shows: allowed, denied, and WHY

7. Enable CloudTrail S3 data events for audit
   → Logs every GetObject, PutObject, DeleteObject
   → Required for SOC 2 compliance
   → Costs ~$0.10 per 100K events

❌ WHAT SHOULD NOT DO

1. DO NOT attach policies directly to IAM users
   → alice-policy, bob-policy, carol-policy = unmanageable at scale
   → When alice leaves and dave joins her role, you must recreate everything
   → Groups abstract the person from the permission

2. DO NOT use "Resource": "*" in IAM policies
   → Grants access to ALL S3 buckets in the account (and possibly others)
   → Always scope to specific bucket ARN + prefix

3. DO NOT confuse bucket ARN formats
   → WRONG: arn:aws:s3:::quickcart-raw-data-prod/* (for ListBucket)
   → RIGHT: arn:aws:s3:::quickcart-raw-data-prod   (for ListBucket)
   → RIGHT: arn:aws:s3:::quickcart-raw-data-prod/* (for GetObject/PutObject)
   → Bucket-level actions need bucket ARN (no /*)
   → Object-level actions need object ARN (with /*)

4. DO NOT rely solely on ALLOW policies
   → Someone with AdministratorAccess bypasses your carefully crafted IAM policies
   → Bucket policy DENY stops even administrators
   → Only the account root user bypasses bucket policy DENY

5. DO NOT forget the s3:ListBucket + Condition trick
   → Without Condition: user can LIST the entire bucket (sees all prefixes)
   → Even if they can't GET objects in other prefixes, they see the NAMES
   → File names alone can leak sensitive info (e.g., "layoff_plan_2025.csv")

6. DO NOT create overly complex policies
   → AWS policy size limit: 6,144 characters (managed policy)
   → AWS policy size limit: 2,048 characters (inline policy)
   → Too many prefix conditions → hit size limits
   → Solution: one policy per access pattern, attach multiple to group

7. DO NOT test with your admin account
   → Admin account has full access — tests always pass
   → Create a test user in each group and test from their perspective
   → Or use AWS Policy Simulator

🚫 WHEN NOT TO USE

1. DO NOT use S3 ACLs for access control
   → Legacy mechanism from pre-IAM era
   → Cannot express conditions, time-based access, or MFA requirements
   → AWS default since April 2023: "Bucket owner enforced" (ACLs disabled)
   → Use IAM Policies + Bucket Policies instead

2. DO NOT use bucket policies as the ONLY access control for same-account users
   → Bucket policies are best for cross-account + DENY rules
   → IAM groups are cleaner for same-account team-based access
   → Combining both = defense in depth

3. DO NOT use S3 Access Points for < 10 consumer applications
   → Management overhead exceeds benefit
   → Each access point = separate DNS name + policy
   → Simple prefix-based IAM policies suffice for small teams

4. DO NOT use IAM policies for temporary/time-limited access
   → Use: Pre-signed URLs (time-limited, per-object)
   → Use: STS AssumeRole with session duration
   → IAM policy changes take effect globally and permanently

5. DO NOT use resource-based policies for fine-grained user-level control
   → Bucket policy with 50 user ARNs = unmanageable
   → Use IAM groups + role-based access instead
   → Bucket policy for: account-level, DENY rules, cross-account

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Access Denied on ListObjects but GetObject works"
→ Cause: policy grants s3:GetObject on bucket/* but NOT s3:ListBucket on bucket
→ User can download a file if they know the EXACT key
→ But cannot browse/list the contents of a prefix
→ Fix: add s3:ListBucket with Condition for the prefix

PITFALL 2: "User can list ALL prefixes even though I restricted GetObject"
→ Cause: s3:ListBucket without Condition on s3:prefix
→ ListBucket is a BUCKET-level action — applies to entire bucket
→ Must add Condition to restrict which prefixes appear in LIST results
→ Fix: add "Condition": {"StringLike": {"s3:prefix": ["allowed_prefix/*"]}}

PITFALL 3: "Bucket policy DENY blocks even the admin"
→ Cause: DENY in bucket policy overrides ALL IAM ALLOW policies
→ This is BY DESIGN — it's the safety net
→ Only the account ROOT USER can bypass bucket policy DENY
→ Fix: carefully scope the DENY with Condition or NotPrincipal

PITFALL 4: "NotPrincipal DENY locks everyone out including me"
→ Cause: "Effect": "Deny", "NotPrincipal": {"AWS": "arn:...:user/alice"}
→ This means: DENY everyone EXCEPT alice
→ But: NotPrincipal doesn't work as expected with assumed roles
→ Fix: use "Condition": {"StringNotEquals": {"aws:PrincipalArn": "..."}}
→ This is a CLASSIC interview trap question

PITFALL 5: "Policy works for IAM user but not for EC2 instance"
→ Cause: EC2 uses an IAM Role (instance profile), not IAM user
→ Bucket policy references IAM user ARN but EC2 presents role ARN
→ Fix: add the role ARN to the policy (covered in Project 6)

PITFALL 6: "Policy change doesn't take effect immediately"
→ IAM policy changes: eventual consistency (usually < 60 seconds)
→ Bucket policy changes: usually immediate but can take minutes
→ Don't test immediately after change — wait 1-2 minutes

PITFALL 7: "I used 'aws:username' condition but it's empty for roles"
→ Cause: roles don't have a username — they have a RoleSessionName
→ Use aws:PrincipalArn or aws:PrincipalTag instead
→ aws:username only works for IAM users

📈 SCALABILITY CONSIDERATIONS

POLICY SIZE LIMITS:
→ IAM managed policy: max 6,144 characters
→ IAM inline policy: max 2,048 characters
→ S3 bucket policy: max 20 KB (20,480 characters)
→ Max managed policies per group: 10
→ Max groups per user: 10
→ Max IAM users per account: 5,000 (soft limit)
→ Max IAM groups per account: 300 (soft limit)

SCALING FROM 4 TEAMS TO 40 TEAMS:
→ 40 IAM groups × 2-3 policies each = 80-120 managed policies
→ AWS limit: 1,500 managed policies per account
→ Still well within limits

→ Bucket policy with 40 DENY statements = approaches 20KB limit
→ Solution: use IAM policies for ALLOW, bucket policy only for critical DENY
→ Or: migrate to S3 Access Points (one per team, each with own policy)

SCALING FROM 1 BUCKET TO 50 BUCKETS:
→ Each bucket has its own bucket policy
→ IAM policies can reference multiple bucket ARNs
→ Use wildcards carefully: arn:aws:s3:::quickcart-*-prod/*
→ Better: use IAM policy variables: ${aws:PrincipalTag/team}

TAG-BASED ACCESS CONTROL (ABAC) — FUTURE SCALING:
→ Instead of explicit prefix per team in policy:
→ Tag S3 objects with "team=finance"
→ Tag IAM users with "team=finance"
→ Policy: allow if object tag matches user tag
→ Zero policy changes when new teams/prefixes added
→ Covered at advanced level in Project 74 (Lake Formation)

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO DEBUG "ACCESS DENIED" ERRORS:

1. AWS Policy Simulator (IAM Console):
   → Select the user/role
   → Select the service (S3) and action (GetObject)
   → Enter the resource ARN
   → Click "Run Simulation"
   → Shows: Allowed/Denied + WHICH policy caused it

2. CloudTrail:
   → Every API call logged with:
     - Who (IAM principal ARN)
     - What (API action)
     - When (timestamp)
     - Result (success/failure)
     - Error code (AccessDenied, etc.)
   → Search: EventName=GetObject, errorCode=AccessDenied

3. S3 Server Access Logs:
   → Per-request logs including:
     - Requester ARN
     - Bucket + Key
     - HTTP status (403 = denied)
     - Error code

4. IAM Access Analyzer:
   → Identifies resources shared with external entities
   → Flags bucket policies that grant public or cross-account access
   → Generates findings with recommended fixes

5. Manual Policy Debugging Checklist:
   → Is there an explicit DENY anywhere? (check all policies)
   → Does the IAM policy ALLOW this specific action?
   → Does the resource ARN match EXACTLY? (bucket vs object level)
   → Is there a Condition that's not met?
   → Is the user in the right group?
   → Did the policy propagate? (wait 60 seconds)

🔑 KEY TAKEAWAY

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ AWS has TWO policy attachment points: identity-based (IAM) + resource-based (bucket)
→ Policy evaluation: Explicit DENY > Explicit ALLOW > Implicit DENY
→ Same-account: EITHER IAM or bucket policy ALLOW is sufficient
→ Cross-account: BOTH IAM AND bucket policy must ALLOW (Project 56)
→ s3:ListBucket is bucket-level; s3:GetObject is object-level — different ARN formats
→ Use Condition on s3:prefix to restrict LIST results to specific "folders"
→ IAM Groups abstract users from permissions — always use groups
→ Bucket policy DENY = safety net that even admin IAM policies cannot override
→ S3 ACLs are legacy — disable them, use IAM + bucket policies only
→ Test with Policy Simulator before deploying to production
