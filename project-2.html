INTERN ACCIDENTALLY DELETES PRODUCTION FILE FROM S3
S3🟢 Phase 1: Very Easy

🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — same startup from Project 1
Timeline: 3 months after Project 1 deployment
Situation:
→ 270+ CSV files now living in s3://quickcart-raw-data-prod/
→ Upload script from Project 1 runs reliably every night
→ New intern joins the data team
→ Intern is tasked: "Delete the TEST files from the dev folder"
→ Intern runs:
   aws s3 rm s3://quickcart-raw-data-prod/daily_exports/orders/ --recursive
→ PRODUCTION orders folder — not dev
→ 90 days of orders CSVs: GONE
→ No backup exists anywhere else
→ CTO: "Can we recover this?"
→ Current answer: NO — versioning was never enabled
→ Real question: How do we make sure this NEVER happens again?

🟡 HOW TO APPROACH IT

Step 1: Understand WHAT actually happened inside S3
→ S3 DeleteObject is a PERMANENT operation (without versioning)
→ Once deleted, the object bytes are removed from all 3+ AZs
→ No recycle bin, no undo, no "restore from trash"
→ AWS Support CANNOT recover unversioned deleted objects

Step 2: Understand the PROTECTION layers available
→ Layer 1: Versioning — keeps every version of every object
→ Layer 2: MFA Delete — requires MFA token to permanently delete
→ Layer 3: Object Lock — WORM (Write Once Read Many) — legal hold
→ Layer 4: Lifecycle Rules — auto-manage old versions
→ Layer 5: Bucket Policy — restrict delete permissions

Step 3: Understand the COST of protection
→ Versioning: every version stored = extra storage cost
→ Without lifecycle rules: old versions accumulate forever
→ Need both: versioning ON + lifecycle to expire old versions

Step 4: Design the solution
→ Enable versioning (can never be fully disabled once enabled — only suspended)
→ Add lifecycle rules to manage version accumulation
→ Restrict s3:DeleteObject permission via IAM (Project 3 covers this deeply)
→ Optionally: MFA Delete for critical data

🟢 HOW CAN WE SOLVE IT

SOLUTION ARCHITECTURE:

BEFORE (Project 1 — no protection):

  DELETE request
       │
       ▼
  S3 Object ──→ PERMANENTLY GONE ──→ NO RECOVERY
  

AFTER (Project 2 — versioning enabled):

  DELETE request
       │
       ▼
  S3 adds "Delete Marker" (a tombstone)
       │
       │  Original object versions STILL EXIST
       ▼
  ┌─────────────────────────────────────────────────┐
  │  Key: daily_exports/orders/orders_20250101.csv  │
  │                                                  │
  │  Version v3 ← DELETE MARKER (latest — "deleted")│
  │  Version v2 ← previous upload (STILL EXISTS)    │

  │                                                  │
  │  GET request → returns 404 (sees delete marker)  │
  │  But: v1 and v2 are FULLY RECOVERABLE           │
  └─────────────────────────────────────────────────┘

RECOVERY:
→ Option A: Delete the delete marker → latest version reappears
→ Option B: GET specific version ID → download any old version

🔵 WHAT COULD BE DONE

Option A: Enable Versioning Only
→ Simplest protection
→ All versions kept indefinitely
→ Risk: storage cost grows unbounded

Option B: Versioning + Lifecycle Rules (RECOMMENDED)
→ Keep current + last 3 versions
→ Auto-delete versions older than 90 days
→ Move non-current versions to cheaper storage after 30 days
→ Balanced protection + cost control

Option C: Versioning + MFA Delete
→ Root account must enable MFA Delete via CLI
→ Permanent version deletion requires MFA token
→ Extremely strong but operationally heavy

Option D: S3 Object Lock (WORM)
→ Legal/compliance requirement (SEC Rule 17a-4, HIPAA)
→ Objects literally CANNOT be deleted for a defined retention period
→ Even root account cannot override in Compliance mode
→ Overkill for QuickCart — designed for regulated industries

DECISION: Option B — Versioning + Lifecycle Rules
→ Reason: protects against accidental deletion
→ Controls cost via automatic version expiration
→ Can always add MFA Delete later for critical datasets

✅ WHAT SHOULD DO

1. Enable versioning IMMEDIATELY on production bucket
   → One API call: put_bucket_versioning(Status='Enabled')
   → Takes effect for ALL new operations instantly
   → Existing objects don't get version IDs until next PUT/DELETE

2. Create lifecycle rules with CLEAR strategy:
   → Rule 1: Transition non-current versions to S3-IA after 30 days
   → Rule 2: Permanently delete non-current versions after 90 days
   → Rule 3: Clean up incomplete multipart uploads after 7 days
   → Rule 4: Remove expired delete markers (keeps bucket clean)

3. Verify versioning status PROGRAMMATICALLY after enabling
   → get_bucket_versioning() must return {'Status': 'Enabled'}

4. Test the recovery process BEFORE a real disaster
   → Upload file → Delete it → Recover it → Confirm contents match
   → Document the recovery runbook

5. Add CloudTrail logging for DeleteObject events
   → Know WHO deleted WHAT and WHEN
   → Covered deeper in Project 7 (IAM)

6. Educate the team:
   → "Delete markers are NOT permanent deletions"
   → "Versioned buckets have TWO types of delete"
   → "Always verify which bucket/prefix before --recursive"

7. Separate environments STRICTLY
   → Production: quickcart-raw-data-prod
   → Development: quickcart-raw-data-dev
   → NEVER same bucket with prefix-based separation for environments

❌ WHAT SHOULD NOT DO

1. DO NOT assume versioning protects against EVERYTHING
   → If someone deletes a specific VERSION ID — that version is PERMANENTLY gone
   → Versioning protects against simple DELETE, not version-targeted DELETE
   → MFA Delete protects against version-targeted DELETE

2. DO NOT enable versioning without lifecycle rules
   → Every PUT overwrites creates a new version
   → Daily 62MB × 365 days = 22.6 GB of CURRENT data
   → But with 5 versions each: 113 GB of VERSIONED data
   → Without lifecycle: versions accumulate FOREVER

3. DO NOT confuse "Suspended" with "Disabled"
   → Versioning has THREE states: Unversioned → Enabled → Suspended
   → Once Enabled, you can NEVER go back to Unversioned
   → Suspended: stops creating new versions but KEEPS existing ones
   → Existing versioned objects remain and still incur cost

4. DO NOT use Object Lock unless legally required
   → Once enabled on bucket creation, CANNOT be disabled
   → Compliance mode: even root account cannot delete
   → Governance mode: can be overridden with special permission
   → Wrong choice → you're locked into paying for storage you can't free

5. DO NOT skip testing recovery
   → "We have versioning" means nothing if nobody knows how to recover
   → Run a disaster recovery drill quarterly

6. DO NOT grant s3:DeleteObject to broad IAM groups
   → Intern had permission because everyone used the same admin role
   → Principle of Least Privilege (Project 3)

7. DO NOT rely on versioning as your ONLY backup strategy
   → Versioning protects within ONE bucket in ONE region
   → Region failure = all versions gone
   → Cross-region replication (Project 47) is the true DR solution

🚫 WHEN NOT TO USE

1. DO NOT enable versioning on TEMPORARY/TRANSIENT buckets
   → Example: ETL staging bucket that holds files for 5 minutes
   → Versioning adds overhead and cost for data you don't need to keep
   → Use: lifecycle rule with 1-day expiration instead

2. DO NOT use Object Lock for non-regulated data
   → QuickCart's CSV dumps don't need SEC-compliant WORM
   → Object Lock adds operational complexity and cost
   → Can't be disabled once enabled on the bucket

3. DO NOT use MFA Delete for buckets accessed by automation
   → Glue jobs, Lambda, Step Functions CANNOT provide MFA tokens
   → MFA Delete = only root user with physical device can delete
   → Use for: audit logs, compliance archives
   → Don't use for: ETL pipelines, active data lakes

4. DO NOT use versioning as a REPLACEMENT for proper data pipeline design
   → "We have versioning so we don't need idempotent writes"
   → Wrong — versioning saves you from disasters, not bad architecture

5. DO NOT keep versions indefinitely for high-churn data
   → A file overwritten 100 times/day = 100 versions/day
   → 365 days = 36,500 versions of ONE key
   → MUST have lifecycle rules

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "I enabled versioning but old objects have version ID = null"
→ Cause: Objects uploaded BEFORE versioning have no version ID
→ They get version ID "null" (literal string)
→ A DELETE on these objects puts a delete marker but the "null" version persists
→ A version-targeted DELETE on versionId=null permanently removes it

PITFALL 2: "Versioning is enabled but storage costs tripled"
→ Cause: No lifecycle rules — every version kept forever
→ Upload script re-uploads same file daily (even if unchanged)
→ 90 days × 62MB/day = 5.58GB current + 5.58GB × N versions
→ Fix: lifecycle rules to expire non-current versions

PITFALL 3: "I suspended versioning thinking it deletes old versions"
→ Suspended only STOPS creating new versions
→ All existing versions REMAIN and continue costing money
→ Must explicitly delete old versions OR use lifecycle expiration

PITFALL 4: "I deleted the delete marker but file still shows old data"
→ Cause: Multiple delete markers stacked (deleted the file multiple times)
→ Must delete ALL delete markers to expose the latest data version
→ Or: copy the specific version you want back to the same key

PITFALL 5: "Lifecycle rule set to expire in 1 day but versions still exist"
→ Cause: Lifecycle rules run ONCE per day (not real-time)
→ S3 processes lifecycle rules asynchronously in batches
→ May take 24-48 hours for rules to fully execute
→ Not suitable for real-time cleanup

PITFALL 6: "Cross-region replication not copying delete markers"
→ By DEFAULT, delete markers are NOT replicated
→ Must explicitly enable "Delete marker replication" in replication config
→ Version-targeted deletes are NEVER replicated (safety measure)

PITFALL 7: "MFA Delete locked us out of our own bucket"
→ Cause: root account MFA device lost/broken
→ Only root account can disable MFA Delete
→ Must contact AWS Support with account verification to recover
→ ALWAYS have backup MFA device for root account

📈 SCALABILITY CONSIDERATIONS

VERSIONING COST MODEL:

Scenario: QuickCart grows to 200 files/day × 50MB average

WITHOUT lifecycle rules:
→ Day 1:   200 files × 50MB = 10 GB (current)
→ Day 30:  10 GB current + 290 GB non-current = 300 GB
→ Day 365: 10 GB current + 3,640 GB non-current = 3,650 GB
→ Cost: 3,650 GB × $0.023 = $83.95/month (JUST versions)

WITH lifecycle rules (transition after 30d, expire after 90d):
→ Day 1:   10 GB at Standard
→ Day 30:  10 GB Standard + 290 GB transitioning to S3-IA
→ Day 90:  10 GB Standard + 580 GB S3-IA + older expired
→ Steady state: ~10 GB Standard + ~600 GB S3-IA
→ Cost: (10 × $0.023) + (600 × $0.0125) = $0.23 + $7.50 = $7.73/month
→ SAVINGS: ~91% compared to no lifecycle rules

VERSION LISTING PERFORMANCE:
→ list_object_versions returns MAX 1000 entries per call
→ Bucket with millions of versions = many paginated calls
→ Each version is a separate index entry in S3's metadata store
→ Heavy versioning + no cleanup = slow listing operations

LIFECYCLE RULE LIMITS:
→ Max 1000 lifecycle rules per bucket
→ Each rule can apply to specific prefix or tag filter
→ Rules evaluated once per day per object
→ Large buckets (billions of objects) may take days for full evaluation
