  - Glue Catalog: shared cross-account ✅ (Project 57)
  - Redshift cluster: "quickcart-prod-cluster"

→ Account B (222222222222): Analytics

  - Redshift Serverless: "quickcart-analytics-workgroup"
    - Used by BI team for ad-hoc analysis

THE PROBLEM:
BI Lead: "Our Redshift in Account B has yesterday's data.
  Account A's Redshift has LIVE curated tables with:
  - Materialized views refreshed every 15 minutes
  - Pre-joined star schema (orders + customers + products)
  - Pre-aggregated daily/weekly/monthly rollups
  - Custom UDFs for business metric calculations
  
  We need to query Account A's Redshift tables FROM Account B's Redshift.
  NOT a copy. NOT an export. LIVE data."

WHAT THEY TRIED AND WHY IT FAILED:

ATTEMPT 1: UNLOAD → S3 → COPY
→ Account A: UNLOAD orders_star_schema TO 's3://...'
→ Account B: COPY orders_star_schema FROM 's3://...'
→ Problems:
  - 500 GB × daily = massive S3 storage cost
  - UNLOAD takes 45 minutes
  - COPY takes 30 minutes
  - Data is 75 minutes stale by the time BI queries it
  - Must repeat for every table (50 tables)
  - Materialized views can't be UNLOADED (they're Redshift-internal)

ATTEMPT 2: Cross-account Redshift JDBC
→ Account B connects to Account A's Redshift via JDBC
→ Problems:
  - Requires VPC Peering (complex network setup)
  - Account A's cluster gets overloaded by Account B's queries
  - No workload isolation — BI queries slow production dashboards
  - No access control at table level

ATTEMPT 3: Federated Query
→ Account B's Redshift queries Account A's via federated query
→ Problems:
  - Every query goes through Account A's cluster
  - Network latency for every row
  - Can't use Account B's compute for heavy queries

THE REAL SOLUTION:
→ Redshift Data Sharing
→ Account A PUBLISHES specific tables as a "datashare"
→ Account B SUBSCRIBES to the datashare
→ Account B queries the data using ITS OWN compute
→ Data stays in Account A — zero copying
→ Real-time: Account B always sees Account A's latest data
→ Workload isolation: Account B's queries don't affect Account A

REQUIREMENTS:
→ Share 15 specific tables (not all 50)
→ Account B gets READ-ONLY access
→ No data movement (zero-copy)
→ Real-time freshness (no lag)
→ Account A controls what's shared
→ Account B uses its own Redshift compute for queries

🟡 HOW TO APPROACH IT

Step 1: Understand what Redshift Data Sharing IS
→ A feature of Redshift that shares LIVE data between clusters
→ Producer cluster: owns the data, creates datashare
→ Consumer cluster: subscribes, queries using own compute
→ Zero-copy: data physically stays on producer's storage
→ Real-time: consumer always reads latest committed data
→ Works: same account, cross-account, cross-region (with limitations)

Step 2: Understand the components
→ DATASHARE: a named container of shared objects (created by producer)
→ DATASHARE OBJECTS: tables, schemas, views, UDFs added to datashare
→ CONSUMER NAMESPACE: unique identifier for consumer cluster
→ DATASHARE DATABASE: consumer creates a local database from the datashare
→ External schema: consumer queries shared tables through this database

Step 3: Understand the authorization flow
→ Producer creates datashare → adds objects → grants to consumer namespace
→ For cross-account: AWS account-level authorization also required
→ Producer authorizes consumer's AWS account (via console/API)
→ Consumer associates the datashare (accepts the sharing invitation)
→ Consumer creates database from datashare → queries

Step 4: Understand what CAN and CANNOT be shared
→ CAN share: tables, views (regular + late-binding), materialized views
→ CAN share: schemas (all tables in schema), UDFs
→ CANNOT share: stored procedures, system tables
→ CANNOT share: cross-database references within shared views
→ Consumer: READ-ONLY (cannot INSERT, UPDATE, DELETE shared tables)

Step 5: Understand compute isolation
→ Producer's queries: use producer's compute (RPU/nodes)
→ Consumer's queries: use CONSUMER's compute (RPU/nodes)
→ Even though data is on producer → consumer's compute reads it directly
→ No network hop through producer's cluster
→ This is the KEY advantage over JDBC/federated query

Option B: VPC Peering + JDBC connection

Option D: Redshift Data Sharing (this project) ✅
→ Zero-copy: data stays in producer

→ Marketplace-based — overkill for internal teams

→ Reason: zero-copy, real-time, compute isolation, simple
→ Only requirement: both clusters must use RA3 nodes or Serverless
→ Does NOT work with dc2 or ds2 node types (legacy)

✅ WHAT SHOULD DO

1. Use RA3 nodes or Serverless on BOTH sides
   → Data Sharing requires Redshift Managed Storage (RMS)
   → RA3 nodes: separate compute + storage (supports data sharing)
   → Serverless: always supports data sharing
   → dc2/ds2: local storage only → data sharing NOT supported
   → Check: SELECT * FROM stv_datashares; (fails on dc2/ds2)

2. Share at SCHEMA level (not individual tables)
   → Share entire "analytics" schema → all tables within it shared
   → New tables added to schema → automatically shared
   → No need to update datashare for every new table
   → Keep shared tables in dedicated schema (separation of concerns)

3. Create DEDICATED schema for shared objects
   → Don't share your "public" schema (contains everything)
   → Create: "analytics_shared" schema with only tables to share
   → Internal tables stay in separate schema (not shared)
   → Clear boundary: shared vs. internal

4. Use meaningful datashare names
   → "quickcart_analytics_share" (describes what's shared)
   → Not: "ds1" or "share_for_account_b"
   → Name should be self-documenting

5. Grant to specific consumer namespace (not account wildcard)
   → Each Redshift cluster/serverless has a unique namespace UUID
   → Grant to namespace = specific cluster
   → More precise than granting to entire account

6. Document shared objects and consumers
   → Maintain catalog: what's shared, with whom, since when
   → Review quarterly: is Account B still using this datashare?
   → Revoke unused shares to reduce surface area

7. Monitor consumer query patterns
   → Producer can see: which shared tables are queried
   → SVL_DATASHARE_USAGE_CONSUMER: consumer-side query tracking
   → SVL_DATASHARE_USAGE_PRODUCER: producer-side access tracking
   → Identify: unused shared tables → candidates for removal

8. Set up notifications for schema changes
   → If producer drops a shared table → consumer queries break
   → Communicate schema changes BEFORE making them
   → Use SNS notification (from Project 54) when shared schema changes

❌ WHAT SHOULD NOT DO

1. DO NOT share system tables or pg_catalog
   → System tables contain: user credentials, cluster config, internal state
   → Sharing these = security risk
   → Only share user-created schemas with business data

2. DO NOT share tables with PII without access controls
   → Data sharing is READ-ONLY but consumer can SELECT all columns
   → If orders_star has customer_email → consumer sees it
   → Redshift doesn't support column-level sharing within datashares
   → Fix: create VIEWS that exclude PII columns → share the views

3. DO NOT assume cross-region data sharing is free
   → Same-region: no data transfer cost
   → Cross-region: data transfer charges apply ($0.01-$0.02/GB)
   → 500 GB queried cross-region daily = $5-$10/day
   → Keep producer and consumer in SAME region when possible

4. DO NOT forget to authorize at BOTH levels
   → Level 1: Redshift datashare grants (within cluster)
   → Level 2: AWS account authorization (for cross-account)
   → Missing either level → consumer can't access
   → Most common failure: forgot account-level authorization

5. DO NOT modify shared tables without communicating
   → Producer drops column → consumer queries referencing it break
   → Producer renames table → consumer references break
   → Treat shared schema like a PUBLIC API — versioned, documented
   → Breaking changes require consumer notification + migration window

6. DO NOT create circular sharing
   → Account A shares with Account B
   → Account B shares derived tables with Account A
   → Creates dependency cycle → changes in either break the other
   → Keep data flow unidirectional: producer → consumer

7. DO NOT grant ALL PRIVILEGES on datashare
   → Only USAGE is needed for consumers
   → GRANT USAGE ON DATASHARE share_name TO NAMESPACE 'uuid'
   → Not: GRANT ALL ON DATASHARE (doesn't exist but conceptually wrong)

🚫 WHEN NOT TO USE REDSHIFT DATA SHARING

1. When data is in S3 (not Redshift)
   → Data Sharing is for Redshift-internal tables only
   → S3 data: use cross-account S3 + Spectrum (Projects 56-57)
   → Don't COPY S3 data into Redshift just to share it via Data Sharing

2. When consumer needs to WRITE to shared tables
   → Data Sharing is READ-ONLY
   → Consumer cannot INSERT, UPDATE, DELETE shared data
   → If write needed: consumer writes to LOCAL tables, producer reads via separate share

3. When using dc2 or ds2 node types
   → Data Sharing requires Redshift Managed Storage (RMS)
   → dc2/ds2 use local SSD storage → no RMS → no data sharing
   → Upgrade to RA3 or migrate to Serverless

4. When cross-region latency is unacceptable
   → Cross-region data sharing: higher latency for every query
   → Large result sets: significant data transfer time
   → If consumer queries are latency-sensitive: replicate to same region

5. When sharing with non-Redshift consumers
   → Data Sharing is Redshift-to-Redshift only
   → Athena, Glue, external tools: can't consume datashares
   → For non-Redshift consumers: UNLOAD to S3, or use Redshift data API

6. When you need row-level security on shared data
   → Data Sharing doesn't support RLS across clusters
   → Consumer sees ALL rows in shared tables
   → Fix: create filtered views in producer → share the views
   → Or: use Lake Formation for row-level security (Project 74)

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

┌──────────────────────────────────────────────────────────────────────────────┐
│            REDSHIFT DATA SHARING — INTERNAL MECHANICS                        │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  COMPONENT 1: REDSHIFT MANAGED STORAGE (RMS)                                 │
│  ═══════════════════════════════════════════                                  │
│  → RA3 nodes and Serverless use RMS (separate from compute)                  │
│  → Data stored in S3-backed managed storage layer                            │
│  → Multiple clusters can read SAME storage blocks                            │
│  → This is what enables zero-copy sharing                                    │
│                                                                              │
│  Without RMS (dc2/ds2):                                                      │
│  ┌──────────┐                                                                │
│  │ Compute  │ ← data stored on LOCAL SSD                                     │
│  │ + Storage│ ← only THIS cluster can read it                                │
│  └──────────┘ ← no sharing possible                                          │
│                                                                              │
│  With RMS (RA3/Serverless):                                                  │
│  ┌──────────┐     ┌──────────────────┐     ┌──────────┐                     │
│  │ Compute  │────→│ Managed Storage  │←────│ Compute  │                     │
│  │ (Prod)   │     │ (shared S3 layer)│     │(Analytics)│                    │
│  └──────────┘     └──────────────────┘     └──────────┘                     │
│  Producer reads     Data lives HERE         Consumer reads                   │
│  and writes         Both clusters access    (read-only)                      │
│                     SAME physical blocks                                     │
│                                                                              │
│                                                                              │
│  COMPONENT 2: DATASHARE OBJECT                                               │
│  ═════════════════════════════                                                │
│  → Named container created on producer cluster                               │
│  → Contains references to schemas, tables, views, UDFs                       │
│  → Does NOT contain actual data (just metadata pointers)                     │
│  → Created via SQL: CREATE DATASHARE share_name                              │
│                                                                              │
│  DATASHARE LIFECYCLE:                                                        │
│  1. CREATE DATASHARE share_name;                                             │
│  2. ALTER DATASHARE share_name ADD SCHEMA schema_name;                       │
│  3. ALTER DATASHARE share_name ADD ALL TABLES IN SCHEMA schema_name;         │
│  4. GRANT USAGE ON DATASHARE share_name TO NAMESPACE 'consumer-uuid';        │
│  5. (Cross-account): authorize via AWS API                                   │
│                                                                              │
│                                                                              │
│  COMPONENT 3: CROSS-ACCOUNT AUTHORIZATION                                    │
│  ════════════════════════════════════════                                     │
│  → Within same account: only Redshift GRANT needed                           │
│  → Cross-account: TWO levels of authorization                                │
│                                                                              │
│  Level 1: Redshift-level (SQL)                                               │
│  → Producer: GRANT USAGE ON DATASHARE TO ACCOUNT '[REDACTED:BANK_ACCOUNT_NUMBER]'              │
│  → This makes the datashare VISIBLE to Account B                             │
│                                                                              │
│  Level 2: AWS account-level (API/Console)                                    │
│  → Producer: authorize_data_share(data_share_arn, consumer_account_id)       │
│  → Consumer: associate_data_share_consumer(data_share_arn, consumer_arn)      │
│  → This establishes the AWS-level trust between accounts                     │
│                                                                              │
│  BOTH levels required for cross-account access                               │
│                                                                              │
│                                                                              │
│  COMPONENT 4: CONSUMER DATABASE CREATION                                     │
│  ════════════════════════════════════════                                     │
│  → Consumer creates a LOCAL database from the datashare                      │
│  → CREATE DATABASE acct_a_shared FROM DATASHARE 'share_name'                 │
│    OF NAMESPACE 'producer-namespace-uuid';                                   │
│  → This database appears in consumer's catalog like any other DB             │
│  → Tables inside are READ-ONLY (backed by producer's storage)                │
│  → Consumer queries: SELECT * FROM acct_a_shared.schema.table               │
│                                                                              │
│                                                                              │
│  COMPONENT 5: QUERY EXECUTION FLOW                                           │
│  ═══════════════════════════════════                                          │
│  ┌──────────────────────────────────────────────────────────────┐            │
│  │ Consumer submits: SELECT * FROM acct_a_shared.analytics     │            │
│  │                   .daily_rollup WHERE order_date = '2025-01-15'          │
│  │                                                              │            │
│  │ 1. Consumer's query planner reads metadata from datashare    │            │
│  │    (table structure, column types, partition info)            │            │
│  │                                                              │            │
│  │ 2. Consumer's compute nodes generate scan plan               │            │
│  │    (which RMS blocks to read, predicate pushdown)            │            │
│  │                                                              │            │
│  │ 3. Consumer's compute nodes read data DIRECTLY from          │            │
│  │    producer's Managed Storage (RMS)                          │            │
│  │    → NO data flows through producer's compute nodes          │            │
│  │    → Consumer reads RMS blocks directly (like reading S3)    │            │
│  │                                                              │            │
│  │ 4. Consumer's compute processes the data (filter, aggregate) │            │
│  │                                                              │            │
│  │ 5. Results returned to consumer's client                     │            │
│  │                                                              │            │
│  │ KEY: Producer's cluster is NOT involved in query execution   │            │
│  │ Producer could be offline → consumer still queries (from RMS)│            │
│  │ (but new writes by producer would be invisible until online) │            │
│  └──────────────────────────────────────────────────────────────┘            │
│                                                                              │
│                                                                              │
│  COMPONENT 6: CONSISTENCY MODEL                                              │
│  ══════════════════════════════                                               │
│  → Consumer sees COMMITTED data from producer                                │
│  → Producer writes new row → COMMIT → consumer immediately sees it           │
│  → In-flight transactions on producer: NOT visible to consumer               │
│  → This is "read committed" isolation across clusters                        │
│  → No eventual consistency — it's transactionally consistent                 │
│                                                                              │
│                                                                              │
│  COMPONENT 7: NAMESPACE                                                      │
│  ════════════════════════                                                     │
│  → Every Redshift cluster/serverless workgroup has a NAMESPACE               │
│  → Namespace = UUID that uniquely identifies the cluster                     │
│  → Format: "a1b2c3d4-e5f6-7890-abcd-ef1234567890"                           │
│  → Used in: GRANT USAGE ON DATASHARE TO NAMESPACE 'uuid'                    │
│  → Used in: CREATE DATABASE ... OF NAMESPACE 'uuid'                          │
│  → Find it: SELECT current_namespace FROM stv_sessions LIMIT 1;             │
│             Or: Redshift console → Namespace column                          │
│                                                                              │
│                                                                              │
│  PRICING:                                                                    │
│  → Data Sharing itself: FREE (no additional charge)                          │
│  → Consumer compute: charged at normal Redshift rate                         │
│  → Cross-region data transfer: standard AWS transfer rates                   │
│  → No storage charge for consumer (data stays on producer)                   │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Cannot create datashare — cluster not RA3/Serverless"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: dc2 or ds2 node types don't support Managed Storage
→ Data Sharing requires RMS → requires RA3 or Serverless
→ Fix: migrate cluster from dc2 → RA3 (elastic resize)
→ Or: create new Serverless workgroup

PITFALL 2: "Datashare visible but consumer can't create database"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: missing AWS account-level authorization
→ Redshift GRANT done ✅ but API authorize_data_share not done ❌
→ Cross-account requires BOTH: SQL grant + API authorization
→ Fix: call redshift:AuthorizeDataShareConsumer API from producer account

PITFALL 3: "Consumer sees tables but queries return 0 rows"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: datashare was created before tables had data
→ Or: tables were truncated after sharing
→ Debug: query from PRODUCER to confirm data exists
→ Also check: is consumer's database pointing to correct namespace?

PITFALL 4: "Consumer query is slow — expected data sharing to be fast"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause 1: consumer cluster too small (not enough RPU/nodes)
→ Consumer's compute does ALL the work — needs adequate sizing
→ Cause 2: cross-region sharing → network latency on every block read
→ Cause 3: shared table lacks sort keys → full scan instead of range scan
→ Fix: size consumer appropriately, keep same region, ensure good table design

PITFALL 5: "Producer dropped shared table → consumer queries break"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Producer ALTERs or DROPs table → immediately affects consumer
→ No versioning, no grace period
→ Consumer's next query: "relation does not exist"
→ Fix: treat shared schema like API — document changes, notify consumers
→ Fix: producer tests changes in staging datashare first

PITFALL 6: "Consumer namespace UUID is wrong"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Provisioned cluster namespace ≠ Serverless namespace
→ Each workgroup in Serverless has its OWN namespace
→ Copy-paste errors in UUID → GRANT fails silently
→ Debug: SHOW DATASHARES; on producer → check granted namespaces
→ Verify: SELECT current_namespace on consumer

📈 SCALABILITY CONSIDERATIONS

DATA VOLUME:
→ 1 GB shared: negligible performance impact
→ 100 GB shared: normal query performance
→ 1 TB shared: ensure consumer has adequate compute
→ 10 TB+ shared: consider materialized views on producer for heavy aggregations
→ No limit on data volume in datashare

CONSUMER COUNT:
→ 1 consumer: simple setup
→ 5 consumers: same datashare, multiple GRANT statements
→ 20+ consumers: manage via automation (script the grants)
→ Max consumers per datashare: no hard limit
→ Each consumer uses own compute → no contention

TABLE COUNT:
→ 1-10 tables: add individually or via schema
→ 50+ tables: share entire schema (auto-includes new tables)
→ 100+ tables: consider multiple datashares by domain
→ No limit on objects per datashare
