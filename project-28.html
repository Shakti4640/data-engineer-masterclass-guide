🟡 HOW TO APPROACH IT

Step 1: Understand the COMPLETE event chain
→ S3 PutObject event fires when file lands
→ S3 sends notification to SNS topic
→ SNS fans out to:
   a) Email subscription (team notification)
   b) SQS queue (processing buffer)
→ EC2 worker polls SQS (long polling — Project 27)
→ Worker receives message → downloads file → processes → deletes message
→ Worker sends success/failure notification via SNS

Step 2: Understand WHY SNS sits between S3 and SQS
→ S3 can send directly to SQS (Project 18)
→ BUT: S3 event notification allows only ONE destination per event type per prefix
→ With SNS in the middle: ONE S3 event → SNS → MULTIPLE consumers
   a) SQS queue (for processing)
   b) Email (for team awareness)
   c) Lambda (for future use — Project 41)
   d) Another SQS queue (for audit logging)
→ This is the SNS → SQS FAN-OUT pattern (Project 14)

Step 3: Map the data flow with error handling
→ Happy path: file → notify → queue → process → load → done
→ Bad file: file → notify → queue → process → VALIDATION FAIL → retry → DLQ
→ Worker crash: file → notify → queue → process (crash) → timeout → retry
→ S3 delete before processing: file → queue → process → 404 → skip
→ Every failure mode must be handled gracefully

Step 4: Design for observability
→ Every step must be VISIBLE:
   - S3 event: CloudTrail logs
   - SNS delivery: CloudWatch metrics
   - SQS message: queue depth metrics
   - Worker: application logs + custom metrics
   - MariaDB load: row count validation
   - End-to-end: latency from S3 event to DB load

🔵 WHAT COULD BE DONE

Option A: S3 → SQS → EC2 (direct, no SNS)
→ Simpler: one fewer service
→ BUT: no email notification
→ BUT: can't add more consumers later without reconfiguring S3
→ REJECTED for production (not extensible)

Option B: S3 → SNS → SQS → EC2 (recommended)
→ SNS enables fan-out: email + SQS + future Lambda
→ Adding consumers = adding SNS subscription (no S3 changes)
→ S3 configured once → never touched again
→ SELECTED ✅

Option C: S3 → EventBridge → Step Functions → EC2
→ More sophisticated orchestration
→ Overkill for single-file processing
→ Better for: multi-step workflows (Project 41, 55, 66)

Option D: S3 → Lambda → MariaDB
→ Serverless — no EC2 management
→ BUT: Lambda 15-min timeout limits large file processing
→ BUT: Lambda → MariaDB requires VPC configuration
→ Good for small files — not for production flexibility

DECISION: Option B — S3 → SNS → SQS → EC2
→ Extensible fan-out pattern
→ Reliable with DLQ
→ EC2 handles large files and long processing
→ Combines all pieces from Projects 17, 18, 27

✅ WHAT SHOULD DO

1. Configure S3 event notification to SNS (not directly to SQS)
   → SNS enables fan-out: multiple consumers from one event
   → Adding email/Lambda/second queue = just add subscription
   → No need to reconfigure S3 ever again

2. Use prefix AND suffix filters on S3 notification
   → Prefix: incoming/shipfast/ (only ShipFast files)
   → Suffix: .csv (only CSV files, ignore .tmp or .metadata)
   → Prevents: processing temp files, checkpoint files, folders

3. Handle SNS-wrapped messages in worker
   → S3 → SNS → SQS: message body is SNS envelope containing S3 event
   → Different from S3 → SQS direct (plain S3 event)
   → Worker must unwrap SNS layer first (coded in Project 27)

4. Validate file BEFORE loading into database
   → Check: file exists in S3 (may have been deleted)
   → Check: file size > 0 (empty file = source error)
   → Check: CSV header matches expected schema
   → Check: row count > 0
   → Check: data types valid (no strings in numeric columns)
   → REJECT invalid files → send to DLQ → alert

5. Use staging table in MariaDB (not production table directly)
   → Load into staging.shipfast_updates first
   → Validate data in staging
   → Then merge into production table
   → If staging load fails: production untouched

6. Track processing state for idempotency
   → Before processing: check "have I already processed this file?"
   → Track in: MariaDB table (processed_files) or S3 marker
   → If already processed: delete SQS message, skip

7. Send completion notification with metrics
   → "File processed: shipping_updates_20250130.csv"
   → Rows loaded: 5,432
   → Processing time: 8.3 seconds
   → Latency from S3 arrival: 47 seconds

❌ WHAT SHOULD NOT DO

1. DO NOT configure S3 events to SQS directly in this pattern
   → Loses fan-out capability
   → Can't add email or Lambda without reconfiguring S3
   → S3 → SNS → SQS gives same reliability + extensibility

2. DO NOT process files without validation
   → Partner sends corrupted file → loads garbage into MariaDB
   → Always validate BEFORE loading
   → Reject early, fail loud

3. DO NOT load directly into production tables
   → Bad data in production = broken dashboards, wrong reports
   → ALWAYS: S3 → staging → validate → production
   → Staging is your safety net

4. DO NOT ignore the SNS wrapper in SQS messages
   → S3 → SNS → SQS wraps the event in SNS envelope
   → Parsing raw body as S3 event → JSON parse error
   → Worker must detect and unwrap SNS layer (Project 27 handles this)

5. DO NOT hardcode partner-specific logic in the worker
   → Today: ShipFast
   → Tomorrow: 3 more partners with different schemas
   → Design worker to: read config per partner, apply correct schema
   → Configuration-driven, not code-driven

6. DO NOT skip the end-to-end test
   → Unit testing individual pieces is not enough
   → Must test: drop file in S3 → verify it appears in MariaDB
   → Automated integration test catches: permission issues, 
     format mismatches, network problems

🚫 WHEN NOT TO USE THIS PATTERN

1. When processing must happen in < 1 second
   → S3 event → SNS → SQS → EC2 poll adds 1-25 seconds latency
   → Use: S3 → Lambda (direct invocation, ~100ms)
   → Or: API Gateway → Lambda for synchronous processing

2. When file arrives via API (not S3 drop)
   → S3 events only fire on S3 operations
   → Use: API Gateway → SQS → EC2 for API-driven workflows

3. When you need exactly-once processing guarantee
   → SNS → SQS Standard may deliver duplicate messages
   → Use: SQS FIFO + deduplication
   → Or: implement idempotency in worker (recommended)

4. When there are 50+ different file types with different schemas
   → Single worker handling all types becomes complex
   → Use: SNS → multiple SQS queues (filter by file type)
   → Each queue has specialized worker (Project 51)

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "S3 event notification not firing"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause A: notification not configured on the correct prefix
→ Cause B: SNS topic policy doesn't allow S3 to publish
→ Cause C: file uploaded to wrong prefix (incoming/ vs incoming/shipfast/)
→ Debug: upload test file → check SNS delivery metrics in CloudWatch
→ Fix: verify S3 notification config + SNS topic policy

PITFALL 2: "SNS delivers to email but NOT to SQS"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: SQS queue policy doesn't allow SNS to send messages
→ The SQS queue needs explicit policy: Allow SNS topic to SendMessage
→ Debug: SNS delivery status logs (if enabled)
→ Fix: update SQS queue policy (coded in this project)

PITFALL 3: "Worker receives message but can't parse S3 event"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: S3 → SNS → SQS wraps event in TWO layers
→ SQS message body = SNS envelope → inside = S3 event JSON
→ Worker from Project 27 handles this (parse_s3_event function)
→ ALWAYS log raw message body on parse failure for debugging

PITFALL 4: "File processed twice after worker restart"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: worker processed file but crashed before delete_message
→ Message reappears → processed again → duplicate data in MariaDB
→ Fix: idempotency check before processing
→ Check: SELECT COUNT(*) FROM processed_files WHERE file_key = ?
→ If exists: skip processing, just delete the message

PITFALL 5: "S3 event fires for .tmp files during multipart upload"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: some upload tools create temp files then rename
→ S3 fires event for EVERY object creation
→ Worker receives .tmp file → processing fails → clogs DLQ
→ Fix: suffix filter in S3 notification (suffix=.csv)
→ Also: worker validates file extension before processing

PITFALL 6: "Email notifications arrive but processing doesn't start"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: SNS delivers to email (works) but SQS delivery fails (silently)
→ Email subscription and SQS subscription are INDEPENDENT
→ One can succeed while other fails
→ Debug: check SQS queue depth — if 0 after file drop, SQS didn't receive
→ Fix: verify SQS subscription is confirmed + queue policy allows SNS
