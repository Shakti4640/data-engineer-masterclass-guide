✅ WHAT SHOULD DO

1. Add audit columns to EVERY table
   → created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
   → updated_at TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
   → WHY: Project 42 (CDC) uses updated_at as watermark column
   → Without these, you CANNOT do incremental extraction

2. Design indexes based on QUERY PATTERNS, not table structure
   → "I search orders by date + status" → INDEX (order_date, status)
   → "I look up reviews by product" → INDEX (product_id, created_at)
   → NEVER index every column — each index slows writes

3. Use FOREIGN KEYS to enforce referential integrity
   → Prevents orphan records (order_item without order)
   → Database enforces rules even if application has bugs
   → ON DELETE CASCADE vs RESTRICT — choose deliberately

4. Use ENUM for finite value sets
   → status ENUM('pending','shipped','completed','cancelled','refunded')
   → Database rejects invalid values — no "compelted" typos
   → Storage: 1-2 bytes vs VARCHAR's variable length

5. Use DECIMAL for money — NEVER FLOAT
   → FLOAT: 0.1 + 0.2 = 0.30000000000000004
   → DECIMAL(10,2): 0.1 + 0.2 = 0.30 (exact)
   → One rounding error × 1 million transactions = lawsuit

6. Use GENERATED columns for derived values
   → line_total GENERATED ALWAYS AS (quantity * unit_price) STORED
   → Database guarantees correctness — application can't miscalculate
   → Already done in Project 8's order_items table

7. Plan for CDC from Day 1
   → Add updated_at column to every table
   → Add _etl_watermarks table to track last extraction time
   → This enables incremental extraction in Projects 31, 42

❌ WHAT SHOULD NOT DO

1. DO NOT create indexes on every column
   → Each index = separate B+Tree on disk
   → Every INSERT/UPDATE must update ALL indexes
   → 10 indexes on a table = 10x write amplification
   → RULE: index only columns in WHERE, JOIN, ORDER BY

2. DO NOT use VARCHAR(255) for everything
   → customer_id is max 20 chars → use VARCHAR(20)
   → state is max 2 chars → use CHAR(2)
   → Smaller columns = more rows per page = faster scans

3. DO NOT store addresses as part of customer table
   → Customer has multiple addresses (billing, shipping, old)
   → Separate table with FK to customer
   → Violating this = nightmare when customer moves

4. DO NOT use FLOAT/DOUBLE for monetary values
   → Already explained in SHOULD DO — this is critical enough to repeat
   → DECIMAL(10,2) for prices, DECIMAL(12,2) for totals

5. DO NOT skip foreign keys "for performance"
   → FK overhead on INSERT: microseconds
   → Cost of orphan data in production: hours of debugging
   → FK creates implicit index on child column — actually HELPS reads

6. DO NOT create redundant indexes
   → INDEX(order_date) is redundant if INDEX(order_date, status) exists
   → Leftmost prefix rule: composite index serves single-column queries
   → MariaDB can use (order_date, status) to search by order_date alone

7. DO NOT use ON DELETE CASCADE without thinking
   → CASCADE on orders → order_items: correct (items belong to order)
   → CASCADE on customers → orders: DANGEROUS (deleting customer deletes orders)
   → Use RESTRICT for critical relationships — force explicit deletion

8. DO NOT add indexes before measuring
   → Run EXPLAIN on slow queries FIRST
   → Identify which queries actually need optimization
   → Add the MINIMUM index that solves the problem

🚫 WHEN NOT TO USE

1. DO NOT use composite index when queries only filter on second column
   → INDEX(order_date, status): works for WHERE order_date = X
   → INDEX(order_date, status): works for WHERE order_date = X AND status = Y
   → INDEX(order_date, status): DOES NOT work for WHERE status = Y alone
   → Leftmost prefix rule — column order in index MATTERS

2. DO NOT use transactions for read-only queries
   → SELECT queries don't need BEGIN/COMMIT
   → InnoDB provides consistent reads via MVCC automatically
   → Wrapping SELECTs in transactions wastes lock resources

3. DO NOT use LOAD DATA INFILE for small batches (< 1000 rows)
   → Setup overhead exceeds benefit
   → executemany() with 1000 rows is fast enough
   → LOAD DATA INFILE shines at 100K+ rows

4. DO NOT use ENUM if values change frequently
   → Adding new ENUM value requires ALTER TABLE
   → ALTER TABLE on 10M row table: locks table for minutes
   → If values change monthly: use VARCHAR + application validation
   → If values are truly fixed (status, tier): ENUM is perfect

5. DO NOT use history tables for frequently queried current state
   → History table grows forever — queries slow down
   → Keep current state in main table (UPDATE)
   → History table is append-only AUDIT LOG
   → Query main table for current, history for timeline

│              DEEP INTERNALS: INDEXES, TRANSACTIONS, BULK LOADS             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ═══════════════════════════════════════════════════════                     │
│  1. COMPOSITE INDEX INTERNALS                                               │
│  ═══════════════════════════════════════════════════════                     │
│                                                                             │
│  INDEX idx_date_status (order_date, status):                                │
│                                                                             │
│  B+Tree is sorted by (date, status) TOGETHER:                               │
│                                                                             │
│  ┌──────────────────────────────────────────────┐                           │
│  │  Leaf nodes (sorted):                        │                           │
│  │  (2025-01-01, cancelled)  → PK: ORD-001     │                           │
│  │  (2025-01-01, completed)  → PK: ORD-002     │                           │
│  │  (2025-01-01, completed)  → PK: ORD-005     │                           │
│  │  (2025-01-01, pending)    → PK: ORD-003     │                           │
│  │  (2025-01-01, shipped)    → PK: ORD-004     │                           │
│  │  (2025-01-02, cancelled)  → PK: ORD-008     │                           │
│  │  (2025-01-02, completed)  → PK: ORD-006     │                           │
│  │  ...                                         │                           │
│  └──────────────────────────────────────────────┘                           │
│                                                                             │
│  WHY (date, status) order matters:                                          │
│                                                                             │
│  Query: WHERE order_date = '2025-01-01' AND status = 'completed'            │
│  → Seek to (2025-01-01, completed) in tree — O(log N)                       │
│  → Scan consecutive entries — all matching rows are ADJACENT                │
│  → Stop at first non-matching entry                                         │
│  → Result: 2 rows examined, 2 rows returned                                │
│                                                                             │
│  Query: WHERE order_date BETWEEN '2025-01-01' AND '2025-01-31'              │
│  → Seek to (2025-01-01, ...) — O(log N)                                    │
│  → Scan forward until (2025-02-01, ...)                                     │
│  → ALL January rows are in consecutive leaf pages                           │
│  → Result: range scan — very efficient                                      │
│                                                                             │
│  Query: WHERE status = 'completed' (NO date filter)                         │
│  → Index sorted by date FIRST, status SECOND                                │
│  → 'completed' entries scattered across all dates                           │
│  → Must scan ENTIRE index — no better than full table scan                  │
│  → THIS is the leftmost prefix rule                                         │
│                                                                             │
│  ═══════════════════════════════════════════════════════                     │
│  2. COVERING INDEX INTERNALS                                                │
│  ═══════════════════════════════════════════════════════                     │
│                                                                             │
│  Normal query with index:                                                   │
│  SELECT * FROM orders WHERE order_date = '2025-01-01'                       │
│  Step 1: Search idx_date_status → find PK values                            │
│  Step 2: Use PK to look up FULL ROW from main table ← EXTRA I/O            │
│  This "Step 2" is called a BOOKMARK LOOKUP                                  │
│                                                                             │
│  Covering index query:                                                      │
│  SELECT order_date, status, COUNT(*)                                        │
│  FROM orders                                                                │
│  WHERE order_date = '2025-01-01'                                            │
│  GROUP BY order_date, status                                                │
│  → All columns (order_date, status) ARE IN the index                        │
│  → No bookmark lookup needed — answer entirely from index                   │
│  → EXPLAIN shows: "Using index" in Extra column                             │
│  → 2-3x faster than bookmark lookup                                        │
│                                                                             │
│  ═══════════════════════════════════════════════════════                     │
│  3. TRANSACTION INTERNALS (InnoDB)                                          │
│  ═══════════════════════════════════════════════════════                     │
│                                                                             │
│  BEGIN;                                                                     │
│    INSERT INTO orders (...) VALUES (...);                                    │
│    INSERT INTO order_items (...) VALUES (...);                               │
│    INSERT INTO order_items (...) VALUES (...);                               │
│  COMMIT;                                                                    │
│                                                                             │
│  WHAT HAPPENS INTERNALLY:                                                   │
│                                                                             │
│  1. BEGIN:                                                                  │
│     → InnoDB assigns transaction ID (incrementing counter)                  │
│     → Creates read view (snapshot of database at this moment)               │
│     → No locks acquired yet                                                 │
│                                                                             │
│  2. First INSERT:                                                           │
│     → New row written to Buffer Pool (RAM)                                  │
│     → Row marked with this transaction's ID                                 │
│     → UNDO LOG entry created (how to reverse this insert)                   │
│     → REDO LOG entry created (how to replay this insert)                    │
│     → Exclusive lock (X-lock) acquired on the row                           │
│     → Other transactions CAN'T see this row yet (isolation)                 │
│                                                                             │
│  3. Second INSERT, Third INSERT:                                            │
│     → Same process — undo log grows, redo log grows                         │
│     → All rows locked, all invisible to other transactions                  │
│                                                                             │
│  4. COMMIT:                                                                 │
│     → Redo log flushed to DISK (durability guarantee)                       │
│     → Transaction marked as committed                                       │
│     → Locks released                                                        │
│     → Rows become visible to other transactions                             │
│     → Dirty pages in Buffer Pool will flush to disk LATER (async)           │
│                                                                             │
│  5. If ROLLBACK instead of COMMIT:                                          │
│     → InnoDB reads UNDO LOG                                                 │
│     → Reverses each operation (delete inserted rows)                        │
│     → Locks released                                                        │
│     → As if nothing happened                                                │
│                                                                             │
│  6. If SERVER CRASHES between INSERT and COMMIT:                            │
│     → On restart: InnoDB reads redo log                                     │
│     → Replays committed transactions                                        │
│     → Rolls back uncommitted transactions (using undo log)                  │
│     → Database is consistent — no half-written orders                       │
│                                                                             │
│  ═══════════════════════════════════════════════════════                     │
│  4. BULK INSERT INTERNALS                                                   │
│  ═══════════════════════════════════════════════════════                     │
│                                                                             │
│  ROW-BY-ROW INSERT (slow):                                                  │
│  for row in 100000_rows:                                                    │
│      cursor.execute("INSERT ... VALUES (%s)", row)                          │
│      connection.commit()                                                    │
│  → 100,000 network round trips                                              │
│  → 100,000 transaction commits                                              │
│  → 100,000 redo log flushes to disk                                         │
│  → ~400 seconds (4 hours for 500K rows)                                     │
│                                                                             │
│  BATCH INSERT (fast):                                                       │
│  cursor.executemany("INSERT ... VALUES (%s)", all_100000_rows)              │
│  connection.commit()                                                        │
│  → 1 network round trip (batched)                                           │
│  → 1 transaction commit                                                     │
│  → 1 redo log flush                                                         │
│  → ~4 seconds (100x faster)                                                 │
│                                                                             │
│  LOAD DATA INFILE (fastest):                                                │
│  LOAD DATA LOCAL INFILE '/path/file.csv' INTO TABLE orders ...              │
│  → MariaDB's native bulk loader                                             │
│  → Bypasses SQL parser for each row                                         │
│  → Direct page writes to InnoDB                                             │
│  → ~1.5 seconds for 100K rows                                               │
│  → BUT: requires local_infile=1 (security risk — disabled in Project 8)     │
│                                                                             │
│  OPTIMIZATION TRICKS FOR BULK INSERT:                                       │
│  → Disable indexes: ALTER TABLE t DISABLE KEYS; (MyISAM only)              │
│  → For InnoDB: insert in PRIMARY KEY order                                  │
│  → Increase bulk_insert_buffer_size                                         │
│  → Set innodb_autoinc_lock_mode = 2 (interleaved)                           │
│  → Commit every 10,000 rows (not every row, not all 1M at once)            │
│                                                                             │
│  ═══════════════════════════════════════════════════════                     │
│  5. FOREIGN KEY ENFORCEMENT INTERNALS                                       │
│  ═══════════════════════════════════════════════════════                     │
│                                                                             │
│  INSERT INTO order_items (order_id, product_id, ...) VALUES ('ORD-999',...) │
│                                                                             │
│  InnoDB does:                                                               │
│  1. Acquire shared lock (S-lock) on parent row in orders table              │
│  2. Check: does order_id='ORD-999' exist in orders?                         │
│  3. If YES: allow insert, release lock                                      │
│  4. If NO: reject with ERROR 1452 (foreign key constraint fails)            │
│                                                                             │
│  ON DELETE CASCADE:                                                         │
│  DELETE FROM orders WHERE order_id = 'ORD-001'                              │
│  1. InnoDB finds all order_items with order_id = 'ORD-001'                  │
│  2. Deletes those child rows FIRST                                          │
│  3. Then deletes parent order row                                           │
│  4. All within same transaction — atomic                                    │
│                                                                             │
│  ON DELETE RESTRICT:                                                        │
│  DELETE FROM customers WHERE customer_id = 'CUST-001'                       │
│  1. InnoDB checks: any orders reference CUST-001?                           │
│  2. If YES: reject with ERROR 1451 (cannot delete parent row)               │
│  3. Must delete/reassign orders FIRST, then delete customer                 │
│                                                                             │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Query uses index for 1000 rows but full scan for 100K"
→ Cause: MariaDB optimizer decides full scan is cheaper when 
   index would return > 20-30% of table
→ This is CORRECT behavior — random I/O from index is slower 
   than sequential full scan above a threshold
→ Fix: accept it OR force with FORCE INDEX (rarely needed)

PITFALL 2: "Transaction deadlock — both transactions killed"
→ Cause: Transaction A locks row 1, waits for row 2
         Transaction B locks row 2, waits for row 1
→ InnoDB detects deadlock, kills one transaction (victim)
→ Fix: always access tables/rows in SAME ORDER across all code
→ Fix: keep transactions SHORT — don't hold locks during API calls

PITFALL 3: "ALTER TABLE adding index locks table for 20 minutes"
→ Cause: InnoDB must build entire B+Tree for new index
→ On 10M rows: can take 10-30 minutes, table is locked
→ Fix: use pt-online-schema-change (Percona toolkit)
→ Fix: schedule during maintenance window
→ In RDS: ALTER TABLE is online for most operations

PITFALL 4: "executemany() is not actually faster"
→ Cause: pymysql's executemany still sends individual INSERTs
→ It just loops cursor.execute() internally
→ Fix: use multi-row INSERT VALUES syntax manually:
   INSERT INTO t VALUES (1,'a'), (2,'b'), (3,'c')
→ OR: set rewriteBatchedStatements=true in connection string

PITFALL 5: "Inserted duplicate order — UNIQUE constraint missing"
→ Cause: application generates order_id, no DB uniqueness check
→ Two concurrent requests create same order_id
→ Fix: PRIMARY KEY or UNIQUE constraint catches this at DB level
→ Application gets error, retries with new ID

PITFALL 6: "History table has 50M rows — queries crawling"
→ Cause: no partition strategy, no archiving
→ Fix: partition by month (PARTITION BY RANGE)
→ Fix: archive old partitions to S3 (Project 20)
→ Fix: only query recent data, use Athena for historical (Project 24)

PITFALL 7: "updated_at not changing on UPDATE"
→ Cause: ON UPDATE CURRENT_TIMESTAMP only fires when 
   row VALUES actually change
→ UPDATE customers SET name='John' WHERE name='John' → no change
→ This is CORRECT behavior — but surprises CDC systems
→ Fix: for CDC, also check row content hash (Project 42)

📈 SCALABILITY CONSIDERATIONS

TABLE SIZE GUIDELINES (MariaDB on t3.medium):
→ < 1M rows per table: no tuning needed, everything fast
→ 1M-10M rows: proper indexing critical, buffer pool must be sized
→ 10M-100M rows: partitioning needed, consider read replicas
→ > 100M rows: MariaDB struggling — extract to Redshift for analytics

INDEX SCALING:
→ Each index adds ~30-50% overhead on INSERT operations
→ 2-3 indexes per table: healthy
→ 5+ indexes per table: writes slow down noticeably
→ 10+ indexes: something is wrong with your query patterns

TRANSACTION SCALING:
→ Short transactions (< 100ms): handle 500+ TPS on t3.medium
→ Long transactions (> 5 seconds): block other operations
→ RULE: transaction should touch < 1000 rows
→ Bulk operations: commit every 10,000 rows

HISTORY TABLE SCALING:
→ 1 year of order history: ~730K rows (2000 orders/day)
→ 3 years: ~2.2M rows — still manageable
→ 5 years: ~3.6M rows — partition by year
→ Archival: old partitions → S3 CSV → query with Athena

WHEN TO MIGRATE OFF EC2 MariaDB:
→ Database > 50 GB: migrate to RDS (managed)
→ Need read replicas: RDS one-click setup
→ Need Multi-AZ failover: RDS only
→ Analytics queries slow down OLTP: extract to Redshift (Project 32)
→ > 500 concurrent connections: Aurora MySQL

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO ANALYZE QUERY PERFORMANCE:

1. EXPLAIN — shows execution plan
   → EXPLAIN SELECT * FROM orders WHERE order_date = '2025-01-15';
   → Look for:
     type: const (best) → eq_ref → ref → range → index → ALL (worst)
     key: which index is used (NULL = no index)
     rows: estimated rows examined
     Extra: "Using index" = covering index (excellent)
            "Using filesort" = sorting in temp space (slow)
            "Using temporary" = temp table created (slow)

2. EXPLAIN ANALYZE (MariaDB 10.1+) — shows ACTUAL execution
   → EXPLAIN ANALYZE SELECT ...
   → Shows real row counts, not estimates
   → Shows actual time per step

3. SHOW INDEX FROM table — list all indexes
   → Cardinality: number of unique values (higher = more selective)
   → Low cardinality index (e.g., status with 5 values) = less useful
   → High cardinality index (e.g., order_id) = very useful

4. slow_query_log — catches problems in production
   → Enabled in Project 8 configuration
   → Any query > 2 seconds logged with full SQL
   → Review weekly — fix top offenders

5. SHOW TABLE STATUS — storage metrics
   → Row count, data size, index size, average row length
   → Useful for capacity planning

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ TRANSACTIONS: BEGIN/COMMIT/ROLLBACK = all-or-nothing operations
   → InnoDB uses redo log (durability) + undo log (rollback)
   → Never commit between related INSERT operations
   → FOR UPDATE locks rows during read-modify-write cycles

→ INDEXES: B+Tree structures that turn O(N) into O(log N)
   → Composite index: column ORDER matters (leftmost prefix rule)
   → Covering index: all SELECT columns in index = no table access
   → EXPLAIN shows: type, key, rows, Extra — learn to read these
   → Never put functions on indexed columns in WHERE

→ BULK OPERATIONS: multi-row INSERT = 100x faster than row-by-row
   → Commit per row = redo log flush per row = catastrophically slow
   → Batch + periodic commit = practical for production loads

→ SCHEMA DESIGN: audit columns + history tables + ETL watermarks
   → updated_at enables CDC (incremental extraction)
   → History tables enable timeline queries
   → Separate tables for 1:many relationships (addresses, reviews)
   → ENUM for finite values, DECIMAL for money, GENERATED for derived
