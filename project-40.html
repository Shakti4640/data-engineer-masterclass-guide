🟡 HOW TO APPROACH IT

Step 1: Understand Redshift resource model
→ Redshift has FIXED compute capacity (2 nodes × 2 slices = 4 slots)
→ Queries share these resources
→ Without WLM: first-come-first-served, no priority
→ One heavy query can monopolize all resources

Step 2: Understand WLM components
→ QUEUES: separate lanes for different workloads
→ SLOTS: how many concurrent queries per queue
→ MEMORY: how much RAM per queue/slot
→ TIMEOUT: max runtime per queue
→ PRIORITY: which queue gets resources first
→ QMR (Query Monitoring Rules): auto-kill runaway queries
→ SQA (Short Query Acceleration): fast-track tiny queries
→ CONCURRENCY SCALING: auto-add capacity for burst traffic

Step 3: Design queue strategy
→ Queue 1: "Dashboard" — high priority, many slots, short timeout
→ Queue 2: "ETL" — medium priority, few slots, long timeout
→ Queue 3: "Ad-hoc" — low priority, few slots, medium timeout
→ Queue 4: "Maintenance" — lowest priority, 1 slot, long timeout
→ Default: catches unclassified queries

Step 4: Dependencies from previous projects
→ Project 10: Redshift cluster
→ Project 32: ETL loading (competes with analysts)
→ Project 38: Spectrum queries (mixed with local queries)
→ Project 39: VACUUM/ANALYZE (competes with analysts)

🔵 WHAT COULD BE DONE

Option A: Increase cluster size (throw hardware at the problem)
→ 2 nodes → 4 nodes: $1,564/month → $3,128/month
→ More resources = less contention
→ Pros: simplest, no configuration
→ Cons: 2x cost, doesn't solve priority problem

Option B: WLM queues only (basic resource isolation)
→ Separate queues for dashboard vs ETL vs ad-hoc
→ Pros: isolates workloads, prevents one type from blocking another
→ Cons: fixed slots → can't handle burst traffic

Option C: WLM + SQA + Concurrency Scaling (RECOMMENDED)
→ WLM: separate queues with priority
→ SQA: fast-track short queries (dashboards)
→ Concurrency Scaling: auto-add capacity for bursts
→ QMR: auto-kill runaway queries
→ Pros: comprehensive, handles all scenarios
→ Cons: more configuration, requires monitoring

Option D: Migrate to Redshift Serverless (Project 67)
→ Auto-scales compute based on demand
→ No WLM configuration needed (serverless manages it)
→ Pros: zero management, auto-scales
→ Cons: potentially higher cost for sustained workloads

DECISION: Option C — WLM + SQA + Concurrency Scaling + QMR
→ Addresses all four complaints from the problem statement
→ Dashboard priority ensures CEO's dashboard always loads
→ ETL isolated from analyst impact
→ Runaway queries auto-killed
→ Burst traffic handled by concurrency scaling

✅ WHAT SHOULD DO

1. Create Redshift USER GROUPS for routing
   → dashboard_users: Tableau service account + analyst users
   → etl_users: Glue service account
   → adhoc_users: analysts with ad-hoc access
   → maintenance_users: admin/maintenance scripts
   → WLM routes queries to queues based on user group

2. Enable Short Query Acceleration (SQA) FIRST
   → Biggest immediate win — zero queue configuration needed
   → Dashboard queries (0.2-0.5s) skip all queues
   → Eliminates 90% of dashboard latency complaints
   → Enable: modify cluster parameter group → SQA = true

3. Configure WLM queues with Automatic WLM (recommended)
   → Automatic WLM: Redshift manages memory/slots dynamically
   → You define: queues, priorities, user group routing
   → Redshift decides: optimal slot count and memory per query
   → Much simpler than Manual WLM

4. Enable Concurrency Scaling for dashboard queue
   → Handles 9 AM burst: 40 queries when normal capacity is 15
   → Auto-adds transient clusters for overflow
   → 1 hour/day free → covers typical morning rush
   → Cost: $0 for first hour, then cluster-rate per second

5. Set Query Monitoring Rules (QMR) as safety nets
   → Kill: queries scanning > 1 billion rows
   → Kill: queries running > 5 minutes in dashboard queue
   → Log: queries with > 10 GB Spectrum scan
   → Hop: move long-running dashboard queries to ad-hoc queue

6. Schedule ETL and maintenance outside peak hours
   → ETL: 2:00-3:00 AM (Project 35 workflow)
   → VACUUM: 3:00-3:30 AM (Project 39 maintenance)
   → ANALYZE: 3:30-3:45 AM
   → If ETL overruns: ETL queue has lower priority → dashboards unaffected

7. Monitor queue utilization with CloudWatch
   → WLM queue metrics: query count, wait time, throughput
   → Concurrency Scaling: scaling events, cost
   → Set alarms: dashboard queue wait time > 5 seconds

❌ WHAT SHOULD NOT DO

1. DO NOT use Manual WLM unless you're an expert
   → Manual WLM: you set fixed slot count + memory per slot
   → Get it wrong: wasted memory or OOM errors
   → Automatic WLM: Redshift manages this dynamically
   → Amazon recommends Automatic WLM for almost all cases

2. DO NOT put all users in the same queue
   → Default: single queue with 5 concurrent slots
   → 40 queries → 35 queued → everyone waits
   → Separate queues ensure priority enforcement

3. DO NOT set dashboard timeout too high
   → Dashboard query should complete in < 5 seconds
   → If it takes > 30 seconds: something is wrong (bad query or table design)
   → Timeout = 30 seconds → kills stuck dashboard queries
   → User sees error → refreshes → fast retry

4. DO NOT enable Concurrency Scaling on ETL queue
   → ETL runs at 2 AM — no contention
   → Concurrency Scaling on ETL = paying for extra clusters at 2 AM
   → Only enable on dashboard queue (where burst happens)

5. DO NOT create more than 8 queues
   → Each queue reserves resources even when empty
   → Too many queues = fragmented resources
   → 4 queues covers 99% of use cases

6. DO NOT forget to assign users to groups
   → WLM routes by user group
   → If Tableau user not in dashboard_users group → goes to default queue
   → Default queue = low priority → slow dashboards
   → Verify: every service account and user is in correct group

7. DO NOT set QMR to kill queries without logging first
   → First: set QMR to LOG (observe what would be killed)
   → After 1 week: review logs → tune thresholds
   → Then: switch to ABORT (actually kill)
   → Prevents accidentally killing legitimate long queries

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "WLM configured but queries still go to default queue"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: User not added to the correct Redshift group
→ WLM routes by user GROUP, not individual user
→ Fix: ALTER GROUP dashboard_users ADD USER tableau_svc;
→ Verify: SELECT * FROM pg_group WHERE groname = 'dashboard_users';

PITFALL 2: "Concurrency Scaling enabled but not activating"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause A: Concurrency Scaling not enabled on the specific queue
→ Cause B: Query type not eligible (DDL, COPY not eligible)
→ Cause C: Free credits exhausted and billing not enabled
→ Fix: Verify queue has concurrency_scaling_mode = 'auto'
→ Check: svl_query_summary → look for concurrency_scaling flag

PITFALL 3: "SQA not accelerating dashboard queries"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Queries predicted as > 5 seconds (SQA threshold)
→ This happens when: table statistics are stale
→ Stale stats → model overestimates query time → not SQA eligible
→ Fix: Run ANALYZE (Project 39) → fresh stats → better SQA predictions

PITFALL 4: "QMR kills legitimate long-running query"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: QMR threshold too aggressive
→ Analyst's complex but valid query exceeds row scan limit
→ Fix: Start with LOG action (observe for 1 week)
→ Then: tune thresholds based on observed patterns
→ For specific users: put in separate queue with higher limits

PITFALL 5: "ETL queue starved — nightly load takes 3x longer"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Dashboard queue has HIGH priority → ETL gets minimum resources
→ During business hours: dashboards consume 90% of resources
→ ETL runs at 2 AM but overruns into 9 AM → gets deprioritized
→ Fix: Schedule ETL to finish before 8 AM (buffer time)
→ Fix: Give ETL queue MEDIUM priority (not LOW)
→ Fix: Set dashboard queue to reduce priority outside business hours

CONCURRENCY SCALING COST:
→ 1 hour/day FREE per cluster (covers morning rush)
→ Additional: $0.25 per transient cluster per second
→ Typical morning burst: 15 minutes → $0 (within free hour)
→ If burst exceeds 1 hour: ~$13.50/hour additional
→ Monthly estimate: $0-$50 depending on burst patterns

WHEN TO ADD NODES vs WLM:
→ Average utilization < 60%: WLM is sufficient
→ Average utilization 60-80%: WLM + Concurrency Scaling
→ Average utilization > 80% sustained: add nodes
→ Spiky (< 30% average, > 90% peak): Concurrency Scaling
→ Unpredictable: consider Redshift Serverless (Project 67)

🔑 KEY TAKEAWAY

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ WLM separates workloads into priority queues (dashboard > ETL > ad-hoc)
→ SQA bypasses all queues for short queries (< 5s predicted) — biggest win
→ Concurrency Scaling auto-adds clusters for burst traffic (1 hr/day free)
→ QMR auto-kills runaway queries (cross-joins, billion-row scans)
→ Automatic WLM is recommended — Redshift manages slots/memory dynamically
→ User GROUPS drive routing — must assign every user to correct group
→ Query GROUPS enable dynamic routing from scripts (SET query_group)
→ Priority doesn't kill lower-priority queries — reduces their resources
→ ANALYZE freshness affects SQA accuracy — stale stats → wrong predictions
→ Start QMR with LOG action → observe 1 week → then switch to ABORT
→ Dashboard queue + Concurrency Scaling = handles any burst size
→ ETL should ALWAYS finish before business hours to avoid contention
→ Monitor: queue wait times, SQA hit rate, QMR violations, scaling cost
