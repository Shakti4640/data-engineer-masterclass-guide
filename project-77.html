🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — now a 500-person company with a mature data platform
Platform: Full Data Mesh (Project 71) across 4 AWS accounts
          Bronze → Silver → Gold lake (Project 44/73)
          Lake Formation governing everything (Project 74)

Situation:
→ 380+ Glue ETL jobs running daily
→ 120+ Redshift tables (some via Spectrum from Project 38)
→ 90+ Athena views layered on top of each other
→ 45+ MariaDB source tables feeding the lake
→ 3 accounts: Ingestion (A), Processing (B), Serving (C)

INCIDENT:
→ CFO dashboard shows "total_revenue_adjusted" = $4.2M
→ Finance team says actual revenue is $5.8M — a $1.6M discrepancy
→ "total_revenue_adjusted" lives in:
     Redshift table: gold.finance_summary
→ That column is derived from:
     Athena view: silver.order_metrics
→ Which joins:
     Glue table: silver.orders_cleaned + silver.refunds_processed
→ "orders_cleaned" comes from:
     Glue ETL job: job_clean_orders (transforms bronze.raw_orders)
→ "raw_orders" comes from:
     MariaDB: ecommerce.orders (Account A, via JDBC — Project 31)
→ "refunds_processed" comes from:
     Another MariaDB: payments.refunds (Account A, different DB)
→ Somewhere in these 5+ hops, a filter is dropping valid rows

CTO: "We need to trace ANY column back to its EXACT origin,
      through every transformation, across every account.
      Build a lineage system."

CHALLENGE:
→ AWS does NOT have a native end-to-end lineage service
→ Glue tracks SOME metadata, CloudTrail tracks API calls
→ But COLUMN-LEVEL lineage across 5 hops? — You must BUILD it

🟡 HOW TO APPROACH IT

Step 1: Understand the TWO types of lineage

  TABLE-LEVEL LINEAGE (coarse):
  → "gold.finance_summary DEPENDS ON silver.order_metrics"
  → Tells you which tables feed which
  → Easier to build, less useful for debugging

  COLUMN-LEVEL LINEAGE (fine):
  → "gold.finance_summary.total_revenue_adjusted
      = SUM(silver.order_metrics.order_total)
        - SUM(silver.order_metrics.refund_amount)"
  → "silver.order_metrics.order_total
      = silver.orders_cleaned.quantity × silver.orders_cleaned.unit_price"
  → "silver.orders_cleaned.quantity
      = bronze.raw_orders.qty (renamed + cast to INT)"
  → "bronze.raw_orders.qty
      = MariaDB ecommerce.orders.quantity (JDBC extract)"
  → FULL TRACE: 5 hops from dashboard to source row

Step 2: Understand ACTIVE vs PASSIVE lineage collection

  PASSIVE (after-the-fact):
  → Parse Glue ETL scripts to infer transformations
  → Analyze CloudTrail logs for data movement
  → Inspect Glue Data Catalog for table dependencies
  → Pros: no code changes needed
  → Cons: incomplete, can't track column transforms

  ACTIVE (instrumented):
  → Each ETL job EMITS lineage events during execution
  → Events say: "I read column X from table A, applied transform T,
                  wrote result as column Y in table B"
  → Pros: precise, column-level, tracks transformations
  → Cons: requires modifying every ETL job

  DECISION: Hybrid approach
  → Active instrumentation for Glue ETL jobs (our code, we control)
  → Passive collection for Redshift/Athena (SQL parsing)
  → Catalog enrichment via Lake Formation tags (Project 74)

Step 3: Choose lineage storage

  → DynamoDB: fast lookups by node ID, good for graph traversal
  → S3 + Athena: cheap, queryable, good for historical lineage
  → Neptune (Graph DB): ideal for graph queries, but adds cost/complexity

  DECISION: DynamoDB for live lineage graph + S3 for historical archive
  → Neptune mentioned but not required for our scale

Step 4: Define the lineage data model

  NODE: any data asset (table, column, view, file)
  EDGE: any relationship (derives_from, feeds_into, transforms_to)
  EVENT: any execution that created an edge (job run, query, COPY)

Step 5: Build the system in layers

  Layer 1: Lineage Event Emitter (inside Glue jobs)
  Layer 2: Lineage Store (DynamoDB + S3)
  Layer 3: Lineage Query API (traverse the graph)
  Layer 4: Passive Collectors (CloudTrail, Catalog, SQL parser)
  Layer 5: Visualization (optional — graph rendering)

🔵 WHAT COULD BE DONE

Option A: Use AWS-native services only (Glue Catalog + CloudTrail + Tags)
→ Free/cheap, no extra infrastructure
→ Limited to table-level lineage, no column transforms
→ Cannot trace 5 hops automatically

Option B: Use Apache Atlas (open-source lineage tool)
→ Powerful, Hadoop-native
→ Heavy — requires dedicated EC2 cluster
→ Overkill for AWS-native stack

Option C: Use OpenLineage + Marquez (open standard)
→ Industry standard for lineage events
→ Requires running Marquez server
→ Good if you plan multi-cloud

Option D: Build custom lineage system (DynamoDB + S3 + Lambda)
→ AWS-native, serverless, cost-effective
→ Full control over column-level tracking
→ Must instrument Glue jobs manually

Option E: Commercial tool (DataHub, Atlan, Collibra)
→ Beautiful UI, auto-discovery
→ $50K-$500K/year licensing
→ Startup cannot afford this yet

DECISION: Option D with Option C's data model (OpenLineage-inspired)
→ Reason: column-level precision, AWS-native, extensible
→ Uses OpenLineage event schema but stores in DynamoDB/S3
→ Can migrate to full OpenLineage later if needed

✅ WHAT SHOULD DO

1. Define lineage at COLUMN level, not just table level
   → Table lineage answers: "what feeds what"
   → Column lineage answers: "WHERE did this specific number come from"
   → Column lineage is what actually solves the CFO's $1.6M question

2. Make lineage emission AUTOMATIC inside Glue jobs
   → Wrap DynamicFrame operations with lineage-aware functions
   → Every read, transform, write emits an event
   → Developer doesn't need to remember to log lineage

3. Store lineage events immutably
   → S3 for append-only historical record
   → DynamoDB for fast traversal queries
   → Never delete lineage — it's your audit trail

4. Include transformation semantics in edges
   → Not just "A → B" but "A.qty RENAMED TO B.quantity, CAST TO INT"
   → Without this, column lineage is useless

5. Assign unique IDs to every data asset
   → Format: {account}:{service}:{database}.{table}.{column}
   → Example: 111111111111:glue:silver.orders_cleaned.quantity
   → Enables cross-account lineage (from Project 71)

6. Track JOB RUN context
   → Which Glue job, which run ID, when, how long
   → Links lineage edges to specific executions
   → Essential for debugging "when did it break"

7. Use Lake Formation tags (Project 74) as classification layer
   → Tag columns as PII, Financial, Derived, etc.
   → Lineage + classification = governance

❌ WHAT SHOULD NOT DO

1. DO NOT try to infer lineage by parsing Glue scripts alone
   → PySpark code has dynamic logic (if/else, UDFs, joins)
   → Static analysis misses runtime behavior
   → Active instrumentation is the only reliable method

2. DO NOT store lineage only in Glue Data Catalog "description" fields
   → Description is free-text, no structure
   → Cannot traverse programmatically
   → Cannot query "show me all ancestors of column X"

3. DO NOT build lineage as an afterthought
   → Retrofitting lineage onto 380 existing jobs is 10x harder
   → Should have started in Project 44 (Medallion Architecture)
   → Lesson: instrument from Day 1

4. DO NOT track only the latest lineage state
   → Lineage changes over time (schema evolution — Project 45)
   → Must keep historical snapshots
   → "What was the lineage on March 15th?" is a real question

5. DO NOT ignore Athena views and Redshift views
   → Views are SQL-based transformations
   → They create lineage edges that are invisible to Glue
   → Must parse SQL definitions to extract column mappings

6. DO NOT make lineage emission synchronous and blocking
   → If DynamoDB write fails, ETL job should NOT fail
   → Lineage is observability — it must not break production
   → Use async/fire-and-forget with retry queue

7. DO NOT build a custom graph database
   → DynamoDB adjacency list pattern handles 95% of queries
   → Neptune only if you need complex graph traversals (PageRank, shortest path)
   → Premature complexity kills projects

🚫 WHEN NOT TO USE

1. DO NOT build column-level lineage for ad-hoc notebooks
   → Jupyter notebooks change constantly
   → Lineage of exploratory code = wasted effort
   → Only track PRODUCTION pipelines

2. DO NOT use lineage as a replacement for data quality checks
   → Lineage tells you WHERE bad data came from
   → Data quality (Project 68) tells you IF data is bad
   → They complement each other, neither replaces the other

3. DO NOT deploy Neptune just for lineage if you have < 1000 nodes
   → DynamoDB adjacency list handles small-to-medium graphs efficiently
   → Neptune minimum cost: ~$200/month (even idle)
   → Our graph: ~2000 nodes (tables + columns) — DynamoDB is fine

4. DO NOT track lineage for temporary/staging tables
   → Staging tables in Redshift (Project 65) are transient
   → They exist for minutes during MERGE operations
   → Track the logical flow: source → final target (skip staging)

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Lineage graph has cycles"
→ Cause: table A feeds table B feeds view C which is read by job that writes A
→ Symptom: traversal runs forever or OOMs
→ Fix: track visited nodes during BFS/DFS, skip already-visited
→ Also: cycles indicate design smell — investigate the pipeline

PITFALL 2: "Lineage says column X comes from column Y, but values don't match"
→ Cause: lineage tracks STRUCTURAL lineage (schema), not DATA lineage (values)
→ A filter/WHERE clause drops rows — lineage won't show that
→ Fix: include row-count metrics in events (rows_read vs rows_written)
→ If rows_written < rows_read, a filter exists — flag it

PITFALL 3: "Glue job uses UDFs or complex PySpark — can't auto-extract lineage"
→ Cause: custom Python functions are black boxes
→ Fix: require developers to annotate UDFs with input→output column mappings
→ Convention: @lineage_udf(inputs=["qty","price"], outputs=["total"])

PITFALL 4: "DynamoDB adjacency list gets hot partition"
→ Cause: one gold table has 500+ upstream edges — all on same PK
→ Fix: use GSI with inverted key, or shard the PK
→ At QuickCart scale (2000 nodes): unlikely to hit this

PITFALL 5: "Lineage emission fails, takes down production ETL"
→ Cause: DynamoDB throttling or network issue during lineage write
→ Fix: ALWAYS use async/fire-and-forget with SQS buffer
→ Lineage is OBSERVABILITY — never let it block data flow

PITFALL 6: "Historical lineage shows wrong graph for past date"
→ Cause: lineage was updated in-place instead of versioned
→ Fix: every lineage event has timestamp + job_run_id
→ Query "lineage as of date X" = filter edges by created_at <= X

PITFALL 7: "Cross-account lineage breaks when account IDs change"
→ Cause: account migration or reorganization
→ Fix: use logical names (alias) alongside account IDs
→ Map: "ingestion-account" → "111111111111" in config

   → Fix: include cancelled orders in revenue but flag them separately
