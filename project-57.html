Glue🟠 Phase 4: Hard

🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — multi-account AWS setup (from Project 56)
Team: 15 engineers, 5 data engineers, 4 analysts, 1 security lead

CURRENT STATE (after Project 56):
→ Account A (111111111111): Production Data
  - S3 bucket with Bronze/Silver/Gold data ✅
  - Glue Data Catalog with 25+ tables (from Project 23 crawlers) ✅
  - Redshift cluster with production warehouse ✅

→ Account B (222222222222): Analytics
  - Cross-account S3 READ access to Account A ✅ (Project 56)
  - Glue ETL jobs for transformation
  - Athena for ad-hoc queries
  - DataBrew for analyst cleaning

THE NEW PROBLEM:
Account B's data engineer writes a Glue ETL job:

    # In Account B's Glue job
    datasource = glueContext.create_dynamic_frame.from_catalog(
        database="quickcart_db",
        table_name="orders_gold"
    )
    # ERROR: Database 'quickcart_db' not found!

WHY IT FAILS:
→ "quickcart_db" and "orders_gold" exist in Account A's Glue Catalog
→ Account B's Glue Catalog is SEPARATE — completely empty
→ Glue Catalog is per-account — Account B can't see Account A's tables
→ Even though S3 data is accessible (Project 56), METADATA is not

WHAT THIS BLOCKS:

THREE POSSIBLE APPROACHES:
1. Copy Catalog metadata from A to B (duplication — bad)
2. Re-run crawlers in Account B against Account A's S3 (redundant — bad)
3. Share Account A's Catalog with Account B (correct — this project)

SECURITY LEAD: "Account B should see Account A's Catalog tables
  but NOT be able to create, modify, or delete them.
  Read-only metadata access. Account A stays in control."

🟡 HOW TO APPROACH IT

Step 1: Understand Glue Data Catalog architecture
→ Each AWS account has ONE Glue Data Catalog (per region)
→ Catalog contains: Databases → Tables → Partitions → Columns
→ Catalog ID = Account ID (e.g., "111111111111")
→ By default: services only see their OWN account's Catalog
→ Cross-account: must explicitly grant access via Resource Policy

Step 2: Understand the access mechanism
→ Glue Data Catalog supports RESOURCE-BASED policies (like S3 bucket policies)
→ Account A puts a Resource Policy on its Catalog
→ Policy says: "Allow Account B's roles to GetDatabase, GetTable, GetPartitions"
→ Account B's services reference Account A's Catalog using CatalogId parameter

Step 3: Understand how consumers reference cross-account Catalog
→ Glue ETL: catalog_id="111111111111" in from_catalog()
→ Athena: register Account A's Catalog as a data source
→ Redshift Spectrum: CREATE EXTERNAL SCHEMA with catalog_id
→ Each service has its own way of specifying the foreign Catalog

Step 4: Understand the permission chain (same as Project 56)
→ Account A's Catalog Resource Policy must ALLOW Account B
→ Account B's identity policy must ALLOW Glue Catalog actions
→ BOTH must allow — cross-account AND logic

Step 5: Design the access scope
→ Account B can: GetDatabase, GetTable, GetPartitions, GetTableVersions
→ Account B CANNOT: CreateDatabase, CreateTable, UpdateTable, DeleteTable
→ Account B sees: quickcart_db.* (all tables in the database)
→ Account B CANNOT see: internal_db.* (sensitive internal metadata)

🔵 WHAT COULD BE DONE

Option A: Copy Catalog metadata from Account A to Account B
→ Export tables as JSON → import in Account B
→ PROBLEMS: metadata gets stale, new partitions not synced
→ Must re-copy every time schema changes
→ Two sources of truth → inevitable drift

Option B: Run Glue Crawlers in Account B against Account A's S3
→ Account B's crawler reads Account A's S3 → creates tables in Account B's Catalog
→ PROBLEMS: duplicate crawling cost, partition sync delay
→ Account B's schema may differ from Account A's (different classifiers)
→ Two catalogs for same data → confusion

Option C: Glue Catalog Resource Policy (this project) ✅
→ Account A shares its Catalog with Account B via resource policy
→ Single source of truth — Account A's Catalog
→ Real-time: Account B always sees latest schema/partitions
→ No duplication, no sync, no drift
→ Read-only: Account B can't modify Account A's metadata

Option D: AWS Lake Formation cross-account grants (Project 74)
→ More granular: column-level, row-level, tag-based permissions
→ Requires Lake Formation setup in both accounts
→ Better for enterprises with 10+ accounts and complex permissions
→ Overkill for 2-3 account setups

Option E: AWS RAM (Resource Access Manager)
→ Share Glue Catalog resources via RAM
→ Works within AWS Organizations
→ Simpler than manual resource policies for Organization-wide sharing
→ But: less granular than direct resource policies

DECISION: Option C — Glue Catalog Resource Policy
→ Reason: simplest, no duplication, real-time, fits our 2-account model
→ Foundation for Lake Formation (Project 74) later

✅ WHAT SHOULD DO

1. Grant ONLY read actions on Catalog
   → Allow: glue:GetDatabase, glue:GetDatabases
   → Allow: glue:GetTable, glue:GetTables, glue:GetTableVersions
   → Allow: glue:GetPartition, glue:GetPartitions
   → Deny: glue:CreateTable, glue:UpdateTable, glue:DeleteTable
   → Deny: glue:CreateDatabase, glue:DeleteDatabase

2. Scope access to SPECIFIC databases (not entire Catalog)
   → Allow Account B to see: quickcart_db
   → Deny Account B from seeing: internal_db, hr_db
   → Use Condition or explicit Resource ARNs

3. Use specific principal ARNs (not entire Account B)
   → Bad: "Principal": {"AWS": "222222222222"} (any role in Account B)
   → Good: "Principal": {"AWS": "arn:aws:iam::222222222222:role/GlueRole-B"}
   → Least privilege — only authorized roles can see metadata

4. Test with Athena first (fastest verification)
   → Athena can reference cross-account Catalog directly
   → No code needed — just SQL with catalog_id prefix
   → If Athena works → Glue ETL will also work

5. Ensure Account B's IAM policies ALSO allow Glue Catalog actions
   → Cross-account = both sides must allow (from Project 56)
   → Account A's resource policy + Account B's identity policy
   → Missing identity policy = AccessDenied even with resource policy

6. Document which databases are shared and with whom
   → Maintain inventory: database → shared with → purpose → granted by
   → Review quarterly — revoke unused access
   → Tag shared databases: "SharedWithAccountB=true"

❌ WHAT SHOULD NOT DO

1. DO NOT grant glue:* in the resource policy
   → Includes: CreateTable, DeleteTable, UpdatePartition
   → Account B could modify Account A's production metadata
   → Corrupted metadata → broken queries for everyone

2. DO NOT share the entire Catalog without database-level filtering
   → Account A may have: hr_db with salary tables, credentials_db
   → Sharing entire Catalog exposes sensitive metadata
   → Even table NAMES can be sensitive ("layoff_candidates")

3. DO NOT create circular dependencies
   → Account A shares Catalog with Account B
   → Account B creates tables pointing to Account A's S3
   → Account A starts using Account B's tables
   → Now: breaking either account breaks both → fragile

4. DO NOT forget about Catalog encryption
   → If Account A's Catalog is encrypted with KMS
   → Account B needs kms:Decrypt permission on Account A's KMS key
   → Missing KMS access = AccessDenied on encrypted metadata
   → This is a frequently missed step

5. DO NOT mix Catalog IDs in Glue job code without abstraction
   → Hardcoding: catalog_id="111111111111" everywhere
   → If account ID changes (account migration) → update 50 scripts
   → Use: environment variable, SSM Parameter Store, or job argument

6. DO NOT assume partitions auto-sync
   → Account A adds new partition → Account B sees it immediately? YES
   → Catalog Resource Policy shares LIVE metadata (not a copy)
   → But: Account B cannot ADD partitions to Account A's tables
   → If Account B needs to add partitions: different pattern needed

🚫 WHEN NOT TO USE CROSS-ACCOUNT CATALOG SHARING

1. When Account B needs to MODIFY Account A's tables
   → Resource policy for read-only is safe
   → Write access cross-account is risky (metadata corruption)
   → Use: AssumeRole pattern with write permissions (controlled)
   → Or: Account B requests Account A's pipeline to update

2. When you need tag-based access control (TBAC)
   → "Share all tables tagged 'public' but not 'confidential'"
   → Glue Resource Policy doesn't support tag conditions
   → Use: Lake Formation tag-based access (Project 74)

3. When sharing between accounts in DIFFERENT AWS Organizations
   → Glue Resource Policy works cross-Organization
   → But: no centralized governance (no SCPs across orgs)
   → Use: explicit trust with External ID (like Project 56 AssumeRole)

4. When Catalog is encrypted with customer-managed KMS key
   → AND you can't grant cross-account KMS access
   → Decryption fails → metadata inaccessible
   → Use: re-crawl in Account B (Option B from approach section)
   → Or: change to AWS-managed KMS key (allows cross-account by default)

5. When data governance requires physical metadata separation
   → Regulation: "analytics team must not see production metadata structure"
   → Cross-account Catalog sharing exposes table names, column names, types
   → Even without data access: metadata IS information
   → Use: curated views/tables in Account B's own Catalog

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "AccessDeniedException: cross-account access not allowed"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Catalog Resource Policy missing or doesn't match caller
→ Debug: aws glue get-resource-policy (in Account A)
→ Verify: Principal ARN exactly matches Account B's role
→ Verify: Actions include the specific API being called
→ Common: forgot glue:GetPartitions (tables visible but partitions fail)

PITFALL 2: "Database visible but tables return empty"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Resource policy allows GetDatabase but Resource ARN for tables wrong
→ Policy grants: "arn:aws:glue:...:database/quickcart_db" ← only database
→ Also needs: "arn:aws:glue:...:table/quickcart_db/*" ← tables within it
→ And: "arn:aws:glue:...:catalog" ← catalog-level for GetDatabases

PITFALL 3: "Works from Athena but fails from Glue ETL"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Athena role has permissions, Glue role doesn't
→ Different services use different IAM roles
→ Resource policy may allow Athena role but not Glue role
→ Fix: add ALL Account B service roles to resource policy

PITFALL 4: "Catalog encrypted with KMS — cross-account fails"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Account A's Catalog encrypted with customer-managed CMK
→ Account B's role needs kms:Decrypt on that CMK
→ CMK key policy must allow Account B's role
→ Three policies to align: Catalog resource policy + IAM + KMS key policy
→ Fix: add Account B's role to CMK key policy with kms:Decrypt

PITFALL 5: "Glue ETL from_catalog() ignores catalog_id parameter"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: older Glue versions don't support catalog_id in from_catalog()
→ Fix: use additional_options={"catalogId": "111111111111"}
→ Or: use boto3 glue client directly to get table, then read S3 manually
→ Or: use Glue job arguments: --catalog-id 111111111111

PITFALL 6: "Resource policy exceeds 10 KB limit"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Sharing with 20+ accounts × 10+ databases = large policy
→ Glue Resource Policy max: 10 KB
→ Fix: use aws:PrincipalOrgID condition (one condition for entire Organization)
→ Fix: use wildcard principals with Condition keys
→ Fix: migrate to Lake Formation for scalable access management

📈 SCALABILITY CONSIDERATIONS

SHARING SCALE:
→ 2 accounts: simple resource policy (this project)
→ 5 accounts: resource policy with multiple principals
→ 20+ accounts: use aws:PrincipalOrgID condition
→ 100+ accounts: Lake Formation (Project 74) or AWS RAM

CATALOG SIZE:
→ 100 tables: no performance impact on cross-account reads
→ 1,000 tables: GetTables pagination needed (100 per page)
→ 10,000+ tables: consider database-level filtering in policy
→ No hard limit on cross-account Catalog reads

POLICY LIMITS:
→ Resource Policy max: 10 KB
→ ~10-15 detailed statements per policy
→ Workaround: use wildcards and Conditions to compress

LATENCY:
→ Cross-account Catalog API: ~50-100ms additional latency vs same-account
→ Negligible for batch ETL (adds 0.1s to a 20-minute job)
→ Noticeable for interactive Athena (adds 0.1s per query planning)
→ Not a concern for data pipeline use cases

COST:
→ Glue Catalog API calls: first 1M/month free, then $1/100K
→ Cross-account calls count the SAME as same-account calls
→ No additional charge for cross-account Catalog access
→ Typical pipeline: ~10K API calls/day = well within free tier
