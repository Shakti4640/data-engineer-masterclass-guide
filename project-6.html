PYTHON SCRIPT ON EC2 MUST READ/WRITE S3 WITHOUT HARDCODED KEYS
S3EC2🟢 Phase 1: Very Easy

🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — same startup from Project 1
Current State:
→ The upload script from Project 1 runs on EC2
→ Developer initially made it work by running:
     aws configure
     → pasted Access Key ID:[REDACTED:AWS_ACCESS_KEY]E
     → pasted Secret Access Key:[REDACTED:AWS_SECRET_KEY]Y
→ These credentials are now stored in ~/.aws/credentials on the EC2 instance
→ The same credentials are ALSO hardcoded in a second script

6 months later:
→ 3 EC2 instances now run various S3 upload/download scripts
→ All 3 use the SAME IAM user's access keys (copy-pasted everywhere)
→ Security audit reveals:
   - Keys are 6 months old (never rotated)
   - Same keys on 3 machines (blast radius = everything)
   - Keys belong to an IAM user with AdministratorAccess (!!!)
   - Keys are committed in a private GitHub repo
   - If ONE machine is compromised, attacker has full AWS access

CISO: "Remove ALL hardcoded keys from EC2 instances by Friday.
       Use IAM Roles instead. Explain to me WHY this is safer."

🟡 HOW TO APPROACH IT

Step 1: Understand WHY hardcoded keys are dangerous
→ Keys are STATIC — they don't expire unless you manually rotate
→ Keys are PORTABLE — anyone who copies them can use them from anywhere
→ Keys are BROAD — typically over-permissioned (lazy admin gave Admin access)
→ Keys are INVISIBLE — no way to know if someone extracted them silently
→ Keys are SHARED — same key on multiple machines = no audit trail per machine

Step 2: Understand the ALTERNATIVE — IAM Roles for EC2
→ IAM Role = identity WITHOUT permanent credentials
→ EC2 Instance Profile = container that delivers role credentials TO the instance
→ Credentials are TEMPORARY — auto-rotated every ~6 hours by AWS
→ Credentials are LOCAL — only accessible from that specific EC2 instance
→ Credentials are SCOPED — role has ONLY the permissions you define
→ Credentials are TRACEABLE — CloudTrail logs show which instance made each call

Step 3: Understand what CHANGES in our code
→ NOTHING changes in boto3 code
→ boto3 automatically discovers credentials via the "credential chain"
→ If Instance Profile exists → boto3 uses it automatically
→ No code change needed — only infrastructure change

Step 4: Plan the migration
→ Create IAM Role with MINIMAL S3 permissions
→ Create Instance Profile and attach the role
→ Attach Instance Profile to EC2 instance
→ Delete ~/.aws/credentials from all instances
→ Remove hardcoded keys from all scripts
→ Verify everything still works
→ Deactivate and delete the old IAM user access keys

🔵 WHAT COULD BE DONE

Option A: Rotate access keys every 90 days manually
→ Still keys → still portable → still dangerous
→ Rotation is manual → people forget
→ 3 machines to update → one always missed
→ VERDICT: band-aid, not a solution

Option B: Use AWS Systems Manager Parameter Store for keys
→ Keys stored centrally, fetched at runtime
→ Better than hardcoded, BUT still static credentials
→ Still need a way to authenticate TO Parameter Store (chicken-and-egg)
→ VERDICT: useful for OTHER secrets, not for AWS API auth

Option C: Use IAM Roles attached to EC2 via Instance Profile ✅
→ No credentials in code, filesystem, or environment variables
→ Temporary, auto-rotated, scoped, traceable
→ boto3 discovers them automatically via metadata service
→ VERDICT: the CORRECT and ONLY recommended approach

Option D: Use AWS SSO / Identity Center
→ Designed for HUMAN users accessing console/CLI from laptops
→ Not for machine-to-machine (EC2 to S3)
→ VERDICT: wrong use case

DECISION: Option C — IAM Role + Instance Profile

✅ WHAT SHOULD DO

1. Create role with LEAST PRIVILEGE policy
   → Only s3:GetObject, s3:PutObject, s3:ListBucket
   → Only on the specific bucket ARN
   → NOT s3:* and definitely NOT *

2. Use trust policy that ONLY allows EC2 service to assume the role
   → Principal: {"Service": "ec2.amazonaws.com"}
   → No human user should be able to assume this role

3. Name the role descriptively
   → QuickCart-EC2-S3ReadWrite-Prod
   → NOT "MyRole" or "test-role-1"

4. One role per responsibility
   → Upload script → role with PutObject
   → Download script → role with GetObject
   → In practice: one role per application is acceptable

5. Verify credential chain BEFORE deleting old keys
   → Attach role → test script works → THEN delete keys
   → Never delete first and hope for the best

6. Use IMDSv2 (Instance Metadata Service v2)
   → v1 is vulnerable to SSRF attacks
   → v2 requires session token → much harder to exploit

7. Log and monitor with CloudTrail
   → Every S3 call from the role is logged
   → AssumedRole ARN includes the instance ID

❌ WHAT SHOULD NOT DO

1. DO NOT give the role AdministratorAccess or PowerUserAccess
   → "It works with Admin" is the most dangerous sentence in AWS
   → A compromised EC2 with Admin = attacker owns your entire account

2. DO NOT create one "shared" role used by all applications
   → Blast radius: if one app is compromised, all permissions are exposed
   → Each application/purpose gets its own role

3. DO NOT hardcode the temporary credentials fetched from metadata
   → Some developers fetch from 169.254.169.254 and cache in variables
   → Let boto3 handle this — it manages refresh automatically

4. DO NOT leave IMDSv1 enabled
   → SSRF vulnerability: attacker tricks app into calling metadata URL
   → 2019 Capital One breach: SSRF → IMDSv1 → stole credentials → 100M records
   → ALWAYS enforce IMDSv2 (HttpTokens = required)

5. DO NOT skip testing after attaching the role
   → Policy might have typo in bucket ARN
   → Policy might be missing s3:ListBucket (causes confusing 403 on list)

6. DO NOT keep old access keys "just in case"
   → Deactivate immediately after role works
   → Delete after 7-day observation period

7. DO NOT attach role to instance without understanding trust policy
   → Trust policy = WHO can assume this role
   → Permission policy = WHAT the role can do
   → Both must be correct — either one wrong = broken or insecure

RULE OF THUMB:
→ If compute runs IN AWS → USE IAM ROLES (never keys)
→ If compute runs OUTSIDE AWS → minimize key scope + rotate

🚫 WHEN NOT TO USE IAM ROLES FOR EC2

1. When application runs OUTSIDE of AWS (on-prem server)
   → EC2 Instance Metadata doesn't exist on-prem
   → Use: IAM User keys (last resort) or SSM Hybrid Activation

2. When you need CROSS-ACCOUNT access
   → Instance Profile alone isn't enough
   → Use: Instance Profile + AssumeRole into target account (Project 56)

3. When you need database credentials (MariaDB password, API keys)
   → IAM Roles provide AWS API credentials, NOT application passwords
   → Use: Secrets Manager or Parameter Store for non-AWS secrets (Project 70)

4. When you need to limit access PER-USER on a shared EC2
   → Instance Profile applies to the INSTANCE, not the SSH user
   → All users on that EC2 get the same role
   → Use: separate instances per team, or containerize with ECS/EKS task roles

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

┌──────────────────────────────────────────────────────────────────────────────┐
│                  IAM ROLE + INSTANCE PROFILE — DEEP MECHANICS               │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  COMPONENT 1: IAM ROLE                                                       │
│  ━━━━━━━━━━━━━━━━━━━━━                                                       │
│  An IAM Role is an identity with TWO types of policies:                      │
│                                                                              │
│  A) TRUST POLICY (who can BECOME this role):                                 │
│     {                                                                        │
│       "Effect": "Allow",                                                     │
│       "Principal": {"Service": "ec2.amazonaws.com"},                         │
│       "Action": "sts:AssumeRole"                                             │
│     }                                                                        │
│     → This says: ONLY the EC2 service can assume this role                   │
│     → A human, Lambda, or Glue CANNOT assume it                             │
│     → The "Principal" field is the GATEKEEPER                                │
│                                                                              │
│  B) PERMISSION POLICY (what the role CAN DO):                                │
│     {                                                                        │
│       "Effect": "Allow",                                                     │
│       "Action": ["s3:GetObject", "s3:PutObject"],                            │
│       "Resource": "arn:aws:s3:::quickcart-raw-data-prod/*"                   │
│     }                                                                        │
│     → This says: whoever assumes this role can read/write objects            │
│     → ONLY in the specified bucket                                           │
│                                                                              │
│  COMPONENT 2: INSTANCE PROFILE                                               │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━                                                │
│  → An Instance Profile is a CONTAINER (wrapper) for an IAM Role              │
│  → EC2 doesn't directly attach roles — it attaches Instance Profiles         │
│  → One Instance Profile holds exactly ONE role                               │
│  → When you "attach role to EC2" in console, AWS auto-creates the profile    │
│  → Via API/CLI, you must create both separately                              │
│                                                                              │
│  RELATIONSHIP:                                                               │
│  ┌──────────────────┐     contains     ┌──────────────────┐                  │
│  │ Instance Profile  │────────────────→│ IAM Role          │                  │
│  │ (wrapper)         │                 │ (actual identity)  │                 │
│  └────────┬─────────┘                 └──────────────────┘                  │
│           │                                                                  │
│           │ attached to                                                      │
│           ▼                                                                  │
│  ┌──────────────────┐                                                        │
│  │ EC2 Instance      │                                                       │
│  └──────────────────┘                                                        │
│                                                                              │
│  COMPONENT 3: EC2 INSTANCE METADATA SERVICE (IMDS)                           │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                            │
│  → Special HTTP endpoint: http://169.254.169.254                             │
│  → ONLY accessible from WITHIN the EC2 instance                              │
│  → Link-local address — never routed to internet                             │
│  → Returns instance metadata: ID, region, role credentials                   │
│                                                                              │
│  CREDENTIAL FLOW (step by step):                                             │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                                             │
│                                                                              │
│  Step 1: EC2 boots with Instance Profile attached                            │
│          └→ AWS associates role with the instance                            │
│                                                                              │
│  Step 2: boto3 script starts on EC2                                          │
│          └→ boto3 searches for credentials (credential chain — see below)    │
│                                                                              │
│  Step 3: boto3 reaches "Instance Metadata" in the chain                      │
│          └→ Makes HTTP call to:                                              │
│             http://169.254.169.254/latest/meta-data/                         │
│             iam/security-credentials/QuickCart-EC2-S3ReadWrite-Prod           │
│                                                                              │
│  Step 4: IMDS calls STS (Security Token Service) internally                  │
│          └→ STS generates TEMPORARY credentials:                             │
│             - AccessKeyId:     ASIA...  (starts with ASIA, not AKIA)         │
│             - SecretAccessKey:  wJal...                                      │
│             - SessionToken:     FwoGZX... (THIS is what makes it temporary)  │
│             - Expiration:       2025-01-15T18:30:00Z (~6 hours from now)     │
│                                                                              │
│  Step 5: boto3 receives these temp credentials                               │
│          └→ Caches them in memory                                            │
│          └→ Uses them for ALL subsequent AWS API calls                       │
│          └→ Includes SessionToken in every request signature                 │
│                                                                              │
│  Step 6: Before expiry, boto3 AUTO-REFRESHES                                 │
│          └→ Calls IMDS again → gets new temp credentials                     │
│          └→ No code change, no restart needed                                │
│          └→ Seamless rotation every ~6 hours                                 │
│                                                                              │
│                                                                              │
│  BOTO3 CREDENTIAL CHAIN (search order):                                      │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                                      │
│  boto3 checks these locations IN ORDER — uses FIRST one found:               │
│                                                                              │
│  Priority 1: Hardcoded in boto3.client(aws_access_key_id=...)                │
│  Priority 2: Environment variables (AWS_ACCESS_KEY_ID)                       │
│  Priority 3: Shared credentials file (~/.aws/credentials)                    │
│  Priority 4: AWS config file (~/.aws/config)                                 │
│  Priority 5: Assume Role provider (role_arn in config)                       │
│  Priority 6: boto3 can assume role via web identity                          │
│  Priority 7: ★ EC2 Instance Metadata Service (IMDS) ★                       │
│  Priority 8: Container credentials (ECS task role)                           │
│                                                                              │
│  KEY INSIGHT:                                                                │
│  → If ~/.aws/credentials EXISTS, it takes PRIORITY over Instance Profile     │
│  → This is why we MUST delete old credential files                           │
│  → Otherwise, boto3 uses the old static keys, NOT the role                   │
│                                                                              │
│                                                                              │
│  IMDSv1 vs IMDSv2:                                                           │
│  ━━━━━━━━━━━━━━━━                                                            │
│  v1 (DANGEROUS — default on older instances):                                │
│  → Simple GET request: curl http://169.254.169.254/...                       │
│  → Any process on the instance can call it                                   │
│  → SSRF attack: trick a web app into calling metadata URL                    │
│  → Capital One breach (2019): SSRF → IMDSv1 → stole creds → 100M records    │
│                                                                              │
│  v2 (SECURE — always enforce this):                                          │
│  → Requires two-step process:                                                │
│     Step A: PUT request to get a SESSION TOKEN (with TTL)                    │
│       TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token"          │
│              -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")               │
│     Step B: GET request WITH token in header                                 │
│       curl -H "X-aws-ec2-metadata-token: $TOKEN"                            │
│              http://169.254.169.254/latest/meta-data/                         │
│  → SSRF can't do PUT + capture token → attack blocked                        │
│  → boto3 handles v2 automatically (no code change)                           │
│                                                                              │
│                                                                              │
│  STS (Security Token Service) — THE ENGINE BEHIND ROLES:                     │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                      │
│  → STS is the service that GENERATES temporary credentials                   │
│  → When EC2 metadata is queried, STS is called behind the scenes             │
│  → STS operations:                                                           │
│     - AssumeRole → for IAM roles (this project)                              │
│     - AssumeRoleWithWebIdentity → for OIDC federation                        │
│     - AssumeRoleWithSAML → for SAML federation                               │
│     - GetSessionToken → for MFA-protected API access                         │
│     - GetCallerIdentity → "who am I?" (debugging tool)                       │
│  → Every temp credential has:                                                │
│     - AccessKeyId (starts with ASIA for temp, AKIA for permanent)            │
│     - SecretAccessKey                                                        │
│     - SessionToken (MUST be included in API calls)                           │
│     - Expiration (UTC timestamp)                                             │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "I attached the role but boto3 still uses old keys"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: ~/.aws/credentials still exists → Priority 3 beats Priority 7
→ Fix: delete or rename ~/.aws/credentials
→ Verify: python -c "import boto3; print(boto3.Session().get_credentials().method)"
   → Should print: "iam-role" (NOT "shared-credentials-file")

PITFALL 2: "AccessDenied even though role has S3 permissions"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause 1: Permission policy has wrong bucket ARN
   → "arn:aws:s3:::quickcart-raw-data-prod" (bucket-level) allows ListBucket
   → "arn:aws:s3:::quickcart-raw-data-prod/*" (object-level) allows Get/Put
   → NEED BOTH for full access
→ Cause 2: S3 bucket policy has explicit Deny that overrides role
   → Explicit Deny ALWAYS wins (Project 3 bucket policy concept)
→ Cause 3: S3 Block Public Access blocks cross-account (not this case)
→ Debug: aws sts get-caller-identity → verify you ARE the role

PITFALL 3: "Works from SSH but fails from cron job"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: cron runs as root, SSH session runs as ec2-user
→ ~/.aws/credentials might exist for ec2-user but not root
→ With IAM Role: works for ALL users → no more user-specific issues
→ Fix: ensure no credential files exist for ANY user

PITFALL 4: "Temporary credentials expired mid-upload"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: uploading a huge file takes > 6 hours
→ boto3 auto-refreshes for NEW API calls, but mid-transfer refresh is tricky
→ Fix: use multipart upload → each part is a separate API call → auto-refreshes

PITFALL 5: "Instance Profile exists but EC2 says 'no role attached'"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Instance Profile created but not ASSOCIATED with EC2 instance
→ Creating a profile ≠ attaching it
→ Must call: associate_iam_instance_profile() or set at launch time

PITFALL 6: "I can assume the role from my laptop too"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Trust policy says Principal: "*" or includes your IAM user
→ Fix: Trust policy MUST only have: {"Service": "ec2.amazonaws.com"}
→ No human principal should be there (unless for debugging, then remove)

PITFALL 7: "IMDSv1 still enabled — security team flagged it"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Default on older instances: IMDSv1 + v2 both enabled
→ Must explicitly set: HttpTokens=required (forces v2-only)
→ Do this at launch time or modify running instance

📈 SCALABILITY CONSIDERATIONS

SINGLE EC2 INSTANCE:
→ One role, one Instance Profile
→ Credentials fetched from IMDS → negligible overhead
→ boto3 caches credentials → no repeated IMDS calls

MULTIPLE EC2 INSTANCES (same function):
→ All can share the SAME Instance Profile
→ Each gets its OWN temporary credentials (different AccessKeyId)
→ CloudTrail distinguishes them by instance ID in the ARN:
   arn:aws:sts::123456789012:assumed-role/QuickCart-EC2-S3Role/i-0abc123def456

AUTO SCALING GROUP:
→ Launch Template specifies Instance Profile
→ Every new instance automatically gets the role
→ No credential distribution needed (unlike access keys)
→ This is WHY roles are essential for auto-scaling (Project 52)

MULTIPLE ROLES ON ONE INSTANCE:
→ EC2 supports ONLY ONE Instance Profile at a time
→ If you need multiple permission sets:
   Option A: combine permissions into one role (simplest)
   Option B: application assumes a DIFFERENT role via STS (advanced)
   → script calls sts.assume_role(RoleArn=...) to switch roles
   → useful for cross-account access (Project 56)

RATE LIMITS:
→ IMDS: no documented rate limit, but throttles under extreme load
→ STS AssumeRole: 1 request per second per role per account (soft limit)
→ boto3 credential refresh: happens ~15 min before expiry
→ For 1000+ instances: consider STS regional endpoints to distribute load

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ NEVER hardcode AWS credentials — not in code, not in files, not in env vars
→ IAM Roles provide TEMPORARY, AUTO-ROTATED, SCOPED credentials
→ Instance Profile = wrapper that delivers a role to EC2
→ boto3 credential chain has 8 priorities — Role is #7 (ensure nothing above it)
→ ALWAYS delete ~/.aws/credentials when using Instance Profiles
→ ALWAYS enforce IMDSv2 — IMDSv1 is the #1 EC2 credential theft vector
→ Trust Policy (who can assume) and Permission Policy (what they can do) are SEPARATE
→ Deactivate before delete — 7-day observation prevents surprises
→ AKIA = permanent key (bad), ASIA = temporary key (good)
→ STS is the engine behind ALL temporary credentials in AWS
