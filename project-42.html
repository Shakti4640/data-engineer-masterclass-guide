🟡 HOW TO APPROACH IT

Step 1: Understand CDC (Change Data Capture) fundamentals
→ CDC = detect and extract ONLY the rows that changed since last run
→ Three approaches exist:
   a) Timestamp-based (watermark column)    ← THIS PROJECT
   b) Log-based (binlog / WAL parsing)      ← Project 79 (DMS)
   c) Trigger-based (database triggers)      ← Not recommended at scale

Step 2: Evaluate which CDC approach fits QuickCart

*Timestamp-based can capture soft deletes (status = 'deleted')
  but NOT hard deletes (DELETE FROM). For QuickCart, orders are never

DECISION: Timestamp-based watermark CDC
→ Low complexity
→ No additional infrastructure (no DMS, no Debezium)
→ Uses existing MariaDB → Glue JDBC path (Project 31)
→ Captures INSERTs and UPDATEs (all we need)

Step 3: Design the watermark mechanism
→ MariaDB orders table needs: updated_at TIMESTAMP column
→ Every INSERT: updated_at = NOW()
→ Every UPDATE: updated_at = NOW() (via ON UPDATE CURRENT_TIMESTAMP)
→ CDC query: SELECT * FROM orders WHERE updated_at > '{last_run_timestamp}'
→ Store last_run_timestamp in: DynamoDB (persistent, queryable)

Step 4: Design the merge strategy for Redshift
→ Changed rows arrive at Redshift
→ Some are NEW (INSERT) → must be added
→ Some are UPDATED → must replace existing rows
→ Redshift has no native MERGE (until recently — conditional)
→ Classic pattern: staging table → DELETE matching → INSERT all from staging
→ Modern pattern: Redshift MERGE statement (available since late 2023)

Step 5: Fit into the existing event-driven pipeline (Project 41)
→ MariaDB CDC export → S3 (only changed rows)
→ S3 file arrival → EventBridge → Step Functions (Project 41)
→ Step Functions → Glue ETL → Redshift MERGE
→ Same architecture, just smarter extraction + loading

🔵 WHAT COULD BE DONE

Option A: Glue Job Bookmarks alone (Project 33)
→ Bookmarks track WHICH S3 FILES were processed
→ They do NOT track WHICH ROWS in MariaDB changed
→ If you do a full dump to S3, bookmark sees "new file" → processes ALL rows
→ Bookmarks solve: "don't reprocess old S3 files"
→ Bookmarks do NOT solve: "extract only changed rows from MariaDB"
→ VERDICT: Insufficient alone — need watermark extraction FIRST

Option B: Full dump + Glue DynamicFrame filter
→ Export all 5.2M rows to S3
→ In Glue, filter where updated_at > watermark
→ Problem: still exporting 480 MB from MariaDB every night
→ Network transfer waste, MariaDB load, S3 storage waste
→ VERDICT: Solves Glue processing time but not extraction waste

Option C: Watermark CDC at extraction point (MariaDB → S3)
→ Query MariaDB: SELECT * WHERE updated_at > watermark
→ Export only 11K rows → 1.5 MB CSV
→ Glue processes 1.5 MB (fast, cheap)
→ Redshift merges 11K rows (fast, no table lock)
→ VERDICT: ✅ Optimal — solves extraction + processing + loading

Option D: AWS DMS (Database Migration Service) with CDC
→ Reads MariaDB binlog in real-time
→ Streams changes to S3/Kinesis/Redshift
→ Captures INSERTs, UPDATEs, and DELETEs
→ Problem: requires DMS replication instance ($$$)
→ Problem: complex setup, ongoing maintenance
→ VERDICT: Overkill for nightly batch — perfect for real-time (Project 79)

DECISION: Option C — Watermark CDC at extraction

✅ WHAT SHOULD DO

1. Add updated_at column to MariaDB tables
   → DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
   → Every INSERT gets current timestamp
   → Every UPDATE automatically refreshes timestamp
   → Zero application code changes needed

2. Add created_at column separately
   → DEFAULT CURRENT_TIMESTAMP (no ON UPDATE)
   → Distinguishes new rows from updated rows
   → Useful for audit and debugging

3. Index the updated_at column
   → CREATE INDEX idx_orders_updated_at ON orders(updated_at)
   → Without index: CDC query does full table scan (defeats the purpose)
   → With index: query returns 11K rows in <100ms

4. Store watermark in DynamoDB (not in S3 or local file)
   → DynamoDB: strongly consistent reads, atomic updates
   → Survives EC2 termination, Glue job restarts
   → Can be queried/audited easily
   → Stores: entity_name, last_watermark, last_run_at, rows_extracted

5. Use overlap window for watermark
   → Don't query WHERE updated_at > exactly_last_watermark
   → Query WHERE updated_at > (last_watermark - 5 minutes)
   → WHY: clock skew, transaction commit delays, replication lag
   → This means some rows are extracted TWICE (overlap)
   → Redshift MERGE handles duplicates idempotently

6. Use Redshift MERGE (not DELETE+INSERT) when possible
   → MERGE is atomic — no window where data is missing
   → DELETE+INSERT has a moment where rows are deleted but not yet re-inserted
   → Concurrent queries during DELETE+INSERT see incomplete data

7. Log extraction metrics
   → rows_extracted, watermark_from, watermark_to, extraction_duration
   → Enables trend analysis: is daily change volume growing?
   → Alerts if rows_extracted = 0 (possible watermark bug)
   → Alerts if rows_extracted > 50% of total (possible full reload)

❌ WHAT SHOULD NOT DO

1. DO NOT use auto_increment ID as watermark for UPDATE detection
   → Auto-increment only increases on INSERT
   → Updated rows keep their original ID
   → You'll miss ALL updates (only catch inserts)
   → Use updated_at timestamp instead

2. DO NOT set watermark to "NOW()" at extraction START
   → If extraction takes 5 minutes (00:00 to 00:05)
   → And a row is updated at 00:03
   → Watermark set to 00:00 at start → 00:03 row is INCLUDED ✅
   → But if watermark set to 00:05 at end → next run starts from 00:05
   → Row updated at 00:04 could be missed if committed after extraction read
   → SAFE: set watermark to MAX(updated_at) from EXTRACTED rows

3. DO NOT rely on MariaDB NOW() matching extraction server clock
   → MariaDB server time ≠ Glue job server time ≠ EC2 time
   → Always use MariaDB's own timestamps from the extracted data
   → Watermark = MAX(updated_at) FROM the data you just extracted

4. DO NOT skip the staging table in Redshift
   → Never MERGE directly from S3 into production table
   → COPY to staging → MERGE from staging → DROP staging
   → Staging allows: validation, row counts, rollback

5. DO NOT forget to handle the FIRST RUN (no watermark exists)
   → First run: watermark = '1970-01-01' (extract everything)
   → Store initial watermark in DynamoDB
   → Subsequent runs: use stored watermark

6. DO NOT hard-delete rows in MariaDB if using timestamp CDC
   → Hard DELETE removes the row — no updated_at to detect
   → Use soft delete: UPDATE status = 'deleted', updated_at = NOW()
   → Timestamp CDC will pick it up

7. DO NOT extract without ORDER BY updated_at
   → Without ORDER BY: MariaDB returns rows in arbitrary order
   → If extraction fails midway, you can't resume reliably
   → With ORDER BY: you know exactly where you stopped

🚫 WHEN NOT TO USE TIMESTAMP WATERMARK CDC

1. When source table has NO timestamp column and you CAN'T add one
   → Legacy system, no schema control
   → Use: full dump + hash comparison (expensive but works)
   → Or: DMS binlog-based CDC

2. When rows are HARD DELETED
   → DELETE FROM orders WHERE id = 123
   → Row disappears — no updated_at to detect
   → Use: DMS CDC (reads DELETE from binlog)
   → Or: full dump comparison (find missing IDs)

3. When you need sub-second latency
   → Watermark CDC is batch (runs every N minutes/hours)
   → Minimum practical interval: ~5 minutes
   → For sub-second: DMS CDC or Debezium streaming

4. When updated_at is not reliably maintained
   → If application code sometimes skips updating the timestamp
   → If bulk UPDATE scripts bypass the ON UPDATE trigger
   → Symptom: rows change but updated_at stays old → missed by CDC
   → Fix: enforce at DB level (ON UPDATE CURRENT_TIMESTAMP)

5. When change volume is > 50% of table
   → If half the table changes daily, CDC overhead > full dump
   → Watermark query + staging + merge > simple COPY + replace
   → Threshold: when CDC rows > ~30% of total, reconsider full dump

│  → Last watermark: 2025-01-14 23:59:59                                          │

│  → Transaction A: begins at 23:59:58, commits at 00:00:01                      │

│    - 23:59:58 < 23:59:59 → MISSED by CDC query!                                │
│                                                                                 │
│  SCENARIO with 5-minute overlap:                                                │
│  → Last watermark: 2025-01-14 23:59:59                                          │
│  → Adjusted query: WHERE updated_at > '2025-01-14 23:54:59'                    │
│  → Transaction A (23:59:58): 23:59:58 > 23:54:59 → CAPTURED ✅                 │
│  → Some rows from 23:55-23:59 are extracted AGAIN (duplicates)                  │
│  → Redshift MERGE handles duplicates: WHEN MATCHED → UPDATE (idempotent)       │

│  → MariaDB max long-running transaction: typically < 30 seconds                 │
│  → Network latency between MariaDB and extraction script: < 1 second            │
│  → Clock skew between servers: < 1 second (NTP synced)                          │
│  → 5 minutes = generous safety margin without excessive overlap                 │
│  → At 8K changes/day ≈ 5.5 changes/minute → overlap ≈ 28 extra rows            │
│  → 28 duplicate rows vs missing rows: always choose duplicates                  │

│                                                                                 │
│  ❹ Redshift MERGE Statement Internals                                          │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                                        │

│  MERGE INTO orders AS target                                                    │
│  USING staging_orders AS source                                                 │
│  ON target.order_id = source.order_id                                           │
│  WHEN MATCHED THEN UPDATE SET ...                                               │
│  WHEN NOT MATCHED THEN INSERT ...                                               │
│                                                                                 │
│  Internal execution:                                                            │
│  1. Redshift hash-joins staging_orders with orders on order_id                  │
│  2. For MATCHED rows: generates UPDATE operations (in-place block rewrite)      │
│  3. For NOT MATCHED rows: generates INSERT operations (append to blocks)        │
│  4. All operations are in a single implicit transaction                         │
│  5. If any step fails: entire MERGE rolls back                                  │
│  6. Concurrent queries see either ALL old data or ALL new data (MVCC)           │
│                                                                                 │
│  PERFORMANCE:                                                                   │
│  → staging_orders: 11K rows → tiny table                                        │
│  → orders: 5.2M rows → distribution key on order_id                             │
│  → Hash join: O(n) where n = staging rows                                       │
│  → Total MERGE time: ~2-5 seconds for 11K rows                                 │
│  → Compare: full table COPY replacement = 45 seconds + table lock               │
│                                                                                 │
│                                                                                 │
│  ❺ Classic DELETE+INSERT Pattern (Fallback for older Redshift)                 │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                      │
│                                                                                 │
│  BEGIN TRANSACTION;                                                             │
│  -- Delete existing rows that have updates                                      │
│  DELETE FROM orders                                                             │
│  WHERE order_id IN (SELECT order_id FROM staging_orders);                       │
│  -- Insert all rows from staging (new + updated)                                │
│  INSERT INTO orders SELECT * FROM staging_orders;                               │
│  COMMIT;                                                                        │
│                                                                                 │
│  WHY this works:                                                                │
│  → BEGIN/COMMIT = atomic (all or nothing)                                       │
│  → DELETE removes old versions of updated rows                                  │
│  → INSERT adds new versions + genuinely new rows                                │
│  → Net effect: same as MERGE                                                    │
│                                                                                 │
│  WHY MERGE is better:                                                           │
│  → Single statement vs three statements                                         │
│  → Optimizer can plan the entire operation together                             │
│  → Less transaction log overhead                                                │
│  → Clearer intent (readable, auditable)                                         │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "CDC query returns 0 rows but I know data changed"
→ Cause #1: updated_at column is NOT set to ON UPDATE CURRENT_TIMESTAMP
   - Column defined as TIMESTAMP DEFAULT CURRENT_TIMESTAMP (no ON UPDATE)
   - INSERT sets it, but UPDATE does NOT refresh it
   - Fix: ALTER TABLE orders MODIFY updated_at TIMESTAMP ... ON UPDATE CURRENT_TIMESTAMP
→ Cause #2: Application uses bulk UPDATE with explicit updated_at = old_value
   - Bypasses ON UPDATE trigger
   - Fix: remove explicit updated_at from UPDATE statements
→ Cause #3: Watermark is AHEAD of actual data (clock skew)
   - Fix: always derive watermark from MAX(updated_at) of extracted data

PITFALL 2: "CDC extracts 5 million rows — same as full dump"
→ Cause: watermark is '1970-01-01' (never updated after first run)
   - DynamoDB write failed silently
   - Fix: verify watermark update AFTER extraction, add error handling
→ Cause: overlap window is too large (e.g., 24 hours)
   - Everything from last 24 hours = nearly everything
   - Fix: reduce overlap to 5 minutes

PITFALL 3: "Duplicate rows appear in Redshift after CDC"
→ Cause: Using INSERT instead of MERGE
   - CDC row already exists in target → duplicate
   - Fix: always use MERGE or DELETE+INSERT pattern
→ NOT a problem if: using MERGE (WHEN MATCHED → UPDATE handles it)

PITFALL 4: "Redshift MERGE fails: duplicate key in staging table"
→ Cause: same order_id appears twice in CDC extract
   - Happens when: row updated twice within overlap window
   - Fix: deduplicate staging table before MERGE
   - SELECT DISTINCT ON (order_id) ORDER BY updated_at DESC

PITFALL 5: "MariaDB slows down during CDC extraction"
→ Cause: CDC query without index does full table scan
   - 5.2M row scan while production traffic is running
   - Fix: ensure INDEX on updated_at exists
→ Cause: extraction runs during peak hours
   - Fix: schedule during low-traffic window OR use read replica

PITFALL 6: "Watermark gets corrupted — stuck in infinite loop"
→ Cause: DynamoDB write succeeded but with wrong value
   - e.g., stored Python datetime object instead of ISO string
   - Fix: always store as ISO 8601 string, validate on read
→ Recovery: manually reset watermark in DynamoDB console

PITFALL 7: "First run after adding CDC takes forever"
→ Expected: first run extracts ALL historical rows (watermark = epoch)
→ This is the "initial load" — equivalent to a full dump
→ After first run, subsequent runs are incremental
→ Solution: accept first run is slow, or do initial load via COPY (Project 21)
  then set watermark to current timestamp in DynamoDB

📈 SCALABILITY CONSIDERATIONS

CURRENT LOAD (QuickCart):
→ 11K changed rows/day → 1.5 MB → processes in 30 seconds
→ Cost: ~$0.02/run

SCALING TO 100K CHANGES/DAY:
→ 100K rows → ~15 MB → processes in ~2 minutes
→ Redshift MERGE: ~10 seconds for 100K rows
→ Still very efficient vs 5.2M full dump

SCALING TO 1M CHANGES/DAY:
→ 1M rows → ~150 MB → Glue needs 4+ DPUs
→ Redshift MERGE: ~30-60 seconds for 1M rows
→ Consider: switch from nightly to hourly batches
→ 1M/24 = ~42K per hour → each batch processes faster

SCALING TO 10M CHANGES/DAY:
→ At this point: timestamp CDC overhead approaches full dump
→ 10M changed out of maybe 20M total = 50% change rate
→ SWITCH TO: DMS binlog-based CDC (Project 79)
→ Or: Kinesis streaming with micro-batch (Project 72)

WATERMARK TABLE (DynamoDB):
→ 1 read + 1 write per entity per run
→ 3 entities × 1 run/day = 3 reads + 3 writes/day
→ DynamoDB on-demand: essentially free at this volume
→ Even at 1000 entities × 24 runs/day = 24K reads + 24K writes/day
→ Cost: ~$0.03/day

STAGING TABLE (Redshift):
→ Created and dropped each run
→ No persistent storage cost
→ If run fails mid-MERGE: staging table persists until next run drops it
→ Add: DROP TABLE IF EXISTS staging_orders at start of each run

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO MONITOR CDC PIPELINE HEALTH:

1. DynamoDB Watermark Audit
   → After each run, DynamoDB stores: last_watermark, rows_extracted, duration
   → Dashboard: plot rows_extracted over time
   → Trend UP = table is growing or more updates
   → Sudden DROP to 0 = watermark bug or source table issue
   → Sudden SPIKE to millions = watermark reset or corruption

2. Extraction Metrics (CloudWatch Custom Metrics)
   → Metric: CDC/RowsExtracted (dimension: entity=orders)
   → Metric: CDC/ExtractionDurationSeconds
   → Metric: CDC/WatermarkLagSeconds (NOW - last_watermark)
   → Alarm: if RowsExtracted = 0 for 2 consecutive runs
   → Alarm: if WatermarkLagSeconds > 48 hours (pipeline stalled)

3. Redshift MERGE Audit
   → After MERGE: SELECT COUNT(*) from staging vs affected rows
   → Log: rows_inserted, rows_updated (from Redshift system tables)
   → stl_merge or stl_insert for detailed operation logs

4. Data Quality Check
   → After MERGE: SELECT COUNT(*) FROM orders (Redshift)
   → Compare with: SELECT COUNT(*) FROM orders (MariaDB)
   → Difference should be 0 (or very small due to timing)
   → Large difference = missed changes or duplicate inserts

   → Each CDC run is a Step Functions execution
   → Click to see: extraction time, row count, merge result
   → Red state = failure, with error details

🔑 KEY TAKEAWAY

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ Timestamp watermark CDC: simplest CDC method for batch pipelines
→ ON UPDATE CURRENT_TIMESTAMP: MariaDB auto-refreshes on any column change
→ INDEX on updated_at: critical — without it, CDC query = full table scan
→ Overlap window (5 min): prevents missed rows from clock skew / long transactions
→ Overlap creates duplicates → MERGE handles them idempotently
→ DynamoDB stores watermarks: strongly consistent, atomic, serverless
→ Watermark = MAX(updated_at) from EXTRACTED data, NOT from system clock
→ Staging table pattern: COPY → deduplicate → MERGE → DROP
→ MERGE is atomic (all-or-nothing), DELETE+INSERT has a brief gap
→ Data consistency verification: compare source vs target counts after merge
