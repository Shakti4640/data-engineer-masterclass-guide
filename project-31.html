EXTRACT MARIADB TABLES TO S3 PARQUET VIA GLUE JDBC
S3MariaDBGlue🔵 Phase 2: Easy

🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — now 18 months old, growing fast
Current State:
→ MariaDB on EC2 (Project 8, 9) holds 12 tables
   - orders (45M rows), customers (2M rows), products (50K rows)
   - order_items, payments, shipments, returns, reviews...
→ S3 data lake has raw CSVs from daily exports (Project 1, 19, 20)
→ Glue Crawlers catalog S3 CSVs (Project 23)
→ Glue ETL converts CSVs to Parquet (Project 25)
→ Athena queries work on S3 Parquet (Project 24, 26)

THE PROBLEM:
→ The nightly EC2 Python script (Project 20) that extracts MariaDB → CSV → S3
   is becoming unreliable:
   - Script crashes mid-export on large tables (45M rows)
   - No parallelism — single-threaded pymysql SELECT
   - CSV files are huge (2.3 GB for orders table)
   - No schema enforcement — CSV has no types
   - If EC2 instance dies, entire pipeline dies
   - No retry, no bookmark, no monitoring

→ Data team says: "We want a MANAGED, SCALABLE way to extract
   MariaDB tables directly into S3 as Parquet — with proper types,
   partitioning, and without managing any servers."

→ CTO says: "Use Glue. That's what we're paying for."

🟡 HOW TO APPROACH IT

Step 1: Understand what Glue JDBC does
→ Glue can connect to ANY JDBC-compatible database
→ MariaDB supports JDBC via MySQL JDBC driver (compatible)
→ Glue spins up Apache Spark workers INSIDE your VPC
→ Workers connect to MariaDB via JDBC, read tables, write to S3

Step 2: Understand the NETWORK path
→ Glue workers need NETWORK ACCESS to MariaDB on EC2
→ MariaDB is in a VPC → private subnet
→ Glue must run in the SAME VPC (or peered VPC)
→ Security Groups must allow Glue → MariaDB on port 3306

Step 3: Understand the authentication path
→ Glue needs MariaDB username + password
→ Stored in a Glue Connection object
→ Better: stored in Secrets Manager (Project 70)
→ For now: Glue Connection properties

Step 4: Understand the output
→ Read: MariaDB table via JDBC → DynamicFrame
→ Transform: ResolveChoice, type casting (Project 34 concepts)
→ Write: S3 as Parquet with partitioning

Step 5: Understand the dependencies
→ S3 bucket exists (Project 1)
→ MariaDB schema exists (Project 9)
→ Glue basics understood (Project 15, 23, 25)
→ Parquet format understood (Project 25)
→ VPC/Subnet/Security Group understood (Project 5, 10)

🔵 WHAT COULD BE DONE

Option A: Glue Visual Editor (Console drag-and-drop)
→ Quick to prototype
→ Generates PySpark code behind the scenes
→ Limited control over JDBC pushdown, partitioning
→ Not reproducible, not version-controllable

Option B: Glue PySpark Script (recommended)
→ Full control over JDBC read parallelism
→ Custom partitioning logic
→ Version controllable in Git
→ Interview-demonstrable

Option C: Glue Python Shell Job
→ Uses pymysql/pandas instead of Spark
→ Single node — no parallelism
→ Max 1 DPU — limited to small tables
→ Same problems as EC2 script

Option D: Glue Crawler on JDBC source + separate ETL
→ Crawler catalogs MariaDB tables in Glue Catalog
→ Then ETL reads from catalog
→ Two steps instead of one — unnecessary complexity here

DECISION: Option B — Glue PySpark Script
→ Parallel reads via JDBC partitioning
→ Parquet output with proper partitioning
→ Handles 45M rows efficiently
→ Production-grade pattern

✅ WHAT SHOULD DO

1. Create a DEDICATED Security Group for Glue
   → Glue workers communicate with each other
   → MUST allow ALL TCP inbound from ITSELF (self-referencing rule)
   → This is the #1 missed step — job fails with "connection timeout"

2. Create S3 VPC Gateway Endpoint
   → Glue workers in private subnet have NO internet access
   → Without this endpoint, Glue cannot write to S3
   → Free — no data transfer charges through Gateway Endpoint

3. Use JDBC hashfield or hashpartitions for parallel reads
   → Default: single JDBC connection reads entire table sequentially
   → With hashfield: Glue splits reads across multiple Spark tasks
   → 10x-50x faster for large tables

4. Store credentials in Glue Connection (minimum) or Secrets Manager (ideal)
   → Never in job script or job parameters

5. Use DynamicFrame over DataFrame for Glue-specific features
   → Handles schema inconsistencies (Project 45)
   → Integrates with Glue Catalog natively

6. Write as PARTITIONED Parquet
   → Partition by extraction date at minimum
   → Enables incremental reads downstream (Athena, Spectrum)

7. Use --enable-metrics and --enable-continuous-cloudwatch-log
   → Monitor DPU usage, memory, shuffle spills
   → Debug failures without guessing

8. Create a DEDICATED MariaDB read-only user for Glue
   → Never use root/admin credentials
   → GRANT SELECT only on specific schema

❌ WHAT SHOULD NOT DO

1. DO NOT put Glue workers in a PUBLIC subnet
   → Glue requires private subnet with NAT Gateway or VPC Endpoints
   → Public subnet = Glue job fails silently

2. DO NOT forget the self-referencing Security Group rule
   → Glue Spark workers must communicate with each other
   → Without this: "Error connecting to Spark" / timeout after 15 min
   → Rule: sg-glue → sg-glue, ALL TCP, 0-65535

3. DO NOT read a 45M row table without JDBC parallelism
   → Single connection = single thread = hours for large tables
   → Use hashfield parameter to split across partitions

4. DO NOT use Glue 2.0 — use Glue 4.0
   → Glue 4.0 = Spark 3.3 + Python 3.10 + faster startup
   → Glue 2.0 is legacy — 10x slower cold start

5. DO NOT skip S3 VPC Endpoint
   → Glue in private subnet cannot reach S3 over internet
   → Job hangs indefinitely on write step
   → Error: "timeout writing to S3" (misleading — it's a network issue)

6. DO NOT use overly broad IAM permissions
   → Bad:  "s3:*" on "*"
   → Good: "s3:PutObject" on "arn:aws:s3:::quickcart-datalake-prod/bronze/*"

7. DO NOT hardcode JDBC URL in the script
   → Use Glue Connection — it manages the URL, creds, VPC config together
   → Connection can be reused across multiple jobs

8. DO NOT write output without a date partition
   → Without partition: every full extract overwrites the last one
   → Or: data accumulates with no way to identify when it was extracted

🚫 WHEN NOT TO USE GLUE JDBC

1. Table has < 10,000 rows
   → Glue minimum billing: 1 DPU × 10 minutes = $0.44
   → A Lambda function with pymysql costs $0.0001
   → Overkill for tiny tables

2. Need sub-second latency extraction
   → Glue cold start: 30-60 seconds (Glue 4.0) to 2-3 min (Glue 2.0)
   → Use: Direct JDBC query from Lambda or EC2

3. Source database has no JDBC driver
   → MongoDB, DynamoDB, Cassandra → use native Glue connectors
   → Or Marketplace connectors

4. Need to extract binary/BLOB columns
   → JDBC handles BLOBs poorly over network
   → Better: export via database-native tools, then upload to S3

5. Source is in a different AWS account with no VPC peering
   → Glue Connection requires VPC access
   → Need VPC Peering (Project 61) or PrivateLink first
   → Alternative: DMS with public endpoint (less secure)

6. You need to write back to the database
   → Glue CAN write to JDBC targets, but it's risky for OLTP
   → Covered in Project 43

│                                                                                  │
│  WHY THIS MATTERS:                                                               │
│  → Glue workers LIVE inside your VPC during job execution                        │
│  → They can reach anything your VPC can reach                                    │
│  → They are subject to YOUR Security Group rules                                 │
│  → They are subject to YOUR Route Table                                          │
│  → They CANNOT reach internet unless NAT Gateway exists                          │
│  → They CANNOT reach S3 unless VPC Endpoint exists                               │

│  STEP 3: JDBC READ (How Spark reads from MariaDB)                                │
│  ────────────────────────────────────────────                                    │
│                                                                                  │
│  WITHOUT hashfield (DEFAULT — SINGLE THREAD):                                    │
│  ┌─────────────────────────────────────────────────────┐                         │
│  │ Executor 1: SELECT * FROM orders                    │ ← ONE connection        │
│  │             → Reads ALL 45M rows sequentially        │ ← Takes 45 minutes     │
│  │ Executor 2: (idle)                                   │                         │
│  │ Executor 3: (idle)                                   │                         │
│  │ ...                                                  │                         │
│  │ Executor 10: (idle)                                  │                         │
│  └─────────────────────────────────────────────────────┘                         │
│  → You're paying for 10 DPUs but using 1                                         │
│  → 9 executors sit idle — massive waste                                          │
│                                                                                  │
│  WITH hashfield="order_id" + hashpartitions=10:                                  │
│  ┌─────────────────────────────────────────────────────┐                         │
│  │ Executor 1:  SELECT * FROM orders                   │                         │
│  │              WHERE MOD(hash(order_id), 10) = 0       │ ← 4.5M rows            │
│  │ Executor 2:  SELECT * FROM orders                   │                         │
│  │              WHERE MOD(hash(order_id), 10) = 1       │ ← 4.5M rows            │
│  │ Executor 3:  ...WHERE MOD(...) = 2                  │ ← 4.5M rows            │
│  │ ...                                                  │                         │
│  │ Executor 10: ...WHERE MOD(...) = 9                  │ ← 4.5M rows            │
│  └─────────────────────────────────────────────────────┘                         │
│  → 10 parallel JDBC connections                                                  │
│  → Each reads ~4.5M rows                                                         │
│  → Total time: ~4.5 minutes (10x faster)                                         │
│  → MariaDB sees 10 concurrent connections                                        │
│                                                                                  │
│  STEP 4: IN-MEMORY PROCESSING                                                    │
│  ────────────────────────────                                                    │
│  → Each executor holds its partition in memory as Spark DataFrame                │
│  → DynamicFrame wraps DataFrame with Glue-specific features                      │
│  → Transforms applied: type casting, column rename, filter                       │
│  → Data serialized as Parquet (columnar, compressed)                             │
│                                                                                  │
│  STEP 5: S3 WRITE                                                                │
│  ────────────────                                                                │
│  → Each executor writes its partition as separate Parquet file                   │
│  → 10 executors → 10 Parquet files (approximately)                               │
│  → Files written via S3 VPC Gateway Endpoint                                     │
│  → No internet traversal → fast + free                                           │
│                                                                                  │
│  STEP 6: CLEANUP                                                                 │
│  ───────────────                                                                 │
│  → Glue releases ENIs from your subnet                                           │
│  → Private IPs returned to subnet pool                                           │
│  → DPUs deallocated — billing stops                                              │
│  → CloudWatch logs finalized                                                     │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Glue job times out after 15 minutes with no error"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Security Group missing self-referencing rule
→ Glue workers cannot talk to each other
→ Spark driver cannot coordinate with executors
→ Fix:
   sg-glue INBOUND: All TCP (0-65535) from sg-glue (ITSELF)
   sg-glue OUTBOUND: All traffic to 0.0.0.0/0

PITFALL 2: "Connection test succeeds but job fails with JDBC error"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Connection test uses 1 ENI in 1 AZ
→ Job may allocate ENIs in DIFFERENT AZ than MariaDB
→ If subnet only exists in 1 AZ, other AZ workers fail
→ Fix: Use same AZ subnet for both Glue and MariaDB
→ OR: specify availabilityZone in connection properties

PITFALL 3: "Job writes to S3 but files are empty / 0 bytes"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: JDBC URL wrong — connected to empty database/schema
→ Debug: Add print(dyf.count()) after read to verify row count
→ Fix: Verify JDBC URL includes correct database name

PITFALL 4: "Job succeeds but MariaDB CPU hits 100% and app slows down"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: 10 parallel JDBC connections each doing full table scan
→ MariaDB isn't designed for 10 concurrent analytical queries
→ Fix: 
   a) Run during off-peak hours (2 AM)
   b) Reduce hashpartitions to 3-4
   c) Use a READ REPLICA instead of primary
   d) Add resource limits to glue_reader user

PITFALL 5: "Glue cannot reach S3 — job hangs on write"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: No S3 VPC Gateway Endpoint
→ Private subnet has no internet access
→ Glue workers can't reach S3 public endpoint
→ Fix: Create S3 Gateway Endpoint attached to your VPC
→ Also need: Glue VPC Endpoint for catalog access (optional but recommended)

PITFALL 6: "Job fails: Not enough IP addresses in subnet"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Each DPU needs 1 ENI = 1 private IP
→ 10 DPUs in a /28 subnet (16 IPs, 11 usable) = not enough
→ Fix: Use at least a /24 subnet (256 IPs) for Glue
→ PLAN: 2 IPs per DPU (worker + shuffle) → 10 DPUs = 20 IPs minimum

PITFALL 7: "MariaDB JDBC URL uses 'localhost' — connection refused"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: 'localhost' resolves to the Glue worker, not MariaDB
→ Fix: Use MariaDB EC2's PRIVATE IP (e.g., 10.0.1.50)
→ Never use public IP (data crosses internet unnecessarily)

PITFALL 8: "Parquet files are 1KB each — thousands of tiny files"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Too many Spark partitions for a small table
→ 10 DPUs reading 5000 rows → 10 files of 500 rows each
→ Fix: coalesce(1) before write for small tables
→ This problem is addressed deeply in Project 62

│ < 100K rows     │ 2        │ Not needed       │ 2-3 minutes      │

│ 1M - 10M rows   │ 10       │ 10               │ 5-10 minutes     │
│ 10M - 100M rows │ 20       │ 20               │ 10-20 minutes    │
│ 100M+ rows      │ 40+      │ 40               │ 20-45 minutes    │
└─────────────────┴──────────┴──────────────────┴──────────────────┘

MARIADB CONNECTION LIMITS:
→ Default max_connections: 151
→ Glue hashpartitions=20 → 20 connections
→ Leave headroom: application needs connections too
→ Rule: Glue connections ≤ 30% of max_connections

SUBNET IP EXHAUSTION:
→ 10 DPUs ≈ 20 ENIs (worker + shuffle service)
→ /24 subnet: 251 usable IPs → safe for up to 100 DPUs
→ /27 subnet: 27 usable IPs → only 10-12 DPUs max
→ Multiple Glue jobs sharing same subnet compound the problem

MULTIPLE TABLES:
→ Option A: One job per table (simpler, parallel execution)
→ Option B: One job extracts all tables (fewer jobs, sequential)
→ Recommended: Option A for large tables, Option B for small tables
→ Orchestrate via Glue Workflow (Project 35) or Step Functions (Project 55)

AUTO SCALING (Glue 4.0):
→ Set NumberOfWorkers as MAX
→ Enable auto-scaling: Glue reduces workers if not needed
→ Saves cost for variable-size tables
→ Set: --enable-auto-scaling = true

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY THE EXTRACTION WORKED:

1. ROW COUNT VERIFICATION
   → In Glue script: print(f"Rows read: {dyf.count()}")
   → In MariaDB: SELECT COUNT(*) FROM orders
   → Compare: must match (or match with WHERE filter)

2. SCHEMA VERIFICATION
   → In Glue: dyf.printSchema()
   → Check: column names match, types make sense
   → Common issue: MariaDB DECIMAL → Glue string (need ResolveChoice)

3. SAMPLE DATA VERIFICATION
   → In Glue: dyf.toDF().show(5)
   → Spot check: do values look right?
   → Check: NULLs, date formats, special characters

4. S3 OUTPUT VERIFICATION
   → aws s3 ls s3://bucket/bronze/mariadb/orders/ --recursive
   → Check: files exist, sizes are reasonable
   → Check: partition structure is correct

5. ATHENA VERIFICATION (Project 24)
   → SELECT COUNT(*) FROM "bronze_mariadb"."orders"
   → SELECT * FROM "bronze_mariadb"."orders" LIMIT 10
   → Must match MariaDB source

6. CLOUDWATCH LOGS
   → Glue job run → "Logs" link → CloudWatch
   → Check: no ERROR lines
   → Check: row counts logged
   → Check: duration is reasonable

7. GLUE JOB METRICS
   → Enable --enable-metrics
   → CloudWatch → Glue Job Metrics
   → Check: DPU utilization, memory usage, data shuffle
