🟡 HOW TO APPROACH IT

Step 1: Understand the GAP between S3 and Athena
→ S3 stores raw bytes — it has NO concept of "columns" or "data types"
→ Athena needs schema metadata to parse those bytes into queryable rows
→ Glue Data Catalog is the metadata store (introduced in Project 15)
→ In Project 15: we MANUALLY created catalog tables
→ Problem: manual is error-prone and doesn't scale

Step 2: Understand what a Glue Crawler does
→ Crawler = automated schema discovery agent
→ You tell it: "look at this S3 prefix"
→ It reads file samples → infers:
   - File format (CSV, Parquet, JSON, ORC, Avro)
   - Column names and data types
   - Partition structure (year=2025/month=01/)
   - Number of rows (approximate)
→ Creates/updates Glue Catalog tables with discovered schema

Step 3: Plan the crawling strategy
→ One crawler per dataset? Or one crawler for everything?
→ Best practice: one crawler per S3 prefix with homogeneous data
→ Why: mixing CSVs and Parquet in one crawler causes confusion
→ QuickCart: 3 crawlers for daily_exports (orders, customers, products)
   + 1 crawler for Parquet exports
   + 1 crawler for MariaDB extracts

Step 4: Understand Crawler output
→ Crawler creates tables in a Glue Database (logical grouping)
→ Table name derived from S3 prefix (e.g., "daily_exports_orders")
→ Once table exists in catalog → Athena can query immediately
→ Re-running crawler: UPDATES existing table if schema changed

✅ WHAT SHOULD DO

1. Create a DEDICATED Glue Database for the data lake
   → Logical grouping: all QuickCart tables in one database
   → Name: quickcart_datalake (lowercase, underscores)
   → Separate from any default database

2. Create SEPARATE crawlers per data format/source
   → Crawler 1: daily_exports/ CSVs
   → Crawler 2: exports/data_science/ Parquet
   → Crawler 3: mariadb_extracts/ CSVs
   → WHY separate: each format needs different classifier settings
   → Mixing formats in one crawler causes misclassification

3. Set TABLE PREFIX on each crawler
   → Prevents naming collisions
   → Crawler for daily_exports/ → prefix "raw_"
   → Creates: raw_orders, raw_customers, raw_products
   → Without prefix: table names derived from S3 folder names only

4. Configure SCHEMA CHANGE POLICY
   → "Update table definition in catalog" — applies schema changes
   → "Add new columns only" — never removes columns (safest)
   → "Log" — just logs changes without modifying catalog
   → Production: "Add new columns only" prevents accidental column deletion

5. Configure CRAWLER RECRAWL POLICY
   → "Crawl all folders" — re-scans everything (thorough but slow)
   → "Crawl new folders only" — only new S3 prefixes (fast)
   → "Event-based" — triggers on S3 events (most efficient)
   → Start with "Crawl all folders" → optimize later

6. Set appropriate IAM Role for crawler
   → Needs: s3:GetObject, s3:ListBucket on target buckets
   → Needs: glue:* for catalog operations
   → Needs: logs:* for CloudWatch logging

7. Run crawler ON-DEMAND first to validate
   → Don't schedule until you verify output is correct
   → Check created tables in Glue console
   → Test with Athena query before scheduling

❌ WHAT SHOULD NOT DO

1. DO NOT point ONE crawler at the entire bucket root
   → s3://quickcart-raw-data-prod/ as single target
   → Crawler tries to merge CSVs + Parquets + manifests into one table
   → Results: schema confusion, wrong data types, garbage tables
   → Rule: one crawler target = one homogeneous data format

2. DO NOT skip the table prefix
   → Without prefix: crawler creates table named "orders"
   → Another crawler also creates table named "orders"
   → Second crawler OVERWRITES first table's schema
   → With prefix: "raw_orders" vs "parquet_orders" — no collision

3. DO NOT trust crawler data type inference blindly
   → Crawler samples a SUBSET of files (not all)
   → If first 1000 rows of "price" column are "9.99", "12.50"
     → Crawler infers: DOUBLE
   → But row 50,001 has "N/A"
     → Athena query fails: "HIVE_BAD_DATA: Field price is not a valid DOUBLE"
   → ALWAYS spot-check inferred types
   → Override in catalog if needed

4. DO NOT run crawler too frequently on unchanged data
   → Each run costs time and API calls
   → Daily data → daily crawler run (max)
   → Hourly data → consider event-based crawling
   → Unchanged data → on-demand only

5. DO NOT let crawler create tables in the "default" database
   → Default database is shared, messy, hard to manage
   → Always specify a dedicated database
   → Makes permissions, discovery, and cleanup easier

6. DO NOT ignore crawler logs when tables look wrong
   → CloudWatch Logs: /aws-glue/crawlers
   → Shows: which files were read, what was inferred, errors
   → Most schema issues are explained in these logs

7. DO NOT use crawler for data with NO consistent schema
   → Free-form JSON with varying keys per record
   → CSVs where column count changes between files
   → Crawler creates unusable tables with merged/conflicting columns
   → Pre-process such data to standardize schema FIRST (Glue ETL — Project 25)

8. DO NOT assume crawler handles HEADER detection perfectly for CSV
   → Crawler may misinterpret first data row as header or vice versa
   → Configure custom classifier to explicitly set "has header" behavior

🚫 WHEN NOT TO USE GLUE CRAWLER

1. When schema is 100% known and NEVER changes
   → Manual catalog definition is simpler and more precise
   → Crawler adds unnecessary complexity for static schemas
   → Example: dimension tables with fixed 5 columns

2. When S3 prefix contains MIXED file formats
   → CSVs + Parquets + JSONs in same prefix
   → Crawler gets confused trying to merge schemas
   → Separate by format first, then crawl each

3. When files have NO consistent structure
   → Log files with varying formats
   → Free-form text files
   → Crawler creates useless tables
   → Pre-process with Glue ETL or Lambda first

4. When you need REAL-TIME catalog updates
   → Crawler takes 30-120 seconds minimum to run
   → For sub-second catalog updates: use boto3 glue.create_table() directly
   → Event-driven architecture: S3 event → Lambda → catalog update

5. When you need COLUMN-LEVEL descriptions/comments
   → Crawler discovers structure but NOT business meaning
   → "col1: string" tells you nothing about what it represents
   → Add descriptions manually via console or API after crawling

6. When cost optimization is critical at extreme scale
   → Crawler with 100,000+ files scans extensively
   → Each scan = S3 GET requests ($0.0004 per 1000)
   → 100K files × daily crawler = cost adds up
   → Consider: crawl ONCE, then manage catalog programmatically

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Crawler created ONE table instead of THREE"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: all CSVs under one prefix have same column count
→ Crawler merges: orders(8 cols) + customers(7 cols) → confused
→ If column counts happen to match → merged into single table
→ Fix: create SEPARATE crawlers for each sub-prefix
   - Crawler 1 target: s3://bucket/daily_exports/orders/
   - Crawler 2 target: s3://bucket/daily_exports/customers/
   - Crawler 3 target: s3://bucket/daily_exports/products/

PITFALL 2: "Column types are all STRING even for numeric data"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: CSV has mixed values — even ONE non-numeric row makes entire column STRING
→ Example: price column has "N/A" in row 500 → entire column → STRING
→ Fix A: clean source data before crawling
→ Fix B: manually update column type in Glue Catalog after crawling
→ Fix C: use Parquet instead of CSV — types are embedded and exact

PITFALL 3: "Crawler treats CSV header row as data"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: built-in CSV classifier failed to detect header
→ Columns named: col0, col1, col2... instead of order_id, customer_id...
→ Fix: create a CUSTOM CSV classifier with explicit header handling
→ Set: "Has header: true" in custom classifier
→ Attach custom classifier to crawler (higher priority than built-in)

PITFALL 4: "Crawler runs for 30 minutes on 100 files"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: "Crawl all folders" policy re-scans everything every time
→ Fix: switch to "Crawl new folders only" after initial crawl
→ Or: use event-based crawling (S3 event → triggers crawler)
→ First run: always slow (initial discovery)
→ Subsequent runs: should be fast if only new files added

PITFALL 5: "Athena query returns 0 rows but table exists in catalog"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause A: table's S3 location points to wrong prefix
→ Cause B: partitioned table but no partitions registered
→ Debug A: check table's "Location" in Glue console → verify S3 path
→ Debug B: run MSCK REPAIR TABLE in Athena to discover partitions
→ Or: re-run crawler to pick up new partitions

PITFALL 6: "Crawler creates duplicate tables on re-run"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: S3 path structure changed between runs
→ Crawler sees new prefix pattern → creates new table
→ Old table still exists → now have: orders AND orders_v2
→ Fix: keep S3 structure stable, use schema change policy "Update"
→ Clean up orphaned tables manually

PITFALL 7: "Crawler IAM role access denied"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Role needs THREE sets of permissions:
   1. S3: s3:GetObject, s3:ListBucket (read source data)
   2. Glue: glue:* (create/update catalog)
   3. Logs: logs:CreateLogGroup, logs:PutLogEvents (CloudWatch)
→ Missing ANY of these → crawler fails
→ Trust policy must allow: glue.amazonaws.com to assume role

PITFALL 8: "Crawler inferred wrong delimiter for CSV"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: data contains commas inside quoted fields
→ Crawler misdetects: pipe-delimited instead of comma
→ Result: all data in one column
→ Fix: custom CSV classifier with explicit delimiter = ","
→ Set quote character = '"'

📈 SCALABILITY CONSIDERATIONS

FILE COUNT SCALING:
━━━━━━━━━━━━━━━━━━
→ 10 files:     crawler runs in ~30 seconds
→ 100 files:    crawler runs in ~1-2 minutes
→ 1,000 files:  crawler runs in ~3-5 minutes
→ 10,000 files: crawler runs in ~10-15 minutes
→ 100,000 files: crawler runs in ~30-60 minutes (expensive!)

COST SCALING:
━━━━━━━━━━━━━
→ Minimum: $0.073 per run (10-minute minimum billing)
→ 1 run/day × 30 days = ~$2.19/month (trivial)
→ 24 runs/day × 30 days = ~$52.56/month (hourly schedule)
→ Optimize: use event-based triggers instead of schedules

PARTITION SCALING:
━━━━━━━━━━━━━━━━━━
→ 365 partitions (daily for 1 year): crawler handles fine
→ 8,760 partitions (hourly for 1 year): crawler slows down
→ 100,000+ partitions: crawler becomes very slow
→ Solution: Partition Projection (Project 26) — no crawling needed
→ Athena computes partitions mathematically instead of scanning catalog

MULTI-TABLE SCALING:
━━━━━━━━━━━━━━━━━━━━
→ 1 crawler can discover multiple tables from one prefix
→ But mixing formats reduces accuracy
→ Best practice: one crawler per homogeneous data source
→ 10 data sources = 10 crawlers (manageable)
→ 100 data sources = consider programmatic catalog management

OPTIMIZATION TIPS:
━━━━━━━━━━━━━━━━━━
→ Use Parquet over CSV — crawling is 10x faster (no data sampling)
→ Use "Crawl new folders only" after initial crawl
→ Use event-based triggers (S3 → EventBridge → crawler)
→ Group related tables under same crawler (same format, same source)
→ Use Partition Projection for high-cardinality partitions
