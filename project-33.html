🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — Pipeline running for 3 months
Current State (after Projects 31, 32):
→ Nightly pipeline: MariaDB → S3 Parquet → Redshift
→ S3 bronze layer accumulates daily partitions:
   bronze/mariadb/orders/year=2025/month=01/day=01/
   bronze/mariadb/orders/year=2025/month=01/day=02/
   ...
   bronze/mariadb/orders/year=2025/month=03/day=31/  (90 partitions)

→ ALSO: A second pipeline exists that processes RAW CSV uploads
   from partner companies:
   raw/partner_feeds/partner_A/feed_20250101.csv
   raw/partner_feeds/partner_A/feed_20250102.csv
   ...
   raw/partner_feeds/partner_A/feed_20250331.csv  (90 files)

THE PROBLEM:
→ Pipeline A (Parquet transform job from Project 25):
   - Reads ALL CSV files from raw/ every run
   - Converts to Parquet, writes to bronze/
   - Day 1: processes 1 file → 2 minutes
   - Day 90: processes 90 files → 45 minutes
   - Day 365: processes 365 files → 3+ hours
   - COST: 10 DPUs × 3 hours = $13.20/day (was $0.73 on Day 1)

→ Pipeline B (Silver layer aggregation):
   - Reads ALL Parquet from bronze/ every run
   - Aggregates into silver/ layer
   - Same exponential growth problem

→ Data engineer says: "90% of our Glue compute cost is 
   RE-PROCESSING files we already processed yesterday.
   Only 1 new file arrives each day.
   We need Glue to REMEMBER what it already processed."

→ CTO looks at the $400/month Glue bill (was $22 in Month 1):
   "Fix this. Yesterday's files haven't changed. Don't touch them."

🟡 HOW TO APPROACH IT

Step 1: Understand the ROOT problem
→ Glue jobs are STATELESS by default
→ Each run starts fresh — no memory of previous runs
→ "Read from S3 path" means "read ALL files at that path"
→ As files accumulate, each run reads more and more

Step 2: Understand the SOLUTIONS landscape
→ Solution A: Glue Job Bookmarks (built-in)
→ Solution B: Custom watermark tracking (manual)
→ Solution C: S3 event-driven processing (per-file trigger)
→ Solution D: Partition-based filtering (date-aware reads)
→ Each has trade-offs — we'll implement ALL four

Step 3: Understand WHEN each solution fits
→ Job Bookmarks: best for S3 source, automatic, zero code
→ Custom watermark: best for JDBC source, flexible, more control
→ Event-driven: best for real-time, file-at-a-time processing
→ Partition filtering: best for date-partitioned data, explicit

Step 4: Understand the dependencies
→ S3 data accumulating over time (Projects 1, 19, 20, 31)
→ Glue ETL jobs processing S3 data (Projects 25, 31, 32)
→ Parquet partitioning understood (Projects 25, 26)
→ Glue job parameters understood (Projects 31, 32)

🔵 WHAT COULD BE DONE

Option A: Just enable Job Bookmarks flag
→ Change: --job-bookmark-option from "job-bookmark-disable" to "job-bookmark-enable"
→ Glue automatically tracks processed files
→ Pros: zero code change, built-in
→ Cons: opaque — hard to debug, can't customize filter logic

Option B: Build custom watermark with DynamoDB
→ Store: {job_name, last_processed_timestamp, last_file_path}
→ Each run: read watermark → filter source → update watermark
→ Pros: full visibility, queryable state, flexible
→ Cons: more code, another AWS service to manage

Option C: Build custom watermark with S3 marker file
→ Store: s3://bucket/state/{job_name}/watermark.json
→ Same logic as Option B but uses S3 instead of DynamoDB
→ Pros: no extra service, simple JSON file
→ Cons: not atomic (race condition if two runs overlap)

Option D: Combine Approaches — Belt and Suspenders
→ Use Job Bookmarks for S3 sources (automatic)
→ Use custom watermark for JDBC sources (Project 42 CDC)
→ Use date partition for Redshift loads (explicit, predictable)

DECISION: Option D — Combined approach
→ Each mechanism for its best use case
→ Most robust in production
→ Interview demonstrates understanding of ALL patterns

✅ WHAT SHOULD DO

1. Enable Job Bookmarks for S3-to-S3 transform jobs
   → Set --job-bookmark-option = "job-bookmark-enable"
   → Use transformation_ctx parameter in EVERY read and write call
   → transformation_ctx is HOW bookmarks track state per source

2. Use transformation_ctx CONSISTENTLY
   → Every create_dynamic_frame call MUST have unique transformation_ctx
   → Bookmark stores: {transformation_ctx → last_processed_file}
   → If you change transformation_ctx, bookmark resets → reprocesses everything

3. Test bookmark behavior BEFORE production
   → Run 1: processes files A, B, C → bookmark saves state
   → Run 2: add files D, E → should process ONLY D, E
   → Run 3: no new files → should process NOTHING (0 rows)
   → Verify all three scenarios

4. Implement bookmark RESET capability
   → Sometimes you NEED to reprocess everything (schema change, bug fix)
   → API: glue.reset_job_bookmark(JobName="...")
   → After reset, next run processes ALL files from scratch

5. Use custom watermark for JDBC sources
   → Job Bookmarks with JDBC track by primary key column
   → Works but is less flexible than custom watermark
   → Custom watermark lets you filter by updated_at timestamp

6. Log bookmark behavior in CloudWatch
   → Print: "Bookmark enabled — processing only new files"
   → Print: "Files processed this run: N"
   → Print: "Rows processed this run: N"
   → Essential for debugging "why did it process 0 rows?"

7. Set job concurrency to 1 for bookmarked jobs
   → Two concurrent runs with same bookmark = corruption
   → One finishes, updates bookmark → other finishes, overwrites bookmark
   → Result: some files skipped permanently

❌ WHAT SHOULD NOT DO

1. DO NOT enable bookmarks without transformation_ctx
   → Bookmark REQUIRES transformation_ctx to track state
   → Without it: bookmark does nothing — all files reprocessed every time
   → This is the #1 "bookmark isn't working" complaint

2. DO NOT change transformation_ctx between runs
   → Run 1: ctx="read_source" → bookmark stores state for "read_source"
   → Run 2: ctx="read_src" → different key → bookmark has no state → full reprocess
   → NEVER rename transformation_ctx after first production run

3. DO NOT use bookmarks with overwrite mode on output
   → Bookmark tracks INPUT files processed
   → If output mode is "overwrite", each run replaces ALL output
   → Result: output only contains data from NEW files, old data gone
   → Use "append" mode or partition-based writes with bookmarks

4. DO NOT run bookmarked jobs concurrently
   → Already explained in "Should Do" #7
   → Set MaxConcurrentRuns=1 on the job

5. DO NOT assume bookmarks work with DataFrame API
   → Bookmarks ONLY work with DynamicFrame API
   → spark.read.parquet() → NO bookmark tracking
   → glueContext.create_dynamic_frame.from_options() → YES bookmark tracking
   → If you convert to DataFrame for transforms, that's fine
   → But the READ must be via DynamicFrame

6. DO NOT forget to handle bookmark FAILURES
   → If job fails AFTER reading but BEFORE committing
   → Bookmark is NOT updated → next run reprocesses same files
   → This is GOOD — at-least-once semantics
   → But your downstream must handle duplicates (idempotent writes)

7. DO NOT use bookmarks for JDBC sources with no monotonic key
   → JDBC bookmark tracks by primary key or timestamp column
   → If table has no auto-increment or updated_at → bookmark can't track
   → Use custom watermark instead

🚫 WHEN NOT TO USE JOB BOOKMARKS

1. Source files are MODIFIED in place (not append-only)
   → Bookmark tracks: "I processed file X with timestamp T"
   → If file X is overwritten with new content but same name
   → Bookmark may skip it (depends on timestamp change)
   → Use: custom watermark with content hash, or event-driven

2. You need exactly-once processing guarantees
   → Bookmarks provide AT-LEAST-ONCE (may reprocess on failure)
   → If downstream can't handle duplicates, bookmarks alone aren't enough
   → Use: bookmarks + idempotent writes (upsert pattern from Project 32)

3. Job processes multiple INDEPENDENT data flows
   → Job reads from 5 different S3 prefixes
   → Bookmark tracks all or nothing — can't reset one source independently
   → Use: separate jobs per source, each with own bookmark

4. You're using spark.read (DataFrame API) not DynamicFrame
   → Bookmarks ONLY work with Glue DynamicFrame API
   → If your job uses pure Spark: implement custom watermark

5. Data arrives out of order
   → File from January arrives in March (late data)
   → Bookmark has already moved past January
   → Late file may be skipped
   → Use: event-driven processing (triggers on file arrival regardless of date)

6. Job writes to multiple destinations
   → Bookmark commits ONCE at job.commit()
   → If write to destination A succeeds but B fails
   → Bookmark still commits → A's data reprocessed on retry
   → Use: separate jobs per destination

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Bookmark enabled but job still processes ALL files"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause #1: Missing transformation_ctx in create_dynamic_frame call
→ Cause #2: Using spark.read.parquet() instead of DynamicFrame API
→ Cause #3: Job parameter set to "job-bookmark-disable" (typo or default)
→ Fix: Verify ALL three:
   a) --job-bookmark-option = "job-bookmark-enable"
   b) transformation_ctx="unique_name" in every read
   c) job.commit() at end of script

PITFALL 2: "Second run processes 0 files — but new files exist"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: New files have timestamps OLDER than bookmark watermark
→ Example: file was created yesterday but uploaded today
→ S3 last_modified shows yesterday → bookmark thinks it's old
→ Fix: Ensure file upload timestamps are monotonically increasing
→ OR: Use event-driven approach instead of bookmark

PITFALL 3: "Job runs concurrently — bookmark gets corrupted"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Two job runs overlap
→ Run A reads files 1-10, Run B reads files 1-10 (same bookmark)
→ Run A commits → bookmark = file 10
→ Run B commits → bookmark = file 10
→ File 11 arrives between A and B → may be skipped
→ Fix: Set MaxConcurrentRuns=1 in job definition

PITFALL 4: "Bookmark works for S3 reads but not for JDBC reads"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: JDBC bookmark requires specific column configuration
→ Must specify: jobBookmarkKeys = [column_name] in connection options
→ Column must be monotonically increasing (auto_increment, timestamp)
→ Without jobBookmarkKeys: entire table re-read every run

PITFALL 5: "After resetting bookmark, job takes 3 hours"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Expected behavior — reset means "reprocess everything"
→ If you have 365 days of files, full reprocess is expensive
→ Fix: instead of reset, manually set a custom watermark
→ OR: move old files to archive prefix, reset, then only new prefix exists

PITFALL 6: "Output has duplicates after job retry"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Job failed after writing output but before job.commit()
→ Bookmark not updated → next run reprocesses same files
→ Output written AGAIN → duplicates
→ Fix: Make writes idempotent:
   a) Write to date-partitioned path → overwrite partition on retry
   b) Use staging + merge pattern (Project 32) for Redshift
   c) Use CTAS with IF NOT EXISTS for Athena tables

━━━━━━━━━━━━━━━━━━━━
→ Bookmark check is an S3 LIST + filter operation
→ LIST returns 1000 objects per page
→ 10,000 files = 10 LIST calls before filtering
→ 100,000 files = 100 LIST calls → several seconds overhead
→ 1,000,000 files = bookmark overhead becomes significant

MITIGATION FOR LARGE FILE COUNTS:
→ Organize files by date prefix:
   raw/partner_feeds/year=2025/month=01/feed_01.csv
→ Bookmark filters within the listed prefix
→ Fewer files to list = faster bookmark check

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY BOOKMARKS ARE WORKING:

1. CLOUDWATCH LOGS — Check row counts per run
   → Run 1: "Rows processed: 300,000" (3 files × 100K)
   → Run 2: "Rows processed: 100,000" (1 new file)
   → Run 3: "Rows processed: 0" (no new files)
   → If Run 2 shows 400,000 → bookmark NOT working

2. GLUE CONSOLE — Check Job Bookmark state
   → Glue → Jobs → Select Job → Job Bookmark tab
   → Shows: last run ID, bookmark state, timestamp
   → If state is empty: bookmark never committed

3. API — Get bookmark programmatically
   → glue.get_job_bookmark(JobName="job-name")
   → Returns bookmark state object
   → Check: JobBookmarkEntry.RunId matches last successful run

4. S3 OUTPUT — Check output file timestamps
   → Each run should produce files with current timestamp
   → If old timestamps appear: job reprocessed old data
   → ls -la on output prefix shows when files were written

5. COST — Check Glue job billing
   → CloudWatch → Glue → DPU-seconds per run
   → With bookmark: DPU-seconds should be roughly constant
   → Without: DPU-seconds grows linearly with data volume

6. CUSTOM WATERMARK — Check watermark file/record
   → DynamoDB: scan watermark table → see last_processed value
   → S3: read watermark.json → see last_processed_timestamp
   → Compare with source data: are we caught up?

🔑 KEY TAKEAWAY

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ Stateless jobs re-process everything — costs grow linearly with data
→ Three approaches to incremental: Bookmarks, Watermarks, Partition Filters
→ Job Bookmarks: zero-code, tracks by S3 file timestamp, needs transformation_ctx
→ Custom Watermark: full control, visible state, works with any source
→ Partition Filter: most explicit, scheduler passes date parameter
→ Bookmark ONLY commits on job.commit() — failure = retry same files
→ Bookmarks need DynamicFrame API — spark.read() bypasses bookmarks
→ MaxConcurrentRuns=1 is MANDATORY for bookmarked jobs
→ APPEND mode required for output — OVERWRITE destroys previous data
→ Watermark update must happen AFTER successful write (never before)
→ Reset capability is essential — schema changes require full reprocess
→ Cost impact: 10x-100x savings as data accumulates over months
