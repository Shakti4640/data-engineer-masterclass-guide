🟡 HOW TO APPROACH IT

Step 1: Understand the export patterns needed
→ Pattern A: Full table export (all rows from a table)
→ Pattern B: Filtered export (WHERE status = 'completed')
→ Pattern C: Aggregated export (GROUP BY summaries)
→ Pattern D: Date-partitioned export (today's data only)
→ All patterns share: SELECT → CSV → S3

Step 2: Understand S3 key structure for exports
→ Raw dump: exports/{table}/data.csv (overwritten daily — BAD)
→ Date-stamped: exports/{table}/{table}_20250115.csv (better)
→ Hive-partitioned: exports/{table}/year=2025/month=01/day=15/data.csv (best)
→ Hive partitioning enables Athena partition discovery (Project 26)

Step 3: Understand streaming vs buffered approach
→ Buffered: SELECT all → write entire CSV → upload file
→ Streaming: SELECT in chunks → write chunks → upload when done
→ For <1M rows: buffered is fine (fits in memory)
→ For >10M rows: streaming with server-side cursor needed

Step 4: Understand the data type handling
→ MariaDB DECIMAL → Python Decimal → CSV string "29.99"
→ MariaDB DATE → Python datetime.date → CSV string "2025-01-15"
→ MariaDB NULL → Python None → CSV empty string ""
→ MariaDB BOOLEAN → Python 0/1 → CSV "true"/"false"
→ Must handle each type explicitly for clean CSV output

Step 5: Plan implementation
→ Connect to MariaDB (reuse Project 19 pattern)
→ Execute SELECT query (parameterized)
→ Fetch results in batches (cursor.fetchmany)
→ Write CSV with proper headers and type handling
→ Upload to S3 with organized key structure
→ Log export metadata
→ Clean up temp files

🔵 WHAT COULD BE DONE

Option A: pymysql SELECT → csv.writer → boto3 upload (programmatic)
→ Full control over queries, transformations, S3 key structure
→ Handles multiple export patterns in one script
→ ✅ WE USE THIS — most flexible, interview-relevant

Option B: mysqldump → gzip → S3 upload
→ Native MariaDB backup tool
→ Outputs SQL (INSERT statements) or CSV
→ Fast but no transformation capability
→ Good for: disaster recovery backups, not team-specific exports

Option C: SELECT INTO OUTFILE (server-side export)
→ MariaDB writes CSV directly to server filesystem
→ Fastest possible export method
→ BUT: writes to SERVER's filesystem, not client's
→ Requires FILE privilege (security risk)
→ Cannot write directly to S3

Option D: Glue JDBC read → S3 write (Project 31 reverse)
→ Distributed Spark reads from MariaDB via JDBC
→ Writes Parquet/CSV to S3
→ Overkill for <1M rows, great for 100M+

Option E: AWS DMS (continuous replication)
→ Streams changes from MariaDB to S3 in near-real-time
→ Covered conceptually in Project 79
→ Overkill for daily batch exports

DECISION: Option A — pymysql + csv + boto3
→ Reuses connection pattern from Project 19 (no new dependencies)
→ Full control over export format and S3 structure
→ Extensible to any query pattern

✅ WHAT SHOULD DO

1. Use Hive-style partition keys for Athena compatibility
   → year=2025/month=01/day=15/data.csv
   → Athena (Project 26) can auto-discover these partitions
   → MSCK REPAIR TABLE or partition projection reads them
   → Future-proofing: no rework when Athena team needs the data

2. Use server-side cursor for large exports
   → Default pymysql: fetches ALL rows into memory at once
   → 10M rows × 500 bytes = 5 GB memory → OOM crash
   → Server-side cursor: pymysql.cursors.SSCursor
   → Fetches rows in chunks → constant memory usage
   → Critical for tables > 1M rows

3. Write CSV with explicit quoting for safety
   → csv.QUOTE_ALL or csv.QUOTE_NONNUMERIC
   → Handles: commas in city names, quotes in product names
   → Without quoting: "New York, NY" becomes two columns

4. Include header row in every CSV
   → Downstream consumers (Athena, pandas, Excel) expect headers
   → Athena: skip.header.line.count = 1 (Project 16 lesson)
   → Self-documenting: anyone can open file and understand columns

5. Add export metadata to S3 object tags
   → source=mariadb, table=orders, export_date=2025-01-15
   → row_count=200000, query_hash=abc123
   → Enables: S3 inventory queries, cost allocation, lineage tracking

6. Compress large exports before upload
   → 200K rows CSV: ~50 MB uncompressed → ~8 MB gzipped
   → S3 storage savings: 85%
   → Athena reads .csv.gz natively (no decompression needed)
   → Trade-off: slightly slower upload (compression CPU) vs less storage

7. Verify export row count matches query count
   → Execute SELECT COUNT(*) with same WHERE clause
   → Compare with rows written to CSV
   → Mismatch → data changed during export (concurrent writes)
   → For critical exports: use READ COMMITTED or snapshot isolation

❌ WHAT SHOULD NOT DO

1. DO NOT SELECT * for exports — always list columns explicitly
   → Schema changes add new columns → downstream parsers break
   → Column order changes → data misalignment
   → SELECT col1, col2, col3 → always same output regardless of schema changes

2. DO NOT export without a WHERE clause on large tables
   → Full table scan on 100M row table → locks rows, slow, memory-heavy
   → Always filter: WHERE order_date = CURDATE() for daily exports
   → Or: use date range for the specific period needed

3. DO NOT write export directly to S3 (stream upload)
   → S3 PutObject requires Content-Length upfront (unless multipart)
   → Write to local temp file first → know exact size → upload
   → Multipart streaming is complex and error-prone for CSV

4. DO NOT use same S3 key for daily exports (overwriting)
   → exports/orders/orders.csv overwritten daily → no history
   → If today's export is corrupted → yesterday's is lost
   → Always include date in key: exports/orders/orders_20250115.csv
   → Or Hive partitions: exports/orders/year=2025/month=01/day=15/

5. DO NOT export sensitive data without access controls
   → Customer emails, addresses → PII (Personally Identifiable Information)
   → Ensure S3 prefix has restricted IAM policies (Project 3)
   → Consider: masking email as c***@example.com in exports
   → Encryption at rest (SSE-S3 minimum, SSE-KMS for sensitive)

6. DO NOT ignore timezone handling
   → MariaDB stores datetime in server timezone (usually UTC)
   → Python datetime objects may be timezone-naive
   → CSV output should specify timezone or always use UTC
   → Inconsistent timezones → "off by one day" errors in daily exports

7. DO NOT leave temp files on disk
   → /tmp fills up on small EC2 instances
   → Stale CSV files from failed exports accumulate
   → Always: try/finally with os.remove()

🚫 WHEN NOT TO USE THIS APPROACH

1. Real-time data sync (changes reflected within seconds)
   → Batch export = minutes/hours delay
   → Use: DMS CDC replication, Debezium, binlog streaming

2. Exporting 100M+ rows daily
   → Single pymysql connection = single-threaded
   → GIL-limited Python CSV writing
   → Use: Glue JDBC (distributed Spark), or parallel chunked exports

3. When destination needs Parquet (not CSV)
   → Python csv module writes CSV only
   → Converting CSV→Parquet in Python (pyarrow) adds complexity
   → Use: Glue ETL which writes Parquet natively (Project 25)

4. When export needs to JOIN across many tables
   → Complex JOINs in MariaDB → heavy query → locks → app slowdown
   → Better: replicate to read replica → export from replica
   → Or: use Glue to read multiple tables and join in Spark

5. When data must never leave the database
   → Compliance: some regulations prohibit data export
   → Use: Athena Federated Query (Project 46) to query in-place
   → Or: Redshift Spectrum with proper access controls

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

┌──────────────────────────────────────────────────────────────────────────────┐
│           MARIADB EXPORT + S3 UPLOAD — INTERNAL MECHANICS                    │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  PHASE 1: QUERY EXECUTION IN MARIADB                                        │
│  ─────────────────────────────────────                                       │
│                                                                              │
│  DEFAULT CURSOR (pymysql.cursors.Cursor):                                    │
│  ┌────────────────────────────────────────────────────────────────┐          │
│  │ cursor.execute("SELECT * FROM orders WHERE order_date = %s")  │          │
│  │                                                                │          │
│  │ 1. SQL sent to MariaDB server over TCP                        │          │
│  │ 2. Server parses, optimizes, executes query                   │          │
│  │ 3. Server streams ALL result rows to client                   │          │
│  │ 4. pymysql buffers ENTIRE result set in Python memory         │          │
│  │ 5. cursor.fetchall() returns list of tuples from buffer       │          │
│  │                                                                │          │
│  │ Memory: O(n) where n = total rows                             │          │
│  │ 200K rows × 500 bytes = 100 MB in Python memory              │          │
│  │ 10M rows × 500 bytes = 5 GB → OOM on t3.medium (4 GB RAM)   │          │
│  └────────────────────────────────────────────────────────────────┘          │
│                                                                              │
│  SERVER-SIDE CURSOR (pymysql.cursors.SSCursor):                              │
│  ┌────────────────────────────────────────────────────────────────┐          │
│  │ cursor = conn.cursor(pymysql.cursors.SSCursor)                │          │
│  │ cursor.execute("SELECT * FROM orders")                        │          │
│  │                                                                │          │
│  │ 1. SQL sent to MariaDB server                                 │          │
│  │ 2. Server executes query but HOLDS result on server side      │          │
│  │ 3. cursor.fetchmany(1000) fetches 1000 rows at a time        │          │
│  │ 4. Only 1000 rows in Python memory at any time               │          │
│  │ 5. Next fetchmany() gets next 1000 from server               │          │
│  │                                                                │          │
│  │ Memory: O(batch_size) — constant regardless of total rows     │          │
│  │ 10M rows with batch=1000 → 500 KB in Python memory           │          │
│  │                                                                │          │
│  │ TRADE-OFF:                                                    │          │
│  │ → Connection MUST stay open during entire fetch               │          │
│  │ → Cannot run other queries on same connection                 │          │
│  │ → If connection drops mid-fetch → partial export              │          │
│  └────────────────────────────────────────────────────────────────┘          │
│                                                                              │
│                                                                              │
│  PHASE 2: DATA TYPE CONVERSION (MariaDB → Python → CSV)                     │
│  ────────────────────────────────────────────────────────                     │
│                                                                              │
│  ┌───────────────────┬────────────────────┬──────────────────────┐           │
│  │ MariaDB Type      │ Python Type        │ CSV Output           │           │
│  ├───────────────────┼────────────────────┼──────────────────────┤           │
│  │ VARCHAR(100)      │ str                │ "John Doe"           │           │
│  │ INT               │ int                │ "42"                 │           │
│  │ DECIMAL(10,2)     │ decimal.Decimal    │ "29.99"              │           │
│  │ FLOAT/DOUBLE      │ float              │ "29.990000000001"    │           │
│  │ DATE              │ datetime.date      │ "2025-01-15"         │           │
│  │ DATETIME          │ datetime.datetime  │ "2025-01-15 00:03:12"│           │
│  │ BOOLEAN/TINYINT(1)│ int (0 or 1)      │ "true" or "false"    │           │
│  │ NULL              │ None               │ "" (empty string)    │           │
│  │ BLOB              │ bytes              │ (not exported)       │           │

│                                                                              │

│  → DECIMAL(10,2) → Python Decimal("29.99") → CSV "29.99" ✅                │
│  → FLOAT → Python float 29.990000000001 → CSV "29.990000000001" ❌          │
│  → Always use DECIMAL for money columns in MariaDB                          │
│  → If stuck with FLOAT: round() in Python before CSV write                  │
│                                                                              │
│                                                                              │
│  PHASE 3: CSV WRITING MECHANICS                                              │
│  ───────────────────────────────                                             │
│                                                                              │
│  csv.writer with QUOTE_NONNUMERIC:                                           │
│  → Quotes all non-numeric fields: "John Doe","New York, NY"                 │
│  → Leaves numbers unquoted: 42,29.99                                        │
│  → Handles embedded commas: "Portland, OR" stays as one field               │
│  → Handles embedded quotes: 'He said "hello"' → '"He said ""hello"""'       │
│  → Handles newlines in fields: enclosed in quotes                           │
│                                                                              │
│  csv.writer with QUOTE_ALL:                                                  │
│  → Quotes EVERY field: "42","29.99","John Doe"                              │

│                                                                              │
│  Standard key:                                                               │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Export has 199,998 rows but table has 200,000"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: concurrent INSERT during export
→ Transaction isolation: default READ COMMITTED sees committed changes
→ Row inserted AFTER cursor started but BEFORE cursor reaches it
→ Fix: use REPEATABLE READ or snapshot isolation
→ Or: accept minor discrepancy for non-critical exports
→ Or: export from read replica with replication lag

PITFALL 2: "CSV has garbled characters: cafÃ© instead of café"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: MariaDB charset utf8mb4 but file opened without encoding
→ Fix: open(filepath, 'w', encoding='utf-8', newline='')
→ The newline='' is CRITICAL for csv.writer (prevents double newlines on Windows)

PITFALL 3: "Decimal values have trailing zeros: 29.990000"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Python Decimal("29.99") → str → "29.99" (correct)
→ BUT: Python float 29.99 → str → "29.99" (lucky)
→ AND: Python float 29.990000000001 → str → "29.990000000001" (bad)
→ Fix: use DECIMAL type in MariaDB (not FLOAT)
→ Fix: format in Python: f"{value:.2f}" for money columns

PITFALL 4: "OOM killed during large export"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: default cursor loads ALL rows into memory
→ 10M rows → 5 GB → EC2 t3.medium has 4 GB → OOM kill
→ Fix: use SSCursor (server-side cursor) + fetchmany()
→ Fix: add WHERE clause to limit rows per export batch

PITFALL 5: "Export file is 0 bytes"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause A: query returned 0 rows (date filter matched nothing)
→ Cause B: file handle not flushed before upload
→ Fix A: check row count before writing → skip if 0
→ Fix B: always close file handle before upload (with statement)

PITFALL 6: "S3 upload succeeds but Athena shows wrong columns"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: CSV column order doesn't match Athena table DDL
→ CSV is positional — column 1 in file maps to column 1 in DDL
→ If MariaDB SELECT order differs from Athena CREATE TABLE order → misalignment
→ Fix: always SELECT columns in exact same order as Athena DDL
→ Fix: use explicit column list (not SELECT *)

PITFALL 7: "Export takes 30 minutes — MariaDB is slow during"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: full table scan during export holds read locks
→ Application queries compete for same resources
→ Fix: schedule exports during low-traffic hours (2 AM)
→ Fix: export from read replica (not primary)
→ Fix: add appropriate indexes for WHERE clause in export query

📈 SCALABILITY CONSIDERATIONS

DATA VOLUME:
━━━━━━━━━━━━
→ 5K rows (products): < 1 second export, < 100 KB file
→ 50K rows (customers): 2-5 seconds, ~5 MB file
→ 200K rows (orders): 10-30 seconds, ~50 MB file
→ 1M rows: 1-3 minutes, ~250 MB file (consider gzip)
→ 10M rows: 10-30 minutes, ~2.5 GB file (MUST use SSCursor + gzip)
→ 100M rows: 1-3 hours → consider Glue JDBC (Project 31)

FILE SIZE & COMPRESSION:
━━━━━━━━━━━━━━━━━━━━━━━━
→ CSV compression ratio: typically 5:1 to 8:1
→ 50 MB CSV → ~8 MB gzipped
→ S3 storage: $0.023/GB → 50 MB = $0.00115/month
→ Gzipped: $0.000184/month (84% savings)
→ Athena reads .csv.gz natively → also saves scan cost

EXPORT FREQUENCY:
━━━━━━━━━━━━━━━━━
→ Daily export of 200K rows: standard pattern (this project)
→ Hourly export: possible but increases file count
→ Real-time: wrong tool → use DMS CDC or binlog streaming
→ Weekly aggregates: lighter on MariaDB, sufficient for reporting

PARALLEL EXPORTS:
━━━━━━━━━━━━━━━━━
→ Export 3 tables sequentially: 1 connection, simple
→ Export 3 tables in parallel: 3 connections, 3x faster
→ Python: threading or multiprocessing (GIL limits CPU, not I/O)

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ MariaDB → CSV → S3 is the reverse of Project 19 (S3 → CSV → MariaDB)
→ SSCursor (server-side cursor) prevents OOM on large exports
→ fetchmany(batch_size) gives constant memory regardless of table size
→ Hive-style partition keys (year=.../month=.../day=...) enable Athena auto-discovery
→ Data type handling is CRITICAL: Decimal→str clean, float→str dirty
→ csv.QUOTE_NONNUMERIC handles embedded commas and quotes safely
→ Always verify: row count from COUNT(*) must match CSV lines written
→ Export configs as dict → extensible without code changes
→ Backfill utility → catch up historical dates without manual effort
→ Cron scheduling → reliable nightly automation
