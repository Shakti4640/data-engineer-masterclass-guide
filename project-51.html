🟡 HOW TO APPROACH IT

Step 1: Understand the difference from Project 14
→ Project 14: Fan-out = same message to ALL subscribers
→ Project 51: Filtered fan-out = SPECIFIC messages to SPECIFIC subscribers
→ Filtering happens INSIDE SNS before delivery — not in consumer code

Step 2: Understand SNS Message Attributes
→ SNS messages have TWO parts:
   a) Message Body — the actual payload (JSON string)
   b) Message Attributes — key-value metadata OUTSIDE the body
→ Filter policies operate on Message Attributes (default)
→ Since 2022: can also filter on Message Body (payload-based filtering)

Step 3: Design the attribute schema
→ We need attributes that SNS can filter on
→ Key attributes:
   - event_type: "etl_event" | "order_event" | "alert_event" | "data_quality_event"
   - severity: "info" | "warning" | "critical"
   - source_system: "glue" | "order_mgmt" | "monitoring" | "data_quality"
   - environment: "prod" | "staging" | "dev"

Step 4: Design filter policies per queue
→ ETL Queue:        event_type = ["etl_event"]
→ Order Queue:      event_type = ["order_event"]
→ Alert Queue:      event_type = ["alert_event", "data_quality_event"]
→ Audit Queue:      NO filter (receives everything)

Step 5: Understand filter policy scope
→ FilterPolicyScope = "MessageAttributes" (default, faster)
→ FilterPolicyScope = "MessageBody" (newer, filters on JSON body)
→ We use MessageAttributes — standard, well-supported

🔵 WHAT COULD BE DONE

Option A: Application-level filtering (current — Project 14)
→ All queues get all messages
→ Consumer code checks: if event_type != "etl": skip
→ PROBLEMS: wasted SQS cost, wasted compute, added latency

Option B: Separate SNS topics per event type
→ quickcart-etl-events topic → ETL queue
→ quickcart-order-events topic → Order queue
→ PROBLEMS: publishers must know which topic to use
→ Adding new event types = new topics = more infra to manage
→ Cross-cutting concerns (severity=critical) require dual publishing

Option C: SNS Message Filtering (this project) ✅
→ Single topic, multiple subscriptions with filter policies
→ SNS evaluates filters server-side before delivery
→ Publishers don't know/care about subscribers
→ Subscribers declare what they want
→ True pub/sub decoupling

Option D: EventBridge rules (Project 66)
→ More powerful content-based routing
→ But heavier, more expensive at high volume
→ Better for orchestration than simple routing

DECISION: Option C — SNS Message Filtering
→ Reason: simplest, cheapest, zero-code routing
→ Filtering is FREE — no extra SNS charge
→ Perfect fit for attribute-based routing

✅ WHAT SHOULD DO

1. Define a CONSISTENT attribute schema across ALL publishers
   → Document: event_type, severity, source_system, environment
   → Enforce via shared Python module / schema definition
   → Inconsistent attributes = messages slip through filters

2. Always include a "catch-all" subscriber (Audit Queue)
   → No filter policy = receives everything
   → Guarantees no message is silently lost
   → Essential for compliance and debugging

3. Use STRING data type for attributes (most compatible)
   → SNS supports: String, Number, Binary, String.Array
   → Filter policies work best with String values
   → Avoid Binary — not filterable

4. Test filter policies BEFORE production deployment
   → Publish test messages with various attribute combinations
   → Verify each queue receives exactly what's expected
   → Verify queues do NOT receive what they shouldn't

5. Use filter policy with OR logic for multi-type subscribers
   → Alert Queue needs: alert_event OR data_quality_event
   → FilterPolicy: {"event_type": ["alert_event", "data_quality_event"]}
   → Array = OR condition

6. Set up DLQ on each filtered queue (from Project 50)
   → Failed processing should not cause message loss
   → DLQ per queue — not one shared DLQ

7. Monitor FilteredMessagesDropped metric
   → CloudWatch: NumberOfNotificationsFilteredOut-NoMessageAttributes
   → High count = publishers forgetting to set attributes

❌ WHAT SHOULD NOT DO

1. DO NOT filter on message body by default
   → Body-based filtering is slower (SNS must parse JSON body)
   → Use MessageAttributes — purpose-built for this
   → Body filtering only when you don't control publisher code

2. DO NOT create overly complex filter policies
   → Max filter policy size: 256 KB
   → Max 5 attribute names per policy
   → Max 150 values across all attribute names
   → Complex policies = hard to debug, easy to misconfigure

3. DO NOT assume unmatched messages go to a DLQ
   → If no subscription matches → message is SILENTLY DROPPED by SNS
   → SNS does NOT have a built-in "unmatched message" DLQ
   → That's why the catch-all Audit Queue is critical

4. DO NOT mix filtered and unfiltered subscriptions carelessly
   → A subscription with NO filter policy = receives EVERYTHING
   → If you add a filter policy with empty object {} = receives NOTHING
   → Empty filter {} ≠ no filter

5. DO NOT forget to set MessageAttributes when publishing
   → Message published WITHOUT attributes:
     - Subscriptions with filters → WILL NOT receive it
     - Subscriptions without filters → WILL receive it
   → Silent data loss if publisher forgets attributes

6. DO NOT use Number type for enum-like values
   → event_type is categorical → use String
   → Number type only for: price > 100, quantity >= 5 (numeric comparisons)

7. DO NOT hardcode filter policies in console
   → Use Infrastructure as Code (boto3, CloudFormation, CDK)
   → Console changes are not tracked, not reproducible

🚫 WHEN NOT TO USE SNS MESSAGE FILTERING

1. When you need content-based transformation (not just routing)
   → SNS filters: YES/NO delivery decision only
   → Cannot modify message before delivery
   → Use: EventBridge input transformer, or Lambda between SNS and SQS

2. When filter logic requires external data lookups
   → "Route to Queue A if customer is in VIP table in DynamoDB"
   → SNS cannot query databases
   → Use: Lambda subscriber that does lookup then forwards

3. When you have >150 possible values for one attribute
   → SNS filter policy hard limit: 150 values across all attributes combined
   → Use: EventBridge or application-level routing

4. When routing must be dynamic (changes per message)
   → SNS filter policies are STATIC per subscription
   → Changing filter = API call to update subscription
   → Use: Lambda-based routing for dynamic decisions

5. When message ordering matters across filtered queues
   → Standard SNS + Standard SQS: no ordering guarantee
   → Even FIFO: ordering is per-message-group, not cross-queue
   → Use: single queue + application routing if strict order needed

│                                                                             │

│                                                                             │
│    Subscription 2: Order Queue                                              │
│    FilterPolicy: {"event_type": ["order_event"]}                            │
│    Message has:   event_type = "order_event"                                │
│    Evaluation:    "order_event" IN ["order_event"]? → YES                   │
│    Result:        ✅ DELIVER to Order Queue                                  │
│                                                                             │
│    Subscription 3: Alert Queue                                              │
│    FilterPolicy: {"event_type": ["alert_event", "data_quality_event"]}      │
│    Message has:   event_type = "order_event"                                │
│    Evaluation:    "order_event" IN ["alert_event","data_quality_event"]? → NO│
│    Result:        ❌ SKIP                                                    │
│                                                                             │
│    Subscription 4: Audit Queue                                              │
│    FilterPolicy: (none)                                                     │
│    Evaluation:    No filter = always match                                  │
│    Result:        ✅ DELIVER to Audit Queue                                  │
│                                                                             │
│  STEP 3: FILTER POLICY EVALUATION RULES                                     │
│  ══════════════════════════════════════                                      │
│                                                                             │
│  RULE 1: String exact match                                                 │
│  Policy:  {"event_type": ["order_event"]}                                   │
│  Matches: event_type = "order_event"                                        │
│  Logic:   value IN [list]                                                   │
│                                                                             │
│  RULE 2: String prefix match                                                │
│  Policy:  {"event_type": [{"prefix": "etl_"}]}                              │
│  Matches: event_type = "etl_event", "etl_started", "etl_anything"           │
│  Logic:   value.startsWith(prefix)                                          │
│                                                                             │
│  RULE 3: Numeric exact match                                                │
│  Policy:  {"price": [{"numeric": ["=", 100]}]}                              │
│  Matches: price = 100                                                       │
│                                                                             │
│  RULE 4: Numeric range                                                      │
│  Policy:  {"price": [{"numeric": [">", 0, "<=", 100]}]}                     │
│  Matches: 0 < price <= 100                                                  │
│                                                                             │
│  RULE 5: Anything-but (exclusion)                                           │
│  Policy:  {"event_type": [{"anything-but": ["test_event"]}]}                │
│  Matches: any event_type EXCEPT "test_event"                                │
│                                                                             │
│  RULE 6: Exists / not-exists                                                │
│  Policy:  {"severity": [{"exists": true}]}                                  │
│  Matches: any message that HAS the severity attribute (any value)           │
│  Policy:  {"severity": [{"exists": false}]}                                 │
│  Matches: any message MISSING the severity attribute                        │
│                                                                             │
│  RULE 7: Multiple attributes = AND logic                                    │
│  Policy:  {"event_type": ["etl_event"], "severity": ["critical"]}           │
│  Matches: event_type="etl_event" AND severity="critical"                    │
│  Logic:   ALL attributes must match                                         │
│                                                                             │
│  RULE 8: Multiple values per attribute = OR logic                           │
│  Policy:  {"event_type": ["alert_event", "data_quality_event"]}             │
│  Matches: event_type="alert_event" OR event_type="data_quality_event"       │
│  Logic:   ANY value in the array can match                                  │
│                                                                             │
│  COMBINED: AND between attributes, OR within attribute values               │
│                                                                             │
│  STEP 4: DELIVERY MECHANICS                                                 │
│  ════════════════════════════                                                │
│  → Matched subscriptions: SNS pushes message to SQS via internal API        │
│  → SQS receives: message body + message attributes + SNS metadata           │
│  → Unmatched subscriptions: message SILENTLY not delivered                   │
│  → No error, no log, no DLQ for unmatched (by design)                       │
│  → CloudWatch metric: NumberOfNotificationsFilteredOut                       │
│                                                                             │
│  STEP 5: FILTER POLICY SCOPE (since 2022)                                   │
│  ════════════════════════════════════════                                    │
│  FilterPolicyScope = "MessageAttributes" (default)                          │
│  → Filters on MessageAttributes dictionary                                  │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Messages disappearing — queue receives nothing"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Publisher publishes WITHOUT MessageAttributes
→ Filter policy requires event_type = "etl_event"
→ Message has NO event_type attribute → filter does NOT match
→ SNS silently drops delivery to this subscription
→ Fix: ALWAYS set MessageAttributes when publishing
→ Detection: monitor NumberOfNotificationsFilteredOut-NoMessageAttributes

PITFALL 2: "Filter policy set but queue still gets everything"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: filter policy was set on wrong subscription ARN
→ Or: filter policy was set but not confirmed yet
→ Debug: aws sns get-subscription-attributes --subscription-arn <arn>
→ Verify: "FilterPolicy" key exists in output

PITFALL 3: "Data type mismatch — String vs Number"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Publisher sets: severity = {"DataType": "Number", "StringValue": "1"}
→ Filter policy: {"severity": ["1"]}  (string match)
→ DOES NOT MATCH — Number "1" ≠ String "1"
→ Fix: use consistent DataType across publisher and filter
→ Best practice: use String for everything except numeric ranges

PITFALL 4: "Empty filter policy {} means no filter"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ WRONG: {} means "filter that matches nothing" → receives NOTHING
→ No filter at all = subscription attribute "FilterPolicy" absent
→ This is a critical distinction
→ To remove filter: delete the FilterPolicy attribute entirely

PITFALL 5: "Cross-account filtered subscription not receiving"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ From Project 59: cross-account SNS→SQS needs queue policy
→ Filter policies work the same cross-account
→ But: SQS queue policy must allow sns:SendMessage from topic ARN
→ If policy is missing: even matched messages fail silently

PITFALL 6: "Raw message delivery breaks attribute forwarding"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ From Project 14: RawMessageDelivery = true
→ With raw delivery: SQS receives message body directly
→ MessageAttributes ARE still forwarded as SQS message attributes
→ But: SNS metadata envelope is stripped
→ Filter still works — filtering happens BEFORE delivery format

PITFALL 7: "Attribute name is case-sensitive"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Publisher sets: "Event_Type" = "etl_event"
→ Filter policy: {"event_type": ["etl_event"]}
→ NO MATCH — "Event_Type" ≠ "event_type"
→ Fix: standardize on lowercase_snake_case everywhere

📈 SCALABILITY CONSIDERATIONS

CURRENT STATE:
→ 200K messages/day through single topic
→ 4 subscriptions with filters
→ SNS handles this trivially

AT 10M MESSAGES/DAY:
→ SNS soft limit: 30,000 publishes/second (adjustable)
→ 10M/day = ~115/second average → well within limits
→ Filter evaluation: <1ms × 4 subscriptions = <4ms overhead
→ No scaling concern

AT 100M MESSAGES/DAY:
→ ~1,157 publishes/second average
→ Peak: maybe 5,000/second
→ Still within default limits
→ Consider: request limit increase if bursts exceed 30K/sec

SUBSCRIPTION SCALING:
→ Max subscriptions per topic: 12,500,000 (soft limit)
→ Each subscription can have its own filter policy
→ Filter evaluation is O(n) where n = number of subscriptions
→ At 100+ subscriptions: consider separate topics by domain

FILTER POLICY LIMITS:
→ Max policy size: 256 KB per subscription
→ Max attribute names in policy: 5
→ Max total values across all attributes: 150
→ Max nesting depth: attribute → condition → value (3 levels)

COST IMPACT:
→ Without filtering (Project 14 pattern):
  200K msgs × 4 queues = 800K SQS deliveries
  SQS cost: 800K × $0.40/million = $0.32/day

→ With filtering (this project):
  200K msgs × avg 2 matching queues = 400K SQS deliveries
  SQS cost: 400K × $0.40/million = $0.16/day
  
→ Savings: 50% SQS cost + eliminated wasted consumer compute
→ At scale (100M msgs/day): savings = thousands of dollars/month

FIFO VARIANT:
→ SNS FIFO topic → SQS FIFO queue with filters
→ Works identically but adds ordering + dedup
→ FIFO throughput: 300 publishes/sec (with batching: 3,000/sec)
→ Use only when ordering matters (Project 13 distinction applies)

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY FILTERS ARE WORKING:

1. Subscription-level verification:
   → aws sns get-subscription-attributes --subscription-arn <arn>
   → Check "FilterPolicy" and "FilterPolicyScope" values
   → Confirm policy JSON matches your intent

2. Publish test messages with known attributes:
   → Publish event_type="etl_event" → verify ONLY ETL Queue receives
   → Publish event_type="order_event" → verify ONLY Order Queue receives
   → Publish with NO attributes → verify ONLY Audit Queue receives

3. CloudWatch metrics per subscription:
   → NumberOfMessagesPublished (topic-level)
   → NumberOfNotificationsDelivered (per subscription)
   → NumberOfNotificationsFilteredOut (per subscription)
   → NumberOfNotificationsFilteredOut-NoMessageAttributes
   → NumberOfNotificationsFilteredOut-InvalidAttributes

4. SQS queue depth monitoring:
   → ApproximateNumberOfMessages on each queue
   → After filtering: each queue should show LOWER depth than total published
   → Audit Queue depth ≈ total published (no filter)

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ SNS Message Filtering routes messages at BROKER level, not application level
→ Filter policies use MessageAttributes (metadata) not message body (by default)
→ Multiple values in one attribute = OR logic
→ Multiple attributes = AND logic
→ Operators: exact, prefix, suffix, anything-but, numeric range, exists
→ No filter policy = receives everything (catch-all)
→ Empty filter policy {} = receives NOTHING (dangerous!)
→ Filtering is FREE — saves SQS cost and consumer compute
→ Always include a catch-all subscriber for compliance/debugging
→ Monitor NumberOfNotificationsFilteredOut-NoMessageAttributes
→ Body-based filtering available when you can't control publisher attributes
