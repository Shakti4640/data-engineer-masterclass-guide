🟡 HOW TO APPROACH IT

Step 1: Understand RTO and RPO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

RTO = Recovery Time Objective = 4 hours
→ "How long can we be DOWN before it's unacceptable?"
→ From moment of failure → platform operational in DR region
→ Includes: detection (15 min) + decision (15 min) + failover (3.5 hours)

RPO = Recovery Point Objective = 1 hour
→ "How much DATA can we lose?"
→ Maximum acceptable data loss = 1 hour of data
→ If outage at 2:14 AM, we must have data up to at least 1:14 AM
→ S3 replication lag must be < 1 hour
→ Redshift snapshots must be < 1 hour old
→ MariaDB replication lag must be < 1 hour

Step 2: Classify Components by DR Strategy
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

TIER 1: DATA (must replicate continuously)
→ S3: Cross-Region Replication (CRR) — continuous
→ Redshift: Cross-Region Snapshots — every 1 hour
→ MariaDB: Read Replica in us-west-2 — continuous
→ Glue Catalog: Export/Import — every 1 hour

TIER 2: COMPUTE (recreate on demand)
→ Redshift Serverless: create workgroup in DR region on failover
→ Kinesis: create new stream in DR region on failover
→ Firehose: create new delivery stream on failover
→ Glue Jobs: deploy from code (IaC) on failover
→ EC2: launch from AMI on failover

TIER 3: CONFIGURATION (backup and restore)
→ Lake Formation: export grants, import in DR
→ IAM Roles: identical in both regions (global service)
→ Step Functions: deploy from definition JSON
→ EventBridge Rules: deploy from IaC
→ SNS/SQS: create new in DR region

Step 3: Design the DR Architecture
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ACTIVE-PASSIVE model:
→ Primary (us-east-2): full production workload
→ DR (us-west-2): data replicated, compute dormant
→ On failover: spin up compute in DR, point to replicated data
→ On recovery: fail back to primary

WHY NOT ACTIVE-ACTIVE:
→ Active-Active doubles cost (two running Redshift clusters, two Kinesis streams)
→ Conflict resolution for writes is complex
→ QuickCart's $3K/month budget doesn't support it
→ Active-Passive meets 4-hour RTO at fraction of cost

Step 4: Calculate DR Budget
━━━━━━━━━━━━━━━━━━━━━━━━━━
→ S3 CRR: replicated data storage ($0.023/GB × 15 TB) = $345/month
→ S3 CRR: replication requests ($0.005/1000 PUT) ≈ $50/month
→ Redshift snapshots in DR: cross-region snapshot storage ≈ $200/month
→ MariaDB Read Replica (t3.medium): $51/month
→ Glue Catalog backup Lambda: $5/month
→ DR automation (Lambda + Step Functions): $20/month
→ Quarterly DR drill cost: ~$200/drill = $67/month amortized
→ TOTAL: ~$738/month (well within $3,000 budget)

→ Both regions running full production workloads
→ Pro: near-zero RTO (instant failover)
→ Con: 2× cost ($15K+/month for dual Redshift, Kinesis, Glue)
→ Con: write conflict resolution complexity
→ ❌ REJECTED — exceeds budget, over-engineered

OPTION C: Backup-Only (No Replication)
→ Nightly S3 backup, weekly Redshift snapshot
→ Pro: cheapest ($100/month)
→ Con: RPO = 24 hours (unacceptable — lose full day of data)

→ On failover: scale up existing resources
→ Pro: faster RTO than pure passive (2-3 hours)
→ Con: slightly higher cost ($1,200/month)
→ ✅ CONSIDERED — could upgrade to this later

DECISION: Option A now, upgrade to Option D if budget allows

✅ WHAT SHOULD DO

1. S3 CROSS-REGION REPLICATION (CRR) — highest priority
   → 15 TB of data lake is the MOST critical asset
   → CRR replicates every new object within 15 minutes (SLA: 99.99% in 15 min)
   → Enable S3 Replication Time Control (RTC) for guaranteed 15-min SLA
   → Replicate: orders-data-prod → orders-data-dr (us-west-2)

2. REDSHIFT CROSS-REGION SNAPSHOTS — every 1 hour
   → Automated snapshot schedule: every 1 hour
   → Copy snapshots to us-west-2 automatically
   → Retain 24 snapshots in DR region (24 hours of history)
   → On failover: restore from latest snapshot

3. MARIADB READ REPLICA — continuous replication
   → Launch MariaDB read replica in us-west-2
   → Binary log replication: < 5 second lag
   → On failover: promote read replica to primary
   → Application reconnects to DR MariaDB

4. GLUE CATALOG BACKUP — every 1 hour
   → Lambda function exports all databases, tables, partitions
   → Stores as JSON in DR S3 bucket
   → On failover: Lambda imports catalog into DR region
   → Include: column tags, table parameters, partition info

5. INFRASTRUCTURE AS CODE — stored in S3 (replicated)
   → All CloudFormation/CDK templates in S3 (auto-replicated via CRR)
   → Kinesis stream definitions
   → Firehose configurations
   → Step Functions state machine definitions
   → EventBridge rules
   → Glue job scripts
   → On failover: deploy stack in DR region

6. FAILOVER RUNBOOK — automated + manual steps
   → Step Functions state machine that orchestrates failover
   → Human approval gate before executing
   → Automated steps: restore Redshift, import catalog, deploy infra
   → Manual steps: promote MariaDB, update DNS, verify

7. QUARTERLY DR DRILLS
   → Actually fail over to DR region
   → Run full pipeline end-to-end
   → Verify data integrity
   → Measure actual RTO/RPO
   → Document gaps and fix

❌ WHAT SHOULD NOT DO

1. DO NOT replicate Kinesis streams cross-region
   → Kinesis is regional — no native cross-region replication
   → During outage: events are LOST (buffered in producer's retry)
   → Accept: real-time clickstream has RPO = outage duration
   → Mitigate: producers retry with exponential backoff
   → On recovery: producers resend buffered events

2. DO NOT keep DR Redshift running 24/7
   → Idle Redshift Serverless costs $0 (auto-pauses)
   → But provisioned Redshift: $180+/month even idle
   → Use Serverless in DR — create on failover, restore from snapshot

3. DO NOT manually copy Glue Catalog
   → 200+ tables with partitions = impossible to manually recreate
   → Automate with Lambda export/import
   → Test import monthly to verify it works

4. DO NOT skip testing the failover
   → "We have DR" means nothing without tested failover
   → Quarterly drills are mandatory
   → Measure actual RTO — it's always longer than planned

5. DO NOT replicate Bronze layer to DR
   → Bronze = raw data, can be re-extracted from source
   → Only replicate Gold layer (curated, business-critical)
   → Saves 60% of CRR storage cost
   → Silver can be re-derived from Gold if needed

6. DO NOT forget DNS / endpoint management
   → Applications hardcode us-east-2 endpoints
   → Use Route 53 DNS aliases that can be switched
   → Or: use parameter store for endpoint configuration
   → On failover: update parameter → apps reconnect

7. DO NOT assume Lake Formation grants replicate
   → Lake Formation is REGIONAL — grants don't cross regions
   → Must backup and restore grants in DR region
   → Include in catalog backup Lambda
