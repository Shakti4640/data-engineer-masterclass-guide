🔵 WHAT COULD BE DONE

Option A: Glue Data Quality within ETL jobs ✅ (CHOSEN — primary)
→ EvaluateDataQuality transform in Glue ETL script
→ DQDL rules defined inline or referenced from Glue Catalog
→ Actions: pass rows to "good" output, route failures to "bad" output
→ Metrics published to CloudWatch automatically
→ Tightest integration with existing Glue pipeline

Option B: Standalone Glue Data Quality job ✅ (CHOSEN — supplement)
→ Separate Glue job that ONLY validates, doesn't transform
→ Runs BEFORE ETL job in Step Function
→ Pass → trigger ETL; Fail → quarantine + alert
→ Clean separation: validation ≠ transformation

Option C: Lambda-based validation
→ Small Lambda reads S3 file → checks row count, schema, samples
→ Lightweight for simple checks (volume, schema)
→ Can't handle: uniqueness across millions of rows (memory limit)
→ Good for: pre-flight checks before Glue starts

Option D: Redshift-based validation (post-load)
→ SQL queries on staging table before merge
→ SELECT COUNT(DISTINCT pk) vs COUNT(*) → duplicate check
→ Works but: bad data already in Redshift staging
→ Better: catch before it reaches Redshift

DECISION: Layered approach
→ Layer 1: Lambda pre-flight (schema, volume, not empty) — fast, cheap
→ Layer 2: Glue DQ within ETL (uniqueness, validity, completeness) — thorough
→ Layer 3: Redshift post-load SQL (cross-table integrity) — final gate

✅ WHAT SHOULD DO

1. DEFINE rules in DQDL (Data Quality Definition Language)
   → Declarative: "RowCount between 3000 and 8000"
   → Not imperative: no Python code for simple checks
   → DQDL rules stored in Glue Catalog → reusable across jobs
   → Version controlled alongside ETL code

2. SEPARATE "hard fail" vs "soft warn" rules
   → HARD FAIL: stops pipeline, quarantines data, alerts team
     Example: duplicate primary keys, row count 10x normal
   → SOFT WARN: continues pipeline but logs warning
     Example: 2% NULL in optional field, slight volume variance
   → Don't fail pipeline for minor issues → alert and continue

3. QUARANTINE bad data — don't delete it
   → Failed rows → s3://lake/quarantine/{source}/{batch_date}/
   → Includes: original row + rule that failed + timestamp
   → Team can inspect quarantined data → fix source → replay
   → Retention: 90 days → lifecycle to Glacier

4. PUBLISH quality metrics to CloudWatch
   → Metric: DQ/PassRate per source per rule
   → Metric: DQ/FailedRows per source
   → Metric: DQ/RowCount per source (trend)
   → Dashboard: quality scores over time
   → Alarm: pass rate < 95% → alert

5. INTEGRATE quality gate into Step Function (Project 66)
   → After source ETL → Quality Check step
   → Quality passes → proceed to gold layer
   → Quality fails → quarantine + alert + STOP
   → Human approves override → manual retry

6. TEST quality rules against KNOWN bad data
   → Create test files with deliberate issues
   → Run quality checks → verify each rule catches its issue
   → Regression test: new rules don't break existing valid data

7. TRACK quality trends over time
   → Partner feed quality score: 98% → 95% → 92% → ALERT
   → Degrading quality = source system problem
   → Catch BEFORE it becomes a data incident

8. DOCUMENT every rule with business justification
   → Rule: "total_amount > 0"
   → Justification: "Negative amounts indicate refunds which should use status='refunded'"
   → Owner: Data Engineering team
   → Last reviewed: 2025-01-15

❌ WHAT SHOULD NOT DO

1. DO NOT validate only at ingestion — validate at EVERY layer
   → Source file may be valid but transformation introduces errors
   → Silver layer valid but gold aggregation has division-by-zero
   → Quality gates at: source → silver → gold

2. DO NOT block pipeline for trivial issues
   → 0.1% NULL in optional email field → don't stop pipeline
   → Set appropriate thresholds: "Completeness('email') >= 95%"
   → Reserve HARD FAIL for truly dangerous data quality issues

3. DO NOT hardcode thresholds without historical context
   → "Row count must be exactly 5,000" → fails on 5,001 rows
   → Better: "Row count between 3,000 and 8,000" (±60% of historical average)
   → Even better: dynamic thresholds based on rolling 30-day average

4. DO NOT skip quality checks in "emergency" situations
   → "CEO needs the data NOW, skip quality checks"
   → This is exactly how the $5M incident happened
   → Quality checks take < 2 minutes — never worth skipping

5. DO NOT run quality checks AFTER data is in production tables
   → "Check quality on Redshift production table after loading"
   → Bad data already visible to analysts → damage done
   → Check BEFORE merge into production (staging → validate → merge)

6. DO NOT create quality rules without monitoring
   → Rules without dashboards = invisible
   → Rule fails silently → nobody notices → same as no rules
   → Every rule must publish metrics → dashboard → alarm

7. DO NOT let quality rules become stale
   → Business changes: new status codes, price ranges, etc.
   → Rules not updated → false positives or missed issues
   → Quarterly review: update rules with current business context

8. DO NOT quarantine without alerting
   → Quarantine = data hidden from pipeline
   → Without alert: team doesn't know data is missing
   → Dashboard shows $0 revenue (quarantined) → worse than wrong data
   → Always: quarantine + alert + manual review path

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Quality check passes but data is still wrong"
→ Cause: rules don't cover the actual failure mode
→ Duplicate rows pass: uniqueness check was on wrong column
→ Fix: map EVERY past incident to a specific rule
→ If incident can't be caught by existing rules → add new rule

PITFALL 2: "Quality check blocks ALL data for 1 bad row"
→ Cause: rule set to HARD FAIL on any failure
→ 1 bad row out of 5,000 → entire batch rejected
→ Fix: use ROW-LEVEL filtering → quarantine bad rows, pass good ones
→ Or: threshold-based: "Completeness >= 99%" (allows 1% bad)

PITFALL 3: "Quality check takes longer than ETL itself"
→ Cause: Uniqueness check on 10M rows → full data shuffle
→ Multiple CustomSql rules → multiple full scans
→ Fix: prioritize rules by cost → run cheap rules first
→ If cheap rules fail → skip expensive rules (already failing)

PITFALL 4: "Row count check fails after legitimate business growth"
→ Cause: hardcoded threshold "between 3000 and 8000"
→ Business grew: now 12,000 orders/day → rule fails daily
→ Fix: dynamic thresholds based on rolling average
→ Or: percentage-based: "within 200% of 30-day average"

PITFALL 5: "Quarantine bucket fills up, nobody reviews it"
→ Cause: soft-fail rules quarantine data → no alert → accumulates
→ 6 months later: 50 GB of quarantined data nobody looked at
→ Fix: alert on EVERY quarantine event → review SLA (24 hours)
→ Lifecycle: auto-delete quarantine after 90 days

PITFALL 6: "False positive: good data flagged as bad"
→ Cause: rule too strict → legitimate edge cases caught
→ Black Friday: 3x normal volume → row count check fails
→ Fix: separate rules for known high-volume periods
→ Or: wider thresholds with anomaly detection instead of hard limits

PITFALL 7: "Quality check passes on sample but fails on full data"
→ Cause: sampling-based checks miss rare issues
→ 1 duplicate in 1M rows → sample of 10K unlikely to catch it
→ Fix: ALWAYS run uniqueness and completeness on FULL data
→ Sampling OK for: distribution checks, statistical rules

PITFALL 8: "Multiple quality gates conflict"
→ Cause: source gate passes (row count OK) but silver gate fails (duplicates)
→ Source gate checked volume, not content → passed
→ Silver gate checked uniqueness → failed → quarantined
→ But: source already loaded to silver → inconsistent state
→ Fix: source gate should include uniqueness check too
→ Or: silver gate quarantines from silver (not source)

PITFALL 9: "DQDL CustomSql rule causes Spark error"
→ Cause: CustomSql uses SQL syntax not supported by Spark SQL
→ DQDL translates CustomSql to Spark → Spark SQL has different functions
→ Fix: test CustomSql rules in Spark context, not Redshift context
→ Use Spark-compatible functions: COALESCE, CASE WHEN, etc.

PITFALL 10: "Quality metrics show 100% pass but CFO still sees wrong data"
→ Cause: quality rules check source data but error is in TRANSFORMATION
→ Source: 5,000 valid rows → Transform: JOIN creates 50,000 rows (fan-out)
→ Quality gate on source passes → transformation introduces duplicates
→ Fix: quality gate on TRANSFORMED output too (silver layer check)

🔑 KEY TAKEAWAY

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ Data quality gates PREVENT bad data from reaching production — not detect after
→ Three-layer defense: Pre-flight (Lambda) → ETL DQ (Glue DQDL) → Post-load (Redshift SQL)
→ DQDL: declarative rule language — RowCount, IsUnique, Completeness, ColumnValues, DataFreshness
→ Row-level filtering: good rows → silver, bad rows → quarantine (don't lose data)
→ HARD_FAIL vs SOFT_WARN: distinguish pipeline-stopping issues from minor warnings
→ Quarantine bucket: failed data preserved for investigation and replay
→ Quality metrics to CloudWatch: trending, alerting, SLA tracking
→ Quality results to S3 JSON: audit trail for compliance
→ Overhead: ~1 minute per source (~$0.02) → prevents incidents costing thousands
→ Step Function integration: quality failure → separate error path → distinct notification
→ Dynamic thresholds: avoid false positives from business growth
→ Weekly rule review: business changes require rule updates
→ NEVER skip quality checks — "just this once" is how incidents happen
