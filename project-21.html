🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — now 18 months old, growing rapidly
Situation:
→ Daily CSV exports land in S3 (established in Project 1)
→ Redshift cluster provisioned with proper schema (established in Project 10, 11)
→ orders table now has 10 MILLION rows accumulated in S3
→ Analytics team says: "We need ALL historical orders in Redshift for dashboards"
→ Junior engineer tried INSERT INTO ... VALUES one row at a time
   → Running for 14 hours, only 200K rows loaded
   → Redshift CPU at 98%, cluster unusable for other queries

CTO: "This is taking forever. Can't we just bulk-load everything?"
YOU: "INSERT row-by-row is the WORST way to load Redshift.
      Redshift has a native COPY command designed for exactly this.
      10 million rows should take 2-3 minutes, not 14 hours."

Data Profile:
→ 30 daily CSV files in S3 (one per day for last month)
→ Each file: ~333K rows, ~50 MB
→ Total: 10M rows, ~1.5 GB uncompressed CSV
→ S3 path: s3://quickcart-raw-data-prod/daily_exports/orders/
→ Redshift cluster: 2x dc2.large nodes (from Project 10)

🟡 HOW TO APPROACH IT

Step 1: Understand WHY row-by-row INSERT is disastrous
→ Redshift is COLUMNAR — optimized for bulk operations
→ Each INSERT is a single transaction with commit overhead
→ Leader node parses SQL → sends to compute nodes → commit → repeat
→ 10M individual transactions = 10M round trips
→ Network + parse + commit overhead PER ROW

Step 2: Understand the COPY command
→ Native Redshift bulk loader
→ Reads directly from S3 (parallel, distributed)
→ Each compute node slice reads a portion of the data
→ Bypasses leader node for data transfer
→ 10M rows in minutes, not hours

Step 3: Understand the PREREQUISITES
→ Redshift cluster must have an IAM Role that can read S3
→ S3 bucket policy must allow that role (or same-account = auto)
→ Target table must exist in Redshift
→ CSV format must match table columns (or use column mapping)

Step 4: Understand OPTIMIZATION levers
→ Split files = parallel loading across slices
→ GZIP compression = less data over network
→ Manifest file = explicit file list (no surprises)
→ COMPUPDATE = auto-detect column encoding
→ STATUPDATE = auto-refresh table statistics

THREE APPROACHES (increasing reliability):
  A) COPY from prefix     — simplest, loads ALL matching files
  B) COPY from manifest   — safest, loads EXACT specified files  
  C) COPY from compressed — fastest, GZIP files before loading

  DECISION: Approach B + C combined (manifest + compression)
  → Production-grade: explicit file list + compressed transfer

🔵 WHAT COULD BE DONE

Option A: COPY from S3 prefix (simplest)
──────────────────────────────────────────
COPY orders FROM 's3://quickcart-raw-data-prod/daily_exports/orders/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftS3ReadRole'
CSV IGNOREHEADER 1;

→ Loads EVERY file under that prefix
→ Risk: stray files (logs, temp files) cause failures
→ Risk: reprocessing already-loaded files

Option B: COPY from manifest file (recommended)
────────────────────────────────────────────────
→ JSON file listing EXACT files to load
→ Redshift loads ONLY those files
→ Fails if any listed file is missing (safety net)
→ Full control over what gets loaded

Option C: COPY with GZIP compression
─────────────────────────────────────
→ Compress CSVs to .gz before loading
→ 50 MB CSV → ~12 MB GZIP (75% reduction)
→ Less S3 read time, less network transfer
→ Redshift decompresses in parallel on each slice

Option D: Pre-split files to match slice count
──────────────────────────────────────────────
→ 2 dc2.large nodes × 2 slices each = 4 slices
→ Optimal: number of files = multiple of 4
→ 30 files ÷ 4 slices = 7-8 files per slice (close enough)
→ WORST case: 1 giant file → only 1 slice works, others idle

DECISION: B + C → Manifest file with GZIP-compressed CSVs

✅ WHAT SHOULD DO

1. ALWAYS use IAM Role — never access keys
   → Attach role to Redshift cluster with S3 read permission
   → Reference role ARN in COPY command
   → Keys in COPY command = security audit failure

2. ALWAYS use MANIFEST for production loads
   → Explicit file list prevents accidental data
   → Fails loudly if a file is missing (better than silent partial load)

3. ALWAYS use IGNOREHEADER 1 for CSVs with headers
   → First row is column names, not data
   → Without this: header row inserted as data → type errors or corrupt data

4. ALWAYS set MAXERROR appropriately
   → MAXERROR 0 (default): fail on first bad row
   → MAXERROR 100: tolerate up to 100 bad rows
   → Production: start with 0, increase only with monitoring

5. SPLIT files for parallel loading
   → Number of files should be MULTIPLE of number of slices
   → 2 dc2.large = 4 slices → ideal: 4, 8, 12, 16... files
   → Each slice processes its assigned files independently

6. COMPRESS files with GZIP before loading
   → Reduces S3 read I/O by 70-80%
   → Redshift auto-detects GZIP when you specify GZIP keyword
   → Upload .gz files to S3, then COPY with GZIP option

7. TRUNCATE before full reload (if replacing data)
   → TRUNCATE is instant (metadata operation)
   → DELETE FROM is row-by-row (extremely slow on 10M rows)
   → DELETE also leaves dead rows requiring VACUUM

8. Run ANALYZE after large loads
   → Updates table statistics for query planner
   → Without this: queries may choose bad execution plans
   → COPY can auto-run via STATUPDATE ON (default)

9. Use a STAGING table for incremental loads
   → Load into staging → validate → merge into production
   → Covered deeply in Project 65 (CDC Merge Pattern)

❌ WHAT SHOULD NOT DO

1. DO NOT use INSERT INTO ... VALUES for bulk loading
   → 10M single-row inserts = 14+ hours
   → COPY for same data = 2-3 minutes
   → INSERT is for single-row application writes, not ETL

2. DO NOT use INSERT INTO ... SELECT from another table for S3 loads
   → This doesn't read from S3 at all
   → Only works for table-to-table within Redshift

3. DO NOT COPY without IGNOREHEADER when CSVs have headers
   → "order_id" gets inserted as a VARCHAR value into an INT column
   → Causes silent data corruption or type mismatch errors

4. DO NOT load one giant file
   → Single file = single slice does all work
   → Other slices sit idle
   → Split into N files where N = multiple of slice count

5. DO NOT forget the REGION parameter if bucket is in different region
   → COPY defaults to cluster's region for S3 reads
   → Cross-region: must specify REGION 'us-west-2'
   → Best practice: keep S3 and Redshift in SAME region (Project 1 advice)

6. DO NOT ignore STL_LOAD_ERRORS after a failed COPY
   → This system table tells you EXACTLY which row failed and why
   → Without checking: you retry blindly and waste hours

7. DO NOT run COPY during peak query hours without WLM setup
   → COPY consumes cluster resources (CPU, I/O, memory)
   → Can starve dashboard queries
   → Use WLM queues to isolate load vs query workloads (Project 40)

8. DO NOT use COPY with access keys in the command
   → COPY orders FROM 's3://...' 
     ACCESS_KEY_ID 'AKIA...' SECRET_ACCESS_KEY '...'
   → These appear in query logs (STL_QUERYTEXT)
   → Anyone with system table access sees your credentials
   → ALWAYS use IAM_ROLE instead

🚫 WHEN NOT TO USE COPY

1. When data needs TRANSFORMATION before loading
   → COPY loads raw data as-is (no SQL transforms during load)
   → Use: Glue ETL → transform → write to S3 → COPY (Project 32)
   → Or: COPY into staging table → INSERT INTO SELECT with transforms

2. When loading < 100 rows
   → COPY has overhead (manifest parsing, S3 listing, slice coordination)
   → For tiny inserts: regular INSERT is fine
   → Threshold: below ~1000 rows, INSERT is acceptable

3. When source is NOT S3, DynamoDB, or EMR
   → COPY only supports these three sources
   → For RDS/MariaDB: extract to S3 first, then COPY (Project 20 → 21 flow)
   → For API data: write to S3 first, then COPY

4. When you need real-time streaming ingestion
   → COPY is batch-oriented (run it, wait, done)
   → For streaming: Kinesis Data Firehose → S3 → COPY micro-batches
   → Covered in Project 72

5. When data has complex nested JSON requiring flattening
   → COPY handles flat JSON and simple nesting
   → Deep nesting with arrays: pre-process in Glue first
   → Then COPY the flattened output

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

┌─────────────────────────────────────────────────────────────────────────────┐
│                    REDSHIFT COPY — INTERNAL EXECUTION FLOW                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  PHASE 1: COMMAND PARSING (Leader Node)                                     │
│  ──────────────────────────────────────                                     │
│  1. Leader receives COPY SQL statement                                      │
│  2. Validates: table exists, columns match, IAM role valid                  │
│  3. Contacts S3 to LIST files matching the prefix or manifest               │
│  4. Determines number of input files                                        │
│  5. Plans distribution: assigns files to compute node slices                │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────┐        │
│  │  FILE ASSIGNMENT ALGORITHM:                                     │        │
│  │                                                                 │        │
│  │  Files:  [f1, f2, f3, f4, f5, f6, f7, f8]                      │        │
│  │  Slices: [S0, S1, S2, S3]                                      │        │
│  │                                                                 │        │
│  │  Assignment (round-robin):                                      │        │
│  │    S0 → f1, f5                                                  │        │
│  │    S1 → f2, f6                                                  │        │
│  │    S2 → f3, f7                                                  │        │
│  │    S3 → f4, f8                                                  │        │
│  │                                                                 │        │
│  │  PERFECT: 8 files / 4 slices = 2 files each (balanced)         │        │
│  │                                                                 │        │
│  │  WORST CASE: 1 file / 4 slices = 1 slice busy, 3 idle          │        │
│  │  → This is why we split files!                                  │        │
│  └─────────────────────────────────────────────────────────────────┘        │
│                                                                             │
│  PHASE 2: PARALLEL DATA INGESTION (Compute Nodes)                           │
│  ─────────────────────────────────────────────────                           │
│  1. Each slice opens DIRECT connection to S3                                │
│     → Data does NOT flow through leader node                                │
│     → Leader only orchestrates; compute nodes do the actual I/O             │
│                                                                             │
│  2. Each slice:                                                             │
│     a. Downloads its assigned files from S3                                 │
│     b. Decompresses (if GZIP/LZO/BZIP2 specified)                          │
│     c. Parses rows according to format (CSV/JSON/AVRO/Parquet)              │
│     d. Applies column mapping / data type conversion                        │
│     e. Writes to local columnar storage                                     │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────┐        │
│  │  NETWORK DATA FLOW:                                             │        │
│  │                                                                 │        │
│  │  S3 ──────┬──────→ Compute Node 1, Slice 0  (files 1,5)        │        │
│  │           ├──────→ Compute Node 1, Slice 1  (files 2,6)        │        │
│  │           ├──────→ Compute Node 2, Slice 2  (files 3,7)        │        │
│  │           └──────→ Compute Node 2, Slice 3  (files 4,8)        │        │
│  │                                                                 │        │
│  │  Leader Node: sits idle during data transfer ✅                 │        │
│  │  → This is why COPY is so fast                                  │        │
│  │  → Contrast with INSERT: every row routes through leader        │        │
│  └─────────────────────────────────────────────────────────────────┘        │
│                                                                             │
│  PHASE 3: DATA DISTRIBUTION (Across Slices)                                 │
│  ──────────────────────────────────────────                                  │
│  After ingestion, rows may need REDISTRIBUTION based on DISTKEY:            │
│                                                                             │
│  → DISTSTYLE EVEN: rows stay where they landed (no redistribution)          │
│  → DISTSTYLE KEY:  rows re-shuffled so same key values → same slice         │
│  → DISTSTYLE ALL:  full copy sent to every slice (small dimension tables)   │
│                                                                             │
│  This redistribution happens AFTER ingestion, adding time for KEY/ALL       │
│                                                                             │
│  PHASE 4: COMMIT & STATISTICS                                               │
│  ────────────────────────────                                                │
│  1. All slices confirm successful load                                      │
│  2. Transaction committed atomically                                        │
│     → Either ALL rows load or NONE (ACID)                                   │
│  3. If STATUPDATE ON: table statistics auto-refreshed                       │
│  4. If COMPUPDATE ON: column encodings analyzed and applied                 │
│  5. Leader returns success + rows loaded count                              │
│                                                                             │
│  PHASE 5: ERROR HANDLING                                                    │
│  ───────────────────────                                                     │
│  If any row fails parsing:                                                  │
│  → Error details written to STL_LOAD_ERRORS system table                    │
│  → If errors ≤ MAXERROR: COPY continues, bad rows skipped                   │
│  → If errors > MAXERROR: entire COPY rolled back (zero rows loaded)         │
│  → Error info: file name, line number, column, raw value, error message     │
│                                                                             │
│  KEY SYSTEM TABLES FOR DEBUGGING:                                           │
│  ┌────────────────────────┬────────────────────────────────────────┐        │
│  │ STL_LOAD_ERRORS        │ Row-level error details                │        │
│  │ STL_LOAD_COMMITS       │ Which files were loaded successfully   │        │
│  │ STL_FILE_SCAN          │ Bytes read per file per slice          │        │
│  │ STL_S3CLIENT_ERROR     │ S3 connectivity issues                 │        │
│  │ SVL_S3LOG              │ S3 request details per file            │        │
│  │ STV_LOAD_STATE         │ Currently running COPY progress        │        │
│  └────────────────────────┴────────────────────────────────────────┘        │
│                                                                             │
│  COPY vs INSERT — WHY 1000x FASTER:                                         │
│  ┌─────────────────────┬────────────────────┬─────────────────────┐         │
│  │                     │ INSERT (per row)   │ COPY (bulk)         │         │
│  │ Parse SQL           │ 10M times          │ 1 time              │         │
│  │ Leader routing      │ 10M times          │ 1 time              │         │
│  │ Transaction commit  │ 10M times          │ 1 time              │         │
│  │ Network round trips │ 10M (app→leader)   │ 4 (S3→slices)      │         │
│  │ S3 connections      │ N/A (no S3)        │ 4 parallel          │         │
│  │ Parallelism         │ None (sequential)  │ Full (all slices)   │         │
│  │ Time for 10M rows   │ ~14 hours          │ ~2-3 minutes        │         │
│  └─────────────────────┴────────────────────┴─────────────────────┘         │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "COPY loads 0 rows but doesn't error"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: S3 prefix has no matching files
→ Debug: 
   SELECT * FROM stl_load_commits WHERE query = pg_last_copy_id();
→ Fix: verify exact S3 path — trailing "/" matters
   's3://bucket/orders'  → matches orders.csv AND orders_backup/
   's3://bucket/orders/' → matches ONLY files inside orders/ prefix

PITFALL 2: "Load Error: Delimiter not found"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: specified DELIMITER '|' but file uses ','
→ Debug:
   SELECT * FROM stl_load_errors ORDER BY starttime DESC LIMIT 10;
→ Fix: match DELIMITER to actual file delimiter

PITFALL 3: "Load Error: Extra column(s) found"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: CSV has 8 columns, table has 7 columns
→ Fix: either add missing column to table or use column mapping
   COPY orders (col1, col2, col3) FROM ...

PITFALL 4: "Load extremely slow despite multiple files"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause A: all files are tiny (< 1MB each) → S3 request overhead dominates
→ Cause B: number of files not aligned with slice count
→ Cause C: COMPUPDATE ON analyzing encodings (slow on first load)
→ Fix A: merge small files into larger ones (64-128 MB each)
→ Fix B: split/merge to get file count = multiple of slice count
→ Fix C: use COMPUPDATE OFF after first load

PITFALL 5: "COPY fails with Access Denied"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Redshift IAM role lacks s3:GetObject on the bucket
→ Debug: check IAM role → trust policy → S3 policy
→ Three things must align:
   1. Role has s3:GetObject + s3:ListBucket policy
   2. Role trust policy allows redshift.amazonaws.com to assume it
   3. Role is ATTACHED to the Redshift cluster (not just exists)

PITFALL 6: "Loaded data has wrong values in wrong columns"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: CSV column order doesn't match table column order
→ COPY maps by POSITION, not by column name
→ Fix: explicit column list in COPY command
   COPY orders (order_id, customer_id, amount) FROM ...

PITFALL 7: "Manifest file COPY fails: S3ServiceException: Forbidden"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: IAM role can read data files but NOT the manifest file
→ Manifest file is also in S3 → role needs access to BOTH:
   - s3://bucket/manifests/load_manifest.json  (manifest itself)
   - s3://bucket/daily_exports/orders/*        (data files)

PITFALL 8: "Table has duplicate data after running COPY twice"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: COPY is append-only — it does NOT check for duplicates
→ Running COPY twice on same files = double the data
→ Fix: 
   Option A: TRUNCATE before COPY (full reload)
   Option B: Load into staging → MERGE into target (Project 65)
   Option C: Track loaded files in a control table

📈 SCALABILITY CONSIDERATIONS

DATA VOLUME SCALING:
━━━━━━━━━━━━━━━━━━━
→ 10 MB:   trivial, single file, 5 seconds
→ 1 GB:    split into 4-8 files, 30-60 seconds
→ 10 GB:   split into 16-32 files + GZIP, 3-5 minutes
→ 100 GB:  split into 64+ files + GZIP, 15-30 minutes
→ 1 TB:    split into 128+ files + GZIP, 1-3 hours
→ 10 TB+:  consider multiple COPY jobs in parallel (different tables)

OPTIMAL FILE SIZE:
━━━━━━━━━━━━━━━━━━
→ Sweet spot: 1 MB – 1 GB per file (after compression)
→ Below 1 MB: S3 GET request overhead dominates
→ Above 1 GB: single slice takes too long, others finish and wait
→ Ideal: 64-128 MB compressed per file

FILE COUNT FORMULA:
━━━━━━━━━━━━━━━━━━
→ Optimal file count = (number of slices) × (integer multiplier)
→ 2x dc2.large = 4 slices → files: 4, 8, 12, 16, 20...
→ 4x dc2.large = 8 slices → files: 8, 16, 24, 32...
→ 2x ra3.xlplus = 4 slices → files: 4, 8, 12, 16...

HOW TO FIND YOUR SLICE COUNT:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ SELECT COUNT(DISTINCT slice) FROM stv_slices;
→ dc2.large: 2 slices per node
→ dc2.8xlarge: 16 slices per node
→ ra3.xlplus: 2 slices per node
→ ra3.4xlarge: 4 slices per node
→ ra3.16xlarge: 16 slices per node

CLUSTER SCALING:
━━━━━━━━━━━━━━━━
→ More nodes = more slices = more parallel S3 readers
→ Double nodes → roughly halve COPY time
→ But: node cost doubles too
→ Balance: optimize file count/size FIRST, add nodes LAST

CONCURRENCY:
━━━━━━━━━━━━
→ Multiple COPY jobs can run simultaneously
→ Each COPY competes for I/O and memory
→ WLM (Project 40) controls how many concurrent COPYs run
→ Recommended: max 2-3 concurrent COPY jobs

2. Check for errors:
   → SELECT * FROM stl_load_errors 
     WHERE query = pg_last_copy_id() 
     ORDER BY starttime DESC;
   → Shows: filename, line_number, colname, raw_field_value, err_reason

3. Check which files were loaded:
   → SELECT * FROM stl_load_commits 
     WHERE query = pg_last_copy_id();
   → Shows: filename, lines_scanned, bytes_scanned

4. Check per-file performance:
   → SELECT * FROM stl_file_scan 
     WHERE query = pg_last_copy_id();
   → Shows: which slice read which file, time taken, bytes read

5. Sample data validation:
   → SELECT * FROM orders ORDER BY order_id LIMIT 10;
   → Verify column values look correct (not shifted)

6. Aggregate validation:
   → Compare S3 row counts vs Redshift row counts
   → Compare SUM(amount) in source vs target
   → Any mismatch = data integrity issue

7. Execution time tracking:
   → SELECT query, starttime, endtime, 
     DATEDIFF(seconds, starttime, endtime) as duration_sec
     FROM stl_query WHERE querytxt LIKE 'COPY%' 
     ORDER BY starttime DESC LIMIT 5;

🔑 KEY TAKEAWAY

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ COPY is the ONLY way to bulk-load S3 → Redshift efficiently
→ INSERT row-by-row: 14 hours for 10M rows
→ COPY: 2-3 minutes for same 10M rows (1000x faster)
→ COPY reads directly from S3 → compute node slices (bypasses leader)
→ Split files to match slice count for maximum parallelism
→ GZIP compression reduces load time by 20-30%
→ MANIFEST gives explicit control over which files load
→ MANIFEST mandatory=true prevents silent partial loads
→ MAXERROR 0 in production (fail loudly, not silently)
→ COMPUPDATE ON only on first load, OFF for all subsequent loads
→ STL_LOAD_ERRORS is your best friend for debugging failures
→ COPY is APPEND-ONLY — running twice = duplicate data
→ Idempotency must be implemented in YOUR code (not COPY's job)
