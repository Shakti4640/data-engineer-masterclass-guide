🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — multi-account data platform maturing

CURRENT STATE (Cron-Based Scheduling):
→ 2:00 AM → Glue CDC job extracts from MariaDB (Project 65)
→ 2:07 AM → MV refresh Lambda (Project 63)
→ 3:00 AM → Sunday full reconciliation (Project 65)
→ 4:00 AM → Sunday compaction job (Project 62)
→ All triggered by Glue Triggers (cron schedules)

ADDITIONAL DATA SOURCES ARRIVING:
→ Partner feed: CSV dropped to S3 daily (arrival time: 1:30-3:00 AM, unpredictable)
→ Payment processor: JSON webhook → SQS → S3 (arrives continuously)
→ Marketing team: campaign export CSV uploaded manually (anytime during business hours)
→ Third-party API: Glue job pulls hourly inventory data

THE PROBLEMS:

PROBLEM 1: Fixed schedule misses variable arrival times
┌────────────────────────────────────────────────────────────────────┐
│ Partner feed CSV arrives at 2:45 AM                                │
│ Glue trigger fires at 2:00 AM → partner file doesn't exist yet   │
│ Result: partner data NOT processed until next day (2:00 AM + 24h) │
│ Business impact: 24-hour delay in partner revenue reporting       │
└────────────────────────────────────────────────────────────────────┘

PROBLEM 2: Early arrival wastes time waiting
┌────────────────────────────────────────────────────────────────────┐
│ MariaDB CDC delta ready at 1:50 AM                                 │
│ Glue trigger fires at 2:00 AM → 10 minutes wasted waiting        │
│ At scale: 10 min × 365 days = 60 hours/year of idle time          │
└────────────────────────────────────────────────────────────────────┘

PROBLEM 3: Cascading delays
┌────────────────────────────────────────────────────────────────────┐
│ CDC job runs at 2:00 AM → finishes at 2:06 AM                     │
│ MV refresh scheduled at 2:07 AM → runs on schedule ✅              │
│ But: if CDC runs long (2:15 AM) → MV refresh already ran          │
│ MV refreshed with STALE data → dashboard shows yesterday's numbers│
│ Must WAIT for CDC before refreshing — cron can't express this     │
└────────────────────────────────────────────────────────────────────┘

PROBLEM 4: No parallelism control
┌────────────────────────────────────────────────────────────────────┐
│ 5 independent data sources → all could process in PARALLEL         │
│ But: Glue Workflow runs steps SEQUENTIALLY by default              │
│ Sequential: 6 + 4 + 8 + 3 + 5 = 26 minutes                       │
│ Parallel: max(6, 4, 8, 3, 5) = 8 minutes                          │
│ Wasted: 18 minutes of sequential overhead                          │
└────────────────────────────────────────────────────────────────────┘

PROBLEM 5: No dynamic routing
┌────────────────────────────────────────────────────────────────────┐
│ Marketing uploads campaign file at 11:30 AM                        │
│ Next scheduled run: 2:00 AM tomorrow                               │
│ Marketing VP: "Why can't I see my data for 14 hours?"              │
│ Need: process IMMEDIATELY when file lands, regardless of schedule  │
└────────────────────────────────────────────────────────────────────┘

CTO: "Our pipeline is clock-driven. I want it EVENT-driven.
      When data arrives → process it. When processing completes → trigger next step.
      Independent sources process in parallel. Dependencies respected.
      No cron. No waiting. No missed files."

🟡 HOW TO APPROACH IT

Step 1: Understand EVENT-DRIVEN vs SCHEDULE-DRIVEN

  SCHEDULE-DRIVEN (current):
  ┌────────────────────────────────────────────────────────────┐
  │ TIME → triggers action                                     │
  │ "At 2:00 AM, run CDC job"                                  │
  │ "At 2:07 AM, refresh MVs"                                  │
  │ Problem: time doesn't know if data is ready                │
  │ Problem: data doesn't know when next schedule fires        │
  └────────────────────────────────────────────────────────────┘

  EVENT-DRIVEN (target):
  ┌────────────────────────────────────────────────────────────┐
  │ EVENT → triggers action                                    │
  │ "When file lands in S3 → process it"                       │
  │ "When CDC job completes → refresh MVs"                     │
  │ "When all sources processed → build gold layer"            │
  │ Advantage: zero wait, zero miss, zero waste                │
  └────────────────────────────────────────────────────────────┘

Step 2: Map events to actions

  ┌──────────────────────────────────┬──────────────────────────────────────┐
  │ EVENT                            │ ACTION                               │
  ├──────────────────────────────────┼──────────────────────────────────────┤
  │ Partner CSV lands in S3          │ Start partner ETL Glue job           │
  │ Payment JSON lands in S3         │ Start payment ETL Glue job           │
  │ Marketing CSV lands in S3        │ Start marketing ETL Glue job         │
  │ Scheduled time (hourly)          │ Start inventory API pull Glue job    │
  │ CDC window opens (2 AM)          │ Start MariaDB CDC Glue job           │
  │ ANY ETL job completes            │ Evaluate: are all sources ready?     │
  │ ALL sources ready                │ Build gold aggregations              │
  │ Gold layer built                 │ Refresh Redshift MVs                 │
  │ MVs refreshed                    │ Send success notification            │
  │ ANY step fails                   │ Send failure alert + retry logic     │
  └──────────────────────────────────┴──────────────────────────────────────┘

Step 3: Choose orchestration technology

  ┌──────────────────────────────┬──────────┬───────────────────────────────────┐
  │ Option                       │ Fit      │ Why                               │
  ├──────────────────────────────┼──────────┼───────────────────────────────────┤
  │ S3 Events → Lambda           │ ⚠️ Basic  │ No orchestration, no parallelism  │
  │ S3 Events → SQS → Lambda     │ ⚠️ Basic  │ Buffered but still no workflow    │
  │ Glue Workflows               │ ⚠️ Limited│ No S3 event trigger, rigid graph  │
  │ EventBridge → Step Functions  │ ✅ Best   │ Event routing + state machine     │
  │ EventBridge → Lambda → Glue   │ ⚠️ Medium │ Works but manual state tracking   │
  │ Apache Airflow (MWAA)         │ ✅ Good   │ Powerful but heavy infrastructure │
  └──────────────────────────────┴──────────┴───────────────────────────────────┘

  DECISION: EventBridge + Step Functions
  → EventBridge: universal event bus — routes events to targets
  → Step Functions: stateful workflow — manages dependencies, parallelism, retries
  → Together: events trigger workflows, workflows manage complex logic

🔵 WHAT COULD BE DONE

Option A: Monolithic Step Function — one giant state machine ⚠️
→ Single state machine handles ALL events and ALL logic
→ Simple to deploy but: massive JSON, hard to debug, tight coupling
→ Change to one source impacts entire workflow

Option B: Micro-orchestration — one Step Function per source + coordinator ✅ (CHOSEN)
→ Per-source Step Function: handles ETL for that source
→ Coordinator Step Function: waits for all sources, builds gold layer
→ EventBridge: routes events to correct per-source Step Function
→ Loose coupling: change one source without affecting others
→ Parallel by design: each source runs independently

Option C: EventBridge → Lambda → Glue (no Step Functions)
→ Lambda acts as "glue" (pun intended) between events and Glue jobs
→ Works for simple pipelines
→ BUT: no built-in state tracking, retry logic, parallelism
→ Must build orchestration logic in Lambda code → reinventing Step Functions

Option D: Apache Airflow (MWAA)
→ Full-featured DAG orchestration
→ Sensor operators can wait for S3 files
→ BUT: $0.49/hr minimum ($353/month) for MWAA environment
→ Overkill for 5 data sources
→ Better when: 50+ DAGs, complex dependency graphs, team familiarity

DECISION: Option B — micro-orchestration
→ Each data source: independent Step Function
→ Coordinator: waits for all, triggers downstream
→ EventBridge: event routing layer
→ Clean separation of concerns

✅ WHAT SHOULD DO

1. USE EventBridge for ALL event routing
   → S3 events → EventBridge (not S3 notifications directly)
   → Glue job state changes → EventBridge (native integration)
   → Scheduled events → EventBridge rules
   → Single event bus → centralized routing, filtering, monitoring

2. ENABLE S3 EventBridge notifications on source buckets
   → Default: S3 events go to SNS/SQS/Lambda only
   → Must EXPLICITLY enable EventBridge delivery per bucket
   → s3.put_bucket_notification_configuration with EventBridge enabled

3. USE Step Functions for workflow logic
   → Parallel branches: process independent sources simultaneously
   → Choice states: route based on event type or source
   → Wait states: pause until external condition met
   → Error handling: Retry + Catch at every step
   → Map state: dynamic parallelism (iterate over list of sources)

4. DESIGN idempotent ETL jobs
   → Event-driven = same event might fire twice (at-least-once delivery)
   → Glue job must handle re-processing gracefully
   → CDC MERGE pattern (Project 65) is naturally idempotent ✅
   → Full load with TRUNCATE+INSERT is idempotent ✅
   → Append-only: must deduplicate (check if file already processed)

5. TRACK pipeline state in DynamoDB
   → Which sources completed for today's batch?
   → DynamoDB table: pipeline_state (batch_date, source_name, status, timestamp)
   → Coordinator queries: "have ALL sources completed for 2025-01-15?"
   → If yes → trigger gold layer build

6. USE EventBridge event patterns for precise filtering
   → Filter by: S3 bucket, key prefix, key suffix
   → Example: only trigger on s3://bucket/partner_feed/*.csv
   → Prevents: triggering on _SUCCESS files, temp files, wrong prefixes

7. SET Step Function execution name to include batch identifier
   → Name: "partner-etl-2025-01-15-abc123"
   → Prevents duplicate executions for same event
   → Enables: easy debugging in console, searchable history

8. IMPLEMENT dead letter queue for failed events
   → EventBridge → Step Function invocation can fail
   → DLQ captures failed invocations
   → Alert + manual retry from DLQ

❌ WHAT SHOULD NOT DO

1. DO NOT use S3 event notifications directly (bypass EventBridge)
   → S3 notifications: limited filtering, no replay, no archive
   → EventBridge: rich filtering, archive, replay, schema registry
   → S3 → EventBridge → Step Functions (correct)
   → S3 → Lambda → Glue (fragile, no state management)

2. DO NOT create one Step Function per file
   → "Every file gets its own execution"
   → 1000 files/day = 1000 executions = expensive + hard to monitor
   → Instead: batch files by source, one execution per batch

3. DO NOT ignore duplicate events
   → S3 EventBridge: may deliver same event 2-3 times
   → Step Function execution: use idempotency key in execution name
   → ETL job: must handle re-processing (CDC MERGE does this naturally)

4. DO NOT hardcode file paths in Step Function definition
   → Step Function receives event payload dynamically
   → Extract bucket/key from event → pass to Glue job as argument
   → Reusable state machine for any file in same pattern

5. DO NOT skip error handling in Step Functions
   → "It's just 5 steps, what could go wrong?"
   → Glue job fails → Step Function stuck in RUNNING state forever
   → ALWAYS: Retry (with backoff) + Catch (with error notification)
   → TimeoutSeconds on every Task state

6. DO NOT trigger gold layer build after EACH source
   → Source A completes → build gold → Source B completes → rebuild gold
   → Wasteful: gold rebuilt 5 times instead of once
   → Instead: coordinator waits for ALL sources → build gold ONCE

7. DO NOT mix EventBridge rules with Glue Triggers
   → "Some jobs triggered by EventBridge, some by Glue cron"
   → Two orchestration systems = confusion, gaps, duplicates
   → Migrate ALL to EventBridge + Step Functions
   → Disable ALL Glue Triggers (cron-based)

8. DO NOT forget to handle the "source never arrives" case
   → Partner feed usually arrives by 3 AM
   → What if it doesn't arrive by 5 AM?
   → Timeout: EventBridge scheduled rule at 5 AM checks state
   → If source missing: alert + proceed without it (or wait with SLA)

🚫 WHEN NOT TO USE EVENT-DRIVEN

1. When pipeline has strict ordering that can't be parallelized
   → "Step A must finish before Step B which must finish before Step C"
   → Sequential-only pipeline: cron + Glue Workflow is simpler
   → Event-driven adds complexity without parallelism benefit

2. When source has no event emission capability
   → On-prem FTP server drops file — no S3 event
   → Must POLL for new files → scheduled check (cron pattern)
   → Hybrid: cron polls → puts to S3 → S3 event → event-driven

3. When debugging requires linear execution trace
   → Event-driven: execution scattered across multiple services
   → Debugging: follow event through EventBridge → Step Function → Glue → Lambda
   → New team members may struggle with distributed tracing
   → Invest in: CloudWatch dashboards, X-Ray tracing

4. When cost of Step Functions exceeds benefit
   → Step Functions pricing: $0.025 per 1000 state transitions
   → Simple pipeline with 5 states × 1 run/day = $0.004/month (negligible)
   → Complex pipeline with 100 states × 100 runs/day = $7.50/month (still cheap)
   → BUT: 10,000 runs/day with 50 states = $375/month → evaluate alternatives

5. When exactly-once processing is critical and hard to achieve
   → Event-driven = at-least-once delivery
   → Must design idempotent consumers
   → If idempotency is impossible: use database transactions + dedup table

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "S3 event never reaches EventBridge"
→ Cause: EventBridge notifications NOT enabled on S3 bucket
→ Default: S3 does NOT send to EventBridge — must opt-in
→ Fix: aws s3api put-bucket-notification-configuration with EventBridgeConfiguration
→ This is the #1 missed step in event-driven S3 pipelines

PITFALL 2: "Step Function triggered twice for same file"
→ Cause: S3 EventBridge delivers at-least-once
→ Same PutObject event delivered twice → two executions start
→ Fix: use file path as execution name (idempotency)
→ Step Functions rejects duplicate execution names → second invocation fails harmlessly

PITFALL 3: "EventBridge rule matches too many events"
→ Cause: rule pattern too broad
→ Pattern: source=aws.s3 → matches ALL S3 events in account
→ Every file upload triggers pipeline → chaos
→ Fix: filter by bucket AND prefix AND suffix
→ Pattern: bucket=quickcart-partner-feeds AND key PREFIX "daily/" AND key SUFFIX ".csv"

PITFALL 4: "Step Function execution times out after 1 year"
→ Cause: .sync task waiting for Glue job that's stuck
→ Standard Step Functions: max 1 year execution
→ But: Glue job stuck in RUNNING → Step Function waits forever
→ Fix: TimeoutSeconds on every Task state (e.g., 3600 = 1 hour)
→ Timeout triggers Catch → error handler → alert

PITFALL 5: "Parallel branches: one slow source blocks everything"
→ Cause: Parallel state waits for ALL branches
→ Partner ETL: 3 min, Payment ETL: 5 min, CDC: 6 min, Marketing: 45 min (huge file)
→ Gold layer build waits 45 minutes for marketing
→ Fix: set per-branch timeout, or decouple gold build from slow sources
→ Or: tolerate failures (ToleratedFailurePercentage in Map state)

PITFALL 6: "EventBridge rule exists but target invocation fails"
→ Cause: EventBridge role lacks permission to start Step Function
→ Rule matches → tries to invoke target → AccessDenied → silent failure
→ Fix: EventBridge rule needs IAM role with states:StartExecution permission
→ Debug: EventBridge → Rules → Monitoring tab shows FailedInvocations metric

PITFALL 7: "File processed by event AND by leftover cron schedule"
→ Cause: old Glue Trigger (cron) still active alongside new EventBridge
→ Same file processed twice: once by event, once by cron
→ Fix: DISABLE all Glue Triggers after migrating to EventBridge
→ Verify: Glue console → Triggers → all should be DEACTIVATED

PITFALL 8: "Step Function Map state: 40 items but only 10 process"
→ Cause: MaxConcurrency = 10 on Map state
→ Only 10 iterations run simultaneously, rest queued
→ If each takes 5 min: 40 items × 5 min / 10 concurrent = 20 min
→ Fix: increase MaxConcurrency (max 40 for inline, unlimited for distributed)
→ Or: use DISTRIBUTED Map mode for > 40 items

PITFALL 9: "Can't pass S3 key with special characters to Glue"
→ Cause: S3 key "daily/partner feed (2025-01-15).csv" has spaces and parens
→ EventBridge passes raw key → Step Function passes to Glue → URL encoding issues
→ Fix: use intrinsic functions States.StringToJson, States.Format
→ Or: URL-encode in Lambda preprocessor

PITFALL 10: "Event storm: 1000 files land simultaneously"
→ Cause: bulk upload to S3 → 1000 S3 events → 1000 Step Function executions
→ Glue concurrent job limit: 200 by default
→ Step Functions all try to start Glue → throttling → failures
→ Fix: SQS between EventBridge and Step Function (buffer)
→ Or: batch files into single execution (Lambda preprocessor)

📈 SCALABILITY CONSIDERATIONS

EVENT VOLUME SCALING:
→ 5 events/day (current): trivial
→ 100 events/day: easily handled
→ 10,000 events/day: need SQS buffering before Step Functions
→ 1M events/day: need EventBridge → SQS → Lambda batch → Step Function

STEP FUNCTION SCALING:
→ Standard: 25,000 state transitions/second (per account)
→ Express: 100,000 state transitions/second (for high-volume)
→ 5 pipelines × 10 states each = 50 transitions/day → negligible
→ Express workflow: for sub-second file processing (not our case)

GLUE JOB CONCURRENCY:
→ Default: 200 concurrent job runs per account
→ 5 parallel sources × 1 job each = 5 concurrent → fine
→ If 50 sources: may hit limit → request increase or batch
