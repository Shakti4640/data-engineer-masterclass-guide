🟡 HOW TO APPROACH IT

Step 1: Understand CDC (Change Data Capture) fundamentals

  WHAT IS CDC?
  → Technique to identify and capture ONLY the data that CHANGED
  → Changed = Inserted, Updated, or Deleted since last extraction
  → Process delta (changed rows) instead of full table

  CDC METHODS:
  ┌──────────────────────────┬──────────┬────────────────────────────────────┐
  │ Method                   │ Accuracy │ How It Works                       │
  ├──────────────────────────┼──────────┼────────────────────────────────────┤
  │ Timestamp-based (chosen) │ High     │ WHERE updated_at > last_watermark  │
  │                          │          │ Requires audit columns on source   │
  │                          │          │ Misses hard deletes                │
  ├──────────────────────────┼──────────┼────────────────────────────────────┤
  │ Binlog-based (DMS)       │ Highest  │ Read MariaDB binary log            │
  │                          │          │ Captures ALL changes including     │
  │                          │          │ deletes, zero source impact        │
  │                          │          │ Requires DMS replication instance  │
  ├──────────────────────────┼──────────┼────────────────────────────────────┤
  │ Trigger-based            │ Highest  │ DB trigger logs changes to table   │
  │                          │          │ High overhead on OLTP performance  │
  │                          │          │ Not recommended for production     │
  ├──────────────────────────┼──────────┼────────────────────────────────────┤
  │ Full diff                │ Highest  │ Compare today's full extract with  │
  │                          │          │ yesterday's → compute diff         │
  │                          │          │ Expensive: reads full table twice  │
  └──────────────────────────┴──────────┴────────────────────────────────────┘

Step 2: Choose timestamp-based CDC

  WHY TIMESTAMP-BASED:
  → QuickCart's MariaDB tables already have:
    - created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
    - updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
  → No additional infrastructure (no DMS replication instance)
  → Glue can handle with simple WHERE clause
  → Catches: INSERTs (created_at > watermark) and UPDATEs (updated_at > watermark)
  → Does NOT catch: hard DELETEs (row disappears — no timestamp to query)

  HANDLING DELETES:
  → Option A: Soft deletes — add is_deleted flag + deleted_at timestamp
    → Application sets is_deleted = TRUE instead of DELETE
    → CDC captures this as an UPDATE (updated_at changes)
    → RECOMMENDED for data warehousing
  → Option B: Periodic full diff — weekly full extract → compare → find deletions
    → Expensive but catches hard deletes
    → Run weekly as supplement to daily CDC
  → Option C: Binlog CDC via DMS — captures DELETE events
    → Best accuracy but adds infrastructure (Project 79)

  DECISION: 
  → Daily: Timestamp-based CDC (INSERTs + UPDATEs)
  → Soft deletes on source tables (application change)
  → Weekly: Full diff reconciliation (catch edge cases)

Step 3: Design the MERGE pattern

  CURRENT (Full Load):
  ┌─────────────────────────────────────────────────────────┐
  │ TRUNCATE target_table;                                  │
  │ INSERT INTO target_table SELECT * FROM source_table;    │
  │                                                         │
  │ DOWNTIME: entire TRUNCATE → INSERT window               │
  │ COST: reads ALL rows every time                         │
  └─────────────────────────────────────────────────────────┘

  NEW (Incremental CDC + Merge):
  ┌─────────────────────────────────────────────────────────┐
  │ 1. Extract ONLY changed rows (WHERE updated_at > mark) │
  │ 2. Load into STAGING table                              │
  │ 3. MERGE staging into target:                           │
  │    - If key exists in staging AND target: UPDATE         │
  │    - If key exists in staging NOT target: INSERT         │
  │    - If staging row has is_deleted=true: mark as deleted │
  │ 4. Update watermark to current timestamp                │
  │                                                         │
  │ ZERO DOWNTIME: target table always has data              │
  │ COST: reads only 200K rows instead of 12M               │
  └─────────────────────────────────────────────────────────┘

✅ WHAT SHOULD DO

1. ADD audit columns to MariaDB source tables
   → created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
   → updated_at TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
   → is_deleted BOOLEAN DEFAULT FALSE
   → These MUST exist before CDC works

2. CREATE indexes on updated_at in MariaDB
   → Without index: WHERE updated_at > X → full table scan on MariaDB
   → With index: range scan → reads only changed rows
   → CREATE INDEX idx_orders_updated_at ON orders(updated_at);
   → CRITICAL for MariaDB performance during extraction

3. USE staging table pattern in Redshift
   → Never merge directly from Glue to target
   → Load delta → staging → MERGE to target
   → Staging table is temporary (CREATE TEMP TABLE or TRUNCATE after)
   → Isolates loading errors from target table

4. PERSIST delta to S3 before loading to Redshift
   → S3 path: s3://lake/cdc/orders/batch_YYYYMMDD_HHMMSS/
   → Serves as: audit trail, replay capability, debugging
   → If Redshift MERGE fails: can replay from S3
   → If need to rebuild: replay all deltas in order

5. TRACK watermarks in a dedicated Redshift table
   → Table: cdc_watermarks (table_name, last_extracted_at, last_run_id)
   → Updated ONLY after successful MERGE
   → If job fails mid-way: watermark stays at previous value
   → Next run re-extracts same delta → idempotent

6. USE Redshift stored procedure for MERGE
   → Encapsulates: staging load → delete+insert → stats update
   → Transaction: all-or-nothing (COMMIT or ROLLBACK)
   → Already covered conceptually in Project 49

7. VALIDATE row counts at every step
   → MariaDB delta count → S3 file row count → staging count → target delta
   → If any mismatch: alert and investigate before proceeding
   → Prevents silent data loss

8. HANDLE late-arriving data
   → Watermark = last_extracted_at, not last_order_date
   → Some rows may have updated_at slightly before watermark
   → SAFETY MARGIN: subtract 5 minutes from watermark
   → WHERE updated_at > (watermark - INTERVAL 5 MINUTE)
   → MERGE handles duplicates gracefully (UPDATE existing row = no harm)

9. RUN weekly full reconciliation
   → Every Sunday: full extract + compare with Redshift
   → Detect: hard deletes, missed updates, data drift
   → Alert if delta > 0.1% of total rows

❌ WHAT SHOULD NOT DO

1. DO NOT use updated_at without an INDEX
   → MariaDB: WHERE updated_at > X without index = FULL TABLE SCAN
   → 12M row full scan = defeats purpose of CDC
   → MariaDB CPU: 95% instead of 5%
   → ALWAYS create index before enabling CDC

2. DO NOT update watermark BEFORE merge completes
   → If merge fails after watermark update: delta is LOST
   → Next run starts from new watermark → skips failed delta
   → ALWAYS: merge first → validate → THEN update watermark

3. DO NOT merge directly into target without staging
   → Direct merge from Glue DynamicFrame → Redshift target
   → If transformation error: corrupted data in target
   → No rollback capability
   → ALWAYS: Glue → S3 → staging → MERGE → target

4. DO NOT trust updated_at alone for deletes
   → Hard DELETE removes the row entirely — no updated_at to query
   → Timestamp CDC misses hard deletes by design
   → Solution: soft deletes (is_deleted flag) + weekly reconciliation

5. DO NOT set watermark safety margin to 0
   → Clock skew between MariaDB and Glue: possible 1-2 seconds
   → Transaction commit timing: row visible after commit
   → 0 margin: may miss rows committed during extraction
   → 5-minute margin: safe overlap, MERGE handles duplicates

6. DO NOT skip S3 persistence of deltas
   → "Just load directly to Redshift staging — faster"
   → If Redshift is down: delta is lost
   → No replay capability
   → No audit trail for compliance
   → S3 write adds ~30 seconds — worth it for reliability

7. DO NOT run CDC and full load simultaneously
   → Full load truncates target → CDC delta references deleted rows
   → Race condition: unpredictable state
   → Use mutex: only one ETL pattern runs at a time

8. DO NOT ignore timezone differences
   → MariaDB updated_at: in server timezone (e.g., UTC)
   → Glue watermark: stored as string or timestamp
   → If mismatch: watermark comparison wrong → miss or duplicate rows
   → ALWAYS: use UTC everywhere, store timezone-aware timestamps

🚫 WHEN NOT TO USE TIMESTAMP-BASED CDC

1. Source tables have NO audit columns and you CAN'T add them
   → No updated_at = no way to identify changes
   → Use: full diff or binlog CDC (DMS)

2. Hard deletes are critical to capture in real-time
   → Timestamp CDC misses DELETE statements
   → Use: binlog CDC (DMS) or soft-delete pattern

3. Source has bulk UPDATE operations that touch millions of rows
   → Monthly batch: UPDATE orders SET status='archived' WHERE order_date < '2024-01-01'
   → 5M rows get updated_at refreshed → CDC treats all as "changed"
   → Delta becomes nearly as large as full table
   → Use: filter by change TYPE, or switch to full load for that batch

4. Sub-minute latency required
   → Timestamp CDC = batch (runs on schedule)
   → Minimum practical interval: 5 minutes
   → For real-time: use DMS CDC or Kinesis streaming

5. Multiple writers update same row within CDC interval
   → Row updated 10 times between CDC runs
   → CDC captures only the FINAL state (latest updated_at)
   → If you need ALL intermediate states: use binlog CDC

┌───────────────────────────────────────────────────────────────────────────────┐

│  ─────────────────────────                                                    │
│                                                                               │

│  → MariaDB CPU: 5% (instead of 95% for full scan)                            │
│                                                                               │
│  WHAT'S IN THE DELTA:                                                         │
│  ┌─────────────────────────────────────────────────────────┐                  │
│  │ 150,000 rows: created_at > watermark (NEW inserts)     │                  │
│  │  45,000 rows: updated_at > watermark, created_at <     │                  │
│  │               watermark (EXISTING rows that were        │                  │

│                                                                               │

│  → TRUNCATE stg_orders; (clean from previous run)                             │
│  → COPY stg_orders FROM 's3://lake/cdc/orders/batch_...'                     │

│  → If staging has issues: target untouched                                    │

│  PHASE 5: THE MERGE                                                           │
│  ──────────────────                                                           │
│                                                                               │
│  Redshift does NOT have native MERGE statement (as of early versions)         │
│  Instead: DELETE + INSERT pattern (equivalent, transactional)                 │
│                                                                               │
│  ┌─────────────────────────────────────────────────────────────┐              │
│  │ BEGIN TRANSACTION;                                          │              │
│  │                                                              │              │
│  │ -- Step 1: DELETE rows in target that exist in staging      │              │
│  │ DELETE FROM orders                                           │              │
│  │ WHERE order_id IN (SELECT order_id FROM stg_orders);         │              │
│  │ -- Removes: existing rows that will be updated              │              │
│  │ -- Also removes: rows that are being soft-deleted           │              │
│  │ -- Rows NOT in staging: untouched ✅                        │              │
│  │                                                              │              │
│  │ -- Step 2: INSERT all staging rows into target              │              │
│  │ INSERT INTO orders                                           │              │
│  │ SELECT * FROM stg_orders                                     │              │
│  │ WHERE is_deleted = FALSE;                                    │              │
│  │ -- New rows: inserted (didn't exist before DELETE)          │              │
│  │ -- Updated rows: inserted with new values                   │              │
│  │ -- Deleted rows: NOT inserted (is_deleted = TRUE)           │              │
│  │ -- Net effect: UPSERT + soft delete handling                │              │
│  │                                                              │              │
│  │ -- Step 3: Update watermark                                 │              │
│  │ UPDATE cdc_watermarks                                        │              │
│  │ SET last_extracted_at = '2025-01-15 02:01:00',               │              │
│  │     last_run_id = 'jr_20250115_020100',                      │              │
│  │     rows_processed = 200000,                                 │              │
│  │     rows_inserted = 150000,                                  │              │
│  │     rows_updated = 45000,                                    │              │
│  │     rows_deleted = 5000                                      │              │
│  │ WHERE table_name = 'orders';                                 │              │
│  │                                                              │              │
│  │ COMMIT;                                                      │              │
│  │ -- ALL changes atomic: if any step fails → ROLLBACK         │              │
│  │ -- Target table is NEVER in inconsistent state              │              │
│  └─────────────────────────────────────────────────────────────┘              │
│                                                                               │
│  WHY DELETE + INSERT (not UPDATE):                                             │
│  → Redshift is COLUMNAR — UPDATE is expensive                                 │
│  → UPDATE in Redshift internally: marks old row deleted + inserts new row     │
│  → DELETE + INSERT does the same thing explicitly                             │
│  → Cleaner, faster, and easier to reason about                                │
│  → Also: COPY into staging + DELETE+INSERT is parallelized by Redshift        │
│                                                                               │
│  ZERO DOWNTIME EXPLAINED:                                                     │
│  → During DELETE+INSERT: Redshift uses MVCC (Multi-Version Concurrency)       │
│  → Concurrent readers see SNAPSHOT of data at transaction start               │
│  → DELETE marks rows as invisible to future transactions                      │
│  → INSERT adds new rows visible after COMMIT                                  │
│  → Reader during merge: sees pre-merge data (consistent)                      │
│  → Reader after merge: sees post-merge data (consistent)                      │
│  → NO moment where table appears empty or partially updated                   │
│                                                                               │
│  PHASE 6: POST-MERGE MAINTENANCE                                              │
│  ───────────────────────────────                                              │
│                                                                               │
│  → ANALYZE orders; — update table statistics for query optimizer               │
│  → VACUUM DELETE ONLY orders; — reclaim space from deleted rows               │
│  → TRUNCATE stg_orders; — clean staging for next run                          │
│  → Refresh Materialized Views (Project 63)                                    │
│                                                                               │
└───────────────────────────────────────────────────────────────────────────────┘

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "CDC extracted 0 rows but MariaDB has new data"
→ Cause: Watermark is in the FUTURE (timezone mismatch)
→ Glue stores watermark as UTC, MariaDB updated_at is in local time
→ WHERE updated_at > '2025-01-15 07:00:00' (UTC) but MariaDB is EST (02:00)
→ 07:00 UTC > all EST timestamps → 0 rows returned
→ Fix: ALWAYS use UTC on both sides
→ Or: CONVERT_TZ(updated_at, 'US/Eastern', 'UTC') in query

PITFALL 2: "MERGE succeeded but row counts don't match source"
→ Cause: Safety margin overlap causes duplicate extraction
→ Same rows extracted in consecutive runs
→ DELETE+INSERT handles this: deletes existing → reinserts → net zero
→ This is EXPECTED — not a bug
→ Validation: compare target COUNT with source COUNT, not delta counts

PITFALL 3: "Some updated rows were missed"
→ Cause: Transaction committed AFTER extraction started
→ Extraction at 02:00:00, transaction commits at 02:00:01
→ updated_at = 02:00:01 > watermark (02:00:00) BUT row wasn't visible during query
→ Fix: safety margin of 5 minutes catches most cases
→ For guaranteed capture: use binlog CDC (DMS)

PITFALL 4: "DELETE+INSERT is slow on 200K rows"
→ Cause: order_id not set as distribution key or sort key
→ DELETE WHERE order_id IN (...) scans all nodes
→ Fix: DISTSTYLE KEY(order_id) and SORTKEY(order_id) on target table
→ DELETE becomes single-node operation per key

PITFALL 5: "Staging table COPY fails — Parquet schema mismatch"
→ Cause: MariaDB column added/renamed but staging table DDL unchanged
→ Glue extracts new column → writes to Parquet → COPY expects old schema
→ Fix: Glue applies ResolveChoice (Project 45) + staging DDL migration
→ Include staging DDL in CI/CD (Project 76)

PITFALL 6: "Watermark updated but merge actually failed"
→ Cause: Watermark UPDATE is OUTSIDE the transaction
→ If merge transaction commits but watermark update fails: OK (re-run captures same delta)
→ If watermark updates but merge fails: LOST DATA
→ Fix: watermark update INSIDE the same transaction as merge
→ COMMIT covers both merge and watermark atomically

PITFALL 7: "Weekly reconciliation finds 50K missing rows"
→ Cause: Hard deletes on MariaDB (application uses DELETE not soft delete)
→ Rows deleted from source but still in Redshift target
→ Fix: weekly full diff identifies rows in target NOT in source → mark as deleted
→ Long-term: migrate application to soft deletes

PITFALL 8: "Glue job bookmark conflicts with manual watermark"
→ Cause: Using BOTH Glue bookmarks AND manual watermark table
→ Bookmark tracks its own state → conflicts with your watermark
→ Fix: use ONE mechanism — manual watermark gives more control
→ Disable Glue bookmarks: --job-bookmark-option = job-bookmark-disable

PITFALL 9: "CDC delta grows to 5M rows unexpectedly"
→ Cause: Bulk UPDATE on source (e.g., price recalculation touching all products)
→ All 5M rows get new updated_at → all appear as "changed"
→ CDC processes 5M instead of usual 200K
→ Fix: detect anomalous delta size → alert → switch to full load for this run
→ Threshold: if delta > 20% of total rows → full load is more efficient

PITFALL 10: "Redshift VACUUM takes 2 hours after merge"
→ Cause: DELETE+INSERT creates "ghost rows" in Redshift storage
→ VACUUM reclaims space but rewrites affected blocks
→ After 200K deletes: significant VACUUM work
→ Fix: VACUUM DELETE ONLY (faster than full VACUUM)
→ Or: schedule VACUUM separately (not blocking ETL)
→ Or: use Redshift automatic VACUUM (enabled by default)

KEY INSIGHT: Full load scales with TOTAL rows
             CDC scales with CHANGED rows (constant if change rate stable)

MARIADB SOURCE IMPACT:
→ Full load: SELECT * FROM orders (12M rows) → 95% CPU for 20 min
→ CDC: SELECT * FROM orders WHERE updated_at > X (200K rows) → 5% CPU for 2 min
→ At 200M rows: full load would cripple MariaDB for hours
→ CDC remains gentle regardless of total table size

GLUE DPU SCALING:
→ Full load 12M: 10 DPUs × 25 min = $1.85/run
→ CDC 200K: 3 DPUs × 6 min = $0.13/run
→ Monthly savings: ($1.85 - $0.13) × 30 = $51.60/month per table
→ 10 tables: $516/month saved

REDSHIFT MERGE SCALING:
→ 200K delta → 200K DELETE + 195K INSERT: ~15 seconds
→ 500K delta → 500K DELETE + 475K INSERT: ~30 seconds
→ 2M delta → 2M DELETE + 1.9M INSERT: ~90 seconds
→ At > 5M delta: consider batch processing (split into chunks)

WATERMARK MANAGEMENT SCALING:
→ 3 tables: simple — one watermark table row per source
→ 50 tables: moderate — parameterized job with table config
→ 200 tables: consider metadata-driven framework
→ Each table can have different CDC frequency (hourly, daily, weekly)

STORAGE SCALING (S3 CDC deltas):
→ 200K rows × 500 bytes avg = ~100 MB/day per table
→ 30 days retention: 3 GB per table
→ 10 tables × 3 GB = 30 GB → $0.69/month (S3 Standard)
→ After 90 days: lifecycle to Glacier ($0.004/GB) → $0.12/month
→ Deltas are cheap insurance for replay capability
