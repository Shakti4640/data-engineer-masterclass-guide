🟡 HOW TO APPROACH IT

Step 1: Understand what Athena Federated Query IS

  NORMAL ATHENA (Project 16, 24, 26):
  → Queries data in S3 via Glue Catalog
  → Only reads: S3 files (CSV, Parquet, JSON, ORC)
  → Only sees: tables defined in Glue Catalog of SAME account

  FEDERATED ATHENA:
  → Queries data ANYWHERE — not just S3
  → Uses LAMBDA CONNECTORS to reach external data sources
  → Each connector: translates Athena SQL → source-specific query
  → Sources: DynamoDB, RDS/MySQL, Redshift, PostgreSQL, other accounts' S3

  ┌─────────────────────────────────────────────────────────────────┐
  │                    ATHENA QUERY ENGINE                          │
  │                                                                 │
  │  SELECT * FROM s3_table       → reads S3 directly (normal)     │
  │  SELECT * FROM dynamo_table   → invokes Lambda connector       │
  │  SELECT * FROM mysql_table    → invokes Lambda connector       │
  │  SELECT * FROM remote_catalog → invokes Lambda connector       │
  │                                                                 │
  │  JOIN across ALL of these     → Athena handles join logic      │
  │                               → Lambda fetches data per source │
  └─────────────────────────────────────────────────────────────────┘

  SELECT c.campaign_name, c.cost, COUNT(o.order_id), SUM(o.total_amount)
  FROM marketing_catalog.campaign_spend c

🔵 WHAT COULD BE DONE

Option A: Deploy ALL connectors as Lambda in Account B ✅ (CHOSEN)
→ Account B is single query hub
→ All Lambdas deployed and managed in Account B
→ Cross-account access via IAM AssumeRole
→ Simplest management — one account owns all connectors

Option B: Deploy connectors in each source account
→ Account A runs MySQL connector Lambda
→ Account C runs DynamoDB connector Lambda
→ Account B invokes cross-account Lambdas
→ More complex: Lambda permissions span 3 accounts

Option C: Copy all data to Account B first, then query
→ ETL from all accounts → Account B S3
→ Query only Account B's S3 via normal Athena
→ Simpler queries but: data duplication, latency, storage cost
→ Already doing this partially (Project 61)

Option D: Redshift Spectrum + Data Sharing for everything
→ Use Redshift as universal query engine
→ Spectrum for S3 data, Data Sharing for cross-account Redshift
→ But: can't reach DynamoDB or MariaDB directly
→ Limited to Redshift ecosystem

DECISION: Option A — Lambda connectors in Account B
→ Most flexible — any source reachable
→ Athena handles JOIN logic across sources
→ Each source exposes data without moving it
→ True federated architecture

✅ WHAT SHOULD DO

1. USE pre-built Athena connector Lambdas (AWS Serverless Application Repository)
   → AWS provides official connectors for:
     MySQL, PostgreSQL, DynamoDB, Redshift, HBase, Redis, etc.
   → Battle-tested, maintained by AWS
   → Deploy via SAR (Serverless Application Repository) or CloudFormation
   → DO NOT write custom connectors unless absolutely necessary

2. DEPLOY connectors with MINIMAL permissions
   → DynamoDB connector: only dynamodb:Query, dynamodb:Scan on specific table
   → MySQL connector: read-only user (glue_reader from Project 61)
   → Cross-catalog: only glue:GetDatabase, glue:GetTable, s3:GetObject

3. USE Spill Bucket for large result sets
   → Connector Lambda has 10 GB /tmp and 15 min timeout
   → For large scans: Lambda "spills" intermediate data to S3
   → Must configure spill bucket in connector config
   → Spill bucket = Account B's S3 (encrypted, lifecycle to delete after 1 day)

4. PUSH DOWN predicates to source
   → Athena sends WHERE clauses to connector Lambda
   → Connector pushes to source: WHERE campaign_id = 'X'
   → Source returns only matching rows (not full table)
   → CRITICAL for DynamoDB: full scan = expensive + slow

5. REGISTER each connector as a DATA SOURCE in Athena
   → Each data source = one Lambda connector
   → Each data source gets a CATALOG NAME
   → Query: SELECT * FROM catalog_name.database.table

6. CREATE dedicated Athena workgroup for federated queries
   → Separate from standard S3-only queries
   → Track costs separately
   → Set query timeout (federated queries can be slow)
   → Configure result location

7. TEST each connector independently before JOIN queries
   → First: SELECT COUNT(*) FROM mysql_acct_a.quickcart.orders
   → Then: SELECT COUNT(*) FROM dynamodb_acct_c.default.campaign_events
   → Then: JOIN them
   → Isolates connector issues from query logic issues

8. CACHE frequently-joined results with CTAS
   → First federated query: slow (crosses network, Lambda invocations)
   → If same query runs daily: use CTAS to materialize in S3
   → Subsequent reads: fast native S3 scan
   → Already covered in Project 36

❌ WHAT SHOULD NOT DO

1. DO NOT scan entire DynamoDB table without filters
   → DynamoDB connector supports Query and Scan
   → Full Scan on 25M items: ~$6.25 per scan (read capacity)
   → Plus: Lambda timeout if table is too large
   → ALWAYS push down filters to reduce scan scope
   → Use partition key (campaign_id) in WHERE clause

2. DO NOT deploy connectors in public subnets
   → Lambda connector may have VPC config (for MySQL)
   → Public subnet = unnecessary internet exposure
   → Use private subnet with VPC endpoints (same as Glue — Project 61)

3. DO NOT give connectors broad cross-account permissions
   → BAD: "arn:aws:dynamodb:*:333333333333:table/*" → all tables
   → GOOD: "arn:aws:dynamodb:us-east-2:333333333333:table/campaign_events"
   → Principle of least privilege per connector

4. DO NOT expect federated queries to be as fast as native S3
   → Native S3 Athena: data locality, optimized scan
   → Federated: network hop → Lambda → source → Lambda → Athena
   → Expect 2-10x slower for same data volume
   → Use for flexibility, not performance

5. DO NOT JOIN large tables across connectors
   → JOIN orders (12M rows, Account A) WITH campaign_events (25M, Account C)
   → Athena must pull ALL rows from both Lambdas → massive data transfer
   → Fix: pre-filter both sides, or materialize one side first

6. DO NOT forget Lambda concurrency limits
   → Default: 1000 concurrent Lambda invocations per account
   → Athena may invoke connector Lambda many times in parallel
   → Heavy federated query + other Lambdas = throttling
   → Reserve concurrency for connector Lambdas

7. DO NOT skip the spill bucket configuration
   → Without spill bucket: Lambda /tmp fills up → query fails
   → For any table > ~1M rows: spill is likely
   → Always configure spill bucket with encryption + lifecycle

8. DO NOT use federated query as replacement for ETL
   → "Why ETL if we can query live?"
   → Federated = slow, expensive, puts load on source systems
   → ETL = pre-processed, fast, no source impact
   → Federated for: ad-hoc exploration, prototyping, low-volume joins
   → ETL for: dashboards, reports, repeated queries (Projects 61, 63)

🚫 WHEN NOT TO USE FEDERATED QUERY

1. When query runs 100+ times/day on same data
   → Lambda invocation cost: $0.20 per 1M requests
   → 100 queries × 50 Lambda invocations each = 5000/day
   → Materialize with CTAS or ETL instead

2. When source system can't handle the read load
   → DynamoDB: Scan consumes read capacity → costs money → throttles app
   → MariaDB: Athena opens parallel connections → overwhelms OLTP DB
   → Solution: query replicas, not primaries

3. When JOINing two large tables across connectors
   → Both tables must be fully scanned → transferred to Athena
   → 12M orders + 25M events = massive Lambda data transfer
   → Solution: pre-filter or pre-aggregate one side

4. When sub-second latency is required
   → Federated query minimum: 3-10 seconds (Lambda cold start + network)
   → Dashboard needs < 2 seconds → use MVs (Project 63)

5. When data sensitivity prevents cross-account access
   → Even with IAM roles: data leaves source account temporarily
   → Athena processes data in Account B
   → If compliance requires data stays in Account C → can't federate
   → Use: Account C's own Athena with their own Glue Catalog

│                │                                                              │
│  ┌─────────────▼───────────────────────────────────────────────────────┐      │
│  │ LAYER 2: Athena Federation SDK (runs inside Lambda)                 │      │
│  │ → Implements standard interface:                                    │      │

│  │ → Receives native query (DDB Scan, SQL SELECT, S3 GET)              │      │

│  │ → Lambda converts to Arrow → returns to Athena                      │      │

│                                                                               │
│  DETAILED QUERY EXECUTION FLOW:                                               │
│                                                                               │
│  User submits:                                                                │
│  SELECT c.campaign_name, SUM(o.total_amount) AS revenue                       │
│  FROM dynamodb_acct_c.default.campaign_events c                               │
│  JOIN analytics_lake.silver_orders o                                          │
│    ON c.user_id = o.customer_id                                               │
│  WHERE c.event_type = 'conversion'                                            │
│    AND o.order_date >= '2025-01-01'                                           │
│  GROUP BY c.campaign_name                                                     │
│                                                                               │
│  Step 1: PARSE                                                                │
│  → Athena identifies two catalogs:                                            │
│    - dynamodb_acct_c → federated (Lambda connector)                           │
│    - analytics_lake → could be native S3 or cross-account Glue               │
│                                                                               │
│  Step 2: PLAN                                                                 │
│  → Athena builds execution plan:                                              │
│    ┌──────────────────────────────────────────┐                               │
│    │ HashAggregate (GROUP BY campaign_name)    │                               │
│    │   └── HashJoin (ON user_id = customer_id)│                               │
│    │        ├── FederatedScan (DynamoDB)       │ ← Lambda invocation           │
│    │        │   filter: event_type='conversion'│ ← PUSHDOWN                   │
│    │        └── NativeScan (S3 Parquet)        │ ← Direct S3 read             │
│    │            filter: order_date>='2025-01-01│ ← Partition pruning           │
│    └──────────────────────────────────────────┘                               │
│                                                                               │
│  Step 3: EXECUTE — Federated Scan (DynamoDB)                                  │
│                                                                               │
│  3a. Athena invokes Lambda: GetTableLayout                                    │
│      → Lambda calls DynamoDB DescribeTable                                    │
│      → Returns: schema (column names, types)                                  │
│                                                                               │
│  3b. Athena invokes Lambda: GetSplits                                         │
│      → Lambda determines how to parallelize:                                  │
│        - DynamoDB: Scan with Segment/TotalSegments                            │
│        - MySQL: ranges on primary key                                         │
│      → Returns: list of "splits" (parallel work units)                        │
│      → Example: 10 splits for DynamoDB parallel scan                          │
│                                                                               │
│  3c. Athena invokes Lambda: ReadRecords (once per split)                      │
│      → 10 splits = 10 Lambda invocations (PARALLEL)                           │
│      → Each Lambda:                                                           │
│        a. Receives: split info + pushed-down predicate                        │
│        b. Calls DynamoDB Scan with:                                           │
│           - Segment=N, TotalSegments=10                                       │
│           - FilterExpression: event_type = 'conversion'  ← PUSHDOWN          │
│        c. Converts DynamoDB items → Apache Arrow record batch                 │
│        d. If data > Lambda memory: spills to S3 spill bucket                  │
│        e. Returns Arrow data to Athena                                        │
│                                                                               │
│  Step 4: EXECUTE — Native Scan (S3)                                           │
│  → Athena reads S3 Parquet directly (no Lambda)                               │
│  → Partition pruning: only reads order_date >= '2025-01-01' partitions        │
│  → Column pruning: only reads customer_id, total_amount columns               │
│  → Much faster than federated scan                                            │
│                                                                               │
│  Step 5: JOIN                                                                 │
│  → Athena receives:                                                           │
│    - DynamoDB data (from Lambda) in Arrow format                              │
│    - S3 data (from direct read) in Arrow format                               │
│  → Performs HashJoin on user_id = customer_id                                 │
│  → Both datasets must fit in Athena's memory                                  │
│  → If too large: Athena spills to its own temp storage                        │
│                                                                               │
│  Step 6: AGGREGATE + RETURN                                                   │
│  → GROUP BY campaign_name, SUM(total_amount)                                  │
│  → Result written to S3 output location                                       │
│  → Result returned to user                                                    │
│                                                                               │
│  ┌─────────────────────────────────────────────────────────────┐              │
│  │           PREDICATE PUSHDOWN — WHY IT MATTERS               │              │
│  │                                                              │              │
│  │  WITHOUT pushdown:                                           │              │
│  │  → Lambda scans ALL 25M DynamoDB items                       │              │
│  │  → Transfers ALL to Athena                                   │              │
│  │  → Athena filters: event_type = 'conversion'                │              │
│  │  → Wasted: 24M items transferred for nothing                │              │
│  │  → DynamoDB cost: 25M items × 4KB = ~$6.25 per query        │              │
│  │                                                              │              │
│  │  WITH pushdown:                                              │              │
│  │  → Lambda sends FilterExpression to DynamoDB                 │              │
│  │  → DynamoDB returns only conversions (~1M items)             │              │
│  │  → Lambda transfers 1M items to Athena                       │              │
│  │  → DynamoDB cost: still scans all (Scan reads all)           │              │
│  │  → BUT: less data transferred → less Lambda memory/time      │              │
│  │                                                              │              │
│  │  WITH DynamoDB Query (not Scan):                             │              │
│  │  → If campaign_id in WHERE clause                            │              │
│  │  → Lambda uses DynamoDB Query instead of Scan                │              │
│  │  → Only reads items for that campaign_id                     │              │
│  │  → Massively cheaper and faster                              │              │
│  │  → THIS is why partition key in WHERE clause matters         │              │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Federated query returns 0 rows but source has data"
→ Cause: Lambda connector can't reach source
→ DynamoDB: Lambda role lacks dynamodb:Scan permission on Account C table
→ MySQL: VPC Peering route missing or SG blocking
→ Debug: invoke connector Lambda directly with test event
→ Check Lambda CloudWatch logs for error details

PITFALL 2: "Query takes 10 minutes and then times out"
→ Cause: DynamoDB full table scan on 25M items
→ No WHERE clause on partition key → Lambda does full Scan
→ Lambda 15-min timeout hit before scan completes
→ Fix: add WHERE campaign_id = 'X' to use DynamoDB Query
→ Or: pre-aggregate DynamoDB data to S3 via ETL

PITFALL 3: "Lambda invocation throttled"
→ Cause: Athena invokes connector Lambda for each split in parallel
→ 50 splits × 10 concurrent queries = 500 concurrent Lambda invocations
→ Account B default Lambda concurrency: 1000
→ Other Lambdas in Account B competing for concurrency
→ Fix: reserve concurrency for connector Lambda (e.g., 200)
→ Or: reduce split count in connector configuration

PITFALL 4: "Cross-account AssumeRole fails with AccessDenied"
→ Cause: Trust policy in Account C doesn't trust Account B
→ Or: Account B Lambda role doesn't have sts:AssumeRole permission
→ Or: ExternalId mismatch (if configured — Project 56)
→ Debug: test AssumeRole manually from Account B CLI first

PITFALL 5: "Column type mismatch between DynamoDB and S3"
→ Cause: DynamoDB is schema-less — items may have varying types
→ Connector infers types from first batch of items
→ Later items have different type → cast error
→ Fix: explicitly define schema in connector's environment variable
→ Or: use CAST() in Athena query to handle type variations

PITFALL 6: "Spill bucket access denied"
→ Cause: Lambda role lacks s3:PutObject on spill bucket
→ Connector tries to spill → fails → query fails
→ Fix: ensure Lambda role has full S3 access to spill bucket
→ Check: bucket policy doesn't DENY the Lambda role

PITFALL 7: "Athena says 'Data source not found: dynamodb_acct_c'"
→ Cause: data source catalog name misspelled in query
→ Or: data source not registered in Athena
→ Verify: Athena console → Data sources → list all registered
→ Catalog names are case-sensitive in some contexts

PITFALL 8: "JOIN produces cartesian product — 300M rows"
→ Cause: JOIN key mismatch between DynamoDB and S3 tables
→ DynamoDB user_id = "USR-001" vs S3 customer_id = "CUST-000001"
→ No matches → cartesian product or empty result
→ Fix: verify JOIN key format matches across sources
→ Pre-process: ensure consistent ID format across systems

PITFALL 9: "Works for small query, fails for large query"
→ Cause: large federated scan exceeds Athena memory
→ Athena has per-query memory limits
→ Fix: break into smaller queries
→ Or: materialize federated data with CTAS first
→ Or: increase Lambda memory for connector

PITFALL 10: "Cost shock — $50 for one federated query"
→ Cause: DynamoDB Scan consumed 25M RCU
→ DynamoDB on-demand: $0.25 per million RCU
→ 25M items × 4KB each = 25M RCU = $6.25
→ Ran query 8 times during debugging = $50
→ Fix: ALWAYS test with LIMIT first
→ Use DynamoDB provisioned capacity with auto-scaling for predictable cost

→ CTAS materializes federated results to S3 Parquet → subsequent queries at native speed
→ Each data source = registered catalog → referenced in SQL as catalog_name.database.table
→ Monitoring: Lambda metrics + Athena query stats + cross-account IAM health checks
→ Use federated for exploration/prototyping, materialize for repeated production queries
→ DynamoDB queries: ALWAYS include partition key in WHERE → Query not Scan → 100x cheaper
