Option A: Cron schedules BOTH export AND COPY (sequential)

→ REJECTED (timing-dependent)

Option B: S3 event → SQS → EC2 worker → COPY (selected)
→ COPY triggers IMMEDIATELY when file lands
→ No timing assumptions — reacts to actual file arrival

→ SELECTED ✅

Option C: S3 event → Lambda → COPY
→ Serverless — no EC2 worker needed
→ But: Lambda needs VPC access for Redshift
→ But: COPY may take > 15 minutes for large tables

Option D: S3 event → Step Functions → Glue → Redshift
→ Most sophisticated orchestration

→ REJECTED (tight coupling)

→ Decoupled: export and load are independent
→ Reliable: SQS retry handles transient failures
→ Extensible: same queue can trigger other consumers later
→ Proven pattern: built on Projects 18, 21, 27

❌ WHAT SHOULD NOT DO

1. DO NOT COPY and TRUNCATE in separate transactions
   → TRUNCATE → (crash) → COPY never runs → table empty!
   → Better: BEGIN → TRUNCATE → COPY → COMMIT (atomic)
   → Or: COPY into staging → swap tables (zero-downtime)

2. DO NOT process files that aren't from today's export
   → S3 event may fire for old files being re-uploaded
   → Check: export_date in S3 key matches today (or yesterday)
   → Reject stale files to prevent loading outdated data

3. DO NOT run two COPYs to same table simultaneously
   → Two COPY commands to public.orders at same time
   → Both TRUNCATE → both COPY → one overwrites the other
   → Use: table-level lock in worker or process sequentially

4. DO NOT ignore COPY errors and delete the SQS message
   → COPY fails → delete message → data not loaded → nobody knows
   → ALWAYS: verify COPY succeeded BEFORE deleting message
   → On failure: leave message → retry → DLQ if persistent

5. DO NOT hardcode Redshift credentials in the script
   → Use: environment variables, Secrets Manager (Project 70), or IAM
   → Credentials in code → credentials in git → security breach

6. DO NOT run COPY during peak Redshift query hours without WLM
   → COPY consumes cluster I/O and memory
   → Dashboard queries compete for same resources
   → 2:00 AM is off-peak — good timing from Project 29
   → For peak-hour loads: configure WLM queues (Project 40)

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "TRUNCATE succeeded but COPY failed — table is empty"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: TRUNCATE and COPY in separate transactions
→ TRUNCATE commits → COPY fails → table empty
→ Dashboard queries return 0 rows → panic
→ Fix: wrap TRUNCATE + COPY in single transaction
→ BEGIN → TRUNCATE → COPY → COMMIT
→ If COPY fails → ROLLBACK → table still has old data ✅

PITFALL 2: "Same table loaded twice — duplicate rows"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: SQS delivers duplicate message (at-least-once)
→ Worker processes COPY twice for same file
→ With TRUNCATE + COPY: second run just replaces first (idempotent ✅)
→ Without TRUNCATE: COPY appends → double the data ❌
→ TRUNCATE + COPY is naturally idempotent for full-replace

PITFALL 3: "COPY takes 10 minutes — visibility timeout expires"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: visibility timeout (300s) < COPY time (600s)
→ At T=300s: message reappears → second worker picks up
→ Two workers run COPY simultaneously → chaos
→ Fix: set visibility timeout = 600s (or extend mid-processing)
→ Use: sqs.change_message_visibility() to extend during long COPY

PITFALL 4: "Worker picks up file from WRONG export date"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: old file re-uploaded triggers new S3 event
→ Worker loads stale data over fresh data
→ Fix: validate export_date in S3 key matches expected date
→ Reject files where export_date < yesterday

PITFALL 5: "COPY fails with Access Denied on S3"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Redshift IAM role lacks s3:GetObject on mariadb_exports/ prefix
→ Project 21 role was configured for daily_exports/ only
→ Fix: update Redshift IAM role to include mariadb_exports/* 
→ Or: use broader s3:GetObject on entire bucket

PITFALL 6: "Export manifest arrives BEFORE all data files"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Project 29 writes manifest after all tables exported
→ Manifest upload triggers S3 event → worker receives
→ But worker expects .csv.gz files, not .json
→ Fix: S3 notification suffix filter = .csv.gz (not .json)
→ Manifest file ignored by worker automatically
