🟡 HOW TO APPROACH IT

Step 1: Understand WHY cross-account access is needed
→ Security: blast radius isolation (compromised Account B can't destroy Account A)
→ Billing: separate cost tracking per team
→ Compliance: data residency and access controls per account
→ Governance: different IAM policies per account
→ This is AWS BEST PRACTICE — not optional for enterprises

Step 2: Understand the TWO cross-account S3 patterns
→ Pattern 1: RESOURCE-BASED policy (Bucket Policy)
  - Account A's bucket policy says: "Account B's role can read"
  - Account B's role accesses bucket directly
  - Objects accessed REMAIN owned by Account A
  - Simpler but less flexible

→ Pattern 2: IDENTITY-BASED + TRUST (AssumeRole via STS)
  - Account A creates a role: "CrossAccountS3ReadRole"
  - Account A's role trust policy says: "Account B can assume me"
  - Account B's entity calls sts:AssumeRole → gets temporary credentials
  - Uses temporary credentials to access Account A's S3
  - More flexible, more secure, more common in production

Step 3: Understand the PERMISSION CHAIN
→ Cross-account access requires BOTH sides to agree:
  - Account A must ALLOW (resource policy OR trust policy)
  - Account B must be ALLOWED to perform the action (identity policy)
  - If EITHER side denies → access denied
  - This is the "two-door" model — both doors must be open

Step 4: Understand STS (Security Token Service)
→ AssumeRole returns TEMPORARY credentials (not permanent)
→ Credentials expire after 1-12 hours (configurable)
→ Each AssumeRole call is logged in CloudTrail (both accounts)
→ Can be revoked by: removing trust policy, deleting role, or waiting for expiry

Step 5: Design the access matrix
→ Account B's Glue role → can READ s3://acct-a-bucket/daily_exports/*
→ Account B's Glue role → can READ s3://acct-a-bucket/cleaned/*
→ Account B's Glue role → CANNOT read s3://acct-a-bucket/internal/*
→ Account B's Glue role → CANNOT write to Account A's bucket
→ Account B's Athena role → same read access as Glue

🔵 WHAT COULD BE DONE

Option A: Copy data from Account A to Account B
→ S3 Replication or manual copy script
→ PROBLEMS: data duplication ($$$), staleness, sync complexity
→ At 100 GB: doubling storage cost + replication time
→ Not scalable — every new dataset needs replication setup

Option B: Make Account A's bucket public
→ ABSOLUTELY NOT — this is the #1 cause of S3 data breaches
→ Anyone on the internet can read your data
→ 2017 Verizon, 2019 Capital One — public S3 = headlines

Option C: Share IAM credentials from Account A to Account B
→ Impossible by design — IAM credentials belong to one account
→ Even if possible: violates security principles (shared credentials)
→ No audit trail of who used the credentials

Option D: Bucket Policy (Pattern 1) ✅ for simple cases
→ Account A grants specific access via bucket policy
→ Simple, no STS calls needed
→ But: Account B accesses as ITSELF (not as Account A's role)
→ Object ownership can be confusing (especially for writes)

Option E: AssumeRole (Pattern 2) ✅ for production use
→ Account A creates a role with specific permissions
→ Account B assumes that role via STS
→ Most flexible, most secure, most common
→ Clear audit trail in BOTH accounts' CloudTrail
→ Temporary credentials — automatically expire

DECISION: Implement BOTH patterns
→ Pattern 1 (Bucket Policy): simpler, good for read-only S3 access
→ Pattern 2 (AssumeRole): production standard, used in Projects 57-61
→ Understanding both is CRITICAL for interviews

✅ WHAT SHOULD DO

1. Use External ID for AssumeRole (confused deputy prevention)
   → External ID: a secret string shared between Account A and B
   → Account A's trust policy: "Allow Account B IF ExternalId matches"
   → Prevents: malicious third party tricking Account B into accessing Account A
   → This is the "confused deputy problem" — External ID solves it
   → AWS recommends External ID for ALL cross-account AssumeRole

2. Use LEAST PRIVILEGE on both sides
   → Account A's bucket policy: only specific prefixes, only specific actions
   → Account A's role: only s3:GetObject + s3:ListBucket (not s3:PutObject)
   → Account B's policy: only sts:AssumeRole on Account A's specific role ARN
   → Never use Resource: "*" in cross-account policies

3. Set session duration appropriately
   → Default: 1 hour
   → For Glue ETL (may run 2+ hours): set DurationSeconds to 3600-7200
   → For interactive Athena: 1 hour is fine
   → Shorter = more secure (credentials expire faster)
   → Longer = more convenient (fewer re-authentication calls)

4. Enable CloudTrail in BOTH accounts
   → Account A CloudTrail: sees who accessed S3 (the assumed role)
   → Account B CloudTrail: sees who called AssumeRole (the original caller)
   → Together: complete audit trail from original user to data access

5. Tag cross-account roles clearly
   → Tag: CrossAccount=true, SourceAccount=222222222222
   → Tag: Purpose=S3ReadOnly, ManagedBy=SecurityTeam
   → Makes it easy to audit all cross-account roles

6. Use Condition keys in trust policies
   → aws:SourceAccount: restrict to specific account
   → sts:ExternalId: prevent confused deputy
   → aws:PrincipalOrgID: restrict to accounts in same Organization
   → aws:PrincipalArn: restrict to specific roles (not any role in Account B)

7. Document the access path
   → Maintain a spreadsheet/wiki of all cross-account access grants
   → Who has access, to what, via which role, when granted, why
   → Review quarterly — revoke unused access

❌ WHAT SHOULD NOT DO

1. DO NOT use root account credentials for cross-account access
   → Root should NEVER be used for operational tasks
   → Root can't be restricted with IAM policies
   → Use IAM roles exclusively

2. DO NOT grant s3:* (all S3 actions) in cross-account policies
   → Account B only needs READ → grant only s3:GetObject + s3:ListBucket
   → s3:DeleteObject in cross-account = Account B can delete Account A's data
   → s3:PutObject with wrong ownership = Account A can't read objects Account B wrote

3. DO NOT hardcode Account IDs in application code
   → Use environment variables or SSM Parameter Store
   → Account IDs change when migrating between environments
   → Hardcoded IDs make testing impossible

4. DO NOT forget the BUCKET SIDE permission when using AssumeRole
   → Common mistake: set up AssumeRole correctly but forget bucket policy
   → With AssumeRole: bucket policy is NOT needed (role's IAM policy is enough)
   → But: if bucket has an explicit DENY → even AssumeRole can't override it
   → Check for explicit denies in bucket policy

5. DO NOT use long-lived access keys for cross-account
   → Access keys don't expire → compromised key = permanent access
   → AssumeRole credentials expire automatically
   → If you MUST use keys: rotate every 90 days (but prefer AssumeRole)

6. DO NOT create overly broad trust policies
   → Bad: "Allow entire Account B to assume this role"
     → Any user/role in Account B can assume it
   → Good: "Allow ONLY arn:aws:iam::222222222222:role/GlueRole-B"
     → Only the specific Glue role can assume it

7. DO NOT skip testing with wrong credentials
   → Test: Account B tries to access DENIED prefix → should fail
   → Test: Account B tries to WRITE → should fail
   → Test: Account C tries to assume Account A's role → should fail
   → Negative testing is MORE important than positive testing for security

🚫 WHEN NOT TO USE CROSS-ACCOUNT S3 ACCESS

1. When data MUST physically exist in both accounts
   → Cross-account access = read in place, no copy
   → Compliance may require: "data must reside in Account B's region"
   → Use: S3 Cross-Account Replication (Project 47)

2. When latency is critical and accounts are in different regions
   → Cross-account in same region: no latency impact
   → Cross-account cross-region: latency + data transfer cost
   → Use: replicate to local region, then access locally

3. When you need to grant access to thousands of objects selectively
   → Bucket policies have 20 KB size limit
   → Can't list 10,000 individual object ARNs
   → Use: prefix-based access (daily_exports/*) or tag-based (S3 access points)

4. When the accessing account is untrusted (public API consumers)
   → Cross-account = trust between known AWS accounts
   → Public consumers: use pre-signed URLs or CloudFront
   → Never create cross-account roles for unknown entities

5. When both accounts are in the SAME account
   → No cross-account setup needed
   → Just use IAM roles with proper policies
   → Cross-account adds complexity for zero benefit in single-account

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "AccessDenied even though bucket policy allows Account B"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Account B's identity policy ALSO needs to allow the action
→ Cross-account = BOTH sides must allow (AND logic, not OR)
→ Bucket policy allows ✅ + Identity policy missing ❌ = DENIED
→ Fix: add s3:GetObject permission to Account B's role policy

PITFALL 2: "AssumeRole succeeds but S3 access denied"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Account A's assumed role lacks S3 permissions
→ AssumeRole gives you the ROLE's permissions, not Account A's admin access
→ Role must have explicit s3:GetObject on the target bucket
→ Fix: attach S3 read policy to the cross-account role in Account A

PITFALL 3: "Trust policy allows Account B but AssumeRole still fails"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause 1: External ID required but not provided
→ Cause 2: Trust policy specifies specific role ARN, caller is different role
→ Cause 3: SCP (Service Control Policy) in Organization blocks sts:AssumeRole
→ Debug: check trust policy conditions EXACTLY match the caller

PITFALL 4: "Bucket policy exceeds 20 KB limit"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: granting access to too many individual principals/prefixes
→ Bucket policy max: 20 KB (about 20-30 policy statements)
→ Fix: use S3 Access Points (each has its own policy, up to 10,000 per bucket)
→ Fix: use AssumeRole pattern (no bucket policy needed)

PITFALL 5: "Objects written by Account B can't be read by Account A"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: S3 object ownership — objects belong to the WRITING account
→ Account B writes object → Account B owns it → Account A can't read
→ Fix 1: bucket-owner-full-control ACL on put
→ Fix 2: S3 Object Ownership = BucketOwnerEnforced (recommended since 2022)
→ Fix 3: use AssumeRole pattern (writes as Account A's role → A owns object)

PITFALL 6: "Temporary credentials expired mid-Glue-job"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Default session: 1 hour
→ Glue job runs for 2 hours → credentials expire → job fails at hour 1
→ Fix: set DurationSeconds in AssumeRole (max 12 hours for IAM roles)
→ Fix: Glue's native cross-account support handles credential refresh
→ Better: use Glue Connection with cross-account role (Project 60)

PITFALL 7: "Works from CLI but fails from Glue/Lambda"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: CLI uses your user credentials (admin). Glue uses its service role.
→ Trust policy allows your user but NOT the Glue service role
→ Fix: trust policy must list the EXACT ARN of the Glue/Lambda role
→ Always test with the ACTUAL service role, not your admin user

STS LIMITS:
→ AssumeRole: 2,000 calls/second per account (adjustable)
→ Session duration: 1-12 hours (IAM roles), 15 min-12 hours (IAM users)
→ Max active sessions per role: no hard limit

🔑 KEY TAKEAWAY

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ Cross-account S3 access has TWO patterns: Bucket Policy and AssumeRole
→ Pattern 1 (Bucket Policy): simpler — Account A's bucket policy grants Account B
→ Pattern 2 (AssumeRole): more secure — Account B assumes a role IN Account A
→ Cross-account requires BOTH sides to allow (AND logic, not OR)
→ External ID prevents confused deputy attacks (mandatory for third-party access)
→ STS returns TEMPORARY credentials that auto-expire (1-12 hours)
→ CloudTrail logs every AssumeRole call and every S3 access (full audit trail)
→ Least privilege: specific prefixes, specific actions, specific roles
→ NEGATIVE testing is MORE important than positive testing for security
→ Object ownership matters for writes (BucketOwnerEnforced recommended)

PATTERN COMPARISON:
━━━━━━━━━━━━━━━━━
→ Bucket Policy: simpler setup, no STS call, good for native service access
→ AssumeRole: more secure, temporary creds, better audit trail, more flexible
→ Production recommendation: AssumeRole for everything except simple Athena/Glue
