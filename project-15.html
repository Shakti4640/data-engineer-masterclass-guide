CATALOG OUR DATA SO OTHER TOOLS KNOW THE SCHEMA
🟢 Phase 1: Very Easy

🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — data platform growing rapidly
Situation:
→ Data lives in MULTIPLE places now:
   - S3: CSV files (Project 1), daily exports, campaign uploads
   - MariaDB: OLTP tables (Projects 8-9)
   - Redshift: OLAP warehouse (Projects 10-11)

→ Multiple tools need to QUERY this data:
   - Athena: SQL on S3 files (Project 16 — coming next)
   - Glue ETL: transform data between systems (Project 25)
   - Redshift Spectrum: query S3 from Redshift (Project 38)
   - EMR/Spark: big data processing (future)

THE PROBLEM — EVERY TOOL ASKS THE SAME QUESTION:
"What is the SCHEMA of this data?"

ANALYST (trying to use Athena):
"I want to query orders_20250715.csv in S3 with SQL.
 But Athena asks: what are the column names? What are the data types?
 What's the delimiter? Where exactly is the file?
 I have to define all of this MANUALLY every time.
 And if the schema changes, my query breaks."

DATA ENGINEER (building Glue ETL):
"Glue needs to know: what tables exist in MariaDB? What are the 
 column types? What tables exist in Redshift? I have to configure
 all of this in every single Glue job. If I have 20 jobs, that's
 20 places to update when a schema changes."

BI DEVELOPER (connecting Tableau):
"I need a list of all available datasets. Where do I look?
 There's no central place that says 'here's what we have.'"

CTO:
"We need a DATA CATALOG — a central registry of all our datasets.
 Every tool reads from the catalog instead of defining schemas 
 independently. Change the schema in ONE place, all tools adapt."

REQUIREMENTS:
→ Central metadata store for ALL datasets (S3, MariaDB, Redshift)
→ Store: database names, table names, column names, data types
→ Store: physical location (S3 path, JDBC connection)
→ Store: file format (CSV, Parquet, JSON)
→ Accessible by: Athena, Glue, Spectrum, EMR, any Hive-compatible tool
→ Programmable: create/update via API (not just console clicks)

Step 2: Understand AWS Glue Data Catalog structure

  CATALOG (one per AWS account per region)
  └── DATABASE (logical grouping — like a schema)
       └── TABLE (metadata about one dataset)
            ├── Column definitions (name, type)
            ├── Storage location (S3 path, JDBC URL)
            ├── File format (CSV, Parquet, JSON, ORC)
            ├── SerDe (serializer/deserializer)
            ├── Partition keys (for partitioned data)
            └── Table properties (custom key-value metadata)

✅ WHAT SHOULD DO

1. Create SEPARATE databases for different data layers
   → quickcart_raw_data: raw CSV files as-is from source
   → quickcart_curated: cleaned/transformed Parquet (Project 25)
   → quickcart_aggregated: pre-computed summaries (future)
   → This maps to Bronze/Silver/Gold pattern (Project 44)

2. Define columns with CORRECT Glue Catalog types
   → Glue types differ from MariaDB and Redshift types
   → Glue: string, int, bigint, double, float, boolean, date, timestamp
   → MariaDB VARCHAR → Glue string
   → MariaDB DECIMAL → Glue double or decimal(precision, scale)
   → MariaDB DATE → Glue date (or string if format varies)
   → Wrong type = Athena query errors or wrong results

3. Set SerDe (Serializer/Deserializer) correctly
   → CSV: org.apache.hadoop.hive.serde2.OpenCSVSerde
   → Or: org.apache.hadoop.hive.serde2.lazy.LazySimpleSerDe
   → Parquet: org.apache.hadoop.hive.ql.io.parquet.serde.ParquetHiveSerDe
   → JSON: org.openx.data.jsonserde.JsonSerDe
   → Wrong SerDe = all columns NULL or parse errors

4. Handle CSV headers properly
   → If CSV has header row: set "skip.header.line.count" = "1"
   → Without this: first row (header) treated as data
   → Athena returns column names as data values — confusing

5. Use table properties for documentation
   → description: human-readable purpose
   → owner: team responsible
   → source_system: where data originates
   → refresh_frequency: how often data updates
   → These help other teams discover and understand datasets

6. Plan partition keys from the start
   → If data will be partitioned by date (Project 26):
     define partition keys NOW even if not partitioned yet
   → Adding partitions later requires table recreation
   → For now: non-partitioned tables (partition added in Project 26)

7. Tag databases and tables for governance
   → Environment: production/staging/dev
   → Classification: public/internal/confidential
   → Team: data-engineering/marketing/finance
   → Tags enable Lake Formation access control (Project 74)

❌ WHAT SHOULD NOT DO

1. DO NOT create one giant database for everything
   → raw + curated + aggregated in one database = confusion
   → Separate databases = clear boundaries, separate permissions
   → Athena query: FROM quickcart_raw_data.orders (clear provenance)

2. DO NOT use Glue Catalog types that don't match actual data
   → Defining order_date as "date" when CSV has "2025-01-15" string
   → If CSV date format isn't ISO 8601: Athena returns NULL
   → Safe approach: use "string" for dates in CSV, cast in queries
   → Use proper "date" type only for Parquet (has native date type)

3. DO NOT forget to set the InputFormat and OutputFormat
   → InputFormat: how Athena reads the files
   → OutputFormat: how Glue writes the files
   → CSV: TextInputFormat + HiveIgnoreKeyTextOutputFormat
   → Parquet: MapredParquetInputFormat + MapredParquetOutputFormat
   → Wrong format = empty results or errors

4. DO NOT define tables pointing to non-existent S3 paths
   → Catalog table is METADATA — doesn't validate S3 path exists
   → Table created successfully even if S3 path is wrong
   → Athena query returns zero rows — misleading
   → Always verify S3 path exists before creating catalog table

5. DO NOT put file names in the table location
   → Location should be the DIRECTORY (prefix), not the file
   → Correct: s3://bucket/daily_exports/orders/
   → Wrong: s3://bucket/daily_exports/orders/orders_20250715.csv
   → Athena scans ALL files under the directory location

6. DO NOT skip the trailing slash in S3 location
   → s3://bucket/orders  ← may match orders_backup/ too
   → s3://bucket/orders/ ← matches only orders/ prefix
   → Always include trailing slash

7. DO NOT create duplicate table definitions
   → Same table name in same database = error
   → If schema changes: UPDATE existing table, don't create new one
   → Use create_or_update pattern (check exists, then create or update)

🚫 WHEN NOT TO USE GLUE DATA CATALOG

1. When data is only in Redshift and queried only from Redshift
   → Redshift has its own internal catalog (pg_catalog)
   → Glue Catalog needed only for Spectrum external tables
   → Don't duplicate Redshift internal tables in Glue Catalog

2. When data is ephemeral (temporary processing files)
   → Catalog is for persistent, queryable datasets
   → Don't catalog staging files that exist for 5 minutes
   → Catalog the OUTPUT of ETL, not intermediate files

3. When you need real-time schema validation
   → Catalog is eventually consistent (not real-time)
   → Schema changes may take seconds to propagate
   → For real-time validation: use application-level checks

4. When data lives outside AWS
   → Glue Catalog is AWS-native — no support for GCP/Azure storage
   → For multi-cloud: consider Apache Hive Metastore or DataHub
   → Glue Catalog IS a Hive Metastore-compatible implementation

5. When dataset count exceeds 1 million tables per account
   → Glue Catalog soft limit: 1,000,000 tables per account
   → Enterprise: request limit increase or use multiple accounts
   → QuickCart: ~10 tables — nowhere near the limit

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Athena returns all NULL values"
→ Cause #1: wrong SerDe — LazySimpleSerDe for CSV with quoted fields
→ Fix: use OpenCSVSerDe
→ Cause #2: wrong delimiter in SerDe parameters
→ Fix: verify separatorChar matches actual file delimiter
→ Cause #3: column count mismatch between catalog and file
→ Fix: catalog column count must match file column count exactly

PITFALL 2: "Athena returns header row as data"
→ Cause: skip.header.line.count not set in table properties
→ Fix: add "skip.header.line.count": "1" to table Parameters
→ Note: this property works with LazySimpleSerDe and OpenCSVSerDe

PITFALL 3: "Glue Catalog table created but Athena can't find it"
→ Cause: table created in different region than Athena workgroup
→ Fix: Glue Catalog is per-region — create in same region as Athena
→ Debug: aws glue get-table --database-name X --name Y --region Z

PITFALL 4: "Table shows 0 rows even though S3 has files"
→ Cause #1: S3 location path doesn't match actual file location
→ Fix: verify trailing slash and exact path
→ Cause #2: files are in subdirectories not matching partition scheme
→ Fix: for non-partitioned tables, files must be directly under location

PITFALL 5: "DOUBLE column returns NULL for decimal values"
→ Cause: OpenCSVSerDe returns ALL columns as STRING
→ Fix: use string type in catalog, CAST in Athena queries
→ Or: use LazySimpleSerDe (supports native types but no quoting)
→ Best: convert to Parquet (Project 25) — native type support

PITFALL 6: "Can't create table — database doesn't exist"
→ Cause: database must exist before creating tables in it
→ Fix: create database first, then tables
→ API: glue.create_database() before glue.create_table()

PITFALL 7: "Column names have spaces or special characters"
→ Cause: CSV header has "Order ID" instead of "order_id"
→ Glue Catalog: column names should be lowercase, no spaces
→ Fix: use underscore naming in catalog, handle mapping in SerDe
→ Best practice: clean column names during ETL (Project 25)

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ GLUE DATA CATALOG: central metadata store for ALL datasets
   → ONE catalog per account per region (implicit, always exists)
   → Databases: logical groupings (raw, curated, aggregated)
   → Tables: metadata about one dataset (columns, location, format)
   → Tables are POINTERS to data — they don't contain data themselves

→ CATALOG HIERARCHY: Catalog → Database → Table → Columns
   → Databases map to data layers (bronze/silver/gold)
   → Tables map to datasets (orders, customers, products)
   → Columns define structure (name, type, comment)

→ SERDE: the parser that reads raw bytes into structured columns
   → OpenCSVSerde: handles quoted fields, returns strings
   → LazySimpleSerDe: simple delimiter, native types
   → ParquetHiveSerDe: binary format, fastest, type-safe
   → Wrong SerDe = all NULL values or parse errors

→ KEY PROPERTIES for CSV tables:
   → skip.header.line.count = "1" (skip CSV header row)
   → S3 location must end with trailing slash
   → Column order must match file column order
   → Column count must match file column count

→ SCHEMA EVOLUTION: add columns, change types, track versions
   → Glue stores version history automatically
   → New columns in catalog → old files return NULL for that column
   → Version tracking via table Parameters

→ HIVE METASTORE COMPATIBLE: any Hive-compatible tool can read
   → Athena, Glue ETL, Spectrum, EMR Spark — all read same catalog
   → Define once, use everywhere
