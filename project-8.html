✅ WHAT SHOULD DO

1. Install from official Amazon Linux repos (not random downloads)
   → dnf install mariadb105-server — vetted, patched, supported

2. Run mysql_secure_installation IMMEDIATELY after install
   → Set root password
   → Remove anonymous users
   → Remove test database
   → Disallow remote root login

3. Create SEPARATE database users for separate purposes
   → root: admin only, never used by applications
   → app_user: application CRUD, limited privileges
   → etl_user: read-only for data extraction (Project 20, 31)

4. Bind to 127.0.0.1 (localhost) initially
   → Only processes ON this EC2 can connect
   → Change to 0.0.0.0 ONLY when remote access needed + SG locked

5. Configure Security Group for port 3306
   → NEVER open 3306 to 0.0.0.0/0 (the entire internet)
   → Allow only VPC CIDR (10.0.0.0/16) or specific IPs

6. Set up daily automated backups with mysqldump
   → Dump to local file → upload to S3 (reuse Project 1 upload logic)

7. Configure InnoDB buffer pool to ~70% of available RAM
   → t3.medium has 4 GB → set innodb_buffer_pool_size = 2.8G

8. Enable slow query log from Day 1
   → Catches performance problems early
   → Critical for optimization later

9. Use InnoDB engine for ALL tables (not MyISAM)
   → InnoDB: ACID, row-level locking, crash recovery
   → MyISAM: legacy, table-level locking, no transactions

❌ WHAT SHOULD NOT DO

1. DO NOT use root user for application connections
   → root has DROP DATABASE, GRANT ALL privileges
   → One SQL injection = entire database destroyed
   → Create app_user with ONLY needed privileges

2. DO NOT open port 3306 to 0.0.0.0/0 in Security Group
   → Bots scan all AWS IP ranges for open MySQL ports
   → Brute force attacks start within MINUTES
   → 2020: thousands of exposed databases ransomed

3. DO NOT skip mysql_secure_installation
   → Default: no root password, anonymous user can login
   → test database is world-writable
   → This is the #1 MySQL/MariaDB security breach vector

4. DO NOT store database on root EBS volume only
   → Root volume = OS + application + database competing for I/O
   → Better: attach separate EBS volume for /var/lib/mysql (if budget allows)

5. DO NOT use MyISAM engine
   → No transactions, no foreign keys, table-level locking
   → Server crash = potential data corruption
   → InnoDB is default since MariaDB 10.2 — keep it that way

6. DO NOT set innodb_buffer_pool_size to more than 80% of RAM
   → OS needs memory, MariaDB threads need memory
   → OOM killer will terminate MariaDB process
   → t3.medium (4 GB): max 3 GB for buffer pool

7. DO NOT forget to enable binary logging (binlog)
   → Needed for point-in-time recovery
   → Needed for replication (future DR setup — Project 75)
   → Small storage overhead, massive recovery benefit

8. DO NOT run MariaDB and heavy Python ETL simultaneously on t3.medium
   → 4 GB RAM is tight — ETL can starve the database
   → Schedule ETL during off-peak hours

OLTP vs OLAP — CRITICAL DISTINCTION:
→ MariaDB = OLTP (Online Transaction Processing)
  → Fast single-row reads/writes, transactions, real-time app
→ Redshift = OLAP (Online Analytical Processing)
  → Fast aggregations across billions of rows, batch analytics
→ NEVER use Redshift as your app database
→ NEVER run GROUP BY on 100M rows in MariaDB

🚫 WHEN NOT TO USE MariaDB ON EC2

1. When you need 99.99% uptime SLA
   → EC2 MariaDB: single instance = single point of failure
   → Use: RDS Multi-AZ (automatic failover in ~60 seconds)

2. When dataset exceeds 1 TB
   → EC2 EBS max practical: ~16 TB, but management nightmare
   → MariaDB single instance struggles above 1 TB
   → Use: Aurora (128 TB auto-scaling) or Redshift (for analytics)

3. When you need read replicas across regions
   → Manual replication setup on EC2 is fragile
   → Use: RDS with read replicas (one-click)

4. When you need automatic backups with point-in-time recovery
   → Manual mysqldump + S3 is your only option on EC2
   → Use: RDS (automated daily snapshots + PITR to any second)

5. When compliance requires encryption at rest + audit logging
   → Possible on EC2 but manual (LUKS, audit plugin)
   → Use: RDS (one checkbox enables encryption, CloudTrail integration)

6. When your team has zero database administration experience
   → EC2 MariaDB requires: patching, monitoring, tuning, backup, recovery
   → Use: RDS — AWS handles all of this

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Can't connect to MySQL server on 'localhost'"
→ Cause: MariaDB service not running
→ Fix: sudo systemctl start mariadb
→ Check: sudo systemctl status mariadb

PITFALL 2: "Access denied for user 'root'@'localhost'"
→ Cause: root password set but you're using wrong one
→ Fix: sudo mysql (uses OS socket auth when run as root)
→ Then: ALTER USER 'root'@'localhost' IDENTIFIED BY 'newpassword';

PITFALL 3: "Can't connect from Python on same EC2"
→ Cause: using host='localhost' tries Unix socket, 
   but pymysql defaults to TCP
→ Fix: use host='127.0.0.1' (forces TCP) or specify socket path

PITFALL 4: "Table uses MyISAM instead of InnoDB"
→ Cause: old default or explicit ENGINE=MyISAM in CREATE TABLE
→ Fix: ALTER TABLE t ENGINE=InnoDB;
→ Verify: SHOW TABLE STATUS WHERE Name='tablename';

PITFALL 5: "Database is slow after a few months"
→ Cause: no indexes on columns used in WHERE clauses
→ Fix: EXPLAIN SELECT ... → look for "type: ALL" (full scan)
→ Add index: CREATE INDEX idx_col ON table(column);

PITFALL 6: "EC2 reboot lost all my data"
→ Cause: data was on instance store (ephemeral) not EBS
→ Fix: ensure /var/lib/mysql is on EBS volume
→ Instance store data is LOST on stop/terminate

PITFALL 7: "Server ran out of disk space — database corrupted"
→ Cause: EBS volume 100% full, InnoDB can't write redo log
→ Prevention: CloudWatch alarm on disk usage > 80%
→ Fix: extend EBS volume + resize filesystem

PITFALL 8: "pymysql.err.OperationalError: too many connections"
→ Cause: connections opened but never closed
→ Fix: always use try/finally or context managers
→ Check: SHOW PROCESSLIST; to see active connections

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY MariaDB IS HEALTHY:

1. Service status:
   → sudo systemctl status mariadb
   → Active: active (running) since...

2. Connection test:
   → mysql -u root -p -e "SELECT 1;"
   → Should return "1"

3. Database size check:
   → SELECT table_schema, 
            ROUND(SUM(data_length + index_length) / 1024 / 1024, 2) AS size_mb
     FROM information_schema.tables
     GROUP BY table_schema;

4. Buffer pool hit ratio:
   → SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_read%';
   → Hit ratio = 1 - (reads / read_requests)
   → Should be > 0.99 (99%)

5. Active connections:
   → SHOW PROCESSLIST;
   → Shows who is connected and what they're doing

6. Slow query log:
   → /var/log/mariadb/slow-query.log
   → Any query taking > 2 seconds logged here

7. Error log:
   → /var/log/mariadb/mariadb.log
   → Check after any unexpected behavior

→ Create separate users for separate purposes (app, etl, admin)
→ NEVER open port 3306 to 0.0.0.0/0
→ Backup to S3 daily — reuse Project 1 upload patterns
