🟡 HOW TO APPROACH IT

Step 1: Understand the IAM chain for cross-account Glue
→ Glue job runs as: Account B's "GlueRole-B"
→ GlueRole-B has permission to: sts:AssumeRole on Account A's cross-account role
→ Inside the job: call sts:AssumeRole → get temp credentials for Account A
→ Use temp credentials: read S3 + Catalog from Account A
→ Use GlueRole-B's own credentials: write to Account B's Redshift

Step 2: Understand Glue's native cross-account Catalog support
→ Glue from_catalog() supports catalog_id parameter
→ But: Glue's service role needs direct permissions on Account A's Catalog
→ Alternative: AssumeRole in job code, then use boto3 for Catalog + Spark for S3
→ We implement BOTH approaches for comparison

Step 4: Design the orchestration

🔵 WHAT COULD BE DONE

Option A: Account A runs ETL and pushes results to Account B
→ Account A owns the transformation logic
→ Problems: Account A doesn't know Account B's requirements
→ Every new Account B requirement = change request to Account A team
→ Tight coupling between teams

Option B: UNLOAD from Account A's Redshift → S3 → COPY to Account B's Redshift
→ From Project 58 discussion — already rejected
→ 500 GB transfer, 75-min stale, no custom transforms

Option C: Account B reads via Data Sharing, transforms locally via Redshift SQL
→ Account B queries Account A's Redshift → joins with local → INSERT INTO local table
→ Works but: Redshift SQL is limited for complex transforms
→ No Spark, no ML libraries, no complex Python logic
→ Good for simple aggregations, bad for complex ETL

Option D: Cross-account Glue ETL (this project) ✅
→ Glue in Account B: reads from A, transforms in B, writes to B's Redshift
→ Full Spark/Python capabilities for complex transforms
→ Account B owns the transformation logic entirely
→ Uses established cross-account patterns (Projects 56-59)
→ Event-driven: triggered by Account A's pipeline completion

Option E: Lambda-based lightweight ETL
→ Lambda reads from A, transforms, writes to B
→ Limited: 15-min timeout, 10 GB memory max
→ Good for small datasets (<1 GB), bad for large transforms
→ Not suitable for 500 GB Gold layer processing

DECISION: Option D — Cross-account Glue ETL
→ Reason: full Spark power, Account B owns logic, event-driven, scalable
→ Combines ALL cross-account patterns from Phase 4

✅ WHAT SHOULD DO

1. Use AssumeRole INSIDE Glue job code (not Glue Connection)
   → Glue Connections for cross-account: limited and finicky
   → AssumeRole in code: full control, explicit, debuggable
   → Create boto3 session with temp credentials for Account A
   → Use default session for Account B operations

2. Separate cross-account and local operations clearly
   → First: do ALL Account A reads (using assumed credentials)
   → Convert to Spark DataFrames (now in Account B's memory)
   → Then: do ALL Account B operations (using local credentials)
   → Never mix credential contexts in the same API call

3. Cache Account A's data in Account B's S3 staging area
   → Read from Account A → write to s3://acct-b/staging/
   → Subsequent operations read from staging (local, fast)
   → Benefits: if Glue job fails mid-transform, don't re-read cross-account
   → Cleanup: delete staging after successful load to Redshift

4. Handle credential expiry for long-running jobs
   → Default AssumeRole session: 1 hour
   → Glue job may run 2+ hours for large datasets
   → Set DurationSeconds=14400 (4 hours) when assuming role
   → Or: re-assume role periodically in long jobs

5. Pass cross-account config via Glue job arguments
   → --source_catalog_id, --source_database, --cross_account_role_arn
   → --external_id (for confused deputy prevention)
   → Don't hardcode account IDs in job script
   → Same script works for dev/staging/prod by changing arguments

6. Add comprehensive logging for cross-account operations
   → Log: "Assuming role ARN: ..."
   → Log: "Reading table from Catalog ID: ..."
   → Log: "Read N rows from Account A's S3"
   → Log: "Writing N rows to Account B's Redshift"
   → Cross-account debugging is HARD without good logs

7. Implement idempotent writes to Redshift
   → If job reruns (retry after failure): don't duplicate data
   → Use: TRUNCATE + INSERT, or MERGE (upsert) pattern
   → Or: staging table → DELETE existing → INSERT new → swap
   → From Project 49: Redshift stored procedures for atomic operations

8. Monitor BOTH accounts' CloudTrail
   → Account A: who assumed the cross-account role? When?
   → Account A: which S3 objects were read? How much data?
   → Account B: Glue job execution, Redshift writes
   → Correlated view: full audit trail of cross-account ETL

❌ WHAT SHOULD NOT DO

1. DO NOT store Account A's credentials in Glue job code
   → No hardcoded access keys EVER
   → Use AssumeRole for temporary credentials
   → Credentials rotate automatically

2. DO NOT read Account A's data directly into Redshift COPY
   → Redshift COPY from cross-account S3: requires Redshift IAM role
   → Complex: Redshift role → AssumeRole → Account A's role → S3
   → Simpler: Glue reads S3 → writes to Account B's staging S3 → COPY locally

3. DO NOT skip error handling for cross-account operations
   → Cross-account failures are MORE common than same-account:
     - Role trust policy changed → AccessDenied
     - Account A's data moved → FileNotFoundError
     - Session expired → ExpiredTokenException
   → Every cross-account call needs try/except with clear error message

4. DO NOT assume Account A's schema never changes
   → Account A drops a column → Glue job breaks
   → Account A renames a table → Glue job breaks
   → Defensive coding: validate schema before processing
   → Alert if schema drift detected (from Project 45)

5. DO NOT run cross-account Glue jobs as admin/superuser role
   → GlueRole-B should have MINIMUM permissions needed
   → Read-only on Account A (via cross-account role)
   → Write-only to specific Account B resources
   → Principle of least privilege across accounts

6. DO NOT process ALL of Account A's data every run
   → Use incremental processing: only new/changed data
   → Glue Job Bookmarks (from Project 33) for S3 sources
   → Watermark columns for database sources (from Project 42)
   → Reduces: cross-account data transfer, compute cost, run time

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Glue reads Catalog successfully but S3 read fails"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Catalog permission ≠ S3 permission
→ GlueRole-B can read Catalog (Project 57) but NOT S3 (Project 56 not done for this role)
→ Catalog returns S3 location → Spark tries to read → AccessDenied on S3
→ Fix: ensure BOTH Catalog AND S3 cross-account access for the same role

PITFALL 2: "Hadoop S3A credentials override default credentials"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ After setting Account A's creds in Hadoop config for S3 read
→ ALL subsequent S3 operations use Account A's credentials
→ Writing to Account B's S3 fails (Account A's role can't write to B)
→ Fix: RESET Hadoop config after cross-account reads are done
→ Or: use separate Spark sessions (complex but clean)

PITFALL 3: "AssumeRole credentials expire mid-job"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Default session: 1 hour
→ Large dataset: 500 GB read takes 90 minutes
→ Credentials expire at 60 minutes → read fails at 60%
→ Fix: set DurationSeconds=14400 (4 hours) in AssumeRole
→ Or: complete ALL cross-account reads in first phase, then transform locally

PITFALL 4: "Glue job writes to Redshift but data is duplicated"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: job reruns after partial failure
→ First run: wrote 50% of data → failed → retried
→ Second run: wrote 100% of data → first 50% now duplicated
→ Fix: use staging table pattern:
  1. Write to staging_table
  2. BEGIN TRANSACTION
  3. DELETE FROM target WHERE date = processing_date
  4. INSERT INTO target SELECT * FROM staging_table
  5. COMMIT
  6. DROP staging_table

PITFALL 5: "Cross-account Glue job takes 3x longer than same-account"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: cross-account S3 reads have higher latency
→ Same-region: ~5-10ms additional latency per request
→ But: Spark reads many small files → thousands of requests → adds up
→ Fix: Account A stores data as large Parquet files (from Project 62)
→ Fix: use s3a:// protocol with readahead and connection pooling
→ Fix: cache in Account B's staging S3 before processing

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ Cross-account Glue ETL uses AssumeRole INSIDE job code for Account A access
→ TWO credential contexts in ONE job: Account A (temp) + Account B (default)
→ Hadoop S3A configuration switches between cross-account and local credentials
→ CRITICAL: reset Hadoop credentials after cross-account reads complete
→ Read ALL cross-account data first → then process locally → then write locally
→ Set DurationSeconds=14400 (4 hours) to prevent credential expiry mid-job
→ Idempotent writes: DELETE + INSERT pattern prevents duplicates on retry
→ Pass config via Glue job arguments (not hardcoded) for environment portability
→ Event-driven trigger: SQS event → Step Functions → Glue job (zero polling)
→ Comprehensive logging for cross-account operations (debugging is harder)
→ Monitor BOTH accounts' CloudTrail for complete audit trail
