🟡 HOW TO APPROACH IT

Step 1: Understand what Step Functions IS
→ Serverless workflow orchestration service
→ Define workflows as STATE MACHINES using JSON (Amazon States Language)
→ Each STATE is a step: run a task, make a decision, wait, branch
→ Visual diagram auto-generated from definition
→ Built-in retry, catch, timeout per state
→ Execution history: every state transition logged
→ Integrates directly with 200+ AWS services (Glue, Athena, SNS, Lambda)

Step 2: Understand State Types
→ Task: do something (run Glue job, publish SNS, execute Athena query)
→ Choice: if/else branching (check result, route accordingly)
→ Parallel: run multiple states simultaneously
→ Wait: pause for N seconds or until a timestamp
→ Pass: transform data, pass through
→ Map: iterate over a list (for-each loop)
→ Succeed: mark execution as successful
→ Fail: mark execution as failed

Step 3: Design the state machine
→ Sequential: Crawl → Clean → Transform → Validate (must run in order)
→ Choice: If quality check passes → continue, else → fail
→ Parallel: Athena CTAS and Redshift COPY can run simultaneously
→ Catch: Any failure → notify → end

Step 4: Understand service integrations
→ Synchronous (.sync): Step Functions WAITS for Glue job to complete
→ Asynchronous: Step Functions starts job and moves on immediately
→ We need .sync for pipeline ordering (wait for each step to finish)
→ Glue: StartJobRun.sync — waits for job to complete or fail
→ Athena: StartQueryExecution.sync — waits for query to finish
→ SNS: Publish — fire and forget (notification)

Step 5: Design error handling
→ Retry: transient failures (network timeout, throttling)
→ Catch: permanent failures (bad data, permission error)
→ Timeout: hung jobs (Glue job running for 3 hours)
→ Each state can have its own retry/catch/timeout independently

🔵 WHAT COULD BE DONE

Option A: Cron + shell script orchestration (current — broken)
→ Hardcoded timing, no dependency awareness
→ No retry, no conditional logic, no visibility
→ The source of Tuesday's incident

Option B: Glue Workflows (Project 35)
→ Can chain Crawlers + ETL Jobs with triggers
→ Limited: only Glue components (no Athena, no Redshift, no SNS)
→ No conditional logic (Choice states)
→ No parallel execution with non-Glue services
→ Good for Glue-only pipelines, insufficient for full platform

Option C: Apache Airflow (MWAA — Managed Workflows for Apache Airflow)
→ Industry standard for data pipeline orchestration
→ DAG-based (Directed Acyclic Graph)
→ Python code defines workflows
→ POWERFUL but complex: requires Airflow knowledge, running cluster
→ Cost: $0.49/hr minimum = ~$360/month (just for scheduler)
→ Overkill for QuickCart's 6-step pipeline

Option D: AWS Step Functions (this project) ✅
→ Serverless — no cluster, no infrastructure
→ Visual workflow designer
→ Native integration with Glue, Athena, SNS, Lambda, Redshift
→ Built-in retry, catch, timeout per state
→ Execution history with full state transition log
→ Cost: $0.025 per 1,000 state transitions → pennies
→ Perfect for 5-15 step data pipelines

Option E: EventBridge Pipes + Rules
→ Event-driven: S3 event → trigger next step
→ Good for simple 2-3 step chains
→ Complex for 6+ step pipelines with branching
→ No visual workflow, harder to debug

DECISION: Option D — Step Functions
→ Reason: serverless, visual, native AWS integration, cheap, right complexity level
→ Can always migrate to Airflow later if pipeline grows to 50+ steps

✅ WHAT SHOULD DO

1. Use Standard Workflows (not Express)
   → Standard: up to 1 year execution, exactly-once, full history
   → Express: up to 5 minutes, at-least-once, sampled history
   → Data pipelines run 30-90 minutes → Standard is correct
   → Express is for high-volume, short-duration (API orchestration)

2. Use .sync integration pattern for Glue and Athena
   → Without .sync: Step Functions starts Glue job and moves to next state
   → With .sync: Step Functions WAITS for Glue job to complete
   → Without .sync: pipeline proceeds before previous step finishes → disaster
   → Resource ARN format: "arn:aws:states:::glue:startJobRun.sync"

3. Set TimeoutSeconds on every Task state
   → Glue job hangs for 6 hours → entire pipeline stuck
   → Timeout: 3600 (1 hour) for Glue ETL
   → Timeout: 600 (10 min) for Crawler
   → Timeout: 1800 (30 min) for Athena CTAS
   → On timeout: state fails → Catch handles it

4. Add Retry with exponential backoff for transient errors
   → Glue: "Glue.ConcurrentRunsExceededException" → retry 3 times
   → Athena: "TooManyRequestsException" → retry 3 times
   → BackoffRate: 2.0 → wait 2s, 4s, 8s between retries
   → MaxAttempts: 3 → fail permanently after 3 retries

5. Add Catch on every Task state
   → Catch "States.ALL" → route to failure notification state
   → Include error details in notification (what failed, why, where)
   → Catch is the SAFETY NET — nothing should fall through unhandled

6. Pass pipeline context through states
   → Input to state machine: {"execution_date": "2025-01-15", "pipeline": "nightly"}
   → Each state receives this context and passes it to the next
   → Failure notification includes: which execution, which date, which step

7. Use EventBridge Rule to trigger on schedule
   → cron(0 1 * * ? *) = 1:00 AM UTC daily
   → EventBridge → Step Functions StartExecution
   → Replaces ALL individual crons with ONE trigger

8. Tag executions for cost and debugging
   → Each execution gets a unique name: "nightly-2025-01-15-001"
   → Tags: Pipeline=nightly-etl, Environment=prod

❌ WHAT SHOULD NOT DO

1. DO NOT use asynchronous integration for sequential steps
   → "arn:aws:states:::glue:startJobRun" (async) → moves on immediately
   → "arn:aws:states:::glue:startJobRun.sync" (sync) → waits for completion
   → Missing ".sync" = pipeline doesn't wait = chaos

2. DO NOT put business logic in the state machine definition
   → State machine = orchestration (WHAT runs in WHAT order)
   → Lambda/Glue = business logic (HOW to process data)
   → Don't try to transform data inside Step Functions Pass states
   → Keep state machine simple, delegate complexity to services

3. DO NOT create monolithic state machines with 100+ states
   → Hard to debug, hard to maintain, hard to visualize
   → Split into sub-state machines (nested execution)
   → Main: Ingest → Transform → Serve (3 states calling child state machines)
   → Child: each contains 5-10 detailed states

4. DO NOT hardcode resource ARNs in state machine definition
   → Use CloudFormation/CDK parameters or SSM Parameter Store
   → Hardcoded ARNs break when deploying to different accounts/regions
   → Reference via input parameters or Lambda lookups

5. DO NOT forget to handle BOTH success and failure paths
   → Happy path only → first failure crashes execution with no notification
   → Every Task must have Catch → route to notification → graceful failure

6. DO NOT use Wait states for polling
   → Bad: Wait 5 min → Check if Glue done → Wait 5 min → Check again
   → Good: Use .sync integration → Step Functions handles polling internally
   → .sync uses optimized polling with service integrations

7. DO NOT ignore execution history limits
   → Standard Workflow: 25,000 events per execution
   → Each state transition = multiple events
   → 100-state pipeline with retries could exceed limit
   → Keep pipeline under 50 states → well within limits

🚫 WHEN NOT TO USE STEP FUNCTIONS

1. When pipeline has >100 steps with complex cross-dependencies
   → Step Functions supports sequential, parallel, branching
   → But NOT arbitrary DAGs (no "step 7 depends on steps 2 AND 5")
   → Use: Airflow (MWAA) for true DAG orchestration

2. When execution must run for >1 year
   → Standard Workflow max: 1 year
   → Use: custom orchestration with DynamoDB state tracking

3. When you need human-in-the-loop approval
   → Step Functions can Wait for callback, but complex to implement
   → Use: Step Functions + Activity Task + manual approval Lambda
   → Or: Airflow with built-in human approval operators

4. When cost is sensitive at very high execution volume
   → 10,000 executions/day × 20 states × $0.025/1000 = $5/day
   → At 1M executions/day: $500/day → consider alternatives
   → For data pipelines: typically 1-10 executions/day → negligible cost

5. When team already has deep Airflow expertise
   → Don't force Step Functions if team knows Airflow
   → Airflow has richer scheduling, better DAG visualization
   → But: MWAA cost ($360+/month) vs Step Functions ($0.50/month)

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Glue job started but Step Functions moved to next state immediately"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: used "glue:startJobRun" instead of "glue:startJobRun.sync"
→ Without .sync: API returns immediately with RunId
→ Next state runs before Glue job finishes
→ Fix: ALWAYS use .sync for jobs that must complete before proceeding

PITFALL 2: "State machine fails with States.Permissions"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Step Functions IAM role missing permissions for target service
→ Needs: glue:StartJobRun, glue:GetJobRun (for .sync polling)
→ Needs: athena:StartQueryExecution, athena:GetQueryExecution
→ Needs: sns:Publish
→ Fix: attach comprehensive policy to Step Functions execution role

PITFALL 3: "Execution stuck in running state for hours"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: .sync integration waiting for job that will never complete
→ Glue job hung, Athena query stuck, service outage
→ Fix: ALWAYS set TimeoutSeconds on every Task state
→ Without timeout: execution waits up to 1 year (Standard max)

PITFALL 4: "Retry keeps retrying a permanently failed job"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: retrying on ALL errors including permanent failures
→ Bad data → Glue fails → retry → fails again → retry → fails again
→ Fix: retry ONLY on transient errors:
  "Glue.ConcurrentRunsExceededException" (temporary)
  "Glue.InternalServiceException" (temporary)
→ DON'T retry: "Glue.InvalidInputException" (permanent — bad config)

PITFALL 5: "Choice state always takes wrong branch"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: condition references wrong path in input JSON
→ Lambda returns: {"body": {"quality_passed": true}}
→ Choice checks: $.quality_passed (wrong — missing .body)
→ Should check: $.body.quality_passed
→ Fix: test with exact JSON structure Lambda returns

PITFALL 6: "Parallel state fails but only one branch had an error"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Behavior: if ANY branch fails, entire Parallel state fails
→ Other branches may still be running → Step Functions cancels them
→ This is BY DESIGN — Parallel is all-or-nothing
→ If branches should be independent: use separate Task states (not Parallel)
→ Or: add Catch on each branch independently

→ Step Functions orchestrates multi-service pipelines as visual state machines
→ Amazon States Language (ASL): JSON-based workflow definition
→ State types: Task, Choice, Parallel, Pass, Wait, Succeed, Fail

→ TimeoutSeconds: prevent hung states from blocking pipeline forever
→ Choice: conditional branching (DQ pass → continue, DQ fail → stop)
→ Parallel: run independent steps simultaneously (Athena + Redshift)
→ EventBridge Rule replaces ALL individual crons with ONE trigger
→ Execution history shows every state transition with input/output
→ Time saved: 5+ hours of buffer time eliminated
→ Cost: ~$0.01/month for daily execution → essentially FREE
