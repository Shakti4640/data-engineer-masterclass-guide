🟡 HOW TO APPROACH IT

Step 1: Understand Data Mesh Principles (Zhamak Dehghani, 2019)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ PRINCIPLE 1: Domain Ownership
  Each team owns the full lifecycle of their data
  → Not "central data team does everything"

→ PRINCIPLE 2: Data as a Product
  Each domain publishes curated, documented, SLA-backed datasets
  → Not raw dumps — polished, versioned, discoverable

→ PRINCIPLE 3: Self-Serve Data Platform
  Central team provides tools, not transforms
  → Shared infra (accounts, catalog, IAM) — not shared pipelines

→ PRINCIPLE 4: Federated Computational Governance
  Global standards (encryption, naming, quality) — local execution
  → Central sets rules, domains enforce them

Step 2: Map Principles to AWS Services
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Domain Ownership      = Separate AWS Account per domain
→ Data as Product       = S3 (Bronze/Silver/Gold) + Glue Catalog per account
→ Self-Serve Platform   = Central Governance Account with Lake Formation
→ Cross-Domain Access   = Cross-Account S3 + Glue Catalog + Redshift Data Sharing
→ Event Notification    = Cross-Account SNS → SQS (data product published)
→ Federated Governance  = Lake Formation tags + Resource Policies

🔵 WHAT COULD BE DONE

OPTION A: Centralized Data Lake (Traditional)
→ One account, one S3 bucket, one Glue Catalog
→ Central team does all ETL
→ Problem: bottleneck, no ownership, no accountability
→ ❌ REJECTED — this is what they're escaping

OPTION B: Copy-Based Sharing
→ Each domain copies data to consumer accounts
→ Problem: data duplication, staleness, storage cost explosion
→ Problem: 4 domains × 3 consumers = 12 copies of same data
→ ❌ REJECTED — doesn't scale

OPTION C: Cross-Account Access-Based Sharing (Data Mesh) ✅
→ Data stays in producer's account
→ Consumers READ from producer's S3 + Glue Catalog
→ No copies, always fresh, single source of truth
→ Governance account controls who reads what
→ ✅ SELECTED

OPTION D: Redshift Data Sharing Only
→ Works for Redshift-to-Redshift
→ Doesn't cover S3/Athena/Glue consumers
→ Partial solution — used WITHIN Option C for Redshift consumers
→ ✅ INCLUDED as component

DECISION: Option C + Option D component
→ S3 cross-account for data lake layer
→ Glue Catalog cross-account for schema discovery
→ Redshift Data Sharing for warehouse consumers
→ SNS→SQS cross-account for event notifications
→ Lake Formation for centralized permission governance

✅ WHAT SHOULD DO

1. ONE AWS ACCOUNT PER DOMAIN — strict blast radius isolation
   → Orders Domain cannot accidentally affect Customer Domain
   → Billing separated per account automatically

2. STANDARDIZE data product structure across ALL domains
   → s3://{domain}-data-{env}/{layer}/{entity}/year=YYYY/month=MM/day=DD/
   → Glue Catalog: database = {domain}_{layer} → table = {entity}
   → Parquet + Snappy compression for Gold layer

3. CENTRAL GOVERNANCE ACCOUNT manages:
   → AWS Organizations (SCPs)
   → Lake Formation permissions
   → Cross-account IAM roles (trust policies)
   → SNS topic for data product events

4. PUBLISH-SUBSCRIBE pattern for data product availability:
   → Producer finishes ETL → publishes to SNS
   → Consumer SQS queues receive notification
   → Consumer triggers their downstream pipeline

5. TAG everything:
   → domain=orders, layer=gold, data_product=daily_orders, version=v2
   → Tags drive Lake Formation access control (Project 74)

6. DATA PRODUCT REGISTRY:
   → DynamoDB table in Governance account
   → Tracks: product name, owner, S3 path, SLA, quality score, version

7. EACH PRODUCER runs their own:
   → Glue Crawlers (schema detection)
   → Glue ETL (Bronze → Silver → Gold) — Project 44
   → Glue Data Quality (validation) — Project 68
   → Only AFTER quality passes → publish event

❌ WHAT SHOULD NOT DO

1. DO NOT create a "shared services" account that does all ETL
   → This recreates the central bottleneck
   → Domains must own their pipelines

2. DO NOT give consumers WRITE access to producer S3
   → Read-only cross-account access
   → Consumers write to THEIR OWN account only

3. DO NOT skip the Governance account
   → Without central permission control = chaos
   → "Everyone can read everything" is NOT a data mesh

4. DO NOT copy data between accounts for sharing
   → Cross-account S3 reads are free (same region)
   → Copying doubles storage cost AND introduces staleness

5. DO NOT let each domain invent their own folder structure
   → Central platform team defines the standard
   → Domains execute it in their accounts

6. DO NOT use cross-account IAM users
   → Always use cross-account IAM ROLES with AssumeRole
   → Never share access keys between accounts (Project 56)

7. DO NOT skip SNS notification on data product publish
   → Without events, consumers resort to polling or guessing
   → Event-driven = reliable, immediate, auditable

8. DO NOT expose raw Bronze/Silver layers as data products
   → Only Gold layer is a data product
   → Bronze/Silver are internal to the domain

🚫 WHEN NOT TO USE DATA MESH

1. SMALL TEAM (< 15 engineers)
   → Overhead of multi-account > benefit
   → Stick with single account + proper IAM
   → Data Mesh is an organizational pattern, not just technical

2. SINGLE DOMAIN
   → If all data is "e-commerce" with no clear boundaries
   → Mesh needs domain decomposition to be meaningful

3. NO PLATFORM TEAM CAPACITY
   → Data Mesh requires a platform team to build self-serve tooling
   → Without it, each domain reinvents the wheel

4. COMPLIANCE REQUIRES SINGLE ACCOUNT
   → Some regulated industries mandate single-account control
   → Data Mesh CAN work in single account (IAM-isolated)
   → But loses blast radius benefit

5. REAL-TIME STREAMING PRIMARY USE CASE
   → Data Mesh is primarily batch/near-real-time oriented
   → Streaming mesh needs Kafka + Schema Registry (different pattern)

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Access Denied" — but policies LOOK correct
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cross-account S3 requires BOTH:
  a) Bucket policy in producer account (resource-based)
  b) IAM role in consumer account (identity-based)
→ Missing EITHER one = Access Denied
→ Debug: use CloudTrail in BOTH accounts to see which check failed

PITFALL 2: Glue Catalog cross-account — "Database not found"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Must specify CatalogId (account ID) when referencing external catalog
→ Without CatalogId, Glue looks in LOCAL catalog only
→ Resource Policy on producer's Glue Catalog must explicitly allow consumer

PITFALL 3: SNS cross-account publish — "AuthorizationError"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Topic Policy must allow sns:Publish from producer account
→ Producer's IAM role must have sns:Publish on the topic ARN
→ BOTH policies checked — like S3 cross-account (Project 56 pattern)

PITFALL 4: Data Product published but consumers don't see it
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Glue Crawler in producer account hasn't run yet
→ Catalog table not updated with new partitions
→ Fix: run MSCK REPAIR or explicit add_partition after ETL (Project 26)

PITFALL 5: "Confused Deputy" attack — one consumer impersonates another
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ ALWAYS use ExternalId in trust policies
→ Without ExternalId, any service in consumer account can assume the role
→ With ExternalId, only the service that knows the secret string can

PITFALL 6: Circular dependencies between domains
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Orders needs Customers, Customers needs Orders
→ Fix: define clear data product contracts with SLAs
→ Use Gold layer snapshots, not live cross-queries
→ Accept T+1 latency for cross-domain references

PITFALL 7: Lake Formation + IAM policy conflict
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ When Lake Formation is enabled, IAM policies for Glue are IGNORED
→ Only Lake Formation grants apply
→ Mixing both = unpredictable access denials
→ Rule: if using Lake Formation, manage ALL Glue permissions there

📈 SCALABILITY CONSIDERATIONS

SCALING DIMENSION 1: Number of Domains
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ 5 domains: manageable manually
→ 20 domains: need automation (Terraform/CDK for account setup)
→ 50+ domains: need self-service portal for domain onboarding
→ Each new domain = new account + standard IAM roles + Glue Catalog + S3

SCALING DIMENSION 2: Number of Data Products
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ 10 products: DynamoDB registry works fine
→ 100 products: need searchable catalog UI (AWS DataZone or custom)
→ 500+ products: need automated lineage tracking (Project 77)

SCALING DIMENSION 3: Cross-Account IAM Roles
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ N producers × M consumers = N×M role pairs
→ 5×5 = 25 roles — manageable
→ 20×20 = 400 roles — MUST automate with Terraform
→ Alternative: use Lake Formation cross-account grants (reduces IAM complexity)

SCALING DIMENSION 4: Event Volume
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ 10 products published daily: single SNS topic fine
→ 1000 products/hour: need SNS message filtering (Project 51)
→ Consumer SQS queues need DLQ (Project 50) for failed processing

SCALING DIMENSION 5: Query Volume
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Athena cross-account: scales infinitely (serverless)
→ Redshift Spectrum: scales with cluster size
→ Redshift Data Sharing: consumer needs own Redshift (cost per consumer)

COST MODEL:
→ S3 storage: paid by producer (they own the bucket)
→ S3 reads: paid by REQUESTER if requester-pays enabled
→ Athena scans: paid by consumer (their workgroup)
→ Glue ETL: paid by the account running the job
→ SNS/SQS: negligible at this volume
→ IAM/STS: free
