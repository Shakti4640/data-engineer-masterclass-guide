🔵 WHAT COULD BE DONE

Option A: All DynamicFrame transforms (Glue-native only)
→ Filter, Map, ApplyMapping, ResolveChoice, DropNullFields
→ Pros: integrates with Glue Catalog, Job Bookmarks
→ Cons: limited join/aggregation support, verbose syntax

Option B: All PySpark DataFrame transforms
→ filter(), join(), groupBy(), window(), withColumn()
→ Pros: full Spark power, familiar API, rich functions
→ Cons: loses Glue-specific features (need DynamicFrame conversion)

Option C: SparkSQL transforms
→ Register DataFrames as temp views → run SQL
→ Pros: familiar for SQL developers, readable
→ Cons: harder to parameterize, less composable

Option D: Hybrid approach (RECOMMENDED)
→ READ with DynamicFrame (bookmark support — Project 33)
→ CONVERT to DataFrame for transforms (full Spark power)
→ CONVERT back to DynamicFrame for write (Glue Catalog integration)
→ Best of both worlds

DECISION: Option D — Hybrid DynamicFrame + PySpark
→ Read/Write via DynamicFrame (Glue features)
→ Transform via PySpark DataFrame (Spark power)
→ This is how production Glue jobs are actually written

✅ WHAT SHOULD DO

1. Apply transforms in correct ORDER
   → Clean BEFORE join (reduce data volume for join)
   → Deduplicate BEFORE join (avoid fan-out from dupes)
   → Join BEFORE aggregate (need joined columns for GROUP BY)
   → Aggregate LAST (final summary)

2. Cache/persist DataFrames used in multiple joins
   → orders_df is joined with BOTH products AND customers
   → Without cache: Spark re-reads from S3 for each join
   → df.cache() keeps it in memory across multiple actions

3. Broadcast small dimension tables
   → products: 50K rows (~2 MB) — tiny
   → customers: 2M rows (~200 MB) — medium
   → Broadcast products → avoids shuffle in join
   → broadcast(products_df) tells Spark to send to all executors

4. Handle schema mismatches with ResolveChoice
   → Bronze data may have inconsistent types across partitions
   → Same column: STRING in one file, INT in another
   → ResolveChoice resolves ambiguity (covered in Project 45 deeply)

5. Write silver tables with OVERWRITE per partition
   → mode("overwrite") + partitionBy("year", "month", "day")
   → Overwrites ONLY today's partition, preserves historical
   → Safe for re-runs (idempotent)

6. Register silver tables in Glue Catalog
   → Athena and Redshift Spectrum need catalog entries
   → Use Glue Crawler or manual registration (Project 23 pattern)

7. Validate row counts at each stage
   → After read: log count
   → After filter: log count + how many removed
   → After dedup: log count + how many duplicates
   → After join: log count + verify no fan-out

❌ WHAT SHOULD NOT DO

1. DO NOT join before cleaning
   → Dirty data + join = amplified mess
   → NULL customer_ids joined with customers table = unexpected NULLs
   → Clean first, then join

2. DO NOT use collect() on large DataFrames
   → collect() pulls ALL data to driver node memory
   → 45M rows × 20 columns = driver OOM crash
   → Use show(5), head(5), take(5) for sampling

3. DO NOT create cartesian/cross joins accidentally
   → join without condition = every row × every row
   → 45M × 2M = 90 TRILLION rows → job crashes
   → Always specify join condition explicitly

4. DO NOT ignore join type implications
   → INNER join: loses orders with unknown product_id
   → LEFT join: keeps all orders, NULL for unknown products
   → For silver layer: LEFT join preserves all transactions

5. DO NOT aggregate at too fine a granularity
   → GROUP BY order_id, customer_id, product_id, date, hour, minute
   → Result: same number of rows as input → no aggregation benefit
   → Aggregate at business-meaningful level: category + month

6. DO NOT hardcode column names in transforms
   → Source schema may change (new columns added)
   → Use column lists from config, not hardcoded strings
   → Project 45 covers schema evolution deeply

7. DO NOT write all silver tables to one S3 prefix
   → silver/everything/*.parquet → impossible to query one table
   → silver/order_details/*.parquet → clean separation
   → Each logical table = separate S3 prefix = separate Catalog table

8. DO NOT skip null handling in join keys
   → NULL == NULL returns FALSE in SQL/Spark joins
   → Rows with NULL join keys silently drop from INNER joins
   → Filter NULLs explicitly BEFORE join or use LEFT join

🚫 WHEN NOT TO USE GLUE FOR TRANSFORMS

1. Simple column rename or type cast only
   → Use Athena CTAS (Project 36) — cheaper, no Glue overhead
   → CREATE TABLE silver AS SELECT CAST(col AS INT) FROM bronze

2. Real-time transformation (< 1 second latency)
   → Glue is batch (30s+ cold start)
   → Use Lambda + custom code, or Kinesis Analytics

3. Transformation requires external API calls per row
   → Spark executors calling REST APIs = bottleneck
   → Use Lambda (parallel by design) or Step Functions

4. Tiny dataset (< 10,000 rows)
   → Glue minimum: 1 DPU × 10 min = $0.44
   → Lambda processing 10K rows costs $0.0001
   → Use Lambda for tiny transforms

5. Transformation is a simple SQL query
   → Athena can do SELECT, JOIN, GROUP BY on S3 data
   → No Glue infrastructure needed
   → Use Athena for ad-hoc or scheduled simple transforms

│  ═════════════════════════                                                       │
│                                                                                  │
│  orders_df.join(products_df, "product_id", "left")                               │
│                                                                                  │
│  Spark Catalyst optimizer chooses join strategy:                                  │
│                                                                                  │
│  Strategy 1: BROADCAST JOIN (for small tables)                                   │
│  ┌─────────────────────────────────────────────────────────────┐                 │
│  │  products_df (50K rows, 2MB) → broadcast to ALL executors   │                 │
│  │                                                             │                 │
│  │  Executor 1: orders_partition_1 + [products copy] → join    │                 │
│  │  Executor 2: orders_partition_2 + [products copy] → join    │                 │
│  │  Executor 3: orders_partition_3 + [products copy] → join    │                 │
│  │  ...                                                        │                 │
│  │                                                             │                 │
│  │  → NO shuffle of orders data                                │                 │
│  │  → Products sent once to each executor (small cost)         │                 │
│  │  → Fastest join strategy for small dimension tables          │                 │
│  │  → Auto-triggered when table < spark.sql.autoBroadcast      │                 │
│  │    JoinThreshold (default: 10MB)                            │                 │
│  └─────────────────────────────────────────────────────────────┘                 │
│                                                                                  │
│  Strategy 2: SORT-MERGE JOIN (for large tables)                                  │
│  ┌─────────────────────────────────────────────────────────────┐                 │
│  │  Both tables shuffled by join key (product_id)              │                 │
│  │                                                             │                 │
│  │  Phase 1: SHUFFLE                                           │                 │
│  │  → All rows with product_id hash=0 → partition 0            │                 │
│  │  → All rows with product_id hash=1 → partition 1            │                 │
│  │  → Network transfer between executors (expensive)           │                 │
│  │                                                             │                 │
│  │  Phase 2: SORT                                              │                 │
│  │  → Each partition sorted by product_id                      │                 │
│  │                                                             │                 │
│  │  Phase 3: MERGE                                             │                 │
│  │  → Sorted partitions merged in linear scan                  │                 │
│  │                                                             │                 │
│  │  → Used when BOTH tables are large                          │                 │
│  │  → Expensive: full data shuffle across network              │                 │
│  └─────────────────────────────────────────────────────────────┘                 │
│                                                                                  │
│  HOW DEDUPLICATION WORKS:                                                        │
│  ════════════════════════                                                        │
│                                                                                  │
│  Method A: dropDuplicates(["order_id"])                                           │
│  → Hash-based: groups by key, keeps FIRST occurrence                             │
│  → Non-deterministic which row is kept if multiple exist                         │
│  → Fast but no control over which duplicate survives                             │
│                                                                                  │
│  Method B: Window + row_number (RECOMMENDED)                                     │
│  → window = Window.partitionBy("order_id").orderBy(desc("updated_at"))           │
│  → df.withColumn("rn", row_number().over(window)).filter("rn = 1")              │
│  → Keeps the LATEST version of each duplicate                                    │
│  → Deterministic, controllable, auditable                                        │
│  → Requires shuffle + sort (more expensive than dropDuplicates)                  │
│                                                                                  │
│  HOW AGGREGATION WORKS:                                                          │
│  ═════════════════════                                                           │
│                                                                                  │
│  df.groupBy("category", "month").agg(sum("revenue"), count("order_id"))          │
│                                                                                  │
│  Phase 1: PARTIAL AGGREGATE (per partition)                                      │
│  → Each executor computes local sum/count for its data                           │
│  → Reduces data volume before shuffle                                            │
│                                                                                  │
│  Phase 2: SHUFFLE (by group key)                                                 │
│  → All rows with same (category, month) go to same partition                     │
│                                                                                  │
│  Phase 3: FINAL AGGREGATE                                                        │
│  → Each partition computes final sum/count                                       │
│  → Result: one row per (category, month) combination                             │
│                                                                                  │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Join produces MORE rows than input — fan-out"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Duplicate keys in the dimension table
→ orders has 1 row with product_id=P001
→ products has 3 rows with product_id=P001 (duplicates)
→ Join result: 3 rows for that order (1 × 3 = 3)
→ Fix: DEDUPLICATE dimension tables BEFORE join
→ Always check: df.count() before and after join

PITFALL 2: "Job OOM — driver runs out of memory"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause A: collect() on large DataFrame
→ Cause B: broadcast() on table > 1GB
→ Cause C: Too many cached DataFrames
→ Fix A: Never collect() — use show(), head(), take()
→ Fix B: Only broadcast tables < 200MB
→ Fix C: unpersist() DataFrames after use

PITFALL 3: "Aggregation returns wrong numbers"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Duplicates in source data not removed before aggregation
→ 3 copies of order_id=ORD001 → revenue counted 3x
→ Fix: ALWAYS deduplicate before aggregate
→ Verify: count(distinct order_id) == count(*)

PITFALL 4: "filter('price > 0') removes rows with NULL price"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: NULL > 0 evaluates to NULL (not TRUE)
→ filter() keeps only TRUE rows → NULLs dropped
→ Fix: filter("price > 0 OR price IS NULL") if you want to keep NULLs
→ Or: fillna(0) before filtering

PITFALL 5: "Join is extremely slow — 2 hours for 45M × 2M"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Both tables shuffled (sort-merge join)
→ 45M rows × 20 columns shuffled across network = slow
→ Fix: broadcast the smaller table
→ from pyspark.sql.functions import broadcast
→ orders_df.join(broadcast(products_df), "product_id")
→ 50K products broadcasted = milliseconds, no shuffle

PITFALL 6: "Window function for dedup runs out of memory"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Skewed data — one order_id has 1M duplicates
→ All 1M rows go to same partition → executor OOM
→ Fix: Pre-filter extreme duplicates first
→ df.groupBy("order_id").count().filter("count > 100") → investigate

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ Hybrid pattern: DynamicFrame for read/write, DataFrame for transforms
→ Clean BEFORE join, dedup BEFORE join — reduce data volume early
→ Window + row_number for deterministic deduplication (keep latest)
→ broadcast() small tables to avoid expensive shuffle joins
→ cache() DataFrames reused in multiple operations, unpersist() after
→ LEFT join preserves all transactions even with missing dimensions
→ Pre-aggregation in silver layer = 40x faster dashboard queries
→ Column rename BEFORE join prevents ambiguous column names
→ Always verify join count ≈ input count (detect fan-out)
→ Write with mode("overwrite") + partitionBy = idempotent per partition
→ Register all silver tables in Glue Catalog for Athena access
