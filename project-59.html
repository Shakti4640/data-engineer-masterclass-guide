🟡 HOW TO APPROACH IT

Step 1: Understand cross-account SNS → SQS pattern
→ Same-account (Project 14): topic + queue + subscription — straightforward
→ Cross-account: TWO additional policies needed
  a) Account A's SNS Topic Policy: "Allow Account B to subscribe"
  b) Account B's SQS Queue Policy: "Allow Account A's SNS to deliver"
→ Both policies must be in place BEFORE the subscription is created

Step 2: Understand the permission chain
→ SUBSCRIBE: Account B's role calls sns:Subscribe on Account A's topic
  → Account A's topic policy must allow sns:Subscribe from Account B
→ DELIVER: Account A's SNS pushes messages to Account B's SQS
  → Account B's queue policy must allow sqs:SendMessage from Account A's SNS
→ Both directions must be authorized

Step 3: Understand who creates the subscription
→ Option A: Account A creates subscription (knows consumer's queue ARN)
→ Option B: Account B creates subscription (knows producer's topic ARN)
→ Cross-account: either side can create it
→ If Account B creates: requires sns:Subscribe permission on Account A's topic
→ If Account A creates: requires knowing Account B's queue ARN
→ BEST PRACTICE: Account B creates (consumer controls what it subscribes to)

Step 4: Understand message filtering (from Project 51)
→ Filter policies work the SAME way cross-account
→ Account B can subscribe with filter: event_type = ["etl_completed", "pipeline_failed"]
→ Account B's queue only receives events it cares about
→ Filtering happens at SNS level (Account A) before delivery

Step 5: Design the event flow
→ Account A's pipeline (Step Functions — Project 55) publishes:
  - "pipeline_started" → severity=info
  - "etl_completed" → severity=info, tables=[orders, customers]
  - "dq_passed" → severity=info, quality_score=94.5
  - "pipeline_failed" → severity=critical, error=...
→ Account B subscribes to: etl_completed + pipeline_failed
→ Account B's queue receives only relevant events
→ Account B's consumer (Project 52) processes and triggers next steps

🔵 WHAT COULD BE DONE

Option A: S3 polling from Account B (current — wasteful)
→ Check every 30 minutes for new data
→ Problems: delay, cost, no context, can't detect failures
→ Already discussed above — this is what we're replacing

Option B: EventBridge cross-account event bus
→ Account A sends events to Account B's event bus
→ More powerful: content-based routing, transforms, 200+ targets
→ But: requires EventBridge setup in both accounts
→ More complex than SNS → SQS for simple event delivery
→ Better for: complex routing rules, many consumers, event transformation
→ Covered conceptually in Project 66

Option C: SQS cross-account direct (Account B polls Account A's queue)
→ Account A puts messages in its own SQS queue
→ Account B's consumer polls Account A's queue directly
→ Problems: tight coupling, Account A manages queue for Account B
→ Doesn't scale to multiple consumers (messages consumed once)
→ No fan-out capability

Option D: Cross-account SNS → SQS (this project) ✅
→ Account A publishes to its existing SNS topic (no changes needed)
→ Account B subscribes its SQS queue to Account A's topic
→ Fan-out: multiple Account B queues (or Account C) can subscribe
→ Filtering: each subscriber gets only relevant events
→ Decoupled: Account A doesn't know or care who subscribes
→ Builds directly on Projects 12, 13, 14, 51

Option E: Lambda cross-account invocation
→ Account A invokes Account B's Lambda directly
→ Problems: tight coupling (Account A must know Account B's Lambda ARN)
→ No buffering (Lambda must be available at invocation time)
→ No retry (if Lambda fails, message lost unless coded)
→ Better: SNS → SQS → Lambda (buffered, retryable)

DECISION: Option D — Cross-account SNS → SQS
→ Reason: decoupled, fan-out capable, filterable, builds on existing patterns
→ Account A's pipeline code: zero changes (already publishes to SNS)
→ Account B: adds queue + subscription + consumer

✅ WHAT SHOULD DO

1. Set BOTH policies before creating subscription
   → Account A: SNS Topic Policy (allow Account B to subscribe + receive)
   → Account B: SQS Queue Policy (allow Account A's SNS to send)
   → Order matters: policies first, then subscription
   → If subscription created before policies → messages silently dropped

2. Use sns:Subscribe permission carefully
   → Allow Account B to subscribe: specific protocol (sqs) only
   → Don't allow: sns:Subscribe with protocol=email (Account B could add emails)
   → Condition: "sns:Protocol": "sqs" restricts to SQS subscriptions only

3. Use aws:SourceArn condition in SQS queue policy
   → Queue policy: "Allow sqs:SendMessage IF source is Account A's SNS topic ARN"
   → Prevents: other SNS topics (from any account) delivering to this queue
   → Without condition: any SNS topic could deliver → security risk

4. Enable RawMessageDelivery for cross-account
   → Same as Project 14: raw delivery = message body directly to SQS
   → Without raw: SQS receives SNS envelope (harder to parse)
   → With raw: message body + message attributes forwarded cleanly

5. Apply filter policy on subscription (from Project 51)
   → Account B only needs: etl_completed + pipeline_failed
   → Filter out: pipeline_started, dq_passed (noisy, not actionable)
   → Filtering happens in Account A's SNS → saves SQS cost in Account B

6. Add DLQ to Account B's queue (from Project 50)
   → Failed message processing → DLQ
   → Monitor DLQ depth with CloudWatch alarm (from Project 54)
   → Cross-account events are CRITICAL — don't lose them

7. Use unique subscription filter per queue
   → Queue 1: event_type = [etl_completed] → triggers pipeline
   → Queue 2: event_type = [pipeline_failed], severity = [critical] → alerts
   → Same topic, different filters, different queues, different actions

8. Test the FULL chain end-to-end
   → Publish test event in Account A → verify arrives in Account B's queue
   → Test with filter match → should arrive
   → Test with filter mismatch → should NOT arrive
   → Test with wrong account → should NOT arrive

❌ WHAT SHOULD NOT DO

1. DO NOT allow sns:* in topic policy (overly broad)
   → sns:* includes: sns:DeleteTopic, sns:SetTopicAttributes
   → Account B could DELETE Account A's topic!
   → Only allow: sns:Subscribe (to subscribe) + delivery happens automatically

2. DO NOT use Principal: "*" in SQS queue policy
   → Allows ANY AWS service/account to send to the queue
   → Massive security risk — anyone can inject messages
   → Always scope: Principal: {"Service": "sns.amazonaws.com"}
   → Plus Condition: {"ArnEquals": {"aws:SourceArn": "<topic-arn>"}}

3. DO NOT forget to confirm cross-account subscriptions
   → SQS subscriptions from same account: auto-confirmed
   → SQS subscriptions cross-account: also auto-confirmed IF queue policy allows
   → Email/HTTPS subscriptions cross-account: require manual confirmation
   → For SQS: auto-confirms if queue policy is correct

4. DO NOT create subscriptions without filter policies
   → Account A publishes 200K events/day (from Project 51)
   → Without filter: Account B receives ALL 200K events
   → Account B only needs ~5K events (etl_completed + pipeline_failed)
   → 195K unnecessary SQS messages/day = wasted cost + processing

5. DO NOT hardcode topic ARNs in application code
   → Use SSM Parameter Store or environment variables
   → Account A's topic ARN may change (recreation, region change)
   → Hardcoded ARN → Account B's subscription breaks silently

6. DO NOT assume message ordering cross-account
   → Standard SNS + Standard SQS: no ordering guarantee
   → Cross-account adds additional routing → even less ordering
   → If order matters: use SNS FIFO → SQS FIFO (but cross-account FIFO is limited)
   → For data pipelines: ordering usually doesn't matter (events are independent)

7. DO NOT create circular event flows
   → Account A → SNS → Account B's SQS → triggers Account B's pipeline
   → Account B → SNS → Account A's SQS → triggers Account A's pipeline
   → If both trigger each other: infinite loop → massive cost
   → Design: unidirectional event flow (producer → consumer)

🚫 WHEN NOT TO USE CROSS-ACCOUNT SNS → SQS

1. When you need guaranteed ordering across accounts
   → Standard SNS/SQS: best-effort ordering (usually FIFO but not guaranteed)
   → FIFO SNS → FIFO SQS: ordered but cross-account support is limited
   → Use: Kinesis Data Streams for ordered cross-account streaming

2. When events need transformation before delivery
   → SNS delivers message AS-IS (no transformation)
   → If Account B needs different format: add Lambda between SQS and consumer
   → Or: use EventBridge (has input transformer built-in)

3. When consumer is not an AWS service
   → SNS → SQS works great for AWS consumers
   → External system (on-prem, other cloud): use SNS → HTTPS endpoint
   → Or: API Gateway → Lambda → external system

4. When event volume exceeds SNS limits
   → SNS: 30,000 publishes/second (adjustable)
   → For 100K+ events/second: use Kinesis Data Streams
   → SNS is for event notification, not high-volume streaming

5. When you need bidirectional communication
   → SNS → SQS is one-way (producer → consumer)
   → If consumer needs to respond: separate return channel needed
   → Use: Step Functions with callback pattern, or separate SNS topic

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Subscription created but messages never arrive"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause 1: SQS queue policy missing → SNS can't deliver
→ Cause 2: SQS queue policy has wrong SourceArn → SNS delivery denied
→ Cause 3: Subscription is "PendingConfirmation" → not active
→ Debug: aws sns list-subscriptions-by-topic --topic-arn <arn>
→ Check: SubscriptionArn should be full ARN (not "PendingConfirmation")
→ Fix: set queue policy BEFORE creating subscription

PITFALL 2: "Subscription confirmed but filter blocks everything"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: publisher doesn't set MessageAttributes
→ Filter: {"event_type": ["etl_completed"]}
→ Publisher: sends body only, no MessageAttributes
→ Filter can't match → message silently not delivered (from Project 51)
→ Fix: ensure Account A's publisher ALWAYS sets MessageAttributes

PITFALL 3: "Messages arrive in SQS but can't be parsed"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: RawMessageDelivery not enabled
→ Without raw: SQS message body = SNS envelope JSON wrapping the actual message
→ Consumer expects raw JSON but gets: {"Type":"Notification","Message":"...escaped..."}
→ Must parse envelope → extract Message field → parse again
→ Fix: set RawMessageDelivery=true on subscription (from Project 14)

PITFALL 4: "Account A topic policy too restrictive — can't subscribe"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: topic policy allows Account B's root but not specific role
→ Or: topic policy missing entirely (default: only topic owner can subscribe)
→ Or: topic policy has Condition that doesn't match
→ Fix: add explicit Allow for Account B's specific role ARN
→ Or: use "AWS": "222222222222" to allow any role in Account B

PITFALL 5: "Circular dependency — policies reference each other"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Topic policy references queue ARN → queue must exist first
→ Queue policy references topic ARN → topic must exist first
→ In practice: this isn't circular — ARNs are predictable
→ Create queue → set queue policy → set topic policy → create subscription
→ ARNs can be constructed without the resource existing

PITFALL 6: "Cross-account subscription counts against Account A's limits"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ SNS subscription limit: 12,500,000 per topic (soft)
→ Each cross-account SQS queue = 1 subscription
→ 5 accounts × 3 queues each = 15 subscriptions → trivial
→ Only a concern at massive scale (10,000+ consumers)
