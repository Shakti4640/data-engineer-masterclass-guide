🟡 HOW TO APPROACH IT

Step 1: Understand the notification landscape on AWS

  OPTION 1: Amazon SNS (Simple Notification Service)
  → Pub/sub messaging service
  → Topics: named channels that messages are published to
  → Subscriptions: who receives messages (email, SMS, SQS, Lambda, HTTP)
  → Fan-out: one publish → multiple subscribers receive
  → Cost: free for email, $0.50/million API calls

  OPTION 2: Amazon SES (Simple Email Service)
  → Full email service (send/receive)
  → HTML templates, bulk sending
  → Overkill for pipeline notifications

  OPTION 3: Custom SMTP (send via Gmail, SendGrid)
  → Requires managing credentials
  → No native AWS integration
  → Not recommended

  OPTION 4: Slack/Teams webhooks
  → Great for team chat notifications
  → Can be an SNS HTTP subscriber
  → Complementary to SNS, not a replacement

  DECISION: Amazon SNS
  → Native AWS integration with every service
  → Multiple delivery protocols (email, SMS, SQS, Lambda)
  → Fan-out pattern (one event → many recipients)
  → Free tier: first 1,000 email notifications free

Step 2: Design the notification architecture
  → Topic 1: pipeline-success → email to data team
  → Topic 2: pipeline-failure → email + SMS to on-call
  → Publish from: Python scripts, Glue jobs, Step Functions
  → Message format: structured JSON with pipeline details

Step 3: Plan for future integration
  → Project 14: SNS → SQS fan-out
  → Project 17: S3 events → SNS notifications
  → Project 28: Full pipeline with notifications
  → Project 54: CloudWatch alarms → SNS
  → Project 78: SLA monitoring with SNS escalation

🔵 WHAT COULD BE DONE

Option A: Single topic for all notifications
→ One topic, all subscribers get everything (success + failure)
→ Pros: simplest setup
→ Cons: on-call gets 365 success emails/year they don't need
→ Can use message filtering (Project 51) but adds complexity

Option B: Separate topics for success vs failure (RECOMMENDED)
→ quickcart-pipeline-success: data team only
→ quickcart-pipeline-failure: on-call + data team
→ Pros: clean separation, different subscriber lists
→ Cons: must publish to correct topic based on outcome

Option C: Single topic with filter policies
→ One topic, subscribers filter by message attributes
→ Pros: one topic to manage
→ Cons: filter policy complexity, covered in Project 51

Option D: EventBridge instead of SNS
→ Event-driven routing with rules
→ Pros: more flexible routing, content-based filtering
→ Cons: overkill for simple notifications, covered in Project 66

DECISION: Option B — separate topics
→ Clear mental model: success topic vs failure topic
→ Easy to add/remove subscribers per topic
→ No filter complexity
→ Extensible: add more topics later (warnings, SLA breaches)

✅ WHAT SHOULD DO

1. Create SEPARATE topics for different severity levels
   → success: informational — data team cares
   → failure: critical — on-call must respond
   → Future: warning (degraded but not failed)

2. Use meaningful topic names with environment prefix
   → quickcart-prod-pipeline-success
   → quickcart-prod-pipeline-failure
   → quickcart-dev-pipeline-failure (dev alerts go to dev team only)

3. Require subscription CONFIRMATION
   → SNS sends confirmation email/SMS
   → Subscriber must click "Confirm" link
   → Prevents spam — you can't subscribe random people

4. Structure messages as JSON
   → Include: pipeline_name, status, timestamp, duration, error_message
   → Parseable by downstream systems (SQS consumers, Lambda)
   → Human-readable subject line for email

5. Add message attributes for filtering
   → Attribute: severity = "info" | "warning" | "critical"
   → Attribute: pipeline = "etl_daily" | "etl_hourly"
   → Enables filter policies in Project 51

6. Tag topics for cost tracking
   → Environment=Production, Team=Data-Engineering
   → SNS is cheap but tagging enables accountability

7. Set delivery retry policy
   → SNS retries failed deliveries (HTTP endpoints)
   → Email: SNS handles retries internally
   → SQS: guaranteed delivery (messages persist in queue)

8. Test notifications BEFORE connecting to real pipelines
   → Publish test message manually
   → Verify all subscribers receive it
   → Check spam folders (SNS emails sometimes land there initially)

❌ WHAT SHOULD NOT DO

1. DO NOT hardcode email addresses in application code
   → Subscribe via SNS — adding/removing subscribers needs no code change
   → Application publishes to TOPIC ARN, not to email addresses
   → Subscriber management is decoupled from publisher code

2. DO NOT send raw error stack traces in notifications
   → Include: error type, error message, pipeline name, timestamp
   → Exclude: full stack trace (put that in CloudWatch Logs)
   → On-call needs to know WHAT failed, not every line of code

3. DO NOT subscribe personal phones to production failure topic
   → Use a shared on-call number/email
   → PagerDuty/OpsGenie integration is better for rotation
   → Personal phone = single point of failure when that person is on vacation

4. DO NOT create topics without encryption
   → Enable SSE (Server-Side Encryption) with KMS
   → Messages may contain pipeline details, error info
   → Encryption at rest is a security best practice

5. DO NOT publish synchronously in the critical path
   → SNS publish is fast (~50ms) but not zero
   → If pipeline has 1000 steps, don't publish after each step
   → Publish ONCE at the end: success or failure

6. DO NOT ignore subscription confirmation
   → Unconfirmed subscriptions receive NOTHING
   → Common mistake: create subscription, forget to confirm, wonder why no emails
   → Check: aws sns list-subscriptions-by-topic → look for "PendingConfirmation"

7. DO NOT use SNS for large payloads
   → Max message size: 256 KB
   → If you need to send large data: publish reference (S3 key) not data itself
   → Pattern: "ETL output available at s3://bucket/path"

🚫 WHEN NOT TO USE SNS

1. When consumer needs to process messages at its own pace
   → SNS is PUSH-based: delivers immediately to all subscribers
   → If consumer is down: email still delivered, but HTTP endpoint misses it
   → Use: SQS (PULL-based) — messages persist until consumer reads them
   → Best: SNS → SQS (fan-out + persistence) — Project 14

2. When you need guaranteed exactly-once processing
   → SNS delivers at-least-once (may deliver duplicates)
   → Use: SQS FIFO for exactly-once processing

3. When message payload > 256 KB
   → SNS max: 256 KB per message
   → Use: S3 + SNS notification (store data in S3, send S3 key via SNS)

4. When you need complex content-based routing
   → SNS filter policies are limited to attribute matching
   → Use: EventBridge for complex rules (Project 66)

5. When you need bidirectional communication
   → SNS is one-way: publisher → subscriber
   → Use: SQS for request/response patterns
   → Use: API Gateway + Lambda for synchronous request/response

6. When you need to replay historical messages
   → SNS doesn't store messages after delivery
   → Use: Kinesis Data Streams (7-day retention, replay from any point)
   → Or: SNS → SQS (SQS retains up to 14 days)

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Subscribed but never receive messages"
→ Cause #1: Subscription not confirmed — still "PendingConfirmation"
→ Fix: check email (including spam folder) for confirmation link
→ Cause #2: filter policy doesn't match message attributes
→ Fix: verify filter policy matches published attributes exactly
→ Debug: aws sns list-subscriptions-by-topic --topic-arn <arn>

PITFALL 2: "Email goes to spam folder"
→ Cause: SNS sends from [email protected]
→ Some email providers flag this as spam
→ Fix: add [email protected] to contacts/safe senders
→ Fix: use SES with custom domain for branded emails (overkill for alerts)

PITFALL 3: "SMS not received"
→ Cause #1: account in SMS sandbox (default for new accounts)
→ Fix: request production SMS access via AWS Support
→ Cause #2: phone number format wrong (must include country code)
→ Fix: use E.164 format: +15551234567 (not 555-123-4567)
→ Cause #3: spending limit reached ($1/month default)
→ Fix: increase SMS spending limit in SNS settings

PITFALL 4: "Published message but publish() returned error"
→ Cause: IAM user/role lacks sns:Publish permission
→ Fix: add policy with sns:Publish on the topic ARN
→ Debug: check IAM Policy Simulator

PITFALL 5: "Message body is empty in email"
→ Cause: published with Message="" (empty string)
→ SNS accepts empty messages — no validation
→ Fix: always validate message content before publishing

PITFALL 6: "Cross-account publish fails"
→ Cause: topic doesn't have resource policy allowing other account
→ Fix: add topic policy allowing sns:Publish from other account
→ Covered in detail in Project 59

PITFALL 7: "Getting duplicate notifications"
→ Cause: SNS delivers at-least-once — duplicates are possible
→ For email: minor annoyance
→ For SQS: consumer must handle idempotency
→ For Lambda: use idempotency key in message

📈 SCALABILITY CONSIDERATIONS

SNS LIMITS:
→ Topics per account: 100,000
→ Subscriptions per topic: 12,500,000
→ Messages per second per topic: 30,000 (default, can increase)
→ Message size: 256 KB max
→ QuickCart: 2 topics × ~10 messages/day = trivially within limits

COST AT SCALE:
→ First 1,000 email notifications: FREE
→ After: $2 per 100,000 emails
→ SMS: $0.00645 per message (US)
→ HTTP/HTTPS: $0.60 per million
→ SNS → SQS: free (same account)
→ QuickCart: ~$0/month (well within free tier)

SCALING PATTERNS:
→ 1-10 pipelines: 2 topics (success/failure) is fine
→ 10-50 pipelines: consider per-pipeline topics or filter policies
→ 50+ pipelines: use filter policies (Project 51) or EventBridge (Project 66)
→ Multi-account: cross-account topic policies (Project 59)

HIGH-VOLUME NOTIFICATIONS:
→ If publishing 1000+ messages/second:
   → SNS handles it (30K msg/sec default)
   → But email subscribers get overwhelmed
   → Use SQS subscriber → batch digest Lambda → single daily email
→ QuickCart: nowhere near this — maybe 10 notifications/day

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY SNS IS WORKING:

1. List topics:
   → aws sns list-topics
   → Verify your topics exist

2. List subscriptions:
   → aws sns list-subscriptions-by-topic --topic-arn <arn>
   → Check: SubscriptionArn is NOT "PendingConfirmation"
   → If pending: subscriber hasn't confirmed yet

3. Publish test message:
   → aws sns publish --topic-arn <arn> --subject "Test" --message "Test body"
   → Should return MessageId
   → Check subscriber's inbox

4. CloudWatch metrics for SNS:
   → NumberOfMessagesPublished
   → NumberOfNotificationsDelivered
   → NumberOfNotificationsFailed
   → Set alarm on NumberOfNotificationsFailed > 0

5. Delivery status logging (for SQS/Lambda/HTTP):
   → Enable in topic delivery status settings
   → Logs to CloudWatch: success/failure per delivery
   → Not available for email (no delivery tracking)

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ SNS is a PUB/SUB messaging service — publishers don't know subscribers
→ TOPICS: named channels that route messages to subscribers
→ SUBSCRIPTIONS: bind protocol + endpoint to a topic
   → Email, SMS, SQS, Lambda, HTTP — all supported
   → Email/HTTP require CONFIRMATION before receiving messages
   → SQS/Lambda are auto-confirmed (same account)
→ MESSAGE ATTRIBUTES: metadata for filtering (Project 51)
→ SEPARATE TOPICS for different severity levels
   → success → data team (informational)
   → failure → on-call (critical, requires response)
→ NOTIFICATION FAILURES SHOULD NEVER CRASH THE PIPELINE
   → Wrap publish in try/except
   → Log the failure but continue pipeline execution
→ STRUCTURED JSON messages enable machine processing
   → Email subscribers read the JSON
   → SQS consumers parse and route
   → Lambda functions process and format

QuickCart now has automated pipeline notifications:
→ Success emails confirm data freshness every morning
→ Failure alerts page the on-call engineer immediately
→ No more silent failures — the Monday morning disaster can't repeat

BUT the data engineer notices a problem:
"SNS delivers messages IMMEDIATELY to all subscribers.
 What if a subscriber is temporarily down?
 What if we need to PROCESS messages, not just display them?
 What if messages arrive faster than the consumer can handle?

 Example: S3 file arrival triggers SNS notification.
 A Python processor on EC2 needs to handle each file.
 If 100 files arrive in 1 minute and the processor takes 
 30 seconds per file — we lose 97 messages because SNS 
 doesn't WAIT for the consumer.

 We need a QUEUE — a buffer between producer and consumer."

→ Project 13: "Buffer Incoming Messages So No Data Is Lost"
→ Introduces: Amazon SQS → Standard vs FIFO → 
   Visibility Timeout → Retention → Long Polling
→ SQS is the complement to SNS: SNS pushes, SQS buffers
→ Together (Project 14): SNS → SQS = fan-out + guaranteed delivery
