🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — continuing from Project 17
Situation:
→ S3 → SNS email alerts working (Project 17)
→ Data team gets emails when files land — great for humans
→ BUT: the data PROCESSING is still manual

Current manual workflow (every morning):
→ 9:00 AM: Analyst checks email → "orders_20250115.csv landed at 00:03 AM"
→ 9:05 AM: Analyst SSH into EC2
→ 9:10 AM: Analyst runs: python load_to_mariadb.py orders_20250115.csv
→ 9:20 AM: Analyst repeats for customers and products files
→ 9:30 AM: Data is ready for queries — 9 HOURS after files landed

Problems:
→ 9-hour delay between file arrival and data availability
→ Human bottleneck — if analyst is sick, data doesn't load
→ Error-prone — analyst sometimes types wrong filename
→ Not scalable — 3 files is manageable, 30 files is nightmare
→ Weekend gap — nobody processes Saturday/Sunday files until Monday

CTO: "I want files processed AUTOMATICALLY within minutes of landing.
      No human in the loop. If processing fails, retry automatically.
      If it fails 3 times, alert us (don't lose the message)."

YOU: "We need S3 events routed to SQS instead of SNS.
      An EC2 worker polls the queue continuously.
      Each message = one file to process.
      Delete message only after successful processing.
      If processing fails → message reappears → automatic retry."

🟡 HOW TO APPROACH IT

Step 1: Understand WHY SQS instead of SNS
→ SNS = push-based → delivers immediately to subscribers
→ SQS = pull-based → messages WAIT until consumer reads them
→ SNS email: fire-and-forget (no retry if human misses it)
→ SQS: persistent → message stays until explicitly deleted
→ SQS: retry-safe → if consumer crashes, message reappears

Step 2: Understand the SQS consumption model
→ Consumer calls ReceiveMessage → gets 1-10 messages
→ Message becomes INVISIBLE for "visibility timeout" period
→ Consumer processes the message
→ Consumer calls DeleteMessage → permanently removes from queue
→ If consumer crashes (no DeleteMessage) → visibility timeout expires
→ Message becomes VISIBLE again → another consumer picks it up
→ This is AUTOMATIC RETRY — built into the protocol

Step 3: Understand the permission chain
→ S3 must have permission to SEND messages to SQS
→ SQS Queue Policy must allow s3.amazonaws.com to sqs:SendMessage
→ Same pattern as Project 17 (SNS Topic Policy)
→ Without this: S3 silently drops events

Step 4: Understand the message structure
→ S3 sends event JSON directly to SQS (no SNS wrapper)
→ SQS message body = S3 event JSON (same structure as Project 17)
→ Simpler than S3 → SNS → SQS (no double-wrapping)

Step 5: Plan the implementation
→ Create SQS Queue
→ Set Queue Policy allowing S3
→ Configure S3 Event Notification → SQS (different prefix from Project 17)
→ Upload test file
→ Poll queue → receive message → parse S3 event → acknowledge

🔵 WHAT COULD BE DONE

Option A: S3 → SQS directly (simplest machine consumption)

Option B: S3 → SNS → SQS (fan-out pattern)
→ S3 sends to SNS → SNS forwards to SQS + email + Lambda
→ More flexible but adds SNS layer
→ Message is DOUBLE-WRAPPED (SNS envelope around S3 event)
→ Covered in Project 28 (full pipeline)

Option C: S3 → Lambda (instant processing)
→ Lambda triggered directly by S3 event
→ No queue, no polling — instant execution
→ Limited to 15 minutes execution time

Option D: S3 → EventBridge → Step Functions
→ Most sophisticated routing

DECISION: Option A — S3 → SQS direct
→ Cleanest path for learning SQS consumption
→ No double-wrapping confusion
→ Foundation for all queue-based patterns
→ Projects 27, 28, 30 build directly on this

✅ WHAT SHOULD DO

1. Set visibility timeout > expected processing time
   → If processing takes 5 minutes → set timeout to 10 minutes
   → Too short: message reappears while still being processed
   → Result: same file processed TWICE simultaneously
   → Rule of thumb: 2x-3x your expected processing time

2. Use long polling (WaitTimeSeconds = 20)
   → Short polling: returns immediately (even if queue empty) → wastes API calls
   → Long polling: waits up to 20 seconds for messages → saves cost + reduces latency
   → Maximum: 20 seconds (SQS hard limit)
   → Reduces ReceiveMessage API calls by 90%+

3. Set message retention period appropriately
   → Default: 4 days → messages auto-deleted after 4 days
   → Max: 14 days
   → If worker is down for 5 days with default 4 days → messages LOST
   → Set to 14 days for safety in non-critical pipelines

4. Delete message ONLY after successful processing
   → Download file → process → verify → THEN delete message
   → If you delete first and processing fails → message gone → file never processed
   → This is the CORE guarantee of SQS-based processing

5. Handle the S3 "test event" that S3 sends during configuration
   → When you configure S3 → SQS notification, S3 sends a TEST event
   → This test event has different structure than real events
   → Contains "Event": "s3:TestEvent" instead of "Records"
   → Your consumer must detect and skip this

6. Use ReceiptHandle for deletion (not MessageId)
   → ReceiptHandle is unique per receive attempt
   → MessageId is unique per message but cannot be used for deletion
   → Every ReceiveMessage returns a new ReceiptHandle for same message

7. Process messages in batches when possible
   → ReceiveMessage can return up to 10 messages per call
   → DeleteMessageBatch can delete up to 10 at once
   → Reduces API calls → reduces cost

❌ WHAT SHOULD NOT DO

1. DO NOT set visibility timeout shorter than processing time
   → Message reappears while still being processed
   → Two workers process same file simultaneously
   → Duplicate data loads, race conditions, wasted compute
   → Example: processing = 5 min, timeout = 2 min
     → At 2 min: message reappears → Worker 2 picks it up
     → At 5 min: Worker 1 finishes → deletes message
     → At 7 min: Worker 2 finishes → tries to delete → message gone
     → Result: file processed TWICE

2. DO NOT delete message before processing completes
   → Delete first → crash during processing → message gone forever
   → File never gets processed → silent data loss
   → ALWAYS: process first, delete last

3. DO NOT use short polling in a tight loop
   → while True: receive_message(WaitTimeSeconds=0)
   → Returns empty immediately → loops again → 1000s of API calls/minute
   → Each API call costs $0.0000004 (but adds up)
   → 1000 calls/min × 60 min × 24 hr = 1.44M calls/day = $0.58/day
   → Long polling (WaitTimeSeconds=20): same loop = ~4,320 calls/day = $0.002

4. DO NOT ignore the S3 test event
   → S3 sends a test event when notification is first configured
   → Structure: {"Service": "Amazon S3", "Event": "s3:TestEvent", ...}
   → If consumer tries to parse as normal event → KeyError → crash
   → Always check for "Records" key before processing

5. DO NOT assume message order
   → SQS Standard Queue does NOT guarantee FIFO ordering
   → File B's event may arrive before File A's event
   → If order matters: use SQS FIFO queue (but throughput limited to 300 msg/sec)
   → For file processing: order usually doesn't matter

6. DO NOT configure overlapping S3 notifications to SNS AND SQS
   → From Project 17: S3 → SNS with prefix="daily_exports/", suffix=".csv"
   → This project: S3 → SQS with SAME prefix and suffix
   → S3 does NOT allow two notifications with overlapping filters
     targeting different service types
   → Error: "Configuration is ambiguously defined"
   → Fix: use DIFFERENT prefixes, OR remove SNS notification first,
     OR use S3 → SNS → SQS fan-out (Project 28)

7. DO NOT hardcode queue URL — construct or discover dynamically
   → Queue URLs change if you recreate the queue
   → Use get_queue_url() to discover at runtime

🚫 WHEN NOT TO USE S3 → SQS

1. When you need INSTANT processing with zero polling delay
   → SQS requires polling (even long polling has up to 20s wait)
   → Lambda triggered by S3 = millisecond invocation
   → Use: S3 → Lambda for instant processing

2. When you need the file processed in guaranteed order
   → S3 → SQS Standard = no ordering guarantee
   → S3 cannot send to SQS FIFO directly
   → Workaround: S3 → SNS → SQS FIFO (complex)
   → Or: add sequence numbers in filename + sort in consumer

3. When you need complex event filtering beyond prefix/suffix
   → S3 notifications only filter on key prefix and suffix
   → Cannot filter on: file size, metadata, content type
   → Use: S3 → EventBridge (filter on any event attribute)

4. When single consumer model is too slow
   → SQS + single EC2 worker = sequential processing
   → 100 files/day each taking 5 min = 500 min = 8.3 hours
   → Scale: multiple workers polling same queue (Project 52)
   → Or: Lambda for parallel processing (each file = separate invocation)

5. When message retention of 14 days is insufficient
   → SQS max retention: 14 days → messages auto-deleted after
   → If worker is down 15 days → messages lost
   → Use: EventBridge Archive (indefinite retention) or S3 event log

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Unable to validate the following destination configurations"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Same as Project 17 but for SQS instead of SNS
→ Cause: Queue Policy doesn't allow s3.amazonaws.com to sqs:SendMessage
→ Fix: set Queue Policy BEFORE configuring S3 notification
→ Also: queue must be in SAME REGION as bucket

PITFALL 2: "Overlapping notification filter conflict"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ If Project 17 SNS notification still exists with same prefix/suffix
→ AND you try to add SQS notification with same prefix/suffix
→ Error: "Configuration is ambiguously defined"
→ S3 does NOT allow overlapping prefix+suffix filters across different targets
→ Fix A: remove Project 17 notification first (17_06_cleanup.py)
→ Fix B: use different prefix for SQS (e.g., "processing/")
→ Fix C: use S3 → SNS → SQS fan-out (Project 28)

PITFALL 3: "Consumer receives S3 test event and crashes"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ When S3 notification is first configured, S3 sends a test event
→ Structure: {"Service":"Amazon S3","Event":"s3:TestEvent","Time":"..."}
→ NO "Records" key → consumer parsing Records[0] → KeyError
→ Fix: always check if "Records" exists before parsing
→ Skip test events gracefully

PITFALL 4: "Message processed but not deleted — processed again"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Consumer processes file successfully but crashes before DeleteMessage
→ Visibility timeout expires → message reappears → processed again
→ Result: duplicate processing (file loaded twice into database)
→ Fix: make processing IDEMPOTENT
→ Idempotent = processing same message twice produces same result
→ Example: use UPSERT instead of INSERT, or check if file already loaded

PITFALL 5: "Messages disappearing without being processed"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause A: retention period expired (default 4 days)
→ Cause B: another consumer deleted the message
→ Cause C: purge_queue() was called accidentally
→ Fix: set retention to 14 days, use single consumer group, monitor queue depth

PITFALL 6: "Long polling returns empty but messages exist"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: visibility timeout hasn't expired for in-flight messages
→ Messages are INVISIBLE (being processed by another consumer)
→ Debug: check ApproximateNumberOfMessagesNotVisible metric
→ This is EXPECTED behavior — not a bug

PITFALL 7: "ReceiveMessage returns fewer messages than MaxNumberOfMessages"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ MaxNumberOfMessages=10 does NOT guarantee 10 messages returned
→ SQS samples a SUBSET of servers — may find fewer
→ This is by design for Standard Queues
→ To drain queue: keep calling ReceiveMessage until empty response

📈 SCALABILITY CONSIDERATIONS

THROUGHPUT:
━━━━━━━━━━
→ SQS Standard: unlimited messages/second (no practical limit)
→ ReceiveMessage: up to 10 messages per call
→ DeleteMessageBatch: up to 10 messages per call
→ QuickCart's 3 messages/day: trivially within limits

CONSUMER SCALING:
━━━━━━━━━━━━━━━━━
→ 1 worker processing 3 files/day: no scaling needed
→ 100 files/day, 5 min each: 1 worker = 500 min/day → just barely fits
→ 1000 files/day: need multiple workers polling same queue
→ SQS supports MULTIPLE consumers on same queue safely
→ Each message delivered to ONLY ONE consumer (at a time)
→ Auto-scaling workers based on queue depth: Project 52

COST:
━━━━━
→ First 1 million requests/month: FREE
→ After: $0.40 per million requests (Standard)
→ Long polling: 1 request every 20 seconds = 4,320/day = 129,600/month
→ Cost: well within free tier
→ Even at scale: 1 million requests = $0.40

MESSAGE SIZE:
━━━━━━━━━━━━━
→ Max message size: 256 KB
→ S3 event JSON: typically 1-2 KB → well within limit
→ If you need larger messages: use Extended Client Library
  (stores payload in S3, message contains pointer)

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ SQS = pull-based message queue — consumer POLLS for messages
→ S3 → SQS: event sent directly as message body (no SNS wrapper)
→ Queue Policy must allow s3.amazonaws.com BEFORE S3 notification config
→ Visibility Timeout = how long message is hidden after receive
→ Delete message ONLY after successful processing (core guarantee)
→ If no delete → timeout expires → message reappears → automatic retry
→ Long polling (WaitTimeSeconds=20) saves 90% API calls vs short polling
→ S3 sends test event on first config → consumer must handle gracefully
→ S3 does NOT allow overlapping prefix+suffix across SNS and SQS targets
→ Message structure: json.loads(body)["Records"][0]["s3"]["object"]["key"]
→ Always URL-decode the object key (unquote_plus)
