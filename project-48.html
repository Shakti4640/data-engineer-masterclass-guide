Step 2: Understand S3 Select capabilities

  WHAT S3 Select CAN do:
  → SELECT specific columns from CSV, JSON, or Parquet objects
  → WHERE clause filtering (equality, comparison, LIKE, IS NULL)
  → Aggregate functions: COUNT, SUM, AVG, MIN, MAX
  → LIMIT clause
  → CAST for type conversion
  → String functions: LOWER, UPPER, TRIM, SUBSTRING, CHAR_LENGTH
  → Date functions (limited)
  → Logical: AND, OR, NOT, IN, BETWEEN

  WHAT S3 Select CANNOT do:
  → JOIN (single object only — no multi-file queries)
  → GROUP BY (aggregates apply to entire result)
  → ORDER BY (results not sorted)
  → Subqueries
  → Window functions
  → User-defined functions
  → Multiple objects in one query (use Athena for that)

Option A: Download file + local processing

→ Problem: slow, wastes bandwidth, needs disk space
→ VERDICT: Works but inefficient for simple inspections

Option B: Athena query on Bronze
→ Create Glue Catalog table for Bronze CSV
→ Run Athena query
→ Problem: need to create/maintain Catalog table for raw CSVs
→ Problem: Athena scans entire 487 MB even for simple filter
→ Problem: Athena startup overhead (~3-5 seconds)
→ VERDICT: Good for repeated queries, overkill for one-time inspection

Option C: S3 Select
→ Run SQL directly on the S3 object via API
→ No Catalog table needed
→ No Athena overhead
→ Returns only matching data (not entire file)
→ Works on CSV, JSON, and Parquet
→ VERDICT: ✅ Perfect for single-file ad-hoc inspection

Option D: S3 Select with Scan Range (partial file read)
→ Read only a PORTION of the file (e.g., first 10 MB)
→ Even faster than full-file S3 Select
→ Problem: may miss matching rows outside scan range
→ VERDICT: Good for sampling, not for complete filtering

DECISION: Option C (S3 Select) + Option D (Scan Range for sampling)

✅ WHAT SHOULD DO

1. Use S3 Select for single-file ad-hoc inspection
   → Quick data quality checks on Bronze CSVs
   → Schema discovery (what columns does this file have?)
   → Row counting without downloading
   → Filtering specific conditions

2. Specify InputSerialization correctly
   → CSV: specify FileHeaderInfo (USE, IGNORE, NONE)
   → USE = first row is header, use column names in SQL
   → IGNORE = first row is header but skip it
   → NONE = no header, use _1, _2, _3 as column names
   → QuickCart CSVs have headers → use FileHeaderInfo="USE"

3. Specify OutputSerialization
   → CSV: get results as CSV (easy to parse)
   → JSON: get results as JSON (structured)
   → For programmatic use: JSON
   → For human inspection: CSV

4. Use CAST for type-aware filtering
   → CSV columns are ALL strings by default
   → WHERE total_amount = 0 → string comparison "0" (misses "0.00")
   → WHERE CAST(total_amount AS DECIMAL) = 0 → numeric comparison ✅
   → CAST works for: INT, FLOAT, DECIMAL, BOOL, TIMESTAMP

5. Handle streaming response correctly
   → S3 Select returns results as a STREAM of events
   → Events: Records (data), Stats (bytes scanned/returned), End
   → Must iterate through EventStream to collect all records
   → boto3: response['Payload'] is an EventStream iterator

6. Use Scan Range for SAMPLING large files
   → ScanRange: {"Start": 0, "End": 10485760}  (first 10 MB)
   → Reads only specified byte range
   → Much faster for sampling (don't need ALL matches)
   → WARNING: may split a row at boundary → S3 handles this gracefully
     (reads partial row at start, complete row at end)

7. Build a reusable utility for the team
   → Wrap S3 Select in a Python function
   → Accept: bucket, key, SQL expression
   → Return: parsed results as list of dicts
   → Team can use for any Bronze file inspection

❌ WHAT SHOULD NOT DO

1. DO NOT use S3 Select for multi-file queries
   → S3 Select works on ONE object at a time
   → For querying across 30 files: use a loop (OK for scripts)
   → For querying across 1000 files: use Athena (much more efficient)

2. DO NOT use S3 Select for Parquet when Athena is available
   → S3 Select on Parquet: limited SQL, no partition pruning
   → Athena on Parquet: full SQL, partition pruning, columnar read
   → S3 Select on Parquet is useful only when Athena is unavailable
     (e.g., no Glue Catalog table exists)

3. DO NOT forget to handle compressed files
   → S3 Select supports: GZIP, BZIP2 compressed CSVs
   → Must specify CompressionType in InputSerialization
   → NONE for uncompressed (default)
   → GZIP for .gz files
   → Cannot use ScanRange with compressed files

4. DO NOT use S3 Select as a replacement for Athena
   → S3 Select: single file, simple SQL, no joins/GROUP BY
   → Athena: multi-file, full SQL, joins, GROUP BY, window functions
   → S3 Select = inspection tool
   → Athena = query engine

5. DO NOT assume S3 Select is free
   → Charged per data SCANNED: $0.002 per GB scanned
   → Charged per data RETURNED: $0.0007 per GB returned
   → 487 MB file: $0.002 × 0.487 = $0.00097 per query
   → Cheap but not zero — 1000 queries = $0.97

6. DO NOT use S3 Select on objects > 256 GB (CSV) or > 1 TB (Parquet)
   → Hard limits for S3 Select
   → For larger files: use Athena

7. DO NOT ignore the 5-column LIMIT for S3 Select on CSV
   → S3 Select: no column limit per se, but performance degrades
     with many columns selected
   → SELECT * on 100-column CSV: returns all columns → slow
   → SELECT col1, col2 on 100-column CSV: much faster
   → Always select only the columns you need

🚫 WHEN NOT TO USE S3 SELECT

1. When querying across MULTIPLE files
   → S3 Select = one object at a time
   → 100 files = 100 API calls = slow
   → Use Athena for multi-file queries

2. When you need GROUP BY, ORDER BY, or JOINs
   → S3 Select doesn't support these
   → Aggregate functions (COUNT, SUM) work but on entire result set
   → For grouped aggregations: use Athena

3. When the file is Parquet and Athena table already exists
   → Athena on Parquet: columnar read, partition pruning → very efficient
   → S3 Select on Parquet: works but less optimized
   → Only use S3 Select on Parquet when Athena isn't set up

4. When file is in Glacier Deep Archive
   → S3 Select doesn't work on Deep Archive
   → Standard Glacier: use Glacier Select (different API)
   → Deep Archive: must restore first, then query

5. When result set is expected to be the ENTIRE file
   → SELECT * FROM s3object (no WHERE) on 487 MB file
   → S3 Select reads 487 MB AND returns 487 MB
   → Slower than just downloading the file
   → S3 Select wins when WHERE clause filters significantly

6. When you need EXACT row ordering
   → S3 Select doesn't guarantee output order
   → For ordered results: download + sort locally, or use Athena

┌─────────────────────────────────────────────────────────────────────────────────┐
│                    INTERNAL MECHANICS: S3 SELECT                                │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  ❶ Request Flow                                                                │
│  ━━━━━━━━━━━━━━━                                                               │
│                                                                                 │
│  Client sends: POST /bucket/key?select&select-type=2                           │
│  Body: {                                                                        │
│    "Expression": "SELECT order_id, total_amount FROM s3object                  │
│                   WHERE CAST(total_amount AS DECIMAL) = 0",                    │
│    "ExpressionType": "SQL",                                                    │
│    "InputSerialization": {                                                      │
│      "CSV": {"FileHeaderInfo": "USE", "FieldDelimiter": ","}                   │
│    },                                                                           │
│    "OutputSerialization": {"CSV": {}}                                           │
│  }                                                                              │
│                                                                                 │
│  S3 internal processing:                                                        │
│  1. S3 receives SelectObjectContent request                                     │
│  2. S3 does NOT download the object to a compute node                          │
│     → Processing happens INSIDE S3 storage infrastructure                      │
│     → This is why it's so fast — no data movement                              │
│  3. S3 reads the object in chunks (streaming)                                   │
│  4. For each chunk:                                                             │
│     a. Parse CSV rows                                                           │
│     b. Evaluate WHERE clause per row                                            │
│     c. If match: project selected columns                                       │
│     d. Buffer matching rows                                                     │
│  5. Stream results back to client as EventStream                               │
│  6. After entire object processed: send Stats event + End event                │
│                                                                                 │
│                                                                                 │
│  ❷ EventStream Response Format                                                 │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                                              │
│                                                                                 │
│  S3 Select returns an EventStream (HTTP chunked transfer encoding):             │
│                                                                                 │
│  Event 1: {"Records": {"Payload": b"ORD-001,0.00\nORD-047,0.00\n..."}}        │
│  Event 2: {"Records": {"Payload": b"ORD-123,0.00\n..."}}                       │
│  ...                                                                            │
│  Event N: {"Stats": {"Details": {                                               │
│              "BytesScanned": 510656512,      ← 487 MB scanned                  │
│              "BytesProcessed": 510656512,    ← 487 MB processed                │
│              "BytesReturned": 2048           ← 2 KB returned                   │
│            }}}                                                                  │
│  Event N+1: {"End": {}}                                                        │
│                                                                                 │
│  KEY INSIGHT:                                                                   │
│  → BytesScanned: 487 MB (must read entire file for WHERE evaluation)           │
│  → BytesReturned: 2 KB (only matching rows sent back)                          │
│  → Network transfer: 2 KB (not 487 MB!)                                        │
│  → You pay for: BytesScanned + BytesReturned                                   │
│                                                                                 │
│                                                                                 │
│  ❸ ScanRange (Partial File Read)                                               │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                                              │
│                                                                                 │
│  ScanRange: {"Start": 0, "End": 10485760}  (first 10 MB)                      │
│                                                                                 │
│  How it works:                                                                  │
│  1. S3 reads ONLY bytes 0 to 10,485,760 of the object                          │
│  2. If Start > 0: S3 seeks to Start, finds next complete row                   │
│     (skips partial row at beginning of range)                                   │
│  3. If End < file size: S3 reads past End to find row boundary                 │
│     (completes the last row that spans the boundary)                            │
│  4. Processes only rows within the range                                        │
│                                                                                 │
│  WHY use ScanRange:                                                             │
│  → Sample first N bytes without scanning entire file                           │
│  → Parallel processing: split file into ranges, query each in parallel         │
│  → Cost: pay only for bytes scanned in range (not entire file)                 │
│                                                                                 │
│  LIMITATIONS:                                                                   │
│  → Cannot use with compressed files (GZIP, BZIP2)                              │
│     (compression prevents random byte-offset access)                           │
│  → Cannot use with JSON objects                                                 │
│  → Works with: CSV (uncompressed) and Parquet                                  │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "S3 Select returns empty results but I know data exists"
→ Cause #1: FileHeaderInfo wrong — column names don't match
   - FileHeaderInfo="NONE" but SQL uses column names → no match
   - Fix: use FileHeaderInfo="USE" when CSV has headers
→ Cause #2: Type mismatch in WHERE clause
   - WHERE total_amount = 0 compares STRING "0.00" to STRING "0" → no match
   - Fix: WHERE CAST(total_amount AS DECIMAL) = 0
→ Cause #3: Extra whitespace in CSV values
   - Column value: " 0.00 " (with spaces) → doesn't match "0.00"
   - Fix: WHERE TRIM(total_amount) = '0.00'

PITFALL 2: "S3 Select is slow — takes 30 seconds for 487 MB"
→ Cause: S3 must scan the ENTIRE file (no index, no column pruning for CSV)
→ This is normal for CSV — row-based format must read all bytes
→ Fix: if speed matters, convert to Parquet first
→ S3 Select on Parquet: reads only needed columns → much faster

PITFALL 3: "S3 Select fails with 'MalformedXML' error"
→ Cause: SQL expression has unescaped characters
→ Single quotes in SQL sent via XML body → XML parsing error
→ Fix: use boto3 (handles escaping) instead of raw API calls

PITFALL 4: "S3 Select truncates results"
→ Cause: MaxBytesReturned limit (default: none, but may be set)
→ Or: streaming response not fully consumed
→ Fix: iterate through ENTIRE EventStream
→ Check Stats event: BytesReturned should match received data

PITFALL 5: "ScanRange returns unexpected number of rows"
→ Cause: ScanRange splits rows at byte boundaries
→ S3 adjusts to complete rows, but some rows may be partially
  included or excluded at boundaries
→ This is by design — ScanRange is for sampling, not exact counts
→ Fix: for exact results, don't use ScanRange (scan entire file)

PITFALL 6: "S3 Select fails on compressed CSV"
→ Cause: CompressionType not specified
→ Default: NONE — S3 tries to parse GZIP bytes as CSV → garbage
→ Fix: specify CompressionType="GZIP" in InputSerialization
→ Note: ScanRange not supported with compression

PITFALL 7: "S3 Select on Parquet fails with 'unsupported type'"
→ Cause: Parquet has complex types (MAP, ARRAY, STRUCT)
→ S3 Select supports only primitive Parquet types
→ Fix: flatten complex types during ETL (Bronze→Silver)
→ Or: use Athena for Parquet with complex types

📈 SCALABILITY CONSIDERATIONS

SINGLE FILE PERFORMANCE:
→ 1 MB CSV: < 0.5 seconds
→ 100 MB CSV: ~1-3 seconds
→ 487 MB CSV: ~2-5 seconds
→ 1 GB CSV: ~5-10 seconds
→ 10 GB CSV: ~30-60 seconds (consider Athena at this size)
→ Max CSV size: 256 GB (S3 Select limit)

MULTI-FILE SCANNING (loop):
→ 30 files × 3 seconds each = 90 seconds (sequential)
→ With concurrent requests: 30 files / 10 parallel = ~9 seconds
→ At 100+ files: switch to Athena (single query, parallel scan)

COST AT SCALE:
→ 1 query on 487 MB CSV: $0.001
→ 100 queries/day on 487 MB: $0.10/day = $3/month
→ 1000 queries/day: $1/day = $30/month → consider building pipeline
→ Rule of thumb: if cost > $10/month for same query → build ETL

PARQUET vs CSV PERFORMANCE:
→ Same data, 15 columns, querying 2 columns:
  - CSV 487 MB: scans 487 MB → $0.001
  - Parquet 150 MB: scans ~20 MB (2/15 columns) → $0.00004
  - Parquet is 25× cheaper for column-specific queries

CONCURRENT S3 SELECT REQUESTS:
→ No hard limit on concurrent S3 Select requests
→ S3 handles parallel requests efficiently
→ Practical limit: Lambda/EC2 concurrency + API throttling
→ 100 concurrent requests: no problem
→ 1000 concurrent: may see throttling → implement retry

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ S3 Select: run SQL directly on individual S3 objects (CSV, JSON, Parquet)
→ No Glue Catalog, no Athena setup, no infrastructure needed
→ Returns ONLY matching rows — 487 MB file → 2 KB returned (99.9% reduction)
→ CSV: S3 must scan entire file (row-based), but only returns matches
→ Parquet: S3 reads only selected columns (columnar) → much less scanning
→ ScanRange: read only a byte range of the file (sampling, not complete filter)
→ ScanRange NOT supported with compressed files (GZIP, BZIP2)
→ FileHeaderInfo="USE": reference columns by name in SQL
→ CAST required for numeric comparisons on CSV (all values are strings)
→ EventStream response: iterate through Records, Stats, End events
→ Cost: $0.002/GB scanned + $0.0007/GB returned (cheaper than Athena per GB)
→ Glacier Select: same concept but for archived objects (slower, different API)
