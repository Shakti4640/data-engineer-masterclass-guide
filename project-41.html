🔵 WHAT COULD BE DONE

Option A: S3 Event Notification → Lambda → Start Glue Job
→ Works but Lambda becomes "glue code" you must maintain
→ No built-in retry/wait/branching
→ If Glue job fails, Lambda has already exited

Option B: S3 Event Notification → SQS → EC2 Poller → Script
→ Already built in Project 30
→ Requires EC2 running 24/7 (cost)
→ Custom orchestration logic in Python (fragile)

Option C: S3 → EventBridge → Lambda → Glue Workflow
→ Triggers Glue Workflow via API
→ But Glue Workflow has limited error handling
→ Can't easily inject dynamic parameters

Option D: S3 → EventBridge → Step Functions → Glue + Lambda + SNS
→ ✅ BEST: Fully managed, visual, auditable
→ Built-in retry, catch, wait, parallel, choice
→ EventBridge → Step Functions is a NATIVE integration
→ No "glue code" for the trigger
→ Dynamic parameters passed from S3 event → State Machine input

DECISION: Option D

✅ WHAT SHOULD DO

1. Enable EventBridge notifications on the S3 bucket
   → One-time bucket-level setting
   → Replaces S3 Event Notifications for EventBridge use cases
   → Both can coexist (S3 Event Notifications still work for SNS/SQS)

2. Use PRECISE EventBridge rule filters
   → Filter by bucket name + prefix + suffix
   → Prevent triggering on non-CSV files or wrong prefixes

3. Pass S3 event details INTO Step Functions as input
   → Bucket name, object key, file size, timestamp
   → State Machine uses these dynamically (no hardcoding)

4. Use Step Functions SDK integrations (not Lambda wrappers)
   → StartCrawler → .sync (waits for completion automatically)
   → StartJobRun → .sync (waits for completion automatically)
   → ".sync" = Step Functions polls Glue on your behalf

5. Implement idempotency
   → Check if this file was already processed (DynamoDB tracking table)
   → Prevent double-processing if S3 event fires twice (rare but possible)

6. Use Step Functions execution name derived from file key
   → Execution name: "orders-20250115-csv-{timestamp}"
   → Makes CloudWatch logs searchable

7. Tag all resources consistently
   → Same tags as Project 1: Environment, Team, CostCenter

❌ WHAT SHOULD NOT DO

1. DO NOT rely on S3 Event Notification → Lambda as orchestrator
   → Lambda has 15-minute timeout
   → Glue jobs can run 30+ minutes
   → Lambda exits, job runs unsupervised

2. DO NOT use polling/cron as backup "just in case"
   → Creates two trigger paths → duplicate processing
   → If you need backup: use Step Functions retry, not cron

3. DO NOT start Step Functions execution with SAME name twice
   → Execution names must be unique within 90 days
   → Duplicate name = API error, NOT idempotent re-run
   → Use: {entity}-{date}-{uuid} pattern

4. DO NOT pass entire S3 file content through Step Functions
   → State Machine payload limit: 256 KB
   → Pass only: bucket, key, metadata
   → Let Glue/Lambda read the actual file

5. DO NOT enable EventBridge on S3 without testing the filter
   → A bad filter triggers on EVERY object in the bucket
   → Including Athena query results, Glue temp files, etc.
   → Test with one file FIRST before going live

6. DO NOT skip the Catch block in Step Functions
   → Uncaught errors = execution fails silently
   → Always catch → log → alert → possibly retry

7. DO NOT hardcode Glue job names or crawler names in state machine
   → Use SSM Parameter Store or Step Functions parameters
   → Enables promotion across dev/staging/prod

🚫 WHEN NOT TO USE THIS PATTERN

1. When files arrive every second (high frequency)
   → Step Functions Standard: charged per state transition ($0.025 per 1000)
   → 86,400 executions/day × 7 states = 604,800 transitions/day = $15/day
   → Use: Step Functions Express (10x cheaper) or Kinesis (Project 72)

2. When the "ETL" is just one Glue job with no chaining
   → Overkill to use Step Functions for a single job
   → Use: S3 → EventBridge → Lambda → StartJobRun (fire-and-forget)

3. When you need sub-second processing latency
   → EventBridge delivery: ~1-3 seconds
   → Step Functions startup: ~200ms
   → Glue job cold start: 30-60 seconds
   → Total: ~1-2 minutes minimum
   → For sub-second: use Kinesis + Lambda (Project 72)

4. When upstream system can call an API directly
   → If MariaDB export script can call Step Functions API directly
   → Skip S3 event entirely — more reliable, less moving parts
   → But: loses the S3 "file as contract" decoupling benefit

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "EventBridge rule never fires"
→ Cause #1: Forgot to ENABLE EventBridge on the S3 bucket
   - Bucket Properties → Amazon EventBridge → Edit → Enable
   - This is a BUCKET-LEVEL setting, not an account-level setting
→ Cause #2: Rule filter is wrong (key prefix mismatch, missing "detail")
→ Debug: CloudWatch Metrics → EventBridge → "TriggeredRules" metric
→ Debug: Remove all filters temporarily, see if rule fires on ANY S3 event

PITFALL 2: "Step Functions execution starts but immediately fails"
→ Cause: IAM role for EventBridge target lacks states:StartExecution
→ Or: IAM role for Step Functions lacks glue:StartJobRun, glue:GetCrawler
→ Fix: check BOTH roles (EventBridge invocation role + Step Functions execution role)

PITFALL 3: "Glue Crawler succeeds but ETL job gets wrong table"
→ Cause: Crawler updates the catalog table, but ETL references old table name
→ Fix: Pass crawler output (database + table name) dynamically through state machine
→ Or: Use fixed naming convention and verify in ETL job

PITFALL 4: "Same file triggers TWO executions"
→ Cause: S3 multipart upload completes → CompleteMultipartUpload event fires
→ AND: S3 may emit both "PutObject" and "CompleteMultipartUpload" events
→ Fix: Filter EventBridge rule by "reason": ["PutObject"] only
→ Or: Implement idempotency check in first state (DynamoDB lookup)

PITFALL 5: "Step Functions .sync task hangs forever"
→ Cause: Glue job is stuck (waiting for DPU capacity, networking issue)
→ Fix: Set TimeoutSeconds on every Task state (e.g., 1800 = 30 min)
→ Without timeout: execution runs for up to 1 year (Standard) and you pay

PITFALL 6: "EventBridge fires for Athena output files too"
→ Cause: Athena writes results to S3 in same bucket
→ These are .csv files under a different prefix
→ Fix: PRECISE prefix filter: "daily_exports/" not just ".csv"

PITFALL 7: "Step Functions can't parse the entity name from S3 key"
→ Cause: Used wrong JSONPath expression in InputPath
→ S3 key: "daily_exports/orders/orders_20250115.csv"
→ Need to extract "orders" — Step Functions has LIMITED string manipulation
→ Fix: Use a Pass state with intrinsic functions (States.ArrayGetItem + States.StringSplit)
→ Or: Use a lightweight Lambda for parsing (adds cost but adds flexibility)

PITFALL 8: "Works in dev, breaks in prod — different account"
→ Cause: IAM roles, bucket names, Glue database names all hardcoded
→ Fix: Parameterize everything via SSM Parameter Store or CloudFormation parameters

📈 SCALABILITY CONSIDERATIONS

CURRENT LOAD:
→ 3 files/day → 3 Step Functions executions → 3 Glue jobs
→ Easily within free tier for EventBridge + Step Functions

SCALING TO 100 FILES/DAY:
→ 100 executions × 8 states = 800 transitions/day
→ Still within free tier (4,000/month)
→ But: 100 concurrent Glue jobs may exceed account limit
→ Default Glue concurrent job runs: 50 (soft limit)
→ Solution: Add a "Choice" state to check Glue capacity before starting
→ Or: Add SQS buffering before Step Functions (rate limiting)

SCALING TO 10,000 FILES/DAY:
→ Step Functions Standard: $6/month (still cheap)
→ But: 10,000 concurrent executions may hit throttling
→ Step Functions Standard: 1,300 state transitions/second (default)
→ Solutions:
   1. Switch to Step Functions Express ($1 per million executions)
   2. Batch files: trigger once per "batch complete" marker file
   3. Fan-out: one parent execution spawns parallel child executions

CONCURRENT EXECUTION LIMITS:
→ Step Functions Standard: 1,000,000 open executions per account per region
→ EventBridge: 10,000 rules per bus (default, can increase)
→ EventBridge target invocations: 10,000/second (soft limit)
→ Glue: 50 concurrent job runs (default, increase via support)
→ Glue Crawlers: 100 concurrent crawlers (default)

MULTI-REGION CONSIDERATION:
→ EventBridge rules are REGIONAL
→ S3 event only goes to EventBridge in the SAME region as the bucket
→ Cross-region: use EventBridge cross-region event bus forwarding

🔑 KEY TAKEAWAY

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ S3 → EventBridge (native since Nov 2021) eliminates need for CloudTrail-based triggers
→ EventBridge → Step Functions is a native integration (no Lambda glue needed)
→ Step Functions .sync pattern waits for Glue job completion automatically
→ Step Functions intrinsic functions (States.StringSplit, States.Format)
  enable dynamic parameter extraction WITHOUT Lambda
→ Crawler polling loop: Start → Wait → GetStatus → Choice → (loop or proceed)
→ Catch blocks on EVERY task state prevent silent failures
→ TimeoutSeconds on task states prevent infinite hangs
→ Diagnostic script systematically checks every link in the chain
→ Event-driven > time-driven when upstream timing is unpredictable

🌉 BRIDGE TO NEXT PROJECT

The pipeline now triggers automatically when files arrive.
But there's a new problem:

Every night, the MariaDB export (Project 29) dumps ALL rows:
→ orders table: 5 million rows → 500 MB CSV
→ But only ~2,000 rows are NEW since yesterday

The Glue ETL job processes ALL 5 million rows every night.
→ 99.96% of the work is redundant
→ Glue job takes 8 minutes (should take 30 seconds)
→ Costs $0.44/run in DPU-hours (should cost $0.02)
→ Redshift COPY replaces entire table (instead of merging changes)

CTO: "We're paying to reprocess the same data every night.
      Can't we detect WHAT CHANGED and only process that?"

YOU: "Yes — that's Change Data Capture (CDC).
      We need a watermark column in MariaDB,
      a filtered export, and a MERGE strategy in Redshift."

→ Project 42: "Capture Only Changed Rows from MariaDB Nightly"
→ Introduces: CDC patterns, watermark columns, Glue bookmarks + WHERE,
