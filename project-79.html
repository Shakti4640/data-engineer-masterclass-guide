🔵 WHAT COULD BE DONE

Option A: Lift-and-shift MariaDB to RDS then build lake
→ Faster app cutover (RDS compatible with MariaDB)
→ But: doesn't solve the analytics problem
→ Still need S3 lake + Redshift anyway
→ Two migrations instead of one

Option B: DMS direct to Redshift (skip S3 lake)
→ Simpler — fewer components
→ But: no raw data preservation (no Bronze layer)
→ Cannot reprocess from source if transformations change
→ Violates medallion architecture (Project 44)

Option C: DMS to S3 Bronze → Glue ETL → Silver → Gold → Redshift
→ Full medallion architecture
→ Raw data preserved forever
→ Reprocessable from any layer
→ More components but proper engineering

Option D: AWS DataSync for bulk + DMS for CDC
→ DataSync for initial 500GB (faster for bulk)
→ DMS for ongoing CDC only
→ Problem: DataSync doesn't understand MariaDB format
→ Need custom export + DataSync + custom import

DECISION: Option C — DMS to S3 + Glue ETL + Redshift
→ Reason: complete architecture, reprocessable, proper governance
→ DMS handles BOTH full load AND CDC in one tool
→ Glue handles all transformations with lineage
→ Redshift Serverless for cost-effective serving

✅ WHAT SHOULD DO

1. Use DMS full-load-and-cdc mode (not sequential)
   → Starts CDC capture WHILE full load is running
   → No gap between historical and real-time data
   → CDC changes are applied after full load completes

2. Target S3 with Parquet format from DMS
   → DMS can write Parquet directly (since 2022)
   → Avoids CSV → Parquet conversion step in Glue
   → Saves processing time and DPU cost

3. Partition DMS output by table AND date
   → S3 path: /raw/{database}/{table}/year=YYYY/month=MM/day=DD/
   → Enables partition pruning in Athena/Glue (Project 26)
   → DMS supports timestamp-based partitioning

4. Run DMS from READ REPLICA, not primary
   → Avoids any performance impact on production
   → Read replica handles the full load scan
   → CDC still comes from primary's binlog (required)

5. Validate row counts at EVERY layer
   → Source (MariaDB) → Bronze (S3) → Silver → Gold → Redshift
   → Automated reconciliation job runs daily
   → Discrepancy > 0.01% triggers alert

6. Build the ENTIRE pipeline before starting DMS
   → Glue jobs, Redshift tables, Athena views — all ready
   → DMS starts flowing → data appears in Bronze → pipeline processes automatically
   → No manual handoff between phases

7. Maintain parallel run for 30 days after cutover
   → On-prem still receives writes (dual-write or app-level routing)
   → Compare AWS results with on-prem daily
   → Rollback plan: switch apps back to on-prem if issues found

8. Tag EVERYTHING with migration context
   → migration-phase: full-load | cdc | cutover | post-cutover
   → source-system: on-prem-mariadb
   → Enables cost tracking per migration phase

❌ WHAT SHOULD NOT DO

1. DO NOT run DMS full load against PRIMARY in production hours
   → 500GB table scan causes I/O contention
   → Use read replica for full load
   → CDC binlog reading has minimal impact on primary

2. DO NOT skip the Bronze layer and transform directly from DMS
   → Raw data must be preserved (audit, reprocessing, debugging)
   → If Glue job has a bug, you need to reprocess from raw
   → Bronze is your safety net — NEVER skip it

3. DO NOT migrate all 45 tables at once
   → Start with smallest tables (reference/lookup) — validate process
   → Then medium tables (customers, products) — validate quality
   → Then large tables (orders, transactions) — validate performance
   → Batch approach reduces risk

4. DO NOT assume on-prem schema matches what Glue expects
   → MariaDB ENUM, SET, GEOMETRY types have no direct Parquet equivalent
   → TIMESTAMP vs DATETIME vs DATE — timezone handling differs
   → Run schema compatibility check BEFORE DMS starts

5. DO NOT forget to enable binary logging on MariaDB primary
   → CDC requires binlog in ROW format (not STATEMENT or MIXED)
   → Check: SHOW VARIABLES LIKE 'binlog_format' → must be ROW
   → Change requires MariaDB restart (plan with DBA)

6. DO NOT cutover on a Friday
   → Murphy's Law: anything that can break will break
   → Cutover on Tuesday/Wednesday — full team available for 3 days
   → Weekend = minimal support

7. DO NOT decommission on-prem until 30-day parallel validation passes
   → Keep on-prem primary running as hot standby
   → Rollback plan must be tested and documented
   → Board approval required before decommission

🚫 WHEN NOT TO USE

1. DO NOT use DMS for transformations
   → DMS is a REPLICATION tool, not an ETL tool
   → It can do basic column mapping and filtering
   → Complex transforms (joins, aggregations, derivations) → Glue

2. DO NOT use Glue for the initial 500GB bulk load
   → DMS is optimized for database → S3 bulk transfer
   → Glue would require JDBC connection (slow for 500GB)
   → DMS parallelizes table reads automatically

3. DO NOT use Redshift for Bronze/Silver storage
   → Redshift is expensive for raw data storage
   → S3 at $0.023/GB vs Redshift at $0.25/GB (RA3)
   → Use Redshift ONLY for Gold (aggregated, query-optimized)

4. DO NOT use real-time streaming (Kinesis) for this migration
   → Batch CDC (every 15 minutes) is sufficient
   → Real-time adds complexity without business value here
   → If sub-minute latency needed later → add Kinesis (Project 72)

5. DO NOT use AWS SCT (Schema Conversion Tool) for S3 target
   → SCT is for database-to-database schema conversion
   → Our target is S3 (no schema enforcement)
   → Schema is handled by Glue Catalog + Crawlers

│  ═══════════════════════════════════════════════════                     │
│                                                                         │
│  STEP 1: DMS connects to MariaDB source (read replica)                  │
│  STEP 2: For each table in scope:                                       │
│    → Executes: SELECT * FROM table (with optional WHERE)                │
│    → Reads in chunks (LOB chunk size configurable)                      │
│    → Converts MariaDB types → Parquet types                             │
│    → Writes to S3 in parallel files:                                    │
│      s3://bronze/raw/ecommerce/orders/LOAD00001/                        │
│        part-00000.parquet (64MB)                                        │
│        part-00001.parquet (64MB)                                        │
│        ...                                                               │
│  STEP 3: DMS writes _CONTROL file when table load completes             │
│  STEP 4: Full load statistics published to CloudWatch                   │
│                                                                         │
│  PARALLELISM:                                                            │
│  → DMS loads multiple tables simultaneously                              │
│  → MaxFullLoadSubTasks controls parallelism (default 8)                  │
│  → Each table gets its own thread                                        │
│  → Large tables are NOT split (single-threaded per table)                │
│  → For 120GB orders table: ~30 min on dms.r5.2xlarge over DX            │
│                                                                         │
│  ESTIMATED TRANSFER TIME:                                                │
│  → 500GB over Direct Connect (1Gbps): ~1.5 hours (theoretical)          │
│  → 500GB over VPN (100Mbps): ~12 hours                                   │
│  → With DMS overhead (conversion, writes): multiply by 1.5-2x            │
│  → Realistic estimate: 3-6 hours (DX) or 18-24 hours (VPN)              │
│                                                                         │
│  ═══════════════════════════════════════════════════                     │
│  PART 2: DMS CDC MECHANICS                                              │
│  ═══════════════════════════════════════════════════                     │
│                                                                         │
│  HOW CDC WORKS WITH MARIADB:                                             │
│  1. MariaDB writes every row change to binary log (binlog)               │
│  2. DMS connects as a replication client (like a read replica)           │
│  3. DMS reads binlog events in real-time                                 │
│  4. Converts events to CDC records:                                      │
│     ┌──────────────────────────────────────────────────┐                │
│     │ {                                                 │                │
│     │   "Op": "U",              ← I=Insert,U=Update,   │                │
│     │                              D=Delete              │                │
│     │   "before": {...},        ← previous values       │                │
│     │   "after": {...},         ← new values             │                │
│     │   "timestamp": "2025-01-15T03:00:01Z",            │                │
│     │   "transaction_id": "trx_12345"                    │                │
│     │ }                                                  │                │
│     └──────────────────────────────────────────────────┘                │
│  5. Batches CDC records and writes to S3:                                │
│     s3://bronze/raw/ecommerce/orders/CDC/                                │
│       2025/01/15/03/  ← hour-partitioned                                 │
│         cdc-00001.parquet                                                │
│         cdc-00002.parquet                                                │
│                                                                         │
│  CDC FILE FREQUENCY:                                                     │
│  → DMS buffers CDC records in memory                                     │
│  → Writes to S3 when buffer is full OR timer expires                     │
│  → CdcMaxBatchInterval: 60 seconds (configurable)                        │
│  → CdcMinFileSize: 32000 KB (configurable)                               │
│  → Result: new S3 file every 1-5 minutes during active changes           │
│                                                                         │
│  BINLOG REQUIREMENTS:                                                    │
│  → binlog_format = ROW (not STATEMENT or MIXED)                          │
│  → binlog_row_image = FULL (captures all columns, not just changed)      │
│  → log_bin = ON                                                          │
│  → binlog retention: ≥ 24 hours (DMS needs to catch up after restart)    │
│                                                                         │
│  ═══════════════════════════════════════════════════                     │
│  PART 3: FULL-LOAD-AND-CDC MODE (Combined)                              │
│  ═══════════════════════════════════════════════════                     │
│                                                                         │
│  TIMELINE:                                                               │
│  T=0:   DMS starts reading binlog position (saves checkpoint)            │
│  T=0:   DMS starts full load (SELECT * FROM each table)                  │
│  T=3h:  Full load completes for all tables                               │
│  T=3h:  DMS applies cached CDC changes (from T=0 to T=3h)               │
│  T=3.5h: CDC caught up — now streaming in real-time                      │
│  T=∞:   Continuous CDC streaming until task stopped                      │
│                                                                         │
│  WHY THIS IS SAFE:                                                       │
│  → Binlog position captured BEFORE full load starts                      │
│  → ALL changes during full load are captured in CDC                      │
│  → After full load, CDC changes are applied in order                     │
│  → Result: S3 has complete, consistent snapshot + all changes            │
│                                                                         │
│  ═══════════════════════════════════════════════════                     │
│  PART 4: GLUE CDC MERGE MECHANICS (Project 65 applied)                  │
│  ═══════════════════════════════════════════════════                     │
│                                                                         │
│  Bronze has TWO types of data:                                           │
│  1. LOAD files (historical full load)                                    │
│  2. CDC files (ongoing changes: I/U/D)                                   │
│                                                                         │
│  Glue ETL processes them differently:                                    │
│                                                                         │
│  INITIAL RUN (after full load):                                          │
│  → Read all LOAD files                                                   │
│  → Clean, transform, write to Silver as Parquet                          │
│  → This is the baseline Silver dataset                                   │
│                                                                         │
│  SUBSEQUENT RUNS (CDC):                                                  │
│  → Read new CDC files (using Job Bookmark — Project 33)                  │
│  → For each CDC record:                                                  │
│    → Op=I (Insert): add new row to Silver                                │
│    → Op=U (Update): find existing row by PK, replace with new values     │
│    → Op=D (Delete): remove row from Silver                               │
│  → MERGE pattern (Project 65):                                           │
│    → Load CDC into staging                                               │
│    → DELETE from Silver WHERE pk IN (staging PKs)                        │
│    → INSERT INTO Silver SELECT * FROM staging WHERE Op != 'D'            │
│    → This handles I/U/D correctly                                        │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "DMS full load took 48 hours instead of expected 6"
→ Cause: VPN bandwidth (100Mbps) instead of Direct Connect
→ 500GB ÷ 100Mbps = 11 hours theoretical + 2x overhead = 22+ hours
→ Fix: provision Direct Connect (1Gbps) — reduces to 3-6 hours
→ Alternative: AWS Snowball for initial load, DMS for CDC only

PITFALL 2: "CDC files have duplicates — same row appears twice"
→ Cause: DMS retry after network blip re-sends some CDC records
→ Fix: CDC merge logic must be IDEMPOTENT
→ Use primary key to deduplicate: last CDC record for each PK wins
→ ORDER BY timestamp DESC, take first

PITFALL 3: "MariaDB ENUM type becomes garbage in Parquet"
→ Cause: DMS doesn't map ENUM to string cleanly
→ Fix: DMS table mapping rule to cast ENUM to VARCHAR before transfer
→ Or: handle in Glue ETL with explicit type mapping

PITFALL 4: "Glue job OOMs processing 120GB orders table"
→ Cause: trying to read entire table into memory
→ Fix: process in partitions (by date range)
→ Use pushdown predicate: read only year=2024 partition per run
→ Scale DPU: 40 DPU for large tables (Project 69 cost optimization)

PITFALL 5: "Redshift COPY from Silver S3 is slow — 2 hours for 120GB"
→ Cause: Silver has too many small files (Project 62 problem)
→ Fix: Glue compaction job before COPY
→ Target: 128MB-512MB files for optimal Redshift COPY
→ Also: use COPY with MANIFEST for parallel loading (Project 21)

PITFALL 6: "CDC stops working after MariaDB failover"
→ Cause: binlog position changed when replica promoted to primary
→ DMS can't find the old binlog position on new primary
→ Fix: enable GTID (Global Transaction ID) on MariaDB
→ GTID survives failover — DMS resumes from correct position

PITFALL 7: "Cutover weekend — applications error rate spikes 500%"
→ Cause: application connection strings pointed to wrong endpoint
→ Or: Redshift cold start (first queries slow on Serverless)
→ Fix: pre-warm Redshift by running representative queries
→ Use DNS CNAME for database endpoint — change once, all apps update
