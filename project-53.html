✅ WHAT SHOULD DO

1. Create a dedicated IAM role for DataBrew
   → Needs: s3:GetObject (read source), s3:PutObject (write output)
   → Needs: glue:GetTable, glue:GetDatabase (if using Catalog)
   → Needs: logs:CreateLogGroup, logs:PutLogEvents (execution logs)
   → Principle of least privilege — only buckets DataBrew needs

2. Run a Profile Job FIRST before building recipes
   → Profile reveals: nulls, duplicates, outliers, distributions
   → Sarah sees the ACTUAL quality issues with statistics
   → Informs which recipe steps are needed
   → Costs pennies — runs on a sample

3. Use Glue Data Catalog as Dataset source (not raw S3 path)
   → Catalog table already exists from Project 23 (Crawler)
   → DataBrew inherits schema, partitioning, metadata
   → Changes in source schema auto-reflected

4. Version recipes
   → Recipe v1: basic dedup + lowercase
   → Recipe v2: add phone normalization
   → Recipe v3: add state mapping
   → Can rollback to any version if new rules break something

5. Output as Parquet (not CSV)
   → Downstream Athena queries 10x faster (Project 26)
   → Redshift COPY 5x faster (Project 21)
   → 70% less storage than CSV
   → Add partition columns (year, month, day) in output

6. Separate Profile Job output from Recipe Job output
   → Profile results → s3://bucket/databrew-profiles/
   → Cleaned data → s3://bucket/cleaned/customers/
   → Don't mix metadata with actual data

7. Schedule Recipe Jobs to run AFTER source data lands
   → Source file lands at midnight (Project 29)
   → DataBrew Recipe Job runs at 2 AM
   → Downstream Glue ETL runs at 4 AM (Project 35)
   → Clear dependency chain

8. Tag all DataBrew resources for cost tracking
   → Environment=Production, Team=Analytics, Owner=Sarah
   → DataBrew costs show up in Cost Explorer under these tags

❌ WHAT SHOULD NOT DO

1. DO NOT give Sarah direct S3 console access
   → She only needs DataBrew console access
   → DataBrew's IAM role handles S3 access
   → Sarah's IAM user/role needs: databrew:* only
   → S3 access via DataBrew role, not her credentials

2. DO NOT use DataBrew for complex multi-table JOINs
   → DataBrew works on single datasets
   → JOINs are limited (can join two datasets but awkward)
   → For complex joins: use Glue ETL (Project 34) or Athena (Project 37)

3. DO NOT run Profile Jobs on full dataset every time
   → Profile on 500K rows takes ~5 min, costs ~$0.50
   → Profile on 50M rows takes ~45 min, costs ~$5.00
   → Use sampling: profile 10% or first 20,000 rows
   → Full profile only when data characteristics change

4. DO NOT hardcode S3 paths in Dataset definition
   → Use parameterized paths with date variables
   → Or use Glue Catalog table (auto-discovers new partitions)
   → Hardcoded path = recipe only works for one day's file

5. DO NOT let Recipe Jobs overwrite source data
   → Source (Bronze): daily_exports/customers/
   → Output (Silver): cleaned/customers/
   → ALWAYS write to different S3 prefix
   → Source is immutable — never modify raw data (Project 44 principle)

6. DO NOT create one giant recipe with 50 steps
   → Hard to debug — which step introduced the issue?
   → Hard to maintain — Sarah can't find the phone step among 50
   → Break into logical groups: dedup recipe, format recipe, validate recipe
   → Chain recipe jobs in sequence

7. DO NOT ignore DataBrew job execution logs
   → Jobs can fail silently (wrong S3 permissions, schema mismatch)
   → Enable CloudWatch logging
   → Set up SNS notification on job failure (from Project 54)

🚫 WHEN NOT TO USE GLUE DATABREW

1. When transformation requires multi-table JOINs
   → DataBrew can join 2 datasets but awkwardly
   → 3+ table joins: use Glue ETL or Athena
   → Star schema joins: definitely Glue ETL

2. When processing >100 GB per job
   → DataBrew scales but becomes expensive at large volumes
   → $0.48/node/hour × 20 nodes × 2 hours = $19.20 per run
   → Glue ETL with auto-scaling DPUs may be cheaper at scale

3. When transformation logic changes frequently via CI/CD
   → DataBrew recipes are created via UI or API — not code files
   → Hard to version-control in Git (no native .py file)
   → Glue ETL PySpark scripts live in Git → CI/CD friendly (Project 76)

4. When you need real-time processing
   → DataBrew is batch only — minimum job = read full dataset
   → For streaming: Kinesis + Lambda (Project 72)

5. When analyst already knows PySpark or advanced SQL
   → DataBrew adds a layer of abstraction they don't need
   → Let them write Glue ETL or Athena SQL directly
   → DataBrew is specifically for NON-technical users

6. When data is in a database (not S3)
   → DataBrew can connect to databases via Glue connection
   → But: extracting large tables is slow
   → Better: Glue ETL JDBC extract → S3 → DataBrew on S3 data

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Recipe job fails with AccessDenied"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: DataBrew IAM role lacks S3 permissions
→ Needs: s3:GetObject on source bucket + s3:PutObject on output bucket
→ Common mistake: gave read permission but forgot write permission
→ Fix: check IAM role policy attached to DataBrew job

PITFALL 2: "Profile shows 0 rows — data not found"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Dataset S3 path doesn't match actual file location
→ Dataset points to: s3://bucket/customers/
→ Actual file is at: s3://bucket/daily_exports/customers/
→ Fix: verify S3 path exactly matches where files exist

PITFALL 3: "Recipe works on sample but fails on full dataset"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: sample had clean data, full dataset has edge cases
→ Example: sample had no NULL phone numbers, full dataset has 5,000
→ Recipe step assumes phone is always present → fails
→ Fix: always add NULL handling BEFORE format transformation steps
→ Fix: run Profile Job first to understand full data characteristics

PITFALL 4: "Session keeps timing out — work lost"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: DataBrew interactive session has idle timeout (default 30 min)
→ Sarah goes to lunch → session ends → unsaved recipe steps LOST
→ Fix: publish recipe frequently (save button)
→ Fix: increase session timeout via API (max 2880 min = 48 hours)
→ But: costs accumulate while session is active

PITFALL 5: "Output has wrong number of columns"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: recipe step accidentally deleted a column
→ Or: SPLIT_COLUMN created extra columns
→ Fix: check recipe steps in order — each step's output is next step's input
→ Fix: preview each step in interactive session before publishing

PITFALL 6: "Scheduled job ran but output is empty"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Dataset path is parameterized but date parameter wasn't set
→ Or: source file for today hasn't landed yet when job runs
→ Fix: schedule DataBrew AFTER source file landing (buffer time)
→ Fix: use S3 event trigger instead of cron (via Step Functions — Project 55)

PITFALL 7: "DataBrew costs more than expected"

NODE CONFIGURATION:

→ More nodes = faster but more expensive

RECIPE COMPLEXITY:
→ 5 steps: minimal overhead
→ 20 steps: noticeable processing time increase
→ 50+ steps: consider splitting into multiple recipe jobs
→ Each step processes full dataset sequentially

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY DATABREW OUTPUT:

1. Profile Job output (S3 JSON):
   → Column-level statistics: nulls, uniques, distributions
   → Compare BEFORE and AFTER profile results
   → Null count should decrease, unique count should change as expected

2. Recipe Job execution details (DataBrew console):
   → Duration, rows processed, rows output
   → Errors per step (if any)
   → Output file location and size

3. Athena verification (from Project 24):
   → Query the cleaned output directly in Athena
   → SELECT COUNT(DISTINCT state) FROM cleaned_customers
   → Should be exactly 50 (not 147 like before)
   → SELECT COUNT(*) WHERE email NOT LIKE '%@%'
   → Should be 0 (all invalid emails flagged or removed)

4. CloudWatch metrics:
   → DataBrew publishes: JobDuration, RowsProcessed, BytesProcessed
   → Monitor for unexpected drops (job ran but processed 0 rows)

5. Data quality comparison:
   → Run Profile Job on source data → save results
   → Run Recipe Job → output cleaned data
   → Run Profile Job on cleaned data → save results
   → Compare: null counts, duplicate counts, pattern distributions

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ Glue DataBrew is a VISUAL, NO-CODE data preparation service
→ Perfect for business analysts who don't know Python/Spark
→ Components: Dataset (source) → Project (workspace) → Recipe (transforms) → Job (execution)
→ Profile Jobs ANALYZE quality without changing data
→ Recipe Jobs APPLY transformations to full dataset
→ 250+ built-in transforms: dedup, case, fill, replace, format, validate
→ Recipes are versioned, reusable, and shareable
→ Schedulable via cron — fully automated nightly cleaning
→ Output: any format (CSV, Parquet, JSON) to S3
→ Cost: $0.48/node-hour — pennies for small datasets
→ Engineers set up infrastructure. Analysts own the recipes.
