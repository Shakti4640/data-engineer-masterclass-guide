🟡 HOW TO APPROACH IT

Step 1: Understand Athena's pricing model
→ $5.00 per TB of data scanned
→ No charge for: DDL, failed queries, metadata operations
→ Minimum charge: 10 MB per query (even if query scans 1 KB)
→ Cancelled queries: charged for data scanned before cancellation
→ NO infrastructure cost — no cluster, no instances, no idle cost

Step 2: Understand what makes Athena cheap vs expensive
→ CHEAP: scan only needed columns from Parquet (columnar)
→ CHEAP: scan only needed partitions (partition pruning)
→ CHEAP: use LIMIT on exploratory queries
→ EXPENSIVE: SELECT * FROM huge_csv_table (full row scan)
→ EXPENSIVE: scanning uncompressed CSV (no column pruning)
→ EXPENSIVE: no partitioning (scans everything for every query)

Step 3: Map analyst workflows to Athena patterns
→ Exploration: "What does this data look like?" → SELECT * LIMIT 10
→ Aggregation: "Total revenue by status" → SELECT status, SUM(amount)
→ Filtering: "Orders from last week" → WHERE order_date BETWEEN ...
→ Joining: "Orders with customer names" → JOIN on customer_id
→ Export: "Give me results as CSV" → Athena auto-saves results to S3

Step 4: Set up governance and cost control
→ Workgroups: isolate teams, set per-query data scan limits
→ Query result location: organized S3 prefix per team
→ Named queries: save common queries as templates
→ Cost monitoring: CloudWatch metrics for data scanned

✅ WHAT SHOULD DO

1. Set up WORKGROUPS per team
   → Isolates query costs (who spent what)
   → Sets per-query scan limits (prevents runaway queries)
   → Separate query result locations per team
   → Enables per-team CloudWatch metrics

2. Set PER-QUERY data scan limits
   → Finance: 1 GB max (small aggregations)
   → Marketing: 5 GB max (larger cohort analysis)
   → Engineering: 50 GB max (full table scans allowed)
   → Prevents accidental SELECT * on 100 GB table

3. ALWAYS select specific columns — never SELECT *
   → CSV: SELECT * scans ALL bytes (no column pruning)
   → Parquet: SELECT col1, col2 scans only those columns
   → Training analysts on this ONE habit saves 80% of cost

4. Use Parquet tables whenever possible
   → Same query on CSV vs Parquet:
     CSV:     SELECT status, SUM(amount) → scans 1.5 GB → $0.0075
     Parquet: SELECT status, SUM(amount) → scans 50 MB  → $0.00025
   → 30x cheaper for same result
   → Project 25 converts CSV → Parquet via Glue ETL

5. Configure query result S3 location with lifecycle
   → Athena saves every result as CSV in S3
   → 200 queries/day = 200 result files/day = clutter
   → Set S3 lifecycle: delete results after 7 days
   → Saves storage cost and keeps bucket clean

6. Save common queries as NAMED QUERIES
   → Analysts reuse instead of writing from scratch
   → Prevents typos and wrong table references
   → Consistent patterns across the team

7. Use EXPLAIN before large queries
   → Shows estimated data scan BEFORE executing
   → Catch expensive queries before they run
   → No cost for EXPLAIN itself

8. Set up CloudWatch alarms on Athena spend
   → Metric: ProcessedBytes per workgroup
   → Alarm: if daily scan > 10 GB → notify via SNS (Project 12)
   → Catches unexpected cost spikes before bill arrives

❌ WHAT SHOULD NOT DO

1. DO NOT let analysts use the "primary" workgroup
   → No scan limits, no cost tracking per team
   → One bad query scans 500 GB = $2.50 surprise
   → ALWAYS assign analysts to their team's workgroup

2. DO NOT use SELECT * on CSV tables
   → CSV is ROW-BASED — Athena reads entire row even for 1 column
   → SELECT * FROM raw_orders → scans 1.5 GB
   → SELECT order_id, total_amount FROM raw_orders → STILL scans 1.5 GB (CSV!)
   → This is WHY we convert to Parquet (Project 25)

3. DO NOT query unpartitioned tables with date filters
   → WHERE order_date = '2025-01-15' on unpartitioned table
   → Athena scans ALL files → filters AFTER scanning
   → With partitioning: scans ONLY that date's folder
   → Project 26 covers partitioning in depth

4. DO NOT forget to set query result encryption
   → Query results land as plain CSV in S3
   → If data contains PII → unencrypted results = compliance violation
   → Set workgroup: ResultConfiguration.EncryptionConfiguration

5. DO NOT run heavy aggregations on raw CSV repeatedly
   → Same GROUP BY query run 10 times/day = 10x cost
   → Use CTAS to materialize results as Parquet table (Project 36)
   → Then query the materialized table (near-zero cost)

6. DO NOT ignore failed queries in cost calculation
   → Cancelled queries STILL charge for data scanned before cancellation
   → Query that scans 5 GB then times out: you pay $0.025
   → Set appropriate timeout in workgroup configuration

7. DO NOT assume Athena is always cheaper than Redshift
   → Athena cost grows linearly with data scanned
   → 1 TB scanned/day × 30 days = 30 TB = $150/month
   → At this scale: Redshift may be cheaper (fixed cost)
   → Rule of thumb: < 1 TB/day scanned → Athena wins
   → > 5 TB/day scanned → Redshift wins
   → 1-5 TB/day → analyze specific workload pattern

8. DO NOT create tables directly in Athena when crawlers exist
   → Athena CREATE TABLE creates catalog entry
   → Crawler also creates catalog entry
   → Running both: duplicate tables, confusion
   → Rule: ONE source of truth for each table's schema

🚫 WHEN NOT TO USE ATHENA

1. When you need sub-second query response
   → Athena cold start: 2-5 seconds minimum
   → Complex queries: 10-60 seconds
   → Use: Redshift (cached), ElastiCache, DynamoDB

2. When same query runs 100+ times/day on same data
   → Each run re-scans S3 → re-charges
   → 100 runs × 1 GB scan = 100 GB/day = $0.50/day
   → Redshift with materialized view: runs once, serves from cache

3. When you need transactions (INSERT/UPDATE/DELETE)
   → Athena is READ-ONLY on S3 data (with few exceptions via Iceberg)
   → CTAS creates NEW tables but doesn't modify existing
   → For writes: use Redshift, MariaDB, or Glue ETL

4. When data volume per query exceeds 5 TB regularly
   → $25 per query at 5 TB scan
   → Redshift with dedicated cluster likely cheaper
   → Unless queries are very infrequent

5. When concurrent query count exceeds 20-30
   → Athena has soft limit of 20-30 concurrent queries per account
   → Can request increase, but not designed for 1000s concurrent
   → Use: Redshift with concurrency scaling for high concurrency

6. When you need query result caching
   → Athena re-scans S3 every time (no built-in cache)
   → Workaround: CTAS to materialize (Project 36)
   → Redshift: result cache returns instantly for identical queries

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Query returns HIVE_BAD_DATA error"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: data in S3 doesn't match catalog schema
→ Example: catalog says column is INT, but CSV has "N/A" in that column
→ Debug: SELECT column_name FROM table WHERE column_name IS NOT NULL LIMIT 100
→ Fix A: update catalog type to STRING (Project 23, Step 6)
→ Fix B: clean source data to remove invalid values
→ Fix C: use try_cast() in Athena v3: try_cast(col AS INTEGER)

PITFALL 2: "Query scans entire table despite WHERE clause"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: table is NOT partitioned — WHERE is applied AFTER full scan
→ WHERE order_date = '2025-01-15' still scans ALL 1.5 GB
→ Athena reads everything → filters in memory → returns matching rows
→ Fix: partition the table by date (Project 26)
→ With partitioning: only scans 50 MB (that date's folder)

PITFALL 3: "Athena returns 0 rows but data exists in S3"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause A: catalog table points to wrong S3 location
→ Cause B: SerDe mismatch (CSV configured but files are Parquet)
→ Cause C: partitioned table with no partitions loaded
→ Debug A: check table location in Glue console
→ Debug B: verify table classification matches actual file format
→ Debug C: run MSCK REPAIR TABLE tablename; to load partitions

PITFALL 4: "Query takes 60 seconds for simple COUNT(*)"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause A: scanning huge uncompressed CSV files
→ Cause B: thousands of tiny files (S3 listing overhead)
→ Cause C: cold start on first query of the day
→ Fix A: convert to Parquet (Project 25)
→ Fix B: compact small files (Project 62)
→ Fix C: accept 2-5 second cold start (serverless trade-off)

PITFALL 5: "Query costs $5 for what should be a simple query"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: SELECT * on 1 TB unpartitioned CSV table
→ $5 per TB scanned = $5 for one query
→ Fix: NEVER SELECT *, always specify columns
→ Fix: partition the table
→ Fix: convert to Parquet
→ Prevention: set workgroup scan limit to catch this

PITFALL 6: "Different analysts get different results for same query"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: S3 data changed between query executions
→ Athena has NO snapshot isolation — reads whatever is in S3 right now
→ If ETL is running while analyst queries → partial/inconsistent data
→ Fix: schedule ETL during off-hours
→ Fix: use separate "published" S3 prefix that ETL atomically swaps

📈 SCALABILITY CONSIDERATIONS

QUERY CONCURRENCY:
━━━━━━━━━━━━━━━━━━
→ Default: 20 concurrent DML queries per account per region
→ Default: 20 concurrent DDL queries per account per region
→ Can request increase to 100+
→ Queued queries: up to 200 in queue (auto-retried)
→ For 10 analysts with 5 concurrent queries each = 50 concurrent → request increase

DATA VOLUME:
━━━━━━━━━━━━
→ No limit on data size in S3
→ Athena can query petabytes of data
→ But cost scales linearly with data scanned
→ 1 PB scan = $5,000 per query (obviously avoid this)
→ Partitioning + columnar format = essential at scale

QUERY COMPLEXITY:
━━━━━━━━━━━━━━━━━
→ Simple aggregation on 10 GB: 5-15 seconds
→ JOIN between two 10 GB tables: 20-60 seconds
→ Complex multi-table JOIN with window functions: 1-5 minutes
→ Timeout: 30 minutes default (configurable)
→ Very complex queries: consider Redshift or Glue Spark instead

RESULT SIZE:
━━━━━━━━━━━━
→ Max result size: limited by S3 (practically unlimited)
→ Console display: limited to 1000 rows in browser
→ API: full result accessible via S3 output file
→ Very large results: use CTAS instead (creates table, not CSV)

COST OPTIMIZATION LADDER:
━━━━━━━━━━━━━━━━━━━━━━━━━
→ Level 0: SELECT * on CSV (100% cost)
→ Level 1: SELECT columns on CSV (100% cost — CSV can't prune)
→ Level 2: SELECT columns on Parquet (30% cost — column pruning)
→ Level 3: Parquet + partitioning (3-10% cost — partition pruning)
→ Level 4: Parquet + partitioning + compression (1-5% cost)
→ Level 5: CTAS materialized tables for repeat queries (~0% marginal)

MONTHLY COST PROJECTIONS:
━━━━━━━━━━━━━━━━━━━━━━━━━
→ 10 analysts × 30 queries/day × 100 MB/query = 30 GB/day
→ 30 GB/day × 30 days = 900 GB/month
→ 900 GB × $0.005/GB = $4.50/month
→ With Parquet optimization: 900 GB → 90 GB → $0.45/month

HOW TO VERIFY ATHENA QUERIES WORK CORRECTLY:

   → Shows: query ID, duration, data scanned, status
   → Filter by workgroup to see team-specific usage

5. CloudWatch metrics
   → ProcessedBytes: total bytes scanned
   → TotalExecutionTime: query duration
   → Per-workgroup breakdown available
