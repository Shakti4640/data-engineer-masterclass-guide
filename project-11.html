🔵 WHAT COULD BE DONE

Option A: SCD Type 1 (Overwrite) for all dimensions
→ Simply UPDATE the dimension row when source changes
→ Pros: simple, small table, easy to query
→ Cons: LOSE history — can't answer "what was tier in January?"
→ Use for: product name changes, typo corrections

Option B: SCD Type 2 (Versioned Rows) for key dimensions
→ Create NEW row for each change, mark old row as expired
→ Pros: FULL history — "customer was silver on Jan 15"
→ Cons: table grows larger, JOINs need effective_date filter
→ Use for: customer tier, address, any analytically important attribute

Option C: SCD Type 3 (Previous Value Column)
→ Add "previous_tier" column alongside "current_tier"
→ Pros: simple schema, no extra rows
→ Cons: only tracks ONE previous value, not full history
→ Use for: rarely changes, only need last value

Option D: SCD Type 6 (Hybrid: 1 + 2 + 3)
→ Versioned rows (Type 2) + current value column (Type 3)
→ Pros: full history + easy access to current value
→ Cons: complex to maintain

DECISION:
→ dim_customers: SCD Type 2 (tier changes are analytically critical)
→ dim_products: SCD Type 1 (product name/price changes just overwrite)
→ dim_dates: Static (dates don't change)

✅ WHAT SHOULD DO

1. SCD Type 2 columns on customer dimension:
   → surrogate_key (BIGINT): unique per VERSION of customer
   → customer_id (VARCHAR): natural business key
   → effective_from (DATE): when this version became active
   → effective_to (DATE): when this version expired (9999-12-31 = current)
   → is_current (BOOLEAN): TRUE for latest version
   → These enable point-in-time JOINs with fact tables

2. Design fact tables for COPY compatibility:
   → All columns nullable except surrogate keys
   → DATE columns as DATE type (not VARCHAR)
   → DECIMAL for money (not FLOAT — same as MariaDB, Project 9)
   → No IDENTITY/AUTO_INCREMENT — generate IDs in ETL
   → Column order matches CSV column order (COPY positional mapping)

3. Create staging tables as EXACT copies of fact/dim structure:
   → No constraints (staging accepts raw data, validation happens after)
   → DISTSTYLE EVEN (no optimization needed for temp data)
   → No SORTKEY (data will be replaced each load)
   → TRUNCATE before each load (idempotent)

4. Create denormalized wide fact table:
   → Pre-join fact_orders + dim_customers + dim_products
   → Redundant storage BUT eliminates 3-way JOINs in 90% of queries
   → Refresh nightly after ETL completes

5. Add audit columns to EVERY table:
   → etl_loaded_at: when this row was loaded by ETL
   → etl_batch_id: which batch loaded this row
   → source_system: where the data came from
   → These are CRITICAL for debugging pipeline issues

6. Create aggregate tables for dashboard performance:
   → agg_daily_revenue: pre-computed daily summaries
   → Dashboards query aggregates (milliseconds) instead of fact tables (seconds)
   → Refresh on schedule or after each ETL run

7. Use ENCODE AUTO for initial setup:
   → Redshift automatically selects best encoding per column
   → After data is loaded: validate with ANALYZE COMPRESSION
   → Override only if ANALYZE suggests better encoding

❌ WHAT SHOULD NOT DO

1. DO NOT use IDENTITY columns in fact tables
   → COPY from S3 can't auto-generate IDENTITY values easily
   → Generate surrogate keys in ETL (hash or sequence)
   → Exception: staging tables where IDs don't matter

2. DO NOT put SCD Type 2 on EVERY dimension
   → Each SCD2 dimension grows with every change
   → More rows = more storage, slower JOINs
   → Only SCD2 for analytically significant attributes
   → Product name typo fix: SCD Type 1 (overwrite)

3. DO NOT JOIN fact table directly to SCD2 dimension on natural key alone
   → fact.customer_id = dim.customer_id returns MULTIPLE rows
     (one per version of the customer)
   → MUST add: AND fact.order_date BETWEEN dim.effective_from AND dim.effective_to
   → OR: store surrogate_key in fact table (preferred)

4. DO NOT create aggregates that are too specific
   → agg_daily_revenue_by_status_by_city_by_tier = too granular
   → Might as well query the fact table
   → Good aggregates: daily by 1-2 dimensions, monthly by 2-3 dimensions

5. DO NOT forget to TRUNCATE staging tables before each load
   → If you don't: previous batch data PLUS new batch = duplicates
   → Always: TRUNCATE → COPY → VALIDATE → MERGE → TRUNCATE

6. DO NOT use TEXT or VARCHAR(MAX) in Redshift
   → Redshift allocates memory for full declared width during queries
   → VARCHAR(65535) for a 20-char column = massive memory waste
   → Always size to actual max + reasonable buffer

7. DO NOT skip the surrogate key in SCD2 dimensions
   → Natural key (customer_id) is NOT unique in SCD2
   → Need surrogate_key (auto-incrementing BIGINT) as true PK
   → Fact table stores surrogate_key for point-in-time accuracy

🚫 WHEN NOT TO USE

1. DO NOT use SCD Type 2 for high-frequency changes
   → Product inventory changes every hour → SCD2 = millions of versions
   → Use: snapshot fact table (daily inventory snapshot) instead
   → Or: SCD Type 1 + separate inventory_history fact table

2. DO NOT use wide fact tables for ad-hoc exploratory queries
   → Wide tables are designed for KNOWN query patterns
   → Ad-hoc queries may need columns not in the wide table
   → Use: normalized star schema for exploration, wide for dashboards

3. DO NOT use aggregate tables if underlying data changes intraday
   → Aggregates are stale until refreshed
   → If orders arrive every minute: aggregate is immediately outdated
   → Use: materialized views (Project 63) with auto-refresh instead

4. DO NOT use staging tables for long-term storage
   → Staging = temporary landing zone
   → TRUNCATE before each load
   → Never query staging from BI tools — always query analytics schema

5. DO NOT create separate schemas for each business domain (yet)
   → At QuickCart's scale: one analytics schema is fine
   → At enterprise scale: data mesh with domain schemas (Project 71)

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "SCD2 JOIN returns duplicate rows"
→ Cause: fact.customer_id = dim.customer_id without date range filter
→ Customer has 3 versions → each fact row joins to ALL 3 versions
→ Fix: use surrogate_key in fact table OR add date range condition

PITFALL 2: "Wide fact table is out of sync with dimension tables"
→ Cause: dimension updated but wide fact not refreshed
→ Customer tier changed yesterday, wide table still shows old tier
→ Fix: rebuild wide table AFTER dimension load completes
→ Use Glue Workflow or Step Functions to enforce order (Project 35, 55)

PITFALL 3: "ANALYZE COMPRESSION suggests different encoding but can't change"
→ Cause: Redshift doesn't allow ALTER COLUMN ENCODE on existing data
→ Fix: CREATE new table with correct encoding → INSERT INTO...SELECT → rename
→ This is called "deep copy" — standard Redshift practice

PITFALL 4: "Staging table has duplicates after COPY"
→ Cause: COPY ran twice (retry after timeout) without TRUNCATE first
→ Fix: ALWAYS TRUNCATE staging before COPY — make loads idempotent
→ Pattern: BEGIN; TRUNCATE stg_orders; COPY stg_orders FROM ...; COMMIT;

PITFALL 5: "Sort key on order_date not working — queries still scan everything"
→ Cause: data loaded out of order (random dates in each COPY batch)
→ Zone maps have overlapping ranges → can't eliminate blocks
→ Fix: VACUUM SORT ONLY fact_orders; (re-sorts physical blocks)
→ Check: SELECT unsorted FROM svv_table_info WHERE tablename = 'fact_orders';
→ unsorted > 20% = need VACUUM

PITFALL 6: "Surrogate key collision in SCD2"
→ Cause: using hash of (customer_id + effective_from) as SK
→ Two changes on same day = same hash = collision
→ Fix: use monotonically increasing sequence or include timestamp
→ Or: generate SK as ROW_NUMBER() OVER (ORDER BY ...) during ETL

PITFALL 7: "Aggregate table has different totals than fact table"
→ Cause: aggregate computed before ETL finished, or rounding differences
→ Fix: always rebuild aggregates AFTER fact table load completes
→ Add verification query: compare agg SUM vs fact SUM

📈 SCALABILITY CONSIDERATIONS

SCD TYPE 2 GROWTH:
→ 500 customers × 1 change/year = 500 new rows/year
→ After 5 years: 500 + (500 × 5) = 3,000 rows
→ Still tiny — DISTSTYLE ALL is fine
→ RISK: if customer attributes change daily (e.g., loyalty points)
   → SCD2 grows to 500 × 365 × 5 = 912,500 rows
   → Still manageable but consider: is daily SCD2 really needed?

WIDE FACT TABLE GROWTH:
→ Same rows as fact_orders BUT wider (more columns per row)
→ 100K orders × 20 columns vs 100K × 8 columns
→ ~2.5x more storage than normalized fact
→ Trade-off: 2.5x storage cost vs eliminating JOINs in 90% of queries
→ At 1 billion rows: storage difference is significant — consider carefully

AGGREGATE TABLE REFRESH:
→ Daily refresh: runs once, query benefit all day
→ Hourly refresh: more current but 24x compute cost
→ Real-time: use materialized views (Project 63) instead
→ At QuickCart scale: daily refresh is fine

STAGING TABLE SIZING:
→ Stage only ONE batch at a time
→ TRUNCATE before each load — never accumulates
→ Size = one day's data (worst case)
→ QuickCart: ~200K orders/day × ~150 bytes/row = ~30 MB staging
→ Trivial at any scale because it's always ONE batch

FACT TABLE AT SCALE:
→ 1 million rows: any design works
→ 100 million rows: distribution key critical, sort key critical
→ 1 billion rows: encoding saves terabytes, VACUUM takes hours
→ 10 billion rows: consider partitioning (date-range tables) or Spectrum

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VALIDATE THE DATA MODEL:

1. SCD2 Validation:
   → Every customer_id must have exactly ONE row where is_current = TRUE
   → SELECT customer_id, COUNT(*) FROM dim_customers_scd2 
     WHERE is_current = TRUE GROUP BY customer_id HAVING COUNT(*) != 1;
   → Should return ZERO rows

2. Surrogate Key Validation:
   → Every fact_orders.customer_sk must exist in dim_customers_scd2.sk
   → SELECT COUNT(*) FROM fact_orders f
     LEFT JOIN dim_customers_scd2 d ON f.customer_sk = d.surrogate_key
     WHERE d.surrogate_key IS NULL;
   → Should return ZERO

3. Distribution Validation:
   → SELECT "table", skew_rows FROM svv_table_info WHERE "schema" = 'analytics';
   → skew_rows < 1.5 = good, > 2.0 = problem

4. Sort Key Validation:

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ SCD TYPE 2: version dimension rows to preserve point-in-time history
   → surrogate_key uniquely identifies each VERSION
   → effective_from/to defines validity window
   → is_current flag simplifies "latest version" queries
   → Fact table stores surrogate_key for accurate historical JOINs

→ STAGING PATTERN: TRUNCATE → COPY → VALIDATE → MERGE → TRUNCATE
   → Staging tables: no constraints, DISTSTYLE EVEN
   → Always idempotent: re-running produces same result

→ WIDE FACT TABLE: denormalize for dashboard performance
   → Pre-join eliminates 3-way JOINs in 90% of queries
   → Trade-off: more storage, simpler queries, must rebuild after loads

→ AGGREGATE TABLES: pre-compute for sub-second dashboards
   → Daily/monthly summaries avoid scanning millions of rows
   → Cross-validate: aggregate totals must match fact totals

→ AUDIT TABLES: track every ETL execution
   → etl_job_log: what ran, when, how many rows, success/failure
   → etl_row_counts: source vs target count validation

→ PHYSICAL DESIGN VALIDATION: prove, don't assume
   → Distribution skew: svv_table_info.skew_rows < 1.5
   → Sort effectiveness: svv_table_info.unsorted < 20%
   → Encoding: ANALYZE COMPRESSION shows optimal vs current
   → EXPLAIN: shows redistribution strategy (DS_DIST_ALL_NONE = good)
