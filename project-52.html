🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — Black Friday approaching
Team: 8 engineers, 3 data engineers

CURRENT ARCHITECTURE (from Projects 27, 28, 51):
→ SNS topic fans out to 4 filtered SQS queues (Project 51)
→ Each queue has ONE EC2 worker polling and processing (Project 27)
→ Workers are manually launched, manually managed

NORMAL DAY (Monday):
→ ETL Queue:   ~5,000 messages/day → 1 worker handles easily
→ Order Queue: ~50,000 messages/day → 1 worker handles with margin
→ Alert Queue: ~200 messages/day → 1 worker barely awake

BLACK FRIDAY (actual numbers from last year):
→ ETL Queue:   ~500,000 messages/day (100x spike at midnight)
→ Order Queue: ~5,000,000 messages/day (100x spike at 6 AM)
→ Alert Queue: ~50,000 messages/day (cascading failures)

WHAT HAPPENED LAST BLACK FRIDAY:
→ 12:01 AM: File landing events flood ETL Queue
→ 12:15 AM: Single ETL worker backlog hits 200,000 messages
→ 12:30 AM: Visibility timeout expires → messages re-appear → duplicates
→ 1:00 AM:  On-call engineer wakes up, SSH into EC2, manually starts 5 more workers
→ 1:30 AM:  Backlog starts draining
→ 6:00 AM:  Order Queue explodes → same problem
→ 6:15 AM:  Engineer manually starts 20 more Order workers
→ 11:00 PM: Traffic normalizes → 25 idle workers still running
→ COST: 25 workers × 23 hours × $0.10/hr = $57.50 wasted on idle machines
→ SLA BREACHED: 90 minutes of processing delay

CTO: "This cannot happen again. Workers must scale UP when queue
      grows and scale DOWN when queue empties — automatically."

REQUIREMENTS:
→ Scale OUT when queue depth > 1,000 messages (add workers)
→ Scale IN when queue depth < 100 messages (remove workers)
→ Minimum: 1 worker always running (baseline)
→ Maximum: 20 workers (budget cap)
→ Scale-out must happen within 3 minutes
→ Scale-in must wait 15 minutes of low traffic (avoid thrashing)
→ Each filtered queue (from Project 51) scales INDEPENDENTLY

🟡 HOW TO APPROACH IT

Step 1: Understand what Auto Scaling Group (ASG) does
→ Manages a FLEET of EC2 instances as one unit
→ Automatically launches new instances (scale out)
→ Automatically terminates idle instances (scale in)
→ Maintains desired count even if an instance dies (self-healing)
→ Distributes across Availability Zones (high availability)

Step 2: Understand the scaling trigger
→ We need: "when SQS queue has many messages → add EC2 workers"
→ SQS publishes metric: ApproximateNumberOfMessagesVisible (to CloudWatch)
→ CloudWatch Alarm watches this metric
→ Alarm triggers ASG scaling policy
→ ASG launches/terminates EC2 instances

Step 3: Understand Launch Template
→ Defines WHAT to launch: AMI, instance type, security group, IAM role
→ Includes User Data script: bootstrap the worker on boot
→ Worker script auto-starts, auto-connects to SQS, auto-processes
→ No SSH needed — fully automated

Step 4: Design the scaling chain
→ SQS Queue (metric source)
   → CloudWatch Alarm (decision maker)
     → ASG Scaling Policy (action taker)
       → EC2 Launch Template (instance blueprint)
         → Worker Script (actual processing)

Step 5: Choose scaling strategy
→ Option A: Simple Scaling — 1 alarm = 1 action (add N instances)
→ Option B: Step Scaling — different actions at different thresholds
→ Option C: Target Tracking — maintain metric at target value (simplest)
→ We use Step Scaling for fine-grained control + Target Tracking for comparison

🔵 WHAT COULD BE DONE

Option A: Manual scaling (current — last Black Friday disaster)
→ Engineer SSHs in, starts more worker processes
→ Forgets to stop them after → wasted cost
→ 3 AM response time → SLA breach

Option B: Cron-based scheduled scaling
→ Scale to 20 workers at 11 PM every Black Friday
→ Scale back at midnight
→ PROBLEMS: doesn't handle unexpected spikes (random viral product)
→ Over-provisions on slow days → wasted cost

Option C: CloudWatch Alarm + ASG Step Scaling (this project) ✅
→ React to ACTUAL queue depth — not predicted load
→ Step policies: proportional response to backlog size
→ Scale-in cooldown prevents thrashing
→ Fully automatic — zero human intervention

Option D: Target Tracking Scaling
→ Tell ASG: "keep backlog per instance at 500 messages"
→ ASG calculates how many instances needed
→ Simpler to configure but less fine-grained control
→ We implement BOTH for comparison

Option E: Lambda consumers instead of EC2
→ Each message triggers a Lambda invocation
→ Infinite auto-scaling built-in
→ PROBLEMS: 15-min max runtime, cold starts, cost at high volume
→ Better for lightweight processing, not heavy ETL

DECISION: Option C (Step Scaling) + Option D (Target Tracking demo)
→ Reason: production-grade, cost-efficient, fully automated
→ Step scaling for fine-grained control on ETL Queue
→ Target tracking for simplicity comparison

✅ WHAT SHOULD DO

1. Use Launch Template (NOT Launch Configuration)
   → Launch Configuration is legacy — cannot be modified after creation
   → Launch Template supports versioning, defaults, overrides
   → Can create new versions without recreating ASG

2. Include FULL worker bootstrap in User Data
   → Install Python, pip, dependencies
   → Download worker script from S3 (not baked into AMI)
   → Start worker as systemd service (survives reboot)
   → This way: update worker script in S3, launch new instance = new code

3. Use IAM Instance Profile (from Project 6)
   → EC2 workers need: sqs:ReceiveMessage, sqs:DeleteMessage, sqs:GetQueueAttributes
   → Plus whatever the processing logic needs (s3:GetObject, etc.)
   → NEVER put credentials in User Data

4. Spread across multiple Availability Zones
   → ASG distributes instances across AZs automatically
   → If one AZ goes down, workers in other AZs continue processing
   → Queue doesn't care which AZ the worker is in

5. Set scale-in cooldown period (300-900 seconds)
   → After removing an instance, WAIT before removing another
   → Prevents "scale in too fast → queue grows → scale out again" thrashing
   → 15 minutes is a good default for data workloads

6. Enable instance scale-in protection for active workers
   → If a worker is mid-processing a batch, don't terminate it
   → Use lifecycle hooks OR application-level protection

7. Tag ASG instances for cost tracking
   → Tag: AutoScalingGroup=quickcart-etl-workers
   → Tag: CostCenter=Data-Engineering
   → Shows up in AWS Cost Explorer per-ASG

8. Set up SNS notification on ASG scaling events
   → Know when instances are launched/terminated
   → Useful for debugging and audit trails
   → Reuses SNS knowledge from Projects 12, 51

❌ WHAT SHOULD NOT DO

1. DO NOT scale on CPU/Memory metrics for queue workers
   → CPU can be low even with 500K message backlog
   → Worker may be I/O-bound (waiting on S3, Redshift)
   → Scale on QUEUE DEPTH — that's the actual demand signal

2. DO NOT set scale-in cooldown too short (<5 minutes)
   → Scenario: queue drains → scale in → new messages arrive → scale out
   → Repeat every 2 minutes = thrashing
   → Each launch/terminate costs time + money
   → Minimum 10-15 minutes for data workloads

3. DO NOT use the same ASG for different queue types
   → ETL workers and Order workers have different:
     - Instance types (ETL needs more memory)
     - Scaling thresholds (ETL tolerates higher backlog)
     - Processing logic
   → One ASG per queue type (from Project 51 filtered routing)

4. DO NOT forget to handle graceful shutdown
   → ASG sends SIGTERM before terminating an instance
   → Worker must: finish current message, delete from queue, then exit
   → If worker is killed mid-processing: message re-appears after visibility timeout
   → Idempotent processing (from Project 27) is critical

5. DO NOT set Max too high without budget approval
   → 20 × t3.medium × 24 hours = $38.40/day = $1,152/month
   → If bug causes infinite scale-out: massive bill
   → Set Max conservatively, increase only when proven needed

6. DO NOT bake worker code into AMI
   → AMI rebuild takes 15-30 minutes
   → Code changes require new AMI → new Launch Template version → rolling update
   → Instead: download latest code from S3 on boot (User Data)
   → Code update = upload to S3 + terminate old instances

7. DO NOT ignore Desired Capacity vs Min Capacity
   → Min = absolute minimum (even during zero traffic)
   → Desired = current target (ASG adjusts this via scaling policies)
   → If you set Desired=10 manually, scaling policies override it
   → Set initial Desired = Min, let policies manage from there

🚫 WHEN NOT TO USE EC2 AUTO SCALING FOR QUEUE PROCESSING

1. When each message takes < 30 seconds to process
   → Lambda is cheaper and simpler
   → Lambda SQS trigger handles scaling automatically
   → No ASG, no Launch Template, no User Data

2. When processing requires GPU
   → GPU instances (p3, g4) have limited availability
   → ASG may fail to launch if no GPU capacity
   → Use: reserved GPU instances + manual management
   → Or: SageMaker Processing for ML workloads

3. When processing requires licensed software per-instance
   → Each new EC2 = new license cost
   → Scaling to 20 instances = 20 licenses
   → Use: fixed fleet with queue prioritization instead

4. When message volume is steady (no spikes)
   → Fixed 2 workers handle 100K messages/day consistently
   → Auto scaling adds complexity for zero benefit
   → Use: simple EC2 fleet with CloudWatch alarm for anomalies

5. When processing is stateful (must resume from exact point)
   → ASG can terminate any instance during scale-in
   → Stateful processing loses in-progress work
   → Use: ECS with persistent volumes, or EBS-backed instances with lifecycle hooks

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Scaled to 20 instances but queue isn't draining faster"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: bottleneck is NOT the worker count
→ Bottleneck is downstream: Redshift connection limit, S3 throttling, MariaDB locks
→ 20 workers all hitting Redshift with COPY = connection exhaustion
→ Fix: identify actual bottleneck BEFORE scaling
→ Rule: scale workers only if workers ARE the bottleneck

PITFALL 2: "Instances launch but never start processing"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: User Data script fails silently
→ pip install fails (no internet → NAT Gateway missing)
→ Worker script has syntax error
→ S3 download of worker code fails (IAM permission missing)
→ Fix: log User Data output to /var/log/user-data.log
→ Debug: check CloudWatch Logs agent output
→ Test: launch ONE instance manually with User Data before creating ASG

PITFALL 3: "Scale-out/scale-in loop — constant oscillation"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: cooldown period too short
→ Timeline:
  T+0: Queue = 2000 → scale out +5 instances
  T+3: Queue = 200 (draining) → scale in -5 instances
  T+6: Queue = 1500 (new messages) → scale out +5 again
→ Fix: set scale-in cooldown to 600-900 seconds (10-15 min)
→ Fix: set scale-out alarm evaluation period to 2+ datapoints

PITFALL 4: "Instance terminated mid-processing → duplicate message"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: ASG terminates instance during scale-in
→ Worker was mid-processing a message
→ Visibility timeout expires → message reappears → another worker processes it
→ Result: DUPLICATE processing
→ Fix 1: Lifecycle hooks — give worker time to finish
→ Fix 2: Idempotent processing (from Project 27 — use message dedup ID)
→ Fix 3: SQS FIFO with exactly-once processing (from Project 13)

PITFALL 5: "Health check kills healthy workers"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ ASG default health check: EC2 status checks (instance reachable?)
→ EC2 is reachable but worker process crashed → ASG thinks it's healthy
→ Fix: use custom health check via CloudWatch agent
→ Worker publishes heartbeat metric → if missing → mark unhealthy

PITFALL 6: "InsufficientInstanceCapacity — can't launch"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: AWS ran out of requested instance type in your AZ
→ Happens during major events (everyone scaling for Black Friday)
→ Fix: use multiple instance types in Mixed Instances Policy
→ Fix: spread across 3+ AZs
→ Fix: use common types (t3, m5) — exotic types have less capacity

📈 SCALABILITY CONSIDERATIONS

SCALING DIMENSIONS:

1. INSTANCE COUNT (horizontal scaling)
   → Min=1, Max=20 for ETL queue
   → Each t3.medium processes ~100 messages/sec
   → 20 instances = 2,000 messages/sec = 7.2M messages/hour
   → If that's not enough: increase Max (with budget approval)

2. INSTANCE SIZE (vertical scaling)
   → t3.medium (2 vCPU, 4 GB) for lightweight processing
   → c5.xlarge (4 vCPU, 8 GB) for CPU-heavy transforms
   → r5.xlarge (4 vCPU, 32 GB) for memory-heavy processing
   → Mixed Instances Policy: ASG picks cheapest available type

3. PROCESSING EFFICIENCY (code optimization)
   → Batch processing: receive 10 messages at once (from Project 27)
   → Batch delete: delete 10 messages in one API call
   → Connection pooling: reuse DB connections across messages
   → 10x efficiency = 10x fewer instances needed

4. QUEUE ARCHITECTURE (from Project 51)
   → Each filtered queue has its OWN ASG
   → ETL ASG: min=1, max=20 (heavy processing)
   → Order ASG: min=2, max=50 (high volume, light processing)
   → Alert ASG: min=1, max=3 (low volume, fast processing)
   → Independent scaling per workload type

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO MONITOR AND DEBUG AUTO SCALING:

1. ASG Activity History:
   → aws autoscaling describe-scaling-activities --auto-scaling-group-name <name>
   → Shows EVERY scaling decision: launch, terminate, failed launch
   → Includes reason: "User requested", "Alarm triggered", "Health check"

2. CloudWatch Dashboard (build custom):
   → Panel 1: SQS ApproximateNumberOfMessagesVisible (queue depth)
   → Panel 2: ASG GroupInServiceInstances (worker count)
   → Panel 3: SQS NumberOfMessagesSent (incoming rate)
   → Panel 4: SQS NumberOfMessagesDeleted (processing rate)
   → Overlay: scaling events as annotations

3. Scaling Policy Evaluation:
   → CloudWatch Alarm state: OK → ALARM → OK
   → Each transition triggers (or doesn't trigger) a scaling action
   → View: aws cloudwatch describe-alarms --alarm-names <name>

4. Per-Instance Monitoring:
   → CloudWatch agent on each worker → CPU, memory, disk
   → Application logs → CloudWatch Logs → filter for errors
   → Worker heartbeat metric → detect crashed workers

5. Cost Tracking:
   → Tag instances with ASG name + queue name
   → AWS Cost Explorer → filter by tag → cost per queue per day
   → Compare: auto-scaled cost vs. fixed-fleet cost
