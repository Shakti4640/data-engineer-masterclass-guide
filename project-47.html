🟡 HOW TO APPROACH IT

Step 1: Understand DR terminology

  RTO (Recovery Time Objective):
  → How long until service is RESTORED after failure
  → CTO wants: < 15 minutes
  → Meaning: within 15 minutes of us-east-2 failure,
    analysts can query data from the backup region

  RPO (Recovery Point Objective):
  → How much DATA can we afford to LOSE
  → CTO wants: < 1 hour
  → Meaning: if us-east-2 fails at 09:00,
    backup region has data up to at least 08:00

  RTO + RPO together define the DR requirements:
  ┌──────────────────────────────────────────────────────────┐
  │  Timeline:                                                │
  │                                                          │
  │  08:00  08:30  09:00  09:15  09:30                       │
  │    │      │      │      │      │                         │
  │    │      │      │      │      └─ Service restored (RTO) │
  │    │      │      │      └─ Failure detected               │
  │    │      │      └─ us-east-2 fails                      │
  │    │      └─ Last replicated data (RPO = 1 hour)         │
  │    └─ Data available in backup (at minimum)               │
  └──────────────────────────────────────────────────────────┘

SOLUTION: CRR with RTC + Pre-configured DR Region

COMPONENTS:

🔵 WHAT COULD BE DONE

Option A: Manual S3 sync via aws s3 sync (cron job)
→ Scheduled copy from primary to DR bucket
→ Problem: not real-time (RPO = sync interval, e.g., 1 hour)
→ Problem: must manage sync script, handle failures
→ Problem: doesn't sync deletes by default
→ VERDICT: Simple but insufficient for < 1 hour RPO guarantee

Option B: S3 CRR without RTC
→ Automatic async replication for new objects
→ No SLA on replication time (usually minutes, but no guarantee)
→ Problem: during high load, replication can lag hours
→ Problem: no CloudWatch metric for replication lag
→ VERDICT: Works 99% of time, but can't guarantee RPO

Option C: S3 CRR with RTC (Replication Time Control)
→ Same as CRR + 15-minute SLA (99.99% of objects)
→ CloudWatch metrics: ReplicationLatency, OperationsFailedReplication
→ S3 Replication Metrics: track replication status per object
→ Additional cost: $0.015 per GB replicated
→ VERDICT: ✅ Best — guaranteed RPO with monitoring

Option D: Multi-region active-active (write to both regions)
→ Every pipeline writes to BOTH us-east-2 AND us-west-2
→ Zero replication lag (data in both regions simultaneously)
→ Problem: double compute cost (Glue runs in both regions)
→ Problem: double storage cost
→ Problem: consistency issues (which region is source of truth?)
→ VERDICT: Overkill for DR — needed only for active-active serving

DECISION: Option C — S3 CRR with RTC

✅ WHAT SHOULD DO

1. Enable VERSIONING on BOTH source and destination buckets
   → CRR REQUIRES versioning on both buckets
   → Without versioning: CRR configuration fails
   → Primary bucket already has versioning (Project 44)
   → DR bucket must also have versioning enabled

2. Use PREFIX-BASED replication rules (not entire bucket)
   → Replicate: silver/ and gold/ (critical for analytics)
   → Skip: bronze/ (regenerable from source, saves 60%+ cost)
   → Skip: _system/ (operational files, not needed for DR)
   → Each prefix gets its own replication rule

3. Enable Replication Time Control (RTC) for Gold zone
   → Gold is most critical (dashboards depend on it)
   → RTC guarantees 99.99% of objects replicated within 15 minutes
   → Silver can use standard CRR (slightly relaxed SLA)
   → Saves cost: RTC costs more per GB

4. Enable S3 Replication Metrics
   → Track: bytes pending replication, replication latency
   → CloudWatch alarm: if ReplicationLatency > 10 minutes → alert
   → CloudWatch alarm: if OperationsFailed > 0 → alert

5. Run S3 Batch Replication for existing objects
   → CRR only replicates NEW objects (written after CRR enabled)
   → Existing Silver + Gold data: must be batch-replicated
   → One-time operation, can take hours for large datasets

6. Pre-configure Glue Catalog in DR region
   → Create same databases: quickcart_silver, quickcart_gold
   → Tables point to DR bucket: s3://quickcart-datalake-dr/silver/...
   → Run Glue Crawlers in DR region to discover replicated data

7. Pre-configure Athena in DR region
   → Create workgroups matching primary region
   → Test queries BEFORE disaster (monthly DR drill)
   → Document: which Athena endpoint to use during failover

8. Create a RUNBOOK for failover
   → Step 1: Confirm us-east-2 is actually down (not false alarm)
   → Step 2: Announce failover to team (Slack, email)
   → Step 3: Analysts switch to us-west-2 Athena console
   → Step 4: Verify data freshness in DR (check latest partition)
   → Step 5: Monitor DR region for issues
   → Step 6: When primary recovers → fail BACK

❌ WHAT SHOULD NOT DO

1. DO NOT replicate Bronze zone (waste of money)
   → Bronze = raw CSV from sources (62% of total storage)
   → Can be regenerated from MariaDB / source systems
   → Replicating Bronze: ~$35/month additional cost
   → Savings: skip Bronze, keep Silver + Gold only

2. DO NOT enable CRR without versioning
   → CRR fails silently without versioning on BOTH buckets
   → Common mistake: enable versioning on source, forget destination
   → Result: replication rule "active" but nothing replicates

3. DO NOT assume CRR replicates existing objects
   → CRR ONLY replicates objects written AFTER rule creation
   → Existing Silver/Gold data: NOT replicated automatically
   → Must run S3 Batch Replication for backfill
   → Without backfill: DR bucket has only recent data

4. DO NOT use the same bucket name for DR
   → S3 bucket names are globally unique
   → Can't have: quickcart-datalake-prod in both regions
   → Use: quickcart-datalake-dr in us-west-2

5. DO NOT skip monthly DR drills
   → DR configuration that's never tested = false confidence
   → Drill: query DR Athena, verify data, measure RTO
   → Common discovery during drills: Glue Catalog tables out of date
   → Drill frequency: monthly (minimum), quarterly (acceptable)

6. DO NOT auto-failover without human confirmation
   → False positive: CloudWatch alarm fires but S3 is actually fine
   → Auto-failover: analysts suddenly on DR data → confusion
   → Better: alert → human confirms → manual failover
   → Exception: if RTO < 5 minutes required → then auto-failover

7. DO NOT replicate delete markers by default
   → If someone deletes Gold data (accidentally or maliciously)
   → CRR can replicate the DELETE to DR → DR data also gone
   → Configure: replicate object creation ONLY, not deletes
   → DR becomes your "safety net" against accidental deletion

8. DO NOT forget to replicate KMS keys if using SSE-KMS
   → If source bucket uses AWS KMS encryption
   → DR bucket needs access to decrypt/re-encrypt
   → Use separate KMS key in DR region + grant CRR role access
   → For SSE-S3 (AES256): no key management needed ✅

🚫 WHEN NOT TO USE S3 CRR

1. When data is easily regenerable from source
   → Bronze zone: can re-extract from MariaDB
   → Glue temp files: recreated every run
   → Don't pay replication cost for regenerable data

2. When RPO of hours is acceptable
   → Weekly reports: data is already days old
   → Historical archives: rarely accessed
   → Use: AWS Backup (daily snapshots) instead of CRR

3. When data volume makes replication prohibitively expensive
   → 100 TB data lake: CRR + RTC = $1,500/month
   → If DR is "nice to have" not "must have" → skip
   → Or: replicate only Gold zone (smallest, most critical)

4. When both regions share the same failure domain
   → Two buckets in us-east-1 → both fail if us-east-1 fails
   → CRR to same region = SRR → doesn't help for regional DR
   → Always replicate to a DIFFERENT region

5. When data has residency requirements
   → EU data must stay in EU (GDPR)
   → Replicating to us-west-2 violates data residency
   → Use: eu-west-1 → eu-central-1 (within EU)

│  → COMPLETED: successfully replicated to destination                           │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "CRR enabled but nothing replicates"
→ Cause #1: Versioning not enabled on destination bucket
→ Cause #2: IAM role lacks s3:ReplicateObject on destination
→ Cause #3: Objects existed before CRR → not eligible for CRR
→ Debug: check x-amz-replication-status on source objects
→ Fix: enable versioning, fix IAM, run batch replication for existing

PITFALL 2: "Replication works for small files but fails for large ones"
→ Cause: large files use multipart upload
→ CRR handles multipart correctly, but IAM role needs:
  s3:GetObjectVersionForReplication (includes multipart parts)
→ Fix: ensure IAM policy includes all required actions

PITFALL 3: "DR bucket has data but Glue Catalog tables point to old paths"
→ Cause: Glue Catalog in DR region wasn't updated after new partitions
→ Source: silver/orders/year=2025/month=01/day=16/ (new partition)
→ DR bucket: data replicated ✅
→ DR Glue Catalog: doesn't know about day=16 partition ❌
→ Fix: run Glue Crawler in DR region periodically (daily)
→ Or: use partition projection (no crawler needed — Project 26)

PITFALL 4: "Failover works but data is 3 days old"
→ Cause: CRR was failing silently for 3 days
→ No monitoring → no one noticed
→ Fix: CloudWatch alarm on OperationsFailedReplication > 0
→ Fix: CloudWatch alarm on BytesPendingReplication > threshold

PITFALL 5: "Deleted data from primary also deleted from DR"
→ Cause: delete marker replication enabled
→ Someone accidentally deleted Gold data → CRR replicated the delete
→ DR bucket now also empty → no recovery possible
→ Fix: DISABLE delete marker replication
→ DR becomes a safety net against accidental deletion

PITFALL 6: "Failback after primary recovery creates duplicates"
→ Cause: during failover, analysts wrote queries/results to DR region
→ When failing back: primary has old data, DR has newer data
→ Fix: failback procedure must sync DR → primary BEFORE switching
→ Or: re-run ETL pipeline from Bronze to rebuild Silver/Gold

PITFALL 7: "DR Athena query costs double because scanning full table"
→ Cause: DR Glue Catalog not partitioned (Crawler didn't detect partitions)
→ Athena scans ALL data instead of just latest partition
→ Fix: use partition projection in DR Catalog tables (no Crawler dependency)
→ Partition projection works identically regardless of Crawler state

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO MONITOR REPLICATION HEALTH:

1. CloudWatch Replication Metrics (enabled with RTC)
   → ReplicationLatency: avg seconds for replication
     - Normal: < 60 seconds
     - Warning: > 300 seconds (5 minutes)
     - Critical: > 900 seconds (15 minutes — RTC SLA breach)
   → BytesPendingReplication: bytes waiting to replicate
     - Normal: < 10 MB
     - Warning: > 100 MB (backlog building)
     - Critical: > 1 GB (significant lag)
   → OperationsFailedReplication: count of failures
     - Normal: 0
     - Any non-zero: investigate immediately

2. S3 Replication Metrics (per-rule dashboard)
   → Enable in replication configuration
   → Shows: objects replicated, bytes replicated, failed count
   → Available in S3 console under Management → Replication

3. DR Bucket Verification (weekly script)
   → Compare object count: source vs destination per prefix
   → Compare latest object timestamp: source vs destination
   → Flag: if DR is > 1 hour behind → alert

4. Monthly DR Drill Report
   → Drill performed: yes/no
   → RTO measured: X minutes (target: < 15)
   → RPO measured: X minutes (target: < 60)
   → Issues found: list
   → Remediation: actions taken

5. Cost Tracking
   → CloudWatch billing: filter by CRR data transfer
   → Compare: CRR cost vs total S3 cost
   → Expected: CRR < 5% of total S3 cost

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ S3 CRR: automatic async replication to another region
→ CRR REQUIRES versioning on BOTH source and destination buckets
→ CRR only replicates NEW objects — use Batch Replication for existing data
→ Replication Time Control (RTC): 99.99% within 15 minutes (SLA)
→ RTC enables CloudWatch metrics: ReplicationLatency, BytesPending, OperationsFailed
→ Delete markers NOT replicated by default — DR is safety net against accidental deletion
→ Prefix-based rules: replicate Gold (RTC) + Silver (standard), skip Bronze + _system
→ Partition projection in DR eliminates need for Crawlers — zero operational overhead
→ Failover runbook: confirm → announce → verify → test → activate → instruct
→ Failback: verify primary → check replication caught up → re-run ETL → switch back
→ Monthly DR drills: test the failover BEFORE you need it
