🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — 1 year old, data growing fast
Situation:
→ MariaDB (Projects 8-9) handles OLTP perfectly:
   - Order lookups: < 1 ms
   - Insert transactions: < 10 ms
   - 500 concurrent app users: no problem

→ BUT the analytics team is suffering:

ANALYST 1 (Marketing):
"I need total revenue by product category, by month, for the last 
 12 months. MariaDB takes 14 seconds. I run this 20 times a day 
 tweaking filters. That's 5 minutes of my day just WAITING."

ANALYST 2 (Finance):
"I need to join orders with customers, products, and addresses
 to build a 360-degree revenue report. Query scans 50 million rows.
 MariaDB times out after 30 seconds."

ANALYST 3 (Data Science):
"I need to run GROUP BY across 200 million historical rows.
 MariaDB crashes with 'out of memory' on the EC2 instance."

DBA (You):
"These analytics queries are KILLING the production MariaDB.
 The app team reports slowdowns during business hours because
 analysts are running full table scans on the OLTP database."

CTO:
"We need a SEPARATE system for analytics. Something designed
 for aggregations, not transactions. Budget: $200/month max."

CORE PROBLEM:
→ MariaDB is ROW-ORIENTED: reads entire rows even for 2-column query
→ Analytics needs COLUMN-ORIENTED: read only the columns needed
→ MariaDB is SINGLE-NODE: one CPU processes the query
→ Analytics needs MPP: multiple CPUs process in parallel
→ MariaDB is OLTP: optimized for single-row operations
→ Analytics needs OLAP: optimized for bulk aggregations

🟡 HOW TO APPROACH IT

Step 1: Understand WHY MariaDB fails at analytics

  MariaDB (Row Store):
  ┌────────────────────────────────────────────────────┐
  │ Row 1: | order_id | customer_id | amount | status | date     |
  │ Row 2: | order_id | customer_id | amount | status | date     |
  │ Row 3: | order_id | customer_id | amount | status | date     |
  │ ... 50 million rows ...                                       |
  └────────────────────────────────────────────────────┘

  Query: SELECT status, SUM(amount) FROM orders GROUP BY status
  → MariaDB reads EVERY column of EVERY row (even order_id, customer_id, date)
  → Reads 50M × 5 columns = 250M column values
  → Only NEEDS 50M × 2 columns = 100M column values
  → 60% of I/O is WASTED

  Redshift (Column Store):
  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
  │ order_id col │ │ amount col   │ │ status col   │
  │ ORD-001      │ │ 99.99        │ │ completed    │
  │ ORD-002      │ │ 149.50       │ │ pending      │
  │ ORD-003      │ │ 299.99       │ │ shipped      │
  │ ...50M vals  │ │ ...50M vals  │ │ ...50M vals  │
  └──────────────┘ └──────────────┘ └──────────────┘

  Same query: SELECT status, SUM(amount) FROM orders GROUP BY status
  → Redshift reads ONLY status column + amount column
  → Reads 50M × 2 columns = 100M column values
  → ZERO wasted I/O
  → Plus: columns compress 5-10x better (similar values together)
  → Plus: multiple nodes process in parallel (MPP)

Step 2: Choose the right OLAP system
  → Amazon Redshift: columnar, MPP, PostgreSQL-compatible
  → Amazon Athena: serverless SQL on S3 (no cluster to manage)
  → Redshift Serverless: pay-per-query Redshift (no cluster)
  → Google BigQuery: competitor (not AWS)
  → Snowflake: competitor (multi-cloud)

Step 3: Evaluate Redshift vs Athena for QuickCart
  → Athena: $5/TB scanned, no infrastructure
    - Best for: ad-hoc queries, infrequent use
    - Covered in Project 16
  → Redshift Provisioned: fixed cluster, predictable cost
    - Best for: repeated queries, dashboards, complex JOINs
    - Sub-second for cached queries
  → Redshift Serverless: pay-per-RPU-hour
    - Best for: variable workloads, no capacity planning

Step 4: Decision
  → Redshift Provisioned with dc2.large nodes
  → 2 nodes × $0.25/hour = $0.50/hour = ~$365/month
  → OR 1 node = ~$183/month (within $200 budget)
  → Start with 1 node, scale to 2 when needed

✅ WHAT SHOULD DO

1. Place Redshift in PRIVATE subnet
   → No public IP needed for internal analytics
   → Access via VPC peering or bastion host
   → Enhanced VPC routing for S3 traffic

2. Create VPC Endpoint for S3
   → COPY/UNLOAD commands transfer data via S3
   → Without endpoint: traffic goes through internet (slow, costs money)
   → With endpoint: traffic stays within AWS network (fast, free)

3. Use separate IAM Role for Redshift
   → Role allows Redshift to read S3 (for COPY — Project 21)
   → Role allows Redshift to write S3 (for UNLOAD — Project 22)
   → Never embed access keys in COPY commands

4. Enable encryption at rest
   → AWS KMS managed key (default)
   → Encrypts all data blocks on disk
   → Zero performance impact

5. Configure WLM (Workload Management) from start
   → Separate queues for dashboards vs ad-hoc queries
   → Prevent one long query from blocking all others
   → Detailed tuning in Project 40

6. Set maintenance window during off-hours
   → Redshift patches during maintenance window
   → Cluster unavailable for 5-15 minutes
   → Schedule: Sunday 3-4 AM

7. Enable audit logging to S3
   → Who ran what query, when, how long
   → Required for compliance and debugging
   → Minimal storage cost

8. Use ENCODINGS on columns from Day 1
   → Redshift compresses columns automatically (ENCODE AUTO)
   → Or specify: LZO for strings, AZ64 for numbers, ZSTD general
   → 3-10x compression = 3-10x less storage cost and faster scans

9. Choose distribution and sort keys deliberately
   → Distribution key: how data is spread across nodes
   → Sort key: how data is ordered on disk within each node
   → Wrong choices = massive performance problems
   → Detailed in this project + deep optimization in Project 39

❌ WHAT SHOULD NOT DO

1. DO NOT make Redshift cluster publicly accessible
   → Bots scan for open Redshift endpoints constantly
   → Use VPC + Security Group + bastion host
   → If BI tool needs access: VPN or SSH tunnel

2. DO NOT use Redshift for OLTP workloads
   → Redshift is NOT a replacement for MariaDB
   → Single-row INSERT: ~50-100ms in Redshift vs <1ms in MariaDB
   → Single-row SELECT by PK: ~50ms vs <1ms
   → Redshift shines at: scan millions of rows, aggregate, GROUP BY
   → MariaDB shines at: lookup one row by key, transactions

3. DO NOT use single-row INSERTs to load data
   → Redshift HATES row-by-row inserts
   → Each INSERT = commit to multiple nodes = expensive
   → Use COPY from S3 for bulk loads (Project 21)
   → Use multi-row INSERT only for small batches (< 1000 rows)

4. DO NOT skip distribution key selection
   → Default: EVEN distribution (round-robin across nodes)
   → JOINs on evenly distributed tables require data shuffling
   → Shuffling = network I/O between nodes = SLOW
   → Choose distribution key = JOIN column (e.g., customer_id)

5. DO NOT create too many small tables
   → Redshift allocates minimum 1 MB per table per slice
   → 100 small tables × 4 slices = 400 MB overhead
   → Combine small reference tables or use DISTSTYLE ALL

6. DO NOT use VARCHAR(MAX) everywhere
   → Redshift allocates memory based on declared column width
   → VARCHAR(65535) for a 10-char column = massive memory waste
   → Size VARCHAR to actual max length + 20% buffer

7. DO NOT run VACUUM/ANALYZE manually in peak hours
   → VACUUM reclaims space from deleted rows
   → ANALYZE updates statistics for query optimizer
   → Both consume cluster resources — schedule during off-peak
   → Redshift auto-vacuum runs in background (but manual helps)

8. DO NOT expect Redshift to handle 100+ concurrent queries
   → Default: 5 concurrent queries per WLM queue
   → Max practical: 15-50 concurrent queries
   → Beyond that: queries queue and wait
   → For high concurrency: use Redshift Serverless or caching layer

🚫 WHEN NOT TO USE REDSHIFT

1. When you need sub-millisecond single-row lookups
   → Redshift minimum query time: ~50ms (leader node overhead)
   → Use: DynamoDB (<10ms) or MariaDB (<1ms with index)

2. When you need real-time ACID transactions
   → Redshift supports transactions but not designed for high-frequency
   → No row-level locking for concurrent updates
   → Use: MariaDB, PostgreSQL, Aurora

3. When data is < 10 GB and queries are infrequent
   → Redshift cluster costs money even when idle
   → 10 GB in Athena: ~$0.05 per full scan
   → Use: Athena for small, infrequent workloads

4. When you need more than 50 concurrent queries
   → Redshift queues queries beyond WLM limit
   → Use: Redshift Serverless (auto-scales) or caching layer

5. When data changes row-by-row in real-time
   → Redshift optimized for bulk loads, not streaming inserts
   → Use: DynamoDB Streams, Kinesis + micro-batch to Redshift

6. When budget is < $100/month and queries are rare
   → Even smallest Redshift cluster: ~$183/month
   → Use: Athena ($5/TB scanned) or Redshift Serverless

│  4. DISTRIBUTION STYLES — HOW DATA SPREADS ACROSS NODES                     │
│  ═══════════════════════════════════════════════════════                     │
│                                                                             │
│  EVEN Distribution (default):                                               │
│  → Round-robin: row 1 → slice 1, row 2 → slice 2, etc.                     │
│  → Equal data on each slice                                                 │
│  → BUT: JOINs require REDISTRIBUTION (shuffling data between nodes)         │
│  → Use for: tables not frequently joined                                    │
│                                                                             │
│  KEY Distribution:                                                          │
│  → All rows with same key value → same slice                                │
│  → Example: DISTKEY(customer_id)                                            │
│    → All orders for CUST-001 on same slice                                  │
│    → JOIN orders ON customers.customer_id: NO data shuffle needed           │
│  → Use for: large fact tables joined on specific column                     │
│  → CAUTION: if key is skewed (90% of orders from 10 customers)              │
│    → One slice gets 90% of data = SKEW = terrible performance               │
│                                                                             │
│  ALL Distribution:                                                          │
│  → ENTIRE table copied to EVERY node                                        │
│  → JOIN with any table: always local (no shuffle)                           │
│  → Use for: small dimension tables (< 5M rows)                              │
│  → CAUTION: updates must propagate to all nodes = slow writes               │
│                                                                             │
│  AUTO Distribution:                                                         │
│  → Redshift decides: small table → ALL, large table → EVEN                  │
│  → Safe default when unsure                                                 │
│  → May not be optimal for specific JOIN patterns                            │

│  │ Block 2: order_date min=2025-01-16     │                                │
│  │                      max=2025-01-31     │                                │
│  │ Block 3: order_date min=2025-02-01     │                                │
│  │                      max=2025-02-14     │                                │
│  └────────────────────────────────────────┘                                │
│                                                                             │
│  Query: WHERE order_date = '2025-01-20'                                     │
│  → Zone map check: Block 1 (01-01 to 01-15) → SKIP                        │
│  → Zone map check: Block 2 (01-16 to 01-31) → READ                        │
│  → Zone map check: Block 3 (02-01 to 02-14) → SKIP                        │
│  → Result: only 1 out of 3 blocks read = 66% I/O saved                    │
│                                                                             │
│  WITHOUT sort key: all blocks have overlapping date ranges                  │
│  → Zone maps can't eliminate any blocks → read everything                   │
│                                                                             │
│  COMPOUND sort key: SORTKEY(order_date, status)                             │
│  → Sorted by date first, then status within each date                       │
│  → Like composite index in MariaDB (Project 9 leftmost prefix)             │
│                                                                             │
│  INTERLEAVED sort key: SORTKEY(order_date, status) INTERLEAVED             │
│  → Equal weight to both columns                                             │
│  → Works for: WHERE date = X, WHERE status = Y, WHERE both                 │
│  → Slower to load, more expensive VACUUM                                    │
│  → Rarely needed — compound is usually better                              │
│                                                                             │
│  ═══════════════════════════════════════════════════════                     │
│  6. QUERY EXECUTION FLOW                                                    │
│  ═══════════════════════════════════════════════════════                     │
│                                                                             │
│  1. Client sends SQL to Leader Node (port 5439)                             │
│  2. Leader parses SQL, checks syntax                                        │
│  3. Leader optimizes: joins order, distribution strategy, sort usage        │
│  4. Leader creates EXECUTION PLAN (like EXPLAIN in MariaDB)                │
│  5. Leader sends plan SEGMENTS to compute node slices                       │
│  6. Each slice executes its segment IN PARALLEL                             │
│     → Scans its local data blocks                                           │
│     → Applies WHERE filters                                                 │
│     → Computes partial aggregates                                           │
│  7. If JOIN requires data from another slice:                               │
│     → REDISTRIBUTION step: data shuffled between nodes via network          │
│     → This is the EXPENSIVE step — minimize with DISTKEY                    │
│  8. Partial results flow back to Leader                                     │
│  9. Leader merges partial results into final result                         │
│  10. Leader returns result to client                                        │
│                                                                             │
│  RESULT CACHING:                                                            │
│  → If same query runs again and data hasn't changed                         │
│  → Leader returns cached result in < 1 second                               │
│  → No compute node involvement                                             │
│  → Cache invalidated when underlying table modified                         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Query takes 30 seconds even with 2 nodes"
→ Cause: data skew — one slice has 80% of the data
→ Diagnose: SELECT slice, COUNT(*) FROM stv_blocklist 
            WHERE tbl = <table_id> GROUP BY slice;
→ Fix: choose a DISTKEY with high cardinality and even distribution

PITFALL 2: "COPY from S3 fails with 'Access Denied'"
→ Cause: Redshift IAM role doesn't have s3:GetObject permission
→ Fix: attach proper IAM role to cluster
→ Verify: IAM role trust policy allows redshift.amazonaws.com

PITFALL 3: "Disk space full — can't insert more data"
→ Cause: dc2.large has only 160 GB per node
→ Diagnose: SELECT SUM(used) FROM stv_partitions WHERE type = 0;
→ Fix: run VACUUM DELETE to reclaim space from deleted rows
→ Long-term: migrate to ra3 nodes (managed storage)

PITFALL 4: "Query is fast first time, slow second time"
→ Cause: first query benefited from result cache
→ Second query had different parameters = cache miss
→ This is normal — not all queries can be cached

PITFALL 5: "JOINs between two large tables are extremely slow"
→ Cause: both tables distributed EVEN — requires full redistribution
→ Fix: set DISTKEY on JOIN column for the larger table
→ Fix: set DISTSTYLE ALL on the smaller table
→ Detailed optimization in Project 39

PITFALL 6: "VARCHAR columns consuming all memory"
→ Cause: declared VARCHAR(65535) but actual data is 10 chars
→ Redshift allocates intermediate memory based on declared width
→ Fix: ALTER TABLE ALTER COLUMN col TYPE VARCHAR(50);
→ Rule: size VARCHAR to actual max + 20%

PITFALL 7: "Single-node cluster has no leader node"
→ Actually: single-node cluster has SHARED leader
→ Leader functions run on same node as compute
→ Some leader-only functions still work
→ Scaling to 2+ nodes gives dedicated leader

PITFALL 8: "Redshift cluster 'paused' but I'm still charged"
→ Cause: paused cluster doesn't charge compute but charges storage
→ dc2: compute + storage bundled — pausing saves most cost
→ ra3: storage charged separately — still billed for managed storage

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY REDSHIFT CLUSTER IS HEALTHY:

1. Cluster status:
   → AWS Console → Redshift → Clusters → Status: "Available"
   → CLI: aws redshift describe-clusters

2. Query performance:
   → STL_QUERY: every query executed with timing
   → SELECT query, elapsed, starttime 
     FROM stl_query ORDER BY starttime DESC LIMIT 10;

3. Disk usage:
   → SELECT owner, name, SUM(rows) as total_rows,
            SUM(sortkey1_enc) as size_mb
     FROM svv_table_info GROUP BY owner, name;

4. Distribution skew:
   → SELECT "table", skew_rows, skew_sortkey1
     FROM svv_table_info ORDER BY skew_rows DESC;
   → skew_rows > 2.0 = problem

5. Query plan:
   → EXPLAIN SELECT ... 
   → Look for: DS_BCAST (broadcast = redistribution = expensive)
   → Look for: DS_DIST_NONE (no redistribution = good)

6. WLM queue status:
   → SELECT * FROM stv_wlm_service_class_state;
   → Shows queued queries, running queries, slots available

7. CloudWatch metrics:
   → CPUUtilization, PercentageDiskSpaceUsed
   → ReadIOPS, WriteIOPS
   → QueriesCompletedPerSecond
   → Set alarms at 80% disk usage (covered in Project 54)
