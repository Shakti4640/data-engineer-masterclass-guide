🟡 HOW TO APPROACH IT

Step 1: Understand WHAT EC2 actually is
→ EC2 = Elastic Compute Cloud
→ Virtual machine (VM) running on AWS physical hardware
→ You choose: OS, CPU, RAM, disk, network
→ You pay: per second of runtime (on-demand) or per hour
→ You control: full root access, install anything, run anything

Step 2: Choose the right INSTANCE TYPE
→ Instance type = combination of CPU, memory, storage, network
→ Format: {family}{generation}.{size}
→ Example: t3.micro = T family, 3rd generation, micro size
→ QuickCart needs: light ETL, Python scripts, small data
→ Best fit: t3.micro (2 vCPU, 1 GB RAM, $0.0104/hour = ~$7.50/month)
→ Even better: t3.small (2 vCPU, 2 GB RAM, $0.0208/hour = ~$15/month)

Step 3: Understand the NETWORKING prerequisites
→ VPC (Virtual Private Cloud): isolated network in AWS
→ Subnet: segment of VPC in a specific Availability Zone
→ Security Group: virtual firewall controlling inbound/outbound traffic
→ Key Pair: SSH authentication (no passwords)
→ Elastic IP (optional): static public IP that persists across stop/start

Step 4: Plan the setup
→ Use default VPC (AWS creates one per region automatically)
→ Create security group allowing SSH (port 22) from team's IPs only
→ Create key pair for SSH authentication
→ Launch t3.small in us-east-2 (same region as S3 bucket — Project 1)
→ Install Python, pip, boto3, and our scripts

Option E: AWS Fargate (serverless containers)

✅ WHAT SHOULD DO

1. Choose AMAZON LINUX 2023 or UBUNTU 22.04 LTS
   → Both are free tier eligible
   → Ubuntu: more familiar to most developers
   → Amazon Linux: tighter AWS integration, SSM agent pre-installed
   → We'll use Ubuntu 22.04 LTS (Long Term Support = 5 years updates)

2. Use t3.small (not t3.micro) for data work
   → t3.micro: 1 GB RAM → Python + pandas kills it on 50MB CSVs
   → t3.small: 2 GB RAM → comfortable for medium ETL
   → t3.medium: 4 GB RAM → if processing grows (upgrade later)

3. Use gp3 EBS volume (not gp2)
   → gp3: 3000 IOPS baseline (free), $0.08/GB/month
   → gp2: 100 IOPS per GB (pay more for IOPS), $0.10/GB/month
   → gp3 is NEWER, CHEAPER, and FASTER baseline
   → 20 GB gp3 = $1.60/month

4. Create DEDICATED security group (not use default)
   → Default SG allows all inbound from same SG → too permissive
   → Create one that ONLY allows SSH from known IPs
   → Add team members' office/VPN IP ranges

5. Use ED25519 key pair (not RSA)
   → ED25519: shorter keys, faster, more secure
   → RSA 2048: legacy, still works but longer keys
   → NEVER share private key via email/Slack/Git

6. Tag EVERYTHING
   → Name, Environment, Team, CostCenter
   → Tags flow to billing → know exactly what costs what
   → Tags enable resource group filtering in console

7. Use USER DATA script to automate initial setup
   → User data: shell script that runs on FIRST boot
   → Install Python, pip, boto3, create directories
   → Eliminates manual SSH-and-install steps
   → Makes instance reproducible (delete and recreate anytime)

8. Consider STOPPING instance when not in use
   → Running 24/7: t3.small = $15/month
   → Running 12 hours/day: t3.small = $7.50/month
   → Stopped: pay only for EBS = $1.60/month
   → For nightly-only scripts: start at 11 PM, stop at 2 AM

❌ WHAT SHOULD NOT DO

1. DO NOT open port 22 (SSH) to 0.0.0.0/0 (the entire internet)
   → Bots scan ALL AWS IPs continuously for open SSH ports
   → Within minutes: brute force login attempts
   → Restrict to: your office IP, VPN IP, or specific CIDR range
   → Better: use AWS Systems Manager Session Manager (no SSH port needed)

2. DO NOT use the default security group
   → Default SG allows all inbound from instances in same SG
   → If another team launches an instance in same VPC with default SG
   → They can access YOUR instance
   → Always create purpose-specific security groups

3. DO NOT use password authentication for SSH
   → Key pair authentication ONLY
   → Disable password auth in /etc/ssh/sshd_config
   → Password brute force is trivial; key brute force is impossible

4. DO NOT store the private key in Git, S3, or shared drives
   → Private key = root access to your server
   → Store in: ~/.ssh/ with chmod 400 permissions
   → If compromised: delete key pair, create new one, update authorized_keys

5. DO NOT launch in us-east-1 "because it's default"
   → Must be SAME REGION as S3 bucket (us-east-2 from Project 1)
   → Cross-region data transfer: $0.02/GB
   → Same-region EC2↔S3: $0.00/GB (free)

6. DO NOT use root user for daily operations
   → Create a non-root user (e.g., "dataeng") on the instance
   → Use sudo when needed
   → Limits blast radius of mistakes

7. DO NOT hardcode AWS credentials on the instance
   → No ~/.aws/credentials with access keys
   → Use IAM Role attached to EC2 (Project 6 — NEXT)
   → For now: understand that credentials are a TEMPORARY gap

8. DO NOT forget to set up automatic security updates
   → Ubuntu: unattended-upgrades package
   → Patches kernel and system packages automatically
   → Prevents known vulnerabilities from being exploited

🚫 WHEN NOT TO USE EC2

1. DO NOT use EC2 for sub-second event processing
   → EC2 takes 30-90 seconds to boot
   → Lambda: cold start 100-500ms, warm start < 10ms
   → For: "process file the instant it lands in S3" → Lambda

2. DO NOT use EC2 for tasks under 15 minutes with no state
   → Lambda: pay per 100ms of execution, zero cost when idle
   → EC2: pay per second even when waiting for next job
   → For: "run this function when SNS message arrives" → Lambda

3. DO NOT use EC2 for containerized microservices
   → ECS/Fargate: container orchestration, auto-scaling, no OS management
   → EC2: you manage the OS, patching, container runtime
   → For: "run 20 Docker containers with auto-scaling" → ECS/Fargate

4. DO NOT use EC2 for managed database hosting
   → EC2 + self-managed MySQL: you handle backups, patches, HA
   → RDS: managed backups, patching, Multi-AZ failover
   → Exception: MariaDB on EC2 for learning (Project 8) is fine

5. DO NOT use EC2 for serverless ETL
   → Glue: managed Spark, auto-scaling, no server management
   → EC2: you manage everything
   → For: "transform 100GB of Parquet files" → Glue (Project 25)

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Cannot SSH — connection timed out"
→ Cause 1: Security group doesn't allow inbound port 22
→ Cause 2: Your IP changed (dynamic home IP) — SG rule outdated
→ Cause 3: Instance in private subnet with no internet gateway
→ Cause 4: NACL blocking traffic (rare with default VPC)
→ Debug: check SG rules → check your current IP → check subnet route table

PITFALL 2: "Permission denied (publickey) on SSH"
→ Cause 1: Wrong username (use "ubuntu" for Ubuntu, "ec2-user" for Amazon Linux)
→ Cause 2: Wrong private key file
→ Cause 3: Private key permissions too open (must be chmod 400)
→ Cause 4: Key pair doesn't match the instance
→ Fix: ssh -i ~/.ssh/correct-key.pem -v ubuntu@<ip> (-v for verbose debug)

PITFALL 3: "Instance running but status check 1/2 failed"
→ System status check: AWS hardware/hypervisor problem → stop/start instance
→ Instance status check: OS-level issue → check logs, reboot
→ Stop/start moves instance to different physical host (fixes hardware issues)

PITFALL 4: "My public IP changed after stop/start"
→ Cause: non-Elastic public IPs are released when instance stops
→ On start: new public IP assigned
→ Fix: allocate Elastic IP ($0.005/hour when NOT attached — free when attached)
→ Or: use DNS name that auto-updates (Route 53)

PITFALL 5: "Instance runs out of disk space"
→ Default root volume: 8 GB (Ubuntu AMI)
→ Python packages + data files fill it fast
→ Fix: specify larger root volume at launch (20-50 GB)
→ Or: attach additional EBS volume for data

PITFALL 6: "User data script didn't run"
→ Cause 1: missing shebang (#!/bin/bash) at top
→ Cause 2: script has Windows line endings (\r\n instead of \n)
→ Cause 3: script exceeds 16 KB limit
→ Debug: SSH in → check /var/log/cloud-init-output.log

PITFALL 7: "EC2 instance can't reach S3"
→ Cause 1: outbound security group too restrictive (blocks HTTPS 443)
→ Cause 2: no internet gateway in VPC (default VPC has one)
→ Cause 3: route table doesn't have 0.0.0.0/0 → IGW route
→ Default VPC: all of these are pre-configured (should just work)

📈 SCALABILITY CONSIDERATIONS

VERTICAL SCALING (scale UP):
→ Stop instance → change instance type → start
→ t3.small → t3.medium → t3.large → m5.xlarge
→ Takes ~2 minutes (stop + type change + start)
→ Same EBS volume, same private IP, same data
→ Public IP changes unless Elastic IP is attached

HORIZONTAL SCALING (scale OUT):
→ Launch multiple instances behind a load balancer
→ Auto Scaling Group: automatically add/remove instances
→ Covered in Project 52 (SQS-based scaling)
→ QuickCart won't need this for a while

DISK SCALING:
→ EBS volumes can be EXPANDED live (no downtime)
→ 20 GB → 100 GB: modify volume, extend filesystem
→ Cannot SHRINK an EBS volume
→ For very large data: use S3, not local EBS

NETWORK SCALING:
→ t3.small: "Up to 5 Gbps" network
→ Sufficient for S3 transfers of files < 1 GB
→ For massive transfers (TB): use S3 Transfer Acceleration or multi-part

COST SCALING:
→ 1 instance (t3.small): $15/month
→ 5 instances (t3.small): $75/month
→ At this point: consider ECS/Fargate for orchestration
→ At 20+ instances: definitely need container orchestration

AMI FOR REPRODUCIBILITY:
→ Once instance is configured perfectly → create AMI (machine image)
→ AMI captures: OS + packages + config + scripts
→ Launch new instances from AMI → identical clones in seconds
→ Essential for auto scaling groups

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO MONITOR EC2 HEALTH:

1. EC2 Console:
   → Instance state: running/stopped/terminated
   → Status checks: 2/2 = healthy
   → Monitoring tab: CPU, network, disk metrics

2. CloudWatch Metrics (free, 5-minute intervals):
   → CPUUtilization: should be < 80% sustained
   → NetworkIn/NetworkOut: data transfer volume
   → StatusCheckFailed: 0 = healthy
   → DiskReadOps/DiskWriteOps: I/O activity

3. CloudWatch Alarms:
   → CPU > 90% for 5 minutes → SNS notification (Project 12)
   → StatusCheckFailed > 0 → auto-recover instance
   → Billing > threshold → alert before bill surprise

4. System Logs (from inside instance):
   → /var/log/syslog: system events
   → /var/log/cloud-init-output.log: user data script output
   → /var/log/auth.log: SSH login attempts (security audit)
   → journalctl: systemd service logs

5. VPC Flow Logs:
   → Network traffic logs for the instance's ENI
   → Shows: source IP, dest IP, port, action (ACCEPT/REJECT)
   → Essential for debugging connectivity issues
   → Delivered to CloudWatch Logs or S3
