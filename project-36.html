🔵 WHAT COULD BE DONE

Option A: Athena CTAS (CREATE TABLE AS SELECT)
→ Creates NEW table from query results
→ Output: Parquet/ORC in S3, registered in Glue Catalog
→ Pros: single SQL statement, automatic catalog registration
→ Cons: creates NEW table each time — must DROP old one first
→ Best for: daily full-refresh materialized tables

Option B: Athena INSERT INTO
→ Appends query results to EXISTING table
→ Pros: incremental — add today's data without replacing yesterday
→ Cons: requires table to exist already, no dedup
→ Best for: event logs, append-only data

Option C: Athena Views (no materialization)
→ Saved SQL — no data copied
→ Pros: always fresh, zero storage cost
→ Cons: still scans full data on every query — no performance gain
→ Best for: convenience, not performance
→ Covered in Project 37

Option D: Glue ETL (already doing this)
→ Project 34 already creates daily_revenue_summary in silver/
→ But: not all analyst queries map to existing silver tables
→ Adding every ad-hoc query to Glue job is overkill
→ Best for: complex transforms, not simple aggregations

Option E: Redshift Materialized Views (Project 63)
→ Auto-refreshed views inside Redshift
→ Best when data already in Redshift
→ Different use case — covered later

DECISION: Options A + B + C combined
→ CTAS for daily-refresh materialized tables (this project)
→ INSERT INTO for append-only historical tracking
→ Views for convenience aliases (Project 37)
→ Each serves different need

❌ WHAT SHOULD NOT DO

1. DO NOT use CTAS without DROP first
   → CTAS fails if table already exists (no "IF NOT EXISTS" for CTAS)
   → Error: "Table already exists"
   → Always DROP TABLE IF EXISTS before CTAS

2. DO NOT use CTAS output location that overlaps with source
   → Source: s3://bucket/silver/order_details/
   → CTAS output: s3://bucket/silver/order_details_summary/
   → If CTAS writes to silver/ prefix → Crawler picks it up → confusion
   → Use separate prefix: s3://bucket/gold/

3. DO NOT materialize tables that change intra-day
   → CTAS creates point-in-time snapshot
   → If silver data updates at 10 AM and 2 PM → gold is stale after 10 AM
   → Only materialize data that refreshes once/day (our nightly pipeline)

4. DO NOT create too many materialized tables
   → Each CTAS costs one full scan of source data
   → 20 materialized tables × $0.075 each = $1.50/day just for CTAS
   → Only materialize: high-frequency + high-scan queries
   → Rule of thumb: materialize if query runs > 5 times/day

5. DO NOT forget to clean up old S3 data after DROP TABLE
   → DROP TABLE removes Catalog entry but NOT S3 data
   → For EXTERNAL tables: S3 files remain → orphaned data
   → For CTAS-created tables: S3 files ARE deleted (Athena manages them)
   → Verify: CTAS tables have 'EXTERNAL' = 'FALSE' in properties

6. DO NOT skip ANALYZE on CTAS output
   → Athena uses table statistics for query optimization
   → CTAS tables may lack accurate statistics
   → Run: ANALYZE gold.revenue_summary after CTAS

7. DO NOT use CTAS for tables > 100 GB output
   → CTAS has a 100-partition limit per statement
   → For very large outputs: use Glue ETL instead
   → CTAS is designed for summary/aggregate tables, not full copies

🚫 WHEN NOT TO USE ATHENA CTAS

1. Source data changes multiple times per day
   → CTAS is a snapshot — stale within hours
   → Use: Redshift Materialized Views (auto-refresh)
   → Or: Athena Views (always fresh, but slower)

2. Output table needs > 100 partitions
   → CTAS hard limit: 100 partitions per statement
   → Workaround: multiple CTAS with WHERE filters
   → Better: Glue ETL for large partitioned outputs

3. Need incremental append (not full replace)
   → CTAS creates new table — replaces entirely
   → Use: INSERT INTO for appending
   → Or: Glue with append mode

4. Complex transform logic (cleaning, dedup, UDFs)
   → Athena SQL is powerful but limited vs Spark
   → No UDFs, no procedural logic, no variables
   → Use: Glue ETL for complex transforms

5. Output table must be ACID-compliant
   → CTAS creates standard Parquet — no transactions
   → Use: Athena ACID tables (Apache Iceberg) for transactional needs
   → Or: Lake Formation Governed Tables

6. Query scans < 1 GB already
   → Materializing saves $0.005 per query (negligible)
   → Storage cost of materialized table may exceed query savings
   → Only materialize when scan > 5 GB AND frequency > 5/day

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "CTAS fails: Table already exists"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: CTAS does not support IF NOT EXISTS
→ If table exists from yesterday's run → error
→ Fix: Always DROP TABLE IF EXISTS before CTAS
→ Note: Must also delete S3 data if using external_location

PITFALL 2: "CTAS succeeds but table has 0 rows"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Source table partitions not registered in Catalog
→ Athena reads Catalog → no partitions → no data
→ Fix: Run MSCK REPAIR TABLE on source before CTAS
→ Or: Ensure Crawler ran (Project 35 workflow includes this)

PITFALL 3: "Partitioned CTAS fails: too many partitions"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: CTAS limit is 100 partitions per statement
→ Partitioning by day for 365 days → 365 partitions → fails
→ Fix: Partition by year or month instead of day
→ Or: Use INSERT INTO with date filter for incremental

PITFALL 4: "CTAS output has 500 tiny files"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Athena parallelism creates one file per worker
→ For small result sets: many empty or tiny files
→ Fix: Use bucketed_by to control file count
→ Or: Accept it — for small tables, file count doesn't matter

PITFALL 5: "Gold table shows stale data"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: CTAS didn't re-run after silver refresh
→ Silver updated at 2:47 AM → CTAS ran at 2:00 AM (before silver)
→ Fix: Schedule CTAS AFTER silver layer completes
→ Add to workflow (Project 35) or schedule separately

PITFALL 6: "DROP TABLE deletes my S3 data"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: CTAS without external_location creates MANAGED table
→ Managed table: DROP deletes both Catalog entry AND S3 data
→ This is EXPECTED for CTAS-managed tables
→ For external_location CTAS: DROP only removes Catalog entry
→ Know the difference — check table type before DROP

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY MATERIALIZED TABLES:

1. ROW COUNT MATCH
   → CTAS result count should match direct query count
   → SELECT COUNT(*) FROM gold.revenue_summary
   → vs: SELECT COUNT(DISTINCT category || month || year)
         FROM silver_quickcart.order_details

2. VALUE MATCH
   → Pick 3 random (category, month) combinations
   → Query gold table → get total_revenue
   → Query silver table → SUM(revenue) WHERE category=X AND month=Y
   → Must match exactly

3. FRESHNESS CHECK
   → Query gold table metadata:
     SELECT "$path" FROM gold.revenue_summary LIMIT 1
   → Check S3 file timestamp matches today's CTAS run

4. PARTITION CHECK
   → SHOW PARTITIONS gold.revenue_summary
   → Should show expected partitions (year=2024, year=2025)

5. COST VERIFICATION
   → Check Athena query history for CTAS query
   → Verify: data scanned matches expected source size
   → Check subsequent analyst queries: data scanned should be tiny
