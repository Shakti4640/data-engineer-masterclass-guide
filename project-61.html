Option A: VPC Peering + Glue JDBC Direct ✅ (CHOSEN)
→ Glue ENI sits in Account B's VPC
→ Peering routes traffic to Account A's MariaDB
→ Single Glue job: read MariaDB → transform → write Redshift
→ Cleanest architecture, fewest moving parts

Option B: Dump MariaDB → S3 (Account A) → Cross-Account S3 Read → Glue (Account B)
→ Avoids VPC Peering entirely
→ But: extra step (dump script), extra storage, extra latency
→ Lose real-time column selection (must dump entire table)
→ Already covered in Projects 20 + 56 — combine those

Option C: AWS DMS (Database Migration Service)
→ Purpose-built for database replication
→ But: DMS replication instance STILL needs VPC Peering to reach source
→ Adds another managed service + cost ($0.018/hr for dms.t3.medium)
→ Better for continuous CDC — overkill for nightly batch

Option D: Transit Gateway
→ Designed for hub-and-spoke with 10+ VPCs
→ $0.05/hr + $0.02/GB — expensive for 2-VPC setup

✅ WHAT SHOULD DO

1. Use NON-OVERLAPPING CIDRs
   → Account A: 10.1.0.0/16
   → Account B: 10.2.0.0/16
   → VPC Peering FAILS if CIDRs overlap — this is NON-NEGOTIABLE

2. Place Glue ENIs in a DEDICATED subnet
   → Create subnet 10.2.6.0/24 specifically for Glue
   → Attach dedicated security group sg-glue-eni
   → Makes security rules surgical and auditable

3. Security groups: reference by CIDR, not by SG ID
   → Cross-account SG referencing works ONLY within same peering
   → Safer to use CIDR blocks explicitly
   → sg-mariadb-ops: allow 10.2.6.0/24 on 3306

4. Enable DNS resolution on peering connection
   → Both sides must enable:
     - "Allow DNS resolution from Account B" (requester side)
     - "Allow DNS resolution from Account A" (accepter side)
   → Without this: Glue can reach IP but not hostname

5. Create MariaDB user specifically for Glue
   → User: glue_reader@'10.2.%'
   → Grants: SELECT only on needed tables
   → No INSERT/UPDATE/DELETE — principle of least privilege

6. Store MariaDB credentials in Account B's Secrets Manager
   → Glue Connection references secret ARN
   → Rotation possible without changing Glue job (Project 70)

7. Enable VPC Flow Logs on BOTH VPCs
   → Proves to security team: traffic flows only on expected ports
   → Helps debug "connection timeout" issues

8. Test connectivity BEFORE building Glue job
   → Launch EC2 in Account B's Glue subnet
   → Attempt: mysql -h 10.1.3.50 -P 3306 -u glue_reader -p
   → If this works → Glue will work
   → If this fails → fix network first, don't blame Glue

❌ WHAT SHOULD NOT DO

1. DO NOT make MariaDB publicly accessible
   → "Just give it an Elastic IP and whitelist Account B's NAT Gateway"
   → This exposes DB to entire internet — one SG misconfiguration = breach
   → VPC Peering keeps traffic 100% private

2. DO NOT use overlapping CIDRs
   → If both VPCs use 10.0.0.0/16 → peering creation fails
   → If both use 172.31.0.0/16 (default VPC) → same problem
   → ALWAYS use custom VPCs with planned CIDR allocation

3. DO NOT forget route tables on BOTH sides
   → Peering connection alone does NOTHING
   → You must add routes in BOTH VPCs pointing to the peering connection
   → Missing route on Account A side = packets go to Account A but can't return

4. DO NOT open all ports in security groups
   → "Allow all traffic from 10.2.0.0/16" — too broad
   → Only allow port 3306 from 10.2.6.0/24 (Glue subnet only)

5. DO NOT use MariaDB root user for Glue
   → Root has DROP DATABASE, GRANT, SHUTDOWN privileges
   → One bug in Glue script = catastrophe
   → Create dedicated read-only user

6. DO NOT skip testing with EC2 before configuring Glue
   → Glue connection errors are vague: "Connection timed out"
   → Could mean: wrong route, wrong SG, wrong subnet, wrong password
   → EC2 test isolates NETWORK from APPLICATION issues

7. DO NOT peer VPCs across regions for this use case
   → Inter-region peering: $0.01/GB data transfer
   → Latency: 50-100ms vs <1ms same-region
   → Both VPCs must be in SAME region (us-east-2)

8. DO NOT create peering from wrong account direction then wonder why
   → Account A REQUESTS, Account B ACCEPTS (or vice versa)
   → Requester and accepter must BOTH modify route tables
   → Peering stays "pending-acceptance" until accepter approves

🚫 WHEN NOT TO USE THIS PATTERN

1. When source DB is in a DIFFERENT region
   → Inter-region peering adds cost + latency
   → Better: dump to S3 in source region → S3 CRR → process in target region

2. When you need REAL-TIME replication (< 1 minute lag)
   → VPC Peering + Glue = batch (minimum ~5 min Glue startup)
   → Use: DMS with CDC mode over peered VPCs
   → Or: Kinesis-based streaming (Project 72)

3. When 10+ accounts need the same source data
   → VPC Peering = mesh topology = N*(N-1)/2 connections
   → 10 accounts = 45 peering connections — unmanageable
   → Use: Transit Gateway or dump-to-S3 + cross-account read

4. When source DB is RDS (not EC2-hosted)
   → RDS supports cross-account access via:
     - RDS Proxy with IAM auth
     - Snapshot sharing + restore
   → These may be simpler than VPC Peering for RDS specifically

5. When CIDR ranges overlap and cannot be changed
   → VPC Peering is IMPOSSIBLE with overlapping CIDRs
   → Use: PrivateLink (doesn't require non-overlapping CIDRs)
   → Or: dump-to-S3 approach

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Peering is Active but Glue still times out"
→ Cause: Route table not updated
→ Peering connection alone does NOTHING — it's just a "virtual cable"
→ You MUST add routes in BOTH VPCs:
   Account A RT: 10.2.0.0/16 → pcx-xxx
   Account B RT: 10.1.0.0/16 → pcx-xxx
→ Verify: check BOTH route tables in console

PITFALL 2: "Route added but still can't connect"
→ Cause: Security group doesn't allow cross-VPC CIDR
→ SG is evaluated AFTER routing
→ MariaDB SG must explicitly allow 10.2.6.0/24 on port 3306
→ Common mistake: SG only allows "sg-mariadb-ops" (self-reference)
   which doesn't match traffic from Account B

PITFALL 3: "Glue job fails: Could not find S3 endpoint"
→ Cause: Glue ENI in private subnet has no internet access
→ Glue needs S3 access for: scripts, temp dir, output
→ Fix: Create S3 VPC Gateway Endpoint in Account B's VPC
→ Attach to route table of Glue subnet
→ This was covered conceptually in Project 31

PITFALL 4: "Connection refused on port 3306"
→ Cause: MariaDB bind-address is 127.0.0.1 (localhost only)
→ Fix: Edit /etc/my.cnf → bind-address = 0.0.0.0
→ Restart MariaDB: sudo systemctl restart mariadb
→ Then: MariaDB listens on all interfaces including 10.1.3.50

PITFALL 5: "Access denied for user 'glue_reader'@'10.2.6.10'"
→ Cause: MariaDB user created with wrong host pattern
→ CREATE USER 'glue_reader'@'localhost' → only allows local connections
→ Fix: CREATE USER 'glue_reader'@'10.2.%' → allows entire Account B range
→ Or: CREATE USER 'glue_reader'@'%' → allows any host (less secure)

PITFALL 6: "Peering request stuck in pending-acceptance for hours"
→ Cause: Accepter account hasn't approved yet
→ Peering auto-expires after 7 days if not accepted
→ Both accounts must coordinate — this is a manual approval step

PITFALL 7: "Glue job reads 10M rows from MariaDB — MariaDB CPU hits 100%"
→ Cause: Glue opens multiple parallel JDBC connections
→ Each connection runs a query → MariaDB overwhelmed
→ Fix: limit Glue DPUs, use hashfield/hashexpression carefully
→ Or: add read replica in Account A, point Glue to replica

PITFALL 8: "VPC Peering created between us-east-2 and eu-west-1"
→ Works — inter-region peering IS supported
→ But: $0.01/GB data transfer + 50-100ms latency
→ For nightly batch: acceptable
→ For real-time: unacceptable
→ Best: keep both VPCs in same region

PITFALL 9: "Overlapping CIDR — peering creation fails"
→ ERROR: "VPC CIDR block conflicts with peering connection"
→ Cannot be fixed without re-creating one VPC
→ THIS is why CIDR planning matters on Day 1
→ Enterprise pattern: central team allocates CIDR blocks per account

PITFALL 10: "Glue connection test succeeds but job fails"
→ Cause: Glue connection test uses a single ENI
→ Job may use multiple ENIs in different AZs
→ Ensure ALL subnets in Glue job config have correct route table entries
→ Best practice: use single subnet for Glue connections

📈 SCALABILITY CONSIDERATIONS

DATA VOLUME SCALING:
→ 1 GB/night from MariaDB: trivial, single Glue job, 2 DPUs, ~5 min
→ 10 GB/night: moderate, increase DPUs to 5-10, use partitioned reads
→ 100 GB/night: heavy, must use parallel JDBC reads (hashfield)
→ 1 TB/night: MariaDB cannot sustain this — need read replica or dump-to-S3

MARIADB PROTECTION AT SCALE:
→ Add read replica specifically for analytics extraction
→ Point Glue JDBC to replica, not primary
→ Primary handles OLTP traffic undisturbed
→ Replica lag: typically < 1 second for MariaDB async replication

VPC PEERING SCALING:
→ No bandwidth limit on peering connection
→ No hourly charge
→ Data transfer: $0.01/GB same-region
→ 100 GB/night = $1/night = $30/month — very reasonable

GLUE JOB SCALING:
→ Max DPUs: 100 (Standard) or auto-scaling with Glue 2.0+
→ Each DPU: 4 vCPU, 16 GB RAM
→ 10 DPUs reading MariaDB in parallel = 10 concurrent connections
→ MariaDB max_connections default: 151
→ Ensure Glue connections < 50% of MariaDB max_connections

MULTI-TABLE SCALING:
→ 3 tables: single Glue job with sequential reads
→ 20 tables: one Glue job per table, run in parallel via Workflow
→ 100 tables: dynamic job generation using Glue Workflow + parameterized jobs

ACCOUNT SCALING:
→ 2 accounts: single VPC peering — current pattern
→ 5 accounts needing same MariaDB: 4 peering connections — manageable
→ 20 accounts: Transit Gateway — hub-and-spoke
→ 50 accounts: dump to S3 + cross-account reads (Project 56 pattern)

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY EACH LAYER WORKS:

LAYER 1 — VPC Peering Active:
→ Console: VPC → Peering Connections → Status = "Active"
→ CLI: aws ec2 describe-vpc-peering-connections --filters Name=status-code,Values=active

LAYER 2 — Routes Exist:
→ Account A: aws ec2 describe-route-tables → find 10.2.0.0/16 → pcx-xxx
→ Account B: aws ec2 describe-route-tables → find 10.1.0.0/16 → pcx-xxx

LAYER 3 — Security Groups Allow Traffic:
→ Account A SG: inbound rule for 10.2.6.0/24 on TCP 3306
→ Account B SG (Glue): outbound rule for 10.1.3.50/32 on TCP 3306
  (default: outbound allows all — but verify if restricted)

LAYER 4 — MariaDB Accepts Remote Connections:
→ From EC2 in Account B Glue subnet:
  mysql -h 10.1.3.50 -P 3306 -u glue_reader -p
→ If works: network is fine, DB is fine
→ If "Connection timed out": network issue (route/SG)
→ If "Access denied": DB user/host issue

LAYER 5 — Glue Connection Test:
→ Console: Glue → Connections → Test Connection
→ Glue creates temporary ENI, attempts JDBC handshake
→ Success = green check, Failure = error message (usually vague)

LAYER 6 — Glue Job Execution:
→ CloudWatch Logs: /aws-glue/jobs/output and /aws-glue/jobs/error
→ Check: rows read from source, rows written to target
→ Glue job metrics: Data Movement, Job Duration, Memory Usage

LAYER 7 — Data Verification in Redshift:
→ SELECT COUNT(*) FROM schema.orders;

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ VPC Peering creates a PRIVATE network bridge between two VPCs in different accounts
→ Peering alone does NOTHING — requires routes + security groups on BOTH sides
→ CIDRs MUST NOT overlap — this is the #1 non-negotiable requirement
→ Glue ENIs live in YOUR VPC — they follow YOUR route tables and security groups
→ Cross-account Glue JDBC works because the NETWORK carries the traffic, not IAM
→ IAM handles permissions (what can this role do)
→ VPC Peering handles connectivity (can packets reach the destination)
→ ALWAYS test with EC2 before blaming Glue — isolate network vs application issues
→ Security: dedicated MariaDB user (SELECT only), dedicated Glue subnet, tight SG rules
→ VPC Peering is free (no hourly charge) — only standard data transfer applies
