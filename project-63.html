🟡 HOW TO APPROACH IT

Step 1: Understand the WASTE

  COMPUTATION ANALYSIS (24-hour period):
  → Data changes: 1 time (2 AM ETL)
  → Dashboard refreshes: 288 times (every 5 min × 24 hours)
  → Each refresh: 12 queries = 3,456 query executions/day
  → Unique results needed: 12 (one per query, refreshed once after ETL)
  → Redundant computations: 3,456 - 12 = 3,444 (99.65% WASTE)

Step 2: Identify the PATTERN

  → Queries are COMPLEX: multi-table JOINs, GROUP BY, window functions
  → Queries are STABLE: same SQL text every refresh
  → Data is BATCH: changes once per day
  → Results are SHARED: all 15 analysts see same dashboard
  → This is the TEXTBOOK use case for materialized views

Step 3: Evaluate solutions

Step 4: Why Materialized Views WIN

  → Pre-computed: query result stored physically in Redshift
  → Auto-refresh: Redshift can detect base table changes and update MV
  → Query rewrite: Redshift optimizer can automatically USE MV
    even when user writes query against base tables
  → No application changes: dashboard queries stay the same
  → Millisecond response: reading pre-computed result, not re-computing
  → Frees cluster: 3,444 redundant computations eliminated

🔵 WHAT COULD BE DONE

Option A: Manual REFRESH after ETL ✅ (CHOSEN for control)
→ Step Functions calls REFRESH after ETL completes
→ Explicit control over timing
→ Can validate data before exposing to dashboards
→ Best for: nightly batch pipelines

Option B: AUTO REFRESH ✅ (supplement for safety)
→ Redshift detects base table changes → auto-refreshes MV
→ No orchestration needed
→ But: refresh timing unpredictable (could be during peak)
→ Best for: tables that change infrequently

Option C: Incremental REFRESH ✅ (for large MVs)
→ Only processes CHANGED rows since last refresh
→ Much faster than full refresh for large datasets
→ Requires MV to be defined with specific constraints
→ Best for: MVs on large tables with small daily deltas

Option D: Query Rewrite (transparent acceleration)
→ Redshift optimizer detects that a base-table query
  can be answered from an existing MV
→ Rewrites query plan to scan MV instead
→ NO dashboard code changes needed
→ Best for: when you can't modify dashboard SQL

DECISION: 
→ Primary: Manual REFRESH triggered by Step Functions after ETL
→ Enable AUTO REFRESH as safety net
→ Use query rewrite for zero-change dashboard integration
→ Incremental refresh for the largest MV (mv_cumulative_revenue)

✅ WHAT SHOULD DO

1. CREATE MVs with the EXACT query the dashboard runs
   → MV definition = dashboard SQL (minus LIMIT if present)
   → Ensures query rewrite can match and redirect
   → If MV query is slightly different → rewrite won't trigger

2. REFRESH MVs AFTER ETL, BEFORE dashboard peak hours
   → ETL: 2:00 - 2:25 AM
   → MV refresh: 2:25 - 2:35 AM
   → Analyst start time: ~8 AM
   → 5.5 hour buffer for issues

3. GRANT SELECT on MVs to analysts
   → MVs are database objects — need explicit GRANT
   → GRANT SELECT ON ALL TABLES IN SCHEMA quickcart_ops TO GROUP analysts;
   → MVs are included in "ALL TABLES"

4. Monitor MV freshness
   → STV_MV_INFO system table: shows last refresh time
   → Alert if MV hasn't been refreshed in 25 hours
   → Stale MV = stale dashboard = wrong business decisions

5. Test query rewrite is working
   → Run EXPLAIN on dashboard query
   → Look for: "Scan mv_daily_revenue" instead of "Scan orders"
   → If rewrite not happening: check MV definition matches query

6. Size MVs appropriately
   → MV stores physical data — uses disk space
   → 12 MVs with tiny results: negligible (< 1 MB total)
   → 1 MV with 100M rows: significant — evaluate if needed

7. Use DISTSTYLE ALL for small MVs
   → Small result sets (< 10K rows): replicate to all nodes
   → Every node has full copy → no redistribution during queries
   → Maximizes query speed for dashboard reads

8. Include WHERE clauses in MV definitions
   → Don't materialize ALL history — only what dashboard needs
   → Q1: last 30 days → WHERE order_date >= CURRENT_DATE - 30
   → Smaller MV = faster refresh = less storage

❌ WHAT SHOULD NOT DO

1. DO NOT create MVs for queries that change every execution
   → "Show me orders for {user-selected date range}"
   → MV has FIXED definition — can't parameterize
   → Use MVs for FIXED dashboard panels, not ad-hoc filters
   → Exception: create MV at granular level, let dashboard filter from MV

2. DO NOT forget to REFRESH after ETL
   → MV without refresh = stale data = wrong numbers
   → Dashboard shows yesterday's revenue as today's
   → Finance team makes decisions on stale data
   → Always: ETL → REFRESH MV → validate → dashboard serves

3. DO NOT create too many MVs
   → Each MV = storage + refresh compute + maintenance overhead
   → 12 MVs for dashboards: fine
   → 500 MVs for every possible query: excessive
   → Rule: MV for queries run > 10 times/day with same parameters

4. DO NOT use MV as replacement for proper table design
   → If base table queries are slow due to bad dist/sort keys (Project 39)
   → Fix the table first, THEN add MVs for remaining complex queries
   → MV masks bad design — tech debt accumulates

5. DO NOT put window functions in AUTO REFRESH MVs
   → Some window functions prevent incremental refresh
   → Redshift falls back to full refresh → slow
   → Design MV queries to be incrementally refreshable when possible

6. DO NOT REFRESH during peak dashboard hours
   → REFRESH acquires locks on MV
   → During refresh: queries may block or return stale data
   → Schedule refresh during off-peak (early morning)

7. DO NOT ignore MV refresh failures
   → If refresh fails: MV retains OLD data
   → No error visible to dashboard users — just wrong numbers
   → Monitor: SVL_MV_REFRESH_STATUS for failures
   → Alert: SNS notification on refresh failure

8. DO NOT assume query rewrite always works
   → Rewrite requires EXACT match between query and MV definition
   → Small differences (column order, alias, extra WHERE) prevent rewrite
   → Always verify with EXPLAIN
   → Fallback: point dashboard directly at MV table name

🚫 WHEN NOT TO USE MATERIALIZED VIEWS

1. Data changes every minute and freshness < 1 minute required
   → MV refresh takes seconds to minutes
   → During refresh: stale data or lock contention
   → Use: direct query with good indexes, or real-time streaming

2. Query is unique every time (ad-hoc exploration)
   → "Show me product X sales in region Y for date range Z"
   → Infinite parameter combinations → can't pre-compute all
   → Use: well-designed base tables + Redshift result cache

3. Base table is tiny (< 10K rows)
   → Base table query already returns in < 100ms
   → MV adds complexity with zero performance gain
   → Use: direct query

4. Storage is critically limited
   → MVs consume disk space proportional to result size
   → On dc2.large: 160 GB per node
   → 12 small MVs: negligible
   → 1 MV with 500M rows: significant — evaluate trade-off

5. MV query uses unsupported features
   → Redshift MVs have restrictions:
     - No LIMIT clause
     - No HAVING clause (in some versions)
     - No correlated subqueries
     - No system functions like GETDATE() (use literal dates)
   → Check documentation for current version restrictions

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

┌───────────────────────────────────────────────────────────────────────────────┐
│              REDSHIFT MATERIALIZED VIEW — INTERNAL MECHANICS                  │
├───────────────────────────────────────────────────────────────────────────────┤
│                                                                               │
│  WHAT IS A MATERIALIZED VIEW?                                                 │
│  → A database object that stores the RESULT of a query physically             │
│  → Unlike a regular view (just stored SQL text, executed on read)             │
│  → MV = pre-computed result set stored as actual table data on disk           │
│  → Reading MV = reading a small table (fast)                                  │
│  → Refreshing MV = re-executing the query and storing new results             │
│                                                                               │
│  CREATE vs READ vs REFRESH lifecycle:                                         │
│                                                                               │
│  1. CREATE MATERIALIZED VIEW mv_daily_revenue AS                              │
│     SELECT order_date, SUM(total_amount)                                      │
│     FROM orders GROUP BY order_date;                                          │
│                                                                               │
│     INTERNALLY:                                                               │
│     a. Redshift executes the SELECT query                                     │
│     b. Takes result set (e.g., 30 rows)                                       │
│     c. Stores result as physical blocks on disk                               │
│     d. Creates metadata entry in pg_class (type = 'm')                        │
│     e. Records base table dependencies                                        │
│     f. MV is now queryable                                                    │
│                                                                               │
│  2. SELECT * FROM mv_daily_revenue;                                           │
│                                                                               │
│     INTERNALLY:                                                               │
│     a. Redshift reads physical blocks (same as table scan)                    │
│     b. NO join, NO aggregation, NO base table access                          │
│     c. 30 rows from disk → return to client                                   │
│     d. Time: < 50ms (vs 3.2 seconds for base query)                          │
│                                                                               │
│  3. REFRESH MATERIALIZED VIEW mv_daily_revenue;                               │
│                                                                               │
│     INTERNALLY:                                                               │
│     a. Redshift re-executes the original SELECT query                         │
│     b. Generates new result set                                               │
│     c. ATOMICALLY replaces old stored result with new                         │
│     d. During refresh: queries still return OLD data (no blocking)            │
│     e. After refresh completes: next query returns NEW data                   │
│     f. Records refresh timestamp in STV_MV_INFO                               │
│                                                                               │
│  ┌─────────────────────────────────────────────────────────────┐              │
│  │            QUERY REWRITE MECHANICS                           │              │
│  │                                                               │              │
│  │  User submits:                                                │              │
│  │  SELECT order_date, SUM(total_amount)                         │              │
│  │  FROM orders                                                  │              │
│  │  GROUP BY order_date;                                         │              │
│  │                                                               │              │
│  │  Redshift optimizer:                                          │              │
│  │  1. Parse query → generate logical plan                       │              │
│  │  2. Check: does any MV match this query?                      │              │
│  │     → Compare: tables referenced ✅                            │              │
│  │     → Compare: columns selected ✅                             │              │
│  │     → Compare: aggregation functions ✅                        │              │
│  │     → Compare: GROUP BY columns ✅                             │              │
│  │     → Compare: WHERE clauses ✅ (MV must be superset)         │              │
│  │  3. Match found: mv_daily_revenue                             │              │
│  │  4. Rewrite plan: Scan mv_daily_revenue instead of orders     │              │
│  │  5. Execute: read 30 rows from MV → return                    │              │
│  │                                                               │              │
│  │  REWRITE CONDITIONS:                                          │              │

│                                                                               │

│  │            AUTO REFRESH MECHANICS                             │              │
│  │                                                               │              │
│  │  When AUTO REFRESH is enabled:                                │              │
│  │                                                               │              │
│  │  1. Redshift tracks which base tables have been modified      │              │
│  │     (INSERT, UPDATE, DELETE, COPY, TRUNCATE)                  │              │
│  │                                                               │              │

│  │     → MV marked as "fresh"                                    │              │

│  │                                                               │              │

│  ┌─────────────────────────────────────────────────────────────┐              │
│  │            INCREMENTAL REFRESH MECHANICS                      │              │
│  │                                                               │              │
│  │  Full refresh:                                                │              │
│  │  → Re-execute entire query → scan ALL base table rows         │              │
│  │  → 12M rows scanned even if only 200K new rows                │              │
│  │                                                               │              │
│  │  Incremental refresh:                                         │              │
│  │  → Redshift tracks which base table rows changed              │              │
│  │  → Only processes delta (new/modified rows)                   │              │
│  │  → Merges delta into existing MV result                       │              │
│  │  → 200K rows processed instead of 12M                         │              │
│  │                                                               │              │
│  │  REQUIREMENTS for incremental refresh:                        │              │
│  │  → MV must use only: SELECT, FROM, JOIN, WHERE, GROUP BY     │              │
│  │  → Aggregates allowed: SUM, COUNT, MIN, MAX, AVG             │              │
│  │  → NO: DISTINCT, HAVING, ORDER BY, LIMIT, UNION              │              │
│  │  → NO: window functions, subqueries                           │              │
│  │  → NO: volatile functions (GETDATE, RANDOM)                   │              │
│  │  → Base tables must not be external (Spectrum) tables         │              │
│  │                                                               │              │
│  │  CHECK: STV_MV_INFO.is_incremental column                    │              │
│  │  → 1 = incremental refresh possible                           │              │
│  │  → 0 = only full refresh                                      │              │
│  └─────────────────────────────────────────────────────────────┘              │
│                                                                               │
│  STORAGE MODEL:                                                               │
│  → MV data stored in Redshift blocks (same as regular tables)                 │
│  → Inherits cluster's compression (ENCODE)                                    │
│  → Can specify DISTSTYLE and SORTKEY on MV                                    │
│  → Disk usage: proportional to MV result size                                 │
│  → 30 rows MV: < 1 KB on disk                                                │
│  → 1M rows MV: several MB on disk                                            │
│                                                                               │
│  CONCURRENCY MODEL:                                                           │
│  → REFRESH acquires exclusive lock on MV                                      │
│  → But: Redshift uses SNAPSHOT ISOLATION                                      │
│  → Concurrent reads see OLD data during refresh                               │
│  → After refresh commits: next read sees NEW data                             │
│  → NO blocking of reads during refresh                                        │

PITFALL 1: "MV created but dashboard still takes 15 seconds"
→ Cause: Query rewrite NOT triggered
→ Dashboard SQL doesn't EXACTLY match MV definition
→ Debug: EXPLAIN <dashboard_query> — look for MV scan vs base table scan
→ Fix: adjust MV definition to match dashboard query exactly
→ Or: point dashboard directly at SELECT * FROM mv_name

PITFALL 2: "REFRESH takes 30 seconds — cluster is busy"
→ Cause: REFRESH re-executes the full query
→ Complex query (3-table JOIN + window functions) = slow refresh
→ Fix: schedule refresh during off-peak
→ Or: use incremental refresh (if MV qualifies)
→ Or: reduce MV query complexity (pre-aggregate in stages)

PITFALL 3: "AUTO REFRESH never triggers"
→ Cause: AUTO REFRESH uses spare capacity
→ If cluster is 85% utilized: no spare capacity → refresh postponed
→ Fix: manual REFRESH via scheduler (more reliable)
→ Or: reduce cluster load so auto refresh has capacity

PITFALL 4: "MV shows yesterday's data at 10 AM"
→ Cause: ETL completed at 2:25 AM but MV refresh failed
→ MV still has data from previous refresh
→ No error visible to dashboard users
→ Fix: monitor SVL_MV_REFRESH_STATUS → alert on failure → retry

PITFALL 5: "CREATE MATERIALIZED VIEW fails with unsupported feature"
→ Cause: MV has restrictions — not all SQL is supported
→ Common: GETDATE() in WHERE clause → volatile function
→ Fix: replace GETDATE() with literal date calculated externally
→ Or: use parameterized stored procedure to refresh with current date

PITFALL 6: "MV query uses CURRENT_DATE but MV returns stale results"
→ Cause: CURRENT_DATE evaluated at CREATE time, not query time
→ MV WHERE order_date >= CURRENT_DATE - 30
→ After 31 days without refresh: MV returns data from wrong window
→ Fix: REFRESH daily (recalculates CURRENT_DATE)
→ Better: avoid date functions in MV — use date literals from scheduler

PITFALL 7: "Disk space alarm after creating 12 MVs"
→ Cause: some MVs materialize large intermediate results
→ mv_cumulative_revenue: 90 days × window function → 12M rows × 90 = large
→ Fix: design MV to produce SMALL result sets
→ Pre-aggregate before window function (staged approach)

PITFALL 8: "Query rewrite stopped working after schema change"
→ Cause: ALTER TABLE on base table invalidates MV
→ Adding/dropping columns on base table → MV marked invalid
→ Fix: DROP and recreate MV after schema changes
→ Include MV recreation in ETL DDL migration scripts (Project 76)

PITFALL 9: "REFRESH MATERIALIZED VIEW hangs for 2 hours"
→ Cause: base table has queued write locks from ETL
→ REFRESH waits for ETL writes to complete
→ If ETL is slow → refresh is blocked
→ Fix: ensure ETL fully commits before refresh starts
→ Use Step Functions: ETL complete → wait → REFRESH (Project 55)

PITFALL 10: "15 analysts see slightly different dashboard numbers"
→ Cause: REFRESH happened mid-session
→ Analysts who queried before refresh: old data
→ Analysts who queried after refresh: new data
→ Fix: schedule refresh before working hours (not during)
→ Or: accept transient inconsistency — numbers converge within minutes

📈 SCALABILITY CONSIDERATIONS

NUMBER OF MVs:
→ 12 MVs (our case): trivial overhead
→ 50 MVs: manageable — refresh takes longer (serialize or parallelize)
→ 200 MVs: significant — REFRESH ALL takes substantial time
→ 500+ MVs: reconsider — likely many are unused
→ Monitor: SVL_MV_REFRESH_STATUS → identify slow/unused MVs

MV SIZE SCALING:
→ 30 rows (daily revenue): < 1 KB storage, < 10ms refresh
→ 10K rows (geo revenue): ~50 KB storage, ~200ms refresh
→ 1M rows (detailed aggregate): ~5 MB storage, ~5s refresh
→ 100M rows (large intermediate): ~500 MB storage, ~60s refresh
→ Rule: MV should be 10-1000x SMALLER than base tables

CONCURRENT USERS:
→ 15 users (our case): MV serves all from same pre-computed data
→ 100 users: same MV, same speed — MV doesn't slow with users
→ 1000 users: may need Redshift concurrency scaling (Project 40)
→ MV BENEFIT MULTIPLIES with user count:
  - 15 users × 12 queries = 180 queries → 12 MV scans
  - 100 users × 12 queries = 1200 queries → 12 MV scans
  - Savings grow linearly with users

REFRESH SCALING:
┌────────────────────┬──────────────┬─────────────────────────────────┐
│ MV Complexity      │ Refresh Time │ Strategy                        │
├────────────────────┼──────────────┼─────────────────────────────────┤
│ Simple agg (SUM)   │ < 1s         │ Full refresh — fast enough      │
│ Multi-table JOIN   │ 2-10s        │ Full refresh — acceptable       │
│ Window function    │ 10-60s       │ Consider staged MVs             │
│ Complex subquery   │ 60s+         │ Break into multiple simpler MVs │
└────────────────────┴──────────────┴─────────────────────────────────┘

CLUSTER SCALING:
→ dc2.large (2 nodes): handles 12 MVs easily
→ If MV refresh competes with ad-hoc queries:
  → Use WLM queue (Project 40): dedicate queue for MV refresh
  → Or: schedule refresh in maintenance window
→ If cluster upgraded to RA3: larger storage → bigger MVs possible
→ If Redshift Serverless (Project 67): RPU auto-scales for refresh

COST IMPACT:
→ Storage: negligible (< 1 MB for 12 small MVs)
→ Compute (refresh): 12 queries × once/day = ~2 minutes of compute
→ Compute (saved): 3,444 redundant queries × ~8s avg = 7.6 hours/day saved
