Company: "QuickCart" — Analytics team growing fast

THE PROBLEM:
→ BI team says: "Athena is too slow for dashboards — 8-15 second query times"
→ Dashboards need sub-second response on pre-aggregated data
→ They tried Redshift COPY (Project 21) manually — works but:
   - Manual process — someone runs COPY commands each morning
   - No transformation between S3 and Redshift
   - No deduplication — duplicate rows if COPY runs twice
   - No pre/post cleanup of staging tables
   - No monitoring or alerting on failures

→ Data team asks: "Can Glue read the Parquet from S3 (Project 31 output)
   and load it into Redshift AUTOMATICALLY with transforms?"

→ CTO says: "One pipeline: MariaDB → S3 (done) → Redshift (need this)"

🟡 HOW TO APPROACH IT

Step 1: Understand what "Glue writes to Redshift" actually means
→ Glue does NOT insert rows one-by-one via JDBC (that would be insanely slow)
→ Internally, Glue does this:
   a) Writes DynamicFrame data to a TEMP S3 location as CSV/Parquet
   b) Issues Redshift COPY command pointing to that temp location
   c) COPY bulk-loads into the target table
   d) Cleans up temp files
→ This is WHY Glue needs BOTH an S3 path AND a Redshift JDBC connection

Step 2: Understand the NETWORK path
→ Glue workers need access to:
   a) S3 — to read source Parquet (VPC Endpoint — Project 31)
   b) S3 — to write temp staging files (same endpoint)
   c) Redshift — to issue COPY commands via JDBC (port 5439)
→ Redshift is in a VPC → Security Group must allow Glue → Redshift

Step 3: Understand the Redshift loading strategy
→ Full Load: TRUNCATE target → COPY all rows (simple but slow for large tables)
→ Upsert/Merge: Load into staging table → MERGE into target (production pattern)
→ Append: INSERT without dedup (for event/log tables only)
→ We'll implement UPSERT — the production-grade approach

Step 4: Understand pre-actions and post-actions
→ Glue supports SQL statements that run BEFORE and AFTER the load
→ Pre-action: CREATE staging table, TRUNCATE staging table
→ Post-action: MERGE from staging to target, DROP staging table
→ This is how deduplication works

Step 5: Map the dependencies
→ S3 Parquet exists (Project 31 output)
→ Redshift cluster exists (Projects 10, 11)
→ Glue basics understood (Project 15, 25, 31)
→ Redshift COPY understood (Project 21)
→ VPC/SG/Endpoint understood (Project 31 — same VPC)

🔵 WHAT COULD BE DONE

Option A: Glue writes directly to Redshift JDBC (row-by-row)
→ connection_type="jdbc" with INSERT statements
→ Painfully slow: ~1,000 rows/second
→ 45M rows = 12.5 HOURS
→ NEVER do this

Option B: Glue writes to S3 → Manual COPY to Redshift
→ Two separate steps: Glue job + SQL script
→ No atomicity — if COPY fails, S3 data is orphaned
→ Manual intervention required
→ This is what Project 21 does — but we want automation

Option C: Glue writes to Redshift using built-in connector (RECOMMENDED)
→ Glue's Redshift connector does the S3-staging + COPY internally
→ Supports pre/post actions for staging table pattern
→ Single job: read → transform → stage → merge → cleanup
→ Fully automated, retryable, monitorable

Option D: Redshift Spectrum reads S3 directly (no loading)
→ No data movement — queries S3 from Redshift
→ Slower than local Redshift tables for hot queries
→ Covered in Project 38 — different use case
→ Good for ad-hoc; bad for dashboards needing <1s response

DECISION: Option C — Glue built-in Redshift connector
→ Production-grade pattern used at scale
→ Handles staging + merge + cleanup atomically
→ Interview gold: demonstrates understanding of ETL internals

✅ WHAT SHOULD DO

1. Create SEPARATE Redshift connection in Glue
   → Even if MariaDB connection exists (Project 31)
   → Different JDBC URL, port (5439 vs 3306), credentials
   → Same VPC, same Glue Security Group can be reused

2. Use STAGING TABLE + MERGE pattern
   → Load into stg_orders first
   → MERGE into fact_orders using primary key
   → Handles duplicates, updates, and inserts in one operation

3. Create Redshift tables with proper DISTKEY and SORTKEY
   → DISTKEY on join column (e.g., customer_id for fact_orders)
   → SORTKEY on query filter column (e.g., order_date)
   → Covered in Project 39 but needed here for table creation

4. Grant Redshift cluster an IAM Role for S3 access
   → Redshift COPY needs to read from temp S3 directory
   → Role: allows s3:GetObject on temp path
   → This is the SAME concept as Project 21

5. Use redshiftTmpDir parameter
   → Tells Glue WHERE to write temp staging files
   → Must be an S3 path the Redshift IAM Role can read
   → Clean up old temp dirs periodically

6. Add pre-actions and post-actions as SQL strings
   → Pre: CREATE TABLE IF NOT EXISTS + TRUNCATE staging
   → Post: MERGE/INSERT-SELECT from staging → target + DROP staging

7. Use a DEDICATED Redshift user for Glue
   → Don't use admin/masteruser
   → GRANT INSERT, SELECT, CREATE, DROP on analytics schema only

8. Test with small table first (products — 50K rows)
   → Verify full flow before running 45M row orders table

❌ WHAT SHOULD NOT DO

1. DO NOT write to Redshift using connection_type="jdbc" with INSERT
   → As explained in Option A — rows trickle in one by one
   → 1,000x slower than COPY-based approach
   → The Glue Redshift connector handles COPY internally

2. DO NOT skip the staging table
   → Direct load into target table means:
     - No deduplication
     - No rollback if load fails halfway
     - Table locked during load → dashboard queries fail
   → Staging pattern isolates the load from live queries

3. DO NOT use the same temp S3 directory for multiple concurrent jobs
   → Job A writes temp files → Job B writes temp files → COPY picks up wrong files
   → Use unique paths: s3://bucket/tmp/glue/{job_name}/{run_id}/

4. DO NOT forget to grant Redshift's IAM Role access to temp S3 path
   → Glue writes temp files using Glue's IAM Role
   → Redshift reads temp files using Redshift's IAM Role
   → BOTH roles need S3 access — different roles, same bucket

5. DO NOT load all historical data every night
   → First run: full load (all historical Parquet partitions)
   → Subsequent runs: only today's partition (incremental)
   → Without this: 45M rows × 365 days = 16 BILLION rows re-loaded yearly

6. DO NOT create Redshift tables without DISTKEY/SORTKEY
   → Default: EVEN distribution → terrible join performance
   → Project 39 covers this deeply
   → At minimum: set DISTKEY on fact table's most-joined column

7. DO NOT ignore Redshift COPY errors
   → COPY loads what it can, skips bad rows silently
   → Check STL_LOAD_ERRORS after every load
   → Post-action should include error check query

8. DO NOT run Glue Redshift loads during peak dashboard hours
   → COPY acquires table locks
   → Dashboard queries queue behind locks
   → Schedule loads at 2-4 AM

🚫 WHEN NOT TO USE GLUE → REDSHIFT

1. Data is already in S3 and queries are infrequent
   → Use Athena or Redshift Spectrum instead
   → Loading into Redshift adds cost (storage + compute)
   → Only load what dashboards actively query

2. Table has < 10,000 rows and no transforms needed
   → Use Redshift COPY directly (Project 21)
   → Glue overhead: $0.73 minimum (10 DPU-minutes)
   → COPY command: free (Redshift already running)

3. Need real-time/streaming ingestion
   → Glue is batch-oriented (minimum ~30s cold start)
   → Use Kinesis Data Firehose → Redshift (Project 72)
   → Or Redshift Streaming Ingestion (native)

4. Source and target schemas are identical — no transform needed
   → Direct COPY from S3 → Redshift is simpler
   → Glue adds value ONLY when transformation is needed

5. Redshift Serverless with low RPU budget
   → COPY from Glue consumes RPUs
   → Large loads on tiny RPU = very slow
   → Size RPUs appropriately or schedule loads

│  → Glue executes your postactions SQL string                                     │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "COPY fails: S3ServiceException: Access Denied"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Redshift's IAM Role cannot read from Glue's temp S3 path
→ Remember: TWO roles involved
→ Glue writes temp files → Redshift reads temp files
→ Fix: Grant Redshift IAM Role s3:GetObject on temp S3 path
→ AND: Ensure Redshift Role is attached to the cluster
→ AND: Pass correct role ARN in aws_iam_role connection option

PITFALL 2: "Job succeeds but Redshift table has 0 rows"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause A: Post-action DELETE deleted everything before INSERT ran
→ Post-actions run as SEPARATE statements, not one transaction
→ Fix: Wrap DELETE + INSERT in BEGIN...COMMIT
→ Cause B: Source Parquet partition was empty for today
→ Fix: Add row count check before write step

PITFALL 3: "Redshift table has duplicate rows"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Job ran twice (retry) without TRUNCATE staging first
→ Second run appends to staging → double rows merged into target
→ Fix: Pre-action MUST include TRUNCATE or DROP+CREATE staging table

PITFALL 4: "COPY loads but some columns have NULL — data looks wrong"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Column order mismatch between Glue output and Redshift table
→ Glue writes CSV with columns in DynamicFrame order
→ Redshift COPY maps columns positionally (column 1 → column 1)
→ Fix: Ensure DynamicFrame column order matches Redshift DDL exactly
→ OR: Use column mapping in COPY command

PITFALL 5: "Job hangs on Redshift connection — timeout after 15 min"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Same as Project 31 Pitfall 1 — Security Group issue
→ Glue SG needs self-referencing rule (already done in Project 31)
→ Redshift SG needs inbound rule: port 5439 from Glue SG
→ Fix: Add sg-redshift INBOUND: TCP 5439 from sg-glue

PITFALL 6: "redshiftTmpDir has thousands of orphaned temp folders"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Glue cleanup step failed or was skipped
→ Each job run creates a new temp-{uuid}/ folder
→ 365 runs/year × 10 files each = 3,650 orphaned files
→ Fix: S3 Lifecycle Rule on tmp/ prefix → delete after 3 days

PITFALL 7: "Dashboard queries are slow during Glue load"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: DELETE + INSERT on large target table acquires locks
→ Dashboard SELECT queries queue behind write lock
→ Fix A: Schedule loads during off-peak hours (2-4 AM)
→ Fix B: Use Redshift WLM (Project 40) to prioritize dashboard queries
→ Fix C: Load into separate table → swap via ALTER TABLE RENAME

PITFALL 8: "Column types don't match — COPY fails with data type error"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Parquet DOUBLE → Redshift DECIMAL(10,2) precision mismatch
→ Cause: Parquet STRING → Redshift VARCHAR(50) — string too long
→ Fix: Cast all columns explicitly in Glue transform step
→ Fix: Use TRUNCATECOLUMNS in COPY to handle overflow gracefully

├─────────────────┼────────────────────────┼────────────────────────────────────┤
│ 500M+ rows      │ Table swap pattern     │ Load into new table → RENAME      │
│                 │                        │ Avoids long locks on target table  │
└─────────────────┴────────────────────────┴────────────────────────────────────┘

REDSHIFT CLUSTER SIZING FOR LOADS:
→ 2 dc2.large nodes: handles up to ~100M rows/load comfortably
→ 4 dc2.large nodes: handles up to ~500M rows/load
→ For larger: consider ra3.xlplus with managed storage

CONCURRENT LOADS:
→ Redshift supports concurrent COPY commands
→ BUT: each COPY acquires write lock on target table
→ Rule: max 1 COPY per target table at a time
→ Different tables can load concurrently (different locks)

TEMP S3 STORAGE:
→ 45M rows as CSV ≈ 2-4 GB temp files
→ Deleted after each run
→ Peak storage: depends on how many jobs run concurrently
→ Budget: negligible ($0.10/day at most)

COST BREAKDOWN:
→ Glue: 10 DPUs × 10 min = $0.73/run
→ Redshift: already running (sunk cost for dashboards)
→ S3 temp: ~$0.001/run
→ Total incremental cost: ~$0.73/run × 30 days = $21.90/month
→ vs. manual process: 1 hour/day × $50/hour engineer = $1,500/month

→ Pre-actions and post-actions wrap the load with setup + cleanup SQL

→ Column ORDER must match between DynamicFrame and Redshift DDL
→ S3 Lifecycle rules prevent orphaned temp file accumulation
