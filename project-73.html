🔵 WHAT COULD BE DONE

OPTION A: Lake-Only Architecture (no Redshift)
→ S3 + Glue + Athena for everything
→ Pro: cheapest, serverless, scales infinitely
→ Con: Athena query latency 5-30 seconds (bad for dashboards)
→ Con: no sub-second BI experience
→ ❌ REJECTED for BI use case

OPTION B: Warehouse-Only Architecture (no S3 lake)
→ Redshift for everything — COPY all data in
→ Pro: fast queries, familiar SQL
→ Con: storage expensive at 15 TB ($4,800/month on RA3)
→ Con: no separation of compute and storage
→ Con: can't use Glue crawlers, Athena, or lake patterns
→ ❌ REJECTED — doesn't scale

OPTION C: Lakehouse — S3 Lake + Redshift Warehouse + Spectrum Bridge ✅
→ S3 for storage (cheap, infinite)
→ Glue Catalog for ALL schemas (unified)
→ Redshift for HOT data (dashboards, materialized views)
→ Spectrum for WARM data (query S3 from Redshift — no COPY needed)
→ Athena for ad-hoc (direct S3 queries)
→ Lake Formation for governance (one place for all permissions)
→ ✅ SELECTED — best of both worlds

OPTION D: Add Apache Iceberg table format
→ Enables ACID transactions on S3
→ Time travel, schema evolution, partition evolution
→ Supported by Athena, Glue, Redshift, EMR
→ ✅ INCLUDED as enhancement for Gold tables

DECISION: Option C + Option D for Gold layer
→ This is the modern "Lakehouse" pattern
→ AWS calls it "Zero-ETL" vision

✅ WHAT SHOULD DO

1. GLUE DATA CATALOG = single schema registry for EVERYTHING
   → Every S3 table registered here
   → Redshift Spectrum reads schemas from here
   → Athena reads schemas from here
   → Lake Formation governs permissions here
   → ONE place to find any table in the platform

2. LAKE FORMATION = single permission layer
   → Replace scattered S3 bucket policies + IAM policies
   → Database-level, table-level, column-level grants
   → Cross-account grants (Project 71 mesh)
   → Audit log: who accessed what, when

3. REDSHIFT SPECTRUM = bridge between warehouse and lake
   → Create external schemas pointing to Glue Catalog databases
   → Query S3 Gold data FROM Redshift SQL
   → JOIN S3 data with Redshift internal tables
   → No COPY needed for historical/cold data

4. SEPARATE HOT vs WARM vs COLD data
   → HOT (Redshift internal): last 90 days, materialized views, dashboards
   → WARM (S3 Gold via Spectrum): 90 days - 2 years, ad-hoc queries
   → COLD (S3 Glacier): 2+ years, compliance/archive (Project 4)

5. USE ICEBERG FOR GOLD TABLES
   → Enables UPDATE/DELETE on S3 (not possible with plain Parquet)
   → Time travel: query data as of yesterday
   → Compaction: automatic small file merging
   → Schema evolution: add columns without breaking consumers

6. UNIFIED QUERY PATTERN FOR ANALYSTS
   → Athena: for exploration and ad-hoc
   → Redshift: for dashboards and BI
   → BOTH read from same Glue Catalog
   → BOTH see same table names and schemas
   → Analyst doesn't need to know where data physically lives

7. ORCHESTRATE EVERYTHING THROUGH STEP FUNCTIONS
   → Nightly batch: Step Functions → Glue ETL → Redshift COPY → SNS
   → Real-time: EventBridge → Step Functions → Glue micro-batch → SNS
   → Quality gates: Glue DQ → pass/fail → conditional next step

❌ WHAT SHOULD NOT DO

1. DO NOT maintain TWO separate catalogs (Glue + Redshift)
   → Redshift Spectrum uses Glue Catalog as external catalog
   → Don't CREATE TABLE in Redshift AND Glue separately
   → Single source of truth: Glue Catalog

2. DO NOT COPY all S3 data into Redshift
   → 15 TB in S3 → loading into Redshift = $4,800/month storage
   → Use Spectrum for cold/warm data
   → Only COPY hot data needed for sub-second dashboard queries

3. DO NOT use Lake Formation AND S3 bucket policies for same tables
   → Lake Formation replaces S3 IAM-based access for Glue/Athena
   → Mixing both creates permission conflicts
   → Choose one: Lake Formation for governed tables, IAM for non-governed

4. DO NOT let each team create their own Glue databases
   → Naming convention enforced: {domain}_{layer}
   → Examples: orders_bronze, orders_silver, orders_gold
   → Platform team creates databases, domain teams create tables

5. DO NOT skip Iceberg for Gold tables that need updates
   → Plain Parquet is immutable — "updates" require full partition rewrite
   → Iceberg: UPDATE/DELETE at row level
   → Iceberg: automatic compaction of small files (Project 62 solved)

6. DO NOT give Spectrum roles more S3 access than needed
   → Spectrum IAM role should only access gold/ prefix
   → Never bronze/ or silver/ — those are internal to ETL

🚫 WHEN NOT TO USE THIS FULL PLATFORM

1. TEAM SIZE < 10 ENGINEERS
   → This platform has 15+ AWS services
   → Operational overhead requires dedicated platform team
   → Smaller teams: stick with S3 + Glue + Athena (no Redshift)

2. DATA VOLUME < 100 GB TOTAL
   → Redshift minimum cost: ~$180/month (Serverless)
   → Athena alone handles 100 GB easily at $0.50/query
   → Add Redshift only when Athena latency becomes unacceptable

3. ALL QUERIES ARE AD-HOC (no dashboards)
   → Redshift value = sub-second BI dashboards
   → If no dashboards: Athena + S3 is sufficient
   → Spectrum adds no value without a Redshift cluster

4. SINGLE DATA SOURCE, NO JOINS
   → Full platform justified when data comes from 5+ sources
   → Single source: simple Glue + Athena pipeline is enough

5. COMPLIANCE PROHIBITS CLOUD DATA LAKE
   → Some industries require on-premises only
   → This architecture is 100% AWS cloud-native

│  WHEN ATHENA QUERIES:                                            │

│                                                                  │

HOW LAKE FORMATION PERMISSION MODEL WORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│  WITHOUT LAKE FORMATION (old way — scattered):                   │
│  ┌────────────┐  ┌────────────┐  ┌────────────────┐            │
│  │ S3 Bucket  │  │ IAM Policy │  │ Glue Resource  │            │
│  │ Policy     │  │ (per user) │  │ Policy         │            │
│  └────────────┘  └────────────┘  └────────────────┘            │
│  → 3 places to check, 3 places to update, 3 places to audit    │
│  → Cross-account = even more complex                             │
│                                                                  │
│  WITH LAKE FORMATION (new way — centralized):                    │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │                LAKE FORMATION                             │   │
│  │                                                           │   │
│  │  GRANT SELECT ON TABLE orders_gold.daily_orders           │   │
│  │  TO ROLE arn:aws:iam::333333333333:role/AnalystRole       │   │
│  │  COLUMNS (order_id, order_total, status)                  │   │
│  │  — column "customer_email" EXCLUDED                        │   │
│  │                                                           │   │
│  │  HOW IT WORKS INTERNALLY:                                 │   │
│  │  1. Lake Formation stores grant in its own permission DB  │   │
│  │  2. When Athena/Spectrum queries this table:              │   │
│  │     → Lake Formation intercepts the Glue GetTable call    │   │
│  │     → Checks: does caller have grant?                     │   │
│  │     → If YES: returns table metadata (filtered columns)   │   │
│  │     → If NO: returns Access Denied                        │   │
│  │  3. Lake Formation also vends temporary S3 credentials    │   │
│  │     → Caller never needs direct S3 IAM permissions        │   │
│  │     → Lake Formation creates scoped-down STS creds        │   │
│  │     → These creds can only read the granted S3 paths      │   │
│  │                                                           │   │
│  │  RESULT:                                                   │   │
│  │  → ONE grant = controls Athena + Spectrum + Glue access   │   │
│  │  → Column-level filtering without data duplication         │   │
│  │  → Full audit trail in CloudTrail                          │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘

HOW ICEBERG TABLE FORMAT WORKS ON S3
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│  PLAIN PARQUET ON S3:                                            │
│  s3://bucket/gold/orders/                                        │
│    year=2025/month=01/day=15/                                    │
│      part-00000.parquet                                          │
│      part-00001.parquet                                          │
│  → Immutable — can't UPDATE or DELETE individual rows            │
│  → "Update" = rewrite entire partition                           │
│  → No transaction isolation                                      │
│  → No time travel                                                │
│                                                                  │
│  ICEBERG ON S3:                                                  │
│  s3://bucket/gold/orders_iceberg/                                │
│    data/                                                         │
│      00000-0-abc.parquet                                         │
│      00001-0-def.parquet                                         │
│    metadata/                                                     │
│      v1.metadata.json          ← snapshot 1                     │
│      v2.metadata.json          ← snapshot 2 (after UPDATE)      │
│      snap-123-0-abc.avro       ← manifest list                  │
│      abc-m0.avro               ← manifest (file-level metadata) │
│                                                                  │
│  HOW UPDATE WORKS:                                               │
│  1. UPDATE orders SET status='shipped' WHERE order_id='ORD-001' │
│  2. Iceberg reads manifest → finds which Parquet file has row   │
│  3. Writes NEW Parquet file with updated row                     │
│  4. Creates NEW manifest pointing to new file + old files        │
│  5. Creates NEW metadata.json (snapshot v3)                      │
│  6. Old files NOT deleted — still referenced by old snapshots    │
│                                                                  │
│  TIME TRAVEL:                                                    │
│  SELECT * FROM orders_iceberg FOR SYSTEM_TIME AS OF              │
│    TIMESTAMP '2025-01-14 00:00:00'                               │
│  → Athena reads snapshot v1 (before the UPDATE)                  │
│  → Returns original data as of that timestamp                    │
│                                                                  │
│  COMPACTION (solves Project 62):                                 │
│  → Iceberg auto-compacts small files into larger ones            │
│  → Background process: rewrite data files, update manifests      │
│  → Readers always see consistent view                            │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: Lake Formation enabled but old IAM policies still active
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Lake Formation has "IAMAllowedPrincipals" super-permission by default
→ This bypasses Lake Formation for everyone — effectively disabled
→ MUST remove IAMAllowedPrincipals to activate Lake Formation governance
→ Then: only Lake Formation grants work (IAM policies ignored for Glue/Athena)

PITFALL 2: Spectrum query returns 0 rows but data exists in S3
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause 1: External schema points to wrong Glue Catalog database
→ Cause 2: Partitions not registered (MSCK REPAIR or add_partition needed)
→ Cause 3: Redshift Spectrum IAM role lacks s3:GetObject permission
→ Debug: query SVL_S3LOG for Spectrum execution details

PITFALL 3: Athena and Spectrum return different results for same table
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Both use Glue Catalog — results SHOULD be identical
→ If different: check Redshift external schema — it may have stale cache
→ Fix: DROP EXTERNAL SCHEMA + recreate, or ALTER TABLE REFRESH

PITFALL 4: Iceberg table compaction never runs
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Iceberg doesn't auto-compact unless you configure it
→ Small files accumulate → queries slow down
→ Fix: schedule Athena OPTIMIZE TABLE or Glue compact job

PITFALL 5: Lake Formation cross-account grant fails silently
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Granting access to external account requires:
  a) Lake Formation grant in producer account
  b) RAM (Resource Access Manager) share accepted in consumer account
  c) Consumer account's Lake Formation admin accepts the shared resource
→ Missing ANY step = access denied with no clear error

PITFALL 6: Redshift COPY and Spectrum competing for same S3 data
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ COPY loads data INTO Redshift (duplicates S3 data)
→ Spectrum queries data IN S3 (no duplication)
→ If both exist: BI queries may hit internal OR external depending on schema
→ Confusion about which version is "current"
→ Fix: clear strategy — COPY for hot data, Spectrum for warm/cold

→ Glue Catalog: no practical limit on tables/databases
→ Partitions: max 10 million per table (use partition projection to avoid this)
→ Lake Formation: no practical limit on grants

→ S3 Glacier for compliance archives (Project 4)
→ Redshift Serverless with min RPU=0 for dev/test (auto-pauses)

→ Iceberg file pruning: metadata-driven skip of irrelevant files
