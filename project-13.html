BUFFER INCOMING MESSAGES SO NO DATA IS LOST
🟢 Phase 1: Very Easy

🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — SNS notifications working (Project 12)
Situation:
→ Multiple data sources now feed into the QuickCart data platform
→ Each source generates "processing requests" that need to be handled

SCENARIO — THE 3 AM FLOOD:
→ 2:00 AM: Marketing uploads 200 campaign CSV files to S3
→ 2:01 AM: S3 event notifications fire via SNS (Project 12 pattern)
→ 2:01 AM: EC2 processing script receives notifications
→ Problem: EC2 script can process 1 file every 30 seconds
→ 200 files × 30 seconds = 100 minutes needed
→ But SNS delivers ALL 200 notifications in < 5 seconds
→ EC2 script picks up the FIRST notification, processes it
→ The other 199 notifications? GONE. SNS already delivered them.
→ Nobody was listening when they arrived.

MONDAY MORNING:
→ Analyst: "Where are the 200 campaign files? I only see 1 loaded."
→ You: "SNS delivered 200 notifications but our processor was busy 
   with the first one. The other 199 were lost."
→ CTO: "We need a BUFFER. Something that HOLDS messages until 
   the processor is ready for them."

SECOND SCENARIO — ORDER OF OPERATIONS:
→ Order system sends: "Order Created" → "Payment Received" → "Shipped"
→ All 3 events arrive within 1 second
→ If processor handles "Shipped" before "Order Created":
   → Tries to update order that doesn't exist yet
   → Processing fails silently
→ CTO: "Some events MUST be processed in ORDER."

THIRD SCENARIO — POISON MESSAGE:
→ One malformed CSV file causes the processor to crash
→ Processor restarts, picks up same file, crashes again
→ Infinite crash loop — nothing else gets processed
→ CTO: "If a message keeps failing, QUARANTINE it. 
   Don't let one bad message kill the entire pipeline."

REQUIREMENTS:
→ Hold messages until consumer is ready (buffering)
→ Don't lose messages if consumer is temporarily down
→ Optionally guarantee message ORDER (FIFO)
→ Quarantine repeatedly failing messages (Dead Letter Queue)
→ Scale: handle bursts of 1000s of messages
→ Cost: pay only for what you use

🟡 HOW TO APPROACH IT

Step 1: Understand the fundamental problem
→ SNS is PUSH-based: delivers immediately, doesn't wait
→ If subscriber is busy/down: message is lost (for HTTP endpoints)
→ Need: PULL-based system where consumer reads at its own pace
→ This is a MESSAGE QUEUE

Step 2: Understand SQS basics
→ Producer SENDS messages to queue
→ Messages PERSIST in queue until consumed (up to 14 days)
→ Consumer POLLS queue for messages
→ Consumer PROCESSES message
→ Consumer DELETES message from queue (acknowledgment)
→ If consumer crashes: message becomes VISIBLE again (retry)

Step 3: Choose Standard vs FIFO
→ Standard Queue:
   → Nearly unlimited throughput
   → At-least-once delivery (may deliver duplicates)
   → Best-effort ordering (mostly in order, no guarantee)
→ FIFO Queue:
   → Exactly-once processing
   → Strict ordering within message groups
   → 300 messages/second (3000 with batching)
   → Name must end with ".fifo"

Step 4: Design for QuickCart
→ File processing queue: Standard (order doesn't matter, need throughput)
→ Order events queue: FIFO (must process in sequence per order)
→ Dead Letter Queue: catches repeatedly failing messages

🔵 WHAT COULD BE DONE

Option A: SQS Standard Queue
→ Nearly unlimited throughput (thousands of msg/sec)
→ At-least-once delivery (consumer must handle duplicates)
→ Best-effort ordering (mostly in order)
→ Best for: file processing, batch jobs, fan-out consumption

Option B: SQS FIFO Queue
→ 300 msg/sec (3000 with batching)
→ Exactly-once processing (deduplication built in)
→ Strict ordering within message groups
→ Best for: order events, financial transactions, state machines

Option C: Amazon Kinesis Data Streams
→ Real-time streaming (sub-second latency)
→ Multiple consumers read same stream
→ 7-day retention, replay from any point
→ Best for: real-time analytics, log aggregation
→ Overkill for QuickCart's batch-oriented needs

Option D: Amazon MQ (managed RabbitMQ/ActiveMQ)
→ Full-featured message broker
→ AMQP, MQTT, STOMP protocols
→ Best for: migrating from on-prem message brokers
→ More complex than SQS

DECISION:
→ Standard Queue for file processing (throughput matters)
→ FIFO Queue for order events (ordering matters)
→ DLQ for both (quarantine poison messages)

✅ WHAT SHOULD DO

1. Create Dead Letter Queue BEFORE main queue
   → DLQ must exist when main queue references it
   → Set maxReceiveCount = 3 (message fails 3 times → moves to DLQ)
   → Set DLQ retention to 14 days (maximum — gives time to investigate)

2. Set visibility timeout based on processing time
   → If processor takes 30 seconds: set visibility to 5× = 150 seconds
   → WHY: if processor crashes at 29 seconds, message becomes visible 
     again at 150 seconds — another consumer picks it up
   → Too short: message becomes visible while still being processed → duplicate
   → Too long: message stuck invisible after crash → delayed retry

3. Use LONG POLLING (WaitTimeSeconds = 20)
   → Short polling: returns immediately (even if queue empty) — wastes API calls
   → Long polling: waits up to 20 seconds for messages — saves money
   → Max: 20 seconds per ReceiveMessage call
   → Reduces API calls by 90%+ for low-volume queues

4. ALWAYS delete message after successful processing
   → receive_message → process → delete_message
   → If you don't delete: message becomes visible again after visibility timeout
   → Consumer processes it AGAIN → duplicate processing

5. Use message attributes for metadata
   → file_type: "csv" | "parquet"
   → source: "marketing" | "sales"
   → Consumers can filter or route based on attributes
   → Don't put metadata in message body — harder to parse

6. Set message retention based on business need
   → Default: 4 days
   → Max: 14 days
   → If not consumed in 14 days: message permanently deleted
   → QuickCart: 7 days (enough time to fix issues)

7. Enable server-side encryption (SSE-SQS)
   → Messages encrypted at rest
   → Transparent to producers/consumers
   → Free with SQS-managed keys
   → Or use KMS for customer-managed keys

8. Monitor queue depth with CloudWatch
   → ApproximateNumberOfMessagesVisible = backlog
   → ApproximateNumberOfMessagesNotVisible = in-flight
   → ApproximateAgeOfOldestMessage = how long oldest message waiting
   → Alarm when backlog exceeds threshold (Project 54)

❌ WHAT SHOULD NOT DO

1. DO NOT use short polling in production
   → Short polling: returns immediately even if empty
   → Empty receives still count as API calls ($0.40/million)
   → 10 consumers × 1 poll/second × 86400 seconds = 864,000 calls/day
   → Long polling: same scenario = ~43,200 calls/day (20x reduction)

2. DO NOT set visibility timeout shorter than processing time
   → Message becomes visible while still being processed
   → Another consumer picks it up → duplicate processing
   → Rule: visibility = 5× expected processing time

3. DO NOT forget to delete messages after processing
   → Undeleted message becomes visible again after timeout
   → Consumer processes it again → infinite reprocessing loop
   → Wastes compute, creates duplicate data

4. DO NOT use SQS as a database
   → SQS is a transit system — messages are meant to be consumed and deleted
   → Don't store messages permanently — use S3 or DynamoDB
   → Max retention: 14 days — messages auto-delete after

5. DO NOT put large payloads in SQS messages
   → Max message size: 256 KB
   → For larger data: put data in S3, send S3 key in SQS message
   → Pattern: {"s3_bucket": "quickcart", "s3_key": "path/to/file.csv"}
   → SQS Extended Client Library handles this automatically

6. DO NOT create FIFO queues when ordering isn't required
   → FIFO: 300 msg/sec limit (3000 with batching)
   → Standard: nearly unlimited
   → FIFO costs 25% more than Standard
   → Only use FIFO when ORDER or EXACTLY-ONCE truly matters

7. DO NOT ignore Dead Letter Queue messages
   → DLQ messages = processing failures that need investigation
   → Set CloudWatch alarm on DLQ depth > 0
   → Review DLQ daily — fix root cause, reprocess if needed

8. DO NOT process messages in batches larger than 10
   → ReceiveMessage max: 10 messages per call
   → DeleteMessageBatch max: 10 per call
   → AWS hard limit — can't change

🚫 WHEN NOT TO USE SQS

1. When you need real-time streaming with sub-second latency
   → SQS long polling adds up to 20 seconds delay
   → Use: Kinesis Data Streams (millisecond latency)

2. When multiple consumers need to read the SAME message
   → SQS: once a message is read and deleted, it's gone
   → Use: SNS fan-out to multiple SQS queues (Project 14)
   → Use: Kinesis (multiple consumers read same stream)

3. When you need to replay historical messages
   → SQS: consumed messages are deleted, no replay
   → Use: Kinesis (replay from any point in retention window)
   → Use: S3 + Athena (query historical event data)

4. When you need pub/sub with human subscribers
   → SQS: consumers must poll programmatically
   → Humans don't poll queues — they read email
   → Use: SNS for human notifications (Project 12)

5. When message size exceeds 256 KB
   → SQS max: 256 KB per message
   → Use: SQS Extended Client (stores payload in S3, sends reference)
   → Or: put data in S3, send S3 key as message

6. When you need request/response pattern
   → SQS is one-way: producer → queue → consumer
   → No built-in response channel
   → Use: API Gateway + Lambda for synchronous request/response

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Messages are processed twice"
→ Cause: visibility timeout too short for processing time
→ Fix: set visibility timeout = 5× average processing time
→ Also: make consumer IDEMPOTENT (processing same message twice = same result)
→ Pattern: check "already processed" flag in database before processing

PITFALL 2: "Messages stuck in queue — consumer not receiving"
→ Cause #1: visibility timeout too long — messages invisible from previous crash
→ Debug: check ApproximateNumberOfMessagesNotVisible (in-flight count)
→ Cause #2: consumer polling wrong queue URL
→ Debug: verify queue URL matches exactly

PITFALL 3: "DLQ fills up but nobody notices"
→ Cause: no monitoring on DLQ
→ Fix: CloudWatch alarm on DLQ ApproximateNumberOfMessagesVisible > 0
→ Alert via SNS (Project 12) when poison messages appear

PITFALL 4: "FIFO queue throughput is too low"
→ Cause: all messages in ONE message group → sequential processing
→ Fix: use meaningful MessageGroupId (e.g., order_id, customer_id)
→ Different groups process in PARALLEL
→ 100 groups × 300 msg/sec = 30,000 msg/sec effective throughput

PITFALL 5: "Consumer deletes message but processing actually failed"
→ Cause: delete_message called BEFORE processing completes
→ Pattern: receive → process → delete (in that order, ALWAYS)
→ Never: receive → delete → process

PITFALL 6: "Queue retention expired — messages lost"
→ Cause: messages sat in queue for > retention period (default 4 days)
→ Consumer was down for a week — messages auto-deleted
→ Fix: set retention to 14 days for critical queues
→ Fix: alarm on ApproximateAgeOfOldestMessage > threshold

PITFALL 7: "Message body is truncated"
→ Cause: message exceeds 256 KB limit
→ send_message silently truncates? No — it returns error
→ Fix: use SQS Extended Client (stores in S3) for large messages
→ Or: send reference (S3 key) instead of actual data

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ SQS is a PULL-BASED message queue — consumers read at their own pace
→ Messages PERSIST until consumed + deleted (up to 14 days)
→ STANDARD Queue: unlimited throughput, at-least-once, best-effort ordering
→ FIFO Queue: exactly-once, strict ordering per MessageGroupId, 300 msg/sec
→ VISIBILITY TIMEOUT: hides message during processing
   → Too short: duplicate processing
   → Too long: delayed retry after crash
   → Rule: 5× expected processing time
→ LONG POLLING: WaitTimeSeconds=20 reduces API calls by 90%
→ DEAD LETTER QUEUE: quarantines messages after N failures
   → maxReceiveCount = 3: three strikes and you're out
   → DLQ messages need manual investigation
   → Redrive: move back to main queue after fixing root cause
→ ALWAYS: receive → process → delete (in that order)
→ NEVER: delete before processing completes
