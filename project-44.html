🟡 HOW TO APPROACH IT

Step 1: Understand the Medallion Architecture (Bronze/Silver/Gold)
→ Invented by Databricks, adopted industry-wide
→ Organizes data lake into PROGRESSIVE QUALITY LAYERS

  ┌─────────────────────────────────────────────────────────────────────┐
  │                   MEDALLION ARCHITECTURE                            │
  │                                                                     │
  │  ┌───────────┐      ┌───────────┐      ┌───────────┐              │
  │  │  BRONZE   │─────→│  SILVER   │─────→│   GOLD    │              │
  │  │  (Raw)    │      │ (Cleansed)│      │ (Curated) │              │
  │  └───────────┘      └───────────┘      └───────────┘              │
  │                                                                     │
  │  BRONZE: Exact copy of source data. No transformations.             │
  │  → "What arrived from the source, byte-for-byte"                   │
  │  → CSV, JSON, Parquet — whatever the source produces               │
  │  → Append-only (never delete, never overwrite)                     │
  │  → Used for: replay, audit, debugging, reprocessing                │
  │                                                                     │
  │  SILVER: Cleaned, deduplicated, typed, standardized                │
  │  → Schema enforced (correct data types, nulls handled)             │
  │  → Duplicates removed (CDC overlap rows, retry duplicates)         │
  │  → Format: always Parquet (columnar, compressed)                   │
  │  → Partitioned by date for efficient querying                      │
  │  → Used for: data engineering, ad-hoc analysis, ML feature input   │
  │                                                                     │
  │  GOLD: Business-level aggregations, ready for consumption          │
  │  → Pre-computed metrics (CLV, churn, revenue by segment)           │
  │  → Joined/denormalized for specific use cases                      │
  │  → Data quality validated (no nulls in required fields)            │
  │  → Used for: dashboards, reports, ML models, reverse ETL           │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘

❌ WHAT SHOULD NOT DO

1. DO NOT allow anyone to write directly to Gold
   → Gold is the "single source of truth" for business
   → If analyst writes ad-hoc output to gold/ → untrusted data mixes in
   → Enforce: only approved Glue jobs can write to gold/

2. DO NOT delete Bronze data to save cost (before lifecycle)
   → Bronze is your INSURANCE POLICY
   → If Silver transformation has a bug → replay from Bronze
   → If Redshift data is corrupted → rebuild from Bronze
   → Cost of Bronze storage << cost of data loss

3. DO NOT skip the Silver layer (Bronze → Gold directly)
   → Tempting: "just transform raw to final in one step"
   → Problem: no intermediate checkpoint
   → If Gold transformation fails midway: must re-read raw data
   → Silver provides: checkpointed, typed, deduplicated data
   → Gold builds ON TOP of Silver (faster, cheaper)

4. DO NOT mix formats within a zone
   → Bronze: ANY format (whatever source produces) — OK
   → Silver: MUST be Parquet (no CSV, no JSON)
   → Gold: MUST be Parquet (no exceptions)
   → Mixing formats in Silver/Gold breaks Athena partition scanning

5. DO NOT put system files in data zones
   → glue-temp/ in bronze/ → Crawler discovers temp files as data
   → athena-results/ in silver/ → Query outputs mixed with ETL outputs
   → _system/ prefix with underscore → Glue Crawlers skip it by default

6. DO NOT create deep partition hierarchies
   → year=2025/month=01/day=15/hour=02/minute=30/ → TOO DEEP
   → Athena/Glue must list every level → slow
   → Sweet spot: year=/month=/day=/ (3 levels max for daily data)
   → For hourly: year=/month=/day=/hour=/ (4 levels max)

7. DO NOT let multiple pipelines write to the same Silver/Gold prefix
   → Pipeline A writes orders/ silver at 2:00 AM
   → Pipeline B overwrites orders/ silver at 2:30 AM
   → Race condition → partial data → broken queries
   → Rule: one pipeline owns one prefix, always

🚫 WHEN NOT TO USE MEDALLION ARCHITECTURE

1. When you have a single pipeline with a single consumer
   → Overhead of 3 zones for 1 table read by 1 dashboard = overkill
   → Just use: raw/ and processed/ (2 zones)

2. When all data is already clean from the source
   → API returns perfectly typed JSON → no cleaning needed
   → Bronze and Silver would be identical → skip Silver
   → Go: Bronze (raw JSON) → Gold (aggregated)

3. When data is purely transient (process and discard)
   → Stream processing: Kinesis → Lambda → DynamoDB
   → No need to persist intermediate states
   → Medallion is for PERSISTENT data lake storage

4. When budget is extremely tight and storage costs matter
   → 3 copies of data (Bronze + Silver + Gold) = 3× storage
   → 100 GB raw → ~100 GB Bronze + ~70 GB Silver + ~20 GB Gold = ~190 GB
   → At $0.023/GB = $4.37/month (usually fine, but consider)
   → Mitigation: lifecycle rules move old data to cheaper tiers

5. When team lacks discipline to maintain zone boundaries
   → Medallion requires GOVERNANCE: who writes where, quality gates
   → Without discipline: people write anywhere → back to data swamp
   → Better to start with 2 simple zones than 3 ungoverned zones

│  S3 Lifecycle Configuration (bucket-level):                                     │
│  → Rules evaluated DAILY at midnight UTC                                        │
│  → Each rule has: prefix filter + action + days threshold                       │
│  → Multiple rules can coexist (evaluated independently)                         │

│  {                                                                              │
│    "ID": "bronze-lifecycle",                                                    │
│    "Filter": {"Prefix": "bronze/"},                                             │
│    "Transitions": [                                                             │
│      {"Days": 30, "StorageClass": "STANDARD_IA"},                              │
│      {"Days": 90, "StorageClass": "GLACIER"}                                   │
│    ],                                                                           │
│    "Expiration": {"Days": 365}                                                  │
│  }                                                                              │
│                                                                                 │
│  Internal execution:                                                            │
│  1. S3 maintains a lifecycle evaluation queue (internal)                        │
│  2. Daily: scans all objects matching each rule's prefix                        │
│  3. For each object: compare (today - LastModified) vs Days threshold           │
│  4. If threshold met: schedule transition/deletion                              │
│  5. Transitions happen asynchronously (may take 24-48 hours)                    │
│  6. Object metadata updated: x-amz-storage-class changes                       │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Silver and Bronze have the same data — why bother?"
→ They look similar for simple CSVs but differ in:
   - Format: Bronze=CSV, Silver=Parquet (10× faster queries)
   - Schema: Bronze=untyped strings, Silver=proper types
   - Dedup: Bronze has CDC overlap duplicates, Silver doesn't
   - Partitioning: Bronze=as-received, Silver=standardized year=/month=/day=
→ Silver is OPTIMIZED for consumption, Bronze is RAW for preservation

PITFALL 2: "Glue Crawler discovers _system/ files as data tables"
→ Cause: Crawler scans entire bucket, finds Athena results as "tables"
→ Fix: Use _system/ prefix (underscore prefix)
→ Configure Crawler to exclude: _system/** pattern
→ Or: set Crawler target to specific zone prefix only

PITFALL 3: "Gold table has NULL values — quality gate didn't work"
→ Cause: quality check runs AFTER write (too late)
→ Fix: validate BEFORE writing to Gold
→ Pattern: write to gold_staging/ → validate → rename to gold/
→ Or: use Glue Data Quality (Project 68) with fail-on-violation

PITFALL 4: "Lifecycle rule deleted Silver data that Gold still needs"
→ Cause: Silver lifecycle=180 days, but Gold job references Silver from 200 days ago
→ Fix: Gold jobs should ONLY reference recent Silver partitions
→ Or: extend Silver lifecycle to match Gold's maximum lookback window

PITFALL 5: "Bronze partition has 50,000 tiny files"
→ Cause: CDC runs hourly, each run produces 1 small file
→ 24 files/day × 30 days = 720 files per month per entity
→ Not terrible, but Silver should compact these (Project 62)
→ Fix: Bronze→Silver ETL reads all daily files → writes 4 Parquet files

PITFALL 6: "Team member writes experiment output to gold/"
→ Cause: no write protection on gold/ prefix
→ Fix: IAM policy denies PutObject on gold/* for non-ETL roles
→ Only the Glue ETL role can write to gold/
→ Enforce via: bucket policy + IAM role restrictions

PITFALL 7: "Migration from old bucket breaks Athena queries"
→ Cause: Athena tables still point to old S3 paths
→ Fix: update Glue Catalog table locations after migration
→ Or: use ALTER TABLE SET LOCATION in Athena
→ Plan: migrate one entity at a time, verify, then proceed

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO KNOW YOUR DATA LAKE IS HEALTHY:

1. Zone Inventory Dashboard
   → For each zone: object count, total size, growth rate
   → CloudWatch custom metrics via daily Lambda scan
   → Alert: if bronze/ grows > 2× expected rate (runaway ingestion)
   → Alert: if gold/ has 0 new objects today (pipeline stalled)

2. Data Freshness per Zone
   → Bronze: latest object timestamp per entity
   → Silver: latest partition per entity
   → Gold: latest partition per domain table
   → SLA: Bronze < 1 hour old, Silver < 4 hours, Gold < 6 hours

3. Schema Consistency
   → Silver tables: schema should be stable (columns don't randomly appear/disappear)
   → Glue Crawler schema change detection (Project 45)
   → Alert: if Silver table schema changes unexpectedly

4. Quality Metrics
   → Bronze→Silver: rows_input vs rows_output (should be close)
   → Silver→Gold: quality check pass rate (should be >99%)
   → Null rate per column (track over time, alert on spikes)

5. Cost Attribution
   → S3 tags per zone: Environment, Zone, Entity
   → AWS Cost Explorer: filter by tag → see cost per zone
   → Expected ratio: Bronze > Silver > Gold (by storage volume)
   → If Gold > Silver → something wrong (Gold should be aggregated/smaller)

6. Access Audit
   → CloudTrail: who accessed which zone, when
   → Alert: if BI Analyst role accesses bronze/ (shouldn't need to)
   → Alert: if unknown role writes to gold/ (unauthorized pipeline)
