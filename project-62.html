🟡 HOW TO APPROACH IT

Step 1: Understand WHY tiny files are created

  SPARK/GLUE INTERNAL:
  → Glue uses Apache Spark under the hood
  → Spark distributes data across PARTITIONS (in-memory chunks)
  → Each PARTITION writes to ONE output file
  → More partitions = more parallelism = more output files

  WHERE DO 48,327 PARTITIONS COME FROM?
  → Source: MariaDB table with 200K rows
  → Glue reads via JDBC with hashpartitions=4 → 4 initial partitions
  → Transformations (filter, join, map) cause SHUFFLES
  → Each shuffle can MULTIPLY partition count
  → GroupBy → default 200 shuffle partitions (spark.sql.shuffle.partitions)
  → Join → another 200 shuffle partitions
  → Final write: 200+ partitions → 200+ files PER DAY
  → Over 30 days with multiple runs/retries → 48,327 files

Step 2: Understand WHY tiny files are SLOW

  ATHENA QUERY EXECUTION:
  ┌────────────────────────────────────────────────────────────────────┐
  │ Step 1: LIST objects in partition prefix                          │
  │   → S3 ListObjectsV2: returns 1000/call → 49 API calls           │
  │   → Each call: ~50ms → 49 × 50ms = 2.5 seconds just to LIST     │
  │                                                                    │
  │ Step 2: For each file, send GET request                           │
  │   → 48,327 HTTP GET requests                                      │
  │   → Each GET: ~20ms overhead (TCP handshake + TLS + headers)      │
  │   → Overhead: 48,327 × 20ms = 966 seconds of pure overhead       │
  │   → But Athena parallelizes: ~100 concurrent connections          │
  │   → Actual: 966 / 100 = ~10 seconds overhead                     │
  │                                                                    │
  │ Step 3: Read file content                                         │
  │   → 4 KB per file — actual data read is instant                   │
  │   → But: Parquet has HEADER + FOOTER metadata per file            │
  │   → Header: ~500 bytes, Footer: ~2 KB                             │
  │   → For 4 KB file: 2.5 KB metadata + 1.5 KB actual data          │
  │   → 62% of each file is OVERHEAD, not data                        │
  │                                                                    │
  │ Step 4: Process data                                               │
  │   → 200 MB of actual data: trivial for Athena                     │
  │   → Processing: < 1 second                                        │
  │                                                                    │
  │ TOTAL: 2.5s (LIST) + 10s (GET overhead) + 35s (metadata) ≈ 47s   │
  │ WITHOUT small file problem: 0.1s (LIST) + 0.5s (GET) + 1s = 1.6s │

🔵 WHAT COULD BE DONE

Option A: coalesce(N) — MERGE partitions without shuffle ✅
→ Reduces partition count by COMBINING adjacent partitions
→ NO data movement across executors — just merges locally
→ FAST, no network shuffle
→ BUT: can cause SKEW if input partitions are uneven
→ BEST FOR: reducing from many to few (200 → 2)

Option B: repartition(N) — REDISTRIBUTE with full shuffle ✅
→ Redistributes data evenly across N partitions via SHUFFLE
→ EXPENSIVE: full data movement across network
→ BUT: guarantees EQUAL-SIZED output files
→ BEST FOR: when data is heavily skewed

Option C: repartition(N, "column") — PARTITION BY COLUMN ✅
→ Same as B but partitions by specific column value
→ Output files grouped by column value
→ BEST FOR: when downstream queries filter by that column

Option D: maxRecordsPerFile Spark option ✅
→ spark.conf.set("spark.sql.files.maxRecordsPerFile", 1000000)
→ Limits rows per output file — Spark auto-splits
→ Doesn't control file SIZE directly — row count proxy
→ BEST FOR: when you know rows-to-size ratio

Option E: S3DistCp (EMR utility) — post-hoc compaction
→ Hadoop DistCp variant optimized for S3
→ Merges small files into large ones
→ Requires EMR cluster — adds infrastructure
→ BEST FOR: legacy Hadoop pipelines

Option F: Athena CTAS — one-time restructure
→ CREATE TABLE AS SELECT ... WITH (format='PARQUET', bucketed_by=...)
→ Creates new table with optimal file sizes
→ Already covered in Project 36
→ BEST FOR: one-time fix, not recurring

DECISION: 
→ Primary: coalesce() for daily ETL (Layer 1)
→ Secondary: Dedicated compaction Glue job (Layer 2)
→ Tertiary: Partition redesign (Layer 3)

✅ WHAT SHOULD DO

1. CALCULATE target file count DYNAMICALLY
   → Don't hardcode coalesce(2)
   → Calculate: total_bytes / target_file_size = N files
   → Target: 128 MB per file
   → 200 MB / 128 MB = 2 files → coalesce(2)
   → 5 GB / 128 MB = 40 files → coalesce(40)
   → Adapts as data volume grows

2. USE coalesce() for REDUCING file count (most common case)
   → coalesce(N) merges N adjacent partitions
   → No shuffle = fast
   → Perfect for: 200 partitions → 2 partitions

3. USE repartition() ONLY when data is skewed
   → If coalesce(2) creates one 190 MB file and one 10 MB file
   → repartition(2) creates two ~100 MB files
   → Trade-off: slower (shuffle) but even output

4. Place coalesce() AFTER all transformations, BEFORE write
   → Wrong: read → coalesce → transform → write (transforms re-partition)
   → Right: read → transform → coalesce → write

5. Name compacted files distinctively
   → Prefix: compacted_ or suffix: _merged
   → Helps debugging: which files are original vs compacted

6. VALIDATE after compaction
   → Row count before compaction == Row count after compaction
   → File count reduced
   → Total bytes approximately equal (Parquet overhead changes slightly)

7. DELETE original small files ONLY after validation passes
   → Atomic swap pattern:
     a. Write compacted files to temp prefix
     b. Validate counts match
     c. Delete original files
     d. Move compacted files to original prefix
   → OR: write to new prefix, update Glue Catalog to point there

8. Monitor file sizes with CloudWatch custom metrics
   → Track: avg file size, file count per partition
   → Alert when avg < 10 MB (small file regression)

❌ WHAT SHOULD NOT DO

1. DO NOT coalesce(1) for all tables
   → Single file = zero parallelism for readers
   → Athena can't parallelize across files if there's only 1
   → 5 GB in 1 file: Athena uses 1 reader = slow
   → 5 GB in 40 files: Athena uses 40 readers = fast

2. DO NOT repartition() when coalesce() suffices
   → repartition() triggers full network shuffle
   → For 200 MB: shuffle takes longer than actual write
   → coalesce() does the same job without shuffle overhead
   → Exception: heavily skewed data

3. DO NOT compact while ETL is writing
   → Race condition: compaction reads files being written
   → Result: partial data in compacted output
   → Fix: schedule compaction AFTER ETL completes (use Glue Workflow triggers)

4. DO NOT delete small files before verifying compacted output
   → If compaction fails silently: data loss
   → Always: validate row counts → then delete originals

5. DO NOT ignore the ROOT CAUSE
   → Compaction treats symptoms, not disease
   → If Glue job creates 48K files daily and you compact daily:
     you're wasting compute
   → Fix the write first (coalesce), compact as safety net

6. DO NOT set spark.sql.shuffle.partitions = 1
   → "I'll just make everything go to one partition"
   → Destroys ALL parallelism in your job
   → GroupBy on 1 partition = single-threaded
   → Set it appropriately: 2-10 for small data, 200 for large data

7. DO NOT over-partition your S3 prefix structure
   → year/month/day/hour/minute = 5 levels
   → If you get 100 rows per minute: each partition = 1 tiny file
   → Right-size: year/month/day is usually sufficient
   → Or even year/month for small tables

8. DO NOT compact across partitions
   → silver/orders/year=2025/month=01/ has 31 daily partitions
   → Compacting ALL days into one giant file breaks partition pruning
   → Compact WITHIN each partition separately

🚫 WHEN NOT TO USE

1. DO NOT compact if files are already 64-256 MB
   → You're in the sweet spot — compaction wastes compute
   → Monitor first, act only when avg < 10 MB

2. DO NOT use coalesce() to INCREASE partitions
   → coalesce(100) on 2 partitions → still 2 partitions
   → coalesce ONLY reduces — it cannot increase
   → To increase: use repartition()

3. DO NOT use compaction on data being actively queried
   → Compaction deletes original files + writes new ones
   → Concurrent Athena query may fail mid-read
   → Solution: write to temp path → swap atomically

4. DO NOT compact Glacier/Deep Archive objects
   → Retrieval costs dominate any benefit
   → Compact before transitioning to cold storage

5. DO NOT apply compaction to append-only streaming tables
   without considering data freshness requirements
   → Compaction reads ALL files → writes new ones
   → During compaction: new files land → not included
   → Need watermark strategy (covered in compaction job)

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "I added coalesce(1) but job takes 3x longer"
→ Cause: ALL data funneled through ONE task on ONE executor
→ That single task must process 200 MB sequentially
→ Meanwhile: 3 other executors sit idle
→ Fix: coalesce(2) or coalesce(4) — balance parallelism vs file count

PITFALL 2: "coalesce(2) creates one 190 MB file and one 10 MB file"
→ Cause: input data is SKEWED — some partitions have more rows
→ coalesce merges ADJACENT partitions — doesn't redistribute
→ Fix: use repartition(2) when skew is observed
→ Detection: check file sizes after write → alert if max/min ratio > 3x

PITFALL 3: "I set shuffle.partitions = 2 globally and joins are now slow"
→ Cause: GroupBy/Join now uses only 2 tasks — no parallelism
→ This setting affects ALL shuffles in the job, not just the final write
→ Fix: set shuffle.partitions appropriately for processing (e.g., 20)
→ Then: coalesce(2) ONLY at the final write step

PITFALL 4: "Compaction job reads 48K files and OOMs"
→ Cause: Spark opens ALL files simultaneously → metadata fills memory
→ 48K Parquet footers × ~2 KB each = ~96 MB just for metadata
→ Plus: Spark creates one FileInputSplit per file → task scheduling overhead
→ Fix: increase driver memory, or compact in batches (1000 files at a time)

PITFALL 5: "Compacted files but Athena still queries old files"
→ Cause: Glue Catalog still points to old partition location
→ OR: both old AND new files exist in same prefix → Athena reads ALL
→ Fix: delete old files BEFORE writing new ones (risky)
→ Better: write to new prefix → update catalog → delete old prefix

PITFALL 6: "After compaction, Glue Job Bookmark re-processes all data"
→ Cause: Job Bookmarks (Project 33) track by file path
→ New compacted files = new paths → Bookmark treats them as new
→ Fix: disable bookmarks for compaction job
→ Or: use separate bookmark-tracked job for incremental, separate for compaction

PITFALL 7: "repartition(2, 'status') creates 5 files not 2"
→ Cause: repartition by COLUMN creates one file per DISTINCT VALUE
→ If status has 5 values → 5 files (ignoring the N parameter for column mode)
→ Actually: repartition(2, col) creates 2 files where each file may contain
  multiple status values — but hash distribution may still vary
→ For exact control: coalesce() after repartition()

PITFALL 8: "Compaction runs during ETL window — data corruption"
→ Cause: ETL writes new files while compaction reads
→ Compaction doesn't see files written after it started
→ Those files get deleted when compaction "cleans up"
→ Fix: enforce ordering — ETL completes → compaction starts
→ Use Glue Workflow (Project 35) or Step Functions (Project 55)

PITFALL 9: "I compacted to 2 files but Athena query is STILL slow"
→ Cause: files are 128 MB each BUT partitioned by year/month/day/hour/minute
→ Athena must scan ALL hour/minute partitions → 60×24 = 1440 partition lookups
→ Fix: this is a PARTITION problem, not a file size problem
→ Reduce partition granularity (covered in Layer 3)

PITFALL 10: "Compaction doubled my S3 storage temporarily"
→ Cause: write new files → validate → delete old files
→ During validation: BOTH sets exist → 2x storage
→ For 200 MB: trivial ($0.005 extra)
→ For 5 TB: significant ($115 extra for even 1 hour)
→ Fix: accept temporary cost OR compact partition-by-partition

→ Older partitions already compacted → skip them

🔑 KEY TAKEAWAY

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ Small files are the #1 silent performance killer in data lakes
→ Root cause: Spark creates one file per partition; shuffles create 200 partitions
→ Impact: Athena/Spectrum/Crawler spend 95% of time on overhead, 5% on data
→ coalesce(N) reduces partitions WITHOUT shuffle — fast and effective
→ repartition(N) redistributes EVENLY with shuffle — use for skewed data
→ NEVER set shuffle.partitions = 1 globally — kills processing parallelism
→ Calculate target file count dynamically: data_size_mb / 128 MB
→ Defense in depth: prevent at write + detect with monitoring + remediate with compaction
→ File sizing affects cost too: 48K GET requests vs 2 GET requests per query
→ Parquet metadata overhead dominates in tiny files (62% overhead at 4 KB)
