🟡 HOW TO APPROACH IT

Step 1: Understand the data flow
→ Source: S3 CSV files (known schema from Project 1)
→ Destination: MariaDB tables (schema from Project 9)
→ Frequency: daily (one file per entity per day)
→ Volume: ~200K orders, ~50K customers, ~5K products per day
→ Mode: append (new rows daily, no updates to old data — yet)

Step 2: Understand the loading strategy options
→ Row-by-row INSERT: 200K individual INSERTs → extremely slow (hours)
→ Batch INSERT: INSERT INTO ... VALUES (...), (...), (...) → fast (minutes)
→ LOAD DATA INFILE: MySQL native bulk load → fastest but requires file on server
→ executemany(): pymysql batch parameter binding → good balance

Step 3: Understand idempotency requirements
→ If script runs twice for same file → must NOT duplicate data
→ Options:
  A. DELETE existing data for that date, then INSERT (replace strategy)
  B. Use INSERT IGNORE (skip duplicates based on primary key)
  C. Use INSERT ... ON DUPLICATE KEY UPDATE (upsert)
  D. Check if file already loaded before starting
→ We use: Option A (DELETE + INSERT in transaction) for simplicity
→ Project 42 covers proper CDC/merge patterns

Step 4: Understand error handling needs
→ S3 download failure → retry, don't touch MariaDB
→ CSV parse failure → log bad rows, continue with good ones
→ MariaDB connection failure → retry with backoff
→ Partial load failure → ROLLBACK transaction, no partial data
→ All errors → logged with enough detail to debug

Step 5: Plan the implementation
→ Download CSV from S3 to local temp file
→ Parse CSV, validate structure
→ Connect to MariaDB
→ Begin transaction
→ DELETE existing data for this file's date (idempotency)
→ Batch INSERT new rows
→ COMMIT transaction
→ Verify row count matches
→ Clean up temp file
→ Log result

✅ WHAT SHOULD DO

1. Use TRANSACTIONS for atomicity
   → BEGIN → DELETE old → INSERT new → COMMIT
   → If any step fails → ROLLBACK → no partial data
   → MariaDB default: autocommit=True → must explicitly manage

2. Use batch INSERT (not row-by-row)
   → INSERT INTO orders VALUES (%s,%s,...), (%s,%s,...), ...
   → executemany() with batch_size of 1000-5000 rows
   → 200K rows at batch_size=1000 → 200 INSERTs instead of 200,000

3. Make loading IDEMPOTENT
   → Running same file twice must produce same result
   → DELETE WHERE order_date = '2025-01-15' THEN INSERT
   → Or: track loaded files in a metadata table

4. Validate CSV structure BEFORE loading
   → Check column count matches expected schema
   → Check for empty file (header only, no data)
   → Check for encoding issues (UTF-8 expected)
   → Reject file early rather than fail mid-load

5. Use connection pooling for repeated connections
   → Single script run: one connection is fine
   → Continuous worker (Project 18 pattern): use connection pool
   → pymysql doesn't have built-in pooling → use DBUtils or SQLAlchemy

6. Log everything with timestamps
   → Start time, end time, row counts, file sizes
   → Which file loaded, to which table, how many rows
   → Essential for debugging when something goes wrong at 2 AM

7. Clean up temporary files
   → Download to /tmp → process → delete
   → If script crashes → stale files accumulate
   → Add cleanup in finally block

8. Use parameterized queries (never string formatting)
   → cursor.execute("INSERT INTO t VALUES (%s, %s)", (val1, val2))
   → NEVER: f"INSERT INTO t VALUES ('{val1}', '{val2}')"
   → String formatting → SQL injection vulnerability
   → Parameterized → safe, handles escaping automatically

❌ WHAT SHOULD NOT DO

1. DO NOT use row-by-row INSERT for large files
   → 200K individual INSERTs → 200K round trips to MariaDB
   → Each INSERT: parse SQL + write to WAL + flush
   → Time: 30-60 minutes for 200K rows
   → Batch INSERT: 2-5 minutes for same data (20-30x faster)

2. DO NOT skip transactions
   → Without transaction: if script crashes after 100K rows
   → Table has 100K partial rows — no way to know which are loaded
   → Must delete all and reload from scratch
   → With transaction: ROLLBACK → table unchanged → retry cleanly

3. DO NOT load into production table directly without validation
   → Bad CSV (wrong columns, corrupt data) → breaks production table
   → Strategy: load into STAGING table → validate → swap/merge
   → For simple cases: validate CSV structure before loading

4. DO NOT store MariaDB password in Python source code
   → Use: environment variables, AWS Secrets Manager (Project 70), 
     or EC2 parameter store
   → Here: we use environment variables as minimum acceptable approach

5. DO NOT forget to handle duplicate loads
   → Cron fires twice, SQS message retried, manual re-run
   → Without idempotency: 400K rows instead of 200K
   → Fix: DELETE + INSERT in transaction, or metadata tracking table

6. DO NOT ignore character encoding
   → CSV may contain: café, naïve, résumé, 日本語
   → MariaDB connection charset must match file encoding
   → Default: charset='utf8mb4' (supports all Unicode)
   → Wrong charset → garbled data or insert failures

7. DO NOT load with autocommit=True for bulk operations
   → Each INSERT auto-committed → 200K commits → slow + no rollback
   → Set autocommit=False → explicit COMMIT after all batches
   → Single commit at end → much faster (one WAL flush)

8. DO NOT ignore MariaDB max_allowed_packet
   → Large batch INSERTs generate large SQL statements
   → Default max_allowed_packet: 16 MB
   → 5000 rows × 500 bytes = 2.5 MB per batch → usually fine
   → 50000 rows × 500 bytes = 25 MB → exceeds default → error
   → Keep batch size at 1000-5000 to stay safe

🚫 WHEN NOT TO USE PYMYSQL BATCH INSERT

1. Loading 100M+ rows from CSV
   → pymysql processes in Python → GIL-limited single thread
   → LOAD DATA INFILE: 10-50x faster (MySQL engine reads file directly)
   → Or: split file → parallel workers → each loads a chunk

2. When source is another database (not CSV)
   → Database-to-database: use DMS or Glue JDBC
   → Don't: dump to CSV → upload to S3 → download → load
   → Direct JDBC-to-JDBC is more efficient

3. When destination is a data warehouse (Redshift, BigQuery)
   → These have native bulk loaders (COPY, bq load)
   → pymysql doesn't speak Redshift protocol
   → Use: psycopg2 for Redshift or Redshift COPY (Project 21)

4. When data needs heavy transformation
   → Complex joins, window functions, pivots during load
   → Better: use pandas or Spark for transformation
   → Then load the transformed result

5. When you need exactly-once delivery guarantee
   → pymysql + DELETE/INSERT = idempotent but not exactly-once
   → For financial transactions: use database-native CDC
   → Or: distributed transaction coordinators

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

┌──────────────────────────────────────────────────────────────────────────────┐
│             PYMYSQL + MARIADB BATCH LOADING — INTERNAL MECHANICS             │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  PHASE 1: CONNECTION ESTABLISHMENT                                           │
│  ─────────────────────────────────                                           │
│  pymysql.connect() →                                                         │
│    1. TCP socket connection to MariaDB port (default 3306)                   │
│    2. MySQL protocol handshake (capability negotiation)                      │
│    3. Authentication (username/password or plugin auth)                      │
│    4. Character set negotiation (utf8mb4 recommended)                        │
│    5. Connection ID assigned by MariaDB server                              │
│                                                                              │
│  Connection object holds:                                                    │
│    → Socket handle                                                           │
│    → Server version info                                                     │
│    → Current database context                                                │
│    → Autocommit state                                                        │
│    → Character encoding                                                      │
│                                                                              │
│                                                                              │
│  PHASE 2: TRANSACTION MECHANICS                                              │
│  ──────────────────────────────                                              │
│  conn.autocommit = False                                                     │
│  → Every SQL statement is part of a transaction                             │
│  → Changes are INVISIBLE to other connections until COMMIT                  │
│  → ROLLBACK undoes ALL changes since last COMMIT                            │
│                                                                              │
│  MariaDB InnoDB transaction flow:                                            │
│  ┌─────────────────────────────────────────────────────────────┐            │
│  │ BEGIN (implicit on first statement)                         │            │
│  │   ├── DELETE FROM orders WHERE order_date = '2025-01-15'    │            │
│  │   │   → Rows marked for deletion in undo log               │            │
│  │   │   → Row locks acquired (prevents other transactions)   │            │
│  │   │   → Original data saved in undo tablespace             │            │
│  │   │                                                         │            │
│  │   ├── INSERT INTO orders VALUES (...), (...), (...)         │            │
│  │   │   → New rows written to buffer pool (memory)           │            │
│  │   │   → Redo log entries written (for crash recovery)      │            │
│  │   │   → NOT visible to other connections yet               │            │
│  │   │                                                         │            │
│  │   ├── ... (more batch INSERTs) ...                          │            │
│  │   │                                                         │            │
│  │   └── COMMIT                                                │            │
│  │       → Redo log flushed to disk (durability guarantee)     │            │
│  │       → Changes become visible to all connections           │            │
│  │       → Row locks released                                  │            │
│  │       → Undo log entries can be purged                      │            │
│  │                                                              │            │
│  │   OR: ROLLBACK                                               │            │
│  │       → Undo log replayed (restores original rows)           │            │
│  │       → New rows discarded from buffer pool                  │            │
│  │       → Row locks released                                   │            │
│  │       → Table returns to pre-transaction state               │            │
│  └─────────────────────────────────────────────────────────────┘            │
│                                                                              │
│                                                                              │
│  PHASE 3: BATCH INSERT MECHANICS                                             │
│  ────────────────────────────────                                            │
│                                                                              │
│  cursor.executemany(sql, data_batch) internals:                              │
│                                                                              │
│  WHAT executemany() ACTUALLY DOES:                                           │
│  → pymysql optimizes INSERT INTO t VALUES (%s,%s) with list of tuples       │
│  → Rewrites as: INSERT INTO t VALUES (v1,v2), (v3,v4), (v5,v6)...          │
│  → Single SQL statement with multiple value sets                            │
│  → Sent as ONE network packet to MariaDB                                    │
│                                                                              │
│  BATCH SIZE IMPACT:                                                          │
│  ┌────────────────┬──────────────┬─────────────┬────────────────┐           │
│  │ Batch Size     │ Network Trips│ Time (200K) │ SQL Size       │           │
│  ├────────────────┼──────────────┼─────────────┼────────────────┤           │
│  │ 1 (row-by-row) │ 200,000      │ ~45 min     │ ~200 bytes     │           │
│  │ 100            │ 2,000        │ ~8 min      │ ~20 KB         │           │
│  │ 1,000          │ 200          │ ~3 min      │ ~200 KB        │           │
│  │ 5,000          │ 40           │ ~2 min      │ ~1 MB          │           │
│  │ 10,000         │ 20           │ ~1.5 min    │ ~2 MB          │           │
│  │ 50,000         │ 4            │ ~1 min      │ ~10 MB         │           │
│  └────────────────┴──────────────┴─────────────┴────────────────┘           │
│                                                                              │
│  SWEET SPOT: 1,000 - 5,000 rows per batch                                   │
│  → Good balance of speed and memory usage                                   │
│  → Stays well under max_allowed_packet (16 MB default)                      │
│  → Each batch is one network round trip                                     │
│                                                                              │
│                                                                              │
│  PHASE 4: LOAD DATA LOCAL INFILE (alternative method)                       │
│  ─────────────────────────────────────────────────────                       │
│  LOAD DATA LOCAL INFILE '/tmp/orders.csv'                                   │
│  INTO TABLE orders                                                          │
│  FIELDS TERMINATED BY ','                                                   │
│  ENCLOSED BY '"'                                                            │
│  LINES TERMINATED BY '\n'                                                   │
│  IGNORE 1 LINES;                                                            │
│                                                                              │
│  HOW IT WORKS:                                                               │
│  → Client reads file from LOCAL filesystem (not server filesystem)          │
│  → Streams file content over MySQL protocol to server                       │
│  → Server parses and inserts directly into table                            │
│  → Bypasses SQL parser for each row → much faster                           │
│  → 200K rows: ~30 seconds vs ~3 minutes for batch INSERT                   │
│                                                                              │
│  REQUIREMENTS:                                                               │
│  → pymysql.connect(local_infile=True)                                       │
│  → MariaDB server: SET GLOBAL local_infile = 1;                             │
│  → Security risk: client can read any local file server allows              │
│                                                                              │
│                                                                              │
│  PHASE 5: IDEMPOTENCY VIA DELETE + INSERT                                   │
│  ─────────────────────────────────────────                                   │
│                                                                              │
│  WHY DELETE + INSERT (not just INSERT):                                      │
│  → File may be reprocessed (SQS retry, manual re-run, cron duplicate)       │
│  → INSERT alone → duplicate rows → wrong aggregations                       │
│  → DELETE matching rows first → then INSERT → same result every time        │
│                                                                              │
│  Transaction guarantees:                                                     │
│  → DELETE + INSERT in same transaction                                      │
│  → If INSERT fails → ROLLBACK → DELETE is also undone                       │
│  → Table never in inconsistent state                                        │
│  → Other connections see old data until COMMIT (isolation)                  │
│                                                                              │
│  Alternative: INSERT ... ON DUPLICATE KEY UPDATE                            │
│  → Requires unique key (e.g., order_id)                                     │
│  → Slower than DELETE + INSERT for full file reload                         │
│  → Better for incremental updates (Project 42)                             │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "pymysql.err.OperationalError: (1040, 'Too many connections')"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: script opens connections but doesn't close them
→ Each run leaks a connection → MariaDB max (default 151) exhausted
→ Fix: always use try/finally or context manager to close connections
→ Debug: SHOW PROCESSLIST; in MariaDB to see open connections

PITFALL 2: "Data truncated for column 'unit_price' at row 1"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: CSV has value like "29.99" but column is INT not DECIMAL
→ Or: value "N/A" in numeric column
→ Fix: match MariaDB column types to actual CSV data
→ Fix: clean data in Python before inserting (convert "N/A" to NULL)

PITFALL 3: "Duplicate entry 'ORD-20250115-0001' for key 'PRIMARY'"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: script ran twice for same file without DELETE first
→ Fix: DELETE WHERE order_date = X before INSERT (our approach)
→ Or: INSERT IGNORE (silently skip duplicates)
→ Or: INSERT ... ON DUPLICATE KEY UPDATE (upsert)

PITFALL 4: "Lock wait timeout exceeded; try restarting transaction"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: another process holds locks on rows we're trying to DELETE
→ Common when: app queries running while bulk load happens
→ Fix: schedule loads during low-traffic periods (midnight)
→ Fix: use shorter transactions (commit every N batches)
→ Fix: increase innodb_lock_wait_timeout

PITFALL 5: "Lost connection to MySQL server during query"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause A: max_allowed_packet exceeded (batch too large)
→ Cause B: wait_timeout exceeded (connection idle too long)
→ Cause C: network hiccup between EC2 and MariaDB
→ Fix A: reduce batch size
→ Fix B: increase wait_timeout or add keepalive pings
→ Fix C: implement retry with exponential backoff

PITFALL 6: "CSV has 9 columns but table has 8"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Source system added a new column without telling data team
→ Script crashes on first INSERT
→ Fix: validate column count before loading
→ Fix: schema evolution handling (Project 45)

PITFALL 7: "Loaded 200K rows but table shows 400K"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: script ran twice (cron fired twice, manual re-run)
→ DELETE step was not implemented → data doubled
→ Fix: always DELETE matching date range before INSERT
→ Fix: add load_metadata table tracking which files were loaded

📈 SCALABILITY CONSIDERATIONS

DATA VOLUME SCALING:
━━━━━━━━━━━━━━━━━━━
→ 5K rows (products): < 1 second
→ 50K rows (customers): 5-10 seconds
→ 200K rows (orders): 2-5 minutes with batch INSERT
→ 1M rows: 10-15 minutes with batch INSERT
→ 10M rows: consider LOAD DATA INFILE (10x faster)
→ 100M rows: split file + parallel loaders + partitioned table

MARIADB TABLE SCALING:
━━━━━━━━━━━━━━━━━━━━━━
→ Under 10M rows: standard InnoDB table works great
→ 10M-100M rows: add proper indexes, optimize queries
→ 100M+ rows: consider table partitioning (by date)
→ 1B+ rows: MariaDB is wrong tool → use Redshift (Project 10)

CONNECTION SCALING:
━━━━━━━━━━━━━━━━━━
→ Single loader script: 1 connection is fine
→ Multiple parallel loaders: connection pool (5-10 connections)
→ MariaDB default max_connections: 151
→ Don't open more than you need → resource exhaustion

NETWORK SCALING:
━━━━━━━━━━━━━━━━
→ EC2 and MariaDB in same VPC: ~0.1ms latency
→ EC2 and MariaDB in different regions: 50-200ms latency
→ Each batch INSERT = 1 round trip
→ 200 batches × 0.1ms = 0.02s overhead (same VPC — negligible)
→ 200 batches × 100ms = 20s overhead (cross-region — significant)
→ Always co-locate compute and database

COST:
━━━━━

HOW TO VERIFY LOADING WORKED CORRECTLY:

1. Row count comparison
   → CSV rows (minus header) must equal MariaDB SELECT COUNT(*)
   → Mismatch → some rows rejected or duplicated
   → Script logs both counts for comparison

2. Spot-check sample rows
   → SELECT * FROM orders WHERE order_date = '2025-01-15' LIMIT 5
   → Compare with first 5 data rows of CSV
   → Check: values match, no truncation, no encoding issues

3. Aggregate comparison
   → CSV: sum of total_amount column (Python calculation)
   → MariaDB: SELECT SUM(total_amount) WHERE order_date = '2025-01-15'
   → Must match (or differ by < $0.01 due to floating point)

4. Load metadata table
   → Track: filename, load_time, row_count, status
   → Query: SELECT * FROM load_metadata ORDER BY load_time DESC
   → Shows full history of what was loaded when

5. MariaDB binary log
   → Contains every INSERT statement executed
   → Can reconstruct exactly what was loaded
   → Useful for: audit, debugging, replication

🔑 KEY TAKEAWAY

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ pymysql batch INSERT is the standard Python→MariaDB loading pattern
→ BATCH_SIZE of 1000-5000 rows balances speed and memory
→ executemany() rewrites multiple VALUES into single INSERT statement
→ Transactions (autocommit=False) give atomicity — ROLLBACK on failure
→ Idempotency via DELETE + INSERT in same transaction
→ load_metadata table tracks what was loaded for audit and dedup
→ CSV validation BEFORE loading prevents corrupting production tables
→ clean_row() handles: empty strings→NULL, type conversion, booleans
→ Connection must ALWAYS be closed (try/finally pattern)
→ Parameterized queries (%s) prevent SQL injection — NEVER use f-strings

KEY DESIGN DECISIONS:
━━━━━━━━━━━━━━━━━━━━
→ DELETE + INSERT (not UPSERT) — simplest for full file reload
→ One connection per load (not pool) — sufficient for single script
→ Local temp file (not streaming) — allows validation before load
→ Metadata table — enables "was this file already loaded?" check
→ Per-entity config dict — extensible to new tables without code change
