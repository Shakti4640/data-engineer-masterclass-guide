🟡 HOW TO APPROACH IT

Step 1: Understand the Two Lake Formation Permission Models
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

MODEL A: Named Resource (what we used in Project 73)
→ GRANT SELECT ON TABLE orders_gold.daily_orders TO role/AnalystRole
→ Explicit: per-table, per-role, per-column
→ Problem: N tables × M roles = N×M grants to maintain
→ Problem: new table → must add grants manually

MODEL B: LF-Tag-Based (what we need now) ✅
→ Tag columns: sensitivity=confidential
→ Tag roles: clearance=internal
→ GRANT: IF tag(sensitivity) IN (public, internal) → allow SELECT
→ Automatic: new table with tagged columns → instantly governed
→ Scales: 1 tag expression covers ALL tables, ALL accounts

Step 2: Design the Tag Taxonomy
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Tag 1: sensitivity
→ Values: public, internal, confidential, restricted
→ Applied to: columns

Tag 2: domain
→ Values: orders, customers, products, analytics
→ Applied to: databases, tables

Tag 3: layer
→ Values: bronze, silver, gold, derived
→ Applied to: databases

Tag 4: pii
→ Values: true, false
→ Applied to: columns
→ Shortcut tag for GDPR/CCPA compliance

🔵 WHAT COULD BE DONE

OPTION A: Named Resource Grants with Column Exclusion (Project 73)
→ Per-table ExcludedColumnNames grants
→ 200 tables × 6 roles = 1,200 grants
→ New table → must manually add 6 grants
→ ❌ REJECTED — doesn't scale, error-prone

OPTION B: LF-Tags (Tag-Based Access Control) ✅
→ Define tags once, assign to columns
→ Define tag expressions per role once
→ New table → tag columns → auto-governed
→ Cross-account → share tags → auto-enforced
→ ✅ SELECTED — scales to any number of tables

OPTION C: Column-Level Encryption (Client-Side)
→ Encrypt PII columns with per-role KMS keys
→ Only authorized roles can decrypt
→ Problem: requires application-level changes
→ Problem: Athena/Spectrum can't decrypt inline
→ ❌ REJECTED — too complex, breaks query tools

OPTION D: Row-Level Security (Redshift RLS)
→ Redshift native row-level security
→ Only works for Redshift internal tables
→ Doesn't cover Athena or S3
→ ❌ REJECTED — doesn't cover the full platform

DECISION: Option B (LF-Tags) as primary + Option D as supplementary for Redshift

✅ WHAT SHOULD DO

1. DEFINE TAGS IN PRODUCER ACCOUNT
   → Tags belong to the account that creates them
   → Other accounts receive tags via cross-account sharing
   → Central governance account could own tags instead (alternative)

2. TAG EVERY COLUMN IN GOLD TABLES
   → sensitivity tag: public/internal/confidential/restricted
   → pii tag: true/false
   → Every column MUST have both tags — no untagged columns
   → Untagged columns default to MOST restrictive (blocked)

3. USE TAG EXPRESSIONS FOR GRANTS (not individual tags)
   → Expression: sensitivity IN (public, internal)
   → This covers ALL tables, ALL databases, ALL columns with matching tags
   → One expression per role — not per table

4. SHARE TAGS CROSS-ACCOUNT BEFORE SHARING DATA
   → Lake Formation cross-account requires:
     a) Share LF-Tags to consumer account
     b) Share table/database to consumer account
     c) Consumer creates resource link
     d) Consumer grants to local roles using shared tags

5. AUTOMATE TAG ASSIGNMENT
   → Use Glue Catalog table parameters as source of truth
   → Script reads column metadata → applies tags
   → Runs after every new table creation
   → Prevents human error in tag assignment

6. AUDIT QUARTERLY
   → Export all LF grants to CSV
   → Cross-reference with HR role list
   → Flag: grants for departed employees
   → Flag: untagged columns in Gold tables

❌ WHAT SHOULD NOT DO

1. DO NOT use LF-Tags AND Named Resource grants on same table
   → They can coexist but create confusion
   → If both exist: UNION of permissions (most permissive wins)
   → Pick ONE model per table — preferably LF-Tags for all

2. DO NOT create too many tag values
   → 4 sensitivity levels is optimal
   → 10+ levels creates unmanageable grant matrix
   → Keep it simple: public, internal, confidential, restricted

3. DO NOT leave columns untagged
   → Untagged columns are governed by table-level grant
   → If table has LF-Tag grant, untagged columns may be VISIBLE
   → Always explicitly tag every column

4. DO NOT forget to share tags BEFORE sharing tables cross-account
   → If consumer doesn't have the tag definition
   → Tag-based grants fail silently
   → Consumer sees Access Denied with no useful error

5. DO NOT grant tag permissions to IAMAllowedPrincipals
   → Defeats the entire purpose
   → Verify IAMAllowedPrincipals is removed (Project 73 Step 2)

6. DO NOT assume Spectrum honors LF-Tags automatically
   → Spectrum uses IAM role → role must be granted via LF-Tags
   → Spectrum role must have lakeformation:GetDataAccess
   → Test column visibility from Spectrum separately

🚫 WHEN NOT TO USE LF-TAGS

1. SIMPLE USE CASE — 5 tables, 2 roles
   → LF-Tags overhead not justified
   → Named Resource grants are clearer
   → Don't over-engineer for small catalogs

2. NEED ROW-LEVEL FILTERING
   → LF-Tags control COLUMN visibility
   → For ROW filtering: use Lake Formation Data Filters
   → Or: create views that filter rows per role

3. NON-GLUE-CATALOG TABLES
   → LF-Tags only work with Glue Data Catalog tables
   → Redshift internal tables: use Redshift native GRANT/RLS
   → DynamoDB: use IAM fine-grained access control

4. REAL-TIME PERMISSION CHANGES
   → LF-Tag changes can take 1-5 minutes to propagate
   → For instant revocation: use IAM deny policy (immediate)
   → LF-Tags are eventually consistent for new grants

📈 SCALABILITY CONSIDERATIONS

TAG LIMITS:
→ Max LF-Tags per account: 50
→ Max values per tag: 50
→ Max tag assignments per resource: 50
→ For QuickCart: 4 tags × 4-5 values = well within limits

GRANT SCALING:
→ Named Resource: O(tables × roles) grants
→ LF-Tags: O(tag_expressions × roles) grants
→ 200 tables, 6 roles: Named = 1,200 grants, Tags = 6 grants
→ 1,000 tables, 20 roles: Named = 20,000 grants, Tags = 20 grants
→ LF-Tags are the ONLY viable option at scale

CROSS-ACCOUNT SCALING:
→ Each consumer account needs: tag share + data share + resource link
→ 10 consumer accounts: 10 shares (manageable)
→ 50 consumer accounts: need automation (Terraform/CDK)
→ Tag grants propagate to ALL consumers — add once, works everywhere

PERFORMANCE:
→ LF-Tag evaluation: < 50ms per query (negligible)
→ First query after grant change: may take 1-5 minutes (cache refresh)
→ Subsequent queries: instant (cached permission evaluation)
→ No performance difference between 4 tags and 40 tags
