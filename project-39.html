🔵 WHAT COULD BE DONE

Option A: Add DISTKEY and SORTKEY to existing tables (ALTER TABLE)
→ Redshift supports ALTER TABLE ... ALTER DISTSTYLE
→ Redshift supports ALTER TABLE ... ALTER SORTKEY
→ Pros: no data movement, no downtime
→ Cons: slow for large tables (rebuilds internally), limited options

Option B: Create new optimized tables, migrate data (DEEP COPY)
→ CREATE new_table (DISTKEY, SORTKEY) → INSERT INTO new_table SELECT * FROM old
→ Rename: old → old_backup, new → original name
→ Pros: clean table, optimal encoding, fresh zone maps
→ Cons: requires downtime (brief rename window), temporary 2x storage

Option C: Use CTAS for migration (CREATE TABLE AS)
→ CREATE TABLE new_fact_orders DISTKEY(customer_id) SORTKEY(order_date)
   AS SELECT * FROM fact_orders
→ Similar to Option B but single statement
→ Pros: simpler syntax, automatic encoding
→ Cons: same downtime concern

Option D: Alter existing + VACUUM FULL + ANALYZE (minimal change)
→ ALTER DISTSTYLE + ALTER SORTKEY on existing tables
→ Run VACUUM FULL to reclaim space and resort
→ Run ANALYZE to refresh statistics
→ Pros: least disruption
→ Cons: ALTER DISTSTYLE on 45M rows takes time

DECISION: Option C (CTAS deep copy) for initial optimization
→ Cleanest result — fresh table with optimal design
→ Then: ongoing VACUUM + ANALYZE maintenance
→ Schedule maintenance after nightly load completes

✅ WHAT SHOULD DO

1. Choose DISTKEY based on most common JOIN column
   → fact_orders JOINs with dim_customers ON customer_id → DISTKEY(customer_id)
   → dim_customers JOINs with fact_orders ON customer_id → DISTKEY(customer_id)
   → MATCHING DISTKEY = co-located join = NO network shuffle
   → This is the SINGLE BIGGEST performance win

2. Choose SORTKEY based on most common WHERE filter column
   → Dashboards filter: WHERE order_date > CURRENT_DATE - 90
   → SORTKEY(order_date) → zone maps know min/max per block
   → Blocks outside date range SKIPPED entirely
   → 90-day filter on 2-year table → skip 82% of blocks

3. Use DISTSTYLE ALL for small dimension tables
   → dim_products: 50K rows × 200 bytes = 10 MB
   → DISTSTYLE ALL: copy to EVERY node
   → JOIN with fact_orders: no network transfer (product data local)
   → Rule: ALL for tables < 5M rows that are frequently joined

4. Run VACUUM after every nightly merge load
   → Project 32 does DELETE + INSERT (staging merge)
   → Deleted rows become ghost rows
   → VACUUM reclaims space + resorts data
   → Schedule: VACUUM FULL fact_orders; after load completes

5. Run ANALYZE after VACUUM
   → ANALYZE updates statistics used by query optimizer
   → Stale statistics → optimizer chooses wrong join strategy
   → Schedule: ANALYZE fact_orders; after VACUUM

6. Use COMPOUND SORTKEY for multiple filter columns
   → COMPOUND SORTKEY(order_date, status)
   → First column gets most benefit
   → Useful when queries filter by date AND status

7. Use EXPLAIN to verify optimization worked
   → Before: DS_DIST_ALL_NONE or DS_DIST_BOTH → network shuffle
   → After: DS_DIST_NONE → co-located join (no shuffle)
   → Before: seq scan → full table scan
   → After: range-restricted scan → zone map pruning

❌ WHAT SHOULD NOT DO

1. DO NOT use DISTKEY on a high-cardinality unique column
   → DISTKEY(order_id) → 45M unique values → even distribution BUT
   → No other table joins on order_id → no co-location benefit
   → DISTKEY should be a JOIN column, not a unique identifier

2. DO NOT use DISTKEY on a skewed column
   → If 80% of orders have customer_id = 'GUEST'
   → DISTKEY(customer_id) → 80% of rows on one slice
   → One slice overloaded, others idle → worse than EVEN
   → Check: SELECT customer_id, COUNT(*) GROUP BY 1 ORDER BY 2 DESC LIMIT 10
   → If top value > 5% of total → consider different DISTKEY

3. DO NOT use DISTSTYLE ALL for large tables
   → ALL copies entire table to every node
   → 45M row fact table × 4 nodes = 180M rows stored
   → 4x storage cost, 4x slower COPY loads
   → ALL only for small tables (< 5M rows)

4. DO NOT skip VACUUM indefinitely
   → Every DELETE + INSERT creates ghost rows
   → After 30 days without VACUUM: table could be 50%+ bloat
   → Queries scan ghost rows → 2x slower
   → Schedule VACUUM weekly at minimum, daily for heavy-write tables

5. DO NOT run VACUUM during peak query hours
   → VACUUM FULL acquires exclusive lock briefly
   → Dashboard queries may queue behind VACUUM
   → Schedule VACUUM at 3-4 AM (after load, before business hours)

6. DO NOT set SORTKEY on a column you never filter by
   → SORTKEY(product_name) → useless if no query filters by name
   → SORTKEY(order_date) → valuable because 90% of queries filter by date
   → SORTKEY adds insert overhead → only use on high-value filter columns

7. DO NOT assume ANALYZE runs automatically
   → Redshift auto-analyzes AFTER COPY but NOT after INSERT...SELECT
   → Project 32 uses Glue write → Redshift COPY internally
   → But staging merge uses INSERT...SELECT → no auto-analyze
   → Always run ANALYZE explicitly after merge operations

8. DO NOT change DISTKEY/SORTKEY on production tables during business hours
   → ALTER DISTSTYLE on 45M rows takes 10-30 minutes
   → Table partially unavailable during redistribution
   → Do it during maintenance window (weekend or 3 AM)

→ Cause B: Data skew — one customer_id has 40% of orders
→ One slice has 40% of data → overloaded
→ Check: SELECT customer_id, COUNT(*) FROM fact_orders GROUP BY 1 ORDER BY 2 DESC LIMIT 5
→ Fix: Choose a more evenly distributed DISTKEY column

PITFALL 2: "SORTKEY set but zone maps not helping"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Data loaded after SORTKEY change is UNSORTED
→ New data appended at end → not in sort order
→ VACUUM needed to re-sort the unsorted region
→ Check: SELECT "table", unsorted FROM svv_table_info WHERE "table" = 'fact_orders'
→ If unsorted > 20% → run VACUUM SORT ONLY fact_orders

PITFALL 3: "VACUUM runs for 6 hours and blocks queries"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: VACUUM FULL on 90M row table (50% bloat) takes hours
→ During VACUUM: table partially locked → queries may slow down
→ Fix: Run VACUUM more frequently (daily) so less work each time
→ Fix: Use VACUUM DELETE ONLY (faster — only removes ghosts, no resort)
→ Then: VACUUM SORT ONLY (resort without delete phase)
→ Split into two smaller operations

PITFALL 4: "After CTAS deep copy, old views break"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: CTAS creates new table with different OID
→ Views reference table by OID, not name
→ Rename new table → views still point to old OID
→ Fix: DROP and recreate views after table swap
→ Or: Use ALTER TABLE instead of CTAS (preserves OID)

PITFALL 5: "ANALYZE says 'Table has not been modified since last ANALYZE'"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Redshift tracks modification flag per table
→ After COPY: flag set → ANALYZE runs
→ After INSERT...SELECT: flag may not set in all cases
→ Fix: Use ANALYZE table_name PREDICATE COLUMNS to force

PITFALL 6: "Query slower AFTER adding SORTKEY"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Query doesn't filter on SORTKEY column
→ SORTKEY adds insert overhead but no scan benefit without filtering
→ GROUP BY customer_city → SORTKEY(order_date) doesn't help
→ Fix: Only SORTKEY on columns that appear in WHERE clauses

OPTIMIZATION DECISION CHEAT SHEET:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  "Which DISTKEY should I use?"
  → What's the most common JOIN between your fact and dimension?
  → fact_orders JOIN dim_customers ON customer_id → DISTKEY(customer_id) on BOTH
  → Can only have ONE DISTKEY per table → choose the most important JOIN

  "Which SORTKEY should I use?"
  → What column appears in WHERE clause most often?
  → WHERE order_date > ... → SORTKEY(order_date)
  → Two columns? → COMPOUND SORTKEY(order_date, status)
  → First column gets 90% of benefit

  "EVEN, KEY, or ALL?"
  → Table < 5M rows AND frequently joined → ALL
  → Table > 5M rows AND frequently joined → KEY (DISTKEY on join column)
  → Table rarely joined → EVEN (default, simplest)
  → Staging/temp tables → EVEN (write-optimized)

  "How often should I VACUUM?"
  → DELETE + INSERT every night → VACUUM DELETE ONLY daily
  → Lots of new data → VACUUM SORT ONLY weekly
  → Heavy maintenance → VACUUM FULL monthly

  "How often should I ANALYZE?"
  → After every VACUUM
  → After every COPY load
  → After every INSERT...SELECT merge
  → When in doubt → ANALYZE (it's fast and never hurts)

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ DISTKEY co-location eliminates JOIN shuffles (20x speedup on joins)
→ SORTKEY enables zone map skip-scanning (19x speedup on filtered queries)
→ DISTSTYLE ALL for small dimensions → every join is local (zero shuffle)
→ VACUUM removes ghost rows from DELETE+INSERT pattern (2x improvement)
→ ANALYZE refreshes statistics → optimizer chooses correct join strategy
→ COMPOUND SORTKEY(col1, col2) benefits first column most
→ DISTKEY must MATCH on BOTH sides of a JOIN for co-location
→ Skewed DISTKEY is worse than EVEN → check data distribution first
→ CTAS deep copy is cleanest optimization path (fresh encoding + zone maps)
→ Views must be recreated after table rename (reference by OID)
→ Daily VACUUM DELETE + ANALYZE is essential for DELETE+INSERT tables
→ Weekly VACUUM SORT maintains zone map effectiveness
→ EXPLAIN reveals: DS_DIST_NONE (good) vs DS_DIST_ALL_NONE (shuffle)
→ Disable result cache during benchmarking for honest measurements
