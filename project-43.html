Step 1: Understand REVERSE ETL
→ Traditional ETL: Operational DB → Warehouse (data IN)
→ Reverse ETL: Warehouse → Operational DB (insights OUT)
→ Why it exists: analytics creates VALUE, but value must reach
  the APPLICATION where users interact

🔵 WHAT COULD BE DONE

Option A: Direct Redshift → MariaDB via Glue JDBC (no S3 intermediate)
→ Glue reads from Redshift JDBC, writes to MariaDB JDBC
→ Problem: Glue holds Redshift connection for entire job duration
→ Redshift has limited connections (500 max)
→ Large result sets (2.5M rows) can timeout the JDBC connection
→ No checkpoint/resume if job fails midway
→ VERDICT: Works for small tables, risky for large ones

Option B: Redshift UNLOAD → S3 → EC2 script → MariaDB LOAD DATA
→ UNLOAD is fast and decouples Redshift from downstream
→ EC2 script downloads CSV and runs LOAD DATA INFILE
→ Problem: custom code, no managed retries, EC2 must be running
→ VERDICT: Works but not cloud-native

Option C: Redshift UNLOAD → S3 → Glue ETL → MariaDB JDBC
→ ✅ UNLOAD decouples Redshift (fast, parallel files)
→ ✅ S3 provides durable intermediate storage (checkpoint)
→ ✅ Glue provides managed compute with JDBC write capability
→ ✅ Fits into existing Step Functions pipeline (Project 41)
→ VERDICT: Best — decoupled, managed, auditable

Option D: Use a Reverse ETL SaaS (Census, Hightouch, etc.)
→ Commercial tools designed exactly for this
→ Problem: cost ($500+/month), vendor lock-in, team must learn new tool
→ VERDICT: Good for large orgs, overkill for QuickCart's 3 tables

DECISION: Option C — UNLOAD → S3 → Glue → MariaDB

✅ WHAT SHOULD DO

1. UNLOAD with Parquet format (not CSV)
   → Parquet preserves data types (decimals, timestamps)
   → CSV causes type ambiguity (is "0.12" a string or float?)
   → Glue reads Parquet natively with correct types
   → Parquet is smaller (compression) → faster S3 transfer

2. Use PARALLEL OFF for UNLOAD (for this use case)
   → Default UNLOAD: splits output into multiple files (parallel)
   → Multiple files = harder to track "is UNLOAD done?"
   → PARALLEL OFF: single file output
   → For 500K rows: single file is fine (< 50 MB)
   → For 2.5M rows: use PARALLEL ON with manifest file

3. Create MariaDB target tables with proper indexes FIRST
   → customer_id index for app reads
   → composite index (customer_id, rank) for recommendations
   → Without index: app reads go from <5ms to >500ms

4. Use table swap (RENAME) for full-replace tables
   → TRUNCATE + INSERT = table empty for seconds during load
   → App reads during this window see NO recommendations
   → RENAME: atomic, <1ms, app always sees complete data

5. Use batch UPSERT for incremental tables
   → INSERT ... ON DUPLICATE KEY UPDATE (MariaDB syntax)
   → Batch size: 1000–5000 rows per INSERT statement
   → Too small: too many round trips to MariaDB
   → Too large: single INSERT takes too long, blocks replication

6. Add computed_at timestamp to all reverse ETL tables
   → Tells the app: "these recommendations were computed at 2:00 AM"
   → Enables freshness monitoring
   → If computed_at is 48+ hours old → pipeline is broken → alert

7. Run reverse ETL AFTER forward CDC completes
   → Forward: MariaDB → Redshift (Project 42) at 00:05
   → Aggregation: Redshift queries at 00:30
   → Reverse: Redshift → MariaDB (this project) at 01:00
   → Ensures aggregations use latest data

❌ WHAT SHOULD NOT DO

1. DO NOT write directly from Redshift to MariaDB via dblink/federation
   → Redshift has no outbound JDBC connection capability
   → Redshift is a SINK, not a source for JDBC writes
   → Always go through S3 as intermediate

2. DO NOT TRUNCATE the production table before loading
   → TRUNCATE customer_recommendations → table is EMPTY
   → Load takes 2 minutes → app shows "no recommendations" for 2 minutes
   → Users see empty "For You" section → support tickets flood in
   → Use RENAME pattern instead (atomic swap)

3. DO NOT write 2.5M rows one-at-a-time via JDBC
   → Single INSERT per row: 2.5M network round trips
   → Time: ~45 minutes (unbearably slow)
   → Batch INSERT (1000 rows per statement): ~90 seconds
   → LOAD DATA INFILE: ~30 seconds (fastest for MariaDB)

4. DO NOT skip type validation between Redshift and MariaDB
   → Redshift DECIMAL(10,2) → MariaDB DECIMAL(10,2): works
   → Redshift FLOAT → MariaDB DECIMAL: precision loss!
   → Redshift TIMESTAMPTZ → MariaDB TIMESTAMP: timezone lost!
   → Validate types BEFORE production deployment

5. DO NOT hold a single MariaDB transaction for 2.5M row insert
   → Huge transaction → locks table → app reads blocked
   → MariaDB InnoDB redo log fills up → performance cliff
   → Use batched commits: INSERT 5000 rows → COMMIT → repeat

6. DO NOT schedule reverse ETL BEFORE aggregation queries finish
   → If aggregation takes 25 minutes and reverse ETL starts at minute 20
   → Reverse ETL reads stale/incomplete aggregation results
   → Use Step Functions to chain: aggregate → UNLOAD → reverse ETL

7. DO NOT forget to monitor MariaDB disk space
   → 2.5M recommendation rows + 500K metric rows = ~500 MB
   → If MariaDB is on a small EBS volume → disk full → crash
   → Table swap temporarily doubles space (old + new table exist)

🚫 WHEN NOT TO USE REVERSE ETL VIA S3

1. When you need real-time sync (< 1 minute latency)
   → UNLOAD + Glue = minimum 5-10 minutes end-to-end
   → For real-time: use Redshift streaming or Lambda trigger
   → Or: compute in MariaDB directly (skip warehouse round trip)

2. When the data is already in S3 and doesn't need Redshift
   → If source is S3 Parquet, skip Redshift entirely
   → Glue can read S3 and write to MariaDB directly
   → Adding Redshift in the middle just for a pass-through is waste

3. When the target is NOT a relational database
   → If target is DynamoDB: use Glue DynamoDB connector (not JDBC)
   → If target is Elasticsearch: use different connector
   → JDBC write pattern is specific to RDBMS targets

4. When data volume exceeds MariaDB capacity
   → 100M+ rows → MariaDB struggles with bulk loads
   → Consider: Aurora MySQL (managed, scales better)
   → Or: DynamoDB (if access pattern is key-value)

5. When security policy forbids warehouse data in operational DB
   → Some compliance frameworks (PCI, HIPAA) restrict
     where aggregated data can be stored
   → Verify with security team before implementing reverse ETL

┌─────────────────────────────────────────────────────────────────────────────────┐
│                    INTERNAL MECHANICS: REVERSE ETL PIPELINE                      │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  ❶ Redshift UNLOAD Mechanics (Recap from Project 22 + new details)             │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━             │
│                                                                                 │
│  UNLOAD ('SELECT ...') TO 's3://bucket/prefix/'                                │
│  IAM_ROLE 'arn:aws:iam::...:role/...'                                          │
│  FORMAT PARQUET                                                                 │
│  ALLOWOVERWRITE;                                                                │
│                                                                                 │
│  Internal execution:                                                            │
│  1. Redshift executes the SELECT query on compute nodes                         │
│  2. Each compute node slice writes its portion to S3 IN PARALLEL               │
│  3. Default: creates one file per slice (e.g., 8 slices = 8 files)             │
│  4. PARALLEL OFF: leader node merges all slices into single file               │
│     → Slower but simpler for downstream consumption                            │
│  5. Files written to S3 with sequential naming:                                │
│     prefix/0000_part_00 , prefix/0001_part_00, etc.                            │
│  6. UNLOAD is READ-ONLY — does not modify Redshift data                        │
│  7. UNLOAD generates a manifest file (optional) listing all output files       │
│                                                                                 │
│  PARQUET vs CSV for reverse ETL:                                                │
│  ┌────────────────┬────────────────────────────┬─────────────────────────┐      │
│  │ Feature        │ Parquet                    │ CSV                     │      │
│  ├────────────────┼────────────────────────────┼─────────────────────────┤      │
│  │ Type safety    │ Schema embedded in file    │ Everything is string    │      │
│  │ Size (500K)    │ ~15 MB                     │ ~45 MB                  │      │
│  │ Decimal prec.  │ Preserved exactly          │ May lose trailing zeros │      │
│  │ Null handling  │ Native null support        │ Empty string ambiguity  │      │
│  │ Glue read      │ Native DynamicFrame        │ Needs CSV SerDe config  │      │
│  └────────────────┴────────────────────────────┴─────────────────────────┘      │
│                                                                                 │
│                                                                                 │
│  ❷ Glue JDBC Write to MariaDB — Internal Mechanics                             │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                             │
│                                                                                 │
│  glueContext.write_dynamic_frame.from_options(                                  │
│      connection_type="mysql",                                                   │
│      connection_options={                                                        │
│          "url": "jdbc:mysql://host:3306/db",                                    │
│          "dbtable": "table_name",                                               │
│          "user": "...", "password": "..."                                       │
│      }                                                                          │
│  )                                                                              │
│                                                                                 │
│  What happens internally:                                                       │
│  1. Glue partitions the DynamicFrame into N partitions (Spark)                 │
│  2. Each Spark executor opens a JDBC connection to MariaDB                     │
│  3. Each executor writes its partition via batch INSERT statements             │
│  4. Default batch size: 1000 rows per INSERT                                   │
│  5. Each batch: INSERT INTO table (col1,col2,...) VALUES (...),(...),...        │
│  6. Executor commits after each batch                                           │
│                                                                                 │
│  PROBLEM with default Glue JDBC write:                                          │
│  → Uses INSERT (not INSERT ... ON DUPLICATE KEY UPDATE)                         │
│  → If row already exists → DUPLICATE KEY ERROR → job fails                     │
│  → Glue does NOT natively support UPSERT for JDBC targets                      │
│                                                                                 │
│  SOLUTIONS for UPSERT:                                                          │
│  a) Pre-action: TRUNCATE table before write (full replace)                      │
│  b) Use Spark foreachPartition with custom pymysql UPSERT logic               │
│  c) Write to temp table → SQL UPSERT from temp to target                       │
│  d) Use Glue custom connection with upsert JDBC parameter                      │
│                                                                                 │
│  SOLUTION WE USE:                                                               │
│  → customer_recommendations: Write to temp → RENAME (full replace)             │
│  → customer_metrics: foreachPartition with ON DUPLICATE KEY UPDATE             │
│                                                                                 │
│                                                                                 │
│  ❸ MariaDB RENAME TABLE — Atomic Table Swap                                   │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                                │
│                                                                                 │
│  RENAME TABLE                                                                   │
│    customer_recommendations TO customer_recommendations_old,                    │
│    customer_recommendations_new TO customer_recommendations;                    │
│                                                                                 │
│  Internal execution:                                                            │
│  1. MariaDB acquires metadata lock on BOTH tables                              │
│  2. Swaps table name pointers in data dictionary                               │
│  3. Releases lock                                                               │
│  4. Total time: < 1 millisecond (no data copy!)                                │
│  5. ATOMIC: both renames happen as one operation                               │
│  6. Concurrent queries: either see ALL old data or ALL new data                │
│  7. No moment where table doesn't exist                                        │
│                                                                                 │
│  WHY this is better than TRUNCATE + INSERT:                                     │
│  ┌─────────────────────┬─────────────────────┬──────────────────────┐           │
│  │ Step                │ TRUNCATE+INSERT     │ RENAME (swap)        │           │
│  ├─────────────────────┼─────────────────────┼──────────────────────┤           │
│  │ Table empty period  │ ~90 seconds         │ 0 seconds            │           │
│  │ App sees stale data │ Yes (during load)   │ Never (atomic swap)  │           │
│  │ Rollback possible   │ No (TRUNCATE is DDL)│ Yes (swap back)      │           │
│  │ Disk space needed   │ 1x table size       │ 2x (old + new exist) │           │
│  └─────────────────────┴─────────────────────┴──────────────────────┘           │
│                                                                                 │
│                                                                                 │
│  ❹ MariaDB ON DUPLICATE KEY UPDATE — Upsert Mechanics                          │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                         │
│                                                                                 │
│  INSERT INTO customer_metrics                                                   │
│    (customer_id, lifetime_value, churn_risk, segment, updated_at)              │
│  VALUES                                                                         │
│    ('CUST-001', 2847.50, 0.12, 'gold', NOW()),                                │
│    ('CUST-002', 142.00, 0.78, 'bronze', NOW()),                               │
│    ...                                                                          │
│  ON DUPLICATE KEY UPDATE                                                        │
│    lifetime_value = VALUES(lifetime_value),                                     │
│    churn_risk = VALUES(churn_risk),                                             │
│    segment = VALUES(segment),                                                   │
│    updated_at = VALUES(updated_at);                                             │
│                                                                                 │
│  Internal execution:                                                            │
│  1. MariaDB checks PRIMARY KEY / UNIQUE index for each row                     │
│  2. If key NOT found → INSERT new row                                           │
│  3. If key found → UPDATE specified columns                                     │
│  4. Uses InnoDB row-level locking (minimal contention)                          │
│  5. Batch of 1000 rows: ~50ms on indexed table                                 │
│  6. Total 500K rows in batches of 1000 = 500 batches = ~25 seconds             │
│                                                                                 │
│  PERFORMANCE TIP:                                                               │
│  → Disable autocommit during batch load                                         │
│  → Commit every 5000-10000 rows                                                │
│  → Reduces fsync overhead significantly                                         │
│                                                                                 │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "RENAME TABLE fails — table doesn't exist"
→ Cause: Glue job failed midway — _new table was never fully created
→ Fix: check table existence before RENAME
→ Pattern: IF EXISTS check, then RENAME, else skip and alert

PITFALL 2: "App returns stale recommendations for 2 hours"
→ Cause: reverse ETL pipeline failed silently
→ App keeps reading old table (which is fine — still valid data)
→ But nobody noticed the failure
→ Fix: monitor computed_at timestamp — alert if > 26 hours old

PITFALL 3: "MariaDB disk full during table swap"
→ Cause: old table + new table both exist simultaneously during swap
→ customer_recommendations: 2.5M rows × 2 copies = ~1 GB
→ Fix: ensure disk has 2× headroom for the largest reverse ETL table
→ Drop _old table immediately after swap

PITFALL 4: "Glue writes 500K rows but MariaDB only has 480K"
→ Cause: Glue JDBC write silently dropped rows with duplicate keys
→ Default mode: INSERT fails on duplicate, Glue catches exception per batch
→ Fix: use ON DUPLICATE KEY UPDATE explicitly
→ Or: use table swap pattern (no duplicates possible on fresh table)

PITFALL 5: "MariaDB replication lag spikes during reverse ETL load"
→ Cause: 2.5M row insert generates massive binlog
→ Read replicas fall behind by minutes
→ App reads from replica → sees OLD data even after swap
→ Fix: monitor replication lag, reduce batch sizes, or
  schedule reverse ETL during low-traffic window

PITFALL 6: "UNLOAD creates 8 files but Glue only reads 3"
→ Cause: UNLOAD with PARALLEL ON creates multiple files
→ Glue glob pattern doesn't match all file names
→ Fix: use manifest file (UNLOAD with MANIFEST option)
→ Or: use PARALLEL OFF for small result sets

PITFALL 7: "Data types mismatch — Redshift FLOAT8 → MariaDB DECIMAL"
→ Cause: Redshift FLOAT8 = double precision (15 digits)
→ MariaDB DECIMAL(10,2) = fixed precision (10 digits, 2 decimal)
→ Score 0.123456789 truncated to 0.12
→ Fix: CAST in Redshift UNLOAD query, OR adjust MariaDB column type

CURRENT LOAD:
→ customer_metrics: 500K rows → ~15 MB Parquet → 25s upsert
→ customer_recommendations: 2.5M rows → ~60 MB Parquet → 90s load + swap
→ Total reverse ETL time: ~3 minutes
→ Runs once nightly

SCALING TO 5M CUSTOMERS:
→ customer_metrics: 5M rows → ~150 MB → 250s upsert
→ customer_recommendations: 25M rows → ~600 MB → 15 min load
→ Table swap still atomic (<1ms)
→ But: MariaDB LOAD DATA is faster than JDBC for this volume
→ Switch from Glue JDBC to: UNLOAD CSV → S3 → EC2 → LOAD DATA INFILE

SCALING TO 50M CUSTOMERS:
→ customer_recommendations: 250M rows → too large for single MariaDB table
→ Options:
  a) Partition table by customer_id range (MariaDB PARTITION BY RANGE)
  b) Switch to DynamoDB (key-value access pattern, unlimited scale)
  c) Switch to ElastiCache/Redis for real-time recommendations
→ At this scale: reconsider whether MariaDB is the right target

PARALLEL WRITES (Glue → MariaDB):
→ Glue default: each Spark executor opens a JDBC connection
→ 10 DPUs = potentially 10 concurrent connections
→ MariaDB max_connections default: 151
→ SAFE: up to 20 concurrent writers
→ DANGEROUS: 100+ concurrent writers → connection pool exhaustion
→ Control via: Glue job DPU count + Spark repartition

FREQUENCY SCALING:
→ Nightly → Hourly: same architecture, 24× more runs
→ Hourly → Every 5 min: switch to streaming (Kinesis → Lambda → MariaDB)
→ Every 5 min → Real-time: skip S3, use Redshift streaming or compute in app

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ Reverse ETL: warehouse insights → operational database → user-facing app
→ UNLOAD to S3 decouples Redshift from downstream consumers (critical pattern)
→ Two load strategies depending on data nature:
   - UPSERT (ON DUPLICATE KEY UPDATE): for incremental tables
   - FULL REPLACE (RENAME swap): for recomputed tables
→ RENAME TABLE is atomic in MariaDB (<1ms, no data gap)
→ TRUNCATE+INSERT has a dangerous window where table is empty
→ Glue JDBC doesn't natively support UPSERT → use foreachPartition + pymysql
→ Freshness monitoring (computed_at) catches silent pipeline failures
→ App query latency (<5ms) validates the reverse ETL architecture
→ Repartition controls parallel JDBC connections → protects MariaDB
