ROOT CAUSES OF EXCESS DOWNTIME:
1. DETECTION: Average 2.5 hours to NOTICE a failure
   → No automated alerting on most pipelines
   → Failures discovered by downstream consumers (dashboards showing stale data)

2. RECOVERY: Average 1.5 hours to FIX after detection
   → Manual intervention required for every failure
   → No auto-retry, no self-healing
   → On-call engineer must SSH, diagnose, fix, re-run

3. ESCALATION: No tiered alerting
   → Either nobody is paged, or the CTO is paged directly
   → No runbook — every incident is "figure it out from scratch"

CTO: "Build a system that:
      → Detects failures within 2 minutes
      → Auto-recovers without humans for 80% of cases
      → Escalates with context when auto-recovery fails

🟡 HOW TO APPROACH IT

Step 1: Define what "uptime" means for a data pipeline

  UPTIME ≠ "servers are running"
  UPTIME = "fresh data is available within SLA window"

  For QuickCart:
  → Gold layer tables must be refreshed by 6:00 AM daily
  → Dashboard queries must return in < 5 seconds
  → Data must be < 2 hours stale at any point

  PIPELINE IS "DOWN" WHEN:
  → A scheduled job fails AND is not recovered within SLA window
  → Data staleness exceeds 2-hour threshold
  → Dashboard queries fail or timeout

Step 2: Decompose the problem into 4 subsystems

  SUBSYSTEM 1: DETECTION (know it's broken)
  → CloudWatch Alarms on every Glue job, Redshift query, S3 event
  → Composite Alarms that combine multiple signals
  → Data freshness checks (is gold layer stale?)

  SUBSYSTEM 2: AUTO-RECOVERY (fix it without humans)
  → Step Functions with retry/catch/fallback patterns
  → Exponential backoff for transient failures
  → Alternative execution paths for persistent failures

  SUBSYSTEM 3: ESCALATION (page the right person)
  → Tiered SNS topics: Info → Warning → Critical → Emergency
  → Time-based escalation: if not recovered in 15 min → page on-call
  → Context-rich alerts: include lineage impact (Project 77)

  SUBSYSTEM 4: MEASUREMENT (prove SLA compliance)
  → SLA metrics dashboard
  → Uptime percentage calculation
  → Incident tracking and MTTR (Mean Time To Recovery)

Step 3: Map failure modes to recovery strategies

  ┌────────────────────────┬──────────────────────────┬──────────────────┐
  │ Failure Mode           │ Auto-Recovery Strategy   │ Escalation If    │
  ├────────────────────────┼──────────────────────────┼──────────────────┤
  │ Glue job OOM           │ Retry with 2x DPU        │ 3rd retry fails  │
  │ Glue job timeout       │ Retry with longer timeout │ 3rd retry fails  │
  │ Redshift disk full     │ Run VACUUM, resize temp   │ VACUUM fails     │
  │ S3 Access Denied       │ Cannot auto-fix           │ Immediate page   │
  │ MariaDB conn timeout   │ Retry 3x with backoff     │ 3rd retry fails  │
  │ Athena query timeout   │ Retry with limit/partition │ 3rd retry fails  │
  │ Data staleness > 2hr   │ Trigger re-run of pipeline │ Re-run fails    │
  │ Network transient      │ Retry with exponential     │ 5th retry fails  │
  └────────────────────────┴──────────────────────────┴──────────────────┘

Step 4: Build incrementally

  Layer 1: Basic alarms (detect failures)
  Layer 2: Step Function with retry (auto-recover)
  Layer 3: Composite alarms + escalation (tiered alerting)
  Layer 4: SLA dashboard (prove compliance)
  Layer 5: Runbook automation (guided human intervention)

🔵 WHAT COULD BE DONE

Option A: CloudWatch Alarms only (detect + alert)
→ Detects failures, pages on-call
→ No auto-recovery — humans fix everything
→ MTTR remains high (1-3 hours)

Option B: Step Functions retry only (auto-recover)
→ Retries failed jobs automatically
→ No detection of SILENT failures (stale data)
→ No escalation if retries exhaust

Option C: Full 4-layer system (detect + recover + escalate + measure)
→ Comprehensive SLA management
→ More infrastructure to maintain
→ Covers 95% of failure modes automatically

Option D: Third-party tool (PagerDuty + Datadog)
→ Excellent UI and escalation
→ $500-2000/month additional cost
→ Still need custom detection for data freshness

DECISION: Option C with Option D for escalation only
→ Build detection + recovery + measurement in AWS-native
→ Use PagerDuty (or SNS→SMS/Email) for human escalation
→ Reason: full control, cost-effective, interview-demonstrable

✅ WHAT SHOULD DO

1. Define SLA in MEASURABLE terms
   → Not "pipelines should work" but "gold.finance_summary refreshed by 6AM UTC"
   → Each pipeline has its own SLA window and staleness threshold
   → Document in a machine-readable config (not a wiki page)

2. Alarm on ABSENCE of success, not just presence of failure
   → A job that never runs produces no failure metric
   → "Expected job_clean_orders to complete by 4AM — no completion event seen"
   → This catches scheduler failures, permission changes, deleted triggers

3. Make auto-recovery IDEMPOTENT
   → Retrying a Glue job must not produce duplicate data
   → Use Job Bookmarks (Project 33) or deduplication logic
   → If COPY to Redshift retries, use staging table + MERGE (Project 65)

4. Include LINEAGE IMPACT in every alert (Project 77)
   → "job_clean_orders failed — affects 12 downstream tables including CFO dashboard"
   → On-call engineer instantly knows severity
   → Prevents: "I'll fix it tomorrow" for critical pipelines

5. Separate TRANSIENT from PERSISTENT failures
   → Transient: network timeout, throttling, temporary OOM
   → Strategy: retry 3x with exponential backoff
   → Persistent: wrong IAM policy, schema change, disk full
   → Strategy: escalate immediately — retrying won't help

6. Track every incident in a structured store
   → DynamoDB table: incident_id, pipeline, start_time, end_time, resolution
   → Enables: MTTR calculation, trend analysis, board reporting
   → Historical data feeds SLA dashboard

7. Test the recovery system regularly
   → "Chaos engineering" for data pipelines
   → Deliberately fail a Glue job monthly — verify auto-recovery works
   → Verify escalation reaches on-call within 2 minutes

❌ WHAT SHOULD NOT DO

1. DO NOT alarm on every single metric
   → "Alarm fatigue" — when everything alarms, nothing alarms
   → 50 alarms firing simultaneously = on-call ignores all of them
   → Use COMPOSITE alarms to reduce noise

2. DO NOT retry indefinitely
   → Infinite retries on a persistent failure = wasted DPU cost
   → Max 3 retries for Glue jobs, then escalate
   → Each retry should change something (more DPU, longer timeout)

3. DO NOT put recovery logic INSIDE the Glue job
   → Glue job should do ONE thing: transform data
   → Recovery orchestration belongs in Step Functions
   → Separation of concerns: job = data logic, Step Function = flow logic

4. DO NOT use CloudWatch Logs as the primary detection mechanism
   → Parsing log lines for "ERROR" is fragile and slow
   → Use STRUCTURED metrics: Glue job state, row counts, timestamps
   → Logs are for DEBUGGING after detection, not for detection itself

5. DO NOT send raw error messages in alerts
   → "java.lang.OutOfMemoryError: GC overhead limit exceeded" means nothing to a manager
   → Translate: "Glue job ran out of memory processing 50M rows. Auto-retry with 2x capacity initiated."
   → Include: what happened, what's being done, what's the impact

6. DO NOT measure uptime as "infrastructure uptime"
   → EC2 running ≠ pipeline healthy
   → Redshift cluster online ≠ data fresh
   → Measure DATA AVAILABILITY, not server availability

7. DO NOT build SLA dashboard that updates monthly
   → Real-time or near-real-time (5-minute granularity)
   → Board wants to see current status, not last month's number
   → CloudWatch Dashboard with auto-refresh

🚫 WHEN NOT TO USE

1. DO NOT use Step Functions for sub-second recovery
   → Step Functions state transitions take 50-200ms
   → For real-time systems: use SQS retry + Lambda (Project 50 DLQ pattern)
   → Data pipelines are batch — Step Functions is perfect

2. DO NOT use CloudWatch Composite Alarms for more than 5 child alarms
   → Maximum 5 child alarms per composite alarm
   → For more complex logic: use Lambda evaluator
   → Composite of composites is allowed (nested)

3. DO NOT use SNS for long-running recovery orchestration
   → SNS delivers a message — that's it
   → Cannot track: "did the retry succeed? should I escalate?"
   → Step Functions maintains state across retries

4. DO NOT use CloudWatch Dashboards for incident management
   → Dashboards show metrics — they don't manage incidents
   → Use DynamoDB incident tracker for: open/close, assign, resolve
   → Dashboard = visualization, DynamoDB = state management

5. DO NOT implement SLA monitoring for non-critical pipelines
   → Not every pipeline needs 99.9% SLA
   → Classify: Tier 1 (CFO dashboard) = 99.9%, Tier 2 (internal) = 99%
   → Over-monitoring creates alert fatigue

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Auto-retry creates duplicate data in Redshift"
→ Cause: Glue job retries and re-runs COPY to Redshift without cleanup
→ Fix: use staging table pattern (Project 65)
→ Step Function must: truncate staging → COPY → MERGE → commit
→ If retry, staging truncate ensures idempotency

PITFALL 2: "Composite alarm stays in ALARM after individual alarms recover"
→ Cause: composite alarm evaluates on child state CHANGES
→ If child transitions OK→ALARM→OK too quickly, composite may miss recovery
→ Fix: set "TreatMissingData" to "notBreaching" on child alarms
→ Also: add a periodic re-evaluation Lambda as safety net

PITFALL 3: "Step Function retry with more DPU costs $$$"
→ Cause: every retry with 2x DPU doubles the Glue cost
→ 3 retries at 2x, 4x, 8x DPU = 14x normal cost for one failure
→ Fix: cap max DPU at 4x normal, and add cost alerting
→ Budget alarm: if daily Glue cost > 2x average → alert

PITFALL 4: "Escalation reaches on-call but they have no context"
→ Cause: alert says "job failed" but not WHY or WHAT'S AFFECTED
→ Fix: include in alert:
   - Error message (first 500 chars)
   - Retry history (attempt 1: OOM, attempt 2: OOM with 2x DPU)
   - Lineage impact (12 downstream tables, CFO dashboard)
   - Runbook URL
   - Direct link to CloudWatch Logs

PITFALL 5: "Data freshness alarm triggers during maintenance window"
→ Cause: planned Redshift resize causes 30-min downtime
→ Data freshness check sees stale data → false alarm
→ Fix: "maintenance mode" flag in DynamoDB
→ Freshness Lambda checks flag before alarming
→ Composite alarm includes NOT(maintenance_mode)

PITFALL 6: "SLA dashboard shows 100% because no incidents were logged"
→ Cause: incident tracking Lambda had a bug — never wrote to DynamoDB
→ Actual uptime was 97% but dashboard showed 100%
→ Fix: DUAL tracking — CloudWatch alarm state history AS WELL AS DynamoDB
→ Cross-validate monthly: alarm state durations vs incident records

PITFALL 7: "Auto-recovery fixes the symptom but not the root cause"
→ Cause: Glue job OOMs because upstream data grew 10x overnight
→ Auto-retry with 2x DPU works TODAY but will fail again tomorrow
→ Fix: after successful auto-recovery, still create a ticket
→ "Auto-recovered, but investigate root cause: data volume anomaly"

🔑 KEY TAKEAWAY

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ SLA for data pipelines = DATA FRESHNESS, not server uptime
→ Detection must be PROACTIVE (alarms) not REACTIVE (user complaints)
→ 80% of failures are TRANSIENT — auto-retry solves them without humans
→ Composite alarms reduce alert fatigue — combine signals before paging
→ Step Functions is the ideal orchestrator for recovery workflows
   → Built-in retry with exponential backoff
   → Catch blocks for escalation when retries exhaust
   → Wait states for time-based escalation tiers
→ Tiered escalation prevents: "nobody was paged" AND "everyone was paged"
→ Every incident must be TRACKED (DynamoDB) for SLA calculation
→ Data freshness checks catch SILENT failures that job metrics miss
   → Job succeeds but writes 0 rows
   → Scheduler stops triggering without error
   → Upstream data missing
→ The board doesn't want technical details — they want ONE NUMBER: 99.94%
→ Auto-recovery must be IDEMPOTENT — retrying must not duplicate data
→ ALWAYS include lineage impact in alerts (Project 77)
   → "12 downstream tables affected including CFO dashboard"
   → On-call instantly knows severity without investigating
