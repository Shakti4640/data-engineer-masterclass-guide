🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — analysts querying S3 via Athena (Project 24)
Situation:
→ Daily CSV exports land in S3 (Project 1):
   - orders/     → 30 files × 50 MB = 1.5 GB (CSV)
   - customers/  → 30 files × 10 MB = 300 MB (CSV)
   - products/   → 30 files × 2 MB  = 60 MB (CSV)

→ Athena cost comparison (proven in Project 24):
   - raw_orders (CSV):     scans 1,500 MB per query → $0.0075
   - curated_orders (Parquet): scans 50 MB per query → $0.00025
   - Same query, 30x cost difference

→ 10 analysts × 30 queries/day × 30 days = 9,000 queries/month
   - On CSV:     9,000 × $0.0075 = $67.50/month
   - On Parquet: 9,000 × $0.00025 = $2.25/month
   - SAVINGS: $65.25/month just from format conversion

→ Problem gets WORSE as data grows:
   - 6 months: CSVs grow to 9 GB → $0.045 per query
   - 1 year: CSVs grow to 18 GB → $0.09 per query
   - Parquet stays proportionally cheap (columnar pruning)

→ Additional problems with raw CSV:
   - No column pruning (Athena reads entire row)
   - No predicate pushdown (no row group statistics)
   - No compression (raw text wastes storage)
   - String types everywhere (crawler couldn't infer dates — Project 23)
   - Schema mismatches possible (source system adds column)

CTO: "Convert everything to Parquet. Automate it."
YOU: "Glue ETL job. Reads CSV from S3, transforms, writes Parquet.
      Runs nightly after data lands. Athena queries Parquet only.
      CSVs stay as raw backup in 'bronze' layer.
      Parquet becomes the 'silver' query layer."

🟡 HOW TO APPROACH IT

Step 1: Understand WHY Parquet is better than CSV
→ CSV: row-based → must read entire row even for 1 column
→ Parquet: columnar → reads only requested columns
→ CSV: no type metadata → everything is string
→ Parquet: embedded schema → exact types (int, double, date, timestamp)
→ CSV: no compression by default → 1.5 GB raw text
→ Parquet: Snappy compression built-in → ~200 MB for same data
→ CSV: no statistics → Athena scans everything
→ Parquet: row group statistics (min/max) → skips irrelevant chunks

Step 2: Understand what Glue ETL does
→ Managed Apache Spark service
→ Reads from S3 (CSV, JSON, Parquet, ORC, Avro)
→ Transforms data (clean, cast types, filter, join, rename)
→ Writes to S3 in optimized format (Parquet, ORC)
→ Updates Glue Catalog automatically
→ Serverless: spins up workers, runs job, shuts down
→ Billed per DPU-second (1 DPU = ~$0.44/hour)

Step 3: Understand Glue's DynamicFrame
→ Glue's extension of Spark DataFrame
→ Handles schema inconsistencies (CSV columns that vary)
→ ResolveChoice: handles ambiguous types (string vs int)
→ ApplyMapping: rename and retype columns
→ Relationalize: flatten nested structures
→ Can convert to/from Spark DataFrame for complex operations

Step 4: Plan the ETL pipeline
→ Source: s3://bucket/daily_exports/orders/*.csv (raw CSV)
→ Transform: cast types, clean nulls, validate
→ Target: s3://bucket/silver/orders/*.parquet (optimized)
→ Catalog: update Glue table to point at Parquet location
→ Schedule: run nightly after CSV upload completes

Option A: Athena CTAS (Create Table As Select)
→ CREATE TABLE silver_orders WITH (format='PARQUET') AS SELECT * FROM raw_orders
→ Simple one-liner — converts and creates table
→ BUT: runs as Athena query → charged per TB scanned ($7.50 for 1.5 GB)
→ BUT: no incremental support — recreates entire table each time
→ BUT: limited transformation capability
→ Good for: one-time conversion, small tables
→ Covered in Project 36

Option B: Glue ETL Job (recommended)
→ Full Spark engine for transformation
→ Supports incremental processing (Job Bookmarks — Project 33)
→ Rich transformation library (DynamicFrame)
→ Handles schema evolution (ResolveChoice — Project 45)
→ Can process multiple tables in one job
→ Billed per DPU-second (cheaper than Athena for large conversions)
→ SELECTED ✅

Option C: EMR Spark Job
→ Full Spark cluster you manage
→ Maximum flexibility
→ But: cluster management overhead, EC2 costs
→ Overkill for simple CSV → Parquet conversion
→ Use when: Glue's managed environment is too limiting

Option D: Lambda function with PyArrow
→ Read CSV → convert to Parquet → write to S3
→ Works for small files (< 500 MB)
→ Lambda memory limit: 10 GB, timeout: 15 minutes
→ Not suitable for 1.5 GB+ datasets
→ Good for: single-file event-driven conversion

DECISION: Option B — Glue ETL Job
→ Right-sized for daily batch conversion
→ Handles schema evolution as data grows
→ Incremental processing with Job Bookmarks
→ Integrates with Glue Catalog natively

✅ WHAT SHOULD DO

1. ALWAYS write Parquet with Snappy compression
   → Snappy: fast compression/decompression, good ratio
   → GZIP: better ratio but slower (use for archival)
   → Snappy is Parquet default — no extra configuration
   → 1.5 GB CSV → ~200 MB Parquet with Snappy

2. ALWAYS cast types during conversion
   → CSV crawler inferred everything as STRING (Project 23)
   → Cast in Glue: order_date STRING → DATE, quantity STRING → INT
   → Parquet stores exact types → Athena queries work correctly
   → No more WHERE order_date > '2025-01-15' string comparison bugs

3. ALWAYS coalesce output files to reasonable count
   → Glue default: creates hundreds of tiny Parquet files
   → Tiny files = slow Athena LIST + per-file overhead
   → Ideal Parquet file size: 128 MB – 512 MB
   → Use repartition() or coalesce() to control output count
   → Detailed in Project 62 (small file problem)

4. ALWAYS update Glue Catalog after writing Parquet
   → Either: run crawler on silver/ prefix (automatic)
   → Or: Glue job writes with catalog update enabled (programmatic)
   → Without catalog update: Athena doesn't see new Parquet table

5. KEEP both CSV and Parquet layers
   → CSV = bronze (raw, immutable backup)
   → Parquet = silver (optimized query layer)
   → If Parquet gets corrupted: regenerate from CSV
   → Never delete source CSVs after conversion

6. SET Glue job parameters externally
   → Don't hardcode S3 paths in job script
   → Use job parameters: --source_path, --target_path
   → Enables reuse across environments (dev/staging/prod)

7. ENABLE Glue Job Bookmarks for incremental processing
   → First run: processes ALL files
   → Subsequent runs: processes only NEW files since last run
   → Prevents reprocessing already-converted CSVs
   → Covered deeply in Project 33

8. LOG transformation metrics
   → Input rows, output rows, dropped rows, type cast failures
   → Write metrics to CloudWatch or S3
   → Essential for data quality monitoring

❌ WHAT SHOULD NOT DO

1. DO NOT delete source CSVs after conversion
   → Parquet is derived data — can be regenerated
   → CSV is the source of truth — cannot be regenerated
   → If conversion had a bug: you need CSVs to re-run
   → Move old CSVs to Glacier (Project 4) — don't delete

2. DO NOT create one giant Parquet file
   → Single 1.5 GB Parquet file → single Athena worker reads it
   → Other workers idle → slow query
   → Split into 4-16 files for parallelism
   → Rule: each file 128-512 MB

3. DO NOT create thousands of tiny Parquet files
   → Glue default with many input CSVs → many output files
   → 1000 × 1 MB files → S3 LIST overhead dominates Athena query
   → Use coalesce() to merge into fewer larger files
   → This is the #1 Glue ETL performance issue (Project 62)

4. DO NOT skip type casting
   → Writing CSV data as-is to Parquet preserves STRING types
   → Parquet with all STRING columns = almost no benefit over CSV
   → The TYPE INFORMATION is what makes Parquet powerful
   → Cast during ETL: string → int, double, date, timestamp, boolean

5. DO NOT use Glue ETL for simple SELECT * conversion
   → If no transformation needed: Athena CTAS is simpler (Project 36)
   → Glue ETL shines when: type casting, filtering, joining, cleaning
   → For pure format conversion: evaluate CTAS first

6. DO NOT hardcode column names in Glue script
   → Source schema may change (new column added)
   → Hardcoded mapping breaks when schema evolves
   → Use DynamicFrame's flexible schema handling
   → Or: explicit mapping with fallback for unknown columns

7. DO NOT over-provision DPUs
   → 2 DPU (minimum) handles 1.5 GB CSV in ~5 minutes
   → 10 DPU for same data: ~2 minutes but 5x cost
   → Start with 2 DPU → increase only if job exceeds time budget
   → Each DPU: 4 vCPU, 16 GB RAM

8. DO NOT ignore Glue job error logs
   → Job fails silently with partial output
   → CloudWatch Logs: /aws-glue/jobs/error
   → Always check: output row count vs input row count
   → Missing rows = type cast failures or filter issues

🚫 WHEN NOT TO USE GLUE ETL

1. When data is already in Parquet/ORC
   → No conversion needed — already optimized
   → Glue ETL adds cost for zero benefit
   → Just crawl and query directly

2. When total data < 100 MB
   → Glue minimum: 2 DPU × 10 min billing = ~$0.15
   → Athena CTAS on 100 MB CSV: ~$0.0005
   → CTAS is 300x cheaper for tiny datasets

3. When you need sub-minute latency
   → Glue job cold start: 30-60 seconds
   → Job execution: 2-10 minutes minimum
   → For real-time: Lambda or Kinesis Firehose

4. When transformation is pure SQL (no Python/Spark needed)
   → Athena CTAS handles SQL-based transformations
   → Glue adds Spark overhead unnecessarily
   → Use Glue only when you need Python logic, loops, API calls

5. When budget is extremely tight and data is small
   → Glue minimum billing: 1 minute per DPU (minimum 2 DPU)
   → $0.44/hour per DPU → $0.015 per minute per DPU
   → Even a 30-second job costs ~$0.03
   → For micro-budgets: Lambda + PyArrow for small files

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Glue creates 500 tiny Parquet files for 1.5 GB input"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: 30 input CSV files → Spark creates many tasks → many output files
→ Each Spark task writes one output file
→ 500 × 3 MB files instead of 4 × 200 MB files
→ Athena impact: slow S3 LIST, per-file overhead, expensive queries
→ Fix: repartition() or coalesce() before writing
→ dyf.toDF().coalesce(4).write.parquet(...)
→ 4 files for 2 dc2.large nodes × 2 slices = optimal for Redshift too

PITFALL 2: "Type cast fails — job errors on 'N/A' in numeric column"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: CSV has "N/A", "", or "null" in numeric columns
→ ApplyMapping with ("amount", "string", "amount", "double") → crash
→ Fix A: ResolveChoice with "cast:double" (casts or nulls invalid)
→ Fix B: Filter invalid rows before type cast
→ Fix C: Use Spark's col().cast("double") which returns null on failure

PITFALL 3: "Output Parquet has all STRING columns"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: no ApplyMapping or type casting in the job
→ Just read CSV → write Parquet → still STRING types
→ Parquet file is smaller (compression) but no column pruning benefit
→ Fix: ALWAYS include explicit type mapping in the job

PITFALL 4: "Job runs successfully but Athena shows old data"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Glue Catalog table still points to old S3 location
→ Job wrote Parquet to new path but catalog not updated
→ Fix: enable catalog update in write_dynamic_frame
→ Or: run crawler on output path after job completes (Project 23)

PITFALL 5: "Job takes 15 minutes for 1.5 GB — expected 5 minutes"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause A: only 2 DPU for data with many columns and complex transforms
→ Cause B: Spark repartition/shuffle on large dataset
→ Cause C: writing too many small files (S3 multipart overhead)
→ Fix A: increase DPU to 4-6 for faster processing
→ Fix B: minimize shuffles — use coalesce() not repartition()
→ Fix C: coalesce to fewer output files

PITFALL 6: "Glue Job Bookmark processes same files again"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: bookmark only works with from_catalog() not from_options()
→ Or: job failed midway → bookmark not committed
→ Fix: use from_catalog() for bookmark support
→ Reset bookmark: aws glue reset-job-bookmark --job-name ...
→ Covered deeply in Project 33

PITFALL 7: "Parquet output is LARGER than CSV input"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: all columns are STRING → Parquet can't compress efficiently
→ Strings compress poorly compared to integers/doubles
→ Fix: cast numeric columns to proper types BEFORE writing
→ int + double columns compress 5-10x better than string equivalents
→ With proper types: 1.5 GB CSV → ~200 MB Parquet
→ Without proper types: 1.5 GB CSV → ~1.2 GB Parquet (barely better)

OUTPUT FILE SIZING:
━━━━━━━━━━━━━━━━━━
→ Target: 128-512 MB per Parquet file
→ Formula: total_data_size / target_file_size = file_count
→ 1.5 GB / 200 MB = ~8 files (use coalesce(8))
→ 10 GB / 256 MB = ~40 files (use coalesce(40))

→ Day 1: process 1.5 GB (30 files) → 5 minutes

→ Day 30: process 50 MB (1 new file) → 2 minutes
→ Without incremental: Day 30 processes 3 GB (60 files) → 8 minutes
→ With incremental: always ~2 minutes (only new files)

COST OPTIMIZATION:
━━━━━━━━━━━━━━━━━━
