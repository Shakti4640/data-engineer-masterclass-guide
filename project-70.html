→ 3 AM page: "All pipelines down — authentication failure"

→ All credential access must be logged (who, when, which credential)

Step 2: Understand Secrets Manager rotation mechanics

  ROTATION STRATEGY: "Alternating Users"
  
  ┌──────────────────────────────────────────────────────────────────┐
  │ TWO database users, alternating:                                 │
  │                                                                  │
  │ User A: glue_reader_a (current active)                           │
  │ User B: glue_reader_b (standby)                                  │
  │                                                                  │
  │ ROTATION STEP 1: Generate new password for User B                │
  │ ROTATION STEP 2: Update User B's password in MariaDB             │
  │ ROTATION STEP 3: Test User B can connect                         │
  │ ROTATION STEP 4: Update secret to point to User B                │
  │ ROTATION STEP 5: User B is now "current", User A is "standby"   │
  │                                                                  │
  │ NEXT ROTATION: swap back — update User A, make it current        │
  │                                                                  │
  │ WHY TWO USERS:                                                   │
  │ → During rotation: User A still works (services using old secret)│
  │ → After rotation: User B works (services get new secret)         │
  │ → Brief overlap: BOTH users valid → zero downtime                │
  │ → No "gap" where neither password works                          │
  └──────────────────────────────────────────────────────────────────┘

🔵 WHAT COULD BE DONE

Option A: Secrets Manager with Lambda rotation ✅ (CHOSEN)
→ AWS-native, fully managed
→ Built-in rotation templates for MySQL, PostgreSQL, Redshift
→ Glue native integration (secret ARN in connection)
→ Redshift Data API native integration (SecretArn parameter)
→ CloudTrail audit automatically
→ Caching: SDK caches secret → reduces API calls

Option B: AWS Systems Manager Parameter Store (SecureString)
→ Cheaper ($0 for standard, $0.05/parameter for advanced)
→ BUT: no built-in rotation mechanism
→ Must build custom rotation logic
→ No native Glue connection integration
→ Good for: static config, not rotating credentials

Option C: HashiCorp Vault
→ Industry standard, cross-cloud
→ Dynamic secrets (generate on-demand, auto-expire)
→ BUT: requires Vault infrastructure (EC2/EKS)
→ Operational overhead: backup, HA, upgrades
→ Good for: multi-cloud enterprises, not AWS-only

Option D: IAM Database Authentication
→ MariaDB + RDS: supports IAM auth (no passwords at all)
→ Redshift: supports IAM auth via GetClusterCredentials
→ BUT: MariaDB on EC2 doesn't support IAM auth natively
→ Would need RDS migration (out of scope)
→ Redshift: can use IAM but Glue Connection still needs JDBC password

DECISION: Option A — Secrets Manager with Lambda rotation
→ Best integration with existing AWS services
→ Built-in rotation for MySQL/MariaDB and Redshift
→ Native Glue + Data API support
→ CloudTrail audit included
→ Cost: $0.40/secret/month + $0.05/10K API calls
→ 4 secrets × $0.40 = $1.60/month (negligible)

✅ WHAT SHOULD DO

1. CREATE secrets in Secrets Manager for EVERY credential
   → quickcart/prod/mariadb/glue-reader
   → quickcart/prod/mariadb/root (admin)
   → quickcart/prod/redshift/etl-writer
   → quickcart/prod/cross-account/external-id
   → Naming convention: {app}/{env}/{service}/{purpose}

2. USE JSON format for secret values (not plain string)
   → {"username":"X", "password":[REDACTED:PASSWORD]Y", "host":"Z", "port":3306, "dbname":"D"}
   → All connection info in one secret → single retrieval
   → Matches Secrets Manager rotation template expectations

3. CREATE separate database users for rotation (alternating users)
   → MariaDB: glue_reader_a + glue_reader_b
   → Both have identical GRANT permissions
   → Rotation alternates between them
   → During rotation: BOTH valid simultaneously → zero downtime

4. UPDATE all services to retrieve secrets at RUNTIME
   → NOT: read secret at deploy time → store in env var
   → YES: call get_secret_value() when connection needed
   → Use caching: AWS SDK caches for 1 hour by default
   → Reduces API calls while staying current

5. USE Glue Connection's native secret reference
   → Glue Connection supports: "SECRET_ARN" property
   → Glue retrieves credentials from secret at job start
   → No password in Glue Connection properties
   → Password change → Glue automatically uses new one

6. USE Redshift Data API's SecretArn parameter
   → execute_statement(SecretArn="arn:...", Database="analytics", Sql="...")
   → No DbUser or password in Lambda code
   → Data API retrieves credentials from Secrets Manager

7. ENABLE rotation schedule on each secret
   → Rotation interval: 90 days (SOC 2 requirement)
   → Rotation Lambda: AWS-provided template (customizable)
   → Test rotation manually first → then enable automatic

8. SET resource policy on secrets for least-privilege access
   → Only specific IAM roles can read each secret
   → GlueCrossAccountETLRole → can read mariadb/glue-reader
   → LambdaMVRefreshRole → can read redshift/etl-writer
   → No blanket "secretsmanager:*" permissions

9. ENABLE CloudTrail for Secrets Manager events
   → Already enabled by default in CloudTrail
   → Log: GetSecretValue, DescribeSecret, RotateSecret
   → Alert: unusual access patterns (new IP, unusual time)

❌ WHAT SHOULD NOT DO

1. DO NOT store passwords in Lambda environment variables
   → Visible in console to anyone with Lambda:GetFunction
   → Visible in CloudFormation template
   → No rotation — must redeploy Lambda to change
   → Instead: Lambda calls get_secret_value() at runtime

2. DO NOT store passwords in Glue job arguments
   → Visible in Glue console, CloudTrail, CloudWatch Logs
   → --redshift_password=[REDACTED:PASSWORD] → logged everywhere
   → Instead: pass SECRET ARN as argument → retrieve in code

3. DO NOT use single database user for rotation
   → Single user rotation: change password → brief window where nothing works
   → Between password change and secret update: connections FAIL
   → Instead: alternating users (A/B) → overlap period → zero downtime

4. DO NOT rotate all secrets simultaneously
   → If rotation Lambda has a bug: ALL credentials break at once
   → Stagger: MariaDB on day 1, Redshift on day 15, etc.
   → Reduces blast radius of rotation failures

5. DO NOT forget to test rotation BEFORE enabling automatic schedule
   → First rotation: run manually → verify all services still work
   → Check: Glue jobs, Lambda functions, Athena connectors, dashboards
   → Only after manual success: enable 90-day automatic rotation

6. DO NOT cache secrets for too long
   → Cache forever: defeats purpose of rotation
   → Cache 0 seconds: excessive API calls ($0.05/10K)
   → Sweet spot: 1 hour cache (default in AWS SDK)
   → After rotation: services pick up new secret within 1 hour

7. DO NOT grant secretsmanager:* to any role
   → "Allow all Secrets Manager actions on all secrets"
   → Overly permissive: role can read ANY secret in account
   → Instead: specific ARN per role per secret

8. DO NOT log secret values
   → print(f"Connecting with password: {password}")
   → Password appears in CloudWatch Logs
   → NEVER log credential values — log secret ARN only
   → Use: print(f"Connecting using secret: {secret_arn}")

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Rotation succeeded but Glue job fails 2 hours later"
→ Cause: Glue Connection caches credentials from PREVIOUS job run
→ Glue doesn't re-fetch secret on every run — uses cached connection
→ Fix: Glue ETL script explicitly calls get_secret_value() at start
→ Or: restart Glue Connection test to force re-cache

PITFALL 2: "Rotation Lambda can't connect to MariaDB"
→ Cause: Rotation Lambda not in VPC, MariaDB on private subnet
→ Rotation Lambda must reach MariaDB to change password (step 2)
→ Fix: Deploy rotation Lambda in same VPC/subnet as Glue ENIs
→ Same VPC Peering path as Glue (Project 61)

PITFALL 3: "testSecret fails: Access denied for glue_reader_b"
→ Cause: glue_reader_b user exists but lacks GRANT permissions
→ Password changed but user can't access database
→ Fix: ensure BOTH alternating users have IDENTICAL grants
→ CREATE USER glue_reader_b → GRANT same as glue_reader_a

PITFALL 4: "Secret rotated but dashboard still uses old password"
→ Cause: Superset JDBC connection caches password in config file
→ Superset doesn't call Secrets Manager — uses static config
→ Fix: custom Superset auth hook that reads from Secrets Manager
→ Or: Superset restart after rotation (less elegant but works)

PITFALL 5: "All 4 secrets rotated on same day — Lambda throttled"
→ Cause: 4 rotation Lambdas invoke simultaneously → API throttle
→ Secrets Manager API: 5,000 requests/second (unlikely to hit)
→ More likely: MariaDB connection limit hit by 4 simultaneous rotations
→ Fix: stagger rotation schedules (day 1, day 8, day 15, day 22)

PITFALL 6: "Rotation works but CloudTrail shows 1000s of GetSecretValue calls"
→ Cause: every Glue task (10 DPU × 4 tasks each) calls get_secret_value()
→ 40 calls per job run × 30 runs/day = 1,200 calls/day
→ Cost: $0.05/10K calls = negligible ($0.006/day)
→ But: CloudTrail log volume increases
→ Fix: use caching library → reduces to 1 call per hour per Lambda

PITFALL 7: "Secret deleted accidentally — all pipelines down"
→ Cause: someone ran aws secretsmanager delete-secret
→ ALL services referencing this secret → authentication failure
→ Fix: enable deletion protection (DeletionPolicy: Retain in CloudFormation)
→ Recovery: Secrets Manager has 7-30 day recovery window before permanent deletion
→ aws secretsmanager restore-secret --secret-id <name>

PITFALL 8: "Rotation Lambda has permission to read ALL secrets"
→ Cause: IAM policy allows secretsmanager:* on Resource: *
→ Rotation Lambda for MariaDB can read Redshift secret (shouldn't need to)
→ Fix: scope each rotation Lambda to its specific secret ARN only

📈 SCALABILITY CONSIDERATIONS

NUMBER OF SECRETS:
→ 4 secrets (current): trivial — $1.60/month
→ 20 secrets (10 services × 2 envs): $8/month
→ 100 secrets (enterprise): $40/month
→ 1000 secrets (large enterprise): $400/month
→ Cost scales linearly — no volume discount

API CALL SCALING:
→ Without caching: N services × M calls/day × 30 days
→ 10 services × 100 calls/day = 30,000/month = $0.15
→ With caching: 10 services × 24 calls/day (hourly) = 7,200/month = $0.04
→ API call cost is NEGLIGIBLE at any scale

ROTATION LAMBDA SCALING:
→ Each secret needs its own rotation Lambda (or shared Lambda with routing)
→ 4 secrets × 1 Lambda = 4 Lambdas (current)
→ 20 secrets: use shared Lambda with secret-type routing
→ Template: AWS provides rotation templates for MySQL, PostgreSQL, Redshift

MULTI-ACCOUNT SCALING:
→ Account A has MariaDB → secret in Account A's Secrets Manager
→ Account B needs to read it → cross-account secret access
→ OR: replicate secret to Account B's Secrets Manager
→ Best practice: secret lives where database lives → cross-account read
