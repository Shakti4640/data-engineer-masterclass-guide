Company: "QuickCart" — Pipeline maturity growing
Current State (after Projects 31, 32, 33, 34):

   Job 1: MariaDB → S3 Bronze (Project 31)
   Job 2: Bronze → Silver transforms (Project 34)
   Job 3: Silver → Redshift load (Project 32)
→ 2 Glue Crawlers needed:
   Crawler A: Crawl bronze/ after Job 1 (discover new partitions)
   Crawler B: Crawl silver/ after Job 2 (register new tables/partitions)
→ Currently orchestrated by Python script (Project 34 Part 3)

THE PROBLEMS:
━━━━━━━━━━━━━

Problem 1 — FRAGILE ORCHESTRATION:
→ Python script runs on EC2 via cron
→ If EC2 reboots at 2 AM → pipeline doesn't run
→ If script crashes mid-way → partial state, no cleanup
→ No automatic retry on transient failures

Problem 2 — NO DEPENDENCY MANAGEMENT:
→ Job 2 should ONLY run if Job 1 SUCCEEDED
→ Crawler B should ONLY run if Job 2 SUCCEEDED
→ Python script checks sequentially — but what if Job 1 takes 2 hours?
→ Script timeout → Job 2 never starts

Problem 3 — MISSING CRAWLERS IN PIPELINE:
→ Job 1 writes new Parquet partitions to bronze/
→ Athena can't query new partitions until Crawler A runs
→ Currently: engineer manually runs Crawler A each morning
→ Sometimes forgets → analysts see stale data

Problem 4 — NO VISUAL MONITORING:
→ Python script logs to CloudWatch but no visual graph
→ "Which step failed?" requires reading 500 lines of logs
→ No at-a-glance pipeline status

Problem 5 — ADDING NEW STEPS IS PAINFUL:
→ Want to add: data quality check between Job 2 and Job 3
→ Requires modifying Python script, testing, redeploying
→ No plug-and-play step insertion

→ Data engineer: "We need a MANAGED workflow that chains
   Crawlers + ETL Jobs with dependencies, retries, and monitoring."

→ CTO: "Isn't that what Glue Workflows do?"

🟡 HOW TO APPROACH IT

Step 1: Understand Glue Workflow components
→ WORKFLOW: Container that groups Crawlers + Jobs + Triggers
→ TRIGGER: Defines WHEN something runs (on-demand, scheduled, conditional)
→ CRAWLER: Auto-discovers schema from S3 data (Project 23)
→ JOB: ETL transformation (Projects 31, 32, 34)
→ ACTION: What a trigger starts (one or more Crawlers/Jobs)

Step 2: Design the execution graph
→ Start trigger (scheduled at 2:00 AM)
  → Job 1: MariaDB → Bronze
    → Conditional trigger (on Job 1 SUCCESS)
      → Crawler A: Crawl bronze/
        → Conditional trigger (on Crawler A SUCCESS)
          → Job 2: Bronze → Silver
            → Conditional trigger (on Job 2 SUCCESS)
              → Crawler B: Crawl silver/
                → Conditional trigger (on Crawler B SUCCESS)
                  → Job 3: Silver → Redshift

Step 3: Understand trigger types
→ SCHEDULED: Cron expression (e.g., "cron(0 2 * * ? *)")
→ ON_DEMAND: Manual start via API/Console
→ CONDITIONAL: Fires when predecessor(s) reach a state
→ EVENT: Fires on EventBridge event (Project 41 pattern)

Step 4: Dependencies from previous projects
→ Project 23: Glue Crawlers (auto-discover schema)
→ Project 31: MariaDB → Bronze job
→ Project 32: Silver → Redshift job
→ Project 34: Bronze → Silver job
→ Project 33: Incremental processing (bookmarks still apply)

🔵 WHAT COULD BE DONE

Option A: Glue Workflow (this project)
→ Native Glue orchestration
→ Visual graph in Console
→ Conditional triggers between steps
→ Pros: tight Glue integration, simple for Glue-only pipelines
→ Cons: limited to Glue Crawlers + Jobs, no Lambda/non-Glue steps

Option B: Step Functions (Project 55)
→ AWS-native orchestration for ANY service
→ Can chain: Glue + Lambda + ECS + SNS + SQS + any AWS API
→ Pros: most flexible, visual designer, complex branching
→ Cons: more setup, separate service to manage

Option C: MWAA (Managed Apache Airflow)
→ Full Airflow on AWS
→ DAG-based orchestration with Python
→ Pros: industry standard, huge community, complex DAGs
→ Cons: expensive ($400+/month minimum), overkill for 5-step pipeline

Option D: EventBridge + Lambda
→ Event-driven: each step completion triggers next step
→ Pros: serverless, event-driven, decoupled
→ Cons: no visual graph, hard to track pipeline state

DECISION: Option A — Glue Workflow
→ Perfect for Glue-only pipelines (our exact case)
→ Zero additional infrastructure cost
→ Visual monitoring in Glue Console
→ Will migrate to Step Functions (Project 55) when we add non-Glue steps

✅ WHAT SHOULD DO

1. Name workflows and triggers descriptively
   → Workflow: "quickcart-nightly-pipeline"
   → Triggers: "trigger-start", "trigger-after-extract", "trigger-after-silver"
   → Makes Console navigation and debugging clear

2. Use CONDITIONAL triggers with SUCCEEDED condition
   → "Start Job 2 ONLY when Job 1 state = SUCCEEDED"
   → If Job 1 FAILS: trigger does NOT fire → pipeline stops
   → This is the dependency enforcement mechanism

3. Pass workflow run properties to all jobs
   → Workflow can pass key-value properties to child jobs
   → Use for: extraction_date, run_id, pipeline_name
   → All jobs share same date parameter → consistency

4. Set appropriate timeouts on each step
   → Job 1 (Extract): 60 min timeout
   → Crawler A: 30 min timeout
   → Job 2 (Silver): 90 min timeout
   → Crawler B: 30 min timeout
   → Job 3 (Redshift): 60 min timeout
   → Total pipeline budget: ~4.5 hours max

5. Create a dedicated Crawler for each layer
   → Crawler A: targets bronze/mariadb/ only
   → Crawler B: targets silver/ only
   → Don't mix — one crawler per data layer

6. Monitor workflow via CloudWatch
   → Glue publishes workflow run metrics
   → Set alarm on: workflow FAILED state
   → Route alarm to SNS (Project 54)

7. Test with ON_DEMAND trigger first
   → Create workflow with on-demand start trigger
   → Manually trigger → watch execution → verify all steps
   → Switch to SCHEDULED trigger after successful test

❌ WHAT SHOULD NOT DO

1. DO NOT create circular trigger dependencies
   → Trigger A starts Job 1 → Trigger B starts Job 2 → Trigger C starts Job 1
   → Creates infinite loop → burns DPUs forever
   → Workflow graph must be a DAG (Directed Acyclic Graph)

2. DO NOT put ALL crawlers and jobs in one mega-workflow
   → Workflow for pipeline A, separate workflow for pipeline B
   → One failure in shared workflow blocks unrelated pipelines
   → Keep workflows focused on one logical pipeline

3. DO NOT skip crawlers between ETL jobs
   → Job 2 reads from Glue Catalog tables populated by Crawler A
   → If Crawler A hasn't run → Job 2 reads stale/missing partitions
   → Crawler ensures catalog is fresh before next job reads it

4. DO NOT use SCHEDULED triggers on INTERMEDIATE steps
   → Only the FIRST trigger should be scheduled
   → All subsequent triggers should be CONDITIONAL (on predecessor success)
   → Scheduled intermediate trigger = step runs even if predecessor failed

5. DO NOT hardcode dates in workflow properties
   → Use dynamic expressions or let each job derive its own date
   → Or pass date via workflow run properties at start time

6. DO NOT forget to handle the "no new data" scenario
   → Job 1 extracts 0 rows (no changes in MariaDB)
   → Job 1 still SUCCEEDS (0 rows is valid)
   → Job 2 runs on empty bronze partition → produces empty silver
   → Job 3 loads empty data to Redshift → no-op
   → All steps should handle 0-row gracefully without failing

7. DO NOT mix Glue versions across jobs in same workflow
   → Job 1: Glue 4.0, Job 2: Glue 2.0 → different Spark versions
   → Parquet files written by Spark 3.3 may have compatibility issues with Spark 2.4
   → Use Glue 4.0 for ALL jobs in the pipeline

🚫 WHEN NOT TO USE GLUE WORKFLOWS

1. Pipeline includes non-Glue steps
   → Lambda function for data quality validation
   → ECS task for ML model inference
   → SNS notification between steps
   → Glue Workflow can ONLY orchestrate Glue Crawlers and Jobs
   → Use: Step Functions (Project 55)

2. Need complex branching logic
   → "If revenue > $1M → run Job A, else → run Job B"
   → Glue Workflow triggers are simple: SUCCESS/FAIL/ANY
   → No data-dependent branching
   → Use: Step Functions with Choice state

3. Need human approval gates
   → "Pause pipeline until manager approves data quality report"
   → Glue Workflows have no wait/approval mechanism
   → Use: Step Functions with Task token + callback

4. Need to orchestrate cross-account resources
   → Glue Workflow can only manage resources in same account
   → Cross-account jobs need external orchestration
   → Use: Step Functions with cross-account IAM roles (Project 60)

5. Need parallel execution with fan-out/fan-in
   → "Run 10 table extractions in parallel, wait for all, then merge"
   → Glue Workflow supports parallel actions BUT limited fan-in
   → Use: Step Functions with Parallel + Map states

6. Need sub-minute scheduling
   → Glue Workflow minimum schedule: 5 minutes
   → For real-time processing: use EventBridge + Lambda

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Workflow runs but triggers never fire"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Triggers not ACTIVATED
→ After creating trigger: state is CREATED (not ACTIVATED)
→ Must call: activate_trigger() for each trigger
→ OR: set State="ACTIVATED" during creation
→ Without activation: trigger exists but never evaluates

PITFALL 2: "Workflow starts but first job never runs"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Start trigger is CONDITIONAL instead of SCHEDULED
→ Conditional trigger waits for predecessor — but there IS no predecessor
→ Fix: First trigger must be SCHEDULED or ON_DEMAND

PITFALL 3: "Jobs run but don't receive workflow properties"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Jobs must explicitly read workflow run properties
→ Workflow properties passed as additional job arguments
→ Job must use getResolvedOptions to read them
→ Properties prefixed with "--" automatically

PITFALL 4: "Crawler succeeds but Catalog still shows old schema"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Crawler configured to ADD new partitions only, not UPDATE
→ SchemaChangePolicy: "UPDATE_IN_DATABASE" vs "ADD_NEW_COLUMNS"
→ Fix: Set UpdateBehavior to "UPDATE_IN_DATABASE"
→ And: RecrawlPolicy to "CRAWL_NEW_FOLDERS_ONLY" for efficiency

PITFALL 5: "Workflow completes but one step was skipped silently"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Trigger predicate uses "ANY" instead of "AND"
→ If predecessor A fails but predecessor B succeeds → trigger fires
→ Downstream job runs without data from A
→ Fix: Always use logical="AND" unless explicitly needed

PITFALL 6: "Scheduled trigger fires but workflow is already running"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Previous run took longer than 24 hours
→ New scheduled run starts while old one still running
→ Two workflows running simultaneously → data conflicts
→ Fix: Check MaxConcurrentRuns on each job (should be 1)
→ Fix: Set workflow timeout shorter than schedule interval

PITFALL 7: "Workflow shows COMPLETED but Job 3 actually failed"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Workflow status is COMPLETED when all triggers are evaluated
→ "COMPLETED" means "workflow finished executing" NOT "all jobs succeeded"
→ A job can FAIL and workflow still shows COMPLETED
→ Fix: Check individual node statuses, not just workflow status
→ Use get_workflow_run() → check each node in Graph.Nodes

📈 SCALABILITY CONSIDERATIONS

WORKFLOW LIMITS:
━━━━━━━━━━━━━━━
→ Max triggers per workflow: 10 (soft limit, can request increase)
→ Max concurrent workflow runs: 1 (by default)
→ Max crawlers/jobs per trigger action: 10
→ Max workflow runs retained: 1000 (oldest auto-deleted)

SCALING STRATEGIES:
→ 5-step pipeline: single workflow (our case) ✅
→ 10-step pipeline: single workflow, optimize with parallel triggers
→ 20+ step pipeline: consider Step Functions (more flexible)
→ 50+ step pipeline: MWAA (Airflow) — designed for complex DAGs

PARALLEL EXECUTION WITHIN WORKFLOW:
→ One trigger can start MULTIPLE jobs/crawlers simultaneously
→ Example: trigger-start → [Job 1A, Job 1B, Job 1C] in parallel
→ Next trigger: CONDITIONAL on ALL three succeeding (AND logic)
→ Useful for extracting multiple independent sources in parallel

COST:
→ Glue Workflow itself: FREE (no additional charge)
→ You only pay for the Crawlers and Jobs that run within it
→ Crawler: $0.44/DPU-hour (same as jobs)
→ Typical crawler run: 2-5 minutes = ~$0.07-$0.15 per run
→ Adding 2 crawlers to pipeline: ~$0.30/day extra

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY WORKFLOW IS WORKING:

1. CONSOLE VISUALIZATION
   → Glue → Workflows → Select workflow → Graph tab
   → Shows visual DAG with status of each node
   → Green = succeeded, Red = failed, Gray = not yet run
   → BEST way to debug pipeline issues

2. WORKFLOW RUN HISTORY
   → Glue → Workflows → Select → History tab
   → Shows all runs with start time, end time, status
   → Click run → see detailed node-by-node execution

3. API MONITORING
   → get_workflow_run() returns full execution graph
   → Each node has: Type, Name, State, StartedOn, CompletedOn
   → Programmatic monitoring for alerting

4. CLOUDWATCH METRICS
   → Glue publishes: glue.workflow.run.status
   → Set alarm on: status != COMPLETED for > 0 runs
   → Covered in Project 54

5. NODE-LEVEL DEBUGGING
   → Click any node in workflow graph
   → Links to: Job Run details, Crawler Run details
   → From there: CloudWatch Logs for full debugging

🔑 KEY TAKEAWAY

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ Glue Workflows chain Crawlers + Jobs with dependency management
→ SCHEDULED trigger starts the pipeline, CONDITIONAL triggers chain steps
→ Triggers MUST be ACTIVATED after creation (default state is CREATED)
→ CONDITIONAL triggers evaluate: predecessor state == SUCCEEDED
→ If ANY step fails → downstream triggers never fire → pipeline stops safely
→ Workflow Run Properties pass shared parameters to ALL child jobs
→ COMPLETED workflow status does NOT mean all jobs succeeded — check nodes
→ Crawlers between jobs ensure Catalog is fresh for downstream consumers
→ CRAWL_NEW_FOLDERS_ONLY makes crawlers fast for incremental pipelines
→ Pause/Resume via trigger activation — no need to delete and recreate
→ Re-run historical dates by starting workflow with custom RunProperties
→ Glue Workflows are FREE — you only pay for the Crawlers and Jobs
