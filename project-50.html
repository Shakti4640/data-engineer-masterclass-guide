Step 2: Understand Dead Letter Queue (DLQ) mechanics

  DLQ = a SEPARATE SQS queue that receives "poison pill" messages
  
  Configuration:
  → Source queue: quickcart-process-q
  → DLQ: quickcart-process-q-dlq
  → Redrive policy: maxReceiveCount = 3
  
  How it works:
  1. Message arrives in source queue
  2. Consumer receives message (receive count = 1)
  3. Consumer fails, doesn't delete → message returns to queue
  4. Consumer receives again (receive count = 2)
  5. Consumer fails again → message returns
  6. Consumer receives again (receive count = 3)
  7. Consumer fails again → receive count EXCEEDS maxReceiveCount
  8. SQS MOVES message to DLQ automatically
  9. Message sits in DLQ until:
     a. Someone processes/deletes it manually
     b. DLQ retention period expires (set to 14 days)
  
  KEY: message is PRESERVED in DLQ, not lost

Step 3: Design the complete solution

  Components:
  1. DLQ queue for each source queue
  2. CloudWatch alarm on DLQ message count (> 0 → alert)
  3. SNS notification when DLQ receives messages
  4. DLQ inspection utility (view failed messages)
  5. DLQ redrive utility (move messages back to source for retry)
  6. DLQ monitoring dashboard

🔵 WHAT COULD BE DONE

Option A: Increase SQS retention to 14 days (no DLQ)
→ Messages stay in source queue longer
→ Problem: failed messages block the queue (re-processed endlessly)
→ Problem: no separation between good and bad messages
→ Problem: no alerting on failures
→ VERDICT: Masks the problem, doesn't solve it

Option B: Application-level retry with error table
→ Worker catches error → writes to DynamoDB "failed_messages" table
→ Separate Lambda retries from DynamoDB
→ Problem: requires code changes in every consumer
→ Problem: if worker crashes before logging → message still lost
→ Problem: reinventing what DLQ already provides
→ VERDICT: Unnecessary complexity when SQS DLQ exists

Option C: SQS Dead Letter Queue (native feature)
→ SQS automatically moves failed messages to DLQ
→ No code changes in consumer (consumer doesn't know about DLQ)
→ Built-in: CloudWatch metrics, redrive capability
→ Zero operational overhead (SQS manages the move)
→ VERDICT: ✅ Perfect — native, simple, reliable

Option D: SQS DLQ + Lambda auto-processor
→ DLQ triggers Lambda that auto-retries with exponential backoff
→ Good for transient errors (quota exceeded, network timeout)
→ Problem: infinite retry loop for permanent errors (bad file format)
→ VERDICT: Good enhancement ON TOP of Option C, not a replacement

DECISION: Option C (SQS DLQ) + CloudWatch alerting + manual redrive utility
→ Option D (Lambda auto-retry) added later if needed

✅ WHAT SHOULD DO

1. Create DLQ for EVERY source queue in the pipeline
   → quickcart-process-q → quickcart-process-q-dlq
   → quickcart-notification-q → quickcart-notification-q-dlq
   → Convention: {source-queue-name}-dlq

2. Set maxReceiveCount based on error type expectations
   → Transient errors (network, quota): maxReceiveCount = 5
     (give more retries — likely to succeed eventually)
   → Mixed errors: maxReceiveCount = 3 (balanced)
   → Permanent errors likely: maxReceiveCount = 2
     (fail fast, don't waste retries)
   → QuickCart: maxReceiveCount = 3 (mix of transient + permanent)

3. Set DLQ retention to 14 days (maximum)
   → Gives engineering team 2 weeks to investigate and fix
   → After 14 days: messages deleted (should be resolved by then)
   → If not resolved in 14 days: bigger organizational problem

4. Set CloudWatch alarm on DLQ ApproximateNumberOfMessagesVisible > 0
   → Alert on FIRST failed message (don't wait for accumulation)
   → SNS notification to data team
   → Include: queue name, message count, timestamp

5. Do NOT change consumer code
   → Consumer doesn't know about DLQ — SQS handles the move
   → Consumer continues: receive → process → delete (on success)
   → Consumer continues: receive → fail → don't delete (on failure)
   → DLQ is TRANSPARENT to the consumer

6. Build a DLQ inspection utility
   → View messages in DLQ without consuming them (peek)
   → Show: message body, attributes, approximate receive count
   → Show: when message was first sent, when it entered DLQ

7. Build a DLQ redrive utility
   → Move messages from DLQ back to source queue (for retry)
   → Use SQS StartMessageMoveTask API (native since 2023)
   → Or: manual receive from DLQ → send to source → delete from DLQ

8. Create a DLQ monitoring dashboard
   → Per-queue: messages in DLQ over time
   → Trend: increasing DLQ messages = systemic issue
   → Age: oldest message in DLQ (if > 7 days → escalate)

❌ WHAT SHOULD NOT DO

1. DO NOT use the same queue as both source and DLQ
   → SQS requires DLQ to be a SEPARATE queue
   → Using same queue: messages loop forever
   → API will reject this configuration

2. DO NOT set maxReceiveCount = 1
   → First failure → immediately to DLQ (no retry)
   → Many errors are transient (network blip, throttling)
   → One retry would have succeeded
   → Minimum recommended: 2–3

3. DO NOT set DLQ retention shorter than investigation time
   → If DLQ retention = 1 day → messages deleted before team notices
   → Weekend failures: nobody looks until Monday
   → 14 days = safe for most organizations

4. DO NOT auto-delete DLQ messages without investigation
   → "Just purge the DLQ every night" = hiding failures
   → Each DLQ message represents LOST DATA
   → Investigate root cause before deleting

5. DO NOT mix DLQ types (Standard vs FIFO)
   → Standard source queue → Standard DLQ
   → FIFO source queue → FIFO DLQ
   → Mismatched types: configuration error

6. DO NOT forget DLQ for FIFO queues
   → FIFO queues ALSO support DLQs
   → Same maxReceiveCount mechanism
   → Same redrive capability
   → FIFO DLQ preserves message group ordering

7. DO NOT rely solely on DLQ — fix root causes
   → DLQ is a SAFETY NET, not a solution
   → If 50 messages/day hit DLQ → systemic problem
   → Fix: increase Glue quota, fix file format validation, etc.
   → DLQ buys time, not a permanent workaround

8. DO NOT process DLQ messages with the same consumer
   → Source queue consumer may have the same bug
   → DLQ processor should: log details, alert, optionally retry with fixes
   → Or: human reviews DLQ messages, fixes issue, then redrives

🚫 WHEN NOT TO USE SQS DLQ

1. When failure is expected and acceptable
   → Some messages are intentionally skipped (filtered by consumer)
   → Consumer receives, checks condition, deletes without processing
   → Not a failure — just a filter
   → DLQ would fill with "valid" skipped messages → noise

2. When you need exactly-once processing guarantee
   → DLQ + redrive can result in duplicate processing
   → Message processed partially → DLQ → redrive → processed again
   → Need idempotent consumers OR deduplication logic
   → DLQ doesn't guarantee exactly-once

3. When messages have no value after a short time
   → Real-time alerts: "Server CPU > 90% at 14:05"
   → By the time DLQ is reviewed (hours later): alert is stale
   → Don't quarantine time-sensitive messages — log and discard

4. When the consumer should never fail
   → If consumer failure = critical bug that needs immediate fix
   → DLQ might mask the urgency ("it's in DLQ, we'll look later")
   → Better: fail loudly, page on-call, fix immediately

5. When using Kinesis or Kafka instead of SQS
   → Kinesis: use enhanced fan-out + checkpoint management
   → Kafka: use built-in DLT (Dead Letter Topic)
   → SQS DLQ is SQS-specific — don't try to adapt it to other systems

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "DLQ has messages but nobody noticed for 5 days"
→ Cause: no CloudWatch alarm configured on DLQ
→ DLQ is passive — it just holds messages, doesn't alert by itself
→ Fix: CloudWatch alarm on ApproximateNumberOfMessagesVisible > 0
→ Alert via SNS → email + Slack

PITFALL 2: "Redrive sends messages back but they fail AGAIN"
→ Cause: root cause not fixed before redriving
→ Message goes: source → DLQ → redrive → source → DLQ (infinite loop)
→ Fix: ALWAYS fix root cause FIRST, then redrive
→ If unsure: process DLQ messages manually, don't redrive blindly

PITFALL 3: "DLQ retention expired — messages lost before investigation"
→ Cause: DLQ retention set to 4 days (default), team took 5 days to investigate
→ Fix: set DLQ retention to 14 days (maximum)
→ 14 days = 2 weekends of buffer

PITFALL 4: "maxReceiveCount=1 and every transient error goes to DLQ"
→ Cause: network blip causes one failure → immediately to DLQ
→ DLQ fills with messages that would have succeeded on retry
→ Fix: maxReceiveCount = 3-5 for queues with transient errors
→ Save DLQ for truly failed messages

PITFALL 5: "Consumer processes message but crashes before DeleteMessage"
→ Cause: processing succeeds, but DeleteMessage call fails
→ VisibilityTimeout expires → message re-processed (duplicate)
→ After maxReceiveCount: goes to DLQ (even though it was processed!)
→ Fix: make consumers IDEMPOTENT (processing same message twice = OK)
→ Fix: delete message as SOON as processing completes (before cleanup)

PITFALL 6: "DLQ message body doesn't contain enough context"
→ Cause: original message was just "process file X"
→ DLQ message: "process file X" — but WHY did it fail?
→ Error details are in consumer logs, not in the message
→ Fix: enrich messages with error context before DLQ
→ Or: correlate DLQ message timestamp with CloudWatch consumer logs

PITFALL 7: "FIFO source queue with Standard DLQ"
→ Cause: FIFO and Standard queue types are incompatible for DLQ
→ SQS rejects the redrive policy configuration
→ Fix: FIFO source → FIFO DLQ, Standard source → Standard DLQ

🔑 KEY TAKEAWAY

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ SQS DLQ: separate queue that receives messages after maxReceiveCount failures
→ DLQ is TRANSPARENT to consumers — no code changes needed
→ Redrive policy: set on SOURCE queue, points to DLQ ARN + maxReceiveCount
→ maxReceiveCount: 3-5 for mixed errors (transient + permanent)
→ DLQ retention: 14 days (maximum) — gives time to investigate
→ CloudWatch alarms: ApproximateNumberOfMessagesVisible > 0 → immediate alert
→ ApproximateAgeOfOldestMessage: track how long failures sit uninvestigated
→ Native redrive (StartMessageMoveTask): SQS moves messages back atomically
→ Manual redrive: receive from DLQ → send to source → delete from DLQ
→ VisibilityTimeout must be >= max processing time (prevents premature DLQ)
→ Delete marker replication disabled: DLQ preserves even if source deletes
→ DLQ cost: essentially $0 (few messages, within free tier)
→ DLQ is a SAFETY NET — always fix root causes, don't just redrive blindly

━━━━━━━━━━━━━━━━━━━━━

BEFORE vs AFTER:
━━━━━━━━━━━━━━
  BEFORE (no DLQ):
  → Message fails 3 times → disappears forever
  → Nobody notified → data silently lost
  → Discovery: days/weeks later during quality check
  → Recovery: manual S3 listing, re-submission (hours of work)

  AFTER (with DLQ):
  → Message fails 3 times → quarantined in DLQ
  → CloudWatch alarm → SNS → team notified immediately
  → Inspection: see what failed and why (minutes)
  → Recovery: fix cause → redrive → automatic reprocessing
