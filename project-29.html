🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — MariaDB has rich operational data
Situation:
→ MariaDB holds 18 months of operational data:
   - orders: 10M+ rows (growing 333K/day)
   - customers: 50K rows (growing ~100/day)
   - products: 5K rows (changes weekly)
   - shipfast_updates: 500K rows (growing ~5K/day from Project 28)

→ Project 20 created a ONE-TIME manual export script:
   - Connect to MariaDB → SELECT → write CSV → upload to S3
   - Works but: run manually, no scheduling, no error handling
   - Data engineer must SSH into EC2 and run it

→ Analytics team needs FRESH data in S3 every morning:
   - Athena queries (Project 24) on yesterday's data
   - Glue ETL converts to Parquet (Project 25)
   - Redshift COPY for dashboard refresh (Project 21)
   - All downstream systems depend on S3 having current data

→ Current pain:
   - Engineer forgets to run export → Athena has stale data
   - Engineer runs it at different times → inconsistent snapshots
   - Export fails silently → nobody knows until analyst complains
   - No record of what was exported when

→ Requirements:
   - Run EVERY night at 2:00 AM UTC (after midnight data cutoff)
   - Export ALL tables to S3 with date-stamped paths
   - Log everything: start time, row counts, duration, errors
   - Alert via SNS on failure (Project 12)
   - Alert via SNS on success (with metrics)
   - Idempotent: re-running for same date overwrites cleanly
   - Track export history for auditing

CTO: "I never want to hear 'the data is stale' again.
      Automate the export. If it fails, wake someone up."

🟡 HOW TO APPROACH IT

Step 1: Understand what needs to be scheduled
→ Python script that: connects to MariaDB → exports tables → uploads to S3

→ Needs: scheduling, error handling, logging, notifications

Step 2: Choose the scheduling mechanism
→ Option A: Linux crontab on EC2 (simplest)
→ Option B: AWS EventBridge + Lambda (serverless)

Step 3: Design the S3 output structure
→ Date-stamped paths enable:
   - Historical snapshots (what did data look like on Jan 15?)
   - Partition-compatible paths for Athena (Project 26)
   - Safe re-runs (overwrite same date, don't append)
→ Pattern: s3://bucket/mariadb_exports/{table}/export_date=YYYY-MM-DD/

Step 4: Design error handling and recovery
→ MariaDB connection fails → retry 3 times → alert
→ S3 upload fails → retry 3 times → alert
→ Partial export (2 of 4 tables succeed) → alert with details
→ Export takes > 30 minutes → timeout → alert
→ Cron doesn't fire (EC2 rebooted) → monitoring catches stale data

Step 5: Design observability
→ Log file: /var/log/mariadb-export/export_YYYYMMDD.log
→ Metrics to CloudWatch: rows exported, duration, success/failure
→ SNS notification: summary email after each run
→ Export manifest: JSON file in S3 documenting what was exported

Option A: Linux crontab + Python script (selected)
→ Simplest, most reliable, zero AWS dependencies for scheduling
→ Works even if AWS APIs are temporarily unavailable
→ Easy to debug: SSH in, run manually, check logs
→ Limitation: tied to single EC2 instance

Option B: EventBridge + Lambda
→ Serverless scheduling — no EC2 needed for scheduler

→ BUT: Lambda 15-min timeout — 10M row export may exceed
→ BUT: Lambda → MariaDB requires VPC config
→ Better for: lightweight scheduled tasks

Option C: EventBridge + Step Functions + Lambda/ECS
→ Full orchestration: parallel table exports, retry per table
→ Best for: complex multi-step workflows

Option D: Managed Airflow (MWAA)
→ Full DAG-based workflow engine
→ $0.49/hour minimum (~$360/month)

DECISION: Option A — crontab + Python
→ Gets the job done reliably
→ Foundation for understanding scheduling patterns
→ Easy to migrate to EventBridge/Step Functions later

✅ WHAT SHOULD DO

1. USE date-stamped S3 paths (Hive-style partitions)
   → export_date=2025-01-30/ not just 2025-01-30/
   → Athena can query these as partitioned tables (Project 26)
   → Glue Crawler auto-detects partition structure (Project 23)
   → Re-running for same date overwrites same folder (idempotent)

2. COMPRESS exports with GZIP before uploading
   → 10M order rows: ~800 MB CSV → ~120 MB GZIP
   → Reduces: upload time, S3 storage cost, downstream read time
   → Athena reads GZIP natively (no decompression needed)

3. WRITE export manifest after each run
   → JSON file documenting: tables exported, row counts, sizes, timestamps
   → Stored in S3: _manifests/export_2025-01-30.json
   → Enables: auditing, debugging, downstream validation

4. USE server-side cursor for large tables
   → Default pymysql: loads ALL rows into memory
   → 10M rows × 100 bytes = 1 GB in Python memory
   → Server-side cursor: streams rows in batches
   → pymysql.cursors.SSCursor → fetches rows on demand

5. SEND SNS notification after EVERY run (success or failure)
   → Success: "Exported 4/4 tables, 10.5M rows, 142 MB in 8m 32s"
   → Failure: "Exported 2/4 tables — orders FAILED: Connection refused"
   → Enables: team awareness without checking logs

6. LOG everything to both file and stdout
   → File: /var/log/mariadb-export/export_YYYYMMDD.log
   → Stdout: captured by cron (mailed to root or redirected)
   → Include: timestamps, row counts, durations, errors

7. USE wrapper shell script for crontab
   → Sets PATH, environment variables, virtualenv
   → Redirects output to log file
   → Captures exit code for monitoring
   → Cron environment is MINIMAL — don't rely on .bashrc

8. IMPLEMENT retry logic per table
   → If orders export fails: retry 3 times with backoff
   → If still fails: skip orders, continue with customers
   → Report: which tables succeeded, which failed
   → Don't let one table failure block all others

❌ WHAT SHOULD NOT DO

1. DO NOT put the Python script directly in crontab
   → Cron has minimal PATH — Python may not be found
   → No environment variables — boto3 can't find credentials
   → Use wrapper shell script that sets up environment

2. DO NOT export without date-stamping the S3 path
   → Without dates: each export overwrites previous
   → Can't answer: "What did data look like last Tuesday?"
   → Can't use as Athena partitioned table
   → Always include date in S3 key path

3. DO NOT load entire large table into memory
   → SELECT * FROM orders → 10M rows → pymysql loads ALL into memory
   → EC2 with 4 GB RAM → OOM kill
   → Use: SSCursor (server-side cursor) or LIMIT/OFFSET pagination
   → Or: export in date chunks (WHERE order_date = ?)

4. DO NOT ignore partial failures
   → Export 4 tables: 3 succeed, 1 fails
   → If script exits on first failure → 3 good exports lost
   → Continue all tables → report which succeeded/failed
   → Downstream can use partial data (3 of 4 tables refreshed)

5. DO NOT rely on cron email for monitoring
   → Cron sends email to local root mailbox
   → Nobody checks EC2 local mail
   → Use: SNS notification + CloudWatch alarm

6. DO NOT schedule exports during peak database hours
   → SELECT * on 10M rows → full table scan
   → Locks may affect application writes
   → Schedule at 2 AM when traffic is lowest
   → Consider: read replica for exports (if available)

7. DO NOT forget to handle timezone
   → Cron runs in EC2's system timezone (usually UTC)
   → Business dates may be in US/Eastern
   → Export "yesterday's data" must account for timezone
   → Always use UTC internally → convert for display

8. DO NOT leave old exports forever
   → 365 days × 4 tables × 150 MB = ~219 GB/year
   → Set S3 lifecycle: move to Glacier after 90 days
   → Delete after 365 days (or per retention policy)
   → Lifecycle rules from Project 4

🚫 WHEN NOT TO USE CRON + SCRIPT

1. When you need sub-minute scheduling precision
   → Cron minimum granularity: 1 minute
   → For second-level: use EventBridge rate(30 seconds)

2. When the EC2 instance may not be running
   → EC2 stopped/terminated → cron doesn't fire
   → Use: EventBridge (runs regardless of EC2 state)
   → Or: ensure EC2 is always running (costly)

3. When you have 20+ scheduled jobs with dependencies
   → Crontab becomes unmanageable at scale
   → Job A must finish before Job B starts
   → Cron can't express dependencies
   → Use: Airflow, Step Functions, or Glue Workflows

4. When you need built-in retry with exponential backoff
   → Cron fires once — if script fails, it's done until next schedule
   → Must build retry INTO the script (we do this)
   → EventBridge + Step Functions have native retry policies

5. When export must run even if EC2 is replaced
   → Auto Scaling Group may terminate/launch new instances
   → Cron config lost on termination (unless in AMI or user data)
   → Use: EventBridge + Lambda or ECS task for persistence

┌─────────────────────────────────────────────────────────────────────────────┐
│                CRON + MARIADB EXPORT — INTERNAL MECHANICS                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  CRON DAEMON MECHANICS:                                                     │
│  ──────────────────────                                                     │
│  1. crond daemon runs continuously on EC2                                   │
│  2. Every minute: checks all crontab entries                                │
│  3. Matches current time against schedule patterns                          │
│  4. If match: forks child process to execute command                        │
│  5. Child process runs with MINIMAL environment:                            │
│     → PATH = /usr/bin:/bin (no /usr/local/bin, no custom paths)             │
│     → HOME = user's home directory                                          │
│     → No .bashrc, .profile, .bash_profile loaded                            │
│     → No virtualenv activated                                               │
│     → No AWS credentials from instance profile (sometimes)                  │
│  6. This is WHY we need a wrapper shell script                              │
│                                                                             │
│  CRON EXPRESSION: 0 2 * * *                                                 │
│  ┌───────────── minute (0)                                                  │
│  │ ┌─────────── hour (2)                                                    │
│  │ │ ┌───────── day of month (* = every)                                    │
│  │ │ │ ┌─────── month (* = every)                                           │
│  │ │ │ │ ┌───── day of week (* = every)                                     │
│  0 2 * * *                                                                  │
│  → Runs at 02:00 every day                                                  │
│                                                                             │
│                                                                             │
│  MARIADB EXPORT MECHANICS:                                                  │
│  ─────────────────────────                                                  │
│                                                                             │
│  SMALL TABLE (< 100K rows):                                                 │
│  → Standard cursor: SELECT * → all rows loaded into Python memory           │
│  → Write to CSV file → upload to S3                                         │
│  → Memory: ~50 MB for 100K rows                                             │
│  → Duration: 5-10 seconds                                                   │
│                                                                             │
│  LARGE TABLE (10M+ rows):                                                   │
│  → Server-side cursor (SSCursor):                                           │
│    1. Python sends: SELECT * FROM orders                                    │
│    2. MariaDB executes query, holds result on SERVER                        │
│    3. Python fetches in batches: cursor.fetchmany(10000)                    │
│    4. Each batch: 10K rows loaded → written to CSV → memory freed           │
│    5. Next batch fetched from server                                        │
│    6. Repeat until all rows consumed                                        │
│  → Memory: ~10 MB constant (regardless of table size)                       │
│  → Duration: 5-15 minutes for 10M rows                                      │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────┐        │
│  │  STANDARD CURSOR vs SERVER-SIDE CURSOR:                         │        │
│  │                                                                 │        │
│  │  Standard (default pymysql):                                    │        │
│  │  SELECT * → MariaDB sends ALL rows → Python stores ALL in RAM   │        │
│  │  10M rows × 100 bytes = 1 GB RAM consumed IMMEDIATELY           │        │
│  │  ❌ OOM kill on 4 GB EC2 instance                               │        │
│  │                                                                 │        │
│  │  Server-side (SSCursor):                                        │        │
│  │  SELECT * → MariaDB holds results on disk/memory                │        │
│  │  Python fetches 10K rows at a time → processes → fetches next   │        │
│  │  Peak memory: ~10 MB (one batch)                                │        │
│  │  ✅ Works on any EC2 instance size                              │        │
│  │                                                                 │        │
│  │  TRADE-OFF:                                                     │        │
│  │  → SSCursor holds MariaDB connection open longer                │        │
│  │  → Other queries may wait for locks                             │        │
│  │  → Run during low-traffic window (2 AM)                         │        │
│  └─────────────────────────────────────────────────────────────────┘        │
│                                                                             │
│  GZIP COMPRESSION DURING WRITE:                                             │
│  ──────────────────────────────                                             │
│  → Don't write CSV then compress → doubles disk I/O                         │
│  → Instead: write directly to gzip.open() stream                            │
│  → Each row → CSV formatted → GZIP compressed → written to disk            │
│  → Output: file.csv.gz (compressed as it's written)                         │
│  → Compression ratio: 5-8x for CSV text                                    │
│  → 800 MB CSV → 120 MB GZIP                                                │
│                                                                             │
│  S3 UPLOAD WITH MULTIPART:                                                  │
│  ──────────────────────────                                                 │
│  → Files > 100 MB: use multipart upload                                     │
│  → boto3 upload_file() handles this automatically                           │
│  → Splits file into 8 MB chunks → uploads in parallel                       │
│  → If one chunk fails → retries that chunk only                             │
│  → All chunks uploaded → S3 assembles into final object                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Cron job runs but Python not found"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: cron's PATH doesn't include /usr/local/bin or virtualenv
→ Fix: use FULL path to Python in wrapper script
→ /opt/venv/bin/python /opt/scripts/export.py (not just "python")

PITFALL 2: "boto3 can't find credentials in cron"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: EC2 instance profile works via metadata service
→ Sometimes cron environment blocks metadata access
→ Fix: ensure IAM role is attached to EC2 (Project 6)
→ Fix: set AWS_DEFAULT_REGION in wrapper script
→ Test: run wrapper script manually first

PITFALL 3: "Export OOM-killed for large table"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: standard cursor loads entire table into Python memory
→ 10M rows × 100 bytes = 1 GB → exceeds EC2 memory
→ Fix: use SSCursor (server-side cursor)
→ Or: export in chunks (WHERE order_date BETWEEN ? AND ?)

PITFALL 4: "Export takes 45 minutes — overlaps with next job"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: table grew larger than expected
→ Fix: add timeout to script (30-minute max)
→ Fix: use lock file to prevent concurrent runs
→ Fix: optimize query (add indexes, use read replica)

PITFALL 5: "S3 upload succeeds but file is corrupt"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: script crashed during GZIP write → partial file
→ S3 received whatever was uploaded → looks like valid object
→ Fix: write to temp file → verify → rename → upload
→ Fix: check uploaded size matches local file size

PITFALL 6: "Cron fires but EC2 was rebooted — missed last night"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: EC2 restarted at 1:55 AM → cron starts at 2:01 AM
→ But: crond doesn't know it missed the 2:00 AM window
→ Fix: monitoring script checks S3 for today's export
→ If missing by 4 AM → SNS alert → manual re-run
→ Or: add @reboot cron entry to run export on startup

📈 SCALABILITY CONSIDERATIONS

TABLE SIZE SCALING:
━━━━━━━━━━━━━━━━━━
→ 100K rows:    standard cursor, 5 sec export,    ~10 MB CSV
→ 1M rows:      SSCursor recommended, 30 sec,     ~100 MB CSV
→ 10M rows:     SSCursor required, 5-10 min,      ~800 MB CSV
→ 100M rows:    chunk by date, 30-60 min,         ~8 GB CSV
→ 1B rows:      use DMS or Glue JDBC (Project 31, 79)

EXPORT OPTIMIZATION:
━━━━━━━━━━━━━━━━━━━━
→ Level 0: SELECT * (full table scan every night) — simple but slow
→ Level 1: SELECT WHERE date >= yesterday (incremental) — faster
→ Level 2: Glue JDBC with Job Bookmarks (Project 31, 33) — managed
→ Level 3: DMS CDC (Project 79) — real-time, zero export needed

STORAGE GROWTH:
━━━━━━━━━━━━━━━
→ 4 tables × 150 MB GZIP/day = 600 MB/day
→ 600 MB × 365 days = 219 GB/year
→ S3 Standard: 219 GB × $0.023 = $5.04/month after 1 year
→ With Glacier lifecycle (90-day transition): ~$1.50/month
→ Lifecycle rules from Project 4

CONCURRENT EXPORTS:
━━━━━━━━━━━━━━━━━━━
→ Export tables sequentially: simple, predictable
→ Export tables in parallel: faster but more DB load
→ For 4 tables: sequential is fine (10-15 min total)
→ For 20+ tables: consider parallel with thread pool
