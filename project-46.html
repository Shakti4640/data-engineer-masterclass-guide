🟡 HOW TO APPROACH IT

Step 1: Understand Athena Federated Query architecture

  NORMAL ATHENA (Projects 16, 24, 26):
  ┌──────────┐      ┌──────────────┐      ┌──────┐
  │ Athena   │─────→│ Glue Catalog │─────→│  S3  │
  │ (SQL)    │      │ (schema)     │      │(data)│
  └──────────┘      └──────────────┘      └──────┘

  FEDERATED ATHENA (this project):
  ┌──────────┐      ┌──────────────┐      ┌──────────────┐      ┌─────────┐
  │ Athena   │─────→│ Data Source  │─────→│ Lambda       │─────→│ MariaDB │
  │ (SQL)    │      │ Catalog      │      │ Connector    │      │ (live)  │
  └──────────┘      └──────────────┘      └──────────────┘      └─────────┘

  How it works:
  1. Analyst writes: SELECT * FROM mariadb.quickcart.orders WHERE order_date = '2025-01-15'
  2. Athena recognizes "mariadb" as a federated data source (not Glue Catalog)
  3. Athena invokes the Lambda connector with the query
  4. Lambda connector:
     a. Connects to MariaDB via JDBC
     b. Pushes down the WHERE clause (predicate pushdown)
     c. Executes: SELECT * FROM orders WHERE order_date = '2025-01-15'
     d. Reads results from MariaDB
     e. Converts to Apache Arrow format
     f. Writes Arrow batches to S3 spill bucket (if results > Lambda memory)
     g. Returns results to Athena
  5. Athena presents results to analyst as normal query output

🔵 WHAT COULD BE DONE

Option A: Give analysts direct MariaDB access (MySQL Workbench)
→ Simple: just create read-only MariaDB user
→ Problem: analysts must learn MySQL syntax (not standard SQL)
→ Problem: can't JOIN with S3/Gold zone data
→ Problem: no query cost tracking, no workgroup isolation
→ Problem: one bad query can lock MariaDB tables
→ VERDICT: Dangerous for production database

Option B: Create a MariaDB read replica, give analysts access to replica
→ Better: queries don't impact production
→ Problem: still can't join with S3 data
→ Problem: managing another database instance
→ Problem: no unified SQL interface
→ VERDICT: Good isolation, but not integrated with data lake

Option C: Run hourly CDC (reduce latency from 13.5h to 1h)
→ Reduce CDC interval from nightly to hourly
→ Problem: 1 hour still too slow for real-time fraud investigation
→ Problem: 24× more Glue job runs = 24× more cost
→ Problem: doesn't solve ad-hoc exploration use case
→ VERDICT: Helps latency, doesn't solve all scenarios

Option D: Athena Federated Query via Lambda connector
→ ✅ Real-time: queries live MariaDB data
→ ✅ Unified: same Athena interface as S3/Gold queries
→ ✅ Cross-source JOINs: MariaDB + S3 in single query
→ ✅ Managed: Lambda auto-scales, no server management
→ ✅ Secure: credentials in Secrets Manager, VPC networking
→ Trade-off: slower than native Athena on S3, loads MariaDB
→ VERDICT: Best for ad-hoc + real-time queries

DECISION: Option D — Athena Federated Query
→ Use for: ad-hoc exploration, real-time investigations, urgent queries
→ NOT for: scheduled dashboards (use Gold zone — Project 44)
→ NOT for: heavy analytical workloads (use Redshift — Project 10)

✅ WHAT SHOULD DO

1. Store MariaDB credentials in Secrets Manager
   → Never hardcode in Lambda environment variables
   → Lambda connector reads from Secrets Manager at runtime
   → Secrets can be rotated without redeploying Lambda (Project 70)

2. Deploy Lambda in the SAME VPC as MariaDB
   → Lambda needs network access to MariaDB on port 3306
   → Place Lambda in private subnet with route to MariaDB subnet
   → Security group: allow Lambda SG → MariaDB SG on port 3306

3. Create a READ-ONLY MariaDB user for federated queries
   → GRANT SELECT ON quickcart.* TO 'athena_reader'@'%'
   → No INSERT, UPDATE, DELETE permissions
   → Prevents accidental data modification via federated query

4. Configure S3 spill bucket for large results
   → Lambda memory: max 10 GB
   → If query returns > Lambda memory: spills to S3 temporarily
   → Spill bucket lifecycle: delete after 1 day

5. Use Athena Workgroups to control federated query access
   → Workgroup "federated-adhoc": allowed to use federated sources
   → Workgroup "dashboard-prod": only allowed Glue Catalog (S3)
   → Prevents dashboards from accidentally querying MariaDB live

6. Enable predicate pushdown
   → WHERE clauses pushed to MariaDB (not filtered in Lambda)
   → Without pushdown: Lambda reads ENTIRE table → filters in memory
   → With pushdown: MariaDB reads only matching rows → much faster

7. Set Lambda concurrency limit
   → Each concurrent Lambda = one MariaDB connection
   → MariaDB max_connections: 151 (default)
   → Lambda reserved concurrency: 10 (prevents connection exhaustion)
   → 10 concurrent federated queries max

❌ WHAT SHOULD NOT DO

1. DO NOT use federated queries for scheduled dashboards
   → Dashboard refreshes every 15 minutes → 96 queries/day on MariaDB
   → Each query scans production database
   → Use Gold zone in S3 for dashboards (faster, cheaper, no DB load)
   → Federated = ad-hoc only

2. DO NOT run SELECT * without WHERE on large tables
   → SELECT * FROM mariadb.quickcart.orders (5.2M rows)
   → Lambda tries to read 5.2M rows via JDBC → timeout
   → MariaDB under heavy read load → production app slows down
   → ALWAYS use WHERE clause to limit results

3. DO NOT join large federated tables with large S3 tables
   → JOIN mariadb.orders (5.2M) WITH gold.customer_360 (500K)
   → Lambda reads 5.2M rows → Athena reads 500K rows → JOIN in engine
   → Extremely slow and expensive
   → Better: filter federated side first (WHERE), then join

4. DO NOT deploy Lambda outside the VPC
   → Lambda outside VPC cannot reach MariaDB on private IP
   → Common mistake: Lambda works in test (public RDS) but fails in prod (private EC2)
   → Always deploy in same VPC with proper subnet + security group

5. DO NOT share MariaDB production credentials with Lambda
   → Create a SEPARATE read-only user for federated queries
   → If Lambda credentials are compromised: read-only limits damage
   → Production app user has INSERT/UPDATE → never share with Lambda

6. DO NOT skip the spill bucket configuration
   → Without spill bucket: queries returning > Lambda memory FAIL
   → Error: "Spill bucket not configured" or OutOfMemoryError
   → Always configure even if you think results will be small

7. DO NOT set Lambda timeout too short
   → Default Lambda timeout: 3 seconds → federated query ALWAYS fails
   → Federated queries need: 30–300 seconds depending on data volume
   → Set timeout to 900 seconds (15 minutes) for safety

🚫 WHEN NOT TO USE ATHENA FEDERATED QUERY

1. When query runs more than 10 times per day on same data
   → Repeated federated queries = repeated MariaDB load
   → Better: ETL once → query S3 unlimited times
   → Rule of thumb: if query frequency > 10/day, build a pipeline

2. When query scans > 1 million rows from source
   → JDBC row-by-row read is inherently slow
   → 1M rows via JDBC: ~30-60 seconds
   → 1M rows from Parquet on S3: ~2-5 seconds
   → For large scans: use ETL pipeline

3. When source database is at capacity
   → MariaDB CPU at 80% → federated queries push it to 100%
   → Production app starts timing out
   → Use read replica for federated queries if DB is stressed

4. When you need sub-second query response
   → Federated query minimum: ~3-5 seconds (Lambda cold start + JDBC)
   → For sub-second: use DynamoDB, ElastiCache, or Redshift materialized views

5. When query requires complex JOINs across multiple federated sources
   → JOIN mariadb.orders WITH postgres.inventory WITH mongodb.sessions
   → Each source returns full result set → Athena joins in memory
   → Extremely slow for large tables
   → Better: ETL all sources to S3, join there

6. When data must be transformed before querying
   → Federated query returns RAW data (no cleaning, no dedup)
   → If you need: type casting, null handling, deduplication
   → Use the ETL pipeline (Bronze → Silver → Gold)

│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                                        │

│  User submits:                                                                  │
│  SELECT * FROM "mariadb_source".quickcart.orders                               │
│  WHERE order_date = '2025-01-15' AND status = 'completed'                      │
│                                                                                 │
│  Step 1: Athena parses SQL                                                      │
│  → Identifies catalog: "mariadb_source" (not "AwsDataCatalog")                 │
│  → Recognizes this is a FEDERATED source                                        │
│  → Looks up Lambda function ARN for "mariadb_source"                           │
│                                                                                 │
│  Step 2: Athena invokes Lambda — METADATA call                                 │
│  → Request type: "ListSchemas" or "ListTables" or "GetTable"                   │
│  → Lambda connects to MariaDB via JDBC                                          │
│  → Executes: SHOW DATABASES or SHOW TABLES or DESCRIBE orders                  │
│  → Returns schema metadata to Athena (column names, types)                     │
│                                                                                 │
│  Step 3: Athena plans the query                                                 │
│  → Creates execution plan with predicate pushdown                              │
│  → Determines: WHERE order_date = '2025-01-15' can be pushed                   │
│  → Determines: AND status = 'completed' can be pushed                          │
│  → Plans: partition the read into N splits (parallel reads)                    │
│                                                                                 │
│  Step 4: Athena invokes Lambda — SPLIT call                                    │
│  → Request type: "GetSplits"                                                    │
│  → Lambda determines how to parallelize the read                               │
│  → For MySQL: typically 1 split per query (no partitioning)                    │
│  → Returns split metadata to Athena                                            │
│                                                                                 │
│  Step 5: Athena invokes Lambda — READ call (per split)                         │
│  → Request type: "ReadRecords"                                                  │
│  → Lambda receives: table name, column list, WHERE predicates                  │
│  → Lambda constructs SQL:                                                       │
│    SELECT order_id, customer_id, ... FROM orders                               │
│    WHERE order_date = '2025-01-15' AND status = 'completed'                    │
│  → Lambda executes against MariaDB via JDBC                                    │
│  → Reads results row by row                                                    │
│  → Converts to Apache Arrow format (columnar, in memory)                       │
│  → If Arrow batch > Lambda memory limit:                                        │
│    → Writes overflow to S3 spill bucket                                        │
│    → Returns S3 reference to Athena                                            │
│  → If Arrow batch fits in memory:                                               │
│    → Returns Arrow batch directly to Athena                                    │
│                                                                                 │
│  Step 6: Athena combines results                                                │
│  → Receives Arrow data from all splits                                          │
│  → Applies any remaining filters not pushed down                               │
│  → Applies ORDER BY, LIMIT, aggregations                                       │
│  → Writes final results to Athena output location (S3)                         │
│  → Returns results to user                                                      │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Lambda timeout — query never returns"
→ Cause: query without WHERE scans entire table
→ Lambda default timeout: 3 seconds → always fails for DB queries
→ Fix: set Lambda timeout to 900 seconds (15 min)
→ Fix: ALWAYS use WHERE clause to limit data

PITFALL 2: "Lambda can't connect to MariaDB"
→ Cause #1: Lambda not in same VPC as MariaDB
→ Cause #2: Security group doesn't allow Lambda → MariaDB on 3306
→ Cause #3: Lambda subnet has no NAT gateway (can't reach Secrets Manager)
→ Debug: test JDBC connection from Lambda with simple test event
→ Fix: VPC configuration must be exact (subnet, SG, route table)

PITFALL 3: "Secrets Manager access denied from Lambda"
→ Cause: Lambda execution role lacks secretsmanager:GetSecretValue
→ Or: Lambda in VPC without VPC endpoint for Secrets Manager
→ Fix: add VPC endpoint for secretsmanager OR add NAT gateway

PITFALL 4: "Federated query returns different types than expected"
→ Cause: MariaDB TINYINT(1) → connector returns BOOLEAN
→ MariaDB DECIMAL(10,2) → connector may return DOUBLE (precision loss)
→ Fix: cast in Athena SQL: CAST(column AS DECIMAL(10,2))
→ Or: configure type mapping in connector properties

PITFALL 5: "Federated query scans 5M rows — takes 5 minutes"
→ Cause: no predicate pushdown (WHERE clause not pushable)
→ Example: WHERE YEAR(order_date) = 2025 → function on column → not pushed
→ Fix: rewrite as WHERE order_date >= '2025-01-01' AND order_date < '2026-01-01'
→ This is an equality/range predicate → pushed to MariaDB

PITFALL 6: "MariaDB CPU spikes to 100% during federated query"
→ Cause: federated query reads large result set via JDBC
→ MariaDB sorts and transfers millions of rows
→ Fix: limit concurrent federated queries (Lambda reserved concurrency)
→ Fix: use read replica for federated queries (separate from production)

PITFALL 7: "Cross-source JOIN is extremely slow"
→ Cause: JOIN between 5M MariaDB rows and 500K S3 rows
→ Athena must read ALL 5M rows from Lambda, ALL 500K from S3
→ Then hash-join in Athena engine
→ Fix: filter federated side FIRST (subquery or CTE with WHERE)
→ Example: WITH recent_orders AS (SELECT * FROM mariadb...orders WHERE date = today)
           SELECT * FROM recent_orders JOIN gold.customer_360 ...

📈 SCALABILITY CONSIDERATIONS

CURRENT USAGE:
→ 3-5 federated queries per day (ad-hoc only)
→ Lambda concurrency: 1-2 at a time
→ MariaDB impact: negligible
→ Cost: ~$0.01/day

SCALING TO 50 QUERIES/DAY:
→ Lambda concurrency: up to 5 at peak
→ MariaDB additional load: ~5% CPU increase
→ Cost: ~$0.10/day
→ Still manageable

SCALING TO 500 QUERIES/DAY:
→ Lambda concurrency: up to 20 at peak
→ MariaDB additional load: ~30% CPU increase → CONCERNING
→ Solutions:
  a) Route federated queries to MariaDB read replica
  b) Reduce Lambda concurrency limit to 10
  c) Convert most-frequent queries to ETL pipeline
→ Rule: if a specific query runs > 10×/day → build a pipeline for it

LAMBDA LIMITS:
→ Memory: 128 MB – 10,240 MB (10 GB)
→ Timeout: max 900 seconds (15 minutes)
→ Payload: 6 MB (synchronous), 256 KB (async)
→ Concurrent executions: 1000 per account (default)
→ Reserved concurrency for connector: set to 10-20

MARIADB CONNECTION LIMITS:
→ max_connections: 151 (default)
→ Each Lambda invocation: 1 connection
→ Reserved concurrency 10 → max 10 connections used
→ Production app needs: ~50-100 connections
→ Safe headroom: 151 - 100 (app) - 10 (federated) = 41 spare

CROSS-ACCOUNT (Project 64):
→ Athena in Account A can query MariaDB in Account B
→ Requires: cross-account Lambda invocation or shared VPC
→ Adds latency: ~200ms for cross-account Lambda call

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO MONITOR FEDERATED QUERIES:

1. Athena Query History
   → Athena console → Query history → filter by data source
   → Shows: query text, execution time, data scanned, cost
   → Identify: slow queries, expensive queries, frequent queries

2. Lambda CloudWatch Metrics
   → Invocations: how many times connector was called
   → Duration: how long each invocation took
   → Errors: failed invocations (connection timeout, OOM)
   → ConcurrentExecutions: peak concurrent federated queries
   → Alert: if Duration > 60 seconds → query is too broad
   → Alert: if Errors > 0 → connection or configuration issue

3. MariaDB Performance Impact
   → Monitor: Threads_connected, Questions, Slow_queries
   → Compare: before vs after enabling federated queries
   → Alert: if Threads_connected > 120 (approaching max)
   → Alert: if Slow_queries increases after enabling federated

4. Cost Tracking
   → Athena workgroup: separate workgroup for federated queries
   → Cost tag: "QueryType=Federated" on workgroup
   → Monthly report: federated query cost vs S3 query cost
   → Decision: if federated cost > ETL cost → build pipeline

5. Query Pattern Analysis
   → Which tables are queried most via federated?
   → Which WHERE clauses are used?
   → Insight: if same query runs 20×/day → candidate for ETL pipeline
   → Insight: if query has no WHERE → educate user on pushdown

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ Athena Federated Query: SQL-query live databases through Athena
→ Architecture: Athena → Lambda Connector → JDBC → MariaDB → Arrow → Results
→ Predicate pushdown: WHERE clauses sent to MariaDB (critical for performance)
→ Non-pushed predicates: functions on columns, LIKE '%abc%', computed expressions
→ Lambda in VPC: must be in same VPC as MariaDB for network access
→ Secrets Manager: credentials stored securely, read by Lambda at runtime
→ Spill bucket: handles results larger than Lambda memory
→ Reserved concurrency: protects MariaDB from connection exhaustion
→ Cross-source JOINs: combine live MariaDB data with S3 Gold zone in one query
→ Separate workgroup: tracks federated costs independently from S3 queries
→ Monitoring: weekly analysis identifies queries that should become ETL pipelines

DECISION FRAMEWORK:
━━━━━━━━━━━━━━━━━━

  → Query runs < 10×/day → Federated ✅
  → Query runs > 10×/day → Build ETL pipeline
  → Need real-time data → Federated ✅
  → Can tolerate hours-old data → S3 Gold zone
  → Query scans < 100K rows → Federated ✅
  → Query scans > 1M rows → ETL pipeline
  → One-time investigation → Federated ✅
  → Recurring dashboard → S3 Gold zone
