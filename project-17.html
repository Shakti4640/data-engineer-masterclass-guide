🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — now 8 months old
Situation:
→ Daily CSV upload pipeline running (Project 1)
→ 3 files land in S3 every midnight: orders, customers, products
→ Athena queries work great (Project 16)
→ BUT: nobody knows WHEN files actually arrive

What's going wrong:
→ Monday: upload script ran at 00:01 AM — files landed at 00:03 AM
→ Tuesday: EC2 had a cron issue — files landed at 02:47 AM
→ Wednesday: S3 throttled — files landed at 00:01, 00:01, 00:14 (staggered)
→ Thursday: upload script crashed — files NEVER landed

Data analyst arrives at 9 AM:
→ "Are today's files ready? Should I run my Athena queries?"
→ Checks S3 console manually → wastes 10 minutes every morning
→ Sometimes runs queries on YESTERDAY's data thinking today's is there

Operations Manager:
→ "Thursday's files never arrived and nobody noticed until Friday"
→ "We need to KNOW the instant files land — or DON'T land"

CTO: "I want an email the moment each file hits S3.
      And I want to know if a file DOESN'T arrive by 1 AM."

YOU: "S3 can fire events when objects are created.
      We route those events to SNS, which sends emails.
      This is S3 Event Notifications."

🟡 HOW TO APPROACH IT

Step 1: Understand what S3 Event Notifications are
→ S3 can emit events when specific things happen to objects
→ Events: ObjectCreated, ObjectRemoved, ObjectRestore, Replication, etc.
→ These events can be ROUTED to: SNS, SQS, Lambda, or EventBridge

Step 2: Understand the event flow
→ File uploaded to S3 (PutObject / CompleteMultipartUpload)
→ S3 generates an event JSON message
→ S3 sends that message to configured destination (SNS in our case)
→ SNS receives the message
→ SNS delivers to all subscribers (email, SQS, Lambda, HTTP, SMS)
→ Analyst receives email: "orders_20250115.csv just landed in S3"

Step 3: Understand filtering
→ We DON'T want alerts for EVERY file in the bucket
→ Athena result files also land in S3 — ignore those
→ S3 Event Notifications support PREFIX and SUFFIX filters
→ Prefix: "daily_exports/" → only our upload path
→ Suffix: ".csv" → only CSV files (ignore .metadata, .crc, etc.)

Step 4: Understand the permission chain
→ S3 must have PERMISSION to publish to SNS
→ SNS Topic Policy must ALLOW s3.amazonaws.com to sns:Publish
→ Without this: S3 silently drops events — no error, no delivery

Step 5: Plan our implementation
→ Create SNS Topic (if not from Project 12, we create fresh one)
→ Add email subscription to SNS Topic
→ Configure SNS Topic Policy to allow S3
→ Configure S3 Event Notification on the bucket
→ Test by uploading a file
→ Receive email alert

🔵 WHAT COULD BE DONE

Option A: S3 Event → SNS (email alert)
→ Simplest notification mechanism
→ Human-readable emails
→ No code to process events
→ ✅ WE USE THIS — teaches event flow fundamentals

Option B: S3 Event → SQS (queue for processing)
→ Machine-to-machine communication
→ Consumer pulls messages and acts on them
→ Covered in Project 18

Option C: S3 Event → Lambda (instant code execution)
→ Run code the moment file lands
→ More complex — requires Lambda function
→ Covered in later projects

Option D: S3 Event → EventBridge (advanced routing)
→ Most flexible — filter on any attribute
→ Route to 20+ AWS services
→ Covered in Project 41

Option E: S3 Event → SNS → SQS Fan-out (multiple consumers)
→ One event reaches email + queue + Lambda simultaneously
→ Covered in Project 14 (concept), Project 28 (full pipeline)

DECISION: Option A
→ Teaches the core S3→SNS integration
→ Foundation for ALL event-driven patterns that follow
→ Every subsequent project (18, 28, 41) builds on this exact wiring

✅ WHAT SHOULD DO

1. Create SNS Topic BEFORE configuring S3 notification
   → S3 notification config requires Topic ARN
   → Cannot reference a non-existent topic

2. Set SNS Topic Policy BEFORE configuring S3 notification
   → S3 validates permissions during put_bucket_notification_configuration
   → If SNS policy doesn't allow S3 → API call FAILS with:
     "Unable to validate the following destination configurations"

3. Use prefix AND suffix filters together
   → Prefix: "daily_exports/" → targets only our upload path
   → Suffix: ".csv" → ignores .metadata, .crc, _SUCCESS marker files
   → More specific = fewer false alerts

4. Use descriptive Event Name IDs
   → "daily-csv-upload-notification" not "notification1"
   → You'll have multiple notifications on one bucket eventually
   → Clear names prevent confusion

5. Confirm email subscription manually
   → SNS sends confirmation email with a link
   → Subscription stays "PendingConfirmation" until clicked
   → No emails delivered until confirmed
   → Check spam folder — AWS emails often land there

6. Test with a real file upload BEFORE telling the team
   → Upload a test file → verify email arrives
   → Verify email contains useful information (filename, time, size)
   → Then announce the feature

7. Log the S3 event structure for debugging
   → The JSON event S3 sends has specific structure
   → Knowing this structure is essential for Projects 18, 28, 41
   → Parse and print it during testing

❌ WHAT SHOULD NOT DO

1. DO NOT configure S3 notification before SNS Topic Policy is set
   → put_bucket_notification_configuration validates destinations
   → Error: "Unable to validate the following destination configurations"
   → This error is CRYPTIC — doesn't tell you it's a permission issue
   → Fix: always set Topic Policy first

2. DO NOT create overlapping prefix filters on same bucket
   → Two notifications: prefix="daily_exports/" AND prefix="daily_exports/orders/"
   → Same file matches BOTH → TWO events fired → TWO emails
   → Duplicate alerts confuse everyone
   → Rule: no notification prefix should be a subset of another

3. DO NOT assume events are delivered exactly once
   → S3 Event Notifications guarantee AT-LEAST-ONCE delivery
   → Rare but possible: same event delivered twice
   → For SNS email: minor annoyance (2 emails)
   → For SQS/Lambda processing: must handle idempotency (Project 18)

4. DO NOT use S3 events for real-time streaming
   → Event delivery latency: typically seconds, can be minutes
   → Not guaranteed sub-second delivery
   → For true real-time: use Kinesis (Project 72)

5. DO NOT put notification config on Athena results bucket
   → Every Athena query writes result files → fires events
   → 50 queries/day = 100 events/day (result + metadata per query)
   → Email inbox floods with irrelevant alerts

6. DO NOT forget: S3 events are per-object, not per-upload-batch
   → Upload 3 files = 3 separate events = 3 separate emails
   → There is NO "batch complete" event from S3
   → If you need "all 3 files arrived": build that logic (Project 28)

7. DO NOT hardcode Topic ARN — construct from components
   → ARN format: arn:aws:sns:{region}:{account_id}:{topic_name}
   → Account ID can be fetched dynamically via STS
   → Makes code portable across accounts (Project 56+)

🚫 WHEN NOT TO USE S3 EVENT NOTIFICATIONS

1. When you need guaranteed ordering of events
   → S3 events are NOT ordered
   → File A uploaded before File B ≠ Event A delivered before Event B
   → Use: SQS FIFO + application-level sequencing

2. When you need to filter on object content (not just key name)
   → S3 events filter on PREFIX and SUFFIX of the key only
   → Cannot filter: "only CSV files > 10 MB" or "only if column X exists"
   → Use: EventBridge (richer filtering) or Lambda (custom logic)

3. When you need sub-100ms event processing
   → S3 event delivery: typically 1-10 seconds, occasionally minutes
   → Not suitable for real-time trading, gaming, IoT
   → Use: Kinesis Data Streams (millisecond latency)

4. When bucket receives 10,000+ objects/second
   → S3 events work at this scale BUT SNS/SQS must handle the throughput
   → SNS standard: soft limit 30,000 messages/second
   → Consider: EventBridge (higher throughput, better filtering)

5. When you need event replay / reprocessing
   → S3 events are fire-and-forget — no event store
   → If SNS delivery fails → message is lost
   → Use: EventBridge Archive + Replay feature
   → Or: S3 → SQS (retains messages up to 14 days)

6. When you need cross-account event routing with complex rules
   → S3 native events are account-local
   → Use: EventBridge for cross-account event bus (Project 59)

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Unable to validate the following destination configurations"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ THE #1 most common error in S3→SNS setup
→ Cause: SNS Topic Policy does not allow s3.amazonaws.com to publish
→ Fix: set Topic Policy BEFORE calling put_bucket_notification_configuration
→ Also check: Topic ARN is correct, Topic exists in same region as bucket
→ Also check: no typo in region (us-east-2 vs us-east-1)

PITFALL 2: "Notifications configured but no emails arrive"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause A: Email subscription not confirmed (still "PendingConfirmation")
→ Cause B: Emails going to spam folder (AWS emails flagged by some providers)
→ Cause C: Prefix/suffix filter doesn't match uploaded file's key
→ Debug: check SNS subscription status → must be "Confirmed"
→ Debug: upload test file matching exact prefix+suffix → check CloudTrail

PITFALL 3: "I get notifications for files I didn't upload"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: prefix filter too broad
→ "daily_exports/" matches: daily_exports/orders/*.csv AND daily_exports/temp/*.csv
→ Fix: narrow the prefix or use multiple specific notifications
→ Or: use EventBridge for richer filtering (Project 41)

PITFALL 4: "put_bucket_notification_configuration REPLACES all existing notifications"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ THIS IS CRITICAL AND DANGEROUS
→ The API is a PUT, not a PATCH
→ If bucket already has Notification A and you add Notification B:
  → You must include BOTH A and B in the API call
  → If you only include B → Notification A is DELETED
→ Fix: always GET existing config first, merge, then PUT
→ Our code handles this correctly (see implementation below)

PITFALL 5: "Duplicate events — same file triggers two notifications"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause A: overlapping prefix filters (two rules match same key)
→ Cause B: S3 at-least-once delivery (rare duplicate)
→ Fix A: ensure no notification prefix is subset of another
→ Fix B: for processing (not alerts), implement idempotency (Project 18)

PITFALL 6: "Object key in event is URL-encoded"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ S3 event encodes special characters in object key
→ Space → "+", "+" → "%2B", etc.
→ Key "my folder/file (1).csv" → "my+folder/file+%281%29.csv"
→ Must URL-decode before using key in S3 GetObject calls
→ Python: urllib.parse.unquote_plus(key)

PITFALL 7: "Events work for PutObject but not multipart upload"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: filter set to "s3:ObjectCreated:Put" specifically
→ boto3 upload_file() uses multipart for files > threshold (default 8 MB)
→ Multipart fires "s3:ObjectCreated:CompleteMultipartUpload", NOT "Put"
→ Fix: use "s3:ObjectCreated:*" to catch ALL creation methods

HOW TO VERIFY S3 EVENT NOTIFICATIONS ARE WORKING:

1. Check notification configuration on bucket
   → aws s3api get-bucket-notification-configuration --bucket BUCKET_NAME
   → Should return JSON with your TopicConfiguration
   → If empty → notification was never set or was overwritten

2. Check SNS subscription status
   → aws sns list-subscriptions-by-topic --topic-arn TOPIC_ARN
   → Each subscription shows: SubscriptionArn, Protocol, Endpoint
   → Status "PendingConfirmation" = email not confirmed yet

3. Check CloudTrail for event delivery
   → CloudTrail logs SNS:Publish calls made by s3.amazonaws.com
   → Proves S3 attempted delivery
   → If no CloudTrail entry → filter didn't match OR config is wrong

4. Upload a test file and time it
   → Upload file matching prefix+suffix filter
   → Start a timer
   → Check email inbox (including spam)
   → Typical: email arrives in 5-30 seconds

5. Check SNS delivery metrics in CloudWatch
   → Metric: NumberOfMessagesPublished (per topic)
   → Metric: NumberOfNotificationsDelivered
   → Metric: NumberOfNotificationsFailed
   → Gap between Published and Delivered = delivery failures

6. Parse the email body
   → Email contains raw JSON event
   → Extract: bucket name, object key, size, event time
   → Verify these match the file you uploaded
