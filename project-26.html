🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — silver Parquet layer now in place (Project 25)
Situation:
→ CSV → Parquet conversion cut query cost by 30x (Project 25) ✅
→ But analysts still see unnecessary data scanning:

   Query: SELECT * FROM silver_orders WHERE order_date = DATE '2025-01-15'
   Expected scan: ~7 MB (one day out of 30)
   Actual scan:   200 MB (entire Parquet dataset)
   
   WHY? All 30 days of data are MIXED across 4 Parquet files
   → Athena must open every file → check row group stats → read matching rows
   → Row group statistics help skip SOME chunks but not entire files
   → For date-filtered queries: 97% of data scanned is WASTED

→ Analyst workflow patterns:
   - 60% of queries filter by order_date (last 7 days, specific date)
   - 20% of queries filter by status
   - 20% are full table scans (monthly reports)

→ Cost projection WITHOUT partitioning:
   9,000 queries/month × average 50 MB scan = 450 GB/month
   450 GB × $0.005/GB = $2.25/month (already good from Parquet)

→ Cost projection WITH date partitioning:
   5,400 date-filtered queries × 7 MB scan = 37.8 GB
   1,800 status-filtered queries × 50 MB scan = 90 GB
   1,800 full-scan queries × 200 MB scan = 360 GB
   Total: 487.8 GB → BUT date-filtered dropped from 270 GB to 37.8 GB
   Savings: ~$1.16/month (47% reduction on already-cheap Parquet)

→ At SCALE (1 year of data, 10x more analysts):
   WITHOUT partitioning: 200 MB → 2.4 GB per query (12 months data)
   WITH partitioning: still 7 MB per query (one day's folder)
   → Partitioning savings GROW as data grows
   → After 1 year: 340x cost reduction vs unpartitioned

CTO: "As data grows, won't our Athena costs grow linearly?"
YOU: "Not with partitioning. Athena only scans the folders it needs.
      1 day query scans 7 MB whether you have 30 days or 3 years of data.
      It's the difference between searching one filing cabinet drawer
      vs dumping the entire cabinet on the floor."

🟡 HOW TO APPROACH IT

Step 1: Understand WHAT partitioning means
→ Physical organization of data into folders by column value
→ S3 layout: s3://bucket/table/partition_col=value/files.parquet
→ Each unique partition value gets its own S3 "folder" (prefix)
→ Athena reads ONLY folders matching WHERE clause filters
→ Folders not matching = never opened = zero cost

Step 2: Understand HOW partitioning saves money
→ Without: 4 Parquet files contain ALL dates mixed
   Query WHERE date='2025-01-15' → opens ALL 4 files → scans 200 MB
→ With: 30 folders, each containing one day's Parquet files
   Query WHERE date='2025-01-15' → opens ONLY that folder → scans 7 MB
→ Data not in the matching folder is NEVER read from S3

Step 3: Choose the RIGHT partition key
→ Must be a column frequently used in WHERE clauses
→ Must have MODERATE cardinality (not too few, not too many)
→ Too few partitions (2-3): minimal benefit
→ Too many partitions (1M+): S3 listing overhead, Glue Catalog bloat
→ Sweet spot: 100-10,000 partitions
→ For QuickCart: order_date (365 values/year) = perfect

Step 4: Choose the partitioning METHOD
→ Method A: Hive-style partitions with MSCK REPAIR TABLE
   → Crawler or manual partition registration
   → Each partition stored in Glue Catalog
   → Works universally but requires catalog management
→ Method B: Partition Projection
   → Athena computes partitions MATHEMATICALLY
   → No catalog entries needed for each partition
   → Faster, no crawler needed, no MSCK REPAIR
   → Recommended for date-based partitioning
→ BOTH methods produce same query cost savings
→ Difference is in MANAGEMENT overhead

Step 5: Modify Glue ETL job to write partitioned output
→ Add partitionKeys=["order_date"] to write options
→ Output structure changes from flat to date-organized
→ Re-run conversion for all historical data
→ Update catalog table definition with partition info

🔵 WHAT COULD BE DONE

Option A: Hive-style partitions + MSCK REPAIR TABLE
→ Standard Hive partitioning (key=value/ folders)
→ MSCK REPAIR TABLE scans S3 to discover new partitions
→ Registers each partition in Glue Catalog
→ Works but: MSCK REPAIR is slow for many partitions
→ Requires re-running after every new partition appears
→ Good for: static or slowly-growing data

Option B: Hive-style partitions + Glue Crawler
→ Crawler detects new partition folders (Project 23)
→ Automatically registers in Glue Catalog
→ Better than MSCK REPAIR (scheduled, automated)
→ But: crawler takes 30-120 seconds per run
→ Good for: daily batch data with scheduled crawlers

Option C: Partition Projection (recommended for date keys)
→ Athena computes partitions MATHEMATICALLY
→ Define: start date, end date, interval
→ Athena generates partition list at query time
→ No catalog entries per partition
→ No MSCK REPAIR or crawler needed
→ Instant recognition of new partitions
→ SELECTED ✅ for date-based partitioning

Option D: Iceberg/Hudi tables with hidden partitioning
→ Modern table formats with built-in partitioning
→ Handles partition evolution, time travel
→ More complex setup
→ Covered conceptually — advanced use case

DECISION: Implement BOTH Option A and Option C
→ Option A: understand the fundamentals (MSCK REPAIR)
→ Option C: production-grade solution (Partition Projection)
→ Same underlying S3 layout — different catalog approach

✅ WHAT SHOULD DO

1. CHOOSE partition key based on query patterns
   → 60% of queries filter by order_date → partition by order_date
   → If queries also filter by status: consider TWO-LEVEL partitioning
     order_date=2025-01-15/status=completed/file.parquet
   → But: more levels = more folders = more management overhead
   → Start with ONE partition key → add more only if needed

2. USE Hive-style naming convention: key=value/
   → order_date=2025-01-15/ (not just 2025-01-15/)
   → Athena, Glue, Spark all auto-detect Hive-style partitions
   → Non-Hive paths (plain date folders) require manual mapping

3. KEEP partition values in standard format
   → Dates: YYYY-MM-DD (ISO 8601)
   → Not: 01-15-2025 or Jan-15 or 20250115
   → Consistent format enables range queries: BETWEEN '2025-01-01' AND '2025-01-31'

4. TARGET one Parquet file per partition (for daily data)
   → 333K rows/day × 85 bytes/row = ~28 MB/day
   → One ~7 MB Parquet file per day (after compression)
   → Sweet spot: 1-4 files per partition
   → Use coalesce(1) in Glue job for daily partitions

5. USE Partition Projection for date-based keys
   → Zero catalog management overhead
   → Instant recognition of new dates
   → No MSCK REPAIR or crawler needed for partitions
   → Define once in table properties → works forever

6. TEST with EXPLAIN before running expensive queries
   → EXPLAIN shows if partition pruning is happening
   → Look for: "partition filter" in plan
   → If missing: query scans all partitions (check WHERE clause)

7. VERIFY partition pruning with data scanned metric
   → Full table scan: ~200 MB
   → Single date partition: ~7 MB
   → If date-filtered query scans 200 MB → pruning NOT working
   → Common cause: wrong data type in WHERE clause

❌ WHAT SHOULD NOT DO

1. DO NOT partition by high-cardinality columns
   → Partitioning by order_id → 10M partitions (one per order)
   → Each partition: 1 tiny file → S3 listing nightmare
   → Athena: "HIVE_EXCEEDED_PARTITION_LIMIT"
   → Rule: < 10,000 partitions per table (soft limit)
   → order_date (365/year) ✅, order_id (10M) ❌

2. DO NOT partition by columns NOT used in WHERE clauses
   → Partitioning by product_id when nobody filters by product
   → Adds folder complexity but zero query benefit
   → Partition key MUST match query filter patterns

3. DO NOT create empty partitions
   → order_date=2025-02-30/ (doesn't exist)
   → Partition Projection may generate invalid dates
   → Athena handles gracefully (returns 0 rows) but wastes LIST calls

4. DO NOT forget to include partition column in Glue write
   → partitionKeys=["order_date"] in write_dynamic_frame
   → Without this: Glue writes flat files (no folders)
   → Then catalog says "partitioned" but S3 says "flat" → 0 rows

5. DO NOT mix partitioned and unpartitioned files in same prefix
   → s3://bucket/silver/orders/some_flat_file.parquet
   → s3://bucket/silver/orders/order_date=2025-01-15/file.parquet
   → Athena gets confused → may miss data or double-count
   → Use SEPARATE prefix: silver_partitioned/orders/

6. DO NOT use MSCK REPAIR TABLE on tables with 100K+ partitions
   → MSCK scans EVERY folder in S3 → very slow
   → 100K folders = 100K S3 LIST calls = minutes to complete
   → Use Partition Projection instead → zero scanning

7. DO NOT partition by date AND time if queries only filter by date
   → order_date=2025-01-15/order_hour=14/ → 365 × 24 = 8,760 partitions/year
   → If nobody filters by hour: 24x more folders for zero benefit
   → Only add partition levels that queries actually use

8. DO NOT use string comparison for date partition filtering
   → WHERE order_date = '2025-01-15' → STRING comparison
   → WHERE order_date = DATE '2025-01-15' → DATE comparison
   → String comparison may miss partition pruning in some engines
   → Always use proper DATE type in WHERE clause

🚫 WHEN NOT TO USE PARTITIONING

1. When table is tiny (< 100 MB)
   → Athena minimum charge: 10 MB per query
   → Partitioning 50 MB into 30 folders: 1.7 MB per partition
   → Still charged 10 MB minimum → no savings
   → Adds folder management complexity for zero benefit

2. When queries NEVER filter by any column
   → All queries are SELECT * or full-table aggregations
   → Partitioning adds S3 folder overhead
   → Every query still scans all partitions
   → No benefit, only downsides

3. When partition key has cardinality > 100,000
   → 100K folders = 100K S3 prefixes
   → S3 LIST operations become slow
   → Glue Catalog bloated (if not using Projection)
   → Consider bucketing instead (not native to Athena though)

4. When data is append-only with no date correlation
   → Log files with sequential IDs but no timestamp
   → No natural partition key that matches query patterns
   → Partitioning by arbitrary ranges adds complexity

5. When using Iceberg/Hudi table format
   → These formats handle partitioning internally
   → Hidden partitioning: Iceberg partitions by month(timestamp)
   → No Hive-style folders needed
   → More advanced — future consideration

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Query scans all partitions despite WHERE clause"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause A: WHERE uses function on partition key
   WHERE YEAR(order_date) = 2025  → NO pruning (function wraps column)
   WHERE order_date >= '2025-01-01' AND order_date < '2026-01-01' → PRUNING ✅
→ Cause B: data type mismatch
   Partition key is STRING, WHERE uses DATE literal → no match
→ Cause C: partition column not declared in table definition
→ Fix: check table properties, use direct comparison, match types

PITFALL 2: "MSCK REPAIR TABLE takes 10 minutes"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: MSCK scans EVERY S3 prefix under table location
→ 1,000 partitions = 1,000 S3 LIST calls
→ Fix: switch to Partition Projection (zero scanning)
→ Or: use ALTER TABLE ADD PARTITION for specific new partitions only

PITFALL 3: "New data arrived but Athena returns 0 rows"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause (Hive): new partition folder exists but not registered in catalog
→ Fix: run MSCK REPAIR TABLE or ALTER TABLE ADD PARTITION
→ Cause (Projection): S3 folder name doesn't match template pattern
→ Fix: verify folder name format matches projection.order_date.format

PITFALL 4: "Partition Projection returns data for dates that don't exist"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: projection generates ALL dates in range, including future dates
→ Query WHERE order_date = '2025-12-31' → Athena tries to read that folder
→ Folder doesn't exist → returns 0 rows (not an error)
→ No cost impact (S3 returns empty listing)
→ Not a bug — expected behavior

PITFALL 5: "Partitioned table is SLOWER than unpartitioned for full scans"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: 365 partitions = 365 S3 LIST calls + 365 file opens
→ Unpartitioned: 4 files = 4 S3 LIST calls + 4 file opens
→ Overhead per partition: ~10-50ms for S3 operations
→ 365 partitions × 30ms = ~11 seconds just for S3 operations
→ Fix: for full-table queries, use unpartitioned copy (or accept overhead)
→ Trade-off: partition-filtered queries are 30x faster but full scans slightly slower

PITFALL 6: "Glue job creates wrong partition folder structure"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: partitionKeys not matching column name exactly
→ Job writes: order_dt=2025-01-15/ but table expects: order_date=2025-01-15/
→ Fix: ensure partitionKeys value matches catalog partition column name EXACTLY
→ Case-sensitive!

PITFALL 7: "Query returns duplicate rows after re-running ETL"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Glue job appends new files to existing partition folders
→ First run: order_date=2025-01-15/part-00000.parquet
→ Second run: order_date=2025-01-15/part-00001.parquet (APPENDED)
→ Now both files read → double the data
→ Fix: delete existing partition files before writing (mode="overwrite")
→ Or: use distinct output path per job run

📈 SCALABILITY CONSIDERATIONS

PARTITION COUNT SCALING:
━━━━━━━━━━━━━━━━━━━━━━━
→ 30 partitions (1 month daily):      trivial, any method works
→ 365 partitions (1 year daily):      fine, MSCK takes ~30 seconds
→ 3,650 partitions (10 years daily):  MSCK slow, use Projection
→ 8,760 partitions (1 year hourly):   Projection mandatory
→ 100,000+ partitions:                Projection mandatory, Hive impossible

PARTITION PROJECTION LIMITS:
━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ No hard limit on projected partition count
→ Athena generates only MATCHING partitions per query
→ WHERE date = '2025-01-15' → generates 1 path (regardless of total range)
→ WHERE date BETWEEN '2025-01-01' AND '2025-12-31' → generates 365 paths
→ Performance: O(matching partitions), not O(total partitions)

DATA GROWTH:
━━━━━━━━━━━━
→ Month 1: 30 partitions × 7 MB = 210 MB total
→ Year 1:  365 partitions × 7 MB = 2.5 GB total
→ Year 3:  1,095 partitions × 7 MB = 7.5 GB total
→ Single-date query: ALWAYS 7 MB regardless of total data size
→ This is the MAGIC of partitioning: query cost is CONSTANT

MULTI-LEVEL PARTITIONING:
━━━━━━━━━━━━━━━━━━━━━━━━━
→ Level 1: order_date (365 values/year)
→ Level 2: status (5 values)
→ Total: 365 × 5 = 1,825 partitions/year
→ Query WHERE date='2025-01-15' AND status='completed' → 1 folder
→ Query WHERE date='2025-01-15' → 5 folders (all statuses)
→ More levels = more pruning but more folders to manage
→ Rule: add level only if queries FREQUENTLY filter by it

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY PARTITION PRUNING IS WORKING:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. Data scanned metric (most reliable):
   → Full table: ~200 MB
   → Single date partition: ~7 MB
   → If date-filtered query scans 200 MB → pruning NOT working

2. EXPLAIN query:
   → EXPLAIN SELECT * FROM table WHERE order_date = DATE '2025-01-15';
   → Look for: "partition filter" or "pushed down filter" in output
   → If absent: partition pruning not happening

3. Query execution time:
   → Full scan: 5-15 seconds
   → Single partition: 1-3 seconds
   → Significant difference confirms pruning

4. S3 request logs:
   → Enable S3 server access logging
   → Check which prefixes were accessed
   → Should only show the matching partition prefix

5. CloudWatch metrics:
   → Athena ProcessedBytes metric
   → Compare partitioned vs unpartitioned queries
   → 90%+ reduction confirms pruning
