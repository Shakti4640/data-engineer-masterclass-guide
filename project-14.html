ONE EVENT MUST REACH MULTIPLE CONSUMERS
🟢 Phase 1: Very Easy

🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — SNS (Project 12) + SQS (Project 13) operational
Situation:
→ Data platform is growing — multiple teams consume the same events
→ A single event now needs to trigger MULTIPLE downstream actions

SCENARIO — FILE ARRIVAL EVENT:
→ Marketing uploads campaign_results_20250715.csv to S3
→ This ONE event must trigger:

CONSUMER 1 — ETL Processor (Data Engineering):
"Load this file into Redshift staging table.
 I need an SQS queue so I can process at my own pace."

CONSUMER 2 — Data Quality Checker (QA Team):
"Validate this file: row count, schema, null percentages.
 If quality fails, block the ETL load.
 I need my OWN SQS queue — independent of ETL."

CONSUMER 3 — Audit Logger (Compliance):
"Record: who uploaded, when, what file, what size.
 Write to audit table in MariaDB.
 I need my OWN SQS queue — compliance has different SLA."

CONSUMER 4 — Email Notification (Data Team):
"Send email: 'New file arrived: campaign_results_20250715.csv'
 I'm a human — I need email, not a queue."

THE PROBLEM:
→ If we publish to SQS directly: only ONE consumer gets the message
→ If we publish to SNS email: machines can't process it
→ We need: ONE publish → FOUR consumers receive simultaneously
→ Each consumer processes independently (different speed, different logic)
→ If Consumer 2 is slow, it shouldn't block Consumer 1

CTO:
"This is the FAN-OUT pattern. Every event-driven architecture uses it.
 Master this and you can build any pipeline."

🟡 HOW TO APPROACH IT

Step 1: Understand the fan-out problem
→ ONE producer generates an event
→ MULTIPLE consumers need that same event
→ Each consumer has different processing needs
→ Consumers must be INDEPENDENT (one's failure doesn't affect others)

Step 2: Why direct SQS doesn't work
→ SQS: one queue, one consumer reads and deletes
→ If Consumer 1 deletes the message, Consumer 2 never sees it
→ Could create 4 queues and publish to all 4 — but producer 
  must know about every consumer (tight coupling)
→ Adding Consumer 5 requires changing the producer code

Step 3: Why SNS alone doesn't work
→ SNS email: great for humans, useless for machines
→ SNS HTTP: machines can receive, but no buffering
   → If machine is down when SNS delivers → message lost
→ SNS doesn't PERSIST messages — delivers and forgets

Step 4: The solution — SNS → SQS fan-out
→ Producer publishes to ONE SNS topic
→ SNS has MULTIPLE SQS queue subscriptions
→ SNS COPIES the message to EVERY subscribed queue
→ Each queue is independent — own retention, own consumer
→ Adding new consumer = subscribe new queue (no producer change)
→ Email subscribers also work alongside SQS subscribers

Step 5: This pattern is EVERYWHERE
→ AWS EventBridge uses this pattern internally
→ Microservices use this for event-driven communication
→ Every data platform uses this for pipeline orchestration
→ Projects 17, 18, 28, 41, 66 all build on this foundation

🔵 WHAT COULD BE DONE

Option A: Producer publishes to multiple SQS queues directly
→ Pros: simple, no SNS dependency
→ Cons: producer must know every queue URL
→ Cons: adding new consumer = modify producer code
→ Cons: producer must handle per-queue errors
→ TIGHT COUPLING — violates event-driven principles

Option B: SNS → SQS fan-out (RECOMMENDED)
→ Pros: decoupled — producer knows only topic
→ Pros: adding consumers = subscribe (no code change)
→ Pros: mixed protocols (SQS + email + Lambda + HTTP)
→ Pros: SNS handles delivery to all subscribers
→ Cons: slight latency (~50-100ms for SNS routing)
→ LOOSE COUPLING — industry standard pattern

Option C: EventBridge → multiple targets
→ Pros: content-based routing, scheduling
→ Pros: native AWS service integration
→ Cons: more complex setup for simple fan-out
→ Best for: complex routing rules (Project 66)

Option D: Kinesis → multiple consumers
→ Pros: multiple consumers read same stream
→ Pros: replay capability
→ Cons: shard management, more expensive
→ Best for: real-time streaming (Project 72)

DECISION: Option B — SNS → SQS fan-out
→ Industry standard for event-driven architectures
→ Perfect for QuickCart's batch-oriented needs
→ Foundation for all future pipeline projects

✅ WHAT SHOULD DO

1. Grant SNS permission to write to each SQS queue
   → SQS queue POLICY must allow sns:SendMessage from topic ARN
   → Without this: SNS publish succeeds but SQS never receives
   → This is the #1 setup mistake in fan-out patterns

2. Enable RAW MESSAGE DELIVERY for SQS subscribers
   → Default: SNS wraps message in SNS envelope JSON
   → Raw delivery: SQS receives EXACTLY what producer published
   → Why raw: consumer doesn't need to unwrap SNS envelope
   → When NOT raw: consumer needs SNS metadata (TopicArn, MessageId)

3. Use message attributes consistently
   → Attributes set by producer flow through to ALL subscribers
   → Enable filter policies (Project 51) for selective delivery
   → Attributes: event_type, source, priority, file_type

4. Create DLQ for EVERY fan-out SQS queue
   → Each queue processes independently — each can fail independently
   → DLQ per queue catches failures specific to that consumer
   → Don't share DLQs across queues — makes debugging impossible

5. Test fan-out with all consumers BEFORE production
   → Publish one test message
   → Verify ALL queues received it
   → Verify email subscribers received it
   → Check message content matches in each queue

6. Monitor each queue independently
   → CloudWatch alarms per queue: backlog, age, DLQ depth
   → One slow consumer shouldn't trigger alerts for others
   → Dashboard showing all fan-out queues side by side

7. Document the fan-out topology
   → Which topic fans out to which queues
   → Who owns each queue (team, service)
   → What each consumer does with the message
   → This documentation prevents "who is consuming this?" confusion

❌ WHAT SHOULD NOT DO

1. DO NOT have producer publish to SQS queues directly
   → Adding Consumer 5 requires modifying producer code
   → Producer handles errors for each queue separately
   → Use SNS as the decoupling layer

2. DO NOT forget the SQS queue policy
   → SNS cannot write to SQS without explicit permission
   → publish() succeeds (SNS accepted the message)
   → But SQS never receives it (permission denied at delivery)
   → SILENT FAILURE — no error returned to publisher

3. DO NOT share one SQS queue among multiple consumers
   → SQS: once Consumer A reads and deletes, Consumer B never sees it
   → Each consumer needs its OWN queue
   → Exception: competing consumers (same logic, load balancing)

4. DO NOT enable raw delivery for ALL subscribers blindly
   → Email subscribers: raw delivery means they get raw JSON
   → Email subscribers usually want the SNS-formatted version
   → SQS subscribers: raw delivery is usually better
   → Decision per subscriber, not per topic

5. DO NOT set different message retention across fan-out queues
   → If ETL queue retains 7 days but audit queue retains 1 day
   → Audit messages expire before investigation
   → Use consistent retention (7 days) unless specific reason

6. DO NOT publish to SNS synchronously in tight loops
   → 1000 files arrive → 1000 SNS publishes → each ~50ms
   → Total: ~50 seconds just for publishing
   → Better: batch processing or async publish

7. DO NOT create circular fan-out
   → Topic A → Queue B → Lambda → publishes to Topic A
   → Infinite loop of messages
   → Always ensure events flow in one direction

🚫 WHEN NOT TO USE SNS → SQS FAN-OUT

1. When there's truly only ONE consumer
   → Fan-out with one queue = unnecessary SNS layer
   → Publish directly to SQS — simpler, one less service
   → Add SNS later when second consumer appears

2. When consumers need real-time streaming
   → SNS → SQS adds ~100-200ms latency
   → For sub-second requirements: Kinesis Data Streams
   → QuickCart's batch ETL: 200ms is negligible

3. When message replay is required
   → SNS doesn't store messages — delivers and forgets
   → SQS: once consumed and deleted, gone forever
   → Use: Kinesis (replay from any point in retention window)

4. When you need request/response pattern
   → Fan-out is fire-and-forget (one-way)
   → No response channel from consumers to producer
   → Use: API Gateway + Lambda for synchronous request/response

5. When total fan-out exceeds 12.5 million subscriptions per topic
   → SNS hard limit: 12.5M subscriptions per topic
   → If you hit this: architectural review needed
   → QuickCart: 4 subscriptions — nowhere close

│                                                                             │
│  ═══════════════════════════════════════════════════════                     │
│  1. PUBLISH FLOW — WHAT HAPPENS INSIDE SNS                                  │
│  ═══════════════════════════════════════════════════════                     │
│                                                                             │
│  Producer calls:                                                            │
│  sns.publish(TopicArn="...", Message="...", MessageAttributes={...})        │
│                                                                             │
│  SNS Internal Process:                                                      │
│  ┌─────────────────────────────────────────────────────┐                   │
│  │ 1. ACCEPT: validate message, check permissions      │                   │
│  │    → Return MessageId to producer IMMEDIATELY       │                   │
│  │    → Producer's job is DONE at this point           │                   │
│  │                                                      │                   │
│  │ 2. ENUMERATE: list all confirmed subscriptions       │                   │
│  │    → Filter: apply filter policies per subscription │                   │
│  │    → Result: list of subscriptions that match       │                   │
│  │                                                      │                   │
│  │ 3. DELIVER: for each matching subscription:          │                   │
│  │                                                      │                   │
│  │    SQS subscription:                                 │                   │
│  │    → SNS calls sqs:SendMessage on queue              │                   │
│  │    → Uses queue's resource policy for authorization  │                   │
│  │    → Message written directly into SQS queue         │                   │
│  │    → If raw delivery: exact message body             │                   │
│  │    → If not raw: wrapped in SNS envelope JSON        │                   │
│  │                                                      │                   │
│  │    Email subscription:                               │                   │
│  │    → SNS sends via SES infrastructure                │                   │
│  │    → Subject = publish Subject parameter             │                   │
│  │    → Body = publish Message parameter                │                   │
│  │                                                      │                   │
│  │    Lambda subscription:                              │                   │
│  │    → SNS invokes Lambda synchronously                │                   │
│  │    → Waits for Lambda response                       │                   │
│  │    → Retries 3 times on failure                      │                   │
│  │                                                      │                   │
│  │ 4. COMPLETE: all deliveries attempted                │                   │
│  │    → Failed deliveries retried per protocol policy   │                   │
│  │    → SNS does NOT notify producer of delivery status │                   │
│  └─────────────────────────────────────────────────────┘                   │
│                                                                             │
│  TIMELINE:                                                                  │
│  T=0ms    Producer calls publish()                                          │
│  T=50ms   SNS returns MessageId to producer                                │
│  T=50ms   Producer continues (doesn't wait for delivery)                   │
│  T=100ms  SNS delivers to SQS Queue 1 (sqs:SendMessage)                   │
│  T=120ms  SNS delivers to SQS Queue 2 (sqs:SendMessage)                   │
│  T=140ms  SNS delivers to SQS Queue 3 (sqs:SendMessage)                   │
│  T=200ms  SNS sends email to subscriber                                    │
│  → All deliveries happen in PARALLEL (not sequential)                      │
│  → Total fan-out latency: ~100-200ms                                       │
│                                                                             │
│  ═══════════════════════════════════════════════════════                     │
│  2. SQS QUEUE POLICY — THE PERMISSION HANDSHAKE                             │
│  ═══════════════════════════════════════════════════════                     │
│                                                                             │
│  WITHOUT queue policy:                                                      │
│  SNS → "I want to send message to queue" → SQS: "Access Denied" ❌        │
│  → publish() returns SUCCESS (SNS accepted it)                             │
│  → But SQS never receives the message                                      │
│  → SILENT FAILURE — hardest bug to debug in fan-out                        │
│                                                                             │
│  WITH queue policy:                                                         │
│  {                                                                          │
│    "Statement": [{                                                          │
│      "Effect": "Allow",                                                     │
│      "Principal": {"Service": "sns.amazonaws.com"},                        │
│      "Action": "sqs:SendMessage",                                          │
│      "Resource": "arn:aws:sqs:...:etl-processing-queue",                   │
│      "Condition": {                                                         │
│        "ArnEquals": {                                                       │
│          "aws:SourceArn": "arn:aws:sns:...:file-events"                    │
│        }                                                                    │
│      }                                                                      │
│    }]                                                                       │
│  }                                                                          │
│                                                                             │
│  → Allows SNS service to send messages to this specific queue              │
│  → Condition: only from this specific SNS topic (not any topic)            │
│  → Without Condition: ANY SNS topic could write to your queue              │
│                                                                             │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Published to SNS successfully but SQS queue is empty"
→ Cause: SQS queue policy doesn't allow sns:SendMessage
→ THIS IS THE #1 FAN-OUT BUG — silent failure
→ SNS returns success (it accepted the message)
→ But delivery to SQS fails (permission denied)
→ Fix: add queue policy allowing SNS topic to send
→ Debug: check SNS delivery status logs (if enabled)

PITFALL 2: "Consumer receives double-wrapped JSON"
→ Cause: raw delivery not enabled + consumer parsing message body as JSON
→ Message body contains SNS envelope with "Message" field inside
→ Consumer parses outer JSON but not inner "Message" string
→ Fix: enable raw delivery on SQS subscription
→ Or: parse two levels: outer SNS envelope → inner Message JSON

PITFALL 3: "Adding new SQS subscription doesn't receive messages"
→ Cause: SQS subscription is auto-confirmed (good)
→ But queue policy not updated to allow the SNS topic
→ Fix: update queue policy BEFORE subscribing

PITFALL 4: "Fan-out works for some queues but not others"
→ Cause: queue policy only allows specific topic ARN
→ Using wrong topic ARN in condition
→ Fix: verify SourceArn condition matches exactly

PITFALL 5: "Message attributes missing in SQS consumer"
→ Cause: raw delivery changes how attributes are delivered
→ With raw: attributes become SQS message attributes
→ Without raw: attributes are in SNS envelope JSON
→ Fix: consistent raw delivery setting + matching consumer code

PITFALL 6: "Cross-account fan-out fails silently"
→ Cause: SQS queue in Account B needs both:
   → SQS queue policy allowing SNS from Account A
   → SNS topic policy allowing subscribe from Account B
→ Missing either = silent failure
→ Covered in detail in Project 59

PITFALL 7: "Duplicate messages in SQS after fan-out"
→ Cause: SNS delivers at-least-once to SQS
→ Under high load or retries: duplicate delivery possible
→ Fix: consumer must be idempotent
→ Pattern: track processed MessageId in database

📈 SCALABILITY CONSIDERATIONS

FAN-OUT LIMITS:
→ Subscriptions per topic: 12,500,000 (soft limit)
→ SNS publishes per second: 30,000 (per topic)
→ SQS receives per second: unlimited (Standard)
→ QuickCart: 4 subscriptions, ~200 messages/day = trivial

FAN-OUT LATENCY:
→ SNS processing: ~50ms
→ SNS → SQS delivery: ~50-100ms per subscription
→ Parallel delivery: all subscriptions delivered simultaneously
→ Total: ~100-200ms from publish to all queues populated
→ For QuickCart batch ETL: negligible

COST AT SCALE:
→ SNS publish: $0.50 per million requests
→ SNS → SQS delivery: free (same account)
→ SQS receive: $0.40 per million requests (per queue)
→ 4 queues × 200 msg/day = 800 SQS requests/day
→ Monthly: 24,000 requests = $0.00 (free tier covers 1M)

SCALING FAN-OUT:
→ 1-5 consumers: simple SNS → SQS setup (this project)
→ 5-20 consumers: same pattern, consider filter policies (Project 51)
→ 20-100 consumers: consider EventBridge for complex routing (Project 66)
→ 100+ consumers: architectural review — probably need event mesh

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY FAN-OUT IS WORKING:

1. Publish test message to topic
   → aws sns publish --topic-arn <arn> --message '{"test": true}'

2. Check EACH queue received the message
   → For each queue:
     aws sqs get-queue-attributes --queue-url <url> 
       --attribute-names ApproximateNumberOfMessages
   → All should show +1

3. Verify message content in each queue
   → aws sqs receive-message --queue-url <url> --max-number-of-messages 1
   → Compare message body across queues — should be identical
   → If raw delivery: exact match
   → If not raw: wrapped in SNS envelope

4. Check SNS delivery status
   → Enable delivery status logging on topic
   → CloudWatch Logs: successful/failed deliveries per subscription
   → Shows exactly which subscriptions received/failed

5. Monitor queue depths after production publish
   → All fan-out queues should have same message count
   → If one queue has fewer: delivery failure to that queue
   → If one queue has more: that consumer is slower (backlog)

🔑 KEY TAKEAWAY

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ FAN-OUT PATTERN: ONE publish → MULTIPLE independent consumers
→ SNS topic: the central routing hub (producer publishes here)
→ SQS queues: independent buffers (one per consumer)
→ DECOUPLING: producer knows ONLY the topic, not the consumers
   → Adding Consumer 5 = subscribe new queue (zero code change)
   → Removing Consumer 2 = unsubscribe (zero code change)

→ CRITICAL SETUP: SQS queue policy MUST allow SNS to send messages
   → Without this: silent failure — publish succeeds, delivery fails
   → #1 debugging issue in fan-out architectures

→ RAW MESSAGE DELIVERY: SQS receives exact published message
   → Without raw: double-wrapped JSON (SNS envelope + message)
   → With raw: clean JSON — consumer parses one level

→ INDEPENDENCE: each consumer has own queue, DLQ, visibility, retention
   → Slow consumer doesn't block others
   → Failed consumer doesn't affect others
   → Each scales independently

→ IDEMPOTENCY: consumers MUST handle duplicate messages
   → SNS delivers at-least-once to SQS
   → SQS delivers at-least-once to consumer
   → Net: duplicates are possible — consumer must handle gracefully

THE PATTERN IN ONE SENTENCE:
━━━━━━━━━━━━━━━━━━━━━━━━━━━
"Publish once to SNS, subscribe N SQS queues,
 each consumer processes independently."
