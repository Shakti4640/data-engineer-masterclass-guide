🔴 HYPOTHETICAL PROBLEM

Company: "QuickCart" — Analyst self-service expanding
Current State (after Projects 31-36):
→ Bronze → Silver → Gold pipeline automated (Glue Workflow)
→ Gold tables handle 80% of analyst queries (Project 36)
→ 8 analysts actively querying Athena daily
→ 3 new junior analysts joining next week

THE PROBLEMS:
━━━━━━━━━━━━━

Problem 1 — SQL DUPLICATION:
→ Every analyst writes the same JOINs repeatedly:
   SELECT o.*, c.customer_name, c.tier, p.product_name, p.category
   FROM silver_quickcart.order_details o
   LEFT JOIN silver_quickcart.clean_customers c ON o.customer_id = c.customer_id
   LEFT JOIN silver_quickcart.clean_products p ON o.product_id = p.product_id
   WHERE o.status = 'completed'
→ 8 analysts × 5 times/day = 40 copies of the same 15-line query
→ If schema changes → must update 40 saved queries across 8 people

Problem 2 — JUNIOR ANALYST ONBOARDING:
→ New analysts don't know the schema
→ Don't know which tables to join
→ Don't know filter conventions (status = 'completed' vs 'complete')
→ Spend 2 weeks learning table relationships before being productive
→ Manager: "Can we give them simple table names that just work?"

Problem 3 — QUERY INCONSISTENCY:
→ Analyst A calculates revenue as: SUM(quantity * unit_price)
→ Analyst B calculates revenue as: SUM(total_amount)
→ Analyst C calculates revenue as: SUM(revenue) WHERE status != 'cancelled'
→ Three different numbers for "total revenue" in three reports
→ CEO: "Why does every report show a different revenue number?"

Problem 4 — NO COST GUARDRAILS:
→ Junior analyst runs: SELECT * FROM silver_quickcart.order_details
→ Scans 15 GB → costs $0.075
→ Meant to add LIMIT 10 but forgot
→ Runs it 20 times debugging → $1.50 wasted
→ No per-query or per-analyst cost limits

Problem 5 — SAVED QUERIES SCATTERED:
→ Each analyst has saved queries in local text files
→ No shared query library
→ No version control on SQL
→ When analyst leaves → their queries disappear

→ Data engineer: "We need VIEWS for standardized access,
   NAMED QUERIES for a shared library,
   and WORKGROUPS for cost control."

🟡 HOW TO APPROACH IT

Step 1: Understand the THREE tools Athena provides
→ VIEWS: Virtual tables defined by SQL — no data stored
→ NAMED QUERIES: Saved SQL templates — shared query library
→ WORKGROUPS: Query execution environments — cost/access control

Step 2: Design the view layer
→ Business views: hide complexity, standardize calculations
→ Security views: restrict columns/rows per team
→ Convenience views: common joins pre-defined

Step 3: Design the workgroup strategy
→ Per-team workgroups with cost limits
→ Shared workgroup for gold tables (cheap queries)
→ Restricted workgroup for silver tables (expensive queries)

Step 4: Dependencies from previous projects
→ Project 16: Athena basics
→ Project 24: Athena querying Glue Catalog
→ Project 26: Partitioned Parquet
→ Project 34: Silver layer tables (source for views)
→ Project 36: Gold layer CTAS (views complement CTAS)

🔵 WHAT COULD BE DONE

Option A: Athena Views only
→ Virtual tables with standardized SQL
→ Pros: simple, immediate, no storage cost
→ Cons: still scans full data on each query

Option B: Views + Named Queries
→ Views for standard access + saved query library
→ Pros: reusable templates, shared across team
→ Cons: no cost control

Option C: Views + Named Queries + Workgroups (RECOMMENDED)
→ Full governance: access, consistency, cost control
→ Views: standardize calculations and joins
→ Named Queries: shared library of common analyses
→ Workgroups: per-team cost limits and result locations
→ Pros: complete solution for analyst self-service
→ Cons: more setup, but worth it at scale

Option D: Views pointing to Gold tables (hybrid)
→ Views on gold tables = virtually free queries
→ Views on silver tables = full scan cost
→ Strategy: gold views for frequent queries, silver views for ad-hoc
→ BEST: combine with Option C

DECISION: Options C + D combined
→ Views on gold tables (frequent, cheap)
→ Views on silver tables (ad-hoc, governed by workgroup limits)
→ Named queries for common analysis patterns
→ Workgroups for per-team cost control

✅ WHAT SHOULD DO

1. Create a dedicated "views" database in Glue Catalog
   → Database: reporting_views
   → All views live here — analysts know where to look
   → Separate from silver/gold data databases

2. Standardize business metric calculations in views
   → revenue = SUM(revenue) WHERE status = 'completed'
   → active_customers = COUNT(DISTINCT customer_id) WHERE last_order < 90 days
   → Define ONCE in view → all analysts get same number

3. Create layered views
   → Base views: simple JOINs (order_details + customers + products)
   → Metric views: standardized aggregations built ON base views
   → Security views: column/row restrictions per team

4. Name views descriptively
   → v_completed_order_details (not "view1")
   → v_monthly_revenue_by_category (not "rev_cat")
   → Prefix with "v_" to distinguish from tables

5. Point frequent views at GOLD tables
   → v_revenue_summary → reads gold_quickcart.revenue_by_category_month
   → Scans 3 KB instead of 15 GB
   → Effectively free for analysts

6. Create one workgroup per team
   → Revenue team: wg-revenue (50 GB/query limit)
   → Junior analysts: wg-junior (1 GB/query limit, enforced)
   → Each workgroup has its own result location in S3

7. Document views with COMMENT in DDL
   → CREATE VIEW v_name COMMENT 'Description of what this view provides'
   → Analysts can see comments in Glue Catalog / Athena Console

8. Use Named Queries for analysis templates
   → "Top N products by revenue" — parameterized
   → "Customer cohort analysis" — reusable
   → Stored in Athena, accessible by all team members

❌ WHAT SHOULD NOT DO

1. DO NOT create views that join 5+ large tables
   → Each additional JOIN multiplies scan volume
   → View with 5 JOINs on silver tables → 50+ GB scan per query
   → Pre-join in silver layer (Project 34) → view reads pre-joined table

2. DO NOT rely on views for performance
   → Views are VIRTUAL — SQL is re-executed every time
   → A view on 15 GB silver table scans 15 GB every query
   → For performance: use CTAS (Project 36) not views
   → Views = consistency + convenience, NOT performance

3. DO NOT create views with SELECT *
   → SELECT * from underlying table includes ALL columns
   → If table adds columns → view automatically exposes them
   → May expose sensitive columns unintentionally
   → Always list columns explicitly in view definition

4. DO NOT skip workgroup cost limits for junior analysts
   → One accidental SELECT * FROM order_details (no LIMIT) = $0.075
   → Junior running 200 queries/day exploring = $15/day
   → Set 1 GB/query limit → max cost per query = $0.005
   → Save $300/month from accidental full scans

5. DO NOT create views in the same database as tables
   → Mixing views and tables in silver_quickcart → confusion
   → "Is this a view or a table?" → analysts don't know
   → Separate database: reporting_views for views only

6. DO NOT use views to bypass security
   → View runs with CALLER's permissions, not creator's
   → If analyst can't access underlying table → view fails too
   → For row-level security: use Lake Formation (Project 74)

7. DO NOT create 100 views for every possible query
   → View explosion → nobody knows which to use
   → Create 10-15 well-designed views covering 90% of needs
   → Named queries cover the remaining 10%

🚫 WHEN NOT TO USE ATHENA VIEWS

1. Need sub-second query response on large data
   → Views re-execute SQL every time → same scan as raw query
   → Use: CTAS (Project 36) or Redshift Materialized Views (Project 63)

2. Need row-level security (user A sees rows X, user B sees rows Y)
   → Athena views run with caller's permissions
   → Cannot enforce different row filters per user via views alone
   → Use: Lake Formation with cell-level security (Project 74)

3. Need to share query results across services (not just Athena)
   → Views only exist in Athena context
   → Lambda, Glue, Redshift cannot query Athena views directly
   → Use: CTAS to create actual tables other services can read

4. Source data changes schema frequently
   → View with explicit column list breaks when column removed
   → Must update view DDL every time source schema changes
   → Use: Schema evolution handling in Glue (Project 45)

5. View would scan > 100 GB per execution
   → Each analyst query costs > $0.50
   → Even with workgroup limits, this is expensive
   → Materialize with CTAS instead

│              ATHENA VIEWS, NAMED QUERIES, WORKGROUPS — INTERNALS                 │
├──────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│  ATHENA VIEWS — HOW THEY WORK:                                                   │
│  ═════════════════════════════                                                   │
│                                                                                  │
│  CREATE OR REPLACE VIEW reporting_views.v_completed_orders AS                    │
│  SELECT order_id, customer_id, revenue                                           │
│  FROM silver_quickcart.order_details                                             │
│  WHERE status = 'completed';                                                     │
│                                                                                  │
│  WHAT HAPPENS:                                                                   │
│  1. Athena validates SQL against Glue Catalog schemas                            │
│  2. View DEFINITION stored in Glue Catalog as table metadata:                    │
│     → TableType: "VIRTUAL_VIEW"                                                  │
│     → ViewOriginalText: the SQL                                                  │
│     → ViewExpandedText: fully qualified SQL                                      │
│  3. NO data is stored, NO S3 files created                                       │
│  4. View appears as a "table" in Glue Catalog                                    │
│                                                                                  │
│  WHEN ANALYST QUERIES THE VIEW:                                                  │
│  SELECT * FROM reporting_views.v_completed_orders WHERE revenue > 100            │
│                                                                                  │
│  1. Athena reads view definition from Glue Catalog                               │
│  2. Expands view SQL inline:                                                     │
│     SELECT * FROM (                                                              │
│       SELECT order_id, customer_id, revenue                                      │
│       FROM silver_quickcart.order_details                                        │
│       WHERE status = 'completed'                                                 │
│     ) WHERE revenue > 100                                                        │
│  3. Optimizes combined query (predicate pushdown)                                │
│  4. Executes against actual S3 data                                              │
│  5. Scans same volume as if analyst wrote the full query                         │
│                                                                                  │
│  KEY INSIGHT: Views are COMPILE-TIME substitution, not runtime caching           │
│  → Zero performance benefit over writing the SQL directly                        │
│  → Value is: consistency, convenience, governance                                │
│                                                                                  │
│  PREDICATE PUSHDOWN THROUGH VIEWS:                                               │
│  ═════════════════════════════════                                                │
│                                                                                  │
│  View: WHERE status = 'completed'                                                │

│                                                                                  │
│  Athena combines: WHERE status = 'completed'                                     │
│                    AND revenue > 100                                              │
│                    AND year = '2025'                                              │
│                                                                                  │
│  → year = '2025' pushes down to partition pruning                                │
│  → Only scans 2025 partitions (not all years)                                    │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "View query is slow — thought views were fast"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Views don't store data — they re-execute SQL every time
→ View on 15 GB silver table = 15 GB scan every query
→ Fix: Point view at GOLD table (3 KB) instead of silver (15 GB)
→ Or: Accept the cost for always-fresh data

PITFALL 2: "View breaks after source table schema changed"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: View references column that was renamed/removed
→ Error: "Column 'old_name' not found in table"
→ Fix: CREATE OR REPLACE VIEW with updated column list
→ Prevention: use stable column aliases in views
→ Project 45 covers schema evolution in depth

PITFALL 3: "Workgroup limit blocks legitimate large queries"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: BytesScannedCutoffPerQuery too restrictive
→ Analyst needs to scan 5 GB but limit is 1 GB
→ Fix A: Increase limit for that workgroup
→ Fix B: Create gold table for that specific query
→ Fix C: Create separate "power-user" workgroup with higher limits

PITFALL 4: "Named query shows in Console but analyst can't find it"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: Named queries are per-workgroup
→ Analyst is in wg-junior but query was created in wg-revenue
→ Fix: Create named queries in the workgroup where analysts work
→ Or: Create in "primary" workgroup (shared)

PITFALL 5: "View on partitioned table doesn't benefit from partition pruning"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: View doesn't include partition column in output
→ Analyst can't filter on partition column → full scan
→ Fix: ALWAYS include partition columns in view SELECT
→ Even if analyst doesn't need them — they enable pruning

PITFALL 6: "CREATE VIEW fails: views cannot be created on views"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Actually: Athena DOES support views on views (up to a depth)
→ Real cause: circular view reference (view A references view B references view A)
→ Fix: Keep view hierarchy shallow (max 2 levels deep)
→ Base views → metric views → STOP (don't add more layers)

📈 SCALABILITY CONSIDERATIONS

VIEW PERFORMANCE AT SCALE:
━━━━━━━━━━━━━━━━━━━━━━━━━
→ Views themselves have ZERO performance overhead
→ Performance = performance of the underlying query
→ View on gold table (3 KB): 0.3s regardless of analyst count
→ View on silver table (15 GB): 12s regardless of analyst count
→ 100 analysts querying same view = 100 separate scans
→ No caching between users (each query is independent)

WORKGROUP SCALING:
→ Max workgroups per account: 1000
→ Max concurrent queries per workgroup: configurable (default 25)
→ Max concurrent queries per account: 25 (default, can increase to 200+)
→ For 50 analysts: 3-5 workgroups sufficient

NAMED QUERY LIMITS:
→ Max named queries per workgroup: unlimited (practically)
→ Max query string length: 262,144 bytes (~256 KB of SQL)

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ Views are VIRTUAL — no data stored, SQL re-executed every query
→ Views provide: consistency, convenience, governance — NOT performance
→ Gold-backed views combine CTAS performance with view standardization
→ Workgroups enforce per-team cost limits (BytesScannedCutoffPerQuery)
→ EnforceWorkGroupConfiguration=true prevents analysts from overriding
→ Named queries create a shared SQL library visible in Athena Console
→ Always include partition columns in views for partition pruning
→ List columns explicitly — never SELECT * in view definitions
→ 12 well-designed views cover 90% of analyst needs
→ Security views restrict columns but NOT true security (use Lake Formation)
→ View health checks catch broken views after schema changes
→ Onboarding guide reduces new analyst ramp-up from 2 weeks to 2 days
