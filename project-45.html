Step 2: Define schema evolution STRATEGY

  Two fundamental approaches:

  A) SCHEMA-ON-WRITE (current — Project 44):
  → Silver has STRICT schema (StructType)
  → Data MUST match schema to be written
  → Any mismatch → job fails
  → Pro: guarantees data quality
  → Con: breaks on any source change

  B) SCHEMA-ON-READ (opposite extreme):
  → Silver stores whatever arrives (like Bronze)
  → Schema enforced only when READING (Athena, Redshift)
  → Pro: never breaks on source changes
  → Con: no quality guarantee, garbage can accumulate

  C) ADAPTIVE SCHEMA-ON-WRITE (our solution — hybrid):
  → Silver has a BASELINE schema (expected columns + types)
  → ETL ADAPTS to changes gracefully:
    - New columns → include with NULL default if not in baseline
    - Missing columns → add as NULL (if nullable) or ALERT (if required)
    - Type changes → attempt safe cast, log failures
    - Renamed columns → use mapping table for known renames
  → Quality gate: validate AFTER adaptation, BEFORE writing
  → Alert on ALL schema changes (even handled ones)
  → Human reviews and updates baseline schema periodically

🟢 HOW CAN WE SOLVE IT

SOLUTION: Multi-layer schema evolution handling

LAYER 1: Schema Registry (source of truth for expected schemas)
→ Store baseline schemas in DynamoDB (or Glue Schema Registry)
→ Version tracked: schema_v1, schema_v2, etc.
→ Includes: column names, types, nullable, known aliases

LAYER 2: Schema Drift Detector (compares incoming vs baseline)
→ Runs at start of every Bronze→Silver ETL
→ Detects: additions, removals, renames, type changes
→ Classifies each change by severity

LAYER 3: Adaptive Transformer (handles changes gracefully)
→ Column added → include in Silver if useful, ignore if not
→ Column removed → fill with NULL if nullable, FAIL if required
→ Column renamed → apply alias mapping from registry
→ Type widened → safe cast (DECIMAL(10,2) → DECIMAL(10,4) is safe)
→ Type changed → attempt cast, quarantine failures

LAYER 4: Quality Gate (validates adapted output)
→ Required columns present and non-null
→ Data types match Silver schema
→ Row count reasonable (not 0, not 10× expected)
→ Pass → write to Silver
→ Fail → write to quarantine, alert team

LAYER 5: Schema Drift Alert (always fires, even when handled)
→ SNS notification with full change report
→ Team reviews and updates baseline if change is permanent
→ Prevents "silent drift" where schemas gradually degrade

🔵 WHAT COULD BE DONE

Option A: Glue Crawlers auto-detect schema changes
→ Crawler re-crawls Bronze, updates Catalog table
→ Problem: Crawler can't ADAPT the ETL logic
→ It tells you the schema changed, but doesn't fix the pipeline
→ VERDICT: Good for detection, insufficient for handling

Option B: Glue DynamicFrame ResolveChoice
→ ResolveChoice handles ambiguous types (e.g., string vs int)
→ Problem: doesn't handle missing columns, renamed columns
→ Only resolves TYPE ambiguity, not structural changes
→ VERDICT: Useful tool but not a complete solution

Option C: AWS Glue Schema Registry
→ Avro/JSON Schema Registry for schema versioning
→ Supports compatibility checks (BACKWARD, FORWARD, FULL)
→ Problem: designed for streaming (Kafka/Kinesis schemas)
→ Doesn't auto-fix ETL logic, just validates
→ VERDICT: Good for versioning, complement to our solution

Option D: Custom adaptive schema handler (our solution)
→ DynamoDB stores baseline schemas + alias mappings
→ Python code detects, classifies, adapts changes
→ Integrated into Bronze→Silver ETL as first step
→ Alerts on all changes, handles safe changes automatically
→ VERDICT: ✅ Most practical for batch ETL pipelines

DECISION: Option D + Option C (Glue Schema Registry for versioning)

❌ WHAT SHOULD NOT DO

1. DO NOT silently swallow schema changes
   → Even if auto-handled, team must be notified
   → Silent handling leads to "schema drift" over months
   → One day: Silver has 30 extra columns nobody knows about

2. DO NOT fail the entire pipeline for a new nullable column
   → Backend adds "promo_code" column → pipeline crashes
   → This is a LOW severity change → handle automatically
   → Only FAIL for truly dangerous changes (required column removed)

3. DO NOT hardcode column names in ETL logic
   → Bad: df.select("order_id", "status", "amount")
   → Good: df.select([c for c in baseline_columns if c in df.columns])
   → Dynamic column selection adapts to schema changes

4. DO NOT update the baseline schema automatically
   → Auto-detection is fine, auto-HANDLING is fine
   → But the BASELINE (source of truth) should only change
     after human review
   → Prevents: source sends garbage → baseline corrupted → 
     garbage becomes the "expected" schema

5. DO NOT ignore type changes even if cast succeeds
   → DECIMAL(10,2) → DECIMAL(10,4): cast succeeds silently
   → But downstream Gold queries may format differently
   → Redshift table DDL may need updating
   → ALWAYS alert, even on compatible type changes

6. DO NOT rely on Glue Crawler alone for schema management
   → Crawler detects changes AFTER the fact
   → Crawler can't FIX the ETL logic
   → Crawler sometimes guesses wrong types (string for everything)
   → Use Crawler as ONE signal, not the sole mechanism

7. DO NOT apply renames without a verified alias map
   → If you GUESS that "status" was renamed to "order_status"
     but actually "status" was REMOVED and "order_status" is NEW
   → Your alias mapping produces WRONG data silently
   → Alias map must be explicitly maintained by the data engineer

🚫 WHEN NOT TO USE ADAPTIVE SCHEMA HANDLING

1. When source schema is contractually stable
   → External vendor API with versioned schema (v1, v2)
   → Schema changes announced months in advance
   → Strict handling is simpler and more predictable

2. When data quality requirements are absolute
   → Financial transaction data — every field must be exact
   → Healthcare records — schema changes require formal review
   → Adaptive handling might mask data quality issues

3. When source produces completely unstructured data
   → JSON with wildly varying fields per record
   → No concept of "baseline" schema exists
   → Use schema-on-read or JSON parsing with explicit field extraction

4. When there's only one source and one consumer
   → Simple pipeline: MariaDB → S3 → Redshift
   → Just fix the ETL when schema changes (faster than building adaptation layer)
   → Adaptation framework has overhead that's only worth it at scale

5. When the adaptation logic itself becomes a maintenance burden
   → 50+ alias mappings, complex type conversion rules
   → The "smart" handler is now more complex than the ETL itself
   → Simplify: establish schema contract with source team instead

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Alias mapping produces wrong data"
→ Cause: assumed "status" renamed to "order_status"
→ Reality: "status" was REMOVED, "order_status" is a NEW column with different semantics
→ "status" was: pending/shipped/completed
→ "order_status" is: open/closed (different enum values!)
→ Fix: alias map requires EXPLICIT human confirmation
→ Never auto-generate alias mappings based on name similarity

PITFALL 2: "Silver schema keeps growing with abandoned columns"
→ Cause: backend adds columns, later removes them, Silver keeps all
→ After 12 months: Silver has 15 columns that source no longer sends
→ All 15 are NULL in new data, wastes storage and confuses analysts
→ Fix: quarterly schema review — mark columns as "deprecated"
→ After 90 days of all-NULL: remove from Silver schema

PITFALL 3: "Type widening cascades break Gold queries"
→ Cause: unit_price widened from DECIMAL(10,2) to DECIMAL(10,4)
→ Silver handles it fine (Parquet supports both)
→ But Gold query: ROUND(unit_price, 2) now rounds differently
→ And Redshift DDL still says DECIMAL(10,2) → COPY fails
→ Fix: schema change alerts must include downstream impact analysis

PITFALL 4: "mergeSchema reads 10,000 Parquet file footers"
→ Cause: Silver has years of partitioned data, mergeSchema reads ALL footers
→ Performance: 10,000 files × 50ms each = 8+ minutes just for schema
→ Fix: read only LATEST partition for schema inference
→ Or: use Glue Catalog as schema authority (don't infer from files)

PITFALL 5: "New column has sensitive data (PII)"
→ Cause: backend adds "shipping_address" → auto-included in Silver
→ Shipping address is PII → must be masked/encrypted
→ Auto-include didn't apply PII handling
→ Fix: new columns default to QUARANTINE (not auto-include)
→ Data engineer must explicitly approve + classify each new column

PITFALL 6: "Schema detection runs AFTER data write (too late)"
→ Cause: ETL writes to Silver, then detects schema changed
→ Silver now has data with inconsistent schema
→ Fix: detect BEFORE write, validate BEFORE write, alert BEFORE write
→ Never write first, check later

PITFALL 7: "Glue Crawler marks all columns as string"
→ Cause: CSV has no type metadata, Crawler can't infer types reliably
→ Crawler sees: "29.99" → could be string or decimal
→ Defaults to string for safety
→ Fix: don't rely on Crawler for Silver schema
→ Use explicit StructType (Project 44) + adaptive handling (this project)

→ Overhead: negligible

SCALING TO 50 ENTITIES:

→ Schema registry: 50 entries, ~200 versions over time

→ Total overhead: 50 entities × 5 seconds = ~4 minutes/day

SCALING TO 500 ENTITIES:
→ Schema changes: potentially daily across all entities
→ Manual review of EVERY change becomes bottleneck
→ Solution: auto-approve LOW severity changes (column added, type widened)
→ Only escalate MEDIUM/HIGH severity to human review
→ Weekly "schema drift summary" instead of per-change alerts

SCHEMA REGISTRY SIZE:
→ DynamoDB: 1 item per entity per version
→ 500 entities × 50 versions = 25,000 items
→ DynamoDB handles this trivially (<1ms reads)
→ Cost: ~$0.00 (on-demand pricing, minimal reads/writes)

SCHEMA COMPARISON PERFORMANCE:
→ Comparing two schemas: O(n) where n = number of columns
→ 100 columns: <1ms comparison time
→ 1000 columns: <10ms comparison time
→ Not a performance concern at any practical scale

WHAT WE LEARNED:
━━━━━━━━━━━━━━━━
→ Schema evolution is INEVITABLE — source systems change constantly
→ Strict schema-on-write (Project 44) breaks on ANY source change
→ Adaptive schema handling: detect → classify → adapt → validate → alert
→ Schema registry (DynamoDB): versioned baseline + alias mappings

→ Type compatibility matrix: widening is safe, narrowing is dangerous
→ Quality gate AFTER adaptation catches data quality issues
→ ALWAYS alert on drift — even auto-handled changes need human review
→ Baseline schema updated MANUALLY by engineer after confirming permanence
→ Drift reports in S3 enable historical schema evolution analysis
