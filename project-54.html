🟡 HOW TO APPROACH IT

Step 1: Understand the monitoring pyramid
→ METRICS: raw numbers (job duration, error count, queue depth)
→ ALARMS: rules on metrics (if error_count > 0 for 5 min → trigger)
→ NOTIFICATIONS: actions when alarms fire (SNS → email/SMS/PagerDuty)
→ DASHBOARDS: visual overview of all metrics in one place
→ RUNBOOKS: what to do when alarm fires (human instructions)

Step 2: Identify metric SOURCES
→ AWS-managed metrics (FREE, auto-published):
  - Glue job metrics → CloudWatch namespace: Glue
  - SQS queue metrics → CloudWatch namespace: AWS/SQS
  - S3 request metrics → CloudWatch namespace: AWS/S3
  - EC2 instance metrics → CloudWatch namespace: AWS/EC2
→ Custom metrics (must publish ourselves):
  - Upload script success/failure
  - Pipeline end-to-end latency
  - Data quality check results
  - Worker heartbeat (from Project 52)

Step 3: Design alarm topology
→ Per-component alarms (detect individual failures)
→ Composite alarm (aggregate: "is the ENTIRE pipeline healthy?")
→ Composite alarm fires ONLY if multiple components are degraded
→ Avoids alert fatigue from transient single-component blips

Step 4: Design notification routing
→ WARNING severity → email to [email protected]
→ CRITICAL severity → email + SMS to on-call engineer
→ SLA BREACH → email + SMS + PagerDuty auto-incident
→ Uses SNS topics with different subscribers per severity
→ Builds on SNS from Project 12 and filtering from Project 51

Step 5: Design the dashboard
→ Single CloudWatch Dashboard showing all pipeline components
→ Widgets: metric graphs, alarm status, recent errors
→ URL bookmarked by CTO, engineers, analysts
→ Auto-refreshes every 60 seconds

🔵 WHAT COULD BE DONE

Option A: Manual log checking (current — failed approach)
→ Engineers SSH into systems, grep logs
→ Only after someone REPORTS an issue
→ Reactive, slow, unreliable

Option B: Simple CloudWatch Alarms on individual metrics
→ One alarm per component
→ Each alarm sends to SNS
→ PROBLEM: 8 components × potential flapping = alert fatigue
→ Engineers start ignoring alerts → defeats the purpose

Option C: CloudWatch Alarms + Composite Alarm + Tiered SNS (this project) ✅
→ Individual alarms detect per-component failures
→ Composite alarm aggregates into "pipeline health" signal
→ Tiered SNS topics route by severity
→ Dashboard provides visual overview
→ Balanced: detailed monitoring + reduced noise

Option D: Third-party monitoring (Datadog, New Relic, PagerDuty)
→ More features: anomaly detection, ML-based alerts
→ Expensive: $15-25/host/month + data ingestion costs
→ Requires agent installation on EC2
→ Better for large organizations with 50+ services
→ Can integrate with Option C (SNS → PagerDuty webhook)

Option E: AWS-native observability (CloudWatch + X-Ray + DevOps Guru)
→ X-Ray: distributed tracing (more for microservices)
→ DevOps Guru: ML-based anomaly detection
→ Overkill for batch data pipelines

DECISION: Option C — CloudWatch + Composite Alarms + Tiered SNS
→ Reason: AWS-native, cost-effective, comprehensive
→ Covers all pipeline components
→ Graduated severity prevents alert fatigue
→ Dashboard satisfies CTO's visibility requirement

✅ WHAT SHOULD DO

1. Create SEPARATE SNS topics per severity level
   → pipeline-warnings: email only
   → pipeline-critical: email + SMS
   → pipeline-sla-breach: email + SMS + PagerDuty
   → Prevents: on-call woken up for non-critical warnings
   → Enables: different subscriber lists per severity

2. Use Composite Alarms for aggregated pipeline health
   → Individual alarm flaps for 30 seconds → noise
   → Composite: "ALARM if (Glue FAILED) AND (DLQ > 0)" → signal
   → Reduces false positives dramatically

3. Set alarm evaluation to MULTIPLE datapoints
   → Don't alarm on first metric breach
   → "3 out of 5 evaluation periods" = sustained problem, not blip
   → Exception: DLQ messages > 0 should alarm on first occurrence (data loss risk)

4. Include CONTEXT in alarm descriptions
   → Bad: "Glue job failed"
   → Good: "Glue ETL job 'quickcart-orders-etl' failed. 
            Runbook: https://wiki/runbook/glue-etl-failure
            Dashboard: https://console.aws.../dashboard/pipeline
            Last successful run: check CloudWatch Logs
            Impact: Gold layer will have stale data until rerun"
   → On-call engineer at 3 AM needs ACTIONABLE information

5. Add OK actions (not just alarm actions)
   → When alarm returns to OK → send "resolved" notification
   → Engineers know problem is fixed without checking manually
   → Prevents: "Was that alarm real? Is it still broken?"

6. Create metric filters for LOG-based errors
   → Glue job logs contain error stack traces
   → Metric filter: count occurrences of "ERROR" or "Exception"
   → Creates a NUMERIC metric from TEXT logs
   → Alarm on that metric like any other

7. Tag all alarms consistently
   → Team=Data-Engineering, Pipeline=nightly-etl, Severity=critical
   → Enables: "show me all critical alarms for data engineering"
   → Enables: cost allocation for monitoring infrastructure

8. Document runbooks for EVERY alarm
   → Each alarm links to a wiki page
   → Runbook contains: what triggered, what to check, how to fix
   → 3 AM engineer should NOT need to think — just follow steps

❌ WHAT SHOULD NOT DO

1. DO NOT create alarms without notification actions
   → Alarm fires → nobody is notified → alarm is useless
   → EVERY alarm must have at least one SNS action

2. DO NOT send ALL alerts to the same SNS topic
   → Warning about slow Athena query at 3 AM → wakes engineer
   → Engineer ignores all future alerts → misses critical failure
   → SEVERITY-BASED ROUTING is essential

3. DO NOT alarm on every transient error
   → Single Glue job retry → self-heals in 5 minutes
   → Alarm on first failure → unnecessary page
   → Use: EvaluationPeriods > 1 for non-critical components
   → Exception: data loss scenarios (DLQ) → alarm immediately

4. DO NOT forget TreatMissingData configuration
   → If metric stops publishing → alarm state = INSUFFICIENT_DATA
   → Default: INSUFFICIENT_DATA doesn't trigger actions
   → For heartbeat metrics: TreatMissingData="breaching"
     (missing heartbeat IS the problem)
   → For error count metrics: TreatMissingData="notBreaching"
     (no data = no errors = probably fine)

5. DO NOT hardcode email addresses in CloudWatch actions
   → CloudWatch alarm actions must be SNS topic ARNs
   → Email addresses are SNS subscriptions, not alarm actions
   → Change recipients by updating SNS subscriptions (no alarm change needed)

6. DO NOT create dashboards without auto-refresh
   → CTO opens dashboard → sees stale data from 2 hours ago
   → Must set auto-refresh period (60 seconds recommended)

7. DO NOT monitor only failures — monitor ABSENCE of success
   → "Glue job didn't fail" ≠ "Glue job ran successfully"
   → Glue job might not have RUN AT ALL (scheduler failure)
   → Monitor: "expected Glue success metric at 4:30 AM, didn't arrive"
   → Use: TreatMissingData="breaching" on success metrics

🚫 WHEN NOT TO USE CLOUDWATCH FOR MONITORING

1. When you need distributed tracing across microservices
   → CloudWatch metrics = per-service
   → Cannot trace a request across Lambda → API GW → DynamoDB
   → Use: AWS X-Ray or Datadog APM

2. When you need log search across millions of records
   → CloudWatch Logs Insights: powerful but expensive at scale
   → 100 GB/day of logs × $0.005/GB scanned = $15/day just for queries
   → Use: OpenSearch (Elasticsearch) for heavy log analytics

3. When you need cross-account centralized monitoring
   → CloudWatch is per-account by default
   → Cross-account dashboards require CloudWatch cross-account setup
   → At scale: use AWS Organizations + delegated CloudWatch admin
   → Or: Datadog/Grafana for multi-account aggregation

4. When you need business-level KPI monitoring
   → "Revenue dropped 20% vs yesterday"
   → CloudWatch monitors infrastructure, not business metrics
   → Use: QuickSight, Tableau, custom BI dashboards

5. When alert rules need complex business logic
   → "Alert if Glue failed AND it's a weekday AND we're in Q4 close"
   → CloudWatch composite alarms support AND/OR but not calendar logic
   → Use: Lambda-based custom alerting

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

┌──────────────────────────────────────────────────────────────────────────────┐
│             CLOUDWATCH MONITORING — INTERNAL MECHANICS                        │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  COMPONENT 1: METRICS                                                        │
│  ════════════════════                                                        │
│  A metric is a TIME SERIES of data points                                    │
│                                                                              │
│  Structure:                                                                  │
│  → Namespace: "AWS/Glue" or "QuickCart/Pipeline" (custom)                    │
│  → MetricName: "glue.ALL.jvm.heap.used" or "PipelineLatencySeconds"         │
│  → Dimensions: key-value pairs that identify the specific resource           │
│    Example: {"JobName": "quickcart-orders-etl", "Type": "count"}            │
│  → Timestamp: when the data point was recorded                               │
│  → Value: the numeric measurement                                            │
│  → Unit: Count, Seconds, Bytes, Percent, etc.                                │
│                                                                              │
│  METRIC RESOLUTION:                                                          │
│  → Standard: 1-minute minimum period (free for AWS services)                 │
│  → High-resolution: 1-second minimum period ($0.30/metric/month)             │
│  → Data retention:                                                           │
│    - 1-second data: 3 hours                                                  │
│    - 1-minute data: 15 days                                                  │
│    - 5-minute data: 63 days                                                  │
│    - 1-hour data: 455 days (15 months)                                       │
│  → Older data is automatically aggregated to coarser resolution              │
│                                                                              │
│  COMPONENT 2: METRIC FILTERS (Log → Metric)                                 │
│  ═══════════════════════════════════════════                                  │
│  → Applied to CloudWatch Log Groups                                          │
│  → Pattern: text match or JSON field match                                   │
│  → When pattern matches a log event → increment a metric                     │
│                                                                              │
│  HOW IT WORKS:                                                               │
│  1. Log event arrives: "ERROR: Glue job failed with OOM"                     │
│  2. Metric filter pattern: "ERROR"                                           │
│  3. Match found → increment metric "GlueErrorCount" by 1                     │
│  4. Metric now has a data point → alarm can evaluate it                      │
│                                                                              │
│  Pattern syntax:                                                             │
│  → Simple text: "ERROR"                                                      │
│  → JSON field: { $.level = "ERROR" }                                         │
│  → Multiple terms: "ERROR" - "WARN" (ERROR but not WARN)                     │
│  → JSON numeric: { $.statusCode >= 500 }                                     │
│                                                                              │
│  COMPONENT 3: ALARMS                                                         │
│  ════════════════════                                                         │
│  → Watches ONE metric (or math expression)                                   │
│  → Evaluates threshold at regular intervals                                  │
│                                                                              │
│  ALARM STATES:                                                               │
│  ┌──────────────────┬──────────────────────────────────────────┐             │
│  │ OK               │ Metric is within acceptable range        │             │
│  │ ALARM            │ Metric breached threshold                │             │
│  │ INSUFFICIENT_DATA│ Not enough data points to evaluate       │             │
│  └──────────────────┴──────────────────────────────────────────┘             │
│                                                                              │
│  STATE TRANSITIONS:                                                          │
│  OK → ALARM:              fires AlarmActions (page the engineer)             │
│  ALARM → OK:              fires OKActions (send "resolved" message)          │
│  * → INSUFFICIENT_DATA:   fires InsufficientDataActions                      │
│                                                                              │
│  EVALUATION LOGIC:                                                           │
│  → Period: how often to evaluate (60s, 300s, etc.)                           │
│  → EvaluationPeriods: how many consecutive periods to check                  │
│  → DatapointsToAlarm: how many of those must breach threshold                │
│  → Example: Period=60s, EvaluationPeriods=5, DatapointsToAlarm=3             │
│    → "3 out of 5 consecutive 1-minute checks must breach"                    │
│    → Tolerates 2 transient spikes without alarming                           │
│                                                                              │
│  COMPONENT 4: COMPOSITE ALARMS                                               │
│  ══════════════════════════════                                               │
│  → Watches OTHER ALARMS (not metrics directly)                               │
│  → Boolean logic: AND, OR, NOT                                               │
│  → Max depth: 1 level (composites can't reference other composites)          │
│                                                                              │
│  RULE SYNTAX:                                                                │
│  → ALARM("alarm-1") AND ALARM("alarm-2")                                     │
│  → ALARM("alarm-1") OR ALARM("alarm-2") OR ALARM("alarm-3")                 │
│  → ALARM("alarm-1") AND NOT ALARM("maintenance-window")                      │
│                                                                              │
│  USE CASE:                                                                   │
│  → "Pipeline unhealthy" = Glue FAILED AND (DLQ > 0 OR DataBrew FAILED)      │
│  → Reduces alert fatigue: single Glue retry ≠ pipeline failure               │
│  → Correlated failures = real incident                                       │
│                                                                              │
│  COMPONENT 5: DASHBOARDS                                                     │
│  ════════════════════════                                                     │
│  → JSON-defined layout of widgets                                            │
│  → Widget types: metric graph, number, alarm status, log query, text         │
│  → Auto-refresh: 10s, 1m, 2m, 5m, 15m                                       │
│  → Shareable via URL (console access required)                               │
│  → Max 500 metrics per dashboard                                             │
│  → FREE: 3 dashboards, then $3/dashboard/month                              │

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Alarm fired but nobody received the notification"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: SNS email subscription not CONFIRMED
→ SNS sends confirmation email → user must click link
→ If not confirmed: subscription is "PendingConfirmation" → no delivery
→ Fix: verify all subscriptions are in "Confirmed" state
→ aws sns list-subscriptions-by-topic --topic-arn <arn>

PITFALL 2: "Too many alerts — engineer ignores them all"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Classic "boy who cried wolf" problem
→ Cause: alarms too sensitive, no severity tiers
→ 50 warnings per night → on-call mutes phone → misses critical
→ Fix: severity-based routing (warning=email, critical=SMS)
→ Fix: composite alarms reduce noise
→ Fix: tune thresholds based on historical data

PITFALL 3: "Alarm in INSUFFICIENT_DATA state — never evaluates"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: metric never published (job didn't run)
→ Alarm needs data points to evaluate → no data = INSUFFICIENT_DATA
→ For scheduled jobs: this IS a problem (expected metric didn't arrive)
→ Fix: TreatMissingData="breaching" for expected-to-arrive metrics
→ Fix: publish a "0" metric on success, so absence = failure

PITFALL 4: "Dashboard shows metric but alarm doesn't fire"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: alarm dimensions don't match metric dimensions EXACTLY
→ Metric published with: {"JobName": "my-job", "Type": "count"}
→ Alarm watches: {"JobName": "my-job"} (missing "Type" dimension)
→ Different dimensions = different metric = alarm sees nothing
→ Fix: ensure alarm dimensions EXACTLY match published metric dimensions

PITFALL 5: "Composite alarm logic is wrong — fires incorrectly"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: AND vs OR confusion
→ "ALARM(glue) OR ALARM(databrew)" fires when EITHER fails → too noisy
→ Wanted: "ALARM(glue) AND ALARM(databrew)" fires only when BOTH fail
→ Fix: carefully design composite rule
→ Test: manually set alarm states and verify composite behavior

PITFALL 6: "SMS notifications cost money and hit limits"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ SNS SMS: $0.00645 per message (US)
→ Default spend limit: $1.00/month → ~155 messages then STOPS
→ If limit hit: critical SMS not delivered
→ Fix: request spend limit increase via AWS Support
→ Fix: use SMS only for critical, not warnings

PITFALL 7: "Metric filter matches too broadly"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Filter pattern: "ERROR"
→ Matches: "ERROR: job failed" ✅
→ Also matches: "No ERROR found" ❌ (false positive)
→ Also matches: "PROCESSING ErrorHandler class" ❌ (false positive)
→ Fix: use specific patterns: "[ERROR]" or { $.level = "ERROR" }
→ Fix: test filter against real logs before creating alarm
