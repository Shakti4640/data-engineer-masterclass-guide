   - Redshift COPY at 5 AM

→ Problem: partner "ShipFast" sends order updates at RANDOM times

   - They drop a file in S3: s3://bucket/incoming/shipfast/updates_*.csv

→ Better approach:
   - S3 event → sends message to SQS queue (Project 18 pattern)
   - EC2 worker POLLS the queue continuously
   - Message arrives ONLY when file lands → zero wasted calls
   - SQS guarantees: no message lost, no duplicate processing
   - If EC2 reboots: unprocessed messages stay in queue

CTO: "We need a reliable worker that processes files as they arrive."
YOU: "SQS + EC2 long polling. The queue holds work items.
      The worker asks the queue: 'Got anything for me?'
      If yes → process. If no → wait quietly for 20 seconds.
      Zero wasted cycles. Zero lost messages."

🟡 HOW TO APPROACH IT

Step 1: Understand the PRODUCER → QUEUE → CONSUMER pattern
→ Producer: S3 event notification sends message to SQS (Project 18)
→ Queue: SQS holds messages until a consumer processes them
→ Consumer: EC2 worker polls queue, receives message, processes, deletes
→ This is the fundamental async processing pattern in AWS

Step 2: Understand SQS polling mechanics
→ Short polling: returns immediately (even if queue empty) — wasteful
→ Long polling: waits up to 20 seconds for a message — efficient
→ Long polling reduces: empty responses, API calls, cost
→ ALWAYS use long polling for worker patterns

Step 3: Understand the message lifecycle
→ Message sent to queue → sits in queue (invisible to no one)
→ Worker calls receive_message → gets message → message becomes INVISIBLE
→ Visibility timeout starts (default 30 seconds)
→ Worker processes the message
→ Worker calls delete_message → message permanently removed
→ If worker CRASHES before delete → visibility timeout expires
→ Message becomes visible again → another worker picks it up
→ This is how SQS guarantees NO MESSAGE LOSS

Step 4: Understand the worker loop
→ while True:
→   messages = sqs.receive_message(WaitTimeSeconds=20)  # Long poll
→   if messages:
→     for msg in messages:
→       process(msg)        # Do the actual work
→       sqs.delete(msg)     # Acknowledge completion
→   # If no messages: loop waits 20 sec then retries (long poll)

✅ WHAT SHOULD DO

1. ALWAYS use long polling (WaitTimeSeconds=20)
   → Short polling returns empty immediately → wasted API call
   → Long polling waits up to 20 seconds → efficient
   → Reduces SQS API costs by 90%+
   → 20 seconds is the maximum wait time

2. ALWAYS set visibility timeout > processing time
   → If processing takes 2 minutes, set timeout to 5 minutes (300s)
   → If timeout < processing time: message reappears while still being processed
   → Result: same message processed TWICE (duplicate work)
   → Rule: visibility timeout = 6× expected processing time

3. ALWAYS delete message AFTER successful processing
   → delete_message() is the acknowledgment
   → Without delete: message reappears after visibility timeout
   → Delete ONLY after confirmed success (file processed, data loaded)

4. ALWAYS configure a Dead Letter Queue (DLQ)
   → Messages that fail repeatedly go to DLQ instead of infinite retry
   → maxReceiveCount=3: after 3 failures → moved to DLQ
   → Monitor DLQ depth with CloudWatch alarm → SNS alert
   → Prevents poison messages from blocking the queue forever

5. ALWAYS handle graceful shutdown
   → Worker receives SIGTERM → finish current message → then exit
   → Don't kill mid-processing → leaves message invisible until timeout
   → Use signal handlers in Python

6. ALWAYS receive messages in batches
   → MaxNumberOfMessages=10 (maximum)
   → One API call returns up to 10 messages
   → Process all 10 → delete all 10
   → Reduces API calls by 10x

7. ALWAYS parse S3 event structure correctly
   → SQS message body contains S3 event JSON
   → May be wrapped in SNS notification (if S3 → SNS → SQS path)
   → Extract: bucket, key, size, event time
   → Handle: nested JSON, multiple records per event

❌ WHAT SHOULD NOT DO

1. DO NOT use short polling in a tight loop
   → while True: sqs.receive_message()  # No WaitTimeSeconds
   → Returns empty instantly → loops 1000s of times per minute
   → SQS charges per API call ($0.40 per million)
   → 1000 calls/min × 1440 min/day = 1.44M calls/day = $0.58/day wasted

2. DO NOT delete message BEFORE processing
   → delete_message() → then process() → process fails
   → Message already gone → work item LOST forever
   → ALWAYS: receive → process → delete (in this order)

3. DO NOT set visibility timeout too short
   → Timeout 30s but processing takes 120s
   → After 30s: message reappears → another worker picks it up
   → Same file processed by two workers simultaneously
   → Data corruption, duplicates, race conditions

4. DO NOT ignore the receipt handle
   → receive_message() returns ReceiptHandle (not MessageId)
   → delete_message() requires ReceiptHandle
   → ReceiptHandle changes every time message is received
   → Using old receipt handle → delete fails

5. DO NOT process messages without idempotency
   → SQS Standard queue may deliver same message TWICE
   → Worker must handle: "I already processed this file"
   → Track processed files in: DynamoDB table, S3 marker, database flag
   → Or use SQS FIFO queue (exactly-once, but lower throughput)

6. DO NOT let the worker run without health monitoring
   → Worker crashes silently → messages pile up in queue
   → Set CloudWatch alarm on: ApproximateAgeOfOldestMessage
   → If oldest message > 30 minutes → worker is probably dead
   → SNS alert → restart worker

7. DO NOT hardcode queue URL
   → Queue URL changes between environments
   → Use environment variable or parameter store
   → Or: look up by queue name via sqs.get_queue_url()

🚫 WHEN NOT TO USE SQS + EC2 POLLING

1. When processing must happen in < 1 second
   → Long polling adds up to 20 seconds latency
   → Use: API Gateway + Lambda (synchronous)
   → Or: Kinesis with sub-second processing

2. When messages must be processed in EXACT order
   → SQS Standard: best-effort ordering (may deliver out of order)
   → Use: SQS FIFO queue (guarantees order within message group)
   → Or: Kinesis (ordered within shard)

3. When you need exactly-once processing guaranteed by the queue
   → SQS Standard: at-least-once (may deliver duplicates)
   → Use: SQS FIFO with deduplication
   → Or: implement idempotency in your worker (recommended regardless)

4. When workload is extremely bursty (0 to 10,000 messages/second)
   → Single EC2 worker can't handle 10K msg/sec
   → Use: Lambda with SQS trigger (auto-scales to concurrency)
   → Or: EC2 Auto Scaling Group based on queue depth (Project 52)

5. When there's no persistent processing state needed
   → EC2 worker is overkill for stateless transforms
   → Lambda is simpler for: parse JSON, write to DynamoDB, send email

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Same file processed twice"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: SQS Standard delivers at-least-once (may duplicate)
→ Fix: implement idempotency
   → Before processing: check if file already processed
   → Track in: DynamoDB table, S3 marker file, or database flag
   → If already processed: delete message, skip processing

PITFALL 2: "Messages pile up — worker can't keep up"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: processing time per message > arrival rate
→ Debug: CloudWatch → ApproximateNumberOfMessagesVisible (growing?)
→ Fix A: increase worker processing speed
→ Fix B: add more workers (Project 52 — auto-scaling)
→ Fix C: batch processing (receive 10 messages, process in parallel)

PITFALL 3: "Worker processes message but crashes before delete"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ This is EXPECTED behavior — SQS designed for this
→ Message reappears after visibility timeout → processed again
→ BUT: work may have been partially done (half the rows loaded)
→ Fix: make processing IDEMPOTENT
   → Truncate target before loading (for full replacement)
   → Or: use upsert/merge logic (for incremental)
   → Or: use transaction: all-or-nothing

PITFALL 4: "Visibility timeout too short — duplicate processing"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Timeout 30s, processing takes 120s
→ At T=30s: message reappears → Worker B picks it up
→ Now Worker A AND Worker B processing same message
→ Fix: set timeout = 6× expected processing time
→ Or: use change_message_visibility() to extend timeout during processing

PITFALL 5: "DLQ fills up but nobody notices"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: no monitoring on DLQ
→ Poison messages sit for 14 days then auto-delete
→ Data silently lost
→ Fix: CloudWatch alarm on DLQ ApproximateNumberOfMessagesVisible > 0
→ → SNS notification → investigate and reprocess

PITFALL 6: "S3 event JSON parsing fails"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Cause: S3 event is nested inside SQS message body
→ If S3 → SNS → SQS: body is SNS wrapper containing S3 event
→ If S3 → SQS directly: body IS the S3 event
→ Must handle BOTH formats
→ Always try-parse and log raw message on failure

📈 SCALABILITY CONSIDERATIONS

SINGLE WORKER THROUGHPUT:
━━━━━━━━━━━━━━━━━━━━━━━━
→ 1 message at a time: ~30 messages/minute (2-sec processing each)
→ 10 messages batch: ~100 messages/minute (parallel processing)
→ With threading: ~200 messages/minute per EC2 instance

SCALING STRATEGIES:
━━━━━━━━━━━━━━━━━━
→ Vertical: larger EC2 instance → more threads → more throughput
→ Horizontal: multiple EC2 instances polling same queue
   → SQS distributes messages across consumers automatically
   → No coordination needed — each gets unique messages
→ Auto-scaling: scale EC2 count based on queue depth (Project 52)
   → CloudWatch metric: ApproximateNumberOfMessagesVisible
   → > 100 messages → add worker
   → 0 messages for 10 min → remove worker

SQS LIMITS:
━━━━━━━━━━━
→ Standard queue: nearly unlimited throughput
→ FIFO queue: 3,000 messages/sec with batching
→ Message size: max 256 KB
→ Retention: 1 minute to 14 days (default 4 days)
→ Visibility timeout: 0 to 12 hours
→ Long poll wait: 0 to 20 seconds

🔍 INTERPRETABILITY CONSIDERATIONS

   → Log: message received, processing started, success/failure, deleted

   → Custom CloudWatch metrics from worker:

     → processing_time_ms (gauge)
     → errors (counter)
